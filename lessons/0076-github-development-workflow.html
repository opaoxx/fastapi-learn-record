0076 GitHub Development Workflow：把课程维护流程写成可测试协作契约

② 核心概念底层原理

项目维护不是只会写代码。课程仓库同时包含源码、测试、讲义、速查、学习记录、索引和 README，任何一个环节漏更新，读者都会在 GitHub 上遇到断链或过期说明。

DEVELOPMENT.md 是给维护者看的协作契约。README 面向学习者，DEVELOPMENT 面向修改者。二者分工清楚，才能避免根 README 变得过长，也避免维护流程散落在聊天记录里。

本节继续使用文档契约测试。原因很简单：流程文档一旦成为公共协作入口，它就不应只靠人肉记忆维护。测试可以保护关键命令、九段结构和 README 链接不被未来改动误删。

文档读者职责
README.md学习者、首次访问者项目定位、学习入口、运行测试入口
DEVELOPMENT.md维护者、贡献者改代码、改课程、改文档、发布前检查
COURSE-STANDARD.md课程作者课程质量原则和写作标准
tests/test_course_docs_contract.py自动化测试保护文档入口和课程结构

复核补强：DEVELOPMENT.md 的价值是把“怎么安全改这个仓库”从个人经验变成团队契约。课程仓库同时有代码和讲义，维护者必须知道改一节课会牵动 lesson、reference、index、record、notes、progress 和测试。

维护契约保护对象不写清楚的后果
Code Change Checklist实现与测试一致功能改了但测试遗漏，回归风险上升。
Lesson Change Checklist课程导航和记录完整新增课程变成孤岛页面。
Documentation Change ChecklistREADME、索引和参考页同步GitHub 入口与真实内容脱节。
Release Readiness发布前验证闭环带着断链、空白错误或过期说明交付。

③ 全套代码逐行精读解析

# Development Workflow

This repository is both a runnable FastAPI project and a course archive.

开头明确仓库双重身份：既是可运行项目，又是课程档案。维护者必须同时考虑代码正确性和学习材料完整性。

## Code Change Checklist

- Update the implementation in `first_api/`.
- Add or update tests in `tests/`.
- Run the targeted test file first.
- Run the full test suite before considering the change complete.

代码修改清单强调小步验证。先跑目标测试，再跑全量测试，可以更快定位错误，也能防止局部通过但整体破坏。

.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q
.\.venv\Scripts\python.exe -m pytest tests\test_course_docs_contract.py -q
.\.venv\Scripts\python.exe -m pytest -q
git diff --check

这些命令是当前仓库的维护底线：provider 逻辑测试、文档契约测试、全量测试和空白检查。写进 DEVELOPMENT 后，新贡献者不用从历史记录里猜。

## Lesson Change Checklist

Every new lesson should include:

- a complete HTML lesson in `lessons/`;
- a quick reference page in `reference/`;
- an entry in `index.html`;
- a learning record in `learning-records/`;
- an update to `NOTES.md`;
- an update to `COURSE_PROGRESS.md`;

课程新增不是只写一个 HTML。lesson、reference、index、learning record、notes、progress 共同组成可追踪学习资料。

1. `① 本节核心知识框架`
...
9. `⑨ 面试高频真题+解析`

DEVELOPMENT 也重复固定九段结构，是为了让维护者在动手前就看到硬标准。这个结构同时被课程页面测试保护。

def test_development_workflow_documents_course_maintenance_contract() -> None:
    workflow = Path("DEVELOPMENT.md").read_text(encoding="utf-8")

测试直接读取 DEVELOPMENT 文本。它不解析 Markdown AST，因为本节保护的是关键短语和命令是否存在。

assert "Code Change Checklist" in workflow
assert ".\\.venv\\Scripts\\python.exe -m pytest -q" in workflow
assert "git diff --check" in workflow
assert "① 本节核心知识框架" in workflow
assert "⑨ 面试高频真题+解析" in workflow

这些断言保护维护流程的关键骨架。未来如果有人删除测试命令或九段结构，文档契约测试会失败。

逐行精读补充：测试同时断言命令和九段式结构，是因为本仓库的质量边界有两类：代码必须能跑、课程必须完整。只测 pytest 命令会漏掉课程写作标准，只测九段结构又无法保护维护者的验证流程。

这里没有测试 DEVELOPMENT 的每一行文字，避免文档被锁死。契约测试只保护不可丢的骨架：检查清单名称、关键命令、发布准备和课程固定结构。

④ 核心机制运行全流程拆解

New contributor
  → README.md
  → DEVELOPMENT.md
  → choose change type
     ├─ code change
     ├─ lesson change
     └─ documentation change
  → update matching files
  → run targeted tests
  → run full pytest
  → run git diff --check

这条流程把维护动作从“凭经验”改成“按清单执行”。对于课程仓库尤其重要，因为课程资料比普通代码项目多了导航、速查、学习记录和进度文件。

修改类型必须关注推荐验证
代码first_api/ 与 tests/目标测试 + 全量测试
课程lessons/、reference/、index.html文档契约测试
README入口链接和命令README 契约测试
发布导航、测试、空白检查git diff --check

易混点错误理解正确理解

⑧ 课后记忆习题+标准答案

1. 为什么需要单独的 DEVELOPMENT.md？

README 面向学习入口，DEVELOPMENT 面向维护流程。分开后根 README 不会过长，维护规则也更集中。

2. 新增课程为什么要同时改多个文件？

课程页、速查页、目录、学习记录、NOTES 和进度共同构成可导航、可追踪的学习产品。

3. 为什么文档契约测试要检查 DEVELOPMENT？

维护命令和九段结构是课程质量底线，不能依赖人工记忆，应该被测试保护。

4. git diff --check 检查什么？

它检查 diff 中的空白问题，例如尾随空格等。当前 Windows LF-to-CRLF 提示不是空白错误。

5. 生产仓库会如何升级这个流程？

生产仓库会用 CI/CD、分支保护、review、自动部署、安全扫描和审计机制强制执行流程。

⑨ 面试高频真题+解析

面试题 1：为什么工程项目需要开发流程文档？

开发流程文档把隐性经验显式化，降低新人上手成本，并减少漏跑测试、漏更新文档、误发布等问题。

面试题 2：README 和 DEVELOPMENT 如何分工？

README 面向使用者和第一次访问者，DEVELOPMENT 面向维护者和贡献者。前者给入口，后者给操作流程。

面试题 3：为什么课程仓库也需要质量门禁？

课程仓库的产品是学习体验。断链、缺速查、漏测试和结构不一致都会破坏学习体验，因此也需要门禁。

面试题 4：如何把本地维护流程迁移到 CI？

把目标测试、文档契约测试、全量测试和空白检查写入 CI workflow，并设置 PR 必须通过检查后才能合并。
