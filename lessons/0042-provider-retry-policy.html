Lesson 0042

Provider Retry Policy：给外部 AI 调用加有限重试

上一课把预测业务编排放进了 service 层，本课继续把外部 provider 的稳定性策略放在 adapter 层：当 AI provider 暂时超时、网络抖动、返回 429 或部分 5xx 时，我们允许有限次数重试；当请求本身错误或响应契约错误时，立刻失败。

0. 本节学习目标

  • 理解 retry 不是“失败了再试试”这么简单，而是一条明确的工程策略。
  • 区分 retryable failure 和 non-retryable failure。
  • 理解为什么 max_attempts 是重试预算，不是无限循环。
  • 学会把 retry 放在 provider adapter 层，而不是散落到 router 或 service。
  • 用 httpx.MockTransport 写测试，证明哪些错误会重试，哪些错误不会重试。

1. 背景：外部 AI provider 的失败常常是“临时”的

真实 AI 服务不是本地函数调用。你的 FastAPI 应用发出一次 provider 请求时，中间会经过 DNS、TCP/TLS、代理、provider 网关、模型排队、限流系统和响应序列化。任意一层短暂抖动，都可能让一次本来可成功的请求失败。

但是 retry 也不是越多越好。AI provider 往往计费、限流，而且一次预测请求可能占用 GPU 排队资源。如果你无限重试，短暂故障会被放大成更高的成本和更重的 provider 压力。

所以本节的核心不是“加一个 for 循环”，而是建立一个判断模型：什么失败值得再试，什么失败必须立刻暴露出来。

2. 核心概念：retry policy 是一组边界规则

Retry policy 指的是调用外部依赖失败后，客户端是否再次尝试、最多尝试几次、哪些错误可以重试、哪些错误不能重试的规则集合。

可以谨慎重试

  • TimeoutException：provider 或网络一段时间没有响应。
  • RequestError：请求发出阶段或网络层发生错误。
  • 429：provider 表示当前请求过多。
  • 500/502/503/504：provider 或网关一侧出现临时服务问题。

不应该重试

  • 400：请求格式或参数错了，再发一次还是错。
  • 401/403：认证或权限错了，重试不能修复密钥。
  • 404：URL 或资源不存在，通常不是瞬时问题。
  • 非法 JSON 或字段校验失败：provider 响应契约坏了，应该暴露。

HTTPX 官方异常层级里，TimeoutException、网络错误和 HTTPStatusError 是不同类型；response.raise_for_status() 会把 4xx/5xx 响应转成 HTTPStatusError。这正好给我们提供了编写 retry policy 的分支边界。

3. 状态码原则：为什么 429 和 5xx 比 400 更适合重试

429 表示 Too Many Requests，也就是请求速率过高。RFC 6585 说明 429 响应可以携带 Retry-After，告诉客户端多久以后再试。503 表示 Service Unavailable，RFC 9110 把它描述为临时过载或维护，并且也可以配合 Retry-After。

本项目现在先做“有限立即重试”：只看 max_attempts，不读取 Retry-After，也不加入指数退避。原因是教学节奏要小步走：先把“哪些错误可以重试”讲清楚，再在后续课程加入 backoff、jitter 和 provider 限流礼貌。

失败类型 本节策略 原因
Timeout / RequestError 重试到 max_attempts 可能是网络或连接的短暂抖动。
429 重试到 max_attempts 限流通常是临时状态，后续会学习 Retry-After。
500/502/503/504 重试到 max_attempts 服务端或网关侧临时失败，下一次可能恢复。
400/401/403/404 不重试 客户端输入、认证、权限或路径问题，不会靠重复请求变好。
非法响应体 不重试 契约已经被破坏，应该尽快暴露给日志、测试和调用方。

4. 运行机制：一次 provider prediction 的 retry 流程

request_provider_prediction()
  -> create or reuse AsyncClient
  -> for attempt in 1..max_attempts
       -> POST provider prediction payload
       -> raise_for_status()
       -> validate provider JSON with Pydantic
       -> success: return internal PredictionResult
       -> timeout/request error: retry if attempts remain
       -> 429 or 5xx: retry if attempts remain
       -> 400/401/403/404: raise AIClientError immediately
       -> invalid JSON/schema: raise AIClientError immediately
  -> close owned client

注意这个流程仍然保留原来的内部错误契约：对外层 service 来说，adapter 失败仍然是 AIClientError，只是 adapter 在抛出错误前多做了有限次数的尝试。

5. 代码精读一：可重试状态码集合

RETRYABLE_PROVIDER_STATUS_CODES = {429, 500, 502, 503, 504}

def is_retryable_provider_status(status_code: int) -> bool:
    return status_code in RETRYABLE_PROVIDER_STATUS_CODES

把状态码集合放在模块顶部，有两个好处。第一，规则是显式的，读者一眼能知道本项目认为哪些 HTTP 状态是临时失败。第二，测试和后续课程可以围绕这个策略扩展，而不是把魔法数字散落在多个 except 分支里。

这里没有把所有 5xx 都重试到失控。我们选了真实网关和 provider 调用中最常见的临时失败：500、502、503、504；再加上 429。

6. 代码精读二：max_attempts 是重试预算

for attempt in range(1, config.max_attempts + 1):
    try:
        response = await active_client.post(...)
        response.raise_for_status()
        body = ProviderPredictionBody.model_validate(response.json())
        return PredictionResult(...)
    except httpx.TimeoutException as exc:
        if attempt < config.max_attempts:
            continue
        raise AIClientError(...) from exc

range(1, config.max_attempts + 1) 表示总尝试次数，而不是“失败后额外重试次数”。如果 max_attempts=1，函数只请求一次；如果 max_attempts=3，最多请求三次。

这种命名对初学者很重要。很多系统会把配置命名为 retries，这时 retries=3 往往表示“第一次之外再试三次”，总请求数可能是四次。我们这里用 max_attempts，避免这个歧义。

7. 代码精读三：HTTP 错误要分状态码

except httpx.HTTPStatusError as exc:
    status_code = exc.response.status_code
    if (
        attempt < config.max_attempts
        and is_retryable_provider_status(status_code)
    ):
        continue
    raise AIClientError(
        message=f"The AI provider returned HTTP {status_code}.",
        error_code="ai_provider_http_error",
    ) from exc

这里不能简单地说“HTTP 错误都重试”。HTTPStatusError 同时覆盖 4xx 和 5xx。400、401、403 这类错误重试没有意义，因为问题来自请求本身或认证状态；503 这类错误才更像 provider 暂时无法服务。

因此，HTTPStatusError 分支先取出 status_code，再用 is_retryable_provider_status() 判断。这个判断就是本节最核心的工程策略。

8. 代码精读四：非法响应不重试

except (ValueError, ValidationError) as exc:
    raise AIClientError(
        message="The AI provider returned an invalid prediction response.",
        error_code="ai_provider_invalid_response",
    ) from exc

response.json() 可能因为不是合法 JSON 而失败，ProviderPredictionBody.model_validate(...) 可能因为字段不符合契约而失败。本节选择不重试这类错误。

为什么？因为这不是“网络路上抖了一下”，而是 provider 已经返回了一个我们不能信任的结果。此时正确动作是暴露错误、保留日志和测试证据，而不是悄悄再发一次，掩盖 provider 契约问题。

9. 为什么 retry 放在 adapter，而不是 router 或 service

上一课我们刚把预测编排抽到 services/predictions.py。那本课为什么不在 run_prediction() 里重试？答案是：retry 的判断依赖 provider HTTP 细节，比如 httpx.TimeoutException、HTTPStatusError、状态码集合和响应体校验。这些都属于 provider adapter 知识。

如果把 retry 放进 router，HTTP 协议翻译层会变胖；如果放进 service，业务编排层会知道太多 provider 传输细节。放在 request_provider_prediction() 里，边界最干净：adapter 负责“如何可靠地和 provider 说话”，service 负责“预测用例怎么跑”，router 负责“怎么翻译成 HTTP 响应”。

10. 测试精读：用 MockTransport 证明 retry 规则

def test_request_provider_prediction_retries_retryable_http_error_then_succeeds():
    attempts = 0

    def handler(request):
        nonlocal attempts
        attempts += 1
        if attempts == 1:
            return httpx.Response(status_code=503, request=request)
        return httpx.Response(
            status_code=200,
            json={"label": "positive", "score": 0.93},
            request=request,
        )

    result = run_async_prediction(client, max_attempts=2)

    assert attempts == 2
    assert result.label == "positive"

这个测试刻意让第一次请求返回 503，第二次返回成功。它验证的不是“函数最后能成功”这么简单，而是验证 adapter 确实把 503 识别成 retryable failure，并且没有超过预算。

本课还加入了三个保护性测试：连续 503 最多请求 max_attempts 次；400 只请求一次；非法响应体只请求一次。它们一起证明 retry policy 的边界没有被写宽。

12. 常见错误与诊断

错误一：所有异常都重试

这会掩盖 400、401、非法 JSON 等确定性错误。诊断方式是写“不可重试错误只调用一次”的测试。

错误二：没有最大次数

无限 retry 会放大故障和成本。任何 retry policy 都必须有预算，例如 max_attempts。

错误三：在 router 里写 retry

router 会被 provider 细节污染。HTTPX 异常、状态码策略和响应体验证更适合放在 adapter。

错误四：忽略 POST 的副作用

不是所有 POST 都适合重试。预测请求在本项目里被设计成读式推理；如果 provider 会创建任务或扣费，策略要更谨慎。

13. 面试视角：怎么解释 retry policy

我会把外部 provider 的 retry 放在 adapter 层，因为它依赖 HTTPX 异常、HTTP 状态码和 provider 响应契约。策略上，我只对超时、网络错误、429 以及常见临时 5xx 做有限重试，用 max_attempts 控制总尝试次数；对 400、401、403、404 和非法响应体不重试。这样可以提升短暂故障下的成功率，同时避免把确定性错误、认证问题或 provider 契约问题隐藏起来。

14. 小测：先回忆，再展开

1. max_attempts=3 表示总共请求几次？

最多请求三次。它是总尝试次数，不是第一次失败后的额外重试次数。

2. 为什么 400 不应该自动重试？

400 通常表示请求格式或参数错误；同一个错误请求再发一次，大概率仍然失败。

3. 为什么非法 provider 响应体不重试？

因为 provider 已经返回了无法通过契约校验的数据，这属于契约问题，应该暴露而不是掩盖。

4. retry 为什么适合放在 adapter 层？

因为 retry 判断依赖 provider HTTP 细节，例如 HTTPX 异常、状态码和响应体验证；这些属于 adapter 的职责。

15. 本节总结

本节给 provider prediction adapter 加入了有限 retry。现在 request_provider_prediction() 会根据 config.max_attempts 控制总尝试次数，对 timeout、request error、429 和常见 5xx 做谨慎重试，对客户端错误和非法响应体立刻失败。更重要的是，你现在知道 retry 是一条工程策略，而不是一个盲目的循环。

下一课建议学习：Backoff Before Retry。我们会讨论为什么真实系统不能连续立即重试，以及如何用等待策略降低雪崩风险。
