0042 Provider Retry Policy：给外部 AI 调用增加有限重试

Retry 不是“失败了再试试”的本能反应，而是一套有边界的工程策略：哪些故障可重试，最多尝试几次，哪些故障必须立即失败，重试逻辑应该放在哪一层。本节把这些问题收敛到 provider adapter 内部。

① 本节核心知识框架

Provider Retry Policy
├─ 重试预算：AIClientConfig.max_attempts
├─ 可重试故障：timeout / request error / 429 / 500 / 502 / 503 / 504
├─ 不可重试故障：400 / 401 / 403 / 404 / invalid JSON / schema validation
├─ 执行位置：request_provider_prediction() adapter
├─ 成功退出：任一尝试成功即返回 PredictionResult
├─ 耗尽退出：最后一次仍失败 → AIClientError
└─ 测试边界：attempts 计数 + MockTransport + caplog

核心结论：重试策略必须同时提升短暂故障下的成功率，并避免把确定性错误、认证错误和契约错误隐藏起来。

② 核心概念底层原理

外部 AI provider 失败来源复杂：网络抖动、DNS、TLS、网关、队列、限流、模型服务临时不可用都可能导致一次请求失败。但重试也会放大流量、成本和供应商压力，所以必须有重试预算。

故障类型是否重试原因
Timeout / RequestError是可能是瞬时网络或连接问题
429是限流通常具有时间窗口，后续可结合 Retry-After
500/502/503/504是服务端或网关临时故障，下一次可能恢复
400/401/403/404否请求、认证、权限或路径错误，重复发送通常仍失败
invalid JSON / ValidationError否provider 契约被破坏，应暴露问题而非掩盖

max_attempts 表示总尝试次数，不是额外 retry 次数。max_attempts=3 代表最多发送 3 次请求，而不是 1 次初始请求加 3 次重试。

③ 全套代码逐行精读解析

RETRYABLE_PROVIDER_STATUS_CODES = {429, 500, 502, 503, 504}

将策略常量显式放在模块顶部，使读者和测试都能看到“哪些 HTTP 状态被视为临时故障”。不要把魔法数字散落在多个 except 分支里。

def is_retryable_provider_status(status_code: int) -> bool:
    return status_code in RETRYABLE_PROVIDER_STATUS_CODES

小函数让策略具名化。后续扩展分类、metrics、日志时，可以复用统一判断。

for attempt in range(1, config.max_attempts + 1):

尝试编号从 1 开始，便于日志和测试表达“第几次尝试”。+ 1 是 Python range 右开区间要求；漏写会少尝试一次。

response = await active_client.post(
    url,
    json=ProviderPredictionPayload(text=text, mode=mode).model_dump(),
)
response.raise_for_status()
body = ProviderPredictionBody.model_validate(response.json())

每次尝试都完整执行一次 provider 调用与校验。只有 HTTP 状态、超时、请求错误进入 retry 判断；响应体校验失败不会 retry。

except httpx.TimeoutException as exc:
    if attempt < config.max_attempts:
        await wait_before_retry(...)
        continue
    log_retry_exhausted(...)
    raise AIClientError(..., error_code="ai_provider_timeout") from exc

超时可重试，但只在还有预算时继续。最后一次仍失败时记录 retry exhausted，再抛稳定错误码。

except httpx.HTTPStatusError as exc:
    status_code = exc.response.status_code
    retryable_status = (
        classify_provider_http_status(status_code) == "retryable_http_status"
    )
    if attempt < config.max_attempts and retryable_status:
        await wait_before_retry(...)
        continue

HTTP 错误不能一概重试。先拿状态码，再通过分类函数判断。400 会 fail fast；503 在预算未耗尽时重试。

except (ValueError, ValidationError) as exc:
    log_prediction_fail_fast(...)
    raise AIClientError(
        message="The AI provider returned an invalid prediction response.",
        error_code="ai_provider_invalid_response",
    ) from exc

非法响应体直接失败。原因是 provider 已经返回了不可被信任的数据，这属于契约破坏，不是瞬时网络故障。

④ 核心机制运行全流程拆解

request_provider_prediction()
  → for attempt in 1..max_attempts
      → POST provider
      → 成功：validate body → record success → return PredictionResult
      → timeout/request error：有预算则等待后重试，否则 exhausted
      → HTTP 429/5xx：有预算则等待后重试，否则 exhausted
      → HTTP 4xx：fail_fast
      → invalid JSON/schema：fail_fast
  → 若循环异常结束：抛 AIClientError

retry 放在 adapter 的原因很明确：它依赖 HTTPX 异常、HTTP 状态码、Retry-After、provider 响应校验，这些都是 provider 通信细节。router 和 service 不应知道这些细节。

⑤ 重难点、易混淆点对比辨析

易混点正确理解风险
max_attempts vs retries本项目表示总尝试次数理解错会多打 provider
所有 HTTP 错误都重试只重试 429 和常见临时 5xx认证错误、请求错误被掩盖
invalid response 也重试不重试，立即暴露契约错误schema 漂移被悄悄吞掉
retry 写在 router应写在 adapterHTTP 协议层被 provider 细节污染

⑥ 开发常见报错+坑点+解决方案

  • 无限循环重试：使用 while True 且退出条件不完整。解决：使用 range(1, max_attempts + 1) 明确预算。
  • 401 被反复重试：把所有 HTTPStatusError 都视为可恢复。解决：用 classify_provider_http_status() 分类。
  • 测试真的 sleep：retry 测试等待变慢。解决：注入 fake async sleep，断言 delay 参数。
  • POST 盲目重试导致副作用：如果 provider 会创建任务或扣费，重复 POST 可能不是安全的。解决：确认接口语义，必要时使用幂等键。

⑦ 生产环境 vs 教学环境 核心差异

维度教学环境生产环境
重试次数ai_max_attempts 1 到 5按 SLA、成本、超时预算精细配置
等待策略后续课程讲 backoff、Retry-After、jitter指数退避、抖动、限流反馈、熔断
幂等性预测视作读式推理任务型或计费型 API 必须考虑幂等键

⑧ 课后记忆习题+标准答案

1. max_attempts=3 表示最多发送几次 provider 请求？

最多 3 次。它是总尝试次数，不是额外重试次数。

2. 为什么 400 不应该自动重试？

400 通常表示请求格式或参数错误，同样请求再次发送大概率仍失败，应快速暴露。

3. 为什么无效 provider 响应不重试？

这是契约问题，provider 已返回不可被信任的数据，重试会掩盖 schema 漂移或供应商 bug。

4. retry 为什么放在 adapter 层？

retry 判断依赖 HTTPX 异常、状态码、响应头和 provider schema 校验，这些属于外部 provider 通信细节。

⑨ 面试高频真题+解析

题 1：哪些错误适合重试，哪些不适合？ 答：timeout、网络请求错误、429、临时 5xx 适合有限重试；400、401、403、404、无效 JSON、schema 校验失败不适合。

题 2：如何避免 retry storm？ 答：设置最大尝试次数、总超时预算、指数退避、jitter、熔断和限流；本节先建立 max_attempts 预算，后续再扩展等待策略。

题 3：POST 请求能否重试？ 答：取决于语义。若是纯推理读取且服务端无副作用，可以谨慎有限重试；若会创建任务、扣费或改变状态，必须使用幂等键或避免自动重试。
