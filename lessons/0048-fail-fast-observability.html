0048 Fail-fast Observability：不可重试失败也要有清晰诊断语义

Fail fast 不是“没做错误处理”，而是系统判断继续尝试没有意义后主动停止。本节为 HTTP 400 类错误与无效 provider 响应补充 provider_prediction_fail_fast 日志，防止它们被误标为 retry exhausted。

① 本节核心知识框架

Fail-fast Observability
├─ fail_fast 场景：non-retryable HTTP status / invalid response
├─ 稳定事件：provider_prediction_fail_fast
├─ 原因字段：fail_fast_reason = http_status | invalid_response
├─ 诊断字段：attempt / max_attempts / status_code / error_code
├─ 边界：不 sleep、不 continue、不 exhausted
└─ 测试：attempts == 1 且 exhausted_records == []

核心结论：不可重试失败必须快速返回，同时留下“为什么没有重试”的证据。否则线上排查会把确定性错误误判成 provider 连续失败。

② 核心概念底层原理

Fail fast 是错误分类后的主动终止：系统识别出继续请求大概率无意义，因此不消耗后续 attempt。它与 exhausted 的本质区别是：前者不该重试，后者本该重试但预算用完。

错误处理原因
HTTP 400/401/403/404fail_fast请求、认证、权限或路径问题，重复发送不能修复
invalid JSONfail_fastprovider 响应不是可解析 JSON
Pydantic ValidationErrorfail_fastprovider 响应不符合内部契约
HTTP 503 最后一次失败retry_exhausted可重试错误耗尽预算

③ 全套代码逐行精读解析

FailFastReason = Literal["http_status", "invalid_response"]

fail-fast 原因被收窄为稳定枚举，避免日志里一会儿写 bad_status、一会儿写 invalid_body。

def log_prediction_fail_fast(
    attempt: int,
    config: AIClientConfig,
    fail_fast_reason: FailFastReason,
    error_code: str,
    status_code: int | None = None,

helper 参数与诊断语义直接对应：第几次尝试、最多几次、为什么 fail fast、最终稳定错误码、是否有 HTTP 状态码。

logger.warning(
    "provider_prediction_fail_fast",
    extra={
        "attempt": attempt,
        "max_attempts": config.max_attempts,
        "fail_fast_reason": fail_fast_reason,
        "status_code": status_code,
        "error_code": error_code,
    },

使用 warning，因为本次 provider prediction 最终失败；事件名不是 exhausted，避免表达“预算用完”的错误语义。

if retryable_status:

HTTP 分支先分类。400 类状态不进入等待、不进入 exhausted，而是直接 fail fast。

except (ValueError, ValidationError) as exc:

无效响应体也只尝试一次。status_code=200 这类日志尤其有价值：它说明传输成功，但业务契约失败。

④ 核心机制运行全流程拆解

HTTP 400 路径
  → attempt 1
  → response.raise_for_status()
  → HTTPStatusError(status_code=400)
  → classify = non_retryable_http_status
  → log provider_prediction_fail_fast(reason=http_status)
  → raise AIClientError(ai_provider_http_error)
  → attempts == 1

Invalid response 路径
  → attempt 1
  → HTTP 200
  → response.json()/Pydantic validation fails
  → log provider_prediction_fail_fast(reason=invalid_response)
  → raise AIClientError(ai_provider_invalid_response)
  → attempts == 1

这两条路径都不会 sleep，不会 continue，也不会写 retry_exhausted。这是本节最重要的语义边界。

⑤ 重难点、易混淆点对比辨析

易混点正确理解误用后果
fail fast vs 无处理fail fast 是主动分类后的停止误以为系统偷懒没有重试
400 vs 503400 多为确定性错误，503 多为临时错误400 重试会制造无意义流量
invalid response 是否重试默认不重试契约漂移被掩盖
日志事件名fail_fast 与 retry_exhausted 必须区分线上排查结论错误

⑥ 开发常见报错+坑点+解决方案

  • 400 被重试 3 次：HTTPStatusError 分支未分类。解决：用 classify_provider_http_status() 判断。
  • invalid response 被记为 exhausted：把所有异常统一打 exhausted。解决：ValueError/ValidationError 单独走 fail fast。
  • 日志缺少 status_code：无法判断 provider 是否返回响应。解决：HTTP 与 invalid response 路径都尽量记录 status_code。
  • 测试只看 error_code：未保护日志语义。解决：断言 fail_fast_records 存在且 exhausted_records == []。

⑧ 课后记忆习题+标准答案

1. fail fast 和 retry exhausted 的本质区别是什么？

fail fast 是错误分类上不应重试；retry exhausted 是错误可重试但 attempt 预算已经用完。

2. 为什么 HTTP 400 不应重试？

它通常表示请求本身或认证权限有问题，同一请求重复发送无法自动修复。

3. 为什么 invalid response 记录 status_code=200 有价值？

它说明 provider 有响应且 HTTP 层成功，但响应体不符合内部业务契约。

4. 为什么测试要断言 exhausted_records == []？

为了保护日志语义，证明 fail-fast 路径没有被误标为 retry exhausted。

⑨ 面试高频真题+解析

题 1：哪些 provider 错误应该 fail fast？ 答：非重试 HTTP 状态码，如 400/401/403/404，以及 provider 返回的非法 JSON 或不符合 schema 的响应体。

题 2：为什么 fail fast 也需要 observability？ 答：它解释系统为什么没有继续重试，避免排查人员误以为 retry 逻辑漏执行。

题 3：如何验证 fail-fast 行为？ 答：构造 400 或 invalid response，断言 attempts 为 1，存在 fail_fast 日志，并且不存在 retry_exhausted 日志。
