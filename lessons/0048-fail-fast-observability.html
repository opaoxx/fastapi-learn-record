Lesson 0048

Fail-fast Observability：让不可重试失败也有清楚语义

上一课我们记录了 retry exhausted，说明一个可重试错误已经用完所有 attempt。本节继续补齐另一条路径：当 provider 返回 400 或非法响应体时，系统应该快速失败，并用结构化日志说明“为什么不重试”。

0. 本节学习目标

  • 理解 fail fast 和 retry exhausted 的边界。
  • 知道为什么不可重试错误也需要可观测性。
  • 掌握 provider adapter 如何记录 provider_prediction_fail_fast。
  • 学会用 caplog 验证 400 和非法响应体没有被误标为 retry exhausted。
  • 理解日志事件名、错误码、状态码和 attempt 字段之间的诊断关系。

1. 背景：不重试不代表不用解释

在 provider 调用里，并不是所有失败都应该重试。前几节课已经建立了 retry policy：timeout、request error、429 和常见 5xx 可以重试；而 400 一类错误通常说明请求本身有问题，继续发送同样的请求很难自动恢复。

还有一种失败是非法响应体。比如 provider 返回了 200，但 JSON 里给出 {"label": "confused", "score": 1.5}。这不是网络抖动，也不是限流，而是 provider 的响应不符合我们内部契约。重试可能偶尔成功，但默认把它当成可重试错误会掩盖真正的问题：provider 契约漂移了，或者 mock/provider 实现写错了。

所以，fail fast 的重点不是“沉默地失败”。它的重点是：快速停止无意义重试，同时留下足够清楚的诊断信号。

2. 核心概念：Fail Fast 是一种分类决策

Fail fast 指系统发现某类错误继续尝试也没有意义时，立即停止当前流程，并把稳定错误交给上层处理。它不是“没有错误处理”，而是“已经判断这类错误不该进入 retry 循环”。

retryable failure:
  attempt 1 -> wait -> attempt 2 -> maybe exhausted

fail-fast failure:
  attempt 1 -> classify as not retryable -> raise stable error

这和 retry exhausted 的区别很关键：

  • retry exhausted：错误本来值得重试，但 retry budget 用完了。
  • fail fast：错误从分类上就不应该重试，所以不会消耗后续 attempt。

如果把 400 或非法响应体也记成 retry exhausted，日志会误导排查者，以为系统已经尝试了多次。真实情况是：系统只尝试了一次，然后因为错误类型不适合重试而停止。

3. 本节新增的日志事件

本节新增一个稳定事件名：

provider_prediction_fail_fast

它表达的语义是：provider prediction adapter 已经识别出不可重试失败，并且不会继续进入下一次 attempt。这个事件不是替代 AIClientError，而是补充内部诊断信息。

事件字段保持小而稳定：

  • attempt：在哪一次尝试上做出 fail-fast 决策。
  • max_attempts：当前配置最多允许几次尝试。
  • fail_fast_reason：为什么快速失败，目前是 http_status 或 invalid_response。
  • status_code：如果已经拿到 HTTP 响应，记录响应状态码。
  • error_code：将要上抛给 service/router 的稳定错误码。

4. 运行机制：400 为什么只尝试一次

以 max_attempts=3、provider 返回 400 为例，运行过程是：

  1. adapter 发起第 1 次 POST。
  2. provider 返回 HTTP 400。
  3. response.raise_for_status() 抛出 HTTPStatusError。
  4. adapter 判断 400 不在可重试状态码集合里。
  5. 记录 provider_prediction_fail_fast，字段里包含 status_code=400。
  6. 抛出 AIClientError(error_code="ai_provider_http_error")。
  7. 不会进入第 2 次或第 3 次 attempt。
attempt 1 -> HTTP 400
retryable_status = False
log provider_prediction_fail_fast
raise AIClientError

5. 运行机制：非法响应体为什么也快速失败

非法响应体的路径稍微不同。HTTP 状态码可能是 200，说明传输层和状态码都成功了，但响应 JSON 不符合我们对 provider 的内部契约。

class ProviderPredictionBody(BaseModel):
    label: PredictionLabel
    score: float = Field(ge=0, le=1)

如果 provider 返回 score=1.5，Pydantic 会抛出 ValidationError。如果 provider 返回的不是合法 JSON，response.json() 会抛出 ValueError。这两种情况都会映射成同一个稳定错误码：ai_provider_invalid_response。

本节给这条路径也补上 fail-fast 日志。这样排查时能看出：HTTP 请求不是 timeout，也不是 5xx exhausted，而是 provider 响应契约不符合内部模型。

6. 代码精读一：FailFastReason

代码先把 fail-fast 原因收敛成一个小的 Literal 类型：

FailFastReason = Literal["http_status", "invalid_response"]

这不是为了复杂化，而是为了防止日志字段随手写散。以后阅读日志时，fail_fast_reason 的值应该稳定可枚举，而不是一会儿叫 bad_status，一会儿叫 invalid_body。

7. 代码精读二：统一的 fail-fast 日志 helper

新增的 helper 只负责一件事：用固定事件名和固定字段写出快速失败日志。

def log_prediction_fail_fast(
    attempt: int,
    config: AIClientConfig,
    fail_fast_reason: FailFastReason,
    error_code: str,
    status_code: int | None = None,
) -> None:
    logger.warning(
        "provider_prediction_fail_fast",
        extra={
            "attempt": attempt,
            "max_attempts": config.max_attempts,
            "fail_fast_reason": fail_fast_reason,
            "status_code": status_code,
            "error_code": error_code,
        },
    )

这里用 warning，因为这次 provider prediction 最终会失败并上抛稳定错误。但事件名不是 retry_exhausted，所以它不会误导我们以为 retry budget 已经被用完。

8. 代码精读三：HTTP 状态错误的分支

HTTP 状态错误现在被分成两条路径：可重试状态码可能 scheduled retry 或 exhausted；不可重试状态码直接 fail fast。

status_code = exc.response.status_code
retryable_status = is_retryable_provider_status(status_code)

if attempt < config.max_attempts and retryable_status:
    await wait_before_retry(...)
    continue

if retryable_status:
    log_retry_exhausted(...)
else:
    log_prediction_fail_fast(
        attempt=attempt,
        config=config,
        fail_fast_reason="http_status",
        error_code="ai_provider_http_error",
        status_code=status_code,
    )

这段代码的关键是 else。它只处理不可重试 HTTP 状态码。于是 400 和 503 会进入不同日志事件：400 是 fail fast，503 在最后一次失败时才是 retry exhausted。

9. 代码精读四：非法响应体的分支

非法响应体的捕获点在 JSON 解析和 Pydantic 校验之后：

except (ValueError, ValidationError) as exc:
    log_prediction_fail_fast(
        attempt=attempt,
        config=config,
        fail_fast_reason="invalid_response",
        error_code="ai_provider_invalid_response",
        status_code=response.status_code,
    )
    raise AIClientError(
        message="The AI provider returned an invalid prediction response.",
        error_code="ai_provider_invalid_response",
    ) from exc

这里记录 status_code=response.status_code 很有用。它能说明请求拿到了 HTTP 响应，而且通常是 200，但响应体没有通过内部模型校验。这比只看最终 503 更容易定位问题。

10. 动手实验：运行 provider adapter 测试

在项目根目录运行：

.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q

本节新增的测试关注两个场景。

第一个场景是 400：

assert attempts == 1
assert fail_fast_records[0].fail_fast_reason == "http_status"
assert fail_fast_records[0].status_code == 400
assert exhausted_records == []

第二个场景是非法响应体：

assert attempts == 1
assert fail_fast_records[0].fail_fast_reason == "invalid_response"
assert fail_fast_records[0].status_code == 200
assert exhausted_records == []

注意最后一行很重要：我们不仅证明 fail-fast 日志存在，还证明它没有被误记成 retry exhausted。

11. 常见错误

  • 把不可重试 HTTP 错误也当成 retry exhausted：这会误导排查者以为系统已经多次尝试。
  • 只断言 attempts 等于 1，不检查日志事件：行为对了，但诊断信号可能缺失。
  • 非法响应体不记录 status code：排查时看不出 provider 是没响应，还是响应了但 body 不合法。
  • 日志里塞完整响应体：provider 响应可能包含敏感内容，课程里只记录诊断字段。
  • 把 fail-fast 细节加入公共 JSON 响应：内部分类会变成调用方依赖的外部契约。
  • 用临时字符串做 reason：后续日志查询和测试会变脆。

12. 面试视角：如何解释“哪些错误不重试”

我会先把 provider 错误分类。timeout、request error、429 和部分 5xx 可以有限重试；400 类错误通常说明请求本身不对，非法响应体说明 provider 响应不符合内部契约。后两类会 fail fast，不消耗后续 attempt，但会记录结构化日志，例如 attempt、max_attempts、fail_fast_reason、status_code 和 error_code。这样既避免无意义重试，也能解释为什么系统没有继续尝试。

这段回答展示的是工程判断：重试不是越多越好，日志也不是越多越好。关键是把错误分类、重试策略和可观测字段对应起来。

13. 记忆练习

1. fail fast 和 retry exhausted 的本质区别是什么？

fail fast 是错误分类上不该重试；retry exhausted 是错误可重试，但所有 attempt 已经用完。

2. 为什么 400 不应该记录为 retry exhausted？

因为 400 通常表示请求本身有问题，继续发送同样请求没有恢复意义，不属于 retry budget 用完。

3. 非法响应体记录 status_code=200 有什么价值？

它说明 provider 已经返回了成功状态码，但 body 没通过内部契约校验，排查方向应该转向响应格式或模型映射。

4. 为什么测试要断言 exhausted_records 为空？

因为本节要保护日志语义，证明快速失败不会被误标成重试耗尽。

14. 本节总结

本节给 provider prediction adapter 补上了 fail-fast 可观测性。现在，不可重试 HTTP 状态和非法 provider 响应都会记录 provider_prediction_fail_fast，同时保留稳定的 AIClientError 映射。测试证明它们只尝试一次，并且不会混入 retry exhausted 日志。
