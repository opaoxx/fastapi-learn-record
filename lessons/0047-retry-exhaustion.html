0047 Retry Exhaustion：当所有重试预算都被用完

重试不是无限循环。对于本来可重试的错误，如果最后一次 attempt 仍然失败，adapter 必须记录 provider_prediction_retry_exhausted，再向上抛出稳定 AIClientError。

① 本节核心知识框架

Retry Exhaustion
├─ retry budget：max_attempts
├─ scheduled：还有下一次 attempt，准备等待后重试
├─ exhausted：错误可重试，但 attempt 已达到 max_attempts
├─ fail fast：错误不应重试，不属于 exhausted
├─ 日志事件：provider_prediction_retry_exhausted
├─ 日志级别：warning
└─ 上抛错误：AIClientError(ai_provider_timeout/http_error/request_error)

核心结论：exhausted 表达的是“可重试错误已用完预算”，不是“任意失败”。把 400 或 invalid response 记为 exhausted 会误导排查。

② 核心概念底层原理

Retry budget 是一次 provider 调用允许消耗的最大尝试次数。max_attempts=3 表示第 1、2 次失败后还可重试，第 3 次失败后预算耗尽。

事件语义示例
retry_scheduled当前失败可重试，且还有下一次机会attempt 1/3 返回 503
retry_exhausted当前失败可重试，但没有下一次机会attempt 3/3 返回 503
fail_fast错误分类上不应重试HTTP 400 或 invalid response

日志级别使用 warning，因为 exhausted 代表一次 provider prediction 最终失败，需要关注；但它仍是业务可处理故障，不等同于进程崩溃。

超时本来可重试；预算未耗尽就 scheduled，预算耗尽就 exhausted。日志与错误码保持同一语义。

retryable_status = (
    classify_provider_http_status(status_code) == "retryable_http_status"
)
if attempt < config.max_attempts and retryable_status:
    await wait_before_retry(...)
    continue
if retryable_status:
    log_retry_exhausted(...)

HTTP 分支必须先判断状态码是否可重试。只有 429/5xx 这类可重试状态在最后一次失败时才写 exhausted；400 不进入该日志。

④ 核心机制运行全流程拆解

max_attempts = 3，provider 连续返回 503

attempt 1
  → retryable_http_status
  → 1 < 3
  → retry_scheduled

attempt 2
  → retryable_http_status
  → 2 < 3
  → retry_scheduled

attempt 3
  → retryable_http_status
  → 3 < 3 为 false
  → retry_exhausted
  → raise AIClientError("ai_provider_http_error")

exhausted 是 retry 的终止事件。它回答“系统有没有重试过、为什么最后失败”：有，且可重试错误已经用完所有 attempt。

⑤ 重难点、易混淆点对比辨析

易混点正确理解错误影响
exhausted vs fail fast前者预算用完，后者分类不重试误判 provider 是否多次失败
warning vs errorwarning 表示需关注但可处理过高等级造成告警噪声
最后一次是否 scheduled不 scheduled，直接 exhausted日志误以为还有下一次
error_code 是否可省不可省无法关联 HTTP 错误响应

⑥ 开发常见报错+坑点+解决方案

  • 所有失败都打 exhausted：400/invalid response 被误归类。解决：只对 retryable failure 的最后一次失败打 exhausted。
  • 最后一次失败仍 sleep：没有下一次 attempt。解决：attempt < max_attempts 才 scheduled。
  • 日志缺少 attempt：无法判断预算是否真正用完。解决：exhausted 事件必须带 attempt 与 max_attempts。
  • 只抛错误不记日志：上层只看到 503，无法知道是否重试耗尽。解决：抛出前写 warning 结构化日志。

⑦ 生产环境 vs 教学环境 核心差异

维度教学环境生产环境

下游策略稳定错误码错误分类驱动重试、降级或人工告警

⑧ 课后记忆习题+标准答案

1. Retry exhaustion 的定义是什么？

错误本来可重试，但本次调用允许的所有 attempt 已经用完，只能停止并上抛失败。

2. 为什么 400 不应记为 exhausted？

400 通常是请求或认证类确定性错误，分类上不该重试，不属于预算耗尽。

3. exhausted 日志为什么要带 error_code？

它让内部日志与上层稳定错误响应对应起来，便于跨层排查。

4. exhausted 为什么用 warning？

它表示一次 provider 调用最终失败，需要关注，但应用进程仍可继续运行。

⑨ 面试高频真题+解析

题 1：如何区分 retry scheduled 与 retry exhausted？ 答：scheduled 表示还有下一次 attempt，准备等待后重试；exhausted 表示可重试错误已经没有下一次 attempt。

题 2：为什么 retry 不能无限进行？ 答：无限 retry 会放大故障、增加成本、占用 worker，并破坏用户请求时延边界。

题 3：线上看到 retry_exhausted 应如何排查？ 答：看 retry_reason、status_code、attempt/max_attempts、retry_after、error_code，判断是 timeout、限流、5xx 还是网络错误导致预算耗尽。
