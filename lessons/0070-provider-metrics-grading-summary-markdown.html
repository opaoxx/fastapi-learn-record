0070 Provider Metrics Grading Summary Markdown：把训练汇总渲染成可复盘报告

本节新增 render_provider_metrics_runbook_grading_summary_markdown()，把 0069 生成的结构化 summary 渲染成 Markdown 报告。它输出总体统计表和逐题明细表，适合复制到 GitHub、PR、issue 或学习记录中做离线复盘。

① 本节核心知识框架

Grading Summary Markdown
├─ 输入：summary
│  ├─ total
│  ├─ answered
│  ├─ passed
│  ├─ failed
│  ├─ unanswered
│  └─ grades
├─ 输出：Markdown report
│  ├─ summary table
│  └─ grade details table
├─ 渲染规则
│  ├─ 数字列右对齐
│  ├─ missing 为空时显示 "-"
│  ├─ grades 必须是 list
│  └─ grade["missing"] 必须是 list
└─ 边界
   ├─ 不重新计算评分
   ├─ 不读取原始 response
   ├─ 不调用 provider
   └─ 不写文件或数据库

② 核心概念底层原理

结构化 summary 适合程序消费，Markdown report 适合人类复盘。两者属于不同层级：summary 是数据契约，report 是展示契约。把它们分开，可以让统计逻辑和展示格式分别测试。

本节渲染函数不重新计算 passed、failed 或 unanswered。原因是计算逻辑已经属于 0069 的 summary helper；展示层如果重复计算，会制造第二套事实源，未来一旦统计规则变化就容易漂移。

报告不包含原始 response 文本，只展示 missing 和 feedback。这是教学项目里提前建立的隐私边界：复盘需要结果和原因，不一定需要传播用户自由文本。

层级输入输出职责
评分层card + responsegrade判断锚点命中
汇总层cards + responsessummary统计训练结果
渲染层summaryMarkdown展示复盘报告

③ 全套代码逐行精读解析

def render_provider_metrics_runbook_grading_summary_markdown(
    summary: dict[str, object],
) -> str:
    lines = [
        "## Provider Metrics Runbook Grading Summary",
        "",
        "| total | answered | passed | failed | unanswered |",
        "| ---: | ---: | ---: | ---: | ---: |",

函数只接收 summary，说明它位于训练链路末端。lines 先放标题、空行、表头和对齐行，确保即使后续没有明细，也能形成合法 Markdown 报告骨架。

(
    "| {total} | {answered} | {passed} | {failed} | {unanswered} |"
).format(
    total=summary["total"],
    answered=summary["answered"],
    passed=summary["passed"],
    failed=summary["failed"],
    unanswered=summary["unanswered"],
),

总体统计直接读取 summary 字段，不重新推导。使用 summary["total"] 这种索引访问可以暴露上游契约错误；如果用 get() 静默给默认值，报告可能把损坏数据伪装成正常统计。

"",
"## Grade Details",
"",
"| card_id | answered | passed | missing | feedback |",
"| --- | --- | --- | --- | --- |",
]

第二个表展示逐题明细。报告分成总体统计和题目明细两段，是为了让读者先看整体训练结果，再定位具体哪张题卡缺少哪些锚点。

grades = summary["grades"]
if not isinstance(grades, list):
    raise ValueError("Grading summary grades must be a list.")

这里做最小结构校验：grades 必须是列表。若上游把它传成字典或字符串，继续渲染会产生难读的错误报告，因此应立刻抛出可诊断异常。

for grade in grades:
    missing = grade["missing"]
    if not isinstance(missing, list):
        raise ValueError("Grading detail missing field must be a list.")

每题的 missing 也必须是列表，因为它代表缺失锚点集合。若它是字符串，", ".join(...) 会按字符拆分，报告会变成错误但不易察觉。

lines.append(
    "| {card_id} | {answered} | {passed} | {missing} | {feedback} |".format(
        card_id=grade["card_id"],
        answered=grade["answered"],
        passed=grade["passed"],
        missing=", ".join(str(item) for item in missing) if missing else "-",
        feedback=grade["feedback"],
    )
)
return "\n".join(lines) + "\n"

每个 grade 渲染成一行。missing 为空时显示 -，避免空单元格难以扫描。最后追加末尾换行，方便写入 Markdown 文件或拼接到其他文档。

④ 核心机制运行全流程拆解

cards + responses
  → summarize_provider_metrics_runbook_exercise_grades(...)
  → summary dict
  → render_provider_metrics_runbook_grading_summary_markdown(summary)
  → Markdown report
  → GitHub / PR / issue / learning record

这条链路体现典型分层：评分函数处理单题，汇总函数处理批量统计，渲染函数只处理展示。任何展示格式变化都不应影响评分结果；任何评分规则变化都不应强迫 Markdown renderer 重写。

阶段数据形态可测试点失败风险

传播文档片段链接和导航报告被误当成数据源

⑧ 课后记忆习题+标准答案

1. 为什么 renderer 不重新计算 passed 和 failed？

统计规则属于 summary helper。renderer 重算会产生第二套事实源，未来规则变化时容易漂移。

2. 为什么 grades 必须校验为 list？

逐题明细需要按列表迭代。若传入字典或字符串，报告结构会错误且难以诊断。

3. 为什么 missing 为空时显示 -？

空单元格不利于扫描，也不利于稳定测试。- 明确表示没有缺失锚点。

4. 为什么报告不展示原始 response？

原始作答可能包含敏感文本和噪声。复盘报告只需要评分结果、缺失项和反馈即可。

5. 生产环境为什么要做 Markdown 转义？

用户或系统字段可能包含 |、换行、HTML 片段等特殊内容，未转义会破坏表格或引入安全风险。

⑨ 面试高频真题+解析

面试题 1：展示层函数为什么应保持纯渲染？

纯渲染函数只把结构化数据转换成文本，易测试、可复现，也避免把业务规则散落到展示层。

面试题 2：Markdown 报告和结构化 summary 的关系是什么？

summary 是机器可消费的数据契约，Markdown 是人类可读展示。报告可以由 summary 生成，但不能反向成为评分数据源。

面试题 3：为什么完整字符串测试适合本节？

本节的业务价值就是稳定报告格式。完整字符串测试能保护标题、表头、列顺序、空行和末尾换行。

面试题 4：生产报告系统还要补哪些能力？

需要字段转义、模板版本、权限控制、审计日志、下载记录、脱敏策略和渲染回归测试。
