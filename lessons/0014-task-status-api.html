任务状态 API：异步结果资源、状态机与分页列表契约

后台任务只有“提交接口”是不完整的。客户端需要通过状态 API 判断任务排队、运行、成功或失败。本节围绕 SummaryTask、GET /tasks/{task_id} 与 GET /tasks 的 envelope 响应，建立可应试、可面试、可扩展的异步任务查询模型。

① 本节核心知识框架

任务状态 API
├─ 领域对象：SummaryTask 是可查询的异步任务资源
├─ 状态字段：queued / running / completed / failed
├─ 单资源查询：GET /tasks/{task_id} -> SummaryTaskRead
├─ 列表查询：GET /tasks?status=&limit=&offset= -> SummaryTaskListResponse
├─ 响应隔离：数据库保存 text，API 不返回 text
├─ 错误语义：不存在返回 404，非法状态/分页参数返回 422
└─ 前端契约：列表必须读取 items，而不是把响应当裸数组

② 核心概念底层原理

异步任务系统的本质是把“处理过程”建模为一个资源。提交接口创建资源，状态接口读取资源，worker 更新资源。只要任务记录存在，客户端就不必依赖原始 HTTP 连接维持处理过程。

SummaryTask 是数据库表模型，承载完整内部数据；SummaryTaskRead 是响应模型，只暴露客户端需要知道的字段。这种分离是 API 安全设计的基础：持久化字段不等于对外字段。

列表接口现在返回 SummaryTaskListResponse envelope，而不是裸数组。原因是分页和筛选必须携带元数据：items 是当前页，count 是匹配总数，limit/offset 是当前查询窗口。没有 envelope，前端无法正确计算下一页、空页和总量。

③ 全套代码逐行精读解析

TaskStatus = Literal["queued", "running", "completed", "failed"]

class SummaryTask(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    text: str
    text_length: int
    source_file_id: int | None = Field(default=None, index=True)
    status: str = Field(default="queued", index=True)
    result: str | None = None
    error: str | None = None
  • TaskStatus：限制查询参数只能是四种合法状态，非法值由 FastAPI/Pydantic 返回 422。
  • id：任务主键，也是客户端轮询状态的稳定句柄。
  • text：内部处理输入，保存在库中供 worker 使用，但不对外返回。
  • text_length：对外暴露输入规模，避免泄漏原文。
  • source_file_id：关联文件来源；手动文本任务为 None，文件任务指向上传文件。
  • status：加索引是为了后续按状态筛选更快。
  • result/error：成功和失败互斥表达；成功看 result，失败看 error。
class SummaryTaskListResponse(BaseModel):
    items: list[SummaryTaskRead]
    count: int = PydanticField(ge=0)
    limit: int = PydanticField(ge=1, le=100)
    offset: int = PydanticField(ge=0)

items 使用响应模型列表，避免泄漏 text；count 不是当前页长度，而是匹配条件下的总数；limit/offset 回显服务端实际使用的分页参数。

@router.get("/{task_id}", response_model=SummaryTaskRead)
def read_summary_task(
    task_id: Annotated[int, Path(ge=1, description="The summary task ID.")],
    session: Annotated[Session, Depends(get_session)],
) -> SummaryTask:
    task = read_summary_task_record(session, task_id)
    if task is None:
        raise HTTPException(status_code=404, detail=f"Task {task_id} was not found.")
    return task

Path(ge=1) 先挡住非正数 id；数据库查不到时返回 404，表示“参数格式合法，但资源不存在”。

@router.get("", response_model=SummaryTaskListResponse)
def list_summary_tasks(
    session: Annotated[Session, Depends(get_session)],
    task_status: Annotated[TaskStatus | None, Query(alias="status")] = None,
    offset: Annotated[int, Query(ge=0)] = 0,
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
) -> SummaryTaskListResponse:
    return list_summary_task_records(session=session, task_status=task_status, offset=offset, limit=limit)

Query(alias="status") 让 URL 使用简洁的 ?status=completed，但 Python 变量仍可命名为 task_status，避免与 status 模块混淆。

④ 核心机制运行全流程拆解

POST /tasks/summaries
  -> 创建 queued 任务
  -> 后台 worker 改为 running
  -> AIClient.summarize 成功：result=摘要，status=completed
  -> AIClientError/异常：error=错误信息，status=failed

GET /tasks/{task_id}
  -> Path 校验 task_id >= 1
  -> session.get(SummaryTask, task_id)
  -> None: 404
  -> 命中: 按 SummaryTaskRead 序列化

GET /tasks?status=completed&limit=5&offset=0
  -> Query 校验 status/limit/offset
  -> select(SummaryTask).where(...).order_by(id.desc)
  -> select(count()).where(...)
  -> 返回 {items,count,limit,offset}

⑥ 开发常见报错+坑点+解决方案

  • 前端报 page.map is not a function：把 envelope 当数组。解决：读取 page.items。
  • ?status=done 返回 422：状态不在 Literal 范围。解决：使用 queued/running/completed/failed。
  • GET /tasks/0 返回 422：Path(ge=1) 拦截非正 id。解决：只查询正整数主键。
  • 分页下一页按钮错误：把 count 当当前页长度。解决：下一页条件应是 offset + items.length < count。
  • 敏感输入泄漏：响应模型返回了 text。解决：只返回 SummaryTaskRead。

⑧ 课后记忆习题+标准答案

1. 为什么任务列表要返回 envelope？

因为分页列表不仅需要当前页数据，还需要总数、limit 和 offset，前端才能判断页码与按钮状态。

2. 为什么状态查询接口不返回原始 text？

原始输入可能包含隐私或大文本，API 输出应遵循最小暴露原则，只返回必要元数据和结果。

3. 查询不存在任务为什么是 404 而不是 422？

因为 task_id 的格式和范围合法，只是对应资源不存在；422 用于参数校验失败。

⑨ 面试高频真题+解析

题 1：异步任务状态表应该包含哪些核心字段？

至少包含任务 id、输入引用或输入摘要、状态、结果、错误、创建/更新时间、用户归属。教学项目当前包含 id、text、text_length、source_file_id、status、result、error。

题 2：为什么列表接口从裸数组演进为 envelope 是兼容性问题？

JSON 顶层类型从 array 变为 object，会破坏旧客户端的解析逻辑，所以要通过版本、迁移说明或契约测试保护这种变更。

题 3：轮询任务状态有什么缺点，如何优化？

轮询会产生额外请求和延迟。可用指数退避、长轮询、Server-Sent Events、WebSocket 或回调通知优化。
