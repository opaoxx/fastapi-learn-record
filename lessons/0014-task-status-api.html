Lesson 0014

任务状态 API：让客户端查询异步结果

后台任务只是半个故事。客户端还需要知道任务现在排队、运行、完成还是失败。本节学习用数据库记录任务状态，并提供查询接口。

0. 本节学习目标

  • 理解任务状态模型：queued、running、completed、failed。
  • 理解为什么异步任务要把状态存进数据库。
  • 看懂 GET /tasks/{task_id} 和 GET /tasks。
  • 理解这个模式如何迁移到真实 AI 推理服务。

1. 为什么需要任务状态

如果接口只返回“任务已提交”，客户端仍然不知道后续发生了什么。任务可能成功、失败、排队很久，或者结果已经生成。

所以常见异步 API 会设计成两步：

POST /tasks/summaries  -> 返回 task_id
GET  /tasks/{task_id} -> 查询状态和结果

这就是很多 AI 平台、文件处理平台、视频生成平台常见的工作流。

2. SummaryTask 表模型

class SummaryTask(SQLModel, table=True):
    id: int | None = Field(default=None, primary_key=True)
    text: str
    text_length: int
    status: str = Field(default="queued", index=True)
    result: str | None = None
    error: str | None = None

字段含义：

  • text：原始输入，后台任务需要它来生成摘要。
  • text_length：返回给客户端的安全摘要信息，不直接暴露全文。
  • status：任务状态。
  • result：成功后的结果。
  • error：失败时的错误信息。

3. 响应模型为什么不返回 text

class SummaryTaskRead(SQLModel):
    id: int
    text_length: int
    status: str
    result: str | None = None
    error: str | None = None

数据库表里保存 text，但响应模型不返回 text。这是接口设计里的一个重要习惯：数据库保存什么，不代表 API 必须暴露什么。

真实 AI 服务里，用户输入可能包含隐私信息，默认不要在状态查询接口里原样返回全文。

4. 状态流转

queued -> running -> completed
queued -> running -> failed

本项目的后台函数会先把状态改成 running，处理成功后改成 completed，如果发生异常就改成 failed 并记录 error。

虽然 demo 很快完成，但真实模型调用可能需要几秒甚至几分钟。状态流转能让前端展示“排队中、处理中、已完成、失败”。

5. 查询单个任务

@router.get("/{task_id}", response_model=SummaryTaskRead)
def read_summary_task(
    task_id: Annotated[int, Path(ge=1)],
    session: Annotated[Session, Depends(get_session)],
) -> SummaryTask:
    task = session.get(SummaryTask, task_id)
    if task is None:
        raise HTTPException(status_code=404, detail=f"Task {task_id} was not found.")
    return task

它和 GET /items/{item_id} 很像：按主键查询，找不到返回 404。这说明你前面学的资源详情接口模式正在复用。

6. 查询任务列表

@router.get("", response_model=list[SummaryTaskRead])
def list_summary_tasks(session: Annotated[Session, Depends(get_session)]):
    return list(session.exec(select(SummaryTask)))

这个接口可以帮助开发者或前端查看已有任务。后续可以继续扩展分页、按状态筛选、按用户筛选。

7. 实验：提交并查询

POST /tasks/summaries
X-API-Key: dev-secret-key

{
  "text": "FastAPI background tasks let an API accept work quickly and continue processing after sending the initial response."
}

GET /tasks/{task_id}
X-API-Key: dev-secret-key

在当前 TestClient 和本地 demo 中，任务通常会很快变成 completed，并出现 result。

8. 和真实 AI 服务的对应关系

这个 demo 可以直接映射到真实 AI 服务：

  • SummaryRequest.text：用户输入或 prompt。
  • SummaryTask：推理任务记录。
  • process_summary_task：真实模型调用函数。
  • result：模型输出。
  • status：前端轮询展示进度。

等项目变复杂后，后台函数可以迁移到 Celery、任务队列或独立 worker，但接口契约可以保持类似。

10. 小测：先回忆再展开

1. 为什么状态查询接口不直接返回原始 text？

因为输入可能包含隐私信息，数据库保存不代表 API 必须暴露。

2. queued 和 running 有什么区别？

queued 表示任务已创建等待处理，running 表示后台函数正在处理。

3. 为什么查询不存在的任务返回 404？

因为 task_id 参数合法，但数据库里没有对应任务资源。

11. 本节总结

异步任务需要状态 API。提交接口负责创建任务，后台函数负责处理任务，查询接口负责展示状态和结果。这个模式是 AI 服务、文件处理、视频生成、批处理任务的常见基础。
