CORS 浏览器前端：同源策略、预检请求与 FastAPI 中间件边界

CORS 错误不是 FastAPI 路由“写错了”，而是浏览器安全模型在生效。本节把 origin、同源策略、预检请求、Access-Control-* 响应头和 CORSMiddleware 的应用级位置讲透，避免把 CORS 误当成鉴权或后端业务逻辑。

① 本节核心知识框架

CORS 机制
├─ Origin = scheme + host + port
├─ 同源策略：浏览器限制网页 JS 读取跨源响应
├─ 跨源示例：http://127.0.0.1:5500 -> http://127.0.0.1:8000
├─ 简单请求：满足特定方法/头/Content-Type，直接带 Origin
├─ 预检请求：OPTIONS + Access-Control-Request-Method/Headers
├─ 服务端授权：Access-Control-Allow-Origin/Methods/Headers
├─ FastAPI 实现：app.add_middleware(CORSMiddleware, ...)
└─ 安全边界：CORS 是浏览器读响应许可，不是身份认证

② 核心概念底层原理

origin 由协议、主机和端口三元组决定。http://localhost:8000 与 http://127.0.0.1:8000 不同源，http://127.0.0.1:5500 与 http://127.0.0.1:8000 也不同源，因为主机或端口不同。

同源策略由浏览器执行：网页 JavaScript 默认不能读取其他源的响应。CORS 是服务端显式声明“哪些源可以读取我响应”的协议。如果使用 PyCharm HTTP Client、curl 或 Python requests，通常不会遇到 CORS，因为它们不是浏览器页面环境。

当请求包含 X-API-Key 这类自定义请求头时，浏览器会先发 OPTIONS 预检请求，询问服务端是否允许真实请求的方法和头部。预检通过后，浏览器才发送真正的 POST。

③ 全套代码逐行精读解析

from fastapi.middleware.cors import CORSMiddleware

导入 FastAPI/Starlette 提供的 CORS 中间件。CORS 是应用横切策略，不应写在单个业务路由里。

allowed_origins = [
    "http://127.0.0.1:5500",
    "http://localhost:5500",
    "http://127.0.0.1:8000",
    "http://localhost:8000",
]
  • 127.0.0.1:5500 与 localhost:5500：允许本地独立静态服务器或 VS Code Live Server。
  • 127.0.0.1:8000 与 localhost:8000：允许同端口本地页面与 API 协作。
  • 白名单必须写完整 origin，不能只写主机或路径；路径不属于 origin。
app.add_middleware(
    CORSMiddleware,
    allow_origins=allowed_origins,
    allow_methods=["*"],
    allow_headers=["*"],
)
  • add_middleware：把 CORS 包在路由外层，请求进入路由前、响应离开路由后都会经过它。
  • allow_origins：允许哪些网页源读取响应。
  • allow_methods=["*"]：允许所有 HTTP 方法通过 CORS 检查；生产中可收紧。
  • allow_headers=["*"]：允许自定义请求头，包括本项目的 X-API-Key。
response = client.options(
    "/files/text",
    headers={
        "Origin": "http://127.0.0.1:5500",
        "Access-Control-Request-Method": "POST",
        "Access-Control-Request-Headers": "X-API-Key",
    },
)

测试模拟浏览器预检。若配置正确，响应头中会包含 access-control-allow-origin: http://127.0.0.1:5500。

④ 核心机制运行全流程拆解

页面源：http://127.0.0.1:5500
API 源：http://127.0.0.1:8000

fetch("/files/text", {method:"POST", headers:{"X-API-Key":...}})
  -> 浏览器发现跨源 + 自定义头
  -> 先发 OPTIONS /files/text
     Origin: http://127.0.0.1:5500
     Access-Control-Request-Method: POST
     Access-Control-Request-Headers: X-API-Key
  -> CORSMiddleware 检查 origin/method/header
  -> 返回 Access-Control-Allow-Origin/Methods/Headers
  -> 浏览器确认允许
  -> 发送真实 POST
  -> 后端执行业务路由和 API Key 校验
  -> 浏览器允许 JS 读取响应

⑤ 重难点、易混淆点对比辨析

概念本质常见误解
CORS浏览器跨源读取响应的许可协议后端身份认证机制
同源协议、主机、端口完全一致只要都在本机就是同源
预检请求浏览器自动发出的 OPTIONS 探测业务代码手动发出的接口
allow_headers允许跨源请求携带哪些头替代 API Key 校验
Postman/curl 正常非浏览器不执行同源限制说明 CORS 配置一定正确

⑥ 开发常见报错+坑点+解决方案

  • No 'Access-Control-Allow-Origin'：origin 不在白名单或中间件未注册。解决：把完整 origin 加入 allowed_origins。
  • Request header field x-api-key is not allowed：allow_headers 未允许自定义头。解决：包含 X-API-Key 或使用 ["*"]。
  • Method POST is not allowed：allow_methods 未包含 POST。解决：显式添加 POST 或教学中使用 ["*"]。
  • localhost 与 127.0.0.1 混用失败：两者不同源。解决：白名单同时覆盖常用本地 origin。
  • 把 CORS 当鉴权：非浏览器客户端可绕过 CORS。解决：敏感接口仍必须使用 API Key、OAuth 或会话认证。

⑧ 课后记忆习题+标准答案

1. Origin 由哪三部分组成？

协议、主机、端口。三者任意一个不同，都不是同源。

2. 为什么带 X-API-Key 的 fetch 常触发 OPTIONS？

因为 X-API-Key 是自定义请求头，浏览器会先预检服务端是否允许该跨源方法和请求头。

3. CORS 能替代 API Key 吗？

不能。CORS 只限制浏览器页面读取跨源响应，不能认证调用者身份，也不能限制 curl、脚本等非浏览器客户端。

⑨ 面试高频真题+解析

题 1：为什么 Postman 能调通接口，浏览器 fetch 却报 CORS？

Postman 不执行浏览器同源策略；浏览器页面环境会检查服务端 CORS 响应头。接口业务本身可能正常，但缺少浏览器所需的跨源许可。

题 2：预检请求经过业务鉴权吗？

通常由 CORS 中间件在进入具体业务路由前处理。预检只询问跨源许可，不应要求业务请求体；真实请求才执行 API Key 等业务鉴权。

题 3：生产环境为什么不建议随便 allow_origins=["*"]？

它会允许任意网站页面读取你的 API 响应，扩大攻击面。带凭证请求还会有更严格限制；生产应使用明确域名白名单。
