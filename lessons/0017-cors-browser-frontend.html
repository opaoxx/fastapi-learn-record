Lesson 0017

CORS：为什么浏览器前端会被后端“拦住”

这一节不急着写页面，先把浏览器安全模型讲清楚：同源、跨源、预检请求、允许来源，以及 FastAPI 的 CORSMiddleware。

0. 本节学习目标

  • 理解浏览器里的 origin 是什么。
  • 解释为什么前端调用后端时会出现 CORS 错误。
  • 看懂 OPTIONS 预检请求。
  • 给 FastAPI 添加 CORSMiddleware。

1. 先建立问题场景

后端 API 单独运行时，你可以在 /docs 里调用接口，也可以用 PyCharm 的 HTTP Client 调用接口。可是当前端页面在浏览器里运行时，情况会变复杂。

假设：

前端页面：http://127.0.0.1:5500
后端 API：http://127.0.0.1:8000

虽然它们都在本机，但端口不同，所以浏览器认为它们是两个不同的来源。浏览器会先问后端：“这个前端能不能访问你？”这个机制就是 CORS。

2. Origin 到底是什么

origin 由三部分组成：

  • 协议：http 或 https。
  • 域名或主机：例如 localhost、127.0.0.1、example.com。
  • 端口：例如 8000、5500。

三者只要有一个不同，就是不同源。下面这些互相都不完全相同：

http://localhost:8000
http://127.0.0.1:8000
http://127.0.0.1:5500
https://127.0.0.1:8000

这不是 FastAPI 故意刁难你，而是浏览器为了保护用户做的安全限制。

3. 为什么 API 客户端能调用，浏览器却不行

CORS 是浏览器执行的限制。也就是说，服务器本身并不神秘地“拒绝 Python 请求”，而是浏览器在保护网页环境。

PyCharm HTTP Client

它不是浏览器，不执行浏览器同源策略，所以通常不会遇到 CORS 拦截。

浏览器 fetch

它运行在网页里，必须遵守浏览器安全模型。跨源调用要看后端是否明确允许。

4. 预检请求：浏览器先发 OPTIONS

当前项目的写接口需要 X-API-Key。这种自定义请求头会让浏览器先发送一个 OPTIONS 请求，询问后端：

Origin: http://127.0.0.1:5500
Access-Control-Request-Method: POST
Access-Control-Request-Headers: X-API-Key

如果后端响应里包含合适的 CORS 头，浏览器才会继续发送真正的 POST 请求。

5. FastAPI 中间件位置

中间件可以理解为包在路由外面的一层处理逻辑。请求进入应用时，先经过中间件，再到具体路由；响应离开应用时，也会经过中间件。

浏览器请求
  -> CORSMiddleware
  -> APIRouter 路由函数
  -> CORSMiddleware 给响应补 CORS 头
  -> 浏览器响应

所以 CORS 不适合写在某一个路由函数里，它是整个应用层面的浏览器访问策略。

6. 本节新增代码

from fastapi.middleware.cors import CORSMiddleware

allowed_origins = [
    "http://127.0.0.1:5500",
    "http://localhost:5500",
    "http://127.0.0.1:8000",
    "http://localhost:8000",
]

app.add_middleware(
    CORSMiddleware,
    allow_origins=allowed_origins,
    allow_methods=["*"],
    allow_headers=["*"],
)

allow_origins 是白名单，表示哪些前端来源可以访问后端。allow_methods 允许请求方法，allow_headers 允许请求头。因为我们有 X-API-Key，所以请求头也要放行。

7. 为什么不直接写 *

* 表示允许任意来源，学习阶段看起来省事，但真实项目通常不这么做。你的后端不应该让任意网站都能从浏览器直接调用。

尤其当接口需要登录态、Authorization 头、Cookie、API Key 或其他敏感能力时，推荐明确列出允许的前端地址。

8. 课堂练习

用 HTTP Client 或测试代码观察一次预检请求：

OPTIONS /files/text
Origin: http://127.0.0.1:5500
Access-Control-Request-Method: POST
Access-Control-Request-Headers: X-API-Key

你应该看到响应中包含 access-control-allow-origin，并且值是 http://127.0.0.1:5500。

9. 小测：先回忆再展开

1. 为什么 127.0.0.1:5500 和 127.0.0.1:8000 不同源？

因为端口不同。origin 由协议、主机和端口共同决定。

2. CORS 是谁主要执行的限制？

浏览器。它会根据后端返回的 CORS 响应头决定是否允许网页 JavaScript 读取响应。

3. 为什么 X-API-Key 会牵出预检请求？

因为它是自定义请求头，浏览器会先用 OPTIONS 询问后端是否允许这个跨源请求头。

10. 本节总结

本节把后端从“只能被 API 工具舒服调用”推进到“可以被浏览器前端安全调用”。CORS 不是业务逻辑，而是浏览器和后端之间的访问许可协议。
