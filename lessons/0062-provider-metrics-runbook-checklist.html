0062 Provider Metrics Runbook Checklist：把指标读数变成可执行排障路径

本节把 provider metrics 从“可观察数据”升级为“可执行排障清单”。reference/provider-metrics-runbook.html 的价值不在于重复解释指标，而在于规定看到 success、retry_scheduled、retry_exhausted、fail_fast 后下一步应该查什么、不能推出什么结论、应该回到哪些测试验证。

① 本节核心知识框架

Provider Metrics Runbook Checklist
├─ Runbook 定位
│  ├─ 不是概念文章：不重复讲 metrics 基础
│  ├─ 不是告警策略：当前信号是教学信号
│  └─ 是排障分支表：把现象映射到下一步动作
├─ 观测入口
│  ├─ GET /provider/metrics：人类优先阅读的 JSON snapshot
│  └─ GET /provider/metrics/prometheus：文本导出形状检查
├─ 核心指标
│  └─ provider_prediction_total{operation,outcome,failure_category,error_code,status_code}
├─ outcome 判断
│  ├─ success：provider 请求成功
│  ├─ retry_scheduled：出现可重试失败并安排下一次尝试
│  ├─ retry_exhausted：重试次数耗尽
│  └─ fail_fast：非重试型失败或响应结构无效
├─ 排障字段
│  ├─ failure_category：失败分类
│  ├─ error_code：稳定错误码
│  └─ status_code：HTTP 状态码或 none
└─ 文档合同测试
   ├─ runbook 必须保留 endpoint、metric name、outcome
   ├─ runbook 必须保留 teaching signal, not a paging alert
   └─ runbook 必须链接文档入口、课程页和测试文件

② 核心概念底层原理

metrics 记录“发生过什么”，runbook 规定“接下来查什么”。一个指标样本本身不能自动等价为故障结论。例如 retry_scheduled 表示系统曾安排重试，不等价于最终用户请求失败；真正判断影响需要继续观察是否出现 retry_exhausted、是否仍有 success、失败分类是否集中在 timeout、429、5xx 或 invalid response。

runbook 的底层结构接近条件分支表：输入是观测信号，判断条件是 label，输出是下一步动作。优秀 runbook 不是堆概念，而是减少人在紧张排障时的自由发挥：先查 JSON snapshot，再按 outcome 分组，再读 failure labels，再回到日志和测试证据，最后判断这是教学信号、临时 provider 抖动、重试耗尽还是非重试错误。

对象回答的问题本节对应物
Metric发生过什么provider_prediction_total
Label按什么维度归类outcome、failure_category、status_code
Endpoint去哪里看信号/provider/metrics、/provider/metrics/prometheus
Runbook看到信号后如何行动provider-metrics-runbook.html
Alert何时自动通知人本课程明确不提供生产 paging alert

本节必须强调边界：当前 counter 是进程内教学实现，runbook 是学习排障顺序，文本导出是 educational export shape。它们能训练监控思维，但不能直接替代生产的多实例聚合、持久化时序库、阈值告警、值班升级和 SLO 管理。

④ 核心机制运行全流程拆解

出现 provider prediction 现象
  → 打开 /provider/metrics
    → 是否有 provider_prediction_total 样本
      ├─ 无样本：确认是否触发过 provider prediction 路径，检查测试隔离/进程重启
      └─ 有样本：按 outcome 分组
          ├─ success
          │  └─ 确认成功请求量是否符合预期
          ├─ retry_scheduled
          │  ├─ 继续查 failure_category/status_code
          │  ├─ 查 retry delay log
          │  └─ 不直接判定最终失败
          ├─ retry_exhausted
          │  ├─ 查 max_attempts
          │  ├─ 查最后一次错误类别
          │  └─ 判断 provider 临时不可用是否持续
          └─ fail_fast
             ├─ 区分 non_retryable_http_status 与 invalid_response
             ├─ 不盲目增加重试次数
             └─ 回看响应结构或 4xx 语义
  → 如需检查导出格式
    → 打开 /provider/metrics/prometheus
  → 如需验证文档来源
    → 回到 provider metrics index 和相关测试

调用链路上，provider prediction adapter 在不同结果路径上记录 metrics；metrics endpoint 只是读取快照，不负责解释；runbook 把读取到的 label 组合翻译成排障动作。这个分层很重要：业务代码不应该内置人工排障说明，文档也不应该伪装成运行时权威。

数据层级样例解释方式
事件provider HTTP 503请求过程中发生可重试服务端错误
指标outcome="retry_scheduled"系统安排了下一次尝试
日志provider_prediction_retry_scheduled包含 delay、attempt、reason 等诊断字段
runbook 动作继续查 retry_exhausted判断是否最终失败，而不是看到 retry 就下结论

⑤ 重难点、易混淆点对比辨析

易混点错误判断正确判断
retry_scheduled用户请求最终失败只表示某次尝试失败并安排重试；必须继续看 success 或 retry_exhausted。
retry_exhausted所有 provider 请求都坏了表示某类可重试失败耗尽最大尝试次数，需要结合 failure_category 和样本数量判断范围。
fail_fast应该增加重试次数通常代表不该重试的 4xx 或无效响应结构，应先修请求/响应契约。
count 增加一定需要告警当前是 teaching signal，不能直接当生产 paging alert。
Prometheus text endpoint等同生产 Prometheus pipeline只是教学用 text-style rendering，不含 scrape、聚合、存储、alert rule。
error_code等同根因它是稳定分类字段；根因还要结合 status_code、日志、provider 响应和上下游状态。

⑥ 开发常见报错+坑点+解决方案

坑点表现解决方案
只看 count，不看 labels无法区分 success、retry、fail-fast先按 outcome 分组，再看失败标签。
把 retry_scheduled 当最终失败误报故障继续检查是否有 retry_exhausted 和后续 success。
对 fail_fast 盲目加重试放大错误请求或无效响应先区分 non_retryable_http_status 与 invalid_response。
把教学 counter 当全局事实多进程/多实例下读数不完整课程中明确记录生产边界；生产使用真实 metrics 后端。
runbook 不链接测试排障建议长期漂移链接 endpoint 测试、provider HTTP 测试和文档合同测试。
忘记测试隔离不同测试之间指标样本互相污染使用 reset fixture 清理 in-process counter。
只查 JSON，不查 text export无法发现 Prometheus label escaping 问题排查导出格式时补查 /provider/metrics/prometheus。

⑧ 课后记忆习题+标准答案

1. runbook 与 documentation index 的区别是什么？

documentation index 负责回答“资源在哪里”；runbook 负责回答“看到某个现象后下一步怎么查”。前者是导航入口，后者是操作分支表。

2. 为什么 retry_scheduled 不能直接等价为最终失败？

它只表示某次尝试失败后系统安排了重试；后续重试可能成功。必须继续观察 retry_exhausted 和 success。

3. fail_fast 出现时为什么不应第一反应增加重试次数？

fail_fast 通常表示非重试型 HTTP 错误或无效响应结构。盲目重试会放大错误请求、浪费资源，并掩盖真正的契约问题。

4. 排查失败样本时应优先看哪些 labels？

先看 outcome，再看 failure_category、error_code、status_code。不要只看 count。

5. 为什么当前 runbook 明确写 teaching signal, not a paging alert？

因为当前指标来自进程内教学 counter，没有生产级聚合、持久化、阈值治理和值班升级机制，不能直接作为生产分页告警信号。

⑨ 面试高频真题+解析

面试题 1：如何设计一个 metrics runbook？

先定义观测入口，再列出关键指标和 label；然后把不同 label 组合映射为排障分支；每个分支写清下一步查什么、不能推出什么结论、对应日志/测试/文档在哪里；最后标明生产边界、告警等级、owner 和升级路径。

面试题 2：看到重试相关指标上升，你如何判断是否影响用户？

不能只看 retry_scheduled。应检查是否有 retry_exhausted，成功样本是否仍在增长，失败分类是否集中在特定状态码或 provider，日志中的 attempt、delay、reason 是否符合预期，并结合接口错误率或用户请求结果判断影响。

面试题 3：为什么教学项目的 metrics 不能直接用于生产告警？

教学项目通常使用单进程内存状态，进程重启会丢失，多实例无法聚合，也没有 scrape、存储、保留周期、SLO、告警抑制和值班升级。生产告警必须建立在可靠的指标采集和治理体系之上。

面试题 4：runbook 为什么也要被测试保护？

runbook 承载排障入口和判断分支，一旦漏掉 endpoint、metric name、关键 outcome 或测试链接，排障流程就会失真。轻量合同测试可以防止核心信号和证据链被无意删除。
