0037 Provider Prediction Adapter：把外部 AI JSON 收敛成内部稳定类型

本节目标不是“会发 HTTP 请求”，而是掌握外部 provider 接入的工程边界：请求出站前要有本项目定义的 payload，响应入站后要经过 Pydantic 二次校验，所有传输异常、HTTP 异常、契约异常统一收敛为 AIClientError，上层永远只依赖 PredictionResult。

① 本节核心知识框架

Provider Prediction Adapter
├─ 出站契约：ProviderPredictionPayload(text, mode)
├─ 入站契约：ProviderPredictionBody(label, score)
├─ 传输层：httpx.AsyncClient.post(url, json=...)
├─ HTTP 校验：response.raise_for_status()
├─ 业务校验：ProviderPredictionBody.model_validate(response.json())
├─ 内部模型：PredictionResult(label, score, source)
└─ 异常收敛：Timeout / HTTPStatus / Request / Validation → AIClientError

核心结论：adapter 是外部系统与内部领域模型之间的防腐层。它不把 provider 原始 JSON 继续向上传播，而是把不可信的外部数据转换为项目内部可测试、可替换、可演进的稳定类型。

② 核心概念底层原理

Adapter 的本质是契约翻译。HTTP 200 只表示传输层请求完成，不表示业务 JSON 正确；provider 返回 {"label":"confused","score":1.5} 时，HTTP 层可以成功，但项目业务层必须拒绝。

层次负责内容失败表现
传输层DNS、TCP/TLS、超时、连接、HTTP 状态码TimeoutException、RequestError、HTTPStatusError
数据层响应是否为合法 JSONValueError
业务契约层label 是否在枚举内，score 是否在 0 到 1ValidationError
内部领域层统一输出 PredictionResult上层不感知 provider 私有字段

与 DTO、schema、client 的区别：DTO 偏数据形状，schema 偏校验规则，client 偏调用能力；adapter 同时承担“构造出站数据、调用外部服务、校验入站数据、转换内部结果、映射异常”的边界职责。

③ 全套代码逐行精读解析

class ProviderPredictionPayload(BaseModel):
    text: str
    mode: str

BaseModel 让出站 JSON 结构显式化。若直接手写字典，字段拼写错误只能在 provider 侧暴露；使用模型后，payload 的业务语义集中在一个类型中。

class ProviderPredictionBody(BaseModel):
    label: PredictionLabel
    score: float = Field(ge=0, le=1)

PredictionLabel 将 provider 标签限制在项目允许集合内；Field(ge=0, le=1) 把置信度约束下沉到模型层。不写这层校验，非法分数会污染 PredictionResponse，前端和测试只能事后补救。

async def request_provider_prediction(
    url: str,
    text: str,
    mode: str,
    config: AIClientConfig,
    client: httpx.AsyncClient | None = None,
    sleep: RetrySleep = asyncio.sleep,
    random_fraction: RetryRandom = random.random,
) -> PredictionResult:

async def 表示外部 HTTP I/O 不阻塞事件循环；client 可选是为了兼容生产复用 lifespan client 与单元测试注入 MockTransport；sleep/random_fraction 是后续 retry/backoff 可测试性的扩展点。

owns_client = client is None
active_client = client or httpx.AsyncClient(
    timeout=config.timeout_seconds,
    trust_env=False,
)

owns_client 贯彻“谁创建谁关闭”。trust_env=False 避免本机代理环境变量影响教学与测试结果；生产环境是否信任代理应由部署策略显式决定。

response = await active_client.post(
    url,
    json=ProviderPredictionPayload(text=text, mode=mode).model_dump(),
)
response.raise_for_status()
body = ProviderPredictionBody.model_validate(response.json())

json= 让 HTTPX 自动设置 JSON 请求体；raise_for_status() 把 4xx/5xx 推入异常分支；model_validate() 对 provider 响应做第二道边界校验。不写最后一步，HTTP 200 下的坏数据会被当成真结果。

return PredictionResult(
    label=body.label,
    score=round(body.score, 2),
    source=f"{config.provider}-provider",
)

返回内部结果而非 provider 原始 JSON。round 固定对外分数粒度；source 保留来源证据，便于后续 headers、日志和测试断言。

④ 核心机制运行全流程拆解

调用方
  → request_provider_prediction(text, mode, config, client)
  → ProviderPredictionPayload.model_dump()
  → HTTP POST provider prediction URL
  → response.raise_for_status()
  → response.json()
  → ProviderPredictionBody.model_validate(...)
  → PredictionResult
  → 上层 router/service

异常流与成功流同等重要：超时映射为 ai_provider_timeout；HTTP 4xx/5xx 映射为 ai_provider_http_error；连接、DNS、协议等请求错误映射为 ai_provider_request_error；非 JSON 或 schema 不合法映射为 ai_provider_invalid_response。

⑥ 开发常见报错+坑点+解决方案

  • coroutine was never awaited：调用 request_provider_prediction() 时忘记 await。解决：从 async endpoint 或 async service 中调用。
  • ValidationError 泄漏到 API：未捕获 provider 响应校验异常。解决：在 adapter 内统一转为 AIClientError(error_code="ai_provider_invalid_response")。
  • 测试访问真实外网：未注入 httpx.MockTransport。解决：测试传入自定义 AsyncClient，只验证本项目行为。
  • 请求体字段错位：手写 dict 使用 message 而 provider 期望 text。解决：统一通过 ProviderPredictionPayload 生成。

⑧ 课后记忆习题+标准答案

1. 为什么 provider 返回 200 后仍然要用 Pydantic 校验？

因为 200 只证明 HTTP 请求成功，不证明业务字段合法。label 枚举、score 范围、JSON 结构都必须在项目边界重新校验。

2. ProviderPredictionPayload 和 ProviderPredictionBody 分别解决什么问题？

前者定义发给 provider 的请求 JSON，后者定义从 provider 收到并必须校验的响应 JSON。

3. 为什么 adapter 返回 PredictionResult 而不是原始 dict？

内部上层代码应依赖项目稳定模型，而不是依赖外部 provider 的字段形状。

4. owns_client 的工程意义是什么？

避免关闭由调用方管理的共享 client，同时确保函数自己创建的临时 client 不泄漏连接资源。

⑨ 面试高频真题+解析

题 1：如何设计一个第三方 AI provider adapter？ 答：定义出站 payload 与入站 body；使用可复用 HTTP client 发起请求；对状态码、JSON 和业务字段分层校验；把成功结果转换为内部领域类型；把底层异常收敛为稳定业务错误码。

题 2：为什么不能让 controller/router 直接解析第三方 JSON？ 答：router 会同时承担协议翻译、provider 契约、异常映射和响应组装，导致职责混乱。provider 变化时公共 API 层被迫变更，测试也更难隔离。

题 3：如果 provider schema 发生变化，应该改哪里？ 答：优先改 adapter 的 provider 请求/响应模型和映射逻辑；只要内部 PredictionResult 不变，上层 service/router 和外部 API 可以保持稳定。
