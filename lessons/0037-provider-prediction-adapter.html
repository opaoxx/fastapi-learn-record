Lesson 0037

Provider Prediction Adapter：把外部 JSON 翻译成内部预测结果

第 36 课我们把 provider health check 接进了 FastAPI。本节先不急着改 /predict 的公共行为，而是在 service 层实现一个真正的 provider prediction adapter：向外部 provider 发送 JSON，请求预测结果，再把外部 JSON 校验并转换成本项目内部稳定的 PredictionResult。

0. 本节学习目标

  • 理解 provider adapter 为什么要负责“本项目契约”和“外部 provider 契约”的转换。
  • 学会用 httpx.AsyncClient.post(..., json=...) 发送 provider 请求 payload。
  • 理解外部响应必须再次校验，不能因为 provider 返回 200 就直接相信 JSON 内容。
  • 学会用 Pydantic 模型校验 provider 返回体。
  • 学会为成功、上游 HTTP 错误、上游无效 JSON 写 adapter 测试。

1. 背景：为什么 health check 之后不能直接“上真 AI”

health check 只回答一个问题：provider 能不能访问。真正的预测调用要复杂得多，因为它至少涉及两次翻译：

  • 把本项目的 PredictionRequest 思路翻译成 provider 接受的请求 JSON。
  • 把 provider 返回的 JSON 翻译成本项目内部的 PredictionResult。

如果把这些翻译逻辑直接写进 /predict 路由，路由会很快变胖：它既要处理 HTTP 请求体，又要懂 provider 字段，又要懂异常映射，还要懂本项目的响应模型。真实项目里这会让代码难测、难换 provider、难解释。

所以本节把 provider prediction 调用放在 services/provider_http.py。它暂时不改变 /predict 的外部行为，只先把“外部预测调用”这块砖烧结实。

2. 核心概念：两个契约之间需要 adapter

这里有两个契约：

内部契约

项目自己的 Python 类型，例如 PredictionResult(label, score, source)。这是 router、tests、任务系统可以依赖的稳定结果。

外部契约

provider 的 HTTP JSON 格式，例如本节约定 provider 返回 {"label": "positive", "score": 0.876}。

adapter 的职责是站在两个契约中间：

local text/mode
  -> ProviderPredictionPayload
  -> HTTP POST JSON
  -> provider JSON response
  -> ProviderPredictionBody validation
  -> PredictionResult

这就是“适配器”的含义：不是把外部对象原样搬进项目，而是把外部形状转换成内部稳定形状。

3. 为什么 provider 返回 200 也不能直接相信

初学者常犯一个错误：只要 response.status_code == 200，就认为结果一定正确。实际上，200 只说明 HTTP 层请求成功，不说明业务 JSON 一定符合你的期待。

provider 可能返回：

  • {"label": "positive", "score": 0.87}：这是我们想要的。
  • {"label": "confused", "score": 1.5}：HTTP 成功，但业务字段非法。
  • {"result": "positive"}：字段名和我们约定不一致。
  • "temporarily unavailable"：返回体根本不是对象。

所以本节专门创建 ProviderPredictionBody，用 Pydantic 对 provider 返回 JSON 再做一次边界校验。

4. 本节项目改动概览

  • 在 services/provider_http.py 中新增 ProviderPredictionPayload。
  • 新增 ProviderPredictionBody，校验 provider 返回的 label 和 score。
  • 新增 request_provider_prediction() 异步函数。
  • 新增 adapter 测试：成功路径、非法 provider JSON、provider HTTP 503。

本节没有修改 /predict。这是有意的：先学清楚 provider adapter 的边界，下一课再讨论如何把 async provider adapter 接进现有预测路由。

5. 代码精读一：provider 请求 payload

class ProviderPredictionPayload(BaseModel):
    text: str
    mode: str

这个模型表达“我们打算发给 provider 的 JSON 长什么样”。本节保持简单，只发送 text 和 mode：

{
  "text": "FastAPI is great and pleasant.",
  "mode": "careful"
}

为什么不直接手写字典？手写字典当然能跑，但模型有两个优点：第一，字段结构有名字；第二，未来 provider payload 变复杂时，可以把字段约束、默认值、嵌套结构集中放在模型里。

6. 代码精读二：provider 响应体校验

class ProviderPredictionBody(BaseModel):
    label: PredictionLabel
    score: float = Field(ge=0, le=1)

这段模型是本节最重要的防线。

  • label: PredictionLabel 表示 label 只能是项目允许的标签，例如 positive 或 neutral。
  • score: float = Field(ge=0, le=1) 表示置信度必须在 0 到 1 之间。

如果 provider 返回 {"label": "confused", "score": 1.5}，HTTP 层可能仍然是 200，但 Pydantic 会拒绝这个业务结果。这样错误会被我们转换成稳定的 AIClientError(error_code="ai_provider_invalid_response")。

7. 代码精读三：发送 POST 并映射成功结果

async def request_provider_prediction(
    url: str,
    text: str,
    mode: str,
    config: AIClientConfig,
    client: httpx.AsyncClient | None = None,
) -> PredictionResult:
    owns_client = client is None
    active_client = client or httpx.AsyncClient(
        timeout=config.timeout_seconds,
        trust_env=False,
    )

    try:
        response = await active_client.post(
            url,
            json=ProviderPredictionPayload(text=text, mode=mode).model_dump(),
        )
        response.raise_for_status()
        body = ProviderPredictionBody.model_validate(response.json())
    finally:
        if owns_client:
            await active_client.aclose()

    return PredictionResult(
        label=body.label,
        score=round(body.score, 2),
        source=f"{config.provider}-provider",
    )

这段代码有几个关键点：

  • client 可传可不传：测试或未来 FastAPI 路由可以传入 lifespan-managed client。
  • owns_client 继续遵守“谁创建，谁关闭”的资源生命周期原则。
  • json=... 让 HTTPX 发送 JSON 请求体，而不是表单或纯字符串。
  • raise_for_status() 把 4xx/5xx 上游响应提升为错误路径。
  • ProviderPredictionBody.model_validate(...) 把外部 JSON 收紧成本项目认可的字段。
  • 返回的是内部 PredictionResult，不是 provider 原始 JSON。

8. 代码精读四：错误映射

except httpx.TimeoutException as exc:
    raise AIClientError(
        message="The AI provider prediction request timed out.",
        error_code="ai_provider_timeout",
    ) from exc
except httpx.HTTPStatusError as exc:
    raise AIClientError(
        message=f"The AI provider returned HTTP {exc.response.status_code}.",
        error_code="ai_provider_http_error",
    ) from exc
except httpx.RequestError as exc:
    raise AIClientError(
        message="The AI provider could not be reached.",
        error_code="ai_provider_request_error",
    ) from exc
except (ValueError, ValidationError) as exc:
    raise AIClientError(
        message="The AI provider returned an invalid prediction response.",
        error_code="ai_provider_invalid_response",
    ) from exc

这里把四类失败都收束成 AIClientError：

  • 超时：provider 太慢。
  • HTTP 错误：provider 返回 4xx/5xx。
  • 请求错误：网络连接、DNS、协议等问题。
  • 响应体错误：provider 返回的 JSON 不符合本项目期待。

这就是“内部错误语言统一”。上层 router 不需要知道 HTTPX 的所有异常类型，只需要捕获 AIClientError。

9. 测试精读一：检查真的发出了正确 JSON

def test_request_provider_prediction_sends_payload_and_maps_response() -> None:
    captured_request_body = {}

    def handler(request: httpx.Request) -> httpx.Response:
        nonlocal captured_request_body
        captured_request_body = json.loads(request.content)
        return httpx.Response(
            status_code=200,
            json={"label": "positive", "score": 0.876},
            request=request,
        )

    transport = httpx.MockTransport(handler)
    client = httpx.AsyncClient(transport=transport)
    result = run_async_prediction(client)

    assert captured_request_body == {
        "text": "FastAPI is great and pleasant.",
        "mode": "careful",
    }
    assert result.label == "positive"
    assert result.score == 0.88
    assert result.source == "demo-provider"

这个测试不仅检查返回值，还检查 adapter 发出去的请求体。原因是 provider adapter 的一半职责就是构造外部请求 payload。如果只测返回值，可能漏掉“请求 JSON 发错字段名”的问题。

10. 测试精读二：200 但 JSON 不合法

def test_request_provider_prediction_maps_invalid_json_to_ai_client_error() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(
            status_code=200,
            json={"label": "confused", "score": 1.5},
            request=request,
        )

    transport = httpx.MockTransport(handler)
    client = httpx.AsyncClient(transport=transport)

    try:
        run_async_prediction(client)
    except AIClientError as exc:
        error = exc

    assert error.error_code == "ai_provider_invalid_response"

这条测试体现了一个非常重要的工程判断：外部服务即使返回 200，也只是通过了 HTTP 层检查；业务字段还必须通过我们的模型校验。

运行 provider adapter 测试：

预期看到：

再运行全量测试：

.\.venv\Scripts\python.exe -m pytest -q

观察点：

  • 成功路径里，provider 返回 0.876，内部结果变成 0.88。
  • 非法 provider label 或 score 会变成 ai_provider_invalid_response。
  • provider HTTP 503 会变成 ai_provider_http_error。

12. 常见错误与诊断

错误一：直接把 provider JSON 返回给前端

这会让前端依赖 provider 的字段形状。provider 一变，自己的公共 API 也跟着震动。

错误二：只检查状态码，不校验响应体

HTTP 200 不等于业务 JSON 合法。外部响应进入项目边界时要再次校验。

错误三：adapter 里吞掉错误

失败不能悄悄变成默认中性结果，否则排查 provider 问题会非常痛苦。

错误四：测试只测 happy path

provider adapter 的价值在边界失败处理。至少要测 HTTP 错误和非法响应体。

13. 面试视角：怎样解释 provider adapter

我不会让路由直接依赖外部 provider 的 JSON 形状，而是会写一个 provider adapter。它负责构造 provider 请求 payload，发送 HTTP 请求，校验 provider 响应体，并把结果转换成项目内部稳定的 PredictionResult。所有 timeout、HTTP 错误、请求错误和响应体校验失败都会映射成统一的 AIClientError，上层 API 只需要处理这个稳定错误类型。

14. 小测：先回忆，再展开

1. 为什么本节没有直接改 /predict？

因为真实 provider 调用涉及 async、外部 JSON 契约和错误映射。先把 adapter 边界讲清楚，下一步再接路由更稳。

2. 为什么 provider 返回 200 后还要用 Pydantic 校验？

因为 200 只说明 HTTP 请求成功，不保证业务字段符合本项目的标签、分数范围和响应结构要求。

3. ProviderPredictionPayload 和 ProviderPredictionBody 分别代表什么？

前者代表发给 provider 的请求 JSON，后者代表从 provider 收到并需要校验的响应 JSON。

4. 为什么 adapter 返回 PredictionResult 而不是原始 JSON？

因为上层代码应该依赖项目内部稳定类型，而不是直接依赖某个外部 provider 的字段形状。

15. 本节总结

本节完成了 provider prediction adapter 的第一版：它能发送 JSON payload，解析并校验 provider 返回 JSON，把成功结果转换成 PredictionResult，并把外部失败统一转换成 AIClientError。这让真实 provider 调用不再是散落在路由里的临时代码，而是一个可测试、可替换、可解释的边界模块。
