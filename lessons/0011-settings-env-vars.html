Lesson 0011

配置管理和环境变量：不要把秘密写死在代码里

真实项目会有数据库地址、环境名称、API Key、模型服务地址等配置。它们不应该散落在代码里，而应该由统一的 Settings 对象管理。

0. 本节学习目标

  • 理解“配置”和“业务代码”的区别。
  • 理解环境变量为什么适合保存不同环境的配置。
  • 看懂 pydantic-settings 的 BaseSettings。
  • 理解 @lru_cache 为什么常用于 settings。

1. 什么是配置

配置是“部署或运行时会变化，但业务逻辑不应该变化”的信息。例如：

  • 应用名称：开发环境可能叫 First FastAPI API，生产环境可能叫正式服务名。
  • 环境名称：development、test、production。
  • API Key：保护接口用的密钥。
  • 数据库地址：本地 SQLite、测试数据库、生产数据库可能都不一样。
  • AI 服务密钥：例如 OpenAI、向量数据库、模型网关的 key。

如果这些值硬编码在代码里，就会导致改环境必须改代码，还可能把秘密提交到 GitHub。

2. 环境变量是什么

环境变量是操作系统或运行环境提供给程序的键值对。程序启动时读取它们，例如：

FIRST_API_ENVIRONMENT=development
FIRST_API_API_KEY=dev-secret-key

FastAPI 官方设置文档推荐使用 Pydantic Settings 来处理应用配置，它可以从环境变量和 .env 文件读取值，并像 Pydantic 模型一样做类型处理。

安全直觉：.env.example 可以提交，因为它只展示变量名和示例值；真正的 .env 不提交，因为它可能包含真实密钥。

3. settings.py 逐行讲解

class Settings(BaseSettings):
    app_name: str = "First FastAPI API"
    environment: str = "development"
    api_key: str = "dev-secret-key"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_prefix="FIRST_API_",
        extra="ignore",
    )

这段代码表示：

  • BaseSettings：让这个类具备从环境变量读取值的能力。
  • app_name、environment、api_key：三个配置字段。
  • env_file=".env"：允许从项目根目录的 .env 文件读取配置。
  • env_prefix="FIRST_API_"：环境变量统一加前缀，避免和系统里其他变量撞名。

例如字段 api_key 对应的环境变量就是 FIRST_API_API_KEY。

4. get_settings 和缓存

@lru_cache
def get_settings() -> Settings:
    return Settings()

@lru_cache 会缓存函数结果。第一次调用 get_settings() 时创建 Settings 对象，后面再调用会复用同一个对象。

官方文档也提到，用缓存可以避免每次请求都重新读取 dotenv 文件。测试时仍然可以通过依赖覆盖或清缓存来调整配置。

5. main.py 如何使用 Settings

settings = get_settings()

app = FastAPI(
    title=settings.app_name,
    ...
)

这表示应用标题不再写死，而是来自配置对象。以后你可以通过环境变量改变应用名称，而不用改业务代码。

6. 公开配置接口

@router.get("/config/public", response_model=PublicConfigResponse)
def read_public_config(settings: Annotated[Settings, Depends(get_settings)]):
    return PublicConfigResponse(
        app_name=settings.app_name,
        environment=settings.environment,
        api_key_configured=bool(settings.api_key),
    )

这个接口展示的是“可以公开的配置”。注意它只返回 api_key_configured，不会返回真实 api_key。这是非常重要的安全边界。

7. 实验：创建 .env

复制 .env.example 为 .env，然后修改：

FIRST_API_APP_NAME="My FastAPI Learning API"
FIRST_API_ENVIRONMENT="local"
FIRST_API_API_KEY="my-local-key"

重启服务后访问：

GET /config/public

你应该看到新的 app_name 和 environment。真实 API Key 不会出现在响应里。

9. 小测：先回忆再展开

1. 为什么真实 .env 不应该提交？

因为它可能包含 API Key、数据库密码、模型服务密钥等敏感信息。

2. FIRST_API_API_KEY 为什么能对应 api_key？

因为 Settings 配置了 env_prefix="FIRST_API_"，字段名会和前缀组合成环境变量名。

3. 为什么公开接口只返回 api_key_configured？

因为客户端可以知道密钥是否配置，但绝不能拿到真实密钥值。

10. 本节总结

配置管理让代码和运行环境解耦。BaseSettings 负责从环境变量和 .env 读取配置，.env.example 负责告诉开发者需要哪些变量，真实 .env 负责保存本地秘密并被 Git 忽略。
