AI 客户端错误契约：把内部模型失败映射为稳定 503 响应

AI 调用失败不是简单的 Python 异常打印问题，而是 API 契约问题。本节把 AIClientError、PredictionServiceError、HTTPException 与 ErrorResponse 串起来，建立“内部失败可变、外部响应稳定”的错误边界。

① 本节核心知识框架

AI 错误处理链路
├─ 内部异常：AIClientError
│  ├─ message：人类可读说明
│  └─ error_code：程序可判断错误码
├─ 服务层异常：PredictionServiceError
│  ├─ backend：demo/provider
│  ├─ elapsed_ms：耗时
│  └─ error_code/message
├─ HTTP 映射：HTTPException(status_code=503)
│  ├─ detail.error_code
│  ├─ detail.message
│  └─ X-Prediction-* 响应头
├─ 响应模型：ErrorResponse
└─ 测试策略：dependency_overrides 注入失败客户端

② 核心概念底层原理

同步接口 POST /predict 的语义是“本次请求立即返回预测结果”。如果模型不可用，错误仍发生在同一个 HTTP 请求周期内，因此 router 可以把内部异常转换成 503 Service Unavailable。503 表示服务当前无法完成请求，常见于上游模型超时、限流、临时不可用或返回非法格式。

不要把 Exception 原文直接返回给客户端。原始异常消息不稳定，可能包含内部类名、路径、堆栈、provider 细节或敏感配置。正确做法是定义稳定错误码，例如 test_ai_unavailable、demo_ai_failure、ai_provider_http_error，让前端、测试和日志系统按错误码处理。

与 422、500 的区别：422 是请求参数不符合契约，问题在客户端输入；500 是服务端未分类 bug；503 是后端依赖或能力暂不可用，通常具备临时性和可重试语义。AI provider 错误多数不应伪装成 422。

③ 全套代码逐行精读解析

class AIClientError(Exception):
    def __init__(
        self,
        message: str = "The AI service is temporarily unavailable.",
        error_code: str = "ai_client_unavailable",
    ) -> None:
        super().__init__(message)
        self.message = message
        self.error_code = error_code
  • class AIClientError(Exception)：把 AI 边界失败从普通异常中分类出来；没有分类时 router 很难判断该返回 503 还是 500。
  • message 默认值提供安全的通用说明；不传时也不会暴露内部异常。
  • error_code 默认值提供程序稳定分支；前端不要依赖英文 message 做逻辑判断。
  • super().__init__(message) 保留 Python 异常标准行为，日志和调试仍可读取异常文本。
  • self.message/self.error_code 固化结构化字段；如果只使用 str(exc)，测试和客户端都缺少稳定键。
class ErrorDetail(BaseModel):
    error_code: str
    message: str

class ErrorResponse(BaseModel):
    detail: ErrorDetail
  • ErrorDetail 对齐 FastAPI 错误响应习惯，把结构化错误放在 detail 内。
  • error_code 是机器读字段，适合前端分支、监控聚合、测试断言。
  • message 是人类读字段，适合 UI 提示和调试摘要。
  • ErrorResponse 让 OpenAPI 文档明确 503 响应形状；不声明会让调用方只能猜错误体。
@router.post(
    "/predict",
    response_model=PredictionResponse,
    responses={status.HTTP_503_SERVICE_UNAVAILABLE: {"model": ErrorResponse}},
)
async def predict(...):
    try:
        result = await run_prediction(...)
    except PredictionServiceError as exc:
        log_prediction_failed(exc)
        raise HTTPException(
            status_code=status.HTTP_503_SERVICE_UNAVAILABLE,
            detail={"error_code": exc.error_code, "message": exc.message},
            headers=build_prediction_headers(...),
        ) from exc
  • responses={503: ...} 把错误模型写入 OpenAPI；生产协作中错误契约和成功契约同等重要。
  • try 包住 service 调用，而不是包住整个函数，避免吞掉响应构造等无关 bug。
  • except PredictionServiceError 只捕获已分类业务失败；未知异常继续交给框架变成 500，便于暴露真实 bug。
  • log_prediction_failed(exc) 在抛 HTTP 错误前记录结构化日志；否则只看访问日志很难知道 provider 和耗时。
  • HTTPException(status_code=503) 把内部失败映射为 HTTP 语义。
  • detail={...} 必须是可 JSON 序列化对象；直接放异常实例会序列化失败。
  • headers=build_prediction_headers(...) 给调用方补充后端、耗时、错误码等调试信息。
  • from exc 保留异常链，服务端日志能追溯根因；不影响对外响应体。

④ 核心机制运行全流程拆解

客户端 POST /predict
  -> FastAPI 校验 PredictionRequest
  -> 注入 settings、ai_client、provider_http_client
  -> run_prediction()
       -> DemoAIClient / provider adapter 执行预测
       -> AIClientError 或 provider 错误
       -> 封装为 PredictionServiceError
  -> router 捕获 PredictionServiceError
       -> log_prediction_failed()
       -> build_prediction_headers()
       -> raise HTTPException(503, detail=...)
  -> FastAPI 生成 JSON 错误响应
  -> 客户端收到 detail.error_code/detail.message

这个链路体现了三层错误边界：provider 原始错误在最内层，service 统一业务错误语义，router 最后映射 HTTP。这样后续把 demo 模型换成真实 provider 时，前端仍读取同一套错误结构。

⑤ 重难点、易混淆点对比辨析

概念正确用法误区

HTTPExceptionrouter 层 HTTP 映射在底层 service 到处抛

⑥ 开发常见报错+坑点+解决方案

  • 前端拿不到 detail.error_code：抛错时 detail 写成字符串。解决：传入字典 {"error_code": ..., "message": ...}。
  • OpenAPI 看不到 503 模型：路由没有配置 responses。解决：补 responses={503: {"model": ErrorResponse}}。
  • 测试误连真实模型：没有覆盖 get_ai_client。解决：使用 dependency_overrides 注入失败替身。
  • 错误码无法聚合：每次 message 文案不同。解决：固定 error_code，message 只做人类提示。
  • 异常链丢失：raise HTTPException(...) 没有 from exc。解决：保留异常因果，便于服务端排障。

⑧ 课后记忆习题+标准答案

1. 为什么 AI 模型失败适合返回 503？

请求本身可能合法，但后端依赖的模型能力暂时不可用；503 能表达临时不可用和潜在重试语义。

2. error_code 和 message 分别服务谁？

error_code 服务程序判断、测试和监控聚合；message 服务人类阅读和 UI 提示。

3. 为什么不要捕获所有 Exception 后都返回 503？

未知异常可能是代码 bug 或数据损坏，全部伪装成 503 会隐藏真实问题。只应把已分类的依赖失败映射为 503。

4. responses={503: {"model": ErrorResponse}} 的价值是什么？

它让 OpenAPI 暴露错误响应契约，调用方可以提前知道失败时的 JSON 结构。

⑨ 面试高频真题+解析

题 1：如何设计 AI 接口的错误响应？

内部用业务异常分类 provider 失败，对外统一状态码和结构化错误体，至少包含稳定 error_code 与安全 message；同时记录日志、耗时、provider、trace id，避免暴露原始异常。

题 2：503 和 500 在 AI 服务中的区别是什么？

503 表示依赖服务暂不可用或过载，调用方可考虑重试；500 表示服务端内部未处理错误。模型超时、限流常映射为 503，代码 bug 不应伪装成 503。

题 3：为什么错误响应也属于 API 契约？

前端、SDK、监控、自动重试都依赖错误结构。如果错误体不稳定，成功接口再稳定也无法支撑可靠系统。
