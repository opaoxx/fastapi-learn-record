Lesson 0050

Provider Metrics Basics：把错误分类变成可统计的指标标签

上一课我们建立了 provider error taxonomy。本节继续向生产可观测性靠近：日志适合解释单次请求，metrics 适合回答“发生了多少次、比例是多少、最近有没有变多”。我们先不引入监控平台，只把指标标签的形状设计清楚并用测试保护。

0. 本节学习目标

  • 理解 logs 和 metrics 的职责差异。
  • 知道什么是 metric label，以及为什么 label 必须低基数。
  • 掌握 provider prediction 指标标签应包含哪些稳定字段。
  • 理解为什么不能把用户输入、响应体或异常全文放进指标标签。
  • 学会用测试保护 metrics label 的形状。

1. 背景：日志回答“这一次”，指标回答“一段时间”

前几节课我们已经给 provider retry、retry exhausted 和 fail fast 加了结构化日志。日志非常适合排查某一次请求：第几次 attempt 失败、状态码是多少、有没有 Retry-After、最终是 exhausted 还是 fail fast。

但当你想回答“今天 provider 503 比昨天多了吗？”、“fail fast 主要来自 400 还是 invalid response？”、“retry exhausted 占所有 provider 失败多少比例？”时，逐条翻日志就不够高效了。这类问题更适合 metrics。

Metrics 可以先理解成随时间记录的数值信号，例如计数器、耗时分布、当前队列长度。本节只做最基础的一步：设计 provider prediction 指标的标签。

2. 核心概念：Metric Label 必须稳定且低基数

Metric label 是附在指标上的维度。例如同一个计数器可以按 outcome、failure_category、status_code 分组统计。

provider_prediction_total{
  operation="prediction",
  outcome="fail_fast",
  failure_category="non_retryable_http_status",
  error_code="ai_provider_http_error",
  status_code="400"
} 17

标签最重要的规则是低基数。基数可以理解为一个标签可能出现多少种不同值。status_code 只有有限几个值，基数低；用户输入文本几乎每次都不同，基数极高。

高基数标签会让监控系统存储和查询成本暴涨，也可能泄露隐私。因此指标标签只应该放稳定、有限、可聚合的分类值。

3. 本节不直接接 Prometheus 的原因

你可能会问：既然讲 metrics，为什么不直接安装 Prometheus client？原因是课程当前重点是 FastAPI、provider 边界、错误分类和可观测性思维。过早接入监控平台会把注意力转移到部署、抓取配置和格式暴露上。

本节先做更底层也更稳定的事：把“应该记录哪些 label”设计成一个小 helper。以后无论接 Prometheus、OpenTelemetry、StatsD，还是云厂商监控，这组 label 都能作为输入。

4. 代码精读一：ProviderMetricLabels

本节新增了一个小 dataclass，表达一次 provider prediction 指标应该携带的标签：

@dataclass(frozen=True)
class ProviderMetricLabels:
    operation: ProviderMetricOperation
    outcome: ProviderMetricOutcome
    failure_category: ProviderMetricFailureCategory
    error_code: str
    status_code: str

这些字段都刻意保持低基数：

  • operation：当前只有 prediction。
  • outcome：成功、准备重试、重试耗尽、快速失败。
  • failure_category：沿用上一课 taxonomy，成功时为 none。
  • error_code：稳定错误码，成功时为 none。
  • status_code：HTTP 状态码字符串，没有时为 none。

5. 代码精读二：as_dict()

很多 metrics client 最终都需要字典形式的 labels，所以 dataclass 提供了 as_dict()：

def as_dict(self) -> dict[str, str]:
    return {
        "operation": self.operation,
        "outcome": self.outcome,
        "failure_category": self.failure_category,
        "error_code": self.error_code,
        "status_code": self.status_code,
    }

这让测试也更直接。我们不需要启动监控服务，只要断言字典形状，就能保护后续接入 metrics backend 时最容易变乱的部分。

6. 代码精读三：build_provider_metric_labels()

helper 的职责是把默认值收拢起来，让成功场景和失败场景都能生成稳定标签：

def build_provider_metric_labels(
    outcome: ProviderMetricOutcome,
    failure_category: ProviderMetricFailureCategory = "none",
    error_code: str = "none",
    status_code: int | None = None,
) -> ProviderMetricLabels:
    return ProviderMetricLabels(
        operation="prediction",
        outcome=outcome,
        failure_category=failure_category,
        error_code=error_code,
        status_code=str(status_code) if status_code is not None else "none",
    )

注意这里把 status_code 转成字符串。很多 metrics 系统的 label 值都是字符串；在 helper 边界统一转换，后面调用方就不用反复处理。

7. 成功和失败的标签形状

成功场景应该没有 failure category、error code 和 status code：

{
  "operation": "prediction",
  "outcome": "success",
  "failure_category": "none",
  "error_code": "none",
  "status_code": "none",
}

HTTP 400 快速失败则可以这样表达：

{
  "operation": "prediction",
  "outcome": "fail_fast",
  "failure_category": "non_retryable_http_status",
  "error_code": "ai_provider_http_error",
  "status_code": "400",
}

这两个例子展示了 metrics 和 logs 的分工：metrics 不保存完整上下文，只保存适合聚合的维度；具体某一次请求发生了什么，仍然交给结构化日志。

8. 动手实验：运行 provider HTTP 测试

在项目根目录运行：

.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q

本节新增两个测试。第一个测试成功标签默认值：

labels = build_provider_metric_labels(outcome="success")

assert labels.as_dict()["failure_category"] == "none"
assert labels.as_dict()["error_code"] == "none"

第二个测试失败标签不会包含用户请求文本：

assert "FastAPI is great and pleasant." not in labels.as_dict().values()

这行断言不是为了检查某个具体功能，而是在提醒一个生产规则：指标标签不应该携带高基数、敏感或不可控的业务文本。

9. 常见错误

  • 把日志和指标当成同一个东西：日志解释单次请求，指标统计一段时间。
  • 把用户输入放进 label：高基数且可能泄露隐私。
  • 把完整异常消息放进 label：异常文本可能包含动态内容，基数不可控。
  • label 名随手变化：仪表盘和告警会失效。
  • 成功场景没有统一默认值：查询时要同时处理空值、缺字段和 none。
  • 过早引入监控平台：还没想清楚 label 形状，就先背上部署复杂度。

10. 面试视角：如何解释 logs 和 metrics 的分工

我会用结构化日志解释单次 provider 请求发生了什么，例如 attempt、Retry-After、delay 和 fail-fast reason；用 metrics 聚合一段时间内的趋势，例如 success、retry_exhausted、fail_fast 的数量和比例。metrics label 必须低基数，所以我只放 outcome、failure_category、error_code、status_code 这类稳定字段，不放用户输入或完整响应体。

这个回答能体现你不仅知道“要监控”，还知道监控数据的形状会影响成本、隐私和后续查询质量。

11. 记忆练习

1. logs 和 metrics 的核心区别是什么？

日志适合解释单次事件的上下文；指标适合统计一段时间内的数量、比例和趋势。

2. 什么是高基数 label？为什么危险？

高基数 label 有大量不同取值，例如用户输入文本。它会增加监控系统成本，也可能泄露隐私。

3. 为什么成功场景也要有 none 默认值？

统一默认值能让指标查询更简单，避免同一个维度出现缺字段、空字符串和 none 混用。

4. 本节为什么不直接接 Prometheus？

因为当前学习重点是指标标签设计。先把低基数、稳定标签想清楚，再接具体监控平台更稳。

12. 本节总结

本节把 provider error taxonomy 继续推进到 metrics label 设计。我们新增了 ProviderMetricLabels 和 build_provider_metric_labels()，用测试保护成功和失败场景的标签形状，并明确禁止把用户文本放进指标标签。
