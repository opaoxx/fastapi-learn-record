0050 Provider Metrics Basics：把错误分类沉淀为低基数指标标签

本节从日志进入指标。日志用于解释单次请求，metrics 用于统计一段时间内的数量、比例和趋势。核心代码是 ProviderMetricLabels 与 build_provider_metric_labels()，它们把 provider outcome、failure category、error code、status code 固定为可聚合、可测试、可迁移的低基数字段。

① 本节核心知识框架

Provider Metrics Basics
├─ 日志 vs 指标
│  ├─ logs：单次事件上下文，适合排查“这一请求发生了什么”
│  └─ metrics：聚合数值信号，适合回答“这类事情发生了多少次”
├─ 标签设计
│  ├─ operation：prediction
│  ├─ outcome：success / retry_scheduled / retry_exhausted / fail_fast
│  ├─ failure_category：none / timeout / request_error / ...
│  ├─ error_code：none / ai_provider_http_error / ...
│  └─ status_code：none / 400 / 503 / ...
├─ 低基数原则
│  ├─ 允许：枚举值、状态码、稳定错误码
│  └─ 禁止：用户输入、异常全文、request id、响应体
└─ 代码落点
   ├─ ProviderMetricLabels(frozen=True)
   ├─ ProviderMetricOutcome
   ├─ ProviderMetricFailureCategory
   └─ build_provider_metric_labels()

② 核心概念底层原理

metric label 是指标的分组维度。一个 counter 加上不同 label set，会形成不同时间序列。时间序列越多，存储、查询、压缩、索引和告警计算成本越高。

基数 是某个标签可能出现的不同取值数量。outcome 只有 4 个值，基数低；用户输入文本几乎每次不同，基数极高；异常全文可能携带动态 URL、堆栈、token、文本片段，也属于高基数和高泄漏风险。

低基数示例：
provider_prediction_total{outcome="fail_fast",status_code="400"} 17

高基数反例：
provider_prediction_total{text="用户原始输入...",request_id="..."} 1

metrics 与 logs 的根本区别是“保真度 vs 可聚合性”。日志可以保留更多上下文，但查询成本更高；指标主动压缩上下文，只保留少数稳定维度，以换取长期聚合、仪表盘和告警能力。

③ 全套代码逐行精读解析

ProviderMetricOutcome = Literal[

    "retry_scheduled",

    "fail_fast",

ProviderMetricFailureCategory = ProviderFailureCategory | Literal["none"]

@dataclass(frozen=True)
class ProviderMetricLabels:
    operation: ProviderMetricOperation
    outcome: ProviderMetricOutcome
    failure_category: ProviderMetricFailureCategory
    error_code: str
    status_code: str

    def as_dict(self) -> dict[str, str]:
        return {
            "operation": self.operation,
            "outcome": self.outcome,
            "failure_category": self.failure_category,
            "error_code": self.error_code,
            "status_code": self.status_code,
        }
代码作用设计原因缺失后的问题
ProviderMetricOutcome限定系统动作/结果success、retry_scheduled、retry_exhausted、fail_fast 能表达生命周期只写 failed 会丢失是否仍在恢复、是否最终失败
ProviderMetricFailureCategory复用错误 taxonomy，并允许成功场景 none成功样本也保持同一组 label keys成功与失败样本字段不一致，查询和响应模型复杂
ProviderMetricOperation当前固定为 prediction为未来 summary、embedding 等操作预留维度后续多个 provider 操作共享指标时无法区分
@dataclass(frozen=True)创建不可变 label 对象后续可作为 Counter 的 key可变对象不可哈希，不能稳定聚合
error_code: str记录稳定错误码与 API 错误响应、日志字段对齐只看 status code 无法区分 timeout、request_error、invalid_response
status_code: str记录 provider 原始状态码或 none多数指标系统 label 值按字符串处理调用方反复处理 int/None/string 混合类型
as_dict()转换为响应、测试、导出通用形状隔离 dataclass 内部结构router 和测试会直接依赖对象字段，迁移成本上升
 
def build_provider_metric_labels(
    outcome: ProviderMetricOutcome,
    failure_category: ProviderMetricFailureCategory = "none",
    error_code: str = "none",
    status_code: int | None = None,
) -> ProviderMetricLabels:
    return ProviderMetricLabels(
        operation="prediction",
        outcome=outcome,
        failure_category=failure_category,
        error_code=error_code,
        status_code=str(status_code) if status_code is not None else "none",
    )

该 helper 把默认值集中到一个边界：成功场景无需传 failure 信息，失败场景只补充受控分类。如果调用方到处手写 dict，极易出现 statusCode、status_code、空字符串、None 混用。

④ 核心机制运行全流程拆解

provider prediction 事件
  → 判断 outcome
       ├─ success
       ├─ retry_scheduled
       ├─ retry_exhausted
       └─ fail_fast
  → 判断 failure_category
       ├─ none
       ├─ timeout / request_error
       ├─ retryable_http_status / non_retryable_http_status
       └─ invalid_response
  → 选择 error_code 与 status_code
  → build_provider_metric_labels()
  → ProviderMetricLabels.as_dict()
  → counter / endpoint / text export / tests

数据流中最关键的边界是 build_provider_metric_labels()。它把“业务事件上下文”压缩成“指标可聚合维度”。用户原文、完整异常、响应体不进入这个函数，因为它们不适合成为 label。

成功样本的标准形状如下：

{
  "operation": "prediction",
  "outcome": "success",
  "failure_category": "none",
  "error_code": "none",
  "status_code": "none"
}

HTTP 400 fail-fast 样本的标准形状如下：

{
  "operation": "prediction",
  "outcome": "fail_fast",
  "failure_category": "non_retryable_http_status",
  "error_code": "ai_provider_http_error",
  "status_code": "400"
}

⑧ 课后记忆习题+标准答案

1. 为什么 metrics label 必须低基数？

因为每组 label set 都可能形成独立时间序列。高基数会显著增加存储、索引、查询和告警计算成本，还可能泄漏用户输入或异常细节。

2. 为什么成功样本也要有 failure_category="none"？

为了让所有样本使用同一组 label keys，查询、响应模型、文本导出和测试都可以按统一形状处理，不需要为成功场景写特殊分支。

3. outcome 和 error_code 分别表达什么？

outcome 表达系统动作或结果，例如 success、retry_scheduled、retry_exhausted、fail_fast；error_code 表达稳定错误原因，例如 ai_provider_http_error。

4. 为什么本节不急着接 Prometheus client？

因为初学阶段更重要的是先设计稳定、低基数、可测试的 label contract。具体后端可以替换，但标签形状会长期影响仪表盘和告警。

⑨ 面试高频真题+解析

面试题 1：日志和指标如何分工？

日志记录单次事件上下文，适合排查某次请求的 attempt、delay、Retry-After、异常摘要；指标记录可聚合数值，适合观察一段时间内 success、retry_exhausted、fail_fast 的数量和比例。日志追因，指标看势。

面试题 2：如何判断一个字段能否做 metric label？

看它是否稳定、有限、可聚合、无敏感信息。枚举 outcome、failure category、稳定错误码、有限状态码适合；用户文本、request id、异常全文、完整 URL、响应体不适合。

面试题 3：为什么要用 helper 构造 labels，而不是到处写 dict？

helper 集中默认值、类型转换和字段命名，避免 None、空字符串、int、字符串状态码混用，也避免字段名漂移。后续接入 counter、endpoint、Prometheus 文本导出时可以复用同一契约。
