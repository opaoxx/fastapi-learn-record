Final Review + CSDN Deep Dive

请求体与 Pydantic：把不可信 JSON 变成稳定接口契约

本节聚焦 POST /predict：客户端提交 JSON，请求体由 Pydantic 校验，服务层执行预测，响应模型保持对外稳定。重点不是“怎么写 BaseModel”，而是掌握 API 边界的契约化设计。

① 本节核心知识框架

请求体与 Pydantic
├─ HTTP 请求体：POST/PUT/PATCH 携带业务数据
├─ Pydantic 模型：字段结构 + 类型约束 + 默认值 + JSON Schema
├─ FastAPI 推断规则：Pydantic 参数默认来自 body
├─ 校验结果
│  ├─ 成功：构造 PredictionRequest 对象，进入路径函数
│  └─ 失败：返回 422，路径函数不会执行
├─ 响应契约：PredictionResponse 固定 label/score/source/text_length
└─ AI 服务意义：先校验输入，再进入成本更高的推理或 provider 调用

② 核心概念底层原理

请求体是 HTTP 消息体，不是 URL 参数。GET /items/1 的关键数据在路径；POST /predict 的关键数据在 JSON body。AI 推理、文件任务、创建资源都通常使用请求体，因为输入结构复杂且可能较长。

Pydantic 是边界层数据校验器。外部 JSON 没有 Python 类型，字段可能缺失、类型可能错误、值可能越界。Pydantic 根据模型定义执行解析、转换、约束检查和错误收集，最后生成 Python 对象或结构化错误。

422 与 400 的区别。FastAPI 默认把模型校验失败映射为 422 Unprocessable Entity：请求 JSON 语法可解析，但语义不符合接口契约。400 更常用于无法理解的通用坏请求或自定义协议错误。

教学 Demo 与真实 AI 服务的相同点。即使当前预测逻辑是 demo，PredictionRequest 仍然模拟真实 provider 请求边界：文本长度、模式枚举、温度范围都是推理入口必须控制的约束。

③ 全套代码逐行精读解析

class PredictionRequest(BaseModel):
    text: str = PydanticField(
        min_length=3,
        max_length=500,
        examples=["FastAPI makes backend APIs pleasant."],
    )
    mode: Literal["fast", "careful"] = "fast"
    temperature: float = PydanticField(default=0.2, ge=0, le=1)

class PredictionResponse(BaseModel):
    label: Literal["positive", "neutral"]
    score: float
    source: str
    text_length: int
  • class PredictionRequest(BaseModel)：声明请求体模型。若不继承 BaseModel，FastAPI 无法按 Pydantic 模型生成 body schema。
  • text: str：主输入必须是字符串。若传数组/对象，校验失败，避免后续 len、分词、provider 调用出现类型错误。
  • PydanticField(...)：这里把 Pydantic 的 Field 起别名，避免与 SQLModel 的 Field 混淆。
  • min_length=3：拒绝过短输入。没有该约束，空文本可能进入模型调用，浪费成本且结果无意义。
  • max_length=500：限制教学接口输入规模。生产中通常还会按 token、文件大小、用户套餐做更细限制。
  • examples=[...]：进入 OpenAPI 示例，提高 GitHub 项目可读性和接口自解释能力。
  • mode: Literal["fast", "careful"]：枚举约束。若用普通 str，调用者传 "slow" 也会进入业务逻辑，后续分支容易漏处理。
  • = "fast"：默认值。请求不传 mode 时，Pydantic 自动补齐。
  • temperature: float：AI 参数雏形。虽然当前服务层不一定使用该值，它仍属于公开请求契约。
  • ge=0, le=1：数值闭区间约束。没有约束时，负数或超大值可能传到真实模型 provider 造成错误或不可控输出。
  • PredictionResponse：成功响应模型。label 限定分类结果，score 表示置信度，source 标识预测来源，text_length 回显输入长度用于调试。

) -> PredictionResponse:

payload: PredictionRequest 是本节核心：FastAPI 看到 Pydantic 模型参数后，从 JSON body 取数据并校验。后面几个参数来自依赖注入，说明真实预测接口不仅依赖 body，还依赖配置、AI 客户端和 provider HTTP client。

⑤ 重难点、易混淆点对比辨析

  • 请求模型 vs 响应模型：请求模型约束外部输入；响应模型约束内部输出。两者不应混用，否则可能泄漏内部字段或限制输入演进。
  • 类型注解 vs 运行时校验：普通 Python 类型注解默认不校验；Pydantic 会在模型构造时做运行时校验。
  • 默认值 vs 可选字段：mode = "fast" 是有默认值；str | None = None 才表示允许为空。
  • Literal vs Enum：Literal 适合少量字符串选项；复杂业务状态建议用 Enum 提升复用和可维护性。

⑥ 开发常见报错+坑点+解决方案

  • 422：字段缺失。请求体缺少必填 text。解决：按 /docs schema 传完整 JSON。
  • 422：枚举错误。mode 传 "slow"。解决：只能传 "fast" 或 "careful"。
  • 422：数值越界。temperature 大于 1 或小于 0。解决：遵守 ge=0, le=1。
  • 415/解析异常：客户端没发 JSON 或 Content-Type 错误。解决：用 Content-Type: application/json 并发送合法 JSON。
  • 把 body 写成 query：/predict?text=... 不会填充 PredictionRequest。解决：POST JSON body。

⑦ 生产环境 vs 教学环境 核心差异

  • 输入限制：教学按字符限制；生产常按 token、字节数、用户额度、请求频率、敏感内容策略限制。
  • 错误响应：教学可直接看 Pydantic 422；生产通常会统一错误格式、隐藏内部字段、加入 trace id。
  • 模型参数：教学只保留 mode/temperature；生产会包含模型名、超时、stream、top_p、max_tokens、工具调用等参数。
  • 兼容性：教学可以直接改字段；生产 API 字段变更需要版本化、灰度、兼容窗口和契约测试。

⑧ 课后记忆习题+标准答案

1. 为什么 PredictionRequest 参数默认来自请求体？

它是 Pydantic 模型类型，FastAPI 会把这类复杂模型参数推断为 body 参数。

2. mode: Literal["fast", "careful"] = "fast" 表达了什么？

字段名为 mode，合法值只能是 fast/careful，未传时默认 fast。

3. 请求体校验失败时业务函数会执行吗？

不会。FastAPI 在进入路径函数前完成校验，失败直接返回 422。

⑨ 面试高频真题+解析

1. Pydantic 在 FastAPI 中承担什么角色？

它承担 API 边界的数据解析、类型转换、约束校验、序列化和 JSON Schema 生成，是接口契约的运行时基础。

2. 为什么 AI 接口更需要严格请求体校验？

AI 调用通常成本高、延迟高、参数多，非法输入若进入 provider 层会浪费资源、放大错误并增加排障难度。

3. 422 是否代表服务器崩溃？

不是。422 表示服务器理解请求格式，但请求数据不满足模型约束，是客户端侧契约错误。
