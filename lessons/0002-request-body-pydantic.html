Lesson 0002 Revised

请求体与 Pydantic 模型：从原理到代码

本节先讲清楚 FastAPI 为什么需要 Pydantic、Pydantic 模型如何工作，再回到 POST /predict 编程实践。目标不是记住几行代码，而是理解“接口契约”这个后端核心概念。

0. 本节学习目标

  • 理解什么是 HTTP 请求体，以及为什么 AI 服务通常使用 POST 接收 JSON。
  • 理解 Pydantic 模型不是普通类，而是“数据结构 + 类型约束 + 校验规则 + 文档 schema”。
  • 理解 PredictionRequest 里 text、mode、temperature 三个字段分别承担什么职责。
  • 能读懂 422 Unprocessable Entity，知道它不是程序崩了，而是请求数据没有通过接口契约。

1. 背景知识：什么是请求体

一个 HTTP 请求可以粗略分成三部分：请求方法、路径、请求体。

POST /predict
Content-Type: application/json

{
  "text": "FastAPI is pleasant",
  "mode": "careful",
  "temperature": 0.2
}

GET /health 适合“向服务器要信息”。但 POST /predict 更像“把一份材料交给服务器处理”。AI 服务正是这种形态：客户端提交文本、图片 URL、模型参数，后端调用模型或规则函数，然后返回结果。

官方依据：FastAPI 官方文档说明，请求体是客户端发送给 API 的数据；声明请求体时使用 Pydantic 模型；发送数据通常使用 POST、PUT、PATCH 等方法。

2. 核心概念：Pydantic 模型是什么

Pydantic 官方把模型描述为继承 BaseModel 的类，字段通过 Python 类型注解定义。你可以先把它理解成后端接口的“数据协议说明书”。

class PredictionRequest(BaseModel):
    text: str = Field(min_length=3, max_length=500)
    mode: Literal["fast", "careful"] = "fast"
    temperature: float = Field(default=0.2, ge=0, le=1)

这段代码同时表达了四层含义：

  1. 结构：请求 JSON 应该有哪些字段。
  2. 类型：每个字段应该是什么 Python 类型，如 str、float。
  3. 约束：字段还要满足额外规则，如长度范围、数值范围、固定枚举值。
  4. 文档：FastAPI 会把这些信息转成 OpenAPI schema，然后显示在 /docs。

所以，Pydantic 模型不只是“装数据的类”。它站在 API 边界上，负责把外部传来的不可信 JSON 转换成后端可以放心使用的 Python 对象。

3. 原理流程：一次 POST 请求发生了什么

当客户端请求 POST /predict 时，FastAPI 和 Pydantic 大致按这个顺序工作：

浏览器 / 前端 / HTTP 客户端
        |
        | 发送 JSON 请求体
        v
FastAPI 根据 @app.post("/predict") 找到 predict 函数
        |
        | 看到参数 payload: PredictionRequest
        v
Pydantic 读取 PredictionRequest 的字段定义
        |
        | 校验 text / mode / temperature
        v
校验成功：构造 payload 对象，进入函数
校验失败：直接返回 422，函数不会执行

这就是为什么你在函数里可以直接写 payload.text、payload.mode：因为函数执行时，payload 已经不是原始 JSON 字符串，而是通过校验后的 PredictionRequest 对象。

4. 字段逐行讲解

4.1 text：核心输入

text: str = Field(
    min_length=3,
    max_length=500,
    examples=["FastAPI makes backend APIs pleasant."],
)

text 是这个预测接口的主输入。AI 场景里，它可以是一段用户评论、一段提示词、一段待分类文本。

  • str 表示它应该是字符串。
  • min_length=3 表示太短的文本没有处理价值。
  • max_length=500 表示暂时限制输入规模，避免接口被过长文本拖慢。
  • examples 会影响文档展示，让调用者知道应该传什么。

4.2 mode：推理模式

mode: Literal["fast", "careful"] = "fast"

这里的 mode 不是 FastAPI 内置参数，是我们自己设计的业务字段。它表示“这次预测希望用什么处理策略”。

  • "fast"：默认模式，表示快速返回。真实 AI 项目里可以对应小模型、短超时、低成本推理。
  • "careful"：谨慎模式，表示愿意多花一点计算换更稳的结果。真实 AI 项目里可以对应大模型、多轮检查、更长超时。
  • = "fast"：这是默认值，不是第三个可选模式。调用者不传 mode 时，Pydantic 会把它补成 "fast"。

Literal["fast", "careful"] 的作用是枚举约束：请求里只能传这两个字符串。如果传 "slow"、"accurate"、1，都会被判定为非法请求。

4.3 temperature：AI 参数的雏形

temperature: float = Field(default=0.2, ge=0, le=1)

temperature 是许多生成式 AI 接口常见的参数。现在我们的 demo 还没有真的调用大模型，但提前学习这个字段很有价值。

  • float 表示它是浮点数。
  • default=0.2 表示调用者不传时默认比较稳定。
  • ge=0 表示 greater than or equal，必须大于等于 0。
  • le=1 表示 less than or equal，必须小于等于 1。

真实大模型里，temperature 往往控制输出随机性。值越低，输出越稳定；值越高，输出越发散。不同模型服务的具体范围可能不同，所以以后要以对应模型文档为准。

5. 对照代码：路径函数如何拿到模型对象

@app.post("/predict", response_model=PredictionResponse)
def predict(payload: PredictionRequest) -> PredictionResponse:
    positive_words = {"good", "great", "love", "pleasant", "nice", "happy"}
    tokens = {word.strip(".,!?;:").lower() for word in payload.text.split()}
    is_positive = bool(tokens & positive_words)
    careful_bonus = 0.04 if payload.mode == "careful" else 0.0

    score = (0.91 if is_positive else 0.55) + careful_bonus

    return PredictionResponse(
        label="positive" if is_positive else "neutral",
        score=round(score, 2),
        source="rule-based-demo",
        text_length=len(payload.text),
    )

关键点是函数签名里的 payload: PredictionRequest。FastAPI 看到这个类型后，会把请求体交给 Pydantic 处理。处理成功后，函数里拿到的就是一个对象，而不是字典。

payload.mode 当前只影响 careful_bonus：如果是 "careful"，分数加 0.04。这只是教学 demo，用来说明“请求参数可以改变后端处理策略”。真实 AI 项目里，mode 可能会决定模型版本、检索深度、最大 token 数、是否启用重排或复核。

6. 实验一：合法请求

打开 http://127.0.0.1:8000/docs，找到 POST /predict，填入：

{
  "text": "FastAPI is pleasant",
  "mode": "careful",
  "temperature": 0.2
}

你应该看到类似结果：

{
  "label": "positive",
  "score": 0.95,
  "source": "rule-based-demo",
  "text_length": 19
}

解释：pleasant 命中了正向词，所以基础分是 0.91；mode 是 careful，额外加 0.04；最终四舍五入为 0.95。

7. 实验二：故意触发 422

把请求体改成：

{
  "text": "hi",
  "mode": "slow",
  "temperature": 2
}

它会失败，因为同时违反了三个约束：

  • text 长度只有 2，小于 min_length=3。
  • mode 是 "slow"，不在 "fast" / "careful" 枚举里。
  • temperature 是 2，大于 le=1。

这时 predict 函数不会执行。FastAPI 会把 Pydantic 的校验错误整理成 HTTP 422 响应返回给调用者。

9. 小测：先回忆再展开

1. BaseModel 的核心作用是什么？

它让一个 Python 类变成 Pydantic 模型，从而支持字段定义、类型校验、数据转换、序列化和 JSON Schema 生成。

2. mode: Literal["fast", "careful"] = "fast" 有几层含义？

字段名是 mode；允许值只能是 "fast" 或 "careful"；默认值是 "fast"。

3. 为什么 Pydantic 对 AI 服务尤其重要？

因为 AI 调用通常成本更高、参数更多，提前校验可以避免非法输入进入昂贵或不可控的推理流程。

10. 本节总结

FastAPI 负责 HTTP 路由和文档生成，Pydantic 负责把外部 JSON 变成可信的 Python 对象。对于后端工程师来说，Pydantic 模型就是接口契约；对于 AI 服务工程师来说，它还是推理入口的安全边界。
