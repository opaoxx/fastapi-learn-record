Lesson 0002

用请求体接收 JSON：Pydantic 模型

今天的目标：让 POST /predict 像一个真正的 AI 服务入口。客户端传 JSON，FastAPI 自动校验，函数拿到干净的 Python 对象。

为什么要学请求体

GET /health 适合“拿信息”。但 AI 服务通常需要客户端传入内容，比如一段文本、一个图片 URL、一组推理参数。这些数据一般放在 HTTP 请求体里，并用 POST 发送。

官方依据：FastAPI 官方文档说明，声明请求体时使用 Pydantic 模型；发送数据常用 POST、PUT、PATCH 等方法，而不是普通 GET。

看这段模型

第二课已经把 first_api/main.py 里的 PredictionRequest 升级了：

class PredictionRequest(BaseModel):
    text: str = Field(
        min_length=3,
        max_length=500,
        examples=["FastAPI makes backend APIs pleasant."],
    )
    mode: Literal["fast", "careful"] = "fast"

这里有三个重点：

  • text: str 表示请求 JSON 里必须有 text，而且它必须是字符串。
  • Field(min_length=3, max_length=500) 表示文本长度要在 3 到 500 之间。
  • mode 只能是 fast 或 careful，默认是 fast。

在 /docs 里试

确保服务正在运行：

.\.venv\Scripts\Activate.ps1
fastapi dev first_api/main.py

打开 http://127.0.0.1:8000/docs，找到 POST /predict，点 Try it out，填入：

{
  "text": "FastAPI is pleasant",
  "mode": "careful"
}

你应该会看到类似结果：

{
  "label": "positive",
  "score": 0.95,
  "source": "rule-based-demo",
  "text_length": 21
}

故意制造一个错误

现在把请求体改成这个：

{
  "text": "hi",
  "mode": "slow"
}

你会得到 422 Unprocessable Entity。这不是坏事，它说明 FastAPI 在函数运行前就帮你挡住了脏数据。

普通后端视角

接口边界越清楚，前端和后端越不容易互相猜。

AI 服务视角

推理参数越早校验，越不容易把非法输入送进昂贵的模型调用。

练习：加一个字段

在 PredictionRequest 里新增一个字段：

temperature: float = Field(default=0.2, ge=0, le=1)

然后在 /docs 里分别测试：

  • "temperature": 0.7 应该通过。
  • "temperature": 2 应该返回 422。

完成标准：你能解释为什么一个通过、一个失败。

小测：先回忆再展开

1. FastAPI 怎么知道请求体应该长什么样？

它读取路径函数参数上的 Pydantic 模型类型，例如 payload: PredictionRequest。

2. Field(min_length=3) 的作用是什么？

它给字段增加校验规则，也会进入自动生成的 API 文档。

3. 为什么 422 对后端项目是有价值的？

它表示非法请求在进入业务逻辑前被拦住，接口边界更稳定。
