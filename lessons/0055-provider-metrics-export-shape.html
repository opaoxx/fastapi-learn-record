0055 Provider Metrics Export Shape：从 JSON 快照到 Prometheus 文本形态

本节把 ProviderMetricsCounter.snapshot() 的 JSON 快照渲染为 Prometheus text 风格输出，并通过 GET /provider/metrics/prometheus 暴露为 text/plain。重点不是“拼字符串”，而是理解内部状态、调试快照、机器采集文本三层形态的边界。

① 本节核心知识框架

Metrics Export Shape
├─ 内部事实：ProviderMetricsCounter 按 label set 累计 count
├─ JSON 快照：GET /provider/metrics 返回 labels + count
├─ 文本导出：GET /provider/metrics/prometheus 返回 text/plain
├─ Prometheus text 组成
│  ├─ # HELP：指标说明
│  ├─ # TYPE：指标类型 counter
│  └─ sample：metric_name{labels} count
├─ 格式稳定性
│  ├─ metric name 固定
│  ├─ label 输出排序
│  ├─ label value 转义
│  └─ 末尾换行
└─ 教学边界：text 风格输出不等于生产级 Prometheus pipeline

② 核心概念底层原理

同一份指标事实可以有多种表现形态。Counter 内部保存的是 ProviderMetricLabels → count；JSON 快照适合教学、调试和 API 查看；Prometheus text 风格输出适合展示“采集器会读取什么样的文本”。

JSON snapshot:
[
  {"labels": {"outcome": "success", ...}, "count": 1}
]

Text export:
provider_prediction_total{outcome="success",...} 1

Prometheus text 不是随便的日志文本。它是机器读取的导出格式：指标名、标签、数值、换行、转义规则都属于契约。只要有监控系统、测试快照、文档示例依赖它，格式漂移就可能破坏下游。

③ 全套代码逐行精读解析

PROVIDER_PREDICTION_METRIC_NAME = "provider_prediction_total"
PROVIDER_PREDICTION_METRIC_HELP = (
    "Provider prediction outcomes recorded by the teaching metrics counter."
)

provider_prediction_total 是对外指标名。provider 指 provider adapter 路径，prediction 指预测操作，total 表示累计型 counter。HELP 文本提供人类可读说明。

def escape_prometheus_label_value(value: object) -> str:
    return str(value).replace("\\", "\\\\").replace("\n", "\\n").replace('"', '\\"')
代码片段作用不处理的后果
str(value)统一把 label value 转成字符串非字符串值拼接时出错或输出不一致
replace("\\", "\\\\")转义反斜杠后续转义序列可能被误读
replace("\n", "\\n")把真实换行变成可见字符一条样本被拆成多行，破坏格式
replace('"', '\\"')转义双引号label value 提前闭合，后续文本失效
 
def render_provider_metrics_prometheus_text(
    samples: list[dict[str, object]],
) -> str:
    lines = [
        f"# HELP {PROVIDER_PREDICTION_METRIC_NAME} {PROVIDER_PREDICTION_METRIC_HELP}",
        f"# TYPE {PROVIDER_PREDICTION_METRIC_NAME} counter",
    ]
    for sample in samples:
        labels = sample["labels"]
        if not isinstance(labels, dict):
            raise ValueError("Metric sample labels must be a dictionary.")
        label_text = ",".join(
            f'{name}="{escape_prometheus_label_value(value)}"'
            for name, value in sorted(labels.items())
        )
        lines.append(
            f"{PROVIDER_PREDICTION_METRIC_NAME}{{{label_text}}} {sample['count']}"
        )
    return "\n".join(lines) + "\n"
行精读
lines = [...]无论是否有样本，都先输出 HELP 和 TYPE，保证指标说明存在。
for sample in samples每个 snapshot sample 渲染成一条文本样本行。
isinstance(labels, dict)防御错误输入；文本渲染函数要求 labels 是字典。
sorted(labels.items())稳定 label 顺序，便于测试、diff 和人工阅读。
escape_prometheus_label_value所有 label value 在进入文本前统一转义。
+ "\n"以换行结束，符合行式文本输出习惯。
 
@router.get(
    "/provider/metrics/prometheus",
    response_class=PlainTextResponse,
    summary="Read teaching provider prediction metrics as text",
)
async def get_provider_metrics_prometheus() -> str:
    return get_provider_metrics_prometheus_text()

PlainTextResponse 明确返回文本，不走 JSON response model。router 不拼接文本，只调用 service 层的导出函数，继续保持 HTTP 层与指标语义分离。

④ 核心机制运行全流程拆解

provider event
  → record_provider_metric(labels)
  → ProviderMetricsCounter.increment()
  → get_provider_metrics_snapshot()
  → render_provider_metrics_prometheus_text(samples)
       ├─ 写 HELP
       ├─ 写 TYPE
       ├─ labels 排序
       ├─ label value 转义
       └─ 生成 sample 行
  → GET /provider/metrics/prometheus
  → text/plain 响应

成功样本最终文本形态：

# HELP provider_prediction_total Provider prediction outcomes recorded by the teaching metrics counter.
# TYPE provider_prediction_total counter
provider_prediction_total{error_code="none",failure_category="none",operation="prediction",outcome="success",status_code="none"} 1

这与 JSON endpoint 表达同一份计数，只是面向不同读者：JSON 给人和程序调试，text export 给采集器和监控生态。

⑤ 重难点、易混淆点对比辨析

对象用途典型消费者风险
内部 Counter聚合 label set服务内部函数不应直接暴露
JSON snapshot结构化调试测试、人、前端调试工具不是标准监控采集格式

⑥ 开发常见报错+坑点+解决方案

坑点后果解决方案

⑧ 课后记忆习题+标准答案

1. JSON snapshot 和 text export 的本质区别是什么？

JSON snapshot 是结构化调试形态；text export 是面向采集器或监控生态的行式导出形态。它们表达同一份指标事实，但消费者和格式约束不同。

2. 为什么空 snapshot 也要输出 HELP 和 TYPE？

因为没有样本不代表没有指标定义。HELP 和 TYPE 说明指标名称、含义和类型，使空状态仍然是合法、可解释的导出。

3. label value 为什么必须转义？

双引号、反斜杠、换行会破坏 label 字符串边界或样本行结构。转义后才能保持文本格式可解析。

4. 为什么 PlainTextResponse 必不可少？

它让 FastAPI 返回 text/plain，避免把 Prometheus text 风格内容当 JSON 序列化或让客户端误判响应类型。

⑨ 面试高频真题+解析

面试题 1：如何从内部 metrics snapshot 导出 Prometheus text？

先输出 HELP 和 TYPE，再遍历 snapshot 样本；每条样本按稳定顺序渲染 labels，转义 label value，最后拼成 metric_name{labels} count 行，并以 text/plain 暴露。

面试题 2：为什么导出格式也要测试？

导出文本是机器消费契约。metric name、label 顺序、转义、换行、Content-Type 变化都可能破坏采集或下游测试，因此必须有格式级回归测试。

面试题 3：手写 text export 的教学意义是什么？

它让初学者看清 metrics 从内部 label/count 到外部采集文本的转换过程。生产中通常应使用成熟库，但理解格式能帮助排查导出和采集问题。
