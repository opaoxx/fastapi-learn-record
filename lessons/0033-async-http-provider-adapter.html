异步 HTTP Provider Adapter：用 httpx.AsyncClient 翻译外部失败

真实 AI provider 通常是 HTTP 外部服务。本节用 check_provider_health() 建立 adapter 模板：异步请求 provider，检查状态码，捕获 timeout/status/request 异常，统一映射为项目内部 AIClientError，并用 MockTransport 做无网络测试。

① 本节核心知识框架

Async HTTP Provider Adapter
├─ 输入：url + AIClientConfig + optional AsyncClient
├─ HTTP 客户端：httpx.AsyncClient(timeout, trust_env=False)
├─ 成功路径
│  ├─ await active_client.get(url)
│  ├─ response.raise_for_status()
│  └─ ProviderHealthResult(ok=True, status_code=...)
├─ 失败映射
│  ├─ TimeoutException -> ai_provider_timeout
│  ├─ HTTPStatusError -> ai_provider_http_error
│  └─ RequestError -> ai_provider_request_error
├─ 生命周期：owns_client 决定是否 aclose
└─ 测试：httpx.MockTransport 模拟 200/timeout/503

② 核心概念底层原理

Adapter 是外部协议到内部协议的翻译器。HTTPX 抛出的 ReadTimeout、ConnectError、HTTPStatusError 不应该扩散到 router；项目内部应该只依赖稳定 AIClientError(error_code, message)。

httpx.AsyncClient 适合在 async FastAPI 路由和 service 中等待外部 HTTP。网络等待期间，协程可以让出执行权，避免把请求线程长期占住。它仍需 timeout，否则外部 provider 卡住会拖住业务请求。

测试外部依赖的目标不是测试互联网是否可用，而是测试我们的错误映射是否正确。MockTransport 允许测试精确模拟 200、503、timeout，不依赖真实网络、费用、限流或 provider 状态。

③ 全套代码逐行精读解析

@dataclass(frozen=True)
class ProviderHealthResult:
    ok: bool
    provider: str
    status_code: int
    message: str

这是内部 health 检查结果，不是 provider 原始响应。ok 表达业务判断，provider 标记来源，status_code 保留 HTTP 信号，message 给后续响应或日志使用。

async def check_provider_health(
    url: str,
    config: AIClientConfig,
    client: httpx.AsyncClient | None = None,
) -> ProviderHealthResult:
    owns_client = client is None
    active_client = client or httpx.AsyncClient(
        timeout=config.timeout_seconds,
        trust_env=False,
    )
  • url 是服务端配置的 provider health 地址，不应来自用户任意输入。
  • config 提供 provider 名称和 timeout 策略。
  • client 可选，方便测试注入 MockTransport client，也方便未来复用 lifespan client。
  • owns_client 标记资源所有权：函数自己创建才自己关闭。
  • trust_env=False 避免本机代理环境干扰教学和测试。
try:
    response = await active_client.get(url)
    response.raise_for_status()
  • await 表示网络等待期间让出执行权。
  • raise_for_status() 把 4xx/5xx 转成异常；不调用会把 503 误当普通响应。
except httpx.TimeoutException as exc:
    raise AIClientError(
        message="The AI provider health check timed out.",
        error_code="ai_provider_timeout",
    ) from exc
except httpx.HTTPStatusError as exc:
    raise AIClientError(
        message=f"The AI provider returned HTTP {exc.response.status_code}.",
        error_code="ai_provider_http_error",
    ) from exc
except httpx.RequestError as exc:
    raise AIClientError(
        message="The AI provider could not be reached.",
        error_code="ai_provider_request_error",
    ) from exc

异常顺序从具体到通用：timeout 是特殊请求错误，HTTPStatusError 表示 provider 已响应但状态失败，RequestError 兜底连接/DNS/协议层问题。from exc 保留服务端调试链路。

finally:
    if owns_client:
        await active_client.aclose()

资源生命周期规则：谁创建，谁关闭。外部传入的 client 可能属于 lifespan 或测试 fixture，函数不能擅自关闭。

④ 核心机制运行全流程拆解

check_provider_health(url, config, client)
  -> 如果未传 client：创建 AsyncClient(timeout, trust_env=False)
  -> await GET provider health URL
  -> raise_for_status()
       2xx：构造 ProviderHealthResult
       timeout：AIClientError(ai_provider_timeout)
       4xx/5xx：AIClientError(ai_provider_http_error)
       connect/DNS/protocol：AIClientError(ai_provider_request_error)
  -> 如果函数拥有 client：aclose()
  -> 调用方只看到内部结果或内部错误

⑤ 重难点、易混淆点对比辨析

概念正确理解易错点
Adapter外部失败到内部错误的翻译层只当 HTTP helper
raise_for_status显式触发 4xx/5xx 错误路径忘记调用导致 503 被当成功

RequestError请求层/连接层失败直接暴露给前端

⑥ 开发常见报错+坑点+解决方案

  • 503 没触发错误处理：忘记 response.raise_for_status()。解决：状态码检查放在 adapter 内。
  • 测试偶发失败：真实访问外部 URL。解决：用 MockTransport。
  • 连接资源泄漏：函数创建 client 但不关闭。解决：finally + aclose()。
  • 关闭了不该关的 client：外部传入 client 被函数关闭。解决：用 owns_client 区分所有权。
  • 本地代理导致测试失败：HTTPX 读取环境代理。解决：教学项目设置 trust_env=False。

⑧ 课后记忆习题+标准答案

1. 为什么 adapter 不应把 HTTPX 异常直接漏给 router？

router 应依赖项目内部错误契约；HTTPX 异常是外部库细节，扩散后会降低可替换性。

2. raise_for_status() 的作用是什么？

把 4xx/5xx 响应转换成 HTTPStatusError，让失败路径显式进入异常映射。

3. owns_client 体现什么原则？

谁创建资源，谁关闭资源；函数不能关闭外部传入并由别人拥有的 client。

4. 为什么测试用 MockTransport？

它能稳定模拟外部 HTTP 成功和失败，不依赖真实网络、费用或 provider 状态。

⑨ 面试高频真题+解析

题 1：如何封装外部 HTTP provider 调用？

放到 adapter/service 中，使用 async client、timeout、状态码检查和异常映射；对外返回内部结果或内部业务异常，router 只做 HTTP 契约转换。

题 2：为什么不在单测里访问真实 provider？

真实 provider 慢、贵、不稳定且不可控。单测应模拟响应验证自己的映射逻辑，少量集成测试再访问真实环境。

题 3：timeout、HTTP status error、request error 如何区分？

timeout 是等待超时；HTTP status error 是收到响应但状态码失败；request error 是连接、DNS、协议等请求层失败。三者排障和重试策略不同。
