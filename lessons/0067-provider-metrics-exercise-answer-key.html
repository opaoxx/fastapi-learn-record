0067 Provider Metrics Exercise Answer Key：让排障训练具备标准答案

本节新增 render_provider_metrics_runbook_exercise_answer_key()，把上一节的题卡渲染成 Markdown 答案键。它让 runbook 训练从“凭感觉讨论”变成“有题目、有评分锚点、有标准答案”的工程化学习流程。

① 本节核心知识框架

Exercise Answer Key
├─ 输入：exercise cards
├─ 输出：Markdown answer key
│  ├─ card id
│  ├─ prompt
│  ├─ expected severity
│  └─ answer
├─ 价值
│  ├─ 统一排障术语
│  ├─ 支撑自测和复盘
│  ├─ 便于 GitHub 离线阅读
│  └─ 可被测试保护
└─ 边界
   ├─ 不评分用户输入
   ├─ 不调用 provider
   ├─ 不读取 counter
   └─ 不替代生产演练平台

② 核心概念底层原理

标准答案是学习闭环的最后一环。没有答案键，题卡只能制造讨论；有答案键，学习者才能对照自己是否把 outcome、severity、finding 和 next_action 串成正确判断链。

本节的答案键仍然是纯渲染函数。它不做自然语言评分，不判断用户输入，只把结构化题卡转换成可读 Markdown。这种边界能避免课程过早进入复杂评分系统。

答案键必须从 cards 渲染，而不是重新读取 findings。原因是 cards 已经决定了题干、评分锚点和标准答案；答案键只负责把同一份训练数据转成复盘文本。若答案键绕过 cards 重新组装，会产生“题目一套、答案一套”的维护风险。

阶段产物作用
观测samples提供指标事实
解释findings给出判断和动作
训练exercise cards提出问题
复盘answer key统一标准答案

复核补强：答案键不是“附录”，而是训练闭环的判定基准。没有答案键，题卡只能激发讨论；有答案键，学习者才能把自己的自由表达映射回统一的 severity、finding 和 next_action。

答案键内容为什么必须存在缺失后果
card['id']定位题目，支持引用和复盘多题材料中无法快速对应错误题。
card['prompt']让答案键自包含读者必须跳回题卡页面才能理解答案。
expected_severity显式暴露优先级判断复盘只看动作，会漏掉事故响应优先级训练。
answer给出完整标准表达学习者不知道应如何组织“原因 + 动作”。

    cards: list[dict[str, object]],

    for card in cards:

函数只接收题卡，不接收 samples 或 findings，说明它位于链路末端。这样每层输入输出清晰，便于测试定位。

lines.append(f"## {card['id']}")
lines.append("")
lines.append(f"Prompt: {card['prompt']}")
lines.append(f"Expected severity: {card['expected_severity']}")
lines.append(f"Answer: {card['answer']}")
lines.append("")

每张题卡渲染为一个二级标题块。空行用于保证 Markdown 段落清晰。答案键保留 prompt，便于复盘时不用跳回题目源数据。

return "\n".join(lines)

输出是单个 Markdown 文本块，可直接写入学习记录、issue 或课程参考页。测试会保护其中关键字符串。

逐行精读补充：函数返回值没有额外拼接 + "\n"，因为每张 card 的末尾已经追加空行；非空输入会自然以换行结束，空输入则返回空字符串。这个差异体现了“空题卡不伪造标题”的设计：调用方决定是否展示空状态，而 renderer 不替业务层编造内容。

本函数用 card['...'] 读取字段，与前两节保持一致：题卡结构坏了就应立即失败。教学项目通过异常暴露契约破坏；生产平台则应在题卡入库或发布前做 schema 校验，把错误报告给内容维护者。

代码点作用不这样写的隐患
lines = []允许按块追加 Markdown 行。若在循环中直接拼接大字符串，后续插入字段时可读性更差。
for card in cards按题卡顺序输出，保证题目与答案顺序一致。若重新排序，学生对照题卡时会产生定位成本。
## {card['id']}把每张题卡变成可跳转的 Markdown 小节。缺少标题会让多题答案键难以扫描和引用。
Prompt:保留题干上下文，让答案键自包含。只给答案会迫使读者回查题卡，复盘效率低。
Expected severity:显式暴露优先级判断标准。只看 next_action 会忽略故障处理优先级。
lines.append("")每题末尾留空行，非空 cards 会自然得到末尾换行。缺少空行时多个答案块会粘连，Markdown 可读性下降。

④ 核心机制运行全流程拆解

samples
  → findings
  → exercise cards
  → render_provider_metrics_runbook_exercise_answer_key(cards)
  → Markdown answer key
  → 学生自测 / 课程复盘 / PR 说明

这条链路形成完整训练闭环：先给观测事实，再给题目，再给标准答案。每一步都是纯函数，测试稳定，适合 GitHub 离线课程。

边界条件也要明确：当 cards 为空时，函数返回空字符串；当 card 缺少 id、prompt、expected_severity 或 answer 时，会抛出 KeyError。教学代码选择暴露契约错误，生产环境可在入口层加 schema 校验并返回可诊断错误。

⑧ 课后记忆习题+标准答案

1. 为什么答案键要保留 Prompt？

答案键需要自包含，复盘时不能要求读者再回到题卡源数据寻找题目。

2. 为什么答案键不负责评分用户输入？

评分涉及模糊匹配和质量判断，会显著增加复杂度。本节只做稳定答案渲染。

3. 为什么答案键从 cards 渲染？

这样可以保证题目、severity 和答案来自同一结构化来源，减少手写漂移。

4. 答案键在面试训练中有什么价值？

它给出标准排障表达，帮助学习者把观察、判断、下一步动作组织成完整回答。

5. 空 cards 输入应该返回什么？

返回空字符串。渲染函数不虚构标题或提示文案，是否展示空状态应由页面或调用方决定。

⑨ 面试高频真题+解析

面试题 1：如何构建一套可维护的故障排查训练材料？

用真实契约生成样本，用 findings 解释样本，用题卡训练判断，用答案键统一复盘。每层都是结构化数据和纯函数，配套测试防止漂移。

面试题 2：答案键为什么不能手写散落在文档里？

手写答案容易与题卡和代码契约漂移。自动从 cards 渲染能保证题目、severity、标准答案一致。

面试题 3：教学答案键和生产演练平台有什么区别？

教学答案键强调标准认知路径；生产演练平台还要记录参与人、时间、响应质量、审批、脱敏和复盘更新。
