② 核心概念底层原理

“课程完结”不是最后一个 HTML 文件存在。对 GitHub 公开课程来说，完结意味着内容、代码、测试、目录、速查、治理文档和发布证据共同形成一个可复查的学习产品。

第 96 节的底层思想是闭环控制。前 95 节已经建立了课程内容、项目代码、测试、GitHub 协作、审计和 RC 流程；本节把这些流程按最终发布顺序串起来，避免最后一步只靠人工记忆。

.github/COURSE_COMPLETION.md 与前几个文档分工明确：FINAL_AUDIT 判断总体质量，LINK_AUDIT 判断静态链接，RELEASE_CANDIDATE 判断候选发布，RELEASE_PROCESS 指导 tag 和 Release，COURSE_COMPLETION 负责最终收官顺序。

# Course Completion Release Runbook

标题定义最终职责：课程完结发布运行手册。Runbook 表示它不是理念说明，而是可按步骤执行的操作文档。

This runbook defines the final completion procedure for the 96-lesson FastAPI course.

这行绑定最终范围：96 节 FastAPI 课程。若不写范围，后续扩展课程可能误用该文档宣布新阶段已完成。

It is used after Lesson 0096 exists and before maintainers announce the course as complete.

这行定义使用时机：第 96 节已经存在，但尚未宣布完成。它是宣布前的最后门禁，不是课程开发中的普通清单。

## Completion Meaning

Completion Meaning 明确定义“完结”含义。没有定义，完结就会退化为主观感觉。

- lessons 0001-0096 exist with no numbering gaps;

编号无缺口是完成的第一事实。测试中新增的编号连续性检查会直接保护这一条。

- each protected lesson keeps the required nine-section review structure;

九段式结构是用户最初定下的质量标准。完结时不能只追求数量，必须保持期末复习笔记和高质量技术博文风格。

- each included lesson has a matching quick reference;

速查表是复习闭环。完整课程不仅能精读，还应能考前快速回忆。

- no real secret, private token, production provider URL, private log, customer data, or personal data is committed;

公开 GitHub 仓库必须把无密钥边界写进完成定义。教学材料传播范围越大，泄漏成本越高。

## Final Run Order

Final Run Order 把多个文档排序。排序很重要：先确认完整，再审计，再进入 RC，再按发布流程执行。

1. Confirm the lesson range is complete.

第一步检查 0001-0096 是否齐全。若范围不完整，后面的 Release、Pages 和公告都没有意义。

2. Review `.github/FINAL_AUDIT.md`.

总审计负责仓库级质量，不只看最后一课，也看课程、源码、测试、治理文档的一致性。

3. Review `.github/LINK_AUDIT.md`.

链接审计确保离线 HTML 和 Pages 场景下，目录、课程和速查能互相抵达。

4. Review `.github/RELEASE_CANDIDATE.md`.

RC 文档负责冻结范围、记录验证证据和确认候选版本是否能进入 tag 阶段。

7. Run the required verification commands.

最终发布不能跳过命令证据。人工审阅能发现语义问题，测试能发现结构、链接和行为回归。

## Release Announcement Draft

发布公告不是营销文案，而是用户理解课程范围和质量证据的入口。它必须写明完成范围、学习路径、验证结果、已知警告和阅读方式。

## Do Not Announce Complete If

该节定义阻断条件。只要缺课、缺速查、断链、测试失败、泄漏 secret 或已知警告不清楚，就不能宣布完成。

## After Completion

完结后治理防止“静默重写已完成版本”。公开学习产品应保留基线，修复通过 patch release 或新 tag 透明发布。

⑧ 课后记忆习题+标准答案

1. 为什么课程完结不等于最后一个 HTML 文件存在？

因为公开课程还需要速查、目录导航、九段式结构、源码一致性、测试证据、无密钥检查和发布文档一致，只有这些共同成立才是学习产品完结。

2. COURSE_COMPLETION 与 RELEASE_PROCESS 的区别是什么？

COURSE_COMPLETION 定义宣布 96 节课程完成前的最终顺序；RELEASE_PROCESS 定义创建 annotated tag、GitHub Release Notes 和 rollback 的发布动作。

3. 完结前必须运行哪些命令？

必须运行文档契约测试、provider HTTP 测试、全量 pytest 和 git diff --check，并记录结果。

4. 为什么要保护 lessons 0001-0096 无缺号？

编号是学习路径、目录、速查、发布范围和用户预期的公共索引。缺号会让课程看起来不完整，也会破坏导航和公告可信度。

5. 完结后发现错误应该怎么处理？

不要静默重写已完成版本，应通过新 tag、patch release 或明确的 follow-up 修复透明处理。

⑨ 面试高频真题+解析

面试题 1：如何判断一个技术课程仓库可以宣布完成？

要同时检查范围完整、内容结构、导航链路、源码一致性、测试结果、发布文档、许可证、贡献流程、安全边界和无密钥状态，而不是只看文件数量。

面试题 2：为什么要把完成流程写成 runbook？

Runbook 把关键步骤顺序化、显式化、可复查化，避免依赖个人记忆；它也让后续维护者能复现同样的完成判断。

面试题 3：发布完成后为什么不能静默改写 tag？

tag 是版本承诺。静默改写会破坏读者、CI、Release Notes 和下游引用的一致性，应通过新版本透明修复。

面试题 4：文档型项目的质量门禁与服务端项目有何不同？

文档型项目更重视结构、链接、导航、术语一致性和阅读入口；服务端项目更重视运行时行为、部署、数据和 SLA。课程仓库两者都要覆盖。

面试题 5：如何处理完结版本中的已知 warning？

先判断是否阻断发布；非阻断 warning 要记录来源、影响和后续处理计划。测试失败、断链、缺课和 secret 泄漏不能作为 known warning 放行。
