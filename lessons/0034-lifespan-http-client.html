Lesson 0034

FastAPI lifespan：给 AsyncClient 一个清楚的生命周期

第 33 课我们写了 async HTTP provider adapter。现在补上生产项目里非常关键的一步：HTTP client 不应该在每次请求里随手创建和泄漏，而应该在应用启动时创建，在应用关闭时清理。

0. 本节学习目标

  • 理解 FastAPI lifespan 的启动阶段、运行阶段和关闭阶段。
  • 理解为什么 httpx.AsyncClient 适合跨请求复用。
  • 学会用 app.state 保存应用级资源。
  • 理解谁创建资源、谁关闭资源的生命周期原则。
  • 知道本地代理环境可能影响 HTTPX client 创建，并学会用 trust_env=False 保持教学项目稳定。

1. 背景：每次请求都创建 client 为什么不好

上一节的 check_provider_health() 可以自己创建 httpx.AsyncClient，这是为了让函数独立可用。但真实服务里，如果每个请求都新建 HTTP client，会有几个问题：

  • 连接池无法复用，每次都可能重新握手。
  • 资源生命周期不清楚，容易忘记关闭。
  • 配置散落在多个地方，timeout、proxy、headers 难以统一。
  • 测试和生产行为更难对齐。

HTTPX 官方 client 文档强调，client 实例会使用连接池，同一个 host 的多次请求可以复用底层 TCP 连接。对长期运行的 FastAPI 应用来说，这就是把 HTTP client 当作应用级资源管理的理由。

2. 核心概念：lifespan 是应用资源的生命周期边界

FastAPI 的 lifespan 是一个 async context manager：

@asynccontextmanager
async def lifespan(app: FastAPI):
    # startup: 创建资源
    yield
    # shutdown: 清理资源

yield 前的代码在应用启动时运行，yield 后的代码在应用关闭时运行。它非常适合管理：

  • 数据库连接池。
  • HTTP client。
  • 模型加载结果。
  • 缓存客户端。
  • 消息队列连接。

3. 本节项目改动概览

  • 在 main.py 的 lifespan 中创建应用级 httpx.AsyncClient。
  • 把 provider、timeout 和 client 对象保存到 app.state。
  • 在 shutdown 阶段调用 aclose()。
  • 新增 GET /provider/http-client 教学端点，观察 client 是否已由 lifespan 创建。
  • 新增测试确认 TestClient 启动 lifespan 后能看到 client ready。
  • 把 provider HTTP adapter 中临时创建的 client 也设置为 trust_env=False，避免本机代理环境影响教学。

4. 代码精读一：lifespan 创建资源

@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncGenerator[None, None]:
    create_db_and_tables()
    ai_client_config = build_ai_client_config()
    app.state.provider_http_client = httpx.AsyncClient(
        timeout=ai_client_config.timeout_seconds,
        trust_env=False,
    )
    app.state.provider_http_client_provider = ai_client_config.provider
    app.state.provider_http_client_timeout_seconds = ai_client_config.timeout_seconds
    try:
        yield
    finally:
        await app.state.provider_http_client.aclose()

这段代码把应用启动分成三部分：

  • 启动前准备：建表、读取 AI provider 配置、创建 AsyncClient。
  • 运行中：yield 期间应用开始接收请求。
  • 关闭清理：应用退出时关闭 AsyncClient。

try/finally 的意义是：只要应用进入运行阶段，退出时就尽量执行清理逻辑。

5. 代码精读二：app.state 保存应用级状态

app.state.provider_http_client = httpx.AsyncClient(...)
app.state.provider_http_client_provider = ai_client_config.provider
app.state.provider_http_client_timeout_seconds = ai_client_config.timeout_seconds

Starlette/FastAPI 应用对象有 app.state，可以存放应用级资源。只要 endpoint 能拿到 Request，就能通过 request.app.state 访问它。

这适合存放“应用启动一次、运行期间复用、应用关闭时清理”的资源。它不适合存放每个用户、每个请求独有的数据。

6. 代码精读三：教学观察端点

@router.get("/provider/http-client", response_model=ProviderHttpClientStateResponse)
def read_provider_http_client_state(request: Request) -> ProviderHttpClientStateResponse:
    provider_http_client = request.app.state.provider_http_client
    return ProviderHttpClientStateResponse(
        provider=request.app.state.provider_http_client_provider,
        timeout_seconds=request.app.state.provider_http_client_timeout_seconds,
        client_ready=not provider_http_client.is_closed,
    )

这个端点不是业务功能，而是教学探针。它告诉我们：

  • lifespan 已经创建了 provider HTTP client。
  • 当前 provider 是什么。
  • timeout 配置是多少。
  • client 还没有关闭。

在真实项目中，你不一定要暴露这个端点；但学习时它能让生命周期从抽象概念变成可观察结果。

7. 代码精读四：trust_env=False 这个小坑

httpx.AsyncClient(
    timeout=ai_client_config.timeout_seconds,
    trust_env=False,
)

这节实现时测试曾经失败过：本机环境变量里存在 SOCKS 代理地址，HTTPX 默认会读取环境代理设置，但项目环境没有安装 SOCKS 支持包，于是 client 创建阶段就报错。

对一个教学项目来说，我们希望测试不被每个人的本机代理配置影响，所以这里设置 trust_env=False。真实生产项目是否信任环境代理，要根据部署方式决定；但你必须知道它会影响 HTTP client 的行为。

8. 生命周期规则：谁创建，谁关闭

第 33 课的 check_provider_health() 里有一个规则：

owns_client = client is None

如果函数自己创建 client，它在 finally 里关闭；如果 client 是外部传入的，函数不关闭。

第 34 课则把长期 client 的创建权交给 lifespan，所以关闭也由 lifespan 负责。这两个设计没有冲突，它们表达同一个原则：谁拥有资源生命周期，谁负责清理。

9. 测试精读

def test_provider_http_client_is_created_by_lifespan() -> None:
    with TestClient(app) as client:
        response = client.get("/provider/http-client")

    assert response.status_code == 200
    assert response.json() == {
        "provider": "demo",
        "timeout_seconds": 10.0,
        "client_ready": True,
    }

with TestClient(app) 很重要。进入 context 时，TestClient 会触发应用 lifespan startup；退出 context 时，它会触发 shutdown。也就是说，测试不是只测 endpoint，还间接验证了生命周期能启动起来。

10. 课堂实验

运行服务后打开：

http://127.0.0.1:8000/provider/http-client

你应该看到：

{
  "provider": "demo",
  "timeout_seconds": 10.0,
  "client_ready": true
}

这个结果说明应用启动阶段已经创建好了 provider HTTP client。

11. 常见错误与诊断

错误一：每个请求都新建 AsyncClient

连接池无法复用，资源生命周期容易混乱。

错误二：创建了 client 但不关闭

长期运行可能泄漏连接资源。应用关闭时应该 aclose()。

错误三：把请求级数据放进 app.state

app.state 是应用级状态，不适合保存用户请求独有数据。

错误四：忽略环境代理影响

HTTPX 默认可能读取环境代理；教学项目可用 trust_env=False 保持稳定。

12. 面试视角：如何回答“HTTP client 怎么管理生命周期？”

我会把长期复用的 AsyncClient 放到 FastAPI lifespan 中管理。应用启动时读取配置并创建 client，存到 app.state；请求处理时通过 request.app.state 或依赖函数获取；应用关闭时统一 aclose。这样可以复用连接池，集中配置 timeout/proxy/headers，并避免每次请求重复创建和泄漏资源。

13. 小测：先回忆，再展开

1. lifespan 中 yield 前后的代码分别什么时候执行？

yield 前在应用启动时执行，yield 后在应用关闭时执行。

2. 为什么长期复用 AsyncClient 有价值？

它能复用连接池，减少重复握手，并集中管理配置和资源关闭。

3. app.state 适合存什么？

适合存应用级、跨请求复用、需要统一生命周期管理的资源。

4. trust_env=False 在本节避免了什么问题？

避免本机环境代理设置影响 HTTPX client 创建和测试稳定性。

14. 本节总结

本节把 async HTTP provider adapter 推向更真实的应用生命周期：长期复用的 AsyncClient 应该在应用启动时创建，在应用关闭时清理。FastAPI lifespan 提供了这个边界，app.state 提供了资源存放位置。

下一课建议学习：把 lifespan-managed AsyncClient 接入 provider adapter。我们可以把 check_provider_health() 从“可传入 client”进一步连接到 FastAPI 依赖，让业务路由真正使用应用级 HTTP client。
