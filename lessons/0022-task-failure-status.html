任务失败状态：异步 API 的状态机、错误落库与最终一致性

同步预测失败可以直接返回 503；后台摘要任务不行。POST /tasks/summaries 返回 202 后，HTTP 响应已经结束，最终成功或失败必须写进任务记录。本节核心是异步任务 API 的状态机设计。

① 本节核心知识框架

异步任务失败建模
├─ HTTP 语义
│  ├─ POST /tasks/summaries -> 202 Accepted
│  └─ GET /tasks/{id} -> 当前任务状态
├─ 状态机
│  ├─ queued：已入库，等待执行
│  ├─ running：后台 worker 已开始
│  ├─ completed：成功，读取 result
│  └─ failed：失败，读取 error
├─ 数据字段
│  ├─ status：阶段
│  ├─ result：成功输出
│  └─ error：失败说明
├─ 错误分类
│  ├─ AIClientError：稳定错误码
│  └─ Exception：通用 unexpected_task_error
└─ 原则：202 是接收证明，不是成功证明

② 核心概念底层原理

202 Accepted 表示请求已被接受处理，但处理尚未完成。对于后台任务，客户端收到 202 时只能确定任务已排队，不能确定摘要最终成功。最终结果需要通过任务查询接口读取，这就是“提交接口”和“查询接口”分离。

异步任务的状态字段不是普通字符串，而是小型状态机。状态机的价值在于限定合法阶段和前端行为：queued/running 显示等待，completed 展示 result，failed 展示 error。如果没有状态机，前端只能猜测空结果到底是未完成、失败还是没有输出。

与同步错误对比：同步 /predict 可在同一次 HTTP 响应中返回 503；异步摘要任务的失败发生在响应之后，不能回头修改已发送的 202，只能更新数据库中的任务记录。

③ 全套代码逐行精读解析

def process_summary_task(task_id: int) -> None:
    with Session(engine) as session:
        run_summary_task(
            session=session,
            task_id=task_id,
            ai_client=get_ai_client(),
        )
  • process_summary_task(task_id) 是后台任务入口，只接收简单可序列化的任务 ID；不要传复杂 Session 或 ORM 对象。
  • with Session(engine) 为后台执行创建独立数据库会话；请求级 Session 可能已关闭，复用会造成生命周期错误。
  • run_summary_task(...) 把真正业务规则交给 service，worker 只负责组装依赖。
  • get_ai_client() 在普通函数中显式调用，因为 Depends 只在 FastAPI 路由依赖解析中自动执行。
def run_summary_task(session: Session, task_id: int, ai_client: AIClient) -> None:
    task = session.get(SummaryTask, task_id)
    if task is None:
        return
  • session.get(SummaryTask, task_id) 按主键读取任务；比手写 select 更直接。
  • if task is None: return 处理任务不存在的边界；后台任务没有客户端等待，不能抛 HTTP 404。
try:
    task.status = "running"
    session.add(task)
    session.commit()
    session.refresh(task)

    task.result = ai_client.summarize(task.text)
    task.status = "completed"
    task.error = None
  • status="running" 让查询接口能看到任务已开始；不提交则外部仍看到 queued。
  • session.add/commit/refresh 把 running 状态落库并同步对象；如果进程随后崩溃，至少能知道任务进入过执行阶段。
  • ai_client.summarize(task.text) 是可能失败的外部能力边界，因此必须在 try 块内。
  • task.result = ... 只在成功后写入；失败任务不应伪造 result。
  • task.status = "completed" 明确最终成功状态。
  • task.error = None 清理旧错误，避免重跑任务后 completed 仍带错误信息。
except AIClientError as exc:
    task.status = "failed"
    task.error = f"{exc.error_code}: {exc.message}"
except Exception:
    task.status = "failed"
    task.error = "unexpected_task_error: The summary task failed unexpectedly."
finally:
    session.add(task)
    session.commit()
  • except AIClientError 捕获已知 AI 失败，保留稳定错误码，方便前端和测试判断。
  • except Exception 捕获未知异常，但只暴露通用错误，避免泄露栈信息。
  • finally 确保无论成功或失败都提交最终状态；缺少它会让任务长期停在 running。
  • session.commit() 是任务状态可查询的关键；没有落库，客户端无法观察最终结果。

④ 核心机制运行全流程拆解

客户端提交摘要任务
  -> POST /tasks/summaries
  -> 创建 SummaryTask(status="queued")
  -> 注册 process_summary_task(task.id)
  -> 返回 202 + task id

后台响应后执行
  -> process_summary_task(task.id)
  -> 新建 Session
  -> run_summary_task()
       -> 读取任务
       -> status = running，提交
       -> ai_client.summarize(text)
       -> 成功：result + completed + error=None
       -> 已知失败：failed + error_code/message
       -> 未知失败：failed + unexpected_task_error
       -> finally 提交最终状态

客户端轮询或查询
  -> GET /tasks/{id}
  -> 根据 status 决定展示 loading/result/error

⑤ 重难点、易混淆点对比辨析

概念正确理解易错点

failed任务最终失败状态用 HTTP 503 修改已结束响应
result成功输出失败时仍保留旧 result
error失败说明成功时不清理旧 error

⑥ 开发常见报错+坑点+解决方案

  • 任务一直 running：异常分支未提交最终状态。解决：在 finally 中统一 commit。
  • 后台任务数据库报错：复用请求 Session。解决：worker 内部使用 with Session(engine)。
  • 前端看到 202 就展示成功结果：混淆接收和完成。解决：收到 202 后继续查询任务详情。
  • 失败信息泄露内部异常：直接写入 str(exc) 或 traceback。解决：已知错误写稳定码，未知错误写通用码。
  • completed 同时带 error：成功路径未清理旧错误。解决：成功后设置 task.error = None。

⑧ 课后记忆习题+标准答案

1. 为什么后台任务失败后不能把 202 改成 503？

后台处理发生在响应发送之后，原始 HTTP 响应已经结束，后端只能更新数据库里的任务状态。

2. queued -> running -> completed/failed 状态机解决什么问题？

它让前端和调用方明确任务阶段，区分等待、处理中、成功和失败，而不是通过空 result 猜测状态。

3. 为什么成功路径要设置 task.error = None？

如果任务重试或对象复用，旧错误可能残留。成功状态必须与错误字段保持语义一致。

4. 为什么后台 worker 只传 task_id？

主键简单、稳定、可序列化，后台函数可用它重新读取最新数据库状态，避免传递过期 ORM 对象。

⑨ 面试高频真题+解析

题 1：如何设计一个异步任务 API？

提交接口返回 202 和 task_id；任务表保存状态、结果、错误、时间戳；后台 worker 更新状态；查询接口返回当前状态；失败使用稳定错误码；必要时支持取消、重试和幂等键。

题 2：FastAPI BackgroundTasks 能替代 Celery 吗？

不能完全替代。BackgroundTasks 适合轻量、响应后执行的本进程任务；Celery 等队列系统支持独立 worker、持久化、重试、分布式调度和故障恢复。

题 3：任务失败信息应该如何对外展示？

对外展示稳定错误码和安全 message；内部日志记录详细异常、堆栈、trace id 和上下文。不要把内部 traceback 放到用户可见字段。
