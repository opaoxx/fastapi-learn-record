Lesson 0022

任务失败状态：异步任务不靠 HTTP 状态码表达最终结果

后台任务的响应早就返回了，所以模型失败时，不能再改 HTTP 响应。正确做法是把失败写进任务记录。

0. 本节学习目标

  • 理解同步接口错误和后台任务错误的区别。
  • 看懂 run_summary_task 如何把异常写成 failed。
  • 理解 status、result、error 三个字段的关系。
  • 能测试后台任务失败路径。

1. 为什么不能返回 503

第 21 课的 /predict 是同步接口：请求进来，模型运行，马上返回结果或错误。所以模型失败时可以直接返回 503。

但后台任务不同：

POST /tasks/summaries
  -> 202 Accepted 已经返回给前端
  -> 后台函数才开始运行
  -> 模型此时失败

HTTP 响应已经发出去了，后端不能穿越时间把刚才的 202 改成 503。所以异步任务要把最终结果写进任务状态。

2. 任务状态字段

status: queued | running | completed | failed
result: str | None
error: str | None

正常完成时：

{
  "status": "completed",
  "result": "summary text",
  "error": null
}

失败时：

{
  "status": "failed",
  "result": null,
  "error": "summary_model_timeout: The summary model timed out."
}

3. run_summary_task 的失败处理

try:
    task.status = "running"
    session.add(task)
    session.commit()
    session.refresh(task)

    task.result = ai_client.summarize(task.text)
    task.status = "completed"
    task.error = None
except AIClientError as exc:
    task.status = "failed"
    task.error = f"{exc.error_code}: {exc.message}"
except Exception:
    task.status = "failed"
    task.error = "unexpected_task_error: The summary task failed unexpectedly."
finally:
    session.add(task)
    session.commit()

这里有一个重要习惯：已知的 AI 客户端错误给出稳定错误码；未知异常给出保守的通用错误，不把内部堆栈暴露给调用方。

4. 状态流转图

queued
  -> running
    -> completed
    -> failed

queued 表示任务已经被接收；running 表示后台函数正在处理；completed 表示有结果；failed 表示处理失败，调用方应该看 error 字段。

5. 测试失败路径

class FailingSummaryClient:
    def summarize(self, text: str) -> str:
        raise AIClientError(
            message="The summary model timed out.",
            error_code="summary_model_timeout",
        )

run_summary_task(
    session=session,
    task_id=task.id,
    ai_client=FailingSummaryClient(),
)

这个测试绕过真实后台调度，直接调用服务函数。它验证的是业务结果：任务最终变成 failed，并且写入稳定错误信息。

6. 前端应该怎么处理

前端查询任务详情时，可以根据 status 分支：

  • queued 或 running：显示“处理中”。
  • completed：显示 result。
  • failed：显示失败提示，并可允许用户重试。

这就是异步任务 API 的核心：提交接口只表示“任务已接收”，查询接口才表示“任务最终结果”。

使用 POST /tasks/summaries 创建一个文本中包含 simulate-ai-failure 的任务，然后查询任务详情，观察 status 和 error。

8. 小测：先回忆再展开

1. 为什么后台任务失败后不能把 202 改成 503？

因为 HTTP 响应已经发送给客户端，后台处理发生在响应之后。

2. 任务失败时应该看哪个字段？

看 status 是否为 failed，再看 error 里的错误说明。

3. 为什么未知异常不直接返回原始内容？

因为原始内容可能不稳定或泄露内部细节，通用错误更适合作为对外契约。

9. 本节总结

本节补齐了异步任务失败路径：同步接口失败用 HTTP 状态码表达，异步任务失败用任务记录表达。这个区分在 AI 后端项目里非常关键。
