Lesson 0022

任务失败状态：异步任务不靠 HTTP 状态码表达最终结果

同步接口可以直接用 HTTP 状态码表达本次请求成功或失败；后台任务不一样。它先返回 202 Accepted，真正的模型调用发生在响应之后，所以最终结果必须写入任务记录。

0. 本节学习目标

  • 区分“请求被接收”和“任务处理成功”这两个概念。
  • 理解为什么后台任务失败后不能再把 202 改成 503。
  • 掌握 queued、running、completed、failed 的状态机含义。
  • 看懂 run_summary_task 如何把异常转换为稳定的任务状态。
  • 能为 AI 服务任务设计“可查询、可重试、可解释”的失败结果。

1. 背景：同步错误和异步错误不是一回事

第 21 课的 /predict 是同步接口。客户端发请求，服务端立刻调用模型，然后马上返回预测结果。如果模型不可用，服务端还能在同一个 HTTP 响应里返回 503 Service Unavailable。

摘要任务接口的语义不同。POST /tasks/summaries 的职责不是“现在就给我摘要”，而是“请接收这个任务，稍后我来查结果”。这也是为什么它返回 202 Accepted。

MDN 对 202 的解释很关键：它表示请求已被接受处理，但处理尚未完成，而且结果之后可能成功也可能失败。FastAPI 的 BackgroundTasks 文档也强调，它适合在响应返回后继续执行一些客户端不必等待的工作。

2. 核心概念：202 是接收证明，不是成功证明

200 OK 通常表示这次请求已经完成，并且响应体就是结果。

202 Accepted 表示服务端已经接收请求，但后续处理还没结束。它更像“排队凭证”或“任务回执”。客户端拿到 202 后，不能假设任务最终一定成功。

所以异步任务 API 通常拆成两个接口：

POST /tasks/summaries
  -> 返回 202 + task_id

GET /tasks/{task_id}
  -> 返回 queued/running/completed/failed + result/error

这就是“提交”和“查询”的分离。这个分离一旦理解清楚，后面的任务历史、分页、筛选都会变得自然。

3. 运行机制：失败发生在响应之后

一次摘要任务从提交到失败，大致经历这条时间线：

  1. 客户端提交 POST /tasks/summaries。
  2. FastAPI 校验 API Key 和请求体。
  3. 服务端在数据库里创建任务，初始状态为 queued。
  4. 路由调用 background_tasks.add_task(...) 登记后台函数。
  5. 服务端返回 202 Accepted 和任务 ID。
  6. HTTP 响应已经到达客户端。
  7. 后台函数才真正调用 ai_client.summarize(...)。
  8. 如果 AI 客户端失败，后台函数把任务改成 failed，并写入 error。

第 6 步是分界线。过了这一步，原始 HTTP 响应已经结束，后端不能再改变它。能改变的是数据库里的任务记录。

4. 任务记录的状态模型

status: queued | running | completed | failed
result: str | None
error: str | None

这三个字段共同表达任务最终状态：

  • status 是任务处于哪个阶段。
  • result 是成功时给用户看的摘要。
  • error 是失败时给用户或前端看的错误说明。

正常完成时：

{
  "status": "completed",
  "result": "summary text",
  "error": null
}

处理失败时：

{
  "status": "failed",
  "result": null,
  "error": "summary_model_timeout: The summary model timed out."
}

注意：不要让 completed 同时带着 error，也不要让 failed 同时带着一个看起来成功的 result。状态字段和结果字段要保持语义一致。

5. 状态机：不要把状态当普通字符串

状态字段看起来只是字符串，但从系统设计角度，它是一个小状态机：

queued
  -> running
    -> completed
    -> failed
  • queued：任务已经被接收，等待后台处理。
  • running：后台函数已经开始处理。
  • completed：任务成功结束，可以读取 result。
  • failed：任务失败结束，应该读取 error。

状态机的价值是让前端和用户知道“现在该做什么”。例如 running 可以显示加载中，failed 可以显示重试按钮，completed 可以展示结果。

6. 代码精读：run_summary_task 如何处理失败

try:
    task.status = "running"
    session.add(task)
    session.commit()
    session.refresh(task)

    task.result = ai_client.summarize(task.text)
    task.status = "completed"
    task.error = None
except AIClientError as exc:
    task.status = "failed"
    task.error = f"{exc.error_code}: {exc.message}"
except Exception:
    task.status = "failed"
    task.error = "unexpected_task_error: The summary task failed unexpectedly."
finally:
    session.add(task)
    session.commit()

这段代码有四个重点：

  1. 先把状态改成 running 并提交，让查询接口能看见任务已开始。
  2. 模型调用成功时写入 result，状态改成 completed。
  3. 已知 AI 客户端错误转成稳定错误码，方便前端和测试判断。
  4. 未知异常只暴露通用错误，不把内部堆栈或敏感细节写入对外字段。

finally 里的提交很重要：不管成功还是失败，任务最终状态都要落库。否则客户端会一直看到 running，像任务卡死了一样。

7. 为什么要区分已知错误和未知错误

AI 服务里常见失败包括超时、限流、上游服务不可用、输入过长、模型拒绝处理等。这些属于“预期内失败”，应该有稳定错误码。例如：

summary_model_timeout
summary_provider_unavailable
summary_input_too_long

未知异常则不同。它可能是代码 bug、数据库问题、序列化问题，甚至是你暂时没想到的情况。对外直接暴露原始异常不稳定，也可能泄露内部实现。更稳的做法是写入通用错误：

unexpected_task_error: The summary task failed unexpectedly.

这不是隐藏问题，而是区分“外部契约”和“内部排障”。外部契约要稳定，内部日志可以更详细。

8. 前端应该如何消费任务状态

前端查询任务详情或任务历史时，可以按状态分支：

if (task.status === "queued" || task.status === "running") {
  showLoading(task);
}

if (task.status === "completed") {
  showResult(task.result);
}

if (task.status === "failed") {
  showError(task.error);
}

这段伪代码背后的原则是：前端不要猜测任务是否完成，而要相信后端任务记录。任务状态就是前后端共享的协议。

9. 测试失败路径为什么要绕过真实后台调度

class FailingSummaryClient:
    def summarize(self, text: str) -> str:
        raise AIClientError(
            message="The summary model timed out.",
            error_code="summary_model_timeout",
        )

run_summary_task(
    session=session,
    task_id=task.id,
    ai_client=FailingSummaryClient(),
)

这个测试没有真的等待 BackgroundTasks 调度，而是直接调用 service 函数。原因是我们要验证业务规则：只要摘要客户端失败，任务就必须变成 failed，并写入稳定错误信息。

测试越靠近规则，越不容易被时间、调度、线程这类因素干扰。以后如果引入 Celery 或 Redis 队列，这条业务规则仍然应该成立。

10. 常见错误与诊断

错误一：把 202 当成功结果

202 只表示任务被接收。要看最终结果，必须查询任务详情。

错误二：失败后不 commit

如果异常分支没有提交，数据库里仍然可能是旧状态，前端会误以为任务还在运行。

错误三：把异常原文直接暴露

原始异常可能不稳定或包含内部信息。对外字段应使用稳定错误码和可读消息。

错误四：状态和字段冲突

failed 应该配 error，completed 应该配 result。不要让语义混乱。

11. 课堂实验

创建一个会触发失败的任务：

POST /tasks/summaries
X-API-Key: dev-secret-key
Content-Type: application/json

{
  "text": "This summary task will simulate-ai-failure after it is accepted."
}

然后查询任务详情：

GET /tasks/{task_id}
X-API-Key: dev-secret-key

观察三件事：

  1. 提交接口返回的是 202，不是 503。
  2. 任务详情最终应该出现 "status": "failed"。
  3. error 字段应该包含稳定错误码，而不是一整段 Python 堆栈。

12. 小测：先回忆再展开

1. 为什么后台任务失败后不能把 202 改成 503？

因为 202 响应已经发送给客户端，后台处理发生在响应之后，后端只能更新任务记录。

2. 任务失败时前端应该看哪个字段？

先看 status 是否为 failed，再读取 error 中的错误说明。

3. 为什么要给已知 AI 错误稳定错误码？

稳定错误码方便前端展示、测试断言、日志检索和后续重试策略设计。

4. finally 中提交任务状态解决了什么问题？

它确保成功或失败的最终状态都会落库，避免客户端一直看到旧状态。

13. 本节总结

本节的核心不是“多加一个 failed 字符串”，而是理解异步 API 的基本契约：提交接口表示任务被接收，查询接口表达最终结果。AI 服务越接近真实世界，越需要这种状态模型，因为外部模型调用天然可能慢、失败、限流或超时。
