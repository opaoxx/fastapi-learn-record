Lesson 0024

任务状态筛选：只看 queued、running、completed 或 failed

分页解决“列表太长”的问题，筛选解决“我只关心其中一类”的问题。任务历史页如果不能筛选失败任务，用户就很难定位问题。

0. 本节学习目标

  • 理解状态筛选为什么属于列表查询条件。
  • 理解 Literal 如何把普通字符串升级为有限枚举。
  • 看懂 Query(alias="status") 解决了什么命名问题。
  • 掌握筛选、排序、分页的组合顺序。
  • 知道为什么非法状态应该返回 422，而不是空列表。

1. 背景：任务历史不只是“列出来”

当任务数量变多后，用户通常不会只想看全部任务。他可能会问：

  • 哪些任务失败了？
  • 哪些任务还在处理中？
  • 最近成功的摘要有哪些？

这些问题都指向同一个资源集合 /tasks，只是观察角度不同。因此我们不需要新建 /failed-tasks、/completed-tasks 这些路径，而是在列表接口上增加筛选条件。

2. 核心概念：筛选是集合视图，不是新资源

推荐写法：

GET /tasks?status=completed
GET /tasks?status=failed
GET /tasks?status=running

这里 /tasks 仍然表示任务集合，status 表示从集合里挑哪一部分。

路径参数更适合定位单个资源，比如 /tasks/12。查询参数更适合描述列表视图，比如分页、筛选、排序、搜索。

3. 用 Literal 限制合法状态

TaskStatus = Literal["queued", "running", "completed", "failed"]

这行代码把 TaskStatus 定义成一个有限集合。它不是任意字符串，只允许四个值。

这对 API 很重要。没有限制时，/tasks?status=complete、/tasks?status=done、/tasks?status=success 都可能悄悄进入业务逻辑。用了 Literal 后，非法值会在请求解析阶段被拒绝。

4. 代码精读一：对外叫 status，对内叫 task_status

task_status: Annotated[
    TaskStatus | None,
    Query(alias="status"),
] = None

为什么不直接把 Python 变量命名为 status？因为路由文件里通常还会从 FastAPI 导入 status 模块，例如 status.HTTP_202_ACCEPTED。如果参数也叫 status，读代码时容易混乱。

Query(alias="status") 让外部 URL 仍然保持简洁：

GET /tasks?status=failed

但 Python 函数内部使用更明确的变量名：

task_status == "failed"

5. 为什么 TaskStatus | None 很重要

None 表示客户端没有传 status，也就是“不筛选”。如果没有 None，这个参数就会变成必填，GET /tasks 反而不能查看全部任务。

因此这段类型表达的是：

status 不传 -> 返回全部任务
status=completed -> 只返回 completed
status=failed -> 只返回 failed
status=unknown -> 422 校验错误

这比在 service 里手写一堆字符串判断更清楚，也更容易被 OpenAPI 文档展示出来。

6. 代码精读二：where 条件只在需要时添加

statement = select(SummaryTask).order_by(SummaryTask.id.desc())

if task_status is not None:
    statement = statement.where(SummaryTask.status == task_status)

statement = statement.offset(offset).limit(limit)

这里的逻辑是动态构造查询：

  1. 先构造基础查询：查任务表，并按最新任务排序。
  2. 如果传了状态，就添加 where 条件。
  3. 最后应用分页。

为什么最后才分页？因为你想要的是“失败任务里的第一页”，而不是“全部任务第一页里刚好失败的那几条”。顺序应该是：筛选 -> 排序 -> 分页。

7. 非法状态为什么应该是 422

假设用户请求：

GET /tasks?status=done

如果返回空列表，用户可能误以为系统里没有 done 状态的任务。但真实问题是：done 根本不是合法状态。

所以 422 更准确。它告诉调用方：你的请求参数形状不符合 API 契约。FastAPI 会根据类型注解和 Query 规则自动做这件事。

8. 筛选和分页组合

GET /tasks?status=completed&limit=10&offset=0
GET /tasks?status=failed&limit=10&offset=10

这两个 URL 体现了列表接口的组合能力：

  • status 决定看哪一类任务。
  • limit 决定这一页最多几条。
  • offset 决定从这类任务中的哪个位置开始。

第 26 课会继续补上 count 查询，让前端知道“当前筛选条件下一共有多少条”。

9. 测试覆盖的意义

completed = client.get(
    "/tasks?status=completed&limit=10",
    headers=API_KEY_HEADERS,
)
invalid = client.get(
    "/tasks?status=unknown",
    headers=API_KEY_HEADERS,
)

assert completed.status_code == 200
assert all(task["status"] == "completed" for task in completed.json()["items"])
assert invalid.status_code == 422

这不是为了“凑测试数量”。它守住了两个契约：

  • 合法状态必须真的筛选出对应任务。
  • 非法状态必须被拒绝，而不是悄悄返回误导性的结果。

10. 常见错误与诊断

错误一：把筛选做成路径

/tasks/failed 容易和任务 ID 路径混淆。筛选更适合 ?status=failed。

错误二：非法状态返回空列表

空列表表示没有数据；非法状态是请求错误，应返回 422。

错误三：先分页再筛选

这会变成“当前页里的失败任务”，不是“失败任务的当前页”。

错误四：变量名遮蔽模块名

内部用 task_status，外部用 alias="status"，更清晰。

11. 课堂实验

分别请求：

GET /tasks?status=completed
GET /tasks?status=failed
GET /tasks?status=running
GET /tasks?status=unknown

观察：

  1. 前三个请求是否返回 200。
  2. 返回的 items 中，每条任务的 status 是否匹配筛选条件。
  3. 最后一个请求是否返回 422。

再思考一个前端问题：如果页面上有“失败任务”按钮，它应该把任务历史请求改成哪个 URL？

12. 小测：先回忆再展开

1. 为什么状态筛选适合做查询参数？

因为它是在查看同一个任务集合时附加的列表视图条件。

2. TaskStatus | None 里的 None 表示什么？

表示没有传状态筛选条件，因此返回全部状态的任务。

3. 非法状态为什么返回 422 而不是空列表？

因为非法状态表示请求参数不符合 API 契约，问题在请求本身，不是数据为空。

4. 筛选、排序、分页的推荐顺序是什么？

先筛选候选数据，再稳定排序，最后用 offset 和 limit 截取当前页。

13. 本节总结

本节让任务列表具备了“按状态查看”的能力。更重要的是，你学到了列表接口设计中的一个基本判断：路径定位资源，查询参数描述视图。状态筛选、分页、搜索、排序都属于集合视图的条件。

推荐主资源：FastAPI Query Parameters、FastAPI Query Validations、SQLModel Limit and Offset。
