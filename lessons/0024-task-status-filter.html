Final Review + CSDN Deep Dive

任务状态筛选：Literal 校验、Query alias 与筛选分页组合查询

分页解决“返回多少”，筛选解决“返回哪类”。本节把 GET /tasks?status=failed 建成正式 API 契约：合法状态由类型系统约束，非法状态返回 422，service 按“筛选 -> 排序 -> 分页”的顺序构造 SQL 查询。

① 本节核心知识框架

任务状态筛选
├─ 合法状态：TaskStatus = Literal["queued", "running", "completed", "failed"]
├─ HTTP 表达：GET /tasks?status=completed
├─ Python 内部名：task_status
├─ Query alias：对外 status，对内 task_status
├─ 可选语义：TaskStatus | None
│  ├─ None：不筛选，返回全部
│  └─ 非 None：添加 where 条件
├─ 查询顺序：where -> order_by -> offset/limit
└─ 错误语义：非法状态是 422，不是空列表

② 核心概念底层原理

状态筛选属于集合资源的视图条件。/tasks 表示任务集合，?status=failed 表示从集合中选择失败任务；它不是新资源路径。把筛选做成 /tasks/failed 容易和 /tasks/{task_id} 产生路由语义冲突。

Literal 把开放字符串变成有限枚举。没有它，status=done 会进入 service，开发者可能手写判断后返回空列表；但空列表表示“合法条件下没有数据”，不是“请求参数非法”。因此非法状态应该在 FastAPI/Pydantic 请求解析阶段返回 422。

TaskStatus | None 表达可选筛选：不传 status 时返回全部任务，传合法 status 时返回对应子集。这个类型信息还会进入 OpenAPI，让调用方看到 status 的枚举值。

③ 全套代码逐行精读解析

TaskStatus = Literal["queued", "running", "completed", "failed"]
  • TaskStatus 是类型别名，不是运行时 enum 类；它主要服务类型检查、Pydantic 校验和 OpenAPI 文档。
  • 四个字符串必须与数据库中 SummaryTask.status 写入值一致；如果 service 写 "done"，筛选契约会断裂。
  • 把状态集中定义，避免多个文件散落魔法字符串。
task_status: Annotated[
    TaskStatus | None,
    Query(alias="status", description="Optional status filter for the task list."),
] = None
  • task_status 是 Python 内部变量名，避免遮蔽 FastAPI 导入的 status 模块。
  • TaskStatus | None 表示参数可缺省；如果没有 None，GET /tasks 会变成缺少必填参数。
  • Query(alias="status") 让外部 URL 保持短而符合习惯，仍写 ?status=failed。
  • description 写入 OpenAPI，帮助调用方理解参数用途。
  • = None 是默认值，语义是“不筛选”。如果默认写成 "completed"，全量列表会被悄悄改成只看成功任务。
statement = select(SummaryTask).order_by(SummaryTask.id.desc())
count_statement = select(func.count()).select_from(SummaryTask)
if task_status is not None:
    statement = statement.where(SummaryTask.status == task_status)
    count_statement = count_statement.where(SummaryTask.status == task_status)
statement = statement.offset(offset).limit(limit)
  • statement 查询当前页数据；count_statement 查询总数，二者必须使用相同筛选条件。
  • if task_status is not None 区分“不传筛选”和“传合法筛选”。不要写 if task_status 作为通用习惯，虽然当前字符串非空，但语义不如显式 None 清楚。
  • where(SummaryTask.status == task_status) 把 API 参数转换为 SQL WHERE 条件。
  • count_statement.where(...) 很容易漏掉；漏掉会导致 items 是失败任务，但 count 是全部任务数量。
  • offset/limit 放在筛选之后，表示“失败任务列表的第一页”，而不是“全部任务第一页里的失败项”。
invalid = client.get(
    "/tasks?status=unknown",
    headers=API_KEY_HEADERS,
)
assert invalid.status_code == 422

测试非法状态不是为了凑覆盖率，而是在保护 API 契约：未知状态必须被拒绝。否则前端拼错参数时会得到空列表，误以为系统没有数据。

④ 核心机制运行全流程拆解

GET /tasks?status=failed&limit=10&offset=0
  -> FastAPI 读取 query 参数 status
  -> alias 映射到 Python 参数 task_status
  -> Pydantic/Literal 校验 failed 是否属于合法集合
  -> 合法：进入 router
       -> service 接收 task_status="failed"
       -> select(SummaryTask)
       -> where(status == "failed")
       -> order_by(id desc)
       -> offset/limit
       -> items 与 count 使用同一 where 条件
       -> 返回 envelope
  -> 非法：直接返回 422，service 不执行

筛选、排序、分页顺序必须稳定：先缩小候选集合，再按确定规则排序，最后切出当前页。任何调整都可能改变 API 语义。

⑥ 开发常见报错+坑点+解决方案

  • /tasks?status=unknown 返回 422：这是契约保护。解决：前端只能发送枚举值中的状态。
  • /tasks 不能访问，提示 status required：参数类型或默认值写错。解决：使用 TaskStatus | None = None。
  • 筛选后分页总数不对：count_statement 没有同步 where 条件。解决：items 查询和 count 查询使用相同过滤条件。
  • 变量名混乱：函数参数命名为 status，遮蔽 FastAPI status 模块。解决：内部用 task_status，外部 alias 为 status。
  • 筛选结果只在当前页内过滤：先分页后筛选。解决：SQL 层先 where，再排序和分页。

⑧ 课后记忆习题+标准答案

1. 为什么 status 筛选适合放 query 参数？

因为它描述的是如何查看 /tasks 这个集合资源，不定位单个新资源。

2. TaskStatus | None 中的 None 表示什么？

表示客户端没有传状态筛选条件，此时返回全部状态的任务。

3. 非法状态为什么应该返回 422？

因为请求参数不符合 API 契约。空列表表示合法条件下没有数据，不能表达参数拼错。

4. 筛选、排序、分页的推荐顺序是什么？

先 where 筛选候选集合，再 order_by 形成稳定序列，最后 offset/limit 切当前页。

⑨ 面试高频真题+解析

题 1：为什么非法枚举值不能返回空列表？

空列表表示查询条件合法但没有匹配数据；非法枚举表示客户端请求不符合契约。返回空列表会隐藏调用方 bug，应该返回 422 或等价参数错误。

题 2：列表接口如何组合筛选和分页？

筛选条件进入 WHERE，排序保证稳定顺序，分页在最后截取窗口。响应应返回 items 与分页元信息，count 也必须基于同一筛选条件。

题 3：新增任务状态时要改哪些地方？

至少更新 TaskStatus、任务状态机写入逻辑、前端展示分支、测试用例、OpenAPI/文档；生产中还要考虑旧数据迁移和旧客户端兼容。
