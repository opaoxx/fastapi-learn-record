Final Review + CSDN Deep Dive

UploadFile 文本上传：multipart、字节边界、UTF-8 校验与最小暴露响应

本节把文件上传从“会用表单传文件”提升到后端工程视角：浏览器用 multipart/form-data 传输文件，FastAPI 用 UploadFile 接收流式文件对象，后端必须做扩展名、大小、编码、存储和响应字段边界控制。

① 本节核心知识框架

文本文件上传
├─ 协议层：multipart/form-data，不是 application/json
├─ FastAPI 参数：UploadFile + File(...)
├─ 安全入口：/files 路由整体依赖 require_api_key
├─ 校验顺序：文件名 -> 读取字节 -> 大小限制 -> UTF-8 解码
├─ 持久化：UploadedTextFile 保存 filename/content_type/size/content/preview
├─ 响应模型：UploadedTextFileRead 不返回完整 content
└─ 生产边界：大文件进对象存储，数据库只保存元数据和引用

② 核心概念底层原理

JSON 适合结构化文本字段，不适合直接携带文件流。文件上传通常使用 multipart/form-data：请求体由多个 part 组成，每个 part 有自己的头部和内容；文件 part 可以包含文件名、媒体类型和原始字节。

UploadFile 是 Starlette/FastAPI 对上传文件的抽象，提供 filename、content_type 与异步 read()。它比直接接收 bytes 更贴近文件语义，也更容易扩展到流式处理。

本项目只接受 UTF-8 .txt，不是因为真实系统只能处理文本，而是为了把学习焦点集中到文件入口的完整工程闭环：鉴权、格式校验、大小限制、编码校验、持久化、最小化响应。

③ 全套代码逐行精读解析

MAX_TEXT_FILE_BYTES = 20_000

router = APIRouter(
    prefix="/files",
    tags=["files"],
    dependencies=[Depends(require_api_key)],
)
  • MAX_TEXT_FILE_BYTES：把大小限制定义为常量，避免魔法数字散落在业务逻辑中。
  • prefix="/files"：文件能力统一挂在 /files 下。
  • dependencies=[Depends(require_api_key)]：路由级鉴权，所有文件接口自动要求 X-API-Key。
@router.post("/text", response_model=UploadedTextFileRead, status_code=status.HTTP_201_CREATED)
async def upload_text_file(
    file: Annotated[UploadFile, File(description="A UTF-8 .txt file.")],
    session: Annotated[Session, Depends(get_session)],
) -> UploadedTextFile:
  • status_code=201：上传成功会创建一个新的文件记录，使用 Created 语义。
  • UploadFile：接收文件 part，而不是 JSON 字段。
  • File(...)：显式告诉 FastAPI 该参数来自 multipart 表单文件。
  • async def：因为 await file.read() 是异步读取。
if not file.filename or not file.filename.lower().endswith(".txt"):
    raise HTTPException(status_code=422, detail="Only .txt files are accepted in this lesson.")

先校验文件名是廉价校验。注意：扩展名不是绝对安全的文件类型判断，生产环境还要检查 MIME、魔数、内容扫描；但教学阶段先用它建立输入约束。

raw_content = await file.read()
if len(raw_content) > MAX_TEXT_FILE_BYTES:
    raise HTTPException(status_code=413, detail=f"Text file must be at most {MAX_TEXT_FILE_BYTES} bytes.")

读取后按字节数限制大小。413 表示请求实体过大，比 422 更贴近“体积超限”的 HTTP 语义。

try:
    content = raw_content.decode("utf-8")
except UnicodeDecodeError as exc:
    raise HTTPException(status_code=422, detail="Text file must be UTF-8 encoded.") from exc

后续摘要逻辑按字符串处理，所以必须解码。用 from exc 保留异常链，便于调试根因。

uploaded_file = UploadedTextFile(
    filename=file.filename,
    content_type=file.content_type or "application/octet-stream",
    size_bytes=len(raw_content),
    content=content,
    preview=content[:160],
)
session.add(uploaded_file)
session.commit()
session.refresh(uploaded_file)
return uploaded_file

preview 是响应展示字段，content 是内部处理字段。提交并刷新后，响应模型输出 id/filename/content_type/size_bytes/preview。

④ 核心机制运行全流程拆解

浏览器/HTTP Client 构造 multipart/form-data
  -> part name 必须是 file
  -> 请求带 X-API-Key
  -> require_api_key 通过
  -> FastAPI 解析 multipart，把文件 part 封装为 UploadFile
  -> 校验 filename 后缀 .txt
  -> await file.read() 得到 bytes
  -> 校验 len(raw_content) <= 20000
  -> raw_content.decode("utf-8")
  -> 写入 UploadedTextFile
  -> 按 UploadedTextFileRead 返回 201

⑤ 重难点、易混淆点对比辨析

概念正确理解易错点

⑥ 开发常见报错+坑点+解决方案

  • 422: field required：表单字段名不是 file。解决：前端 formData.append("file", file)。
  • 422: Only .txt files：文件名不以 .txt 结尾。解决：上传正确扩展名的文本文件。
  • 413: file too large：超过 20000 字节。解决：缩小测试文件或提高常量并补测试。
  • 422: UTF-8 encoded：文件不是 UTF-8。解决：用编辑器转换编码为 UTF-8。
  • 401 Unauthorized：文件路由整体受 API Key 保护。解决：带上 X-API-Key。
  • 手写 Content-Type 导致上传失败：浏览器无法自动补 boundary。解决：使用 FormData 时不要手写 Content-Type。

⑧ 课后记忆习题+标准答案

1. 文件上传为什么使用 multipart/form-data？

它能把文件字节和表单元数据拆成多个 part 传输，比把文件塞进 JSON 更符合浏览器和 HTTP 客户端的上传模型。

2. 为什么 UploadFile 接口要限制大小？

防止单个请求消耗过多内存、磁盘或数据库空间，也为后续 AI 处理控制成本。

3. 为什么响应模型不返回完整 content？

完整上传内容可能含隐私且体积较大，响应应遵循最小暴露原则，仅返回元数据和预览。

⑨ 面试高频真题+解析

题 1：设计文件上传接口时必须考虑哪些安全点？

鉴权、大小限制、类型校验、编码校验、文件名清洗、内容扫描、存储隔离、访问控制、日志审计和响应最小化。

题 2：UploadFile 与 bytes 参数有什么区别？

bytes 更像直接把内容读入内存；UploadFile 保留文件元数据并提供文件对象接口，更适合真实上传场景和后续流式扩展。

题 3：为什么不能信任 content_type？

它来自客户端声明，攻击者可以伪造；生产环境需要结合扩展名、魔数、解析结果和安全扫描综合判断。
