0051 Provider Metrics Counters：按 Label Set 聚合事件次数

本节实现教学版 ProviderMetricsCounter。它不是生产监控系统，而是用最小代码解释 counter 的本质：同一指标名 + 同一组 labels 对应一条计数序列，事件发生一次就递增一次，快照只暴露可序列化样本，不暴露内部状态。

① 本节核心知识框架

ProviderMetricsCounter

│  ├─ 只递增，不递减
│  ├─ 统计事件发生次数
│  └─ 不保存单次事件明细
├─ 聚合键

   └─ snapshot 与内部 Counter 隔离

② 核心概念底层原理

Counter 是单调递增的累计型指标，适合记录“发生过多少次”。provider prediction 的 success、retry_scheduled、retry_exhausted、fail_fast 都是事件，天然适合用 counter。

一条指标样本由 metric name 与 label set 唯一确定。本课程暂时把 metric name 固定为 provider prediction 计数，重点观察 label set 如何决定聚合维度。

同一 label set：
  outcome=success, failure_category=none, error_code=none, status_code=none
  第一次事件 → count=1
  第二次事件 → count=2

不同 label set：
  outcome=fail_fast, failure_category=non_retryable_http_status, status_code=400
  独立样本 → count=1

Counter 与 Gauge 必须区分。Counter 只表达累计次数；Gauge 表达当前值，可上升也可下降，例如队列长度、连接数、内存占用。把 counter 设计成可递减会破坏监控查询语义。

③ 全套代码逐行精读解析

@dataclass
class ProviderMetricsCounter:
    _counts: Counter[ProviderMetricLabels] = field(default_factory=Counter)

    def increment(self, labels: ProviderMetricLabels, amount: int = 1) -> None:
        if amount < 1:
            raise ValueError("Metric increment amount must be positive.")
        self._counts[labels] += amount

    def snapshot(self) -> list[dict[str, object]]:
        samples = sorted(
            self._counts.items(),
            key=lambda item: (
                item[0].operation,
                item[0].outcome,
                item[0].failure_category,
                item[0].error_code,
                item[0].status_code,
            ),
        )
        return [
            {
                "labels": labels.as_dict(),
                "count": count,
            }
            for labels, count in samples
        ]

    def reset(self) -> None:
        self._counts.clear()
代码作用为什么这样写不这样写的风险
@dataclass生成轻量状态对象教学场景无需复杂类样板代码手写初始化会增加噪声
field(default_factory=Counter)为每个实例创建独立 Counter避免多个 counter 实例共享同一个可变默认值用 Counter() 直接作为默认值会产生共享状态风险
Counter[ProviderMetricLabels]以 label set 为聚合 keyProviderMetricLabels frozen 后可哈希如果 labels 可变，聚合 key 可能不可用或被修改后失效
amount < 1拒绝 0 和负数保持 counter 单调递增语义0 会隐藏无效调用，负数会把 counter 伪装成 gauge
self._counts[labels] += amount相同 labels 自动累加复用标准库 Counter 的计数语义手写 dict 容易漏初始化或写错 key
sorted(...)稳定快照顺序测试、文档、导出输出可预测无序输出会导致快照测试脆弱
labels.as_dict()返回普通 dict labels让 router、JSON、Prometheus text 不依赖 dataclass 内部直接返回对象不利于序列化和隔离
reset()清空进程内计数服务测试隔离与演示重置测试之间会互相污染

④ 核心机制运行全流程拆解

build_provider_metric_labels(outcome="success")
  → ProviderMetricLabels(...)
  → counter.increment(labels)
  → Counter 使用 labels.__hash__ 定位 key
  → 已存在：count += 1
  → 不存在：初始化 count=0 后 += 1
  → snapshot()
       ├─ 按 operation/outcome/failure_category/error_code/status_code 排序
       ├─ labels.as_dict()
       └─ 返回 [{"labels": ..., "count": ...}]

测试中的三次递增：

counter.increment(success_labels)
counter.increment(success_labels)
counter.increment(fail_fast_labels)

会形成两条样本，而不是三条事件明细：

[
  {"labels": fail_fast_labels.as_dict(), "count": 1},
  {"labels": success_labels.as_dict(), "count": 2}
]

这正是 metrics 与 logs 的边界：metrics 聚合，不负责保存事件流水；单次请求细节交给结构化日志。

⑧ 课后记忆习题+标准答案

1. 为什么 ProviderMetricLabels 必须可哈希？

因为它被用作 Counter 的 key。只有可哈希对象才能稳定定位同一组 label set，从而把相同标签的事件累加到同一计数。

2. 为什么 increment() 不允许 amount=0？

0 增量没有事件含义，允许它会掩盖无效调用。Counter 应表达事件发生，事件发生至少加 1。

3. 为什么 snapshot 要排序？

排序让测试断言、文档展示和后续文本导出可预测。否则同样的计数可能因遍历顺序不同导致快照测试不稳定。

4. 进程内 Counter 为什么不能直接当生产方案？

它无法自动跨进程、跨实例聚合，进程重启会丢失状态，也缺少成熟监控系统的采集、存储、查询和告警能力。

⑨ 面试高频真题+解析

面试题 1：Counter 的核心语义是什么？

Counter 表示累计发生次数，只能递增，常用于请求总数、错误总数、重试次数等事件型信号。若需要表示当前队列长度或连接数，应使用 Gauge，而不是让 Counter 递减。

面试题 2：如何用 label set 做聚合？

metric name 加 label set 唯一确定一条序列。同一 label set 的事件累加到同一 count，不同 label set 形成不同样本。标签必须低基数，否则会产生过多时间序列。

面试题 3：为什么快照函数不直接返回内部 Counter？

直接返回内部 Counter 会泄漏实现细节，并允许调用方修改内部状态。快照应返回序列化友好的副本形状，如 labels dict 加 count，保护模块边界。
