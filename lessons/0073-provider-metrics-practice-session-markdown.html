0073 Provider Metrics Practice Session Markdown：把完整训练会话导出为 GitHub 复盘文档

本节新增 render_provider_metrics_runbook_practice_session_markdown()，把 0072 的 practice session 包渲染成完整 Markdown 文档。它将 findings、题卡、答案键和评分报告整合到一份可复制到 GitHub、PR 或学习记录的离线材料中。

① 本节核心知识框架

│  ├─ findings_markdown
│  ├─ cards
│  ├─ answer_key
│  └─ report_markdown
├─ 输出：完整 Markdown 文档
│  ├─ 标题
│  ├─ Findings
│  ├─ Exercise Cards
│  ├─ Answer Key
│  └─ Grading Report
├─ 设计原则
│  ├─ 不重新生成 session
│  ├─ 不重新评分
│  ├─ 题卡区不暴露 next_action
│  └─ 保持 GitHub 可读格式
└─ 验证重点

   ├─ 表头稳定
   ├─ answer key 不丢
   └─ grading report 不丢

② 核心概念底层原理

practice session 是机器可消费的结构化对象，Markdown package 是人类可阅读的发布格式。二者分离后，训练流水线可以被测试和复用，展示格式也可以独立演进。

本节 renderer 只消费 session，不负责构建 session。这样能避免展示层重新调用 samples、findings、grading 等函数，防止同一个练习在不同入口产生不同结果。

题卡区只展示 id、prompt 和 expected_severity，不展示 expected_next_action。因为题卡段面向练习，答案动作应留在 Answer Key 和 Grading Report 中，避免题干提前泄露完整答案。

内容块来源字段用途
Findingsfindings_markdown复盘指标解释
Exercise Cardscards给学习者自测题目
Answer Keyanswer_key提供标准答案
Grading Reportreport_markdown复盘作答结果

复核补强：本节要防止一个常见误区：Markdown 包不是重新执行训练，而是对已有 session 的发布投影。导出层如果重新构造 session，会破坏“同一份结构化数据导出多种格式”的工程边界。

发布块可见内容隐藏内容或边界
Findings已解释的指标现象不重新读取 counter 或 provider。
Exercise Cards题目、题号、severity 锚点不泄露 expected_next_action。
Answer Key标准答案不参与评分计算。
Grading Report评分结果和反馈不保存原始 response 文本。

③ 全套代码逐行精读解析

def render_provider_metrics_runbook_practice_session_markdown(
    session: dict[str, object],
) -> str:
    lines = [
        "# Provider Metrics Runbook Practice Session",
        "",
        "## Findings",
        "",
        str(session["findings_markdown"]).rstrip(),

函数接收完整 session 包，返回单个 Markdown 字符串。开头先建立文档标题和 Findings 章节，rstrip() 去掉已有片段末尾多余换行，避免组合后空行失控。

"",
"## Exercise Cards",
"",
"| card_id | prompt | expected_severity |",
"| --- | --- | --- |",
]

Exercise Cards 章节使用稳定表格结构。只放 expected_severity，不放 expected_next_action，是为了保持练习区和答案区分层。

cards = session["cards"]
if not isinstance(cards, list):
    raise ValueError("Practice session cards must be a list.")

cards 必须是列表，因为表格需要逐题渲染。若调用方传入错误结构，立即抛出可诊断错误，而不是输出损坏表格。

for card in cards:
    lines.append(
        "| {card_id} | {prompt} | {expected_severity} |".format(
            card_id=card["id"],
            prompt=card["prompt"],
            expected_severity=card["expected_severity"],
        )
    )

每张题卡渲染为一行。使用字段索引读取可以暴露 session 契约错误；如果题卡缺 id 或 prompt，说明上游练习包生成已经损坏。

lines.extend(
    [
        "",
        "## Answer Key",
        "",
        str(session["answer_key"]).rstrip(),
        "",
        "## Grading Report",
        "",
        str(session["report_markdown"]).rstrip(),
        "",
    ]
)
return "\n".join(lines)

最后追加答案键和评分报告。返回值保留末尾换行，因为列表最后一个元素是空字符串，join 后自然以换行结束，便于直接写入 Markdown 文件。

逐行精读补充：str(session["findings_markdown"]).rstrip() 不是为了吞掉坏类型，而是为了组合已经生成的 Markdown 片段。真正的 session 结构完整性应由 0072 的构造函数和后续发布检查保护。

cards 类型被显式校验为 list，但单张 card 字段仍用索引读取。这样可以把“cards 整体结构错”和“某张题卡字段缺失”区分开：前者给 ValueError，后者暴露 KeyError，定位更具体。

④ 核心机制运行全流程拆解

responses
  → build_provider_metrics_runbook_practice_session(responses)
  → session
  → render_provider_metrics_runbook_practice_session_markdown(session)
  → full Markdown package
  → GitHub / PR / learning record

完整链路把训练执行和训练发布分开。session 负责携带结构化数据，Markdown renderer 负责发布格式。这个边界让测试可以分别锁定数据正确性和文本格式稳定性。

阶段产物检查点常见风险
会话生成session dict字段齐全中间产物缺失
报告组合Markdown str章节顺序重复计算或漏章节
发布复盘GitHub 文档可读性题干泄露完整答案

易混点错误理解正确理解
session Markdown等于重新执行练习它只渲染已有 session。
Exercise Cards应展示全部标准答案练习区只展示题目和必要锚点。

⑧ 课后记忆习题+标准答案

1. 为什么本节 renderer 只接收 session？

这样可以避免展示层重新执行练习流水线，保证同一份 session 在不同渲染入口结果一致。

2. 为什么 Exercise Cards 表不展示 next_action？

next_action 是学生需要推导的关键答案，提前放入题卡区会破坏训练价值。

3. 为什么要把 Answer Key 和 Grading Report 都放入完整包？

Answer Key 说明标准答案，Grading Report 说明本次作答结果，二者服务不同复盘需求。

4. rstrip() 在组合 Markdown 时解决什么问题？

它去掉嵌入片段末尾多余换行，避免多段 Markdown 组合后产生不可控空行。

5. 生产环境如何升级这个 Markdown renderer？

应引入模板版本、字段转义、权限控制、报告归档、审批流程和快照测试。

⑨ 面试高频真题+解析

面试题 1：为什么导出层不应该重新计算业务结果？

导出层的职责是格式转换。重新计算会制造第二套事实源，导致报告和结构化 session 可能不一致。

面试题 2：如何设计 GitHub 友好的技术报告？

应使用稳定标题、清晰章节、表格化关键信息、可复制文本和测试保护的输出格式。

面试题 3：为什么完整复盘包需要中间材料？

只看最终评分无法解释为什么这么判。Findings、题卡和答案键能提供从事实到结果的可追溯链路。

面试题 4：生产报告系统为什么需要模板治理？

报告一旦用于协作或审计，就需要版本、审批、权限、脱敏、归档和回归测试，不能靠散乱字符串拼接。
