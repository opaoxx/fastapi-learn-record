0081 GitHub Issue Templates：把问题入口改造成可复现、可分派、可维护的协作协议

本节新增 .github/ISSUE_TEMPLATE/，用 GitHub Issue Forms 把 bug report、lesson request、documentation fix 三类输入结构化。PR template 约束“提交代码前交代什么”，CODEOWNERS 约束“谁来审查”，Issue Templates 约束“问题进入仓库时必须提供什么证据”。

② 核心概念底层原理

Issue 是开源仓库的任务输入单元。低质量 issue 通常只写“这里有问题”“看不懂”“报错了”，维护者必须反复追问环境、步骤、期望、实际结果、影响文件和验证命令。Issue Template 的核心价值不是美化页面，而是把维护者需要的最小上下文转化为创建 issue 时的强约束字段。

GitHub 支持 Markdown 模板和 Issue Forms。Markdown 模板是自由文本预填充，适合轻量仓库；Issue Forms 使用 YAML 描述表单字段，能定义输入框、文本域、下拉框、复选框和必填校验。对于课程仓库，Issue Forms 更适合，因为课程反馈既有 bug，也有学习缺口和文档质量问题，必须在入口处先分类。

本节的三个模板分别对应三种不同任务语义。Bug report 强调可复现性，核心字段是 steps、expected、actual；Lesson request 强调学习差距，核心字段是 learning gap、required depth、expected output；Documentation fix 强调文档准确性，核心字段是 affected file、documentation issue、suggested fix。字段不是随意命名，而是由后续维护动作倒推得到。

模板类型底层问题必须收集的证据后续动作
Bug report代码、测试、页面或流程可复现失败。复现步骤、期望结果、实际结果、验证命令。复现、定位、修复、回归测试。
Lesson request学习者无法理解、实现或应试某个知识点。关联课程、学习缺口、所需深度、输出类型。补课、审稿、重构、追加速查。
Documentation fix文档过期、浅、错、漏或导航不一致。受影响文件、问题描述、建议修复、课程标准影响。改文档、同步索引、补契约测试。

Issue Template 与 PR Template 的差异必须分清：Issue Template 面向“问题进入仓库前”，负责把输入变成可分派任务；PR Template 面向“改动进入主分支前”，负责让贡献者交代实现、验证和风险。前者降低需求澄清成本，后者降低审查成本。

③ 全套代码逐行精读解析

name: Bug report

name 是 GitHub issue 创建页显示的模板名称。名称必须短而明确，否则新手不知道该选哪个入口。这里不用“Problem”这种泛词，而是直接写 Bug report，表示只处理可复现异常。

description: Report a reproducible problem in code, tests, lessons, or repository workflow.

description 是模板说明。它把 bug 的范围限定到 code、tests、lessons、repository workflow。若不限定范围，学习建议、需求讨论、文档错字都会混进 bug 队列，后续 triage 成本上升。

title: "[Bug]: "
labels: ["bug"]

title 预填标题前缀，便于 issue 列表扫描。labels 自动打上分类标签，让维护者可以按 bug 筛选。生产仓库中标签必须预先存在，否则自动打标可能失败或表现不符合预期。

body:
  - type: markdown
    attributes:
      value: |
        Use this form when something is broken and can be reproduced.

body 是表单字段数组。第一个字段是 markdown，只展示说明，不收集输入。它强调“broken and reproducible”，把不可复现的主观感受挡在 bug 模板之外。

  - type: textarea
    id: summary
    attributes:
      label: Problem summary
      description: State the broken behavior in one or two precise sentences.
      placeholder: The provider metrics lesson link points to a missing file.
    validations:
      required: true

textarea 适合多句描述。id 是字段稳定标识，后续自动化脚本或导出工具可依赖它。validations.required: true 使问题摘要必填；没有摘要的 bug 无法进入维护队列。

  - type: textarea
    id: steps
    attributes:
      label: Reproduction steps
      description: List the exact commands, clicks, or file paths needed to reproduce the problem.
      placeholder: |
        1. Open index.html
        2. Click Lesson 0081
        3. Observe the missing page
    validations:
      required: true

复现步骤是 bug report 的核心字段。没有 steps，维护者只能猜测环境和路径。placeholder: | 使用 YAML 多行字符串，保留编号格式；如果缩进错误，GitHub 可能无法解析表单。

  - type: textarea
    id: expected
    attributes:
      label: Expected behavior
      description: Describe the correct behavior or expected learning outcome.
    validations:
      required: true

expected 描述正确结果。它让维护者区分“程序真的错了”和“用户期望与设计不一致”。课程仓库还要写 expected learning outcome，因为文档 bug 可能表现为学习路径不完整。

  - type: textarea
    id: actual
    attributes:
      label: Actual behavior
      description: Include error messages, failing assertions, broken links, or screenshots when relevant.
    validations:
      required: true

actual 记录真实结果。报错文本、失败断言、断链和截图都属于实际证据。若缺少 actual，维护者无法判断严重程度，也无法设计回归测试。

  - type: textarea
    id: verification
    attributes:
      label: Verification already tried
      description: Paste relevant test commands, browser checks, or GitHub workflow results.
      placeholder: |
        .\.venv\Scripts\python.exe -m pytest tests\test_course_docs_contract.py -q
        git diff --check
    validations:
      required: false

verification 不强制必填，因为新手可能还不会跑测试；但 placeholder 给出本仓库高价值命令。教学环境允许用户只提交可复现现象，生产环境通常会要求版本、环境、日志、trace id 和影响范围。

name: Lesson request
labels: ["course", "lesson-request"]

课程需求模板使用两个标签：course 表示课程域，lesson-request 表示新增或加深内容的需求。与 bug 分离后，维护者可以按“教学内容债务”集中处理。

  - type: input
    id: lesson-range
    attributes:
      label: Related lesson or range
      placeholder: Lesson 0081, Lessons 0077-0080, GitHub issue workflow
    validations:
      required: true

input 适合短文本。lesson-range 必填是为了把反馈绑定到课程位置。若没有范围，重构请求会变成泛泛意见，无法进入具体文件修改。

  - type: textarea
    id: learning-gap
    attributes:
      label: Learning gap
      description: Explain what a learner still cannot understand, implement, or answer after reading the current material.
    validations:
      required: true

learning-gap 是课程仓库最关键字段。它不问“想加什么”，而问“读完后仍然不能做什么”。这种问法能把需求从偏好转成能力缺口，更适合期末复习与面试导向课程。

  - type: dropdown
    id: output-type
    attributes:
      label: Expected output
      options:
        - New lesson HTML + quick reference
        - Existing lesson audit and refactor
        - Reference page only
        - README or workflow documentation update
    validations:
      required: true

dropdown 限定输出类型，避免 issue 讨论发散。新增课程、复核重构、只补速查、只改 README/流程文档是四类不同工作量，提前区分有助于排期。

name: Documentation fix
labels: ["documentation"]

文档修复模板只打 documentation 标签，不与 bug 混用。文档问题可能影响学习效果，但不一定意味着代码行为错误。

  - type: input
    id: file
    attributes:
      label: Affected file
      placeholder: lessons/0081-github-issue-templates.html
    validations:
      required: true

文档问题必须绑定文件路径。课程仓库文件多，若只说“这一节浅了”，维护者难以定位。路径字段也是后续 PR 检查 reference、index、NOTES 是否同步的入口。

  - type: checkboxes
    id: course-standard
    attributes:
      label: Course standard impact
      options:
        - label: The nine-section lesson structure may need to be checked.
        - label: The quick reference page may need to be updated.
        - label: index.html, README.md, NOTES.md, or COURSE_PROGRESS.md may need to be updated.

复选框把课程质量标准显式化。文档修复经常不是单文件改动：lesson 改了可能要同步 reference，新增页可能要同步 index，公开入口变化可能要补契约测试。把影响面写进模板能降低遗漏率。

blank_issues_enabled: false

config.yml 控制 issue 创建入口。禁用 blank issue 是为了避免无结构反馈绕过表单。教学仓库可以保留结构化入口；大型生产仓库也常禁用空白 issue，迫使报告人选择正确模板。

④ 核心机制运行全流程拆解

User clicks New issue
  -> GitHub reads .github/ISSUE_TEMPLATE/config.yml
  -> Blank issue is disabled
  -> User chooses one form:
       bug_report.yml
       lesson_request.yml
       documentation_fix.yml
  -> GitHub renders YAML body fields
  -> Required fields block incomplete submissions
  -> Created issue receives title prefix and labels
  -> Maintainer triages by label and field content
  -> Fix becomes a pull request
  -> PR template collects implementation evidence
  -> CI and branch protection decide merge readiness

Issue Forms 位于协作链路最前端。它不修复问题，也不验证代码；它把输入变得可读、可筛选、可复现。后续 PR、CI、CODEOWNERS、branch protection 才负责修复、验证、审查和合并。

阶段输入处理输出
创建 issue用户选择模板。GitHub 渲染 YAML 表单。结构化字段。
提交校验required 字段。GitHub 阻止缺失必填项。最低质量 issue。
维护者分流labels + field ids。按 bug、course、documentation 分类。可排期任务。
修复交付issue 上下文。创建 PR 并填写 PR template。可审查改动。
合并门禁PR diff + tests。CI、CODEOWNERS、branch protection。合并或阻断。

⑤ 重难点、易混淆点对比辨析

易混点错误理解正确理解
Issue Template 与 PR Template都是模板，作用重复。Issue Template 收集问题输入；PR Template 收集修复提交证据。
Markdown 模板与 Issue Forms只是写法不同。Issue Forms 有字段类型、必填校验和结构化数据，更适合开源协作。
label只是装饰。label 是 triage、筛选、自动化和维护优先级的基础信号。
required字段越多越专业。只强制维护者必须知道的信息；过度必填会降低提交率。
bug 与 docs文档错也是 bug，统一入口即可。可复现行为错误和文档质量问题的处理链路不同，应分流。
blank issue开放空白入口更友好。课程仓库更需要结构化反馈；空白入口会制造大量不可执行描述。

⑥ 开发常见报错+坑点+解决方案

坑点表现解决方案
YAML 缩进错误GitHub 无法渲染 issue form 或字段丢失。保持两个空格层级；多行 placeholder 使用 | 并正确缩进。
字段 id 不稳定后续自动化无法可靠读取字段。使用语义化短横线或小写 id，如 learning-gap、output-type。
必填字段过多新手放弃提交，或随便填充无效内容。只对摘要、复现、期望、实际、学习缺口等关键字段 required。
模板分类过细用户不知道该选哪个模板。先保留 bug、lesson request、documentation fix 三类高频入口。
标签不存在或命名混乱issue 无法自动进入预期分类。在生产仓库预先创建标签，并保持 README/维护文档中的名称一致。
config 链接误导contact link 指向错误页面。生产仓库应替换为真实文档站、讨论区或安全报告入口。
只加模板不加测试后续误删字段无人发现。使用文档契约测试保护模板文件、关键字段、labels 和 required 字段。

⑦ 生产环境 vs 教学环境 核心差异

维度教学仓库生产仓库
模板数量保留三类高频入口，帮助新手理解协作模型。可能增加 incident、security、feature、performance、design proposal 等模板。
字段强制只强制复现、学习缺口、受影响文件等核心字段。可能强制版本、环境、日志、trace id、影响范围、回滚方案。
自动化用文档契约测试保护模板存在与关键字段。可结合 GitHub Actions、triage bot、project board、SLA 标签自动流转。
安全入口本节不处理安全漏洞披露。安全问题通常走 private vulnerability reporting 或专门安全流程。
标签治理bug、course、lesson-request、documentation 足够教学使用。需要 severity、priority、area、team、status、release-blocker 等多维标签。
用户群体新手学习者，模板要解释字段含义。内部工程师、客户、外部贡献者并存，模板要兼顾权限和隐私。

⑧ 课后记忆习题+标准答案

1. Issue Template 解决的核心问题是什么？

它把模糊反馈结构化为可复现、可分类、可分派的任务输入，降低维护者反复追问上下文的成本。

2. 为什么本节选择 Issue Forms 而不是普通 Markdown 模板？

Issue Forms 能定义字段类型、必填校验、下拉框、复选框和稳定字段 id，更适合课程仓库的结构化反馈与后续契约测试。

3. Bug report 中最关键的三个证据字段是什么？

复现步骤、期望行为、实际行为。三者共同决定问题是否能被稳定复现和验证修复。

4. Lesson request 为什么要问 learning gap？

因为课程需求的本质是能力缺口。问学习者“读完后仍不能理解、实现或回答什么”，比泛泛问“想加什么”更可执行。

5. 禁用 blank issue 的目的是什么？

防止用户绕过结构化字段提交不可执行描述，使仓库问题入口保持最低信息质量。

⑨ 面试高频真题+解析

面试题 1：Issue Template、PR Template、CODEOWNERS、CI 分别解决什么问题？

Issue Template 结构化问题输入；PR Template 结构化修复证据；CODEOWNERS 路由人工审查责任；CI 自动验证代码与文档契约。四者分别对应输入、交付、审查、验证。

面试题 2：为什么高质量 bug report 必须包含 expected 和 actual？

expected 定义正确行为，actual 描述真实结果。没有 expected 无法判断用户期望是否合理；没有 actual 无法定位失败表现或编写回归测试。

面试题 3：Issue Forms 的 required 字段应该如何设计？

只强制维护者必须依赖的信息，例如摘要、复现步骤、期望、实际、学习缺口、影响文件。过度必填会制造无效填充，降低反馈质量。

面试题 4：为什么 label 是工程协作信号而不是装饰？

label 支撑筛选、分派、优先级、自动化、项目看板和统计分析。没有稳定 label，issue 队列会退化为无法治理的文本列表。

面试题 5：教学仓库的 issue 模板和生产仓库有什么差异？

教学仓库强调学习路径、课程深度和文档同步；生产仓库更强调环境版本、影响范围、严重级别、日志、监控、回滚、安全与 SLA。
