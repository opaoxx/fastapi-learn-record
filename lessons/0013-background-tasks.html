Lesson 0013

BackgroundTasks：先响应，再处理

AI 服务经常会遇到耗时任务。与其让客户端一直等，不如先创建任务并返回任务 ID，再在后台处理。本节先学 FastAPI 内置的 BackgroundTasks。

0. 本节学习目标

  • 理解为什么有些工作不适合阻塞 HTTP 响应。
  • 理解 FastAPI BackgroundTasks 的执行时机。
  • 看懂 background_tasks.add_task(...)。
  • 知道 BackgroundTasks 适合入门和轻量任务，不等于生产级任务队列。

1. 为什么需要后台任务

普通 CRUD 接口通常很快：查数据库、返回 JSON。但 AI 场景里的任务可能更慢，比如摘要、分类、图片理解、向量检索、长文本分析。

如果请求一直等待模型跑完，会有几个问题：

  • 客户端体验差：页面一直转圈。
  • 请求容易超时：网关、浏览器、服务器都可能设置超时。
  • 并发能力下降：慢请求占住资源。

后台任务的入门方案是：接口先确认“我收到了”，然后后台继续处理。

2. BackgroundTasks 是什么

FastAPI 官方文档说明，BackgroundTasks 可以定义“在返回响应之后运行”的任务。典型场景包括发送邮件、写日志、处理不需要客户端立刻等待的工作。

def create_summary_task(
    payload: SummaryRequest,
    background_tasks: BackgroundTasks,
    session: Annotated[Session, Depends(get_session)],
) -> SummaryTask:
    ...
    background_tasks.add_task(process_summary_task, task.id)
    return task

BackgroundTasks 由 FastAPI 注入。你不需要自己创建它，只要在路径函数参数里声明即可。

3. 202 Accepted

创建摘要任务的接口返回 202 Accepted：

@router.post(
    "/summaries",
    response_model=SummaryTaskRead,
    status_code=status.HTTP_202_ACCEPTED,
)

202 的语义是：请求已经被接受处理，但处理还没有完成。它很适合“提交任务”这类接口。

区别：201 Created 更像“资源已经创建完成”；202 Accepted 更像“任务已接收，结果稍后看”。

4. 创建任务的流程

task = SummaryTask(
    text=payload.text,
    text_length=len(payload.text),
    status="queued",
)
session.add(task)
session.commit()
session.refresh(task)

background_tasks.add_task(process_summary_task, task.id)
return task

逐行解释：

  • 先创建数据库记录，状态是 queued。
  • commit 保存，让任务拥有真实 id。
  • refresh 把数据库生成的 id 带回 Python 对象。
  • add_task 把真正处理函数排到响应之后执行。
  • 接口立刻返回任务信息。

5. 后台函数做什么

def process_summary_task(task_id: int) -> None:
    with Session(engine) as session:
        task = session.get(SummaryTask, task_id)
        ...
        task.status = "running"
        ...
        task.result = build_demo_summary(task.text)
        task.status = "completed"
        session.commit()

后台任务不能复用请求里的数据库 Session，因为请求响应结束后，请求 Session 的生命周期也结束了。所以后台函数自己创建新的 Session。

现在的摘要函数只是 demo：取前 12 个单词做摘要。以后可以把 build_demo_summary 替换成真实大模型调用。

6. BackgroundTasks 的边界

它适合入门和轻量工作，但有几个限制要知道：

  • 任务和 FastAPI 应用运行在同一个进程里。
  • 服务重启时，正在运行的后台任务可能丢失。
  • 它没有内置重试、队列监控、分布式 worker。

真实生产 AI 任务经常会使用 Celery、RQ、Arq、云队列或专门的任务系统。但 BackgroundTasks 很适合帮你先理解“提交任务”和“后台处理”的基本形状。

7. 实验：提交摘要任务

写接口仍然需要 API Key：

POST /tasks/summaries
X-API-Key: dev-secret-key

{
  "text": "FastAPI background tasks let an API accept work quickly and continue processing after sending the initial response."
}

你应该得到 202 和一个任务 ID。

8. 课堂练习

请你分别测试：

  1. 不带 X-API-Key 提交任务，观察 401。
  2. 带正确 key 提交任务，观察 202。
  3. 把文本改得很短，观察 422。

9. 小测：先回忆再展开

1. BackgroundTasks 在什么时候执行？

在响应返回给客户端之后执行。

2. 提交任务为什么用 202？

因为请求已被接受，但任务处理可能还没有完成。

3. 后台任务为什么重新创建数据库 Session？

因为请求 Session 的生命周期属于当前请求，后台函数需要自己的数据库访问上下文。

10. 本节总结

BackgroundTasks 让接口能“先响应，再处理”。本项目用它创建摘要任务，返回 202 和任务 ID，再在后台生成结果。它不是生产级任务队列，但非常适合理解 AI 服务异步任务的第一层模型。
