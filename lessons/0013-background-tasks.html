BackgroundTasks：202 接收语义、任务落库与响应后执行边界

本节把“先返回 task_id，再后台处理”的异步任务入口整理成期末复习级笔记：重点不是记住 add_task 语法，而是理解 HTTP 请求生命周期、数据库任务记录、后台函数 Session 隔离和教学级任务队列的生产边界。

① 本节核心知识框架

BackgroundTasks 异步入口
├─ 问题域：耗时任务不应长时间占用 HTTP 请求
├─ HTTP 语义：POST /tasks/summaries 返回 202 Accepted
├─ 持久化：先创建 SummaryTask(status="queued")，再返回 id
├─ 调度点：background_tasks.add_task(process_summary_task, task.id)
├─ 执行点：响应发送后，由 Starlette/FastAPI 在同进程内调用后台函数
├─ 状态机：queued -> running -> completed / failed
└─ 边界：轻量教学机制，不等价于 Celery/RQ/云队列

② 核心概念底层原理

BackgroundTasks 来自 Starlette，FastAPI 将它作为特殊参数注入到路径函数中。路径函数执行时只负责注册后台回调；真正的回调在响应对象准备完成并发送给客户端之后执行。因此客户端先拿到 202 与任务记录，后端再继续处理耗时逻辑。

202 Accepted 的语义是“请求已被接受，但处理尚未完成”。它区别于 200 OK 的同步完成语义，也区别于 201 Created 的资源已创建完成语义。本项目确实创建了任务记录，但摘要结果尚未生成，所以使用 202 更精确。

底层设计的关键约束是：后台函数仍运行在同一个应用进程内，不具备跨进程持久队列、重试、延迟调度、失败补偿和 worker 伸缩能力。它适合教学、日志、邮件通知、轻量任务；生产 AI 推理任务通常需要外部队列与独立 worker。

③ 全套代码逐行精读解析

@router.post(
    "/summaries",
    response_model=SummaryTaskRead,
    status_code=status.HTTP_202_ACCEPTED,
    summary="Create a summary task from raw text",
)
def create_summary_task(
    payload: SummaryRequest,
    background_tasks: BackgroundTasks,
    session: Annotated[Session, Depends(get_session)],
) -> SummaryTask:
    task = create_summary_task_record(
        session=session,
        text=payload.text,
    )

    background_tasks.add_task(process_summary_task, task.id)
    return task
  • @router.post("/summaries")：注册 POST /tasks/summaries，因为路由器本身带有 prefix="/tasks"。
  • response_model=SummaryTaskRead：输出只暴露 id/text_length/source_file_id/status/result/error，不泄漏完整 text。
  • status_code=202：明确告诉客户端“已接收，稍后查询结果”。如果写成 200，语义会误导调用方以为摘要已经完成。
  • payload: SummaryRequest：Pydantic 先校验文本长度，短文本或超长文本不会进入业务函数。
  • background_tasks: BackgroundTasks：由 FastAPI 注入。不要手动实例化，否则不会挂到当前响应生命周期。
  • session: Depends(get_session)：请求级数据库会话，只用于创建任务记录，不传给后台函数。
  • create_summary_task_record(...)：服务层完成 add/commit/refresh，确保任务拥有数据库生成的 id。
  • add_task(process_summary_task, task.id)：只传 task.id，不传 ORM 对象，避免后台阶段使用已过期的请求 Session 对象。
  • return task：响应序列化时按 SummaryTaskRead 过滤字段。
def create_summary_task(session: Session, text: str, source_file_id: int | None = None) -> SummaryTask:
    task = SummaryTask(text=text, text_length=len(text), source_file_id=source_file_id, status="queued")
    session.add(task)
    session.commit()
    session.refresh(task)
    return task

commit() 之前任务只是 Python 对象；refresh() 之后才能稳定拿到数据库生成的主键。不这么做，后台函数可能拿到 None 或无法查询到任务。

④ 核心机制运行全流程拆解

客户端 POST /tasks/summaries
  -> require_api_key 校验 X-API-Key
  -> SummaryRequest 校验 text 长度
  -> get_session 创建请求级 Session
  -> create_summary_task_record 插入 queued 任务并提交
  -> add_task 注册 process_summary_task(task.id)
  -> 返回 202 + SummaryTaskRead
  -> 响应发送后执行 process_summary_task
  -> worker 新建 Session，读取 task_id
  -> run_summary_task: running -> completed / failed
  -> 客户端 GET /tasks/{id} 查询最终状态

核心数据流是“请求体文本 → 数据库任务记录 → 后台 worker 通过主键重新读取 → 更新状态与结果”。调用链中最容易漏掉的是 Session 生命周期：请求函数中的 Session 在响应完成后就不应继续复用。

⑤ 重难点、易混淆点对比辨析

概念本质常见误区
BackgroundTasks响应后同进程回调误以为是持久化任务队列
async/await协程等待模型误以为等于后台任务
202 Accepted任务已接收但未完成误写 200，导致客户端按同步结果处理
任务记录可查询的状态资源只返回“ok”而不保存 task_id，后续无法追踪
传 task.id跨生命周期的稳定标识把 ORM 实例传给后台函数，造成 Session 失效隐患

⑥ 开发常见报错+坑点+解决方案

  • 401 Unauthorized：缺少 X-API-Key 或值不等于配置。解决：请求头加 X-API-Key: dev-secret-key。
  • 422 Unprocessable Content：text 长度小于 10 或超过 2000。解决：按 SummaryRequest 约束构造 JSON。
  • 任务一直 queued：后台函数未执行或进程中断。解决：确认响应是否完成、服务是否重启、worker 函数是否抛异常。
  • 后台函数查不到任务：未提交事务就注册任务，或传递了错误 id。解决：保持 commit() 后再 add_task。
  • SQLite Session 错用：把请求 Session 传进后台函数。解决：后台函数内部新建 Session(engine)。

⑧ 课后记忆习题+标准答案

1. 为什么创建摘要任务返回 202 而不是 200？

因为请求只是被接受并排入后台处理，摘要结果尚未保证完成；202 精确表达异步处理语义。

2. 为什么后台函数只接收 task.id？

主键是跨 Session、跨生命周期稳定的标识；ORM 对象绑定请求 Session，响应结束后继续使用有失效风险。

3. BackgroundTasks 的最大生产边界是什么？

它不是持久化队列，应用重启或进程退出可能丢任务，也没有内建重试、监控和 worker 扩容。

⑨ 面试高频真题+解析

题 1：如何设计一个耗时 AI 摘要接口，避免 HTTP 请求超时？

用提交任务接口返回 202 与 task_id，把输入持久化为任务记录；后台 worker 处理任务并更新状态；客户端轮询或订阅状态接口。教学项目可用 BackgroundTasks，生产项目应使用持久队列。

题 2：后台任务为什么不能直接复用请求中的数据库 Session？

请求 Session 的生命周期由依赖注入管理，响应结束后应关闭；后台任务晚于响应执行，复用会导致连接状态不可控、事务边界混乱和对象过期问题。

题 3：BackgroundTasks 与消息队列的区别是什么？

BackgroundTasks 是应用内响应后回调；消息队列是独立基础设施，提供消息持久化、消费者解耦、重试、死信和水平扩展能力。
