0063 Provider Metrics Scenario Fixtures：用固定样本训练监控排障判断

上一节已经把 provider metrics 整理成 runbook。本节继续补齐“可练习数据”：新增 build_provider_metrics_runbook_scenario_samples()，用一个无副作用纯函数生成四类核心 outcome 样本，使学习者不用真的打出 503、400、重试耗尽等路径，也能对照 runbook 训练排障判断。

├─ 问题背景
│  ├─ runbook 有步骤，但初学者缺少稳定样本练习
│  ├─ 真实触发 retry/fail-fast 路径需要 MockTransport 或外部 provider 状态
│  └─ 固定 scenario fixtures 可以把排障训练从运行偶然性中解耦

│  ├─ success count=3

② 核心概念底层原理

fixture 是测试或练习中使用的稳定输入数据。它的价值不在于模拟世界全部复杂性，而在于把某个知识点所需的最小状态固定下来。对于 metrics runbook，最小状态不是一个随机数字，而是一组带有稳定 label 的样本：成功、安排重试、重试耗尽、快速失败。

scenario fixture 与普通 mock 不同。mock 常用于替换外部依赖行为，例如让 provider 返回 503；scenario fixture 则直接提供已经形成的观测样本，重点训练“看到这些 label 后如何判断”。因此本节不新增 endpoint，也不污染全局 counter，只新增一个纯函数。

概念目的本节选择
MockTransport模拟 provider HTTP 响应适合验证运行链路，不适合每次手动练 runbook
全局 counter记录真实请求过程产生的指标用于运行时，不应被练习样本污染
Scenario Fixture提供可重复、可解释的观测样本本节新增纯函数生成固定样本
Snapshot某一时刻指标读数fixture 输出保持与 snapshot 相同 shape
Runbook把观测样本映射为排障动作fixture 是 runbook 的练习数据

底层设计原则是观测 shape 复用、状态来源隔离。fixture 使用真实的 build_provider_metric_labels() 构造 label，确保与生产代码的 label contract 一致；但 fixture 不调用 record_provider_metric()，避免把教学练习样本混入真实 counter。

③ 全套代码逐行精读解析

def build_provider_metrics_runbook_scenario_samples() -> list[dict[str, object]]:
    return [
        {
            "labels": build_provider_metric_labels(outcome="success").as_dict(),
            "count": 3,
        },
        {
            "labels": build_provider_metric_labels(
                outcome="retry_scheduled",
                failure_category="retryable_http_status",
                error_code="ai_provider_http_error",
                status_code=503,
            ).as_dict(),
            "count": 2,
        },
        {
            "labels": build_provider_metric_labels(
                outcome="retry_exhausted",
                failure_category="retryable_http_status",
                error_code="ai_provider_http_error",
                status_code=503,
            ).as_dict(),
            "count": 1,
        },
        {
            "labels": build_provider_metric_labels(
                outcome="fail_fast",
                failure_category="non_retryable_http_status",
                error_code="ai_provider_http_error",
                status_code=400,
            ).as_dict(),
            "count": 1,
        },
    ]
代码行作用为什么这样写 / 不这样写的隐患
def build_provider_metrics_runbook_scenario_samples()声明场景样本构造函数用 build 前缀表达“生成数据”，不是读取当前运行状态。
list[dict[str, object]]返回与 metrics snapshot 兼容的样本列表保持 lesson、测试、text renderer 能复用同一 shape。
return [...]直接返回新列表每次调用生成独立对象，避免调用方修改样本后影响后续练习。
build_provider_metric_labels(...)复用真实 label builder如果手写 label dict，容易漏字段或写错低基数契约。
.as_dict()转成 endpoint snapshot 使用的字典形状保持与 /provider/metrics 返回结构一致。
count=3构造成功基线成功样本比失败样本多，方便练习“失败存在但系统仍可能有成功请求”的判断。
retry_scheduled 503构造可重试失败已安排重试用于训练“retry_scheduled 不等于最终失败”。
retry_exhausted 503构造同一状态码的重试耗尽用于比较 scheduled 与 exhausted 的语义差异。
fail_fast 400构造非重试型客户端错误用于训练“不应靠增加重试次数解决 fail_fast”。
 
def test_provider_metrics_runbook_scenario_samples_cover_core_outcomes() -> None:
    samples = build_provider_metrics_runbook_scenario_samples()

    assert {sample["labels"]["outcome"] for sample in samples} == {
        "success",
        "retry_scheduled",
        "retry_exhausted",
        "fail_fast",
    }

这个测试保护样本覆盖范围。若未来有人删掉 retry_exhausted 或 fail_fast，runbook 练习数据就会丢失关键分支，测试会立即失败。

def test_provider_metrics_runbook_scenario_samples_render_as_text() -> None:
    text = render_provider_metrics_prometheus_text(
        build_provider_metrics_runbook_scenario_samples()
    )

    assert "provider_prediction_total" in text
    assert 'outcome="retry_scheduled"' in text
    assert 'outcome="retry_exhausted"' in text
    assert 'outcome="fail_fast"' in text

实际测试使用完整字符串断言，确保场景样本不仅 JSON shape 正确，也能被现有 Prometheus text renderer 渲染。这样 fixture 与前面课程的 export shape、export tests、runbook 形成闭环。

④ 核心机制运行全流程拆解

学习者想练 runbook
  → 调用 build_provider_metrics_runbook_scenario_samples()
    → 函数生成四个固定 sample
      → 每个 sample 复用 build_provider_metric_labels()
      → 每个 labels 调用 as_dict()
      → 每个 sample 包含 labels + count
  → 学习者按 runbook 读取样本
    → 先按 outcome 分组
    → 再读 failure_category/error_code/status_code
    → 判断 retry_scheduled 与 retry_exhausted 差异
    → 判断 fail_fast 不应盲目重试
  → 如需练 text export
    → render_provider_metrics_prometheus_text(samples)
    → 得到 provider_prediction_total 文本样本

这条流程刻意绕开真实 provider 请求。真实请求链路用于验证代码正确性；scenario fixture 用于训练观察与判断。两者共享 label builder，所以语义一致；两者分离状态来源，所以不会互相污染。

阅读阶段samples按 outcome 分类排障分支入口
判断阶段failure labels对照 runbook下一步动作
导出阶段samplestext rendererPrometheus text-style 样本
验证阶段函数输出pytest 断言稳定练习数据

⑧ 课后记忆习题+标准答案

1. 为什么本节使用纯函数生成 scenario samples？

纯函数没有副作用，不会污染全局 metrics counter；每次调用都得到稳定练习数据，适合课程、测试和 runbook 对照。

2. 为什么 fixture 要复用 build_provider_metric_labels()？

这样可以复用真实 metrics label contract，避免手写 dict 导致字段缺失、默认值不一致或低基数约束被破坏。

3. 为什么样本同时包含 retry_scheduled 和 retry_exhausted？

两者语义不同：scheduled 表示安排重试，不等于最终失败；exhausted 表示最大尝试次数用完。放在同一练习集中可以训练对比判断。

4. 为什么本节不新增 /provider/metrics/scenarios endpoint？

因为场景样本是课程练习数据，不是运行时公共 API。过早暴露 endpoint 会扩大外部契约面，并可能让读者误以为它是生产监控数据。

5. 如何验证 scenario samples 与文本导出兼容？

把 fixture 输出传入 render_provider_metrics_prometheus_text()，断言输出包含稳定 HELP/TYPE 行和四类 outcome 的文本样本。

⑨ 面试高频真题+解析

面试题 1：测试 fixture 的设计原则是什么？

fixture 应稳定、最小、语义明确、与真实契约复用、无不必要副作用。它不追求模拟全部真实世界，而是固定某个测试或学习目标所需的关键状态。

面试题 2：为什么练习样本不应该写入全局 metrics counter？

全局 counter 表示运行过程中真实发生的事件。练习样本若写入 counter，会污染 endpoint 返回值，使测试顺序相关，也会混淆教学数据和运行数据。

面试题 3：如何让 fixture 与生产代码契约保持一致？

复用生产代码中的 builder、dataclass 或 schema，而不是复制字段字符串；同时用测试锁定输出 shape 和关键语义。如果契约变更，fixture 测试会引导同步更新。

面试题 4：scenario fixture 与端到端测试分别适合验证什么？

scenario fixture 适合验证固定观测样本的解释、渲染和文档训练；端到端测试适合验证真实请求经过 router、adapter、retry、counter 到 endpoint 的完整链路。两者互补，不应互相替代。
