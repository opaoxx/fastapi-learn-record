0080 GitHub CODEOWNERS：把文件审查责任写成可执行所有权规则

本节新增 .github/CODEOWNERS，把后端源码、测试、课程讲义、速查页、学习记录、根文档和 GitHub 协作配置映射到对应 owner。PR template 解决“贡献者应说明什么”，CODEOWNERS 解决“谁应该审查什么”。

② 核心概念底层原理

CODEOWNERS 是 GitHub 的路径到审查责任人的映射文件。创建 PR 后，GitHub 会根据变更文件路径匹配 CODEOWNERS 规则，并自动请求匹配 owner 审查。它把“这类文件该找谁看”从口头经验变成仓库级协作规则。

匹配机制近似路径模式匹配：左侧是文件或目录 pattern，右侧是一个或多个 GitHub 用户或团队。更具体的路径规则通常用于覆盖宽泛规则。本节先用 * 给全仓库设置默认课程维护者，再对 first_api/、tests/、.github/ 等关键区域指定更合适的 owner。

CODEOWNERS 不是权限系统本身。它不会自动让 owner 拥有仓库权限，也不会让占位用户名生效。生产仓库必须把 @course-maintainers 等占位符替换成真实 GitHub 用户或组织 team，并在 branch protection 中启用 code owner review required，才能把 ownership 变成合并门禁。

文件区域审查关注点推荐 owner 类型
first_api/FastAPI 行为、服务边界、schema、依赖注入。后端维护者。
tests/回归覆盖、契约测试、测试隔离。质量维护者。
lessons/九段结构、代码精读、课程深度。课程维护者。
reference/速查准确性、导航一致性。课程维护者。
.github/CI、PR 模板、分支保护、owner 规则。仓库治理维护者。

③ 全套代码逐行精读解析

# CODEOWNERS teaching template.

首行说明这是教学模板。原因是本地仓库无法知道你 GitHub 组织里的真实用户名和 team slug。直接写占位 owner 时，必须在课程中说明生产使用前要替换。

# Replace these placeholder teams with real GitHub users or organization teams
# before relying on automatic review requests.

这两行是关键安全说明。若 owner 不存在或没有仓库权限，GitHub 不会产生有效审查请求。教学模板可以保存结构，生产落地必须替换成真实 owner。

* @course-maintainers

* 是全仓库兜底规则。任何没有被更具体规则覆盖的文件，默认由课程维护者关注。没有兜底规则时，新目录可能完全没有 owner。

/first_api/ @backend-maintainers

/first_api/ 匹配后端示例项目。该目录包含 FastAPI app、router、schema、service、database 和 frontend 静态资源，改动可能影响运行行为，应由后端 owner 审查。

/tests/ @quality-maintainers

/tests/ 由质量维护者负责。测试文件不仅验证代码，还保护 OpenAPI、文档结构、CI、README 和 GitHub 协作规则。测试变更可能降低质量门槛，因此不能只由功能作者自审。

/lessons/ @course-maintainers
/reference/ @course-maintainers
/learning-records/ @course-maintainers

课程内容由课程维护者负责。lesson 决定深度，reference 决定复习效率，learning-records 决定进度追踪。三者缺一都会破坏 GitHub 学习体验。

/README.md @repository-maintainers
/DEVELOPMENT.md @repository-maintainers

根入口文档由仓库维护者审查。README 是外部学习者第一入口，DEVELOPMENT 是贡献者维护入口；任何错误命令或断链都会影响新人上手。

/.github/ @repository-maintainers

.github/ 是协作治理目录，包含 CI、branch protection 文档、PR template 和 CODEOWNERS。该目录变更会影响整个协作流程，应由仓库治理 owner 审查。

def test_codeowners_maps_repository_ownership() -> None:
    owners = Path(".github/CODEOWNERS").read_text(encoding="utf-8")

文档契约测试读取 CODEOWNERS。这里不验证 GitHub 服务器是否启用 code owner review，因为那属于远程仓库设置；本地测试保护的是 owner 映射文件必须存在并保留核心路径。

④ 核心机制运行全流程拆解

Contributor opens PR
  -> GitHub computes changed files
  -> GitHub reads .github/CODEOWNERS
  -> Match each changed path against owner rules
  -> Request reviews from matched owners
  -> CI runs required checks
  -> Branch protection evaluates:
       required status check passed?
       required code owner review approved?
       conversations resolved?
  -> merge allowed or blocked

CODEOWNERS 处在“人类审查路由”层。它不决定测试是否通过，也不解释变更意图；它根据文件路径把 PR 送到合适 reviewer 面前。复杂项目中，这是避免错误 owner 审查关键文件的基础设施。

输入处理输出
PR changed files路径匹配 CODEOWNERS pattern。匹配 owner 列表。
owner 列表GitHub 发起 review request。对应团队或用户进入 review 流程。
branch protection检查 code owner review 是否满足。允许或阻断合并。

⑧ 课后记忆习题+标准答案

1. CODEOWNERS 解决什么问题？

它把文件路径映射到负责审查的用户或团队，让 PR 自动请求合适 owner review，避免关键文件无人或错人审查。

2. CODEOWNERS 会自动授予仓库权限吗？

不会。它只定义审查请求规则。owner 必须已经是有仓库访问权限的真实 GitHub 用户或团队。

3. 为什么本节文件里要保留 * @course-maintainers？

这是兜底规则，保证新增或未细分路径仍有默认课程维护者负责。

4. CODEOWNERS 如何和 branch protection 联动？

GitHub 先根据 CODEOWNERS 请求 owner review；branch protection 若启用 required code owner review，则没有 owner 批准的 PR 不能合并。

5. 为什么 .github/ 目录需要专门 owner？

因为它控制 CI、PR 模板、分支保护说明和 CODEOWNERS，本质上影响整个仓库协作治理。

⑨ 面试高频真题+解析

面试题 1：CODEOWNERS、reviewer、approver 有什么区别？

CODEOWNERS 是路径到责任人的规则；reviewer 是被请求审查的人；approver 是实际批准 PR 的人。开启 code owner required review 后，匹配 owner 的批准才满足该门禁。

面试题 2：为什么大仓库需要 CODEOWNERS？

大仓库中模块多、风险边界不同，普通 reviewer 很难覆盖全部上下文。CODEOWNERS 能把数据库、安全、前端、后端、文档等改动路由到懂该区域的人。

面试题 3：为什么 CODEOWNERS 不能替代测试？

owner review 关注设计合理性、风险和领域知识；测试负责可重复验证行为。人类审查不能稳定覆盖所有回归路径。

面试题 4：如何设计课程仓库的 owner 边界？

应围绕交付物划分：后端行为、测试契约、课程讲义、速查页、学习记录、根文档和 GitHub 治理文件。每类文件的审查重点不同。

面试题 5：CODEOWNERS 文件中的占位团队有什么风险？

如果占位 team 不存在或无权限，GitHub 不会产生有效审查请求。生产使用前必须替换为真实 GitHub 用户或组织 team。
