② 核心概念底层原理

最终质量审计不是普通待办清单，而是仓库级验收协议。协议的价值在于把隐含经验转成显式规则：课程是否完整、链接是否闭合、测试是否覆盖、发布是否可重复、协作规则是否一致，都必须能被人或测试复核。

课程仓库不同于普通代码仓库。普通代码仓库主要保护运行行为；课程仓库还要保护知识结构、叙述顺序、学习入口、导航路径和示例一致性。第 93 课的核心就是把“文档质量”提升为与代码质量同级的公共接口。

.github/FINAL_AUDIT.md 放在 .github/ 下，说明它属于 GitHub 协作和发布治理层；它不会替代 README.md，也不会替代 CHANGELOG.md。README 面向首次进入仓库的人，CHANGELOG 记录变更，FINAL_AUDIT 负责发布前验收。

文档核心职责错误使用
README.md学习入口、项目介绍、运行方式堆放每次执行日志
DEVELOPMENT.md开发、课程维护、测试命令替代代码注释或任务进度
CHANGELOG.md面向发布的变更账本记录每一步临时操作
COURSE_PROGRESS.md当前运行批次证据和交接记录当作对外正式发布说明
.github/FINAL_AUDIT.md完结发布前质量验收门禁替代自动化测试

底层设计思想是“单一事实源 + 多层校验”。编号完整性由文件系统和测试保护；九段式结构由文档契约测试保护；源码行为由 pytest 保护；公开发布边界由 README、PAGES、RELEASE_PROCESS 和 FINAL_AUDIT 共同约束。

③ 全套代码逐行精读解析

# Repository Final Quality Audit

标题定义该文件的唯一职责：仓库最终质量审计。使用 Repository 而不是 Lesson，说明审计对象不是单节课，而是课程、源码、测试、文档和 GitHub 治理的整体一致性。

This checklist is the final pre-release gate for the 96-lesson FastAPI course repository.

这行把审计绑定到 96 节课目标。若不写明确范围，审核者无法判断“完成”到底指当前已有文件，还是最终课程全量。

It is not a substitute for tests; it defines what reviewers must verify before the course is tagged, published, or promoted through GitHub Pages.

这行划清审计与测试边界。审计清单负责人类判断和发布语义，测试负责机器可验证事实。不这样写会造成“有清单就不用跑测试”的危险误解。

## Audit Purpose

Audit Purpose 是审计动机层，先回答为什么存在该文件。发布前文档如果直接进入命令列表，新手会执行命令但不知道命令保护什么。

- Confirm that lessons 0001-0096 form one complete learning path.

该行保护课程编号和学习路径。编号断层会导致目录、导航、学习记录和用户预期同时失效。

- Confirm that every lesson keeps the required review structure.

该行绑定用户要求的九段式结构。没有这一项，后续课程容易退化成浅层网课式讲解。

- Confirm that runnable code, tests, lessons, references, learning records, and GitHub governance documents agree with each other.

这行是仓库一致性核心。课程材料不是孤立 HTML，必须与真实 first_api/ 行为、测试、速查表和治理文档同步。

- Confirm that the public repository does not expose secrets, private provider URLs, personal data, or customer data.

该行建立公开仓库安全边界。教学项目也可能因示例、日志、截图、复制粘贴而泄漏真实密钥，所以必须作为发布门禁。

## Lesson Coverage

Lesson Coverage 负责课程完整性。它不是检查内容是否“看起来很多”，而是检查编号、文件、引用、结构和固定标记是否全部存在。

- `lessons/` contains lessons 0001-0096 with no numbering gaps.

编号无断层是目录可预测性的前提。若缺少 0094，下一课链接和学习计划都会产生断裂。

- `reference/` contains the matching quick reference for every protected lesson.

速查表是复习入口。没有匹配 reference，课程就缺少期末复习场景下的快速回忆材料。

- Each lesson contains the nine-section structure:

这一行把结构要求转成审核条款。九段式不是排版偏好，而是知识学习闭环：框架、原理、代码、流程、辨析、故障、生产差异、习题、面试。

- Each lesson keeps the marker `Final Review + CSDN Deep Dive`.

固定标记用于测试和人工扫描。若某节课缺少该标记，说明它可能没有被纳入最终复习风格。

## Navigation Integrity

Navigation Integrity 保护静态站点可读性。HTML 课程没有框架路由，所有跳转都依赖显式 <a href>，任何漏改都会成为断链。

- Top and bottom lesson navigation return to `index.html`.

顶部和底部导航都要能回目录。长文阅读时用户可能在任意位置结束，底部导航能降低回退成本。

- Previous/next lesson links are correct for adjacent lessons.

相邻课程链接保护线性学习路径。只新增新课而不更新上一课，会让第 92 课无法自然进入第 93 课。

## Source and Test Integrity

Source and Test Integrity 绑定“讲义”和“项目源码”。课程说的接口、配置、错误码、测试命令，如果不与真实代码一致，就会误导读者。

- Documentation behavior described in lessons is protected by `tests/test_course_docs_contract.py` when it is a durable repository contract.

这行解释文档契约测试的职责：保护结构、链接、公共文档内容和发布约定。它不是单元测试，但与单元测试一样承担回归防线。

## Governance Documents

Governance Documents 列出公开协作所需文件。GitHub 推广面向陌生学习者时，行为规范、贡献流程、安全报告、许可证和发布流程都是信任基础设施。

## No Secrets

No Secrets 是公开仓库底线。教学默认值可以提交，真实密钥、生产 URL、客户数据和私人日志不能提交。

## Required Verification

Required Verification 把审计落到命令层。没有命令的清单容易停留在主观判断；命令让发布前状态可复现。

.\.venv\Scripts\python.exe -m pytest tests\test_course_docs_contract.py -q
.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q
.\.venv\Scripts\python.exe -m pytest -q
git diff --check

四条命令分别保护文档契约、Provider HTTP 行为、全项目回归和空白格式。若跳过 docs contract，最容易漏掉断链和九段式退化；若跳过 full pytest，可能只保护了文档却破坏了 API。

## Release Decision

Release Decision 定义通过条件。只有测试通过、最新链路一致、无秘密、Pages 边界准确、已知警告记录清楚，才能进入 tag 或 GitHub Release。

④ 核心机制运行全流程拆解

maintainer finishes a lesson
  -> create lesson HTML
  -> create quick reference
  -> create learning record
  -> update previous lesson/reference navigation
  -> update index.html
  -> update README / DEVELOPMENT / CHANGELOG / NOTES when public contract changes
  -> add docs contract coverage
  -> run targeted docs tests
  -> run provider tests
  -> run full pytest
  -> run git diff --check
  -> update COURSE_PROGRESS with evidence
  -> final audit before release/tag/Pages promotion

这条流程体现“每节课都是一个可发布切片”。新增 HTML 只是第一步，真正完成必须包含速查、目录、相邻导航、学习记录、测试契约和进度证据。

审计层输入输出失败信号
文件层lessons/、reference/编号连续、文件存在缺号、重名、速查缺失
结构层HTML 文本九段式 + 固定标记缺章节、浅层讲解、无代码精读
导航层<a href>目录、上一课、下一课、速查闭合404、无法继续学习
源码层first_api/、tests/讲义与真实行为一致课程代码过时、测试不匹配
发布层README、Pages、Release docs可读、可复查、可分享GitHub 上只看到源码、发布后断样式

⑧ 课后记忆习题+标准答案

1. 为什么最终审计不能替代自动化测试？

审计清单定义发布前需要验证的事实和人工判断，自动化测试提供可重复的机器证据。清单没有执行能力，测试没有完整发布语义，两者必须共同存在。

2. 课程完结时为什么要检查 lessons 0001-0096 无编号断层？

编号是学习顺序、目录导航、速查映射和发布说明的公共索引。断层会破坏线性学习路径，也会让用户误以为缺课。

3. FINAL_AUDIT 与 README 的职责差异是什么？

README 面向首次进入仓库的学习者，提供入口和运行方式；FINAL_AUDIT 面向发布审核者，定义完结前的质量验收门禁。

4. 发布前至少应运行哪些命令？

运行 tests/test_course_docs_contract.py、tests/test_provider_http.py、全量 pytest 和 git diff --check。

5. 为什么公开教学仓库也必须检查 No Secrets？

教学仓库会被复制、Fork、搜索和长期缓存。一旦提交真实密钥、生产 URL 或客户数据，影响不低于生产代码仓库泄漏。

⑨ 面试高频真题+解析

面试题 1：如何设计一个课程仓库的发布前质量门禁？

应分层设计：文件完整性、文档结构、导航链路、源码行为、测试覆盖、公开文档、无密钥检查和发布说明。可机器验证的内容进入 CI 或 pytest，需人工判断的内容进入审计清单。

面试题 2：为什么文档也需要契约测试？

当文档承载公共入口、API 说明、发布流程或课程结构时，它就是用户依赖的接口。契约测试能防止断链、缺章节、旧命令和治理文档不一致。

面试题 3：教学 Demo 项目的安全审计重点是什么？

重点不是复杂攻防，而是防止真实密钥、私有 URL、私人日志、个人数据、客户样本和危险生产实践进入公开材料，并明确教学默认值与生产配置的差异。

面试题 4：CHANGELOG、Release Notes、COURSE_PROGRESS 如何分工？

CHANGELOG 是版本前后的可读变更账本；Release Notes 是某个 tag 的对外发布摘要；COURSE_PROGRESS 是本轮执行过程、测试结果和下一步交接证据。

面试题 5：如何判断一个静态课程站点是否可发布？

检查入口存在、所有 lesson/reference 文件可达、CSS 路径正确、上一课/下一课链路闭合、GitHub Pages 发布源正确、文档契约测试通过、无敏感信息。
