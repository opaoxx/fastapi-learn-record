Lesson 0029

OpenAPI 作为 API 契约：自动文档背后的生成机制

第 27、28 课让浏览器页面调用 GET /tasks。这一节换一个更工程化的角度：前端、后端、测试和未来 AI 调用方到底如何确认接口“应该长什么样”？答案不是口头约定，而是可读取、可测试的 OpenAPI 契约。

0. 本节学习目标

  • 理解 /docs 和 /openapi.json 的关系。
  • 知道 FastAPI 从哪些代码信息生成 OpenAPI：路由、类型注解、Pydantic/SQLModel 模型、参数声明、状态码、安全依赖。
  • 能读懂 GET /tasks 在 OpenAPI 中的参数、响应模型和鉴权方案。
  • 学会给接口补充 summary、description、response_description 和 tag 元信息。
  • 用自动化测试守住关键契约，避免以后不小心改坏前端依赖的 JSON 形状。

1. 背景：自动文档不是装饰，是协作协议

很多初学者第一次打开 http://127.0.0.1:8000/docs 时，会把它理解成“FastAPI 送了一个漂亮页面”。这只说对了一半。

真正重要的是 /openapi.json。它是一份机器可读的 JSON 文档，描述 API 有哪些路径、每个路径支持什么方法、参数在哪里、请求体是什么模型、成功响应是什么结构、可能有哪些错误响应、是否需要安全方案。

/docs 只是把这份 JSON 渲染成人能操作的 Swagger UI。换句话说：

Python 代码
  -> FastAPI 收集路由、模型、参数、安全依赖
  -> 生成 /openapi.json
  -> /docs 把 /openapi.json 渲染成交互式页面

对真实项目来说，这份契约可以服务四类人：

  • 后端：确认自己暴露了什么接口。
  • 前端：确认应该传什么参数、读取什么响应字段。
  • 测试：把关键契约写成自动化断言。
  • 未来的 AI 服务调用方：不用读后端源码，也能知道任务接口如何调用。

2. 核心概念：契约不是实现，契约是边界

实现是服务内部怎么做事，比如 SQL 怎么查、后台任务怎么运行、AI 客户端怎么生成摘要。契约是外部调用者能依赖的边界，比如：

GET /tasks?status=completed&limit=5&offset=0

这个接口的契约至少包含：

  • status 是可选查询参数，只能是 queued/running/completed/failed。
  • limit 是查询参数，最小 1，最大 100，默认 20。
  • offset 是查询参数，最小 0，默认 0。
  • 成功响应是 SummaryTaskListResponse，里面有 items/count/limit/offset。
  • 接口需要 X-API-Key 请求头。

前端第 28 课的分页按钮完全依赖这些事实。如果后端突然把 count 改名为 total，前端分页就会坏。这就是为什么 API 契约必须被认真对待。

3. FastAPI 如何生成 OpenAPI

FastAPI 的 OpenAPI 不是手写出来的。它来自你已经写在代码里的结构化信息：

代码来源 OpenAPI 中体现为什么 本项目例子
路径操作装饰器 路径、HTTP 方法、状态码、summary、description @router.get("", response_model=SummaryTaskListResponse)
函数参数和 Query 查询参数、默认值、最小值、最大值、说明 limit: Annotated[int, Query(ge=1, le=100)] = 20
Path 路径参数、约束和说明 task_id: Annotated[int, Path(ge=1)]
Pydantic/SQLModel 模型 请求体和响应体 JSON Schema SummaryTaskListResponse
response_model 成功响应的 schema response_model=SummaryTaskRead
APIKeyHeader 依赖 安全方案和接口 security 要求 X-API-Key

官方文档也强调了这些来源：应用级 title/summary/description/version 会进入 OpenAPI 和自动文档；路径操作可以配置 summary、响应描述等元信息；Query 的校验会同时参与运行时校验和文档生成；response_model 会用于文档、校验、序列化和过滤。

4. 运行机制：一次请求文档时发生了什么

  1. 浏览器访问 /openapi.json。
  2. FastAPI 遍历应用里注册过的 path operation。
  3. 它读取每个 operation 的路径、HTTP 方法、tag、summary、description、状态码。
  4. 它读取函数签名，把 Query、Path、请求体模型转换为参数 schema。
  5. 它读取 response_model，生成响应 JSON Schema 引用。
  6. 它读取安全依赖，例如 APIKeyHeader，生成 components.securitySchemes。
  7. 返回一份完整 JSON。Swagger UI 再用这份 JSON 渲染 /docs 页面。

这也是 FastAPI 教学里特别值得重视的地方：你写类型注解和模型，不只是为了编辑器提示，也是为了让接口契约能被机器读取。

6. 代码精读一：应用级元信息

openapi_tags = [
    {
        "name": "system",
        "description": "Health checks and public runtime configuration.",
    },
    {
        "name": "tasks",
        "description": "Protected background summary tasks with status, filtering, and pagination.",
    },
]

app = FastAPI(
    title=settings.app_name,
    summary="A learning API that grows from backend basics into AI-service endpoints.",
    description="A tiny learning API for backend basics and future AI service endpoints.",
    version="0.1.0",
    openapi_tags=openapi_tags,
    lifespan=lifespan,
)

title、summary、description 和 version 是 API 级别的信息，会出现在 /openapi.json 的 info 区域，也会显示在 /docs 顶部。

openapi_tags 则是给接口分组写说明。以前只有 tags=["tasks"]，文档知道“这个接口属于 tasks 组”，但不知道这个组是干什么的。现在 tag 有了描述，读者更容易理解系统边界。

7. 代码精读二：安全方案也属于契约

api_key_header = APIKeyHeader(
    name="X-API-Key",
    scheme_name="ApiKeyAuth",
    description="Local learning API key required by protected write, file, and task endpoints.",
    auto_error=False,
)

name="X-API-Key" 表示客户端必须把 key 放在请求头里。scheme_name="ApiKeyAuth" 是 OpenAPI 里的安全方案名称。这个名字会进入：

"securitySchemes": {
  "ApiKeyAuth": {
    "type": "apiKey",
    "in": "header",
    "name": "X-API-Key"
  }
}

这很重要。前端、测试工具、甚至未来的客户端生成器都能从契约里知道：调用受保护接口时，不是把 key 放进 query，也不是放进 body，而是放到 X-API-Key 请求头。

auto_error=False 仍然保留。它让我们可以在自己的 require_api_key 函数里统一返回项目想要的错误消息。

8. 代码精读三：任务列表接口的契约说明

@router.get(
    "",
    response_model=SummaryTaskListResponse,
    summary="List summary tasks",
    description=(
        "Returns a newest-first page of summary tasks. Clients can filter by task status "
        "and use limit/offset pagination."
    ),
    response_description="A paginated task-list response envelope.",
)
def list_summary_tasks(
    session: Annotated[Session, Depends(get_session)],
    task_status: Annotated[
        TaskStatus | None,
        Query(alias="status", description="Optional status filter for the task list."),
    ] = None,
    offset: Annotated[int, Query(ge=0, description="Zero-based number of matching tasks to skip.")] = 0,
    limit: Annotated[int, Query(ge=1, le=100, description="Maximum number of tasks to return.")] = 20,
) -> SummaryTaskListResponse:

这里最值得慢慢看的是四层信息：

  • response_model=SummaryTaskListResponse：告诉调用者成功响应不是裸数组，而是分页响应信封。
  • summary：一句话说明接口用途，适合文档列表快速扫描。
  • description：解释接口行为，比如“newest-first”“status filter”“limit/offset pagination”。
  • Query(...)：同时提供运行时校验和文档信息。

注意 response_description 不是接口整体描述，而是 200 响应的描述。FastAPI 官方文档也区分了 description 和 response_description 的用途。

9. 代码精读四：响应模型如何变成 JSON Schema

class SummaryTaskListResponse(BaseModel):
    items: list[SummaryTaskRead]
    count: int
    limit: int
    offset: int

这段模型会进入 OpenAPI 的 components.schemas。任务列表接口的 200 响应不会把所有字段重复写一遍，而是引用这个组件：

"responses": {
  "200": {
    "description": "A paginated task-list response envelope.",
    "content": {
      "application/json": {
        "schema": {
          "$ref": "#/components/schemas/SummaryTaskListResponse"
        }
      }
    }
  }
}

这就是 response_model 的工程价值。它不只是“返回时转一下 JSON”，它还是文档、测试和客户端理解响应结构的公共来源。

11. 课堂实验二：从 /docs 看同一份契约

打开：

http://127.0.0.1:8000/docs

找到 tasks 分组里的 GET /tasks，观察：

  1. 接口标题是否是 List summary tasks。
  2. status 是否显示为可选查询参数。
  3. limit 是否显示最小值 1、最大值 100。
  4. 右上角或锁形按钮是否能看到 API key 安全方案。
  5. 200 响应是否引用 SummaryTaskListResponse。

这一步的重点是建立直觉：/docs 页面上的多数内容都不是手工写出来的，而是从你的代码结构推导出来的。

12. 代码精读五：用测试守住契约

def test_openapi_documents_task_list_contract() -> None:
    with TestClient(app) as client:
        response = client.get("/openapi.json")

    schema = response.json()
    operation = schema["paths"]["/tasks"]["get"]

    assert operation["summary"] == "List summary tasks"
    assert operation["responses"]["200"]["content"]["application/json"]["schema"] == {
        "$ref": "#/components/schemas/SummaryTaskListResponse"
    }
    assert operation["security"] == [{"ApiKeyAuth": []}]

这个测试不是测试“文案好不好看”，而是在测试最关键的接口边界：

  • 任务列表接口还存在。
  • 它仍然返回 SummaryTaskListResponse。
  • 它仍然需要 API key。
  • limit 和 offset 的边界还在。
  • status 的枚举值还没有乱改。

以后如果有人把 response_model 改成 list[SummaryTaskRead]，这个测试就会失败。这种失败很有价值，因为它在提醒你：这不是简单重构，而是 API 契约变化。

13. 常见错误与诊断

错误一：把 /docs 当成唯一文档

/docs 是展示层；真正的机器可读契约是 /openapi.json。

错误二：忘记 response_model

没有响应模型时，文档很难稳定表达响应结构，前端也更容易依赖“碰巧返回”的字段。

错误三：只写类型，不写 Query 约束

int 只能说明类型，不能说明 limit 的最小值和最大值。

错误四：文档写得很漂亮，但测试不守契约

关键契约要进测试。否则文档和代码会慢慢漂移。

14. 面试视角：如何回答“FastAPI 自动文档怎么来的？”

可以这样答：

FastAPI 会根据 path operation、函数签名、类型注解、Pydantic 模型、参数声明和安全依赖生成 OpenAPI schema。/openapi.json 是机器可读的 API 契约，/docs 是基于它渲染出来的交互式文档。response_model 会影响响应的文档、校验、序列化和字段过滤，Query、Path 这类声明既做运行时校验，也会进入 OpenAPI 参数说明。

如果能再补一句项目经验，就更像真实工程师：

我会给关键接口写 OpenAPI 契约测试，比如确认分页接口仍然返回响应信封、查询参数约束没有丢、安全方案仍然要求 X-API-Key。

16. 小测：先回忆，再展开

1. /docs 和 /openapi.json 谁更接近真正的 API 契约？

/openapi.json。/docs 是把它渲染成交互式页面。

2. response_model 对 OpenAPI 有什么作用？

它告诉 FastAPI 成功响应的 schema，并参与文档、校验、序列化和字段过滤。

3. 为什么 Query(ge=1, le=100) 比只写 int 更适合契约？

因为它表达了参数边界，既能运行时校验，也能进入 OpenAPI 文档。

4. APIKeyHeader 在 OpenAPI 里会变成什么？

会变成 components.securitySchemes 里的 apiKey 安全方案，并标出 header 名称。

17. 本节总结

你现在应该把 FastAPI 自动文档理解成“由代码生成的 API 契约”，而不只是一个调接口页面。类型注解、Pydantic/SQLModel 模型、Query、Path、response_model、APIKeyHeader 都会参与契约生成。
