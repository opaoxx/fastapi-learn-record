OpenAPI 作为 API 契约：FastAPI 自动文档背后的生成机制

/docs 不是漂亮页面本身，真正的契约是 /openapi.json。FastAPI 会从路由、类型注解、Pydantic/SQLModel 模型、参数约束和安全依赖生成机器可读 API schema，让前端、测试、SDK 和未来 AI 调用方对齐同一份事实。

① 本节核心知识框架

OpenAPI 契约生成
├─ 代码来源
│  ├─ FastAPI(title, summary, description, version)
│  ├─ APIRouter(tags, prefix, dependencies)
│  ├─ path operation：summary/description/response_description
│  ├─ Query/Path：参数位置、默认值、边界
│  ├─ response_model：成功响应 schema
│  ├─ Pydantic/SQLModel：components.schemas
│  └─ APIKeyHeader：securitySchemes
├─ 输出
│  ├─ /openapi.json：机器可读契约
│  └─ /docs：Swagger UI 展示层
└─ 保护方式：tests/test_openapi_contract.py

② 核心概念底层原理

OpenAPI 是 HTTP API 的结构化描述格式。它描述路径、方法、参数、请求体、响应体、状态码、安全方案和 schema。FastAPI 的价值在于把 Python 类型系统和 Pydantic 模型转成 OpenAPI 文档，避免手写文档与代码长期漂移。

/openapi.json 是机器可读源，/docs 是基于它渲染出来的人机交互页面。前端、自动化测试、客户端生成器应该以 OpenAPI JSON 为可验证契约，而不是只靠口头说明。

契约与实现要区分：实现是 service 怎么查数据库、worker 怎么跑任务；契约是调用方能依赖的边界，例如 GET /tasks 需要 X-API-Key、支持 status/limit/offset、返回 SummaryTaskListResponse。

③ 全套代码逐行精读解析

openapi_tags = [
    {"name": "system", "description": "Health checks and public runtime configuration."},
    {"name": "tasks", "description": "Protected background summary tasks with status, filtering, and pagination."},
]
  • openapi_tags 给接口分组补描述；如果只有 tag 名字，读者知道分组但不知道边界。
  • tasks 描述中明确 protected、background、status、filtering、pagination，直接对应课程项目能力。
app = FastAPI(
    title=settings.app_name,
    summary="A learning API that grows from backend basics into AI-service endpoints.",
    description="A tiny learning API for backend basics and future AI service endpoints.",
    version="0.1.0",
    openapi_tags=openapi_tags,
    lifespan=lifespan,
)
  • title 进入 OpenAPI info.title，也是 Swagger UI 顶部标题。
  • summary/description 给 API 总体定位；不写会让文档像零散接口集合。
  • version 是契约版本信号，不等同代码包版本，但能帮助调用方识别 API 发布状态。
  • lifespan 与 OpenAPI 无直接关系，但体现应用启动/关闭资源管理仍在同一 app 对象上。
api_key_header = APIKeyHeader(
    name="X-API-Key",
    scheme_name="ApiKeyAuth",
    description="Local learning API key required by protected write, file, and task endpoints.",
    auto_error=False,
)
  • name="X-API-Key" 声明密钥在 header，不在 query/body。
  • scheme_name="ApiKeyAuth" 是 OpenAPI 安全方案名称，测试会保护它。
  • description 让文档解释该 key 保护哪些接口。
  • auto_error=False 允许项目自定义 401 错误；若设为 True，FastAPI 会自动抛默认错误。
@router.get(
    "",
    response_model=SummaryTaskListResponse,
    summary="List summary tasks",
    description="Returns a newest-first page of summary tasks...",
    response_description="A paginated task-list response envelope.",
)
  • response_model 生成 200 响应 schema 引用。
  • summary 适合文档列表扫描。
  • description 说明排序、筛选、分页行为。
  • response_description 专门描述成功响应含义，和接口整体描述不同。
limit: Annotated[int, Query(ge=1, le=100, description="Maximum number of tasks to return.")] = 20

Query 同时参与运行时校验和 OpenAPI 参数 schema。只写 int 只能表达类型，不能表达最小值、最大值和说明。

④ 核心机制运行全流程拆解

浏览器访问 /openapi.json
  -> FastAPI 遍历已注册 routes
  -> 收集 path/method/tags/summary/description
  -> 读取函数签名中的 Query/Path/Body
  -> 将 Pydantic/SQLModel 模型转 JSON Schema
  -> 收集 APIKeyHeader 为 securitySchemes
  -> 生成 OpenAPI JSON
  -> /docs 读取该 JSON 并渲染 Swagger UI

因此，写好类型注解、响应模型和参数约束，不只是让 Python 代码更清楚，也是让机器可读契约更完整。

⑤ 重难点、易混淆点对比辨析

⑥ 开发常见报错+坑点+解决方案

  • 文档看不到响应结构：缺少 response_model。解决：为成功响应声明 Pydantic/SQLModel 模型。
  • OpenAPI 参数没有最大值：只写了 int。解决：使用 Query(ge=..., le=...)。
  • 前端不知道要传 API Key：安全依赖未进入 OpenAPI 或命名混乱。解决：使用 APIKeyHeader 并设置 scheme_name。
  • 代码改了文档没变：实际没有改到类型/模型/装饰器。解决：检查 /openapi.json，并补 OpenAPI 契约测试。
  • 文案漂亮但契约不受保护：没有测试。解决：断言关键路径、参数、schema 和 security。

⑧ 课后记忆习题+标准答案

1. /docs 和 /openapi.json 谁是机器可读契约？

/openapi.json。/docs 是 Swagger UI 对该 JSON 的可视化渲染。

2. FastAPI 从哪些代码生成 OpenAPI？

从 app 元信息、路由装饰器、函数签名、Query/Path/Body、Pydantic/SQLModel 模型、response_model 和安全依赖生成。

3. 为什么安全方案也属于 API 契约？

调用方必须知道密钥放在哪里、方案叫什么、哪些接口需要鉴权，否则无法正确调用受保护接口。

4. OpenAPI 契约测试应该保护什么？

保护关键路径、HTTP 方法、参数边界、响应 schema、安全要求和重要错误响应，而不是所有文案逐字锁死。

⑨ 面试高频真题+解析

题 1：FastAPI 自动文档是怎么生成的？

FastAPI 根据 path operation、类型注解、Pydantic 模型、参数声明和安全依赖生成 OpenAPI schema；/openapi.json 是契约，/docs 是基于它的 Swagger UI。

题 2：为什么要测试 OpenAPI？

因为前端和外部调用方依赖的不只是运行时返回，还包括文档和 SDK 看到的 schema。OpenAPI 测试能发现 response_model、参数约束、安全方案被误改。

题 3：response_model 有哪些作用？

它用于响应文档、序列化、字段过滤和 schema 生成，让接口输出从“碰巧返回某个 dict”变成稳定契约。
