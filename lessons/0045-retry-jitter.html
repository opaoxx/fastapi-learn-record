Lesson 0045

Retry Jitter：给重试等待加一点随机扰动

Backoff 让客户端失败后慢一点再试，但如果所有客户端都用同一套公式，它们仍然可能在同一秒整齐重试。本节给本地 backoff 加入可配置 jitter，让 provider 重试时间分散开。

0. 本节学习目标

  • 理解为什么只有 exponential backoff 仍然可能出现重试峰值。
  • 理解 jitter 的作用：打散多个客户端的重试时间。
  • 掌握本项目采用的简单 additive jitter 公式。
  • 学会把随机数生成器注入测试，避免测试不可重复。
  • 理解为什么 Retry-After 不额外加 jitter。

1. 背景：指数退避仍然可能“排队齐步走”

第 43 课引入了指数退避：第一次失败等 0.1 秒，第二次失败等 0.2 秒，第三次失败等 0.4 秒。它能减少立即重试带来的冲击。

但如果 100 个客户端在同一时刻失败，并且都使用同样的 base delay 和 max delay，那么它们仍然会在 0.1 秒后一起重试，在 0.2 秒后一起重试，在 0.4 秒后一起重试。峰值被推迟了，但没有真正被打散。

AWS Architecture Blog 对这个问题有一个经典解释：capped exponential backoff 能让请求变少，但仍会产生请求簇；加入 jitter 的目的，是把这些峰值打散成更平滑的请求流。Google Cloud 的 retry 指南也建议使用带 jitter 的截断指数退避，避免客户端同步重试。

2. 核心概念：jitter 是可控的随机扰动

Jitter 指在计算出的 backoff 等待时间上加入一个小的随机量。它不是为了“让代码玄学”，而是为了让多个客户端不要在同一时间点重试。

without jitter:
client A: wait 1.0s
client B: wait 1.0s
client C: wait 1.0s

with jitter:
client A: wait 1.08s
client B: wait 1.31s
client C: wait 1.47s

本项目采用初学者更容易理解的 additive jitter：

delay = min(exponential_delay + random_fraction * jitter_seconds, max_delay)

其中 random_fraction 在 0 到 1 之间，jitter_seconds 是最大随机扰动范围。

3. 为什么不直接上 full jitter

AWS 文章里比较了 full jitter、equal jitter、decorrelated jitter 等策略。真实 SDK 可以做得更完整，但课程项目的目标是先让你理解“为什么要随机化”和“怎么让随机性可测试”。

所以本节先实现简单 additive jitter。它保留了上一课的指数退避形状，只在本地 backoff 上加一点可配置的扰动。等你理解这个版本后，再学习 full jitter 会更自然。

4. 运行机制：只有本地 backoff 加 jitter

if Retry-After is valid:
    delay = min(provider_delay, max_delay)
else:
    delay = min(exponential_delay + jitter_delay, max_delay)

这里有一个重要边界：Retry-After 是 provider 的明确提示，所以不额外加 jitter。否则 provider 说“2 秒后再试”，客户端却随机变成 2.4 秒，就会让测试、排查和协议语义都变得不清楚。

jitter 只用于本地估算的 backoff。也就是：provider 没有给明确时间时，客户端用自己的公式估算，并加入少量随机扰动。

5. 代码精读一：配置项

class Settings(BaseSettings):
    ai_retry_jitter_seconds: float = Field(default=0.0, ge=0)

jitter 也是 provider 调用策略，所以放在 settings 里。默认值是 0，表示不启用随机扰动；在 .env.example 中给出推荐示例：

FIRST_API_AI_RETRY_JITTER_SECONDS="0.1"

这样课程项目默认行为保持可预测，但读者可以清楚看到真实项目里应该如何打开它。

6. 代码精读二：随机扰动计算

RetryRandom = Callable[[], float]

def calculate_retry_jitter(
    config: AIClientConfig,
    random_fraction: RetryRandom = random.random,
) -> float:
    jitter_seconds = max(config.retry_jitter_seconds, 0.0)
    if jitter_seconds == 0:
        return 0.0

    fraction = min(max(random_fraction(), 0.0), 1.0)
    return jitter_seconds * fraction

random.random() 来自 Python 标准库，会生成 0 到 1 之间的伪随机浮点数。这里把它包装成 random_fraction 参数，是为了测试可以传入固定函数，例如 lambda: 0.4。

我们还把随机值限制在 0 到 1。生产代码里的 random.random() 本来就在这个范围，但测试或未来调用者可能传入别的函数；边界保护能让策略更稳。

7. 代码精读三：jitter 加到本地 backoff

exponential_delay = base_delay * (2 ** (attempt - 1))
jitter_delay = calculate_retry_jitter(config, random_fraction=random_fraction)
return min(exponential_delay + jitter_delay, max_delay)

例子：base=1.0，第 2 次失败的指数退避是 2.0；jitter_seconds=0.5，随机比例是 0.4，那么 jitter 是 0.2，最终等待就是 2.2 秒。

最后仍然用 max_delay 截断，因为 jitter 不应该突破客户端自己的最大等待边界。

8. 代码精读四：把随机函数传到 retry 分支

async def request_provider_prediction(
    ...,
    random_fraction: RetryRandom = random.random,
) -> PredictionResult:

和上一节的 fake sleep 一样，随机数也要可注入。生产代码不传参数，就使用 random.random；测试代码传 lambda: 0.4，就能准确断言最终 sleep 时间。

这就是测试随机逻辑的关键：不要让测试依赖真正随机数，而是把随机源变成依赖。

9. 测试精读一：直接测试 jitter 数学

def test_calculate_retry_jitter_uses_bounded_random_fraction():
    config = AIClientConfig(retry_jitter_seconds=0.5)

    jitter = calculate_retry_jitter(config, random_fraction=lambda: 0.4)

    assert jitter == 0.2

这条测试保护最小数学规则：最大扰动 0.5 秒，随机比例 0.4，所以 jitter 是 0.2 秒。

10. 测试精读二：真实 retry 分支里的 jitter

result = run_async_prediction(
    client,
    max_attempts=2,
    retry_base_delay_seconds=1.0,
    retry_max_delay_seconds=3.0,
    retry_jitter_seconds=0.5,
    sleep=fake_sleep,
    random_fraction=lambda: 0.4,
)

assert sleep_calls == [1.2]

这里第一次 provider 返回 503。第 1 次失败的指数退避是 1.0，jitter 是 0.2，所以 fake sleep 记录 1.2。这证明随机扰动真正进入了 adapter 的 retry 等待流程。

12. 常见错误与诊断

错误一：测试使用真实随机数

测试会偶发失败。把随机函数注入进去，用固定返回值验证计算。

错误二：jitter 突破最大等待

jitter 后仍然要 cap。最大等待是客户端自己的服务边界。

错误三：给 Retry-After 再加 jitter

provider 已经给了明确提示，额外随机化会让协议语义变模糊。

错误四：把 jitter 当安全随机数

Python random 适合普通扰动，不适合密码学或安全令牌。

13. 面试视角：怎么解释 jitter

指数退避能减少立即重试，但多个客户端用同一公式时仍可能同步重试。jitter 的作用是给本地 backoff 加入小的随机扰动，打散请求峰值。我会把 jitter 做成配置，并把随机数生成函数注入 adapter，这样生产环境使用 random.random，测试环境使用固定函数，保证随机逻辑可测试。

14. 小测：先回忆，再展开

1. 为什么 exponential backoff 还需要 jitter？

因为多个客户端如果同时失败并使用同一公式，仍会在相同时间点重试，形成请求峰值。

2. jitter_seconds=0.5，随机比例 0.4，jitter 是多少？

是 0.2 秒，也就是 0.5 * 0.4。

3. 为什么测试里要注入 random_fraction？

这样测试可以固定随机结果，避免因为真实随机数导致偶发失败。

4. 为什么 Retry-After 不额外加 jitter？

因为它是 provider 的明确重试时间提示，额外随机化会削弱协议语义和可排查性。

15. 本节总结

本节把 provider retry 从“有等待”升级成“等待时间能打散”。现在本地指数退避会根据 retry_jitter_seconds 加入随机扰动，并且仍受最大等待上限保护。测试通过注入固定随机函数，验证了随机逻辑不必让测试变得随机。
