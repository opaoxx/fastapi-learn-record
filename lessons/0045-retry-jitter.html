0045 Retry Jitter：给本地退避加入可控随机抖动

指数退避能推迟重试，但如果所有客户端都按同一公式等待，它们仍会在同一时间点重试。jitter 的作用是打散重试峰值；本项目采用 additive jitter，并把随机函数注入 adapter，使随机策略可测试。

① 本节核心知识框架

Retry Jitter
├─ 问题：相同 backoff 公式会造成同步重试
├─ 策略：jitter = retry_jitter_seconds * random_fraction()
├─ 本地 delay：min(exponential_delay + jitter, max_delay)
├─ 随机源：RetryRandom = Callable[[], float]
├─ 测试控制：random_fraction=lambda: 0.4
└─ 边界：Retry-After 不额外加 jitter

核心结论：jitter 不是“让代码随机一点”，而是分布式系统中降低重试同步峰值的可靠性手段。

② 核心概念底层原理

如果 100 个请求在同一时刻失败，且都等待 1 秒后重试，峰值只是被推迟到 1 秒后。jitter 在等待时间上加入小范围随机量，让请求分布到不同时间点。

策略示例效果
无 jitter1.0s, 1.0s, 1.0s重试仍同步
additive jitter1.1s, 1.3s, 1.4s峰值被打散
full jitter0 到 cap 内随机更强打散，策略更复杂

课程采用 additive jitter，是为了保持公式可读：先算指数退避，再加一个小随机扰动，最后用最大等待截断。

③ 全套代码逐行精读解析

class Settings(BaseSettings):
    ai_retry_jitter_seconds: float = Field(default=0.0, ge=0)

jitter 默认关闭，保证教学行为确定；生产环境可通过 FIRST_API_AI_RETRY_JITTER_SECONDS 开启。

RetryRandom = Callable[[], float]

随机源被抽象成无参函数，返回一个浮点比例。这样生产用 random.random，测试用固定 lambda。

def calculate_retry_jitter(
    config: AIClientConfig,
    random_fraction: RetryRandom = random.random,
) -> float:
    jitter_seconds = max(config.retry_jitter_seconds, 0.0)
    if jitter_seconds == 0:
        return 0.0

负值保护与零值短路。未启用 jitter 时不调用随机函数，避免无意义随机依赖。

fraction = min(max(random_fraction(), 0.0), 1.0)
return jitter_seconds * fraction

即使测试或未来调用方传入异常随机函数，也把比例夹在 0 到 1。若 jitter_seconds=0.5 且比例为 0.4，则 jitter 为 0.2 秒。

exponential_delay = base_delay * (2 ** (attempt - 1))
jitter_delay = calculate_retry_jitter(config, random_fraction=random_fraction)
return RetryDelayPlan(
    delay_seconds=min(exponential_delay + jitter_delay, max_delay),
    source="local_backoff",
)

jitter 只加入本地 backoff 分支。加完后仍用 max_delay 截断，避免抖动突破服务边界。

if provider_delay is not None:
    return RetryDelayPlan(
        delay_seconds=min(provider_delay, max_delay),
        source="retry_after",
    )

合法 Retry-After 不额外加 jitter，因为那是 provider 的明确协议提示，随意扰动会弱化语义。

④ 核心机制运行全流程拆解

retryable failure
  → build_retry_delay_plan()
      ├─ valid Retry-After：delay = provider hint capped
      └─ no valid Retry-After：
          → exponential_delay = base * 2 ** (attempt - 1)
          → jitter = jitter_seconds * random_fraction()
          → delay = min(exponential_delay + jitter, max_delay)
  → log/sleep
  → next attempt

随机性从 request_provider_prediction(..., random_fraction=...) 传入，再一路传给 wait_before_retry()、build_retry_delay_plan()、calculate_retry_jitter()。这条调用链让随机行为可控、可测、可替换。

⑤ 重难点、易混淆点对比辨析

易混点正确理解隐患
jitter vs backoffbackoff 推迟，jitter 打散只有 backoff 仍可能同步重试
随机测试注入固定随机函数真实随机会导致偶发失败
Retry-After 加 jitter不加破坏 provider 明确提示
安全随机数这里不是安全用途不要用普通 random 生成密钥

⑥ 开发常见报错+坑点+解决方案

  • 测试结果偶发变化：直接调用真实 random.random()。解决：测试传 lambda: 0.4。
  • jitter 后 delay 超过最大等待：先 cap 再加 jitter。解决：加完 jitter 后再 min(..., max_delay)。
  • Retry-After 被随机扰动：把 provider hint 和 local backoff 混在一起。解决：provider_delay 可用时直接返回 retry_after source。
  • jitter 配置过大：等待波动过大。解决：根据 API 总超时和 SLA 设置小范围扰动。

⑧ 课后记忆习题+标准答案

1. 为什么有指数退避还需要 jitter？

多个客户端可能同时失败并按同一公式等待，仍会同步重试；jitter 用随机扰动打散峰值。

2. jitter_seconds=0.5、随机比例 0.4 时 jitter 是多少？

0.2 秒。

3. 为什么测试要注入 random_fraction？

固定随机结果，避免测试因真实随机数偶发失败。

4. 为什么 Retry-After 不额外加 jitter？

它是 provider 明确给出的重试提示，额外随机化会削弱协议语义和可排查性。

⑨ 面试高频真题+解析

题 1：jitter 解决什么问题？ 答：解决多个客户端按同一 backoff 公式同步重试造成的请求峰值问题。

题 2：如何让随机重试策略可测试？ 答：把随机数生成器作为依赖注入，生产用真实随机函数，测试用固定返回值函数。

题 3：additive jitter 有什么局限？ 答：它简单可读，但打散能力不如 full jitter。大规模生产 SDK 常会采用更成熟的 jitter 变体。
