0074 Provider Metrics Practice Release Checklist：GitHub 发布前检查完整训练包

本节新增 build_provider_metrics_runbook_practice_release_checklist()，用于检查完整 runbook practice package 是否具备 GitHub 发布条件。它不生成材料，只检查 findings、题卡、答案键、评分校验、评分报告和 Markdown 章节是否齐全。

① 本节核心知识框架

│  └─ package_markdown
├─ 检查项

├─ 输出

│  └─ checks

│     ├─ passed
│     └─ evidence
└─ 边界

   ├─ 不重新渲染 Markdown
   ├─ 不修复缺失材料
   └─ 不替代生产发布流水线

② 核心概念底层原理

发布检查清单不是业务逻辑，也不是文档渲染器。它属于质量门禁：在材料进入 GitHub、PR 或学习记录之前，检查关键产物是否齐全。

本节选择返回结构化 checks，而不是只返回一个布尔值。布尔值只能说明“能不能发布”，不能解释“为什么不能发布”。发布前检查必须能定位缺失项，否则新手只能反复猜。

ready 是所有检查项的合取结果：只有每一项 passed=True，完整包才被视为可发布。这个设计比“只要报告存在就发布”更严格，因为 GitHub 教程需要可学习、可复盘、可验证。

检查对象为什么需要缺失后果
Findings解释指标事实读者不知道题目来源
Exercise Cards提供训练入口无法自测
Answer Key提供标准答案无法复盘对错
Grading Report展示本次作答结果无法衡量训练效果
Markdown Sections保证发布结构GitHub 页面难以阅读

复核补强：发布检查的核心不是“再跑一遍业务”，而是把发布条件变成机器可读门禁。ready 只是汇总结果，真正有价值的是每个 check 的 id、passed 和 evidence，它们能直接转成 PR 评论或 CI 输出。

门禁维度检查方式阻止的问题
内容存在检查 findings、cards、answer key、report材料缺页但仍被发布。
数据可信要求 validation 为 {"valid": True, "errors": []}错误 summary 被包装成正式复盘。
文档结构检查四个固定 Markdown 章节GitHub 页面缺少学习闭环。
失败可解释返回逐项 evidence只知道不能发布，不知道哪里坏。

③ 全套代码逐行精读解析

def build_provider_metrics_runbook_practice_release_checklist(
    session: dict[str, object],
    package_markdown: str,
) -> dict[str, object]:
    checks = [

函数接收已生成的 session 和 package Markdown。它不主动生成材料，说明它位于导出之后、发布之前，是一个独立质量检查层。

{
    "id": "findings_markdown_present",
    "passed": bool(str(session.get("findings_markdown", "")).strip()),
    "evidence": "Findings Markdown is present.",
},

第一项检查 findings 报告是否存在。使用 get() 是因为本函数目标是检查缺失，而不是在第一个缺字段处抛异常。

{
    "id": "exercise_cards_present",
    "passed": bool(session.get("cards")),
    "evidence": "Exercise cards are present.",
},

题卡必须存在，否则完整包只有解释和答案，没有练习入口。bool([]) 为 False，能自然识别空题卡。

{
    "id": "answer_key_present",
    "passed": "## provider-metrics-runbook-" in str(session.get("answer_key", "")),
    "evidence": "Answer key contains runbook card sections.",
},

答案键不是只检查非空，而是检查是否包含题卡标题模式。这样可以避免随便一段文本被误认为标准答案。

{
    "id": "grading_summary_valid",
    "passed": session.get("validation") == {"valid": True, "errors": []},
    "evidence": "Grading summary validation passed.",
},

发布前必须确认 summary 校验通过。若 validation 失败，即使 Markdown 看起来完整，也不能把它作为可信复盘材料发布。

{
    "id": "package_sections_present",
    "passed": all(
        section in package_markdown
        for section in [
            "## Findings",
            "## Exercise Cards",
            "## Answer Key",
            "## Grading Report",
        ]
    ),
    "evidence": "Practice package contains all required sections.",
},

完整包必须包含四个固定章节。这里使用 all() 表示任意章节缺失都会导致发布检查失败。

return {
    "ready": all(check["passed"] for check in checks),
    "checks": checks,
}

ready 汇总所有检查项，checks 保留逐项证据。生产中这类结构可直接进入 CI 报告、PR comment 或发布审批页面。

逐行精读补充：all(check["passed"] for check in checks) 明确采用“全项通过”策略，而不是给检查项设置权重。教学发布包的每个环节都服务学习闭环，缺任意一项都会破坏复盘完整性。

本函数多处使用 session.get(...)，是因为 release checklist 面对的正是“不完整包”。它应把缺失转为失败检查项，而不是在第一个缺失字段处中断整个发布诊断。

④ 核心机制运行全流程拆解

session
  → render_provider_metrics_runbook_practice_session_markdown(session)
  → package_markdown
  → build_provider_metrics_runbook_practice_release_checklist(session, package_markdown)
  → ready/checks
  → GitHub publish decision

这条链路把“生成材料”和“检查材料”分开。生成层负责产物，检查层负责门禁，发布层再根据 ready 决定是否进入 GitHub。

阶段产物职责失败处理
会话生成session组织训练数据检查函数链路
文档导出package_markdown生成可读材料检查章节和格式
发布检查checklist判断是否可发布定位失败检查项

⑧ 课后记忆习题+标准答案

1. 为什么 release checklist 不自动生成缺失材料？

检查层只负责报告问题，自动生成会掩盖上游导出或会话构建错误。

2. 为什么 ready 必须依赖所有检查项？

发布材料面向他人学习，任何关键章节缺失都会破坏复盘闭环，因此必须全项通过。

3. 为什么 answer key 要检查标题模式？

非空文本不一定是答案键，题卡标题模式能更准确确认包含标准答案章节。

4. 为什么检查结果要保留 evidence？

evidence 让检查结果可解释，方便在 PR、学习记录或 CI 输出中定位问题。

5. 生产 CI 比本节 checklist 多什么？

生产 CI 还要覆盖自动化测试、安全扫描、链接检查、权限、合规、构建产物和发布审批。

⑨ 面试高频真题+解析

面试题 1：发布检查清单和单元测试有什么区别？

单元测试验证代码行为是否符合预期；发布检查清单验证待发布材料是否完整、可信、适合交付给读者或用户。

面试题 2：为什么发布前检查需要结构化结果？

结构化结果可以被 CI、PR comment、页面和日志复用，也能让失败项具备机器可读和人类可解释两种价值。

面试题 3：为什么 validation 失败必须阻止发布？

validation 失败说明 summary 契约不可信。继续发布会把错误统计包装成正式材料，破坏复盘可信度。

面试题 4：如何把教学 checklist 升级为生产发布门禁？

需要接入 CI、权限系统、安全扫描、构建产物校验、链接检查、负责人审批和审计留痕。
