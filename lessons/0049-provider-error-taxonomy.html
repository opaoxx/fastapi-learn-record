0049 Provider Error Taxonomy：外部 AI 调用错误分类体系

本节把 timeout、网络错误、HTTP 429/5xx、HTTP 400/4xx、非法响应体统一进可测试的 failure taxonomy。核心目标不是“多写几个异常名”，而是让重试策略、结构化日志、指标标签、测试断言和面试表达使用同一套稳定语言。

① 本节核心知识框架

Provider Error Taxonomy
├─ 失败发生层次
│  ├─ 传输层：timeout / request_error
│  ├─ HTTP 状态层：retryable_http_status / non_retryable_http_status
│  └─ 响应契约层：invalid_response
├─ 决策维度
│  ├─ 是否可重试：429、500、502、503、504 可有限重试
│  ├─ 是否 fail fast：400、401、404、422、非法响应体
│  └─ 是否耗尽：可重试错误在 max_attempts 用完后 retry_exhausted
├─ 代码落点
│  ├─ RETRYABLE_PROVIDER_STATUS_CODES
│  ├─ ProviderFailureCategory
│  ├─ is_retryable_provider_status()
│  └─ classify_provider_http_status()
└─ 工程价值
   ├─ 日志字段稳定
   ├─ 指标标签低基数
   ├─ 测试保护命名契约
   └─ 面试可解释“分类 → 决策 → 观测 → 上抛”链路

② 核心概念底层原理

taxonomy 是分类语言，不是异常处理语法糖。外部 provider 失败必须先回答“失败发生在哪一层”，再回答“系统该做什么”。如果直接把所有异常统称为 provider error，重试、报警、日志检索和指标聚合都会丢失语义。

层次典型输入底层含义策略
传输层httpx.TimeoutException请求没有在预算内完成，可能是网络抖动或 provider 慢有限重试
传输层httpx.RequestError连接、DNS、TLS、代理等请求发送链路失败有限重试
HTTP 状态层429/500/502/503/504provider 接收了请求，但当前无法稳定处理有限重试
HTTP 状态层400/401/404/422请求参数、鉴权、路径或契约不匹配fail fast
响应契约层JSON 字段缺失、label 非法、score 越界HTTP 成功不代表业务响应满足内部模型fail fast

retryable_http_status 与 retry_scheduled 不是同一类词。前者描述错误性质，后者描述系统动作。同一个 503 在第 1 次 attempt 上可能触发 retry_scheduled，在最后一次 attempt 上可能触发 retry_exhausted，但 failure category 始终是 retryable_http_status。

③ 全套代码逐行精读解析

RETRYABLE_PROVIDER_STATUS_CODES = {429, 500, 502, 503, 504}

ProviderFailureCategory = Literal[
    "timeout",
    "request_error",
    "retryable_http_status",
    "non_retryable_http_status",
    "invalid_response",
]

def is_retryable_provider_status(status_code: int) -> bool:
    return status_code in RETRYABLE_PROVIDER_STATUS_CODES

def classify_provider_http_status(status_code: int) -> ProviderFailureCategory:
    if is_retryable_provider_status(status_code):
        return "retryable_http_status"
    return "non_retryable_http_status"
代码作用为什么这样写不这样写的隐患
RETRYABLE_PROVIDER_STATUS_CODES集中声明可重试 HTTP 状态码集合集合查询是 O(1) 平均复杂度，语义清晰散落在 if 分支中会导致 429、503 等策略不一致
429限流通常可结合 Retry-After 延迟重试当成 fail fast 会浪费 provider 明确给出的恢复窗口
500/502/503/504服务端临时错误表示 provider 或网关暂时不可用无重试会降低瞬时故障下的成功率
ProviderFailureCategory定义失败类别的闭合集合Literal 让编辑器和测试更容易发现拼写漂移自由字符串会产生 retryable、server_error 等不兼容别名
is_retryable_provider_status()隔离“状态码是否可重试”的布尔判断调用方无需知道状态码集合细节重复判断会让后续扩展 408、425 等状态码时漏改
classify_provider_http_status()把 HTTP status 转成 taxonomy 名称后续日志、指标、测试都依赖这个稳定分类直接返回 bool 会丢失语义，指标无法区分失败类别
return "non_retryable_http_status"默认把未列入集合的 HTTP 错误视为不可重试保守策略避免对参数错误、鉴权错误进行无意义重试默认重试所有未知状态会放大 provider 压力

HTTP 分支中的核心使用方式如下：

status_code = exc.response.status_code
retryable_status = (
    classify_provider_http_status(status_code)
    == "retryable_http_status"
)

这里先分类再决策。若把条件写成 status_code >= 500，会漏掉 429；若把所有 4xx 都 fail fast，又要明确 429 是特殊的限流重试状态。

④ 核心机制运行全流程拆解

provider POST /predict
  → response.raise_for_status()
  → 捕获 httpx.HTTPStatusError
  → 读取 exc.response.status_code
  → classify_provider_http_status(status_code)
  → retryable_http_status ?
       ├─ attempt < max_attempts：wait_before_retry() + retry_scheduled metric/log
       ├─ attempt == max_attempts：log_retry_exhausted() + 上抛 AIClientError
       └─ 否：log_prediction_fail_fast() + 上抛 AIClientError

关键数据流是：provider 原始 status code 不直接泄漏为业务响应语义，而是先转成内部 failure category；内部 category 再驱动 retry/fail-fast；最终对上层仍使用稳定 AIClientError(error_code="ai_provider_http_error")。

因此，调用方看到的本项目 HTTP 503 与 provider 原始 400 可以同时成立：项目对外表达“AI provider 能力不可用”，metrics/logs 保留 provider 原始状态用于诊断。

⑤ 重难点、易混淆点对比辨析

易混点正确理解判断口诀
failure_category vs outcomecategory 是错误性质，outcome 是系统动作或结果“是什么错”与“系统做了什么”分开
provider HTTP 400 vs 本项目 HTTP 503400 是上游原始状态；503 是当前 API 对调用方的服务不可用表达外部契约与内部诊断不是同一层
429 vs 普通 4xx429 是限流，通常可重试；400/401/404/422 通常不可重试429 看恢复窗口，其他 4xx 看请求契约
timeout vs request_errortimeout 是预算耗尽；request_error 是请求链路异常一个是“等不到”，一个是“发不稳”
invalid_response vs HTTP errorinvalid_response 可能发生在 HTTP 200 后状态码成功不等于响应体契约成功

⑧ 课后记忆习题+标准答案

1. 为什么 provider 错误不能统一叫 provider_error？

因为 timeout、request error、retryable HTTP status、non-retryable HTTP status、invalid response 的策略完全不同。统一命名会丢失重试决策、日志检索和指标聚合所需的语义。

2. 429 为什么属于可重试，而不是普通 4xx？

429 表示限流，provider 通常希望客户端等待后再试，甚至会返回 Retry-After。400、401、404、422 更常表示请求或鉴权契约问题，重复请求通常不会自动恢复。

3. retryable_http_status 与 retry_exhausted 的区别是什么？

retryable_http_status 是失败类别，说明错误具备可重试性质；retry_exhausted 是 outcome，说明系统已经用完最大尝试次数并最终失败。

4. HTTP 200 后为什么仍可能 fail fast？

HTTP 200 只说明传输层和状态码层成功；如果响应体无法通过 ProviderPredictionBody 校验，例如 label 非法或 score 越界，仍属于 invalid_response。

⑨ 面试高频真题+解析

面试题 1：你如何设计外部 AI provider 的错误分类？

先按失败层次分类：传输层 timeout/request_error，HTTP 状态层 retryable/non-retryable status，响应契约层 invalid_response。再用分类驱动策略：可重试错误在预算内 retry_scheduled，预算耗尽 retry_exhausted；不可重试错误 fail_fast。对外返回稳定错误码，对内保留 status_code、failure_category 和日志/指标字段。

面试题 2：为什么不直接按 5xx 重试、4xx 不重试？

这个规则不完整。429 是 4xx 但通常可重试；部分 provider 也可能定义特殊可恢复状态。生产中还要考虑幂等性、请求成本、Retry-After、重试预算和熔断。课程中通过 RETRYABLE_PROVIDER_STATUS_CODES 显式列出可重试集合，避免隐含规则。

面试题 3：为什么 taxonomy 名称需要测试保护？

taxonomy 名称会进入日志、指标、文档、仪表盘和排障流程。随意改名不是普通重构，而是观测契约变更。测试能让重命名变成显式决策，避免无意破坏下游查询和学习材料。
