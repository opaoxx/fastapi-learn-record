0039 Prediction Backend Switch：用配置选择 demo 或 provider 后端

本节把真实 provider 能力收回到正式入口 POST /predict：调用方继续使用同一个 URL、同一个请求体、同一个响应体；服务端通过 FIRST_API_AI_PREDICTION_BACKEND 决定内部走 demo client 还是 provider adapter。

① 本节核心知识框架

Prediction Backend Switch
├─ 公共入口：POST /predict
├─ 配置开关：ai_prediction_backend = "demo" | "provider"
├─ demo 分支：AIClient.predict_sentiment(text, mode)
├─ provider 分支：request_provider_prediction(...)
├─ 输出归一：PredictionResult → PredictionResponse
└─ 测试策略：override get_settings + override get_provider_http_client

核心结论：后端实现可以切换，但公共 API 契约不能跟着抖动。配置开关属于服务端部署策略，不应暴露成普通用户可控请求参数。

② 核心概念底层原理

后端选择属于环境边界：开发环境默认 demo，便于离线学习；集成环境或生产环境可切 provider，验证真实供应商。若让前端传 ?backend=provider，等于把服务部署策略交给调用方，可能绕过成本控制、稳定性策略和安全边界。

方案优点隐患
两个长期入口学习时直观契约重复、文档分裂、权限和日志容易漏改
请求参数选择 backend调试方便调用方可随意切换内部实现
服务端配置选择 backend公共 API 稳定，部署可控需要测试覆盖不同配置分支

③ 全套代码逐行精读解析

class Settings(BaseSettings):
    ai_provider: Literal["demo"] = "demo"
    ai_prediction_backend: Literal["demo", "provider"] = "demo"

Literal 把合法配置值限制在有限集合。若环境变量拼成 prodvider，配置加载阶段就会暴露错误；如果写成普通 str，错误可能直到请求进来才表现为奇怪分支。

async def predict(
    payload: PredictionRequest,
    response: Response,
    settings: Annotated[Settings, Depends(get_settings)],
    ai_client: Annotated[AIClient, Depends(get_ai_client)],
    provider_http_client: Annotated[httpx.AsyncClient, Depends(get_provider_http_client)],
) -> PredictionResponse:

/predict 升级为 async def，因为 provider 分支要等待异步 HTTP。demo 分支虽然是同步轻量函数，但可以在 async endpoint 中直接调用。

result = await run_prediction(
    text=payload.text,
    mode=payload.mode,
    settings=settings,
    ai_client=ai_client,
    provider_http_client=provider_http_client,
)

路由不再自己写 if backend == ...，而是把选择逻辑交给 service。此处仍注入两类依赖，是为了让 service 拥有执行任意分支所需的资源。

active_backend = backend or settings.ai_prediction_backend
if active_backend == "provider":
    config = AIClientConfig(...)
    prediction = await request_provider_prediction(...)
else:
    prediction = ai_client.predict_sentiment(text, mode)

backend 参数允许 /provider/predict 强制 provider；普通 /predict 不传该参数，使用 settings。两个分支最终都必须生成 PredictionResult，这就是输出归一。

def build_prediction_response(
    prediction: PredictionResult,
    text: str,
) -> PredictionResponse:
    return PredictionResponse(
        label=prediction.label,
        score=prediction.score,
        source=prediction.source,
        text_length=len(text),
    )

响应构造集中化，避免 demo/provider 两个分支各自拼 JSON。若未来响应增加字段，只改一个函数即可。

④ 核心机制运行全流程拆解

POST /predict
  → PredictionRequest 校验
  → get_settings() 读取 ai_prediction_backend
  → run_prediction()
      ├─ demo：AIClient.predict_sentiment()
      └─ provider：request_provider_prediction()
  → PredictionServiceResult(prediction, backend, elapsed_ms)
  → router 设置 headers / logs
  → PredictionResponse

配置从环境进入 Settings，再进入 service，而不是从请求参数进入业务分支。这个方向很关键：环境决定服务行为，用户请求只携带预测输入。

⑤ 重难点、易混淆点对比辨析

易混点辨析工程判断

⑥ 开发常见报错+坑点+解决方案

  • 环境变量拼错导致启动失败：Literal 会拒绝非法值。解决：检查 FIRST_API_AI_PREDICTION_BACKEND 是否为 demo 或 provider。
  • provider 分支访问外网：测试只覆盖 settings，忘记覆盖 HTTP client。解决：同时覆盖 get_settings 与 get_provider_http_client。
  • 两个分支响应不一致：分别手写返回体。解决：统一经过 build_prediction_response()。
  • 把 backend 暴露给前端：短期方便，长期破坏治理。解决：后端配置或专门的管理侧开关控制。

⑧ 课后记忆习题+标准答案

1. 为什么 backend 选择不应该由普通请求参数控制？

因为它是服务部署与治理策略，不是业务输入。暴露给调用方会带来成本、稳定性和权限风险。

2. Literal["demo", "provider"] 的价值是什么？

限制配置合法值，让拼写错误在配置加载阶段暴露，避免运行时进入未知分支。

3. 为什么两个后端都要转成 PredictionResult？

这样 router 只依赖统一内部结果，公共响应构造逻辑不会关心底层实现来自 demo 还是 provider。

4. 测试 provider 分支时为什么要覆盖两个依赖？

覆盖 settings 是为了进入 provider 分支，覆盖 provider client 是为了把外部 HTTP 调用替换成 MockTransport。

⑨ 面试高频真题+解析

题 1：如何让同一个 API 支持多个后端实现？ 答：保持公共请求/响应契约不变，在服务端通过配置或策略选择内部实现，各实现输出统一内部模型，再由 router 统一转响应。

题 2：为什么配置开关要有类型约束？ 答：类型约束可以把错误从运行时请求阶段前移到配置加载阶段，减少未知状态和隐式默认分支。

题 3：backend switch 后路由为什么会变胖，下一步怎么处理？ 答：分支选择、耗时、错误包装会堆进 router。下一步应抽到 service 层，让 router 只做 HTTP 翻译。
