Lesson 0039

Prediction Backend Switch：用配置开关切换 demo 与 provider 后端

第 38 课我们新增了 /provider/predict 来验证真实 provider 链路。本节把能力收回到正式入口 /predict：通过 FIRST_API_AI_PREDICTION_BACKEND 配置选择 demo 后端或 provider 后端，同时继续保持 PredictionResponse 稳定。

0. 本节学习目标

  • 理解为什么真实项目需要“后端选择开关”，而不是长期保留多个相似入口。
  • 学会用 settings 控制同一个 endpoint 的内部实现路径。
  • 理解公共 API 契约稳定与内部实现可切换之间的关系。
  • 学会把同步 demo client 和异步 provider adapter 放在同一个 async endpoint 中协调。
  • 学会用依赖覆盖测试配置分支，而不改真实 .env。

1. 背景：两个预测入口不能一直并排长大

上一课以后，项目里有两个预测入口：

  • POST /predict：走本地 DemoAIClient。
  • POST /provider/predict：走外部 provider adapter。

这在学习阶段很好，因为它能把 demo 路径和 provider 路径分开观察。但真实项目长期保留两个相似入口，会慢慢产生问题：

  • 前端不知道应该调用哪个入口。
  • 两个入口都要维护请求体、响应体和错误契约。
  • 以后加鉴权、日志、限流、追踪时容易漏改其中一个。
  • 文档里会出现“两个都能预测”的歧义。

更成熟的做法是：保留一个正式公共入口 /predict，内部通过配置选择使用 demo 后端还是 provider 后端。

2. 核心概念：公共接口稳定，后端实现可换

这节课的核心是一个工程原则：

调用方依赖的是 API 契约，不应该依赖服务内部到底使用 demo、provider、本地模型还是队列任务。

所以我们的目标不是让前端选择 /predict 或 /provider/predict，而是让服务端根据配置决定内部后端：

POST /predict
  request:  PredictionRequest
  response: PredictionResponse

FIRST_API_AI_PREDICTION_BACKEND=demo
  -> DemoAIClient

FIRST_API_AI_PREDICTION_BACKEND=provider
  -> request_provider_prediction()

这就是“同一个公共契约，多种内部实现”。它也是以后做灰度发布、A/B 测试、本地开发和生产环境切换的基础心智模型。

3. 配置开关不是普通 if，而是环境边界

本节新增配置：

FIRST_API_AI_PREDICTION_BACKEND="demo"

它只有两个合法值：

  • demo：本地演示、离线学习、GitHub 读者开箱即用。
  • provider：真实 provider 调用，需要配置 provider prediction URL。

为什么用配置，而不是请求参数 ?backend=provider？因为预测后端属于服务部署策略，不应该让普通调用方随意切换。调用方只负责提交预测请求；服务端负责决定当前环境用哪个实现。

4. 运行机制：/predict 现在怎样选择后端

POST /predict
  -> parse PredictionRequest
  -> get_settings()
  -> if settings.ai_prediction_backend == "provider"
       -> get_provider_http_client()
       -> await request_provider_prediction(...)
     else
       -> get_ai_client()
       -> ai_client.predict_sentiment(...)
  -> build PredictionResponse

注意一个细节：因为 provider 分支需要 await 异步 HTTP 调用，所以 /predict 从同步 def 升级成了 async def。demo 分支仍然可以在 async endpoint 中调用普通同步函数；这没有问题，因为 demo 逻辑非常轻量。

5. 代码精读一：Settings 新增后端选择

class Settings(BaseSettings):
    app_name: str = "First FastAPI API"
    environment: str = "development"
    api_key: str = "dev-secret-key"
    ai_provider: Literal["demo"] = "demo"
    ai_prediction_backend: Literal["demo", "provider"] = "demo"
    ai_provider_health_url: str = "https://example.com"
    ai_provider_prediction_url: str = "https://example.com/predict"

Literal["demo", "provider"] 很重要。它不是随便写一个字符串字段，而是把合法值限制住。这样如果有人把环境变量写成 prodvider 这种拼错值，应用配置加载阶段就能暴露问题，而不是等请求进来才出现诡异行为。

默认值是 demo，这是为了保证新手克隆项目后不配置真实 provider 也能继续使用 /predict。

6. 代码精读二：统一响应构造函数

def build_prediction_response(
    prediction: PredictionResult,
    text: str,
) -> PredictionResponse:
    return PredictionResponse(
        label=prediction.label,
        score=prediction.score,
        source=prediction.source,
        text_length=len(text),
    )

这个小函数的作用是去重。无论预测来自 demo client 还是 provider adapter，最终都应该变成同一个 PredictionResponse。如果两条分支各自手写一遍响应拼装，未来字段变化时很容易一个分支改了，另一个分支漏了。

这也是契约稳定的代码表达：不同后端先统一成 PredictionResult，再统一转成 PredictionResponse。

7. 代码精读三：/predict 的后端选择逻辑

@router.post("/predict", response_model=PredictionResponse)
async def predict(
    payload: PredictionRequest,
    settings: Annotated[Settings, Depends(get_settings)],
    ai_client: Annotated[AIClient, Depends(get_ai_client)],
    provider_http_client: Annotated[
        httpx.AsyncClient,
        Depends(get_provider_http_client),
    ],
) -> PredictionResponse:
    try:
        if settings.ai_prediction_backend == "provider":
            config = AIClientConfig(
                provider=settings.ai_provider,
                timeout_seconds=settings.ai_timeout_seconds,
                max_attempts=settings.ai_max_attempts,
            )
            prediction = await request_provider_prediction(
                url=settings.ai_provider_prediction_url,
                text=payload.text,
                mode=payload.mode,
                config=config,
                client=provider_http_client,
            )
        else:
            prediction = ai_client.predict_sentiment(payload.text, payload.mode)
    except AIClientError as exc:
        raise HTTPException(status_code=503, detail={...}) from exc

    return build_prediction_response(prediction, payload.text)

这段代码的职责分层如下：

  • settings 决定走哪条分支。
  • ai_client 是 demo 分支的实现。
  • provider_http_client 是 provider 分支需要的长期 HTTP client。
  • request_provider_prediction() 负责 provider HTTP 调用和外部 JSON 校验。
  • build_prediction_response() 保证两个分支的输出契约一致。

这里有一个可以继续优化的点：FastAPI 会先解析所有依赖，所以即使 demo 分支不使用 provider_http_client，它也会被注入。对当前教学项目来说这没问题，因为 lifespan 本来就会创建 provider client。更大型项目可以进一步把后端选择抽成一个更深的 service dependency。

9. 测试精读二：用 dependency_overrides 切到 provider

def test_predict_can_switch_to_provider_backend() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(
            status_code=200,
            json={"label": "positive", "score": 0.914},
            request=request,
        )

    def override_provider_settings() -> Settings:
        return Settings(ai_prediction_backend="provider")

    replacement_client = httpx.AsyncClient(
        transport=httpx.MockTransport(handler),
        trust_env=False,
    )
    app.dependency_overrides[get_settings] = override_provider_settings
    app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

    response = client.post("/predict", json={...})

    assert response.json()["source"] == "demo-provider"

这里没有修改真实 .env。测试通过 app.dependency_overrides 替换了两个依赖：

  • get_settings：让本次测试进入 provider 分支。
  • get_provider_http_client：让 provider HTTP 调用使用 MockTransport，不访问外网。

这正是 FastAPI 依赖系统的强大之处：运行时代码用真实依赖，测试代码可以替换依赖边界。

12. 常见错误与诊断

错误一：让前端传 backend

后端选择是部署策略，不应该让普通请求随意控制。用服务端配置更清楚。

错误二：两个分支返回不同 JSON

无论 demo 还是 provider，公共响应都应该是 PredictionResponse。

错误三：测试直接改 .env

测试更适合覆盖 get_settings 依赖，这样不会污染本地环境。

错误四：配置值没有限制

用 Literal["demo", "provider"] 限制合法值，避免拼写错误进入运行时。

13. 面试视角：怎样解释后端选择开关

我会让 /predict 保持稳定的请求和响应契约，然后用配置项选择内部预测后端。默认 demo 方便本地开发和教学；provider 模式则通过 lifespan-managed HTTP client 调用外部 provider adapter。测试时我不会改真实环境变量，而是用 dependency_overrides 替换 settings 和 HTTP client。这样公共 API 稳定，内部实现可以按环境切换。

14. 小测：先回忆，再展开

1. 为什么后端选择用配置，而不是请求参数？

因为后端选择属于服务端部署策略，不应该让普通调用方随意控制内部实现。

2. 为什么 /predict 变成了 async def？

因为 provider 分支需要 await 异步 HTTP adapter；demo 分支仍可在 async endpoint 中调用轻量同步函数。

3. build_prediction_response() 解决了什么问题？

它让 demo 和 provider 两个分支统一转换成同一个 PredictionResponse，避免响应拼装重复和不一致。

4. 测试 provider 分支为什么要覆盖 get_settings？

因为测试需要临时切换到 provider 后端，但不应该修改真实 .env 或污染本地配置。

15. 本节总结

本节把 /predict 从单一 demo 实现升级成可配置后端入口：默认仍走 demo，配置为 provider 时走真实 provider adapter。外部调用方看到的请求体、响应体和错误契约保持稳定，内部实现则可以根据环境切换。

下一课建议学习：给预测调用加入更清楚的可观测信息，例如 provider backend、耗时、错误码在日志或响应调试字段中的表达。真实 AI 服务不只要能调用，还要能排查。
