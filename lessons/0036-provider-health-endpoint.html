Lesson 0036

Provider Health Endpoint：把外部服务状态翻译成稳定 API 契约

第 33 课写了 provider HTTP adapter，第 34 课用 lifespan 管理 HTTP client，第 35 课用 Depends 把 client 交给路由。本节把这三块拼起来：新增 GET /provider/health，用配置好的 provider health URL 检查外部服务，并把外部错误转换成稳定的 503 JSON 响应。

0. 本节学习目标

  • 理解 provider health endpoint 在 AI 服务项目中的作用。
  • 学会把 service adapter 接入 FastAPI endpoint。
  • 理解为什么外部服务失败通常要在 API 边界转换成稳定错误契约。
  • 理解为什么 health URL 应该来自配置，而不是来自用户请求参数。
  • 学会用 MockTransport 和依赖覆盖测试外部 HTTP 调用。

1. 背景：真实 AI endpoint 之前，先学会观测 provider

真实 AI 服务不是只写一个 /predict 就结束。只要接口背后依赖外部模型 provider，你就需要回答几个工程问题：

  • provider 当前能不能访问？
  • provider 返回 500、503 时，我们自己的 API 要怎么响应？
  • provider 超时时，客户端看到的是随机异常，还是稳定 JSON？
  • 测试时能不能不访问真实 provider？

本节新增的 GET /provider/health 不是最终业务功能，而是一个训练用的外部依赖观测端点。它让你先学会“如何把外部 HTTP 调用接进 FastAPI”，再进入真正的 AI 请求转发。

2. 核心概念：API 边界是翻译层

一个后端 API 往往夹在两类世界之间：

内部世界

Python 函数、dataclass、异常类型、HTTPX 错误、provider adapter、配置对象。

外部世界

HTTP 状态码、JSON 响应体、OpenAPI 文档、前端和第三方调用方能依赖的契约。

endpoint 的重要职责之一，就是把内部世界翻译成外部世界。比如本节的转换关系是：

ProviderHealthResult
  -> 200 ProviderHealthResponse

AIClientError("ai_provider_timeout")
  -> 503 {"detail": {"error_code": "...", "message": "..."}}

这比让 HTTPX 异常直接穿透到客户端要好得多。客户端不应该知道你的项目内部用了 HTTPX，也不应该被迫理解 Python 异常栈。

3. 安全设计：为什么不让用户传入 health URL

一个直觉写法可能是：

GET /provider/health?url=https://somewhere.example/health

学习时看起来很灵活，但真实项目里这是危险设计。因为这个接口会让你的服务器代替用户去请求任意地址，攻击者可能让它访问内网地址、本机管理端口、云厂商 metadata 地址等。这类问题通常叫 SSRF，也就是 Server-Side Request Forgery。

所以本节采用更稳的设计：health URL 来自服务端配置。

FIRST_API_AI_PROVIDER_HEALTH_URL="https://example.com"

用户只能调用 /provider/health，不能通过请求参数控制服务器去访问哪里。这是一个小项目里的小选择，但背后是很真实的安全边界意识。

4. 运行机制：一次请求怎么走完整链路

GET /provider/health
  -> FastAPI 匹配 system router
  -> Depends(get_settings) 读取配置
  -> Depends(get_provider_http_client) 获取 lifespan-managed AsyncClient
  -> 构造 AIClientConfig
  -> check_provider_health(url, config, client)
       -> await client.get(url)
       -> response.raise_for_status()
  -> 成功：返回 ProviderHealthResponse
  -> 失败：AIClientError 转 HTTPException(503)

这里把前几节的知识串起来了：配置管理、依赖注入、lifespan、app.state、async HTTP、错误映射、Pydantic 响应模型都在同一条路径上协作。

5. 代码精读一：配置 provider health URL

class Settings(BaseSettings):
    app_name: str = "First FastAPI API"
    environment: str = "development"
    api_key: str = "dev-secret-key"
    ai_provider: Literal["demo"] = "demo"
    ai_provider_health_url: str = "https://example.com"
    ai_timeout_seconds: float = Field(default=10.0, gt=0)
    ai_max_attempts: int = Field(default=1, ge=1, le=5)

ai_provider_health_url 是服务端配置项。它有两个作用：

  • 让不同环境可以用不同 provider 地址，例如本地、测试、生产。
  • 避免把请求目标交给用户输入，降低 SSRF 风险。

教学项目默认使用 https://example.com，只是为了让本地实验有一个稳定的 200 响应占位。真实项目里你会把它改成 provider 官方给出的 health 或 models endpoint。

6. 代码精读二：成功响应模型

class ProviderHealthResponse(BaseModel):
    ok: bool
    provider: str
    status_code: int
    message: str

这个模型把 provider 检查结果压成一个稳定 JSON 形状：

{
  "ok": true,
  "provider": "demo",
  "status_code": 200,
  "message": "The AI provider responded successfully."
}

注意它没有把原始响应体直接返回给客户端。health endpoint 只回答“能不能访问”和“状态如何”，不负责暴露 provider 的所有细节。

7. 代码精读三：endpoint 作为翻译层

@router.get(
    "/provider/health",
    response_model=ProviderHealthResponse,
    responses={503: {"model": ErrorResponse}},
)
async def read_provider_health(
    settings: Annotated[Settings, Depends(get_settings)],
    provider_http_client: Annotated[
        httpx.AsyncClient,
        Depends(get_provider_http_client),
    ],
) -> ProviderHealthResponse:
    config = AIClientConfig(
        provider=settings.ai_provider,
        timeout_seconds=settings.ai_timeout_seconds,
        max_attempts=settings.ai_max_attempts,
    )

    try:
        result = await check_provider_health(
            url=settings.ai_provider_health_url,
            config=config,
            client=provider_http_client,
        )
    except AIClientError as exc:
        raise HTTPException(
            status_code=503,
            detail=ErrorDetail(
                error_code=exc.error_code,
                message=exc.message,
            ).model_dump(),
        ) from exc

    return ProviderHealthResponse(
        ok=result.ok,
        provider=result.provider,
        status_code=result.status_code,
        message=result.message,
    )

这段代码可以分成四层理解：

  • 依赖层：settings 和 provider_http_client 都由 FastAPI 注入。
  • 配置层：把 settings 转成 adapter 需要的 AIClientConfig。
  • 调用层：把配置 URL、配置对象和长期复用的 client 传给 check_provider_health()。
  • 翻译层：成功返回响应模型，失败抛出 HTTP 503。

endpoint 没有直接写 await provider_http_client.get(...)，这是有意的。外部 HTTP 细节已经属于 adapter 的职责；endpoint 负责 HTTP API 契约。

8. 代码精读四：测试不用访问真实 provider

def test_provider_health_endpoint_uses_dependency_injected_client() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(200)

    replacement_client = httpx.AsyncClient(
        transport=httpx.MockTransport(handler),
        trust_env=False,
    )
    app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

    try:
        with TestClient(app) as client:
            response = client.get("/provider/health")
    finally:
        app.dependency_overrides.clear()
        asyncio.run(replacement_client.aclose())

这段测试同时使用了两个技术点：

  • MockTransport：让 HTTPX 请求不出网，而是交给测试里的 handler 返回假响应。
  • dependency_overrides：让 endpoint 使用测试 client，而不是 lifespan 创建的真实 client。

这就是前几节铺垫的价值。因为 route 依赖的是 get_provider_http_client()，所以测试可以替换它；因为 adapter 接收外部传入的 client，所以它不会强行创建真实网络连接。

9. 错误路径：provider 503 怎么变成我们自己的 503

def test_provider_health_endpoint_maps_provider_errors_to_503() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(503)

    replacement_client = httpx.AsyncClient(
        transport=httpx.MockTransport(handler),
        trust_env=False,
    )
    app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

    try:
        with TestClient(app) as client:
            response = client.get("/provider/health")
    finally:
        app.dependency_overrides.clear()
        asyncio.run(replacement_client.aclose())

    assert response.status_code == 503
    assert response.json() == {
        "detail": {
            "error_code": "ai_provider_http_error",
            "message": "The AI provider returned HTTP 503.",
        }
    }

这里 provider 返回 503，HTTPX 的 raise_for_status() 会触发状态码异常，adapter 把它转换成 AIClientError，endpoint 再转换成 FastAPI 的 HTTPException。

为什么我们自己的 API 也返回 503？因为从客户端视角看，这个服务当前依赖的上游不可用。MDN 对 503 的定义就是服务暂时无法处理请求。对于 AI provider 暂时不可用、超时、维护等场景，503 比 500 更能表达“依赖服务不可用”的语义。

11. 常见错误与诊断

错误一：把 provider URL 做成用户参数

这会让服务器代替用户请求任意地址，真实项目中可能引入 SSRF 风险。优先使用服务端配置。

错误二：endpoint 里直接写外部 HTTP 细节

这样会让路由变胖。更清楚的方式是 adapter 负责外部调用，endpoint 负责 API 契约转换。

错误三：测试真的访问外网

外网测试慢且不稳定。用 MockTransport 模拟 provider 响应更适合单元和接口测试。

错误四：失败时返回杂乱异常

客户端需要稳定错误码和消息。把内部异常转换成固定 JSON 契约，前端和调用方才好处理。

12. 面试视角：怎样解释 provider health endpoint

我会把 provider health endpoint 设计成一个 API 边界层。provider URL 来自服务端配置，避免让用户控制请求目标；HTTP client 由 lifespan 创建并通过 Depends 注入；外部调用放在 adapter 中；endpoint 只负责把成功结果转换成响应模型，把 provider 超时、HTTP 错误或连接错误转换成稳定的 503 JSON。测试时用 dependency_overrides 和 MockTransport 替换真实网络。

13. 小测：先回忆，再展开

1. 为什么 /provider/health 不接收用户传入的 URL？

因为让服务器请求用户指定的任意地址可能引入 SSRF 风险。health URL 更适合来自服务端配置。

2. 本节 endpoint 为什么要捕获 AIClientError？

因为 AIClientError 是内部服务层错误，endpoint 需要把它翻译成客户端能理解的 HTTP 状态码和 JSON 错误契约。

3. MockTransport 在测试中解决了什么问题？

它让 HTTPX 请求不访问真实网络，而是由测试代码返回预设响应，从而让测试快速、稳定、可重复。

4. 为什么 provider 不可用时返回 503，而不是随便返回 500？

503 更准确表示服务暂时无法处理请求，适合上游 provider 不可用、超时或维护这类依赖失败场景。

14. 本节总结

本节把第 33 到第 35 课的铺垫连成了一条真实链路：配置提供 provider URL，lifespan 提供长期 HTTP client，Depends 把 client 注入 endpoint，adapter 负责外部 HTTP 调用，endpoint 负责把内部结果和错误翻译成稳定 API 契约。
