Provider Health Endpoint：配置 URL、依赖 client 与稳定 503 错误契约

GET /provider/health 把前几节串成真实链路：Settings 提供 provider health URL，lifespan 提供 AsyncClient，Depends 注入资源，adapter 调用外部 HTTP，endpoint 把内部结果或 AIClientError 翻译成稳定 API 响应。

① 本节核心知识框架

Provider Health Endpoint
├─ 配置：FIRST_API_AI_PROVIDER_HEALTH_URL
├─ 依赖：Depends(get_settings)
├─ 依赖：Depends(get_provider_http_client)
├─ 配置转换：Settings -> AIClientConfig
├─ Adapter：check_provider_health(url, config, client)
├─ 成功响应：ProviderHealthResponse
├─ 失败响应：HTTPException(503) + ErrorResponse
├─ 安全原则：URL 来自服务端配置，避免用户控制 SSRF 风险
└─ 测试：MockTransport + dependency_overrides

② 核心概念底层原理

health endpoint 是外部依赖观测边界。它不执行业务预测，只回答 provider 当前是否可访问，并把外部 HTTP 成功/失败压缩成内部 API 可理解的结果。

provider URL 必须来自服务端配置，而不是用户请求参数。让用户控制服务器访问的 URL，可能形成 SSRF 风险：攻击者诱导服务器请求内网地址、本机端口或云 metadata 服务。

endpoint 是 HTTP 契约翻译层。adapter 抛 AIClientError，endpoint 将其转换成 HTTPException(status_code=503, detail={error_code,message})。客户端不需要知道 HTTPX、provider 原始异常或内部调用栈。

③ 全套代码逐行精读解析

ai_provider_health_url: str = "https://example.com"

该字段位于 Settings，通过 FIRST_API_AI_PROVIDER_HEALTH_URL 配置。默认值只是教学占位；生产中应指向 provider 官方 health/models endpoint。

class ProviderHealthResponse(BaseModel):
    ok: bool
    provider: str
    status_code: int
    message: str
  • ok：项目内部判断 provider 是否健康。
  • provider：标记被检查的 provider。
  • status_code：保留外部 HTTP 状态码，便于排障。
  • message：安全、稳定的人类可读说明。
@router.get(
    "/provider/health",
    response_model=ProviderHealthResponse,
    responses={503: {"model": ErrorResponse}},
    summary="Check the configured AI provider health URL",
)
  • response_model 声明成功响应 schema。
  • responses={503: ...} 声明失败响应 schema；错误契约也必须文档化。
  • summary 进入 OpenAPI，说明该端点检查配置好的 provider health URL。
async def read_provider_health(
    settings: Annotated[Settings, Depends(get_settings)],
    provider_http_client: Annotated[httpx.AsyncClient, Depends(get_provider_http_client)],
) -> ProviderHealthResponse:

两个依赖分别提供配置和 HTTP client。endpoint 不直接创建 client，也不读取环境变量，职责保持清晰。

config = AIClientConfig(
    provider=settings.ai_provider,
    timeout_seconds=settings.ai_timeout_seconds,
    max_attempts=settings.ai_max_attempts,
)

把 Settings 转成 adapter 需要的 config，避免 provider service 依赖整个应用配置对象。

try:
    result = await check_provider_health(
        url=settings.ai_provider_health_url,
        config=config,
        client=provider_http_client,
    )
except AIClientError as exc:
    raise HTTPException(
        status_code=503,
        detail=ErrorDetail(error_code=exc.error_code, message=exc.message).model_dump(),
    ) from exc
  • await check_provider_health 调用 async adapter。
  • url=settings.ai_provider_health_url 使用服务端配置，避免用户传 URL。
  • client=provider_http_client 复用 lifespan-managed client。
  • except AIClientError 捕获内部边界错误。
  • HTTPException(503) 将依赖不可用映射为 HTTP 语义。
  • ErrorDetail(...).model_dump() 保证错误体结构稳定且可 JSON 序列化。

④ 核心机制运行全流程拆解

GET /provider/health
  -> FastAPI 解析路由
  -> Depends(get_settings)
  -> Depends(get_provider_http_client)
       -> request.app.state.provider_http_client
  -> Settings -> AIClientConfig
  -> check_provider_health(configured_url, config, client)
       -> await client.get(url)
       -> raise_for_status()
  -> 成功：ProviderHealthResult -> ProviderHealthResponse -> 200
  -> 失败：AIClientError -> HTTPException 503 -> ErrorResponse

⑥ 开发常见报错+坑点+解决方案

  • provider 503 返回成 500：未捕获 AIClientError。解决：endpoint 映射为 HTTPException(503)。
  • 测试访问真实 example.com：未覆盖 HTTP client。解决：用 dependency_overrides[get_provider_http_client]。
  • SSRF 风险：用户控制 URL。解决：health URL 固定在服务端配置中。
  • OpenAPI 不显示错误模型：缺少 responses={503: {"model": ErrorResponse}}。
  • client 被关闭后复用：测试替换 client 生命周期错误。解决：测试结束清理 overrides 并关闭替换 client。

⑧ 课后记忆习题+标准答案

1. 为什么 health URL 不让用户传？

用户控制服务端请求目标可能导致 SSRF，health URL 应由服务端配置固定。

2. endpoint 为什么捕获 AIClientError 而不是 HTTPX 异常？

HTTPX 异常已由 adapter 翻译为内部错误，endpoint 只处理项目自己的错误契约。

3. provider 不可用为什么返回 503？

请求本身可能合法，但服务依赖的上游暂不可用，503 更准确表达临时不可用。

4. MockTransport 在本节测试什么？

它模拟 provider 成功或失败响应，验证 endpoint 和 adapter 的转换逻辑，不访问真实网络。

⑨ 面试高频真题+解析

题 1：如何设计依赖外部 provider 的 health endpoint？

URL 来自服务端配置，HTTP client 由 lifespan 管理并依赖注入，adapter 负责外部调用和错误映射，endpoint 返回稳定成功模型或 503 错误模型。

题 2：什么是 SSRF，为什么本节要避免用户传 URL？

SSRF 是攻击者诱导服务器请求非预期地址。若 health endpoint 接收任意 URL，可能访问内网、metadata 或管理端口。

题 3：为什么 adapter 和 endpoint 错误转换不合并？

adapter 处理外部协议到内部错误，endpoint 处理内部错误到 HTTP 响应。分层后 service 可复用，HTTP 契约也更清晰。
