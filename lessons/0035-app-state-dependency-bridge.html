app.state 依赖桥：把应用级资源封装成可替换 Depends

app.state 能存资源，但 endpoint 不应该到处直接读取内部状态。本节用 get_provider_http_client(request) 把 lifespan-managed client 封装成 FastAPI 依赖，让路由声明“我需要一个 AsyncClient”，测试通过 dependency_overrides 替换资源。

① 本节核心知识框架

app.state Dependency Bridge
├─ lifespan：拥有资源生命周期
├─ app.state：保存应用级资源
├─ dependency：get_provider_http_client(request)
│  └─ return request.app.state.provider_http_client
├─ endpoint：Depends(get_provider_http_client)
├─ 测试：app.dependency_overrides
└─ 设计原则：资源所有权、存储位置、使用点分离

② 核心概念底层原理

资源所有权和资源使用权应分离。lifespan 负责创建与关闭 HTTP client，app.state 负责保存资源，依赖函数负责把资源暴露给路由，endpoint 只使用资源。

如果路由到处写 request.app.state.provider_http_client，资源存储细节会扩散。未来改成容器、provider manager 或多 client 路由时，需要改大量 endpoint。依赖函数把这个细节集中到一个出口。

FastAPI 依赖函数也能接收 Request。因此依赖函数可以读取 request.app.state，再把结果作为普通参数注入 endpoint。测试可以覆盖依赖，而无需篡改 app 内部状态。

③ 全套代码逐行精读解析

def get_provider_http_client(request: Request) -> httpx.AsyncClient:
    return request.app.state.provider_http_client
  • request: Request：当前 HTTP 请求对象，可访问应用实例。
  • request.app：FastAPI/Starlette 应用对象。
  • request.app.state：上一课 lifespan 存放应用级资源的位置。
  • provider_http_client：长期复用的 AsyncClient。
  • 函数本身不创建、不关闭 client，只负责取出资源；否则会混淆所有权。
def read_provider_http_client_state(
    request: Request,
    provider_http_client: Annotated[
        httpx.AsyncClient,
        Depends(get_provider_http_client),
    ],
) -> ProviderHttpClientStateResponse:
  • Annotated 同时表达 Python 类型和 FastAPI 依赖来源。
  • httpx.AsyncClient 让编辑器和读者知道参数类型。
  • Depends(get_provider_http_client) 让 FastAPI 在调用 endpoint 前先解析依赖。
  • 保留 request 是为了读取 provider/timeout 元信息；业务 endpoint 通常应进一步减少直接 state 读取。
app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

这行是测试价值所在：只要 endpoint 通过依赖拿 client，测试就能替换它。若 endpoint 直接读 app.state，覆盖依赖不会生效。

④ 核心机制运行全流程拆解

HTTP request
  -> FastAPI 匹配 /provider/http-client
  -> 发现参数 provider_http_client 依赖 get_provider_http_client
  -> 调用 get_provider_http_client(request)
       -> request.app.state.provider_http_client
  -> 把返回的 AsyncClient 注入 endpoint
  -> endpoint 读取 client.is_closed
  -> 返回 ProviderHttpClientStateResponse

测试路径
  -> dependency_overrides 替换 get_provider_http_client
  -> endpoint 收到 replacement_client
  -> 验证路由使用依赖入口

⑤ 重难点、易混淆点对比辨析

层次职责不应承担
lifespan创建/关闭资源处理业务请求
app.state保存应用级资源保存请求级状态
dependency提供资源入口拥有长期资源生命周期
endpoint表达 HTTP 契约知道所有内部存储细节
test override替换依赖污染后续测试

⑥ 开发常见报错+坑点+解决方案

  • 依赖覆盖无效：endpoint 未使用 Depends，直接读 state。解决：资源读取封装成依赖。
  • 测试互相污染：覆盖后未清理。解决：finally: app.dependency_overrides.clear()。
  • 依赖函数创建新 client：所有权混乱。解决：依赖只取资源，lifespan 负责创建和关闭。
  • state 属性不存在：lifespan 未执行。解决：用 TestClient context 或确保服务器正常启动。
  • 业务代码依赖 state 路径：存储细节扩散。解决：统一从依赖函数拿资源。

⑧ 课后记忆习题+标准答案

1. 为什么不要在多个 endpoint 里直接读 request.app.state.xxx？

这会扩散资源存储细节，降低可测试性和可演进性；依赖函数能集中封装资源入口。

2. get_provider_http_client 是否拥有 client 生命周期？

不拥有。它只从 app.state 取出 client，生命周期仍由 lifespan 管理。

3. dependency_overrides 为什么能测试替换资源？

FastAPI 会用 overrides 中的函数替代原依赖，因此 endpoint 收到测试提供的资源。

4. Annotated[T, Depends(...)] 如何理解？

T 是 Python 类型，Depends(...) 是 FastAPI 运行时参数来源。

⑨ 面试高频真题+解析

题 1：如何把 lifespan 创建的资源注入 endpoint？

startup 时放入 app.state，写依赖函数从 Request 中读取 app.state，再用 Depends 注入 endpoint。测试时可覆盖依赖。

题 2：为什么区分资源所有权和使用权？

所有权决定创建和关闭，使用权决定业务调用。混在一起会导致泄漏、重复创建、难测试和难替换。

题 3：依赖函数和 service 函数有什么不同？

依赖函数负责为请求解析参数或资源；service 函数表达业务用例。依赖可以调用 service，但二者职责不同。
