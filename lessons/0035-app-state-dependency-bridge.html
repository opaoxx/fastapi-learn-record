Lesson 0035

把 app.state 变成依赖：从“能访问”走向“会设计”

第 34 课我们已经能在 lifespan 里创建长期复用的 httpx.AsyncClient，并把它放进 app.state。本节要补上更像真实项目的一步：不要让每个路由都直接伸手去拿 request.app.state，而是把应用级资源封装成一个 FastAPI 依赖。

0. 本节学习目标

  • 理解 lifespan、app.state、Depends 三者的职责边界。
  • 理解为什么“资源所有权”和“资源使用权”应该分开设计。
  • 学会编写从 Request 中读取应用级资源的依赖函数。
  • 理解依赖覆盖 app.dependency_overrides 如何让外部资源在测试中可替换。
  • 能回答面试问题：为什么不要在路由里到处散落 request.app.state.xxx？

1. 背景：上一课的代码“能用”，但还不够好

第 34 课的教学端点大概是这样写的：

def read_provider_http_client_state(request: Request) -> ProviderHttpClientStateResponse:
    provider_http_client = request.app.state.provider_http_client
    return ProviderHttpClientStateResponse(
        provider=request.app.state.provider_http_client_provider,
        timeout_seconds=request.app.state.provider_http_client_timeout_seconds,
        client_ready=not provider_http_client.is_closed,
    )

这段代码没有错。FastAPI 的 Request 可以拿到底层 ASGI 应用对象，应用对象上有 state，所以路由确实能访问 lifespan 保存的资源。

问题在于：如果后面的十几个路由都这样写，项目会慢慢出现三个坏味道。

  • 知识散落：每个路由都要知道资源存在 app.state.provider_http_client 这个具体位置。
  • 测试困难：测试想替换 client 时，必须修改 app 的内部状态，而不是替换一个清楚的依赖入口。
  • 演进困难：如果以后资源从 app.state 迁移到容器、工厂或 provider manager，所有路由都可能要改。

真实后端项目最怕的不是“某一行代码不好看”，而是内部实现细节到处扩散。扩散之后，未来改动就会从局部修改变成全局搜刮。

2. 核心概念：资源所有权 vs 资源使用权

本节的关键不是语法，而是边界。

资源所有权

谁负责创建资源、保存资源、关闭资源。对于长期复用的 HTTP client，本项目交给 lifespan。

资源使用权

某个 endpoint 或 service 在处理请求时需要使用资源。使用者不应该关心资源究竟存在哪里、什么时候创建、什么时候关闭。

这就是本节要建立的心智模型：

lifespan owns the resource
app.state stores the resource
Depends exposes the resource
endpoint uses the resource

换句话说：app.state 是储藏室，不是每个业务函数都应该直接翻找的抽屉。Depends 是对外提供资源的稳定入口。

3. FastAPI 运行机制：依赖函数是请求处理前的准备步骤

FastAPI 官方文档把依赖注入描述为：声明路径操作函数需要什么，FastAPI 在处理请求时会帮你执行这些依赖，并把返回值传给函数。我们可以把一次请求想成下面这条链路：

HTTP request
  -> route matching
  -> resolve dependencies
       -> get_provider_http_client(request)
       -> return app.state.provider_http_client
  -> call endpoint(provider_http_client=...)
  -> serialize response

这里有一个容易被初学者忽略的点：依赖函数本身也可以接收 Request。所以我们不需要在 endpoint 里直接读 request.app.state，可以把这个动作移到依赖函数里。

这样 endpoint 的签名就从“我知道内部状态在哪里”变成“我需要一个 HTTP client”。这个变化很小，但设计含义很大。

4. 本节项目改动概览

  • 在 first_api/dependencies.py 中新增 get_provider_http_client()。
  • 让 GET /provider/http-client 通过 Depends 接收 httpx.AsyncClient。
  • 补充测试：用 app.dependency_overrides 替换这个依赖，证明路由已经通过依赖入口获取 client。

注意，本节没有急着发起真实外部 HTTP 请求。原因很简单：先把“资源怎样进入路由”讲清楚，再接真实 provider endpoint，学习负担会低很多。

5. 代码精读一：依赖函数作为 app.state 的出口

from typing import Annotated

import httpx
from fastapi import Query, Request

from .schemas import ItemCategory


def get_provider_http_client(request: Request) -> httpx.AsyncClient:
    return request.app.state.provider_http_client

这一小段代码值得逐行看。

  • Request 不是请求体模型，而是当前 HTTP 请求的运行时对象。
  • request.app 指向当前 FastAPI/Starlette 应用对象。
  • request.app.state 是应用级状态容器，上一课的 lifespan 已经把 provider_http_client 放进去了。
  • 函数返回值标注为 httpx.AsyncClient，这让路由签名和编辑器提示都更清楚。

这个函数非常短，但它把一个重要事实隔离起来：资源当前存储在 app.state.provider_http_client。以后如果换存储方式，优先改这个函数，而不是改所有路由。

6. 代码精读二：路由声明“我需要一个 client”

@router.get(
    "/provider/http-client",
    response_model=ProviderHttpClientStateResponse,
    summary="Inspect the lifespan-managed provider HTTP client",
)
def read_provider_http_client_state(
    request: Request,
    provider_http_client: Annotated[
        httpx.AsyncClient,
        Depends(get_provider_http_client),
    ],
) -> ProviderHttpClientStateResponse:
    return ProviderHttpClientStateResponse(
        provider=request.app.state.provider_http_client_provider,
        timeout_seconds=request.app.state.provider_http_client_timeout_seconds,
        client_ready=not provider_http_client.is_closed,
    )

Annotated[httpx.AsyncClient, Depends(get_provider_http_client)] 可以读成一句话：

这个参数在 Python 类型上是 httpx.AsyncClient，在 FastAPI 运行时由 get_provider_http_client 这个依赖函数提供。

这就是 FastAPI 里 Annotated 常见的双层含义：第一层是给 Python 和编辑器看的类型，第二层是给 FastAPI 看的额外参数来源或校验规则。

这里还保留了 request: Request，因为这个教学端点要展示 provider 和 timeout 元信息，它们也存在 app.state 中。后面的真实 provider 调用端点，通常只需要注入 client 和 settings，不一定要让路由自己读取这么多内部状态。

7. 代码精读三：依赖覆盖让测试不碰真实资源

def test_provider_http_client_dependency_can_be_overridden() -> None:
    replacement_client = httpx.AsyncClient(timeout=0.1, trust_env=False)
    asyncio.run(replacement_client.aclose())
    app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

    try:
        with TestClient(app) as client:
            response = client.get("/provider/http-client")
    finally:
        app.dependency_overrides.clear()

    assert response.status_code == 200
    assert response.json()["client_ready"] is False

这个测试有点“故意刁钻”，但非常适合学习依赖注入。

  • 先创建一个 replacement_client。
  • 马上关闭它，让它的 is_closed 变成 True。
  • 用 app.dependency_overrides 把 get_provider_http_client 替换成返回这个关闭 client 的函数。
  • 请求接口后，如果响应里的 client_ready 是 false，说明路由确实使用了被覆盖的依赖。

如果路由仍然直接使用 request.app.state.provider_http_client，这个测试就不会观察到替换 client 的关闭状态。

8. 为什么这对 AI 服务特别重要

AI 服务端点经常会依赖外部资源：

  • 模型 provider 的 HTTP client。
  • embedding 服务 client。
  • 向量数据库连接。
  • 对象存储 client。
  • 限流器、缓存、队列连接。

如果所有 endpoint 都直接读内部状态，项目后期会很难替换真实 provider、demo provider、测试 fake provider。依赖函数的价值就是把“外部资源”变成一个清晰的参数来源。

这是从玩具项目走向工程项目的关键一步：业务函数应该依赖抽象的能力，而不是到处依赖资源藏在哪里。

10. 常见错误与诊断

错误一：在 endpoint 里到处写 app.state

短期方便，长期会让存储细节扩散。优先把它封装成依赖函数。

错误二：依赖函数偷偷创建新 client

本节的依赖函数只“取出”资源，不“拥有”资源。长期 client 的创建和关闭仍然属于 lifespan。

错误三：测试后忘记清理 overrides

app.dependency_overrides 是全局字典。测试结束要清理，否则会影响后面的测试。

错误四：把请求级数据放进 app.state

app.state 是应用级共享状态，不适合保存某个用户、某次请求独有的数据。

11. 面试视角：怎样解释这个设计

我会让 lifespan 管理长期资源的生命周期，例如创建和关闭 HTTP client；用 app.state 保存应用级资源；再用依赖函数把资源暴露给路由。这样 endpoint 不需要知道资源存储细节，测试时也可以通过 dependency_overrides 替换资源。这个设计把资源所有权、存储位置和业务使用点分开了。

这段回答比“我用 Request 能拿到 app.state”更像工程师，因为它解释了为什么这样设计，而不只是解释能不能访问。

12. 小测：先回忆，再展开

1. 本节中，谁拥有 AsyncClient 的生命周期？

lifespan。它在应用启动时创建 client，在应用关闭时关闭 client。

2. get_provider_http_client() 的主要职责是什么？

从当前请求的 request.app.state 中取出应用级 HTTP client，并把它作为依赖返回给路由。

3. 为什么不要在很多路由里直接写 request.app.state.provider_http_client？

因为这会让资源存储细节扩散，降低可测试性和可演进性；封装成依赖后，路由只声明自己需要一个 client。

4. app.dependency_overrides 在测试中解决了什么问题？

它允许测试用 fake、mock 或替换资源覆盖真实依赖，从而避免测试依赖真实外部服务或真实应用资源。

13. 本节总结

本节没有增加新的业务接口，但它修正了一个非常重要的架构姿势：应用级资源可以保存在 app.state，但 endpoint 不应该到处直接读取内部状态。更好的方式是用依赖函数提供资源，让路由通过 Depends 声明自己需要什么。
