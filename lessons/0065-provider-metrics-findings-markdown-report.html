0065 Provider Metrics Findings Markdown Report：把排障判断渲染成可复盘表格

本节新增 render_provider_metrics_runbook_findings_markdown()，把上一节生成的 findings 渲染成 Markdown 表格。它不改变指标、不读取 counter、不做告警，只负责把机器结构化判断变成适合 PR、学习记录和 runbook 复盘的文本报告。

① 本节核心知识框架

Findings Markdown Report
├─ 输入：runbook findings
│  ├─ outcome
│  ├─ count
│  ├─ severity
│  ├─ finding
│  └─ next_action
├─ 输出：Markdown table
│  ├─ header row
│  ├─ alignment row
│  └─ one row per finding
├─ 设计目标
│  ├─ 让排障判断可复制到 PR/issue/学习记录
│  ├─ 保持输出确定性，便于测试
│  └─ 不把人类解释写回 metric label
└─ 验证点
   ├─ 表头稳定
   ├─ 行顺序稳定
   ├─ severity 文案稳定
   └─ next_action 不丢失

② 核心概念底层原理

Markdown report 是解释层的展示格式。metrics 适合机器聚合，findings 适合人类理解，Markdown report 适合传播和复盘。三者分层后，任何一层变更都能被测试定位。

表格格式的优势是列稳定：读者可以横向比较 outcome、severity 和 next_action。相较长段落，它更适合期末复习、面试回答和 GitHub 文档。

本函数默认输入已经通过 build_provider_metrics_runbook_findings() 生成，因此依赖字段契约：outcome、count、severity、finding、next_action 必须存在。若直接传入手写字典并漏字段，Python 会在索引处抛出 KeyError；这不是渲染层要兜底的业务错误，而是调用方违反结构化数据契约。

层级数据形态主要读者
Metric sample{labels,count}程序、测试、监控系统
Finding{severity,finding,next_action}开发者、排障者
Markdown report文本表格课程读者、PR reviewer、复盘记录

③ 全套代码逐行精读解析

def render_provider_metrics_runbook_findings_markdown(
    findings: list[dict[str, object]],
) -> str:
    lines = [
        "| outcome | count | severity | finding | next_action |",
        "| --- | ---: | --- | --- | --- |",
    ]

函数名使用 render，表示把结构化数据转换为文本。返回值是 str，说明它是展示层函数，不应再承载业务状态。

第二行 ---: 让 count 右对齐，符合数字列阅读习惯。若表头不稳定，测试和文档引用都会漂移。

for finding in findings:
    lines.append(
        "| {outcome} | {count} | {severity} | {finding_text} | {next_action} |".format(
            outcome=finding["outcome"],
            count=finding["count"],
            severity=finding["severity"],
            finding_text=finding["finding"],
            next_action=finding["next_action"],
        )
    )
return "\n".join(lines) + "\n"

每个 finding 对应一行，保留原顺序，保证 report 与输入样本可一一追溯。最后补换行，便于写入文件或拼接到 Markdown 文档。

代码点作用不这样写的隐患
lines = [...]先写固定表头和对齐行，建立输出骨架。若表头在循环中拼接，空输入时可能没有合法 Markdown 表格。
finding["outcome"]读取排障分支，保持与 metric outcome 对齐。若改用 get() 静默返回 None，会把上游契约错误伪装成合法报告。
finding["count"]保留观测规模，避免只看 severity 导致误判。丢掉 count 后，读者无法区分偶发样本和批量故障。
finding_text=finding["finding"]避开 finding 变量名与字段名混淆。若格式变量也叫 finding，阅读时容易误以为传入整张字典。
+ "\n"让输出符合文本文件和 Markdown 拼接习惯。缺少末尾换行时，后续追加内容可能黏在表格最后一行。

④ 核心机制运行全流程拆解

scenario samples
  → build_provider_metrics_runbook_findings(samples)
  → render_provider_metrics_runbook_findings_markdown(findings)
  → Markdown table
  → PR / issue / lesson / runbook review

调用链保持单向：样本到 finding，finding 到 report。report 不反向修改样本，也不触发任何 provider 请求。

步骤输入输出风险控制
构造samplesfindings分支测试保护
渲染findingsMarkdown完整字符串测试保护
传播Markdown文档片段不进入 metrics label

数据流的关键约束是“只降维、不回写”：metric sample 是事实，finding 是解释，Markdown 是表达。表达层最多改变排版，不能改变 outcome、count 或 next_action，否则同一份观测数据会在不同展示入口产生不同结论。

⑧ 课后记忆习题+标准答案

1. 为什么本节选择 Markdown 表格？

表格能稳定展示 outcome、count、severity、finding、next_action，适合复习、review 和复盘。

2. 为什么 report 函数不读取全局 counter？

读取全局状态会让输出不可复现；report 应只渲染传入 findings。

3. 为什么 finding 文本不能进入 metric label？

metric label 要低基数且稳定，finding 是人类解释文本，变化频繁且过长。

4. 为什么要测试完整 Markdown 输出？

报告格式本身是课程和文档契约，完整测试能保护列名、顺序和动作建议。

5. 如果 finding 中出现竖线 |，生产环境应如何处理？

应在渲染前做 Markdown 转义，或改用 HTML table / JSON payload 等结构化输出；不能让原始文本直接破坏表格列。

⑨ 面试高频真题+解析

面试题 1：监控 findings 如何进入工程协作流程？

先从 metrics 生成结构化 findings，再渲染成稳定文本或 dashboard panel，进入 PR、issue、incident record。展示层不能反向污染采集层。

面试题 2：为什么 report renderer 应该是纯函数？

纯函数可测试、可复现、无环境依赖。监控报告需要稳定 diff，不能依赖当前进程状态。

面试题 3：Markdown 报告和生产 dashboard 的关系是什么？

Markdown 报告适合教学和文本协作；生产 dashboard 适合实时查询和可视化。两者都消费观测数据，但服务不同场景。
