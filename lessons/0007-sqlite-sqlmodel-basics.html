SQLite 与 SQLModel：从内存变量到关系型持久化

本节把“数据为什么要进数据库”讲透：SQLite 提供本地关系型存储，SQLModel 把 Python 类型、Pydantic 校验和 SQLAlchemy ORM 映射合并到同一套模型体系中。理解这一节，后续 CRUD、测试隔离、任务状态持久化才有底层依据。

① 本节核心知识框架

持久化基础
├─ 内存变量：进程内状态，重启即丢失
├─ SQLite：单文件关系数据库，适合教学、本地开发、轻量项目
├─ SQLModel
│  ├─ SQLModel 基类：兼具 Pydantic 与 SQLAlchemy 能力
│  ├─ table=True：声明数据库表模型
│  ├─ Field(primary_key=True)：主键列
│  └─ Field(index=True)：索引提示
├─ Engine：数据库连接工厂
├─ metadata.create_all(engine)：按模型元数据建表
├─ seed_items：空库初始化样例数据
└─ lifespan：应用启动阶段执行建表和种子数据

② 核心概念底层原理

持久化的本质是把状态移出 Python 进程。内存字典只存在于当前解释器进程；热重载、重启、崩溃都会丢失。数据库把状态写入独立存储介质，应用进程只负责读写它。

SQLite 是嵌入式关系数据库。它不需要单独数据库服务，数据库就是一个本地文件。它支持 SQL、事务、表、索引和约束，适合教学和小型工具；但高并发写入、分布式部署、复杂权限不是它的强项。

SQLModel 不是“另一个 Pydantic”。它建立在 Pydantic 和 SQLAlchemy 之上：Pydantic 负责类型与数据校验，SQLAlchemy 负责 ORM 映射和 SQL 执行，SQLModel 用更接近 FastAPI 的类型写法把两者合并。

表模型与请求/响应模型要区分。Item 是数据库表模型，携带 table=True 和主键；ItemCreate 是输入模型；ItemRead 是输出模型。把三者混成一个模型会导致客户端能提交不该提交的字段，也容易泄漏内部字段。

③ 全套代码逐行精读解析

class ItemBase(SQLModel):
    name: str
    category: str = Field(index=True)
    price: float = Field(ge=0)
    in_stock: bool

class Item(ItemBase, table=True):
    id: int | None = Field(default=None, primary_key=True)
  • ItemBase(SQLModel)：抽出商品公共字段，避免表模型、创建模型、读取模型重复声明。
  • name: str：商品名称列。当前表模型不放长度约束，创建模型里会进一步限制。
  • category: str = Field(index=True)：数据库中按字符串保存分类，并声明索引意图。若经常按分类查询，索引能减少扫描成本。
  • price: float = Field(ge=0)：非负约束在模型层防止非法价格进入数据库。
  • in_stock: bool：库存状态，映射为数据库布尔兼容类型。
  • Item(..., table=True)：告诉 SQLModel 这个类对应真实数据库表。没有 table=True，create_all 不会为它建表。
  • id: int | None：创建前为 None，提交后由数据库生成整数主键。
  • primary_key=True：声明主键列。没有主键，ORM 很难稳定追踪对象身份。
DATABASE_URL = f"sqlite:///{Path(__file__).with_name('first_api.db')}"
engine = create_engine(
    DATABASE_URL,
    connect_args={"check_same_thread": False},
)
  • Path(__file__).with_name('first_api.db')：把数据库文件固定到 first_api/first_api.db，不依赖启动命令所在目录。
  • sqlite:///...：SQLAlchemy 数据库 URL，三条斜杠表示本地文件路径。
  • create_engine(...)：创建 engine。engine 不是一次查询连接，而是连接与方言配置入口。
  • check_same_thread=False：SQLite 默认限制连接只在创建线程使用；FastAPI/TestClient 场景会跨线程访问，教学项目需要关闭该检查。生产不是靠这个解决并发，应使用合适数据库和连接池。
def create_db_and_tables() -> None:
    SQLModel.metadata.create_all(engine)
    migrate_runtime_schema()
    with Session(engine) as session:
        seed_items(session)
  • SQLModel.metadata：收集所有 table=True 模型的表结构元数据。
  • create_all(engine)：缺表则建表；已有表通常不会自动执行复杂迁移。
  • migrate_runtime_schema()：教学项目里的轻量运行时迁移，用于补旧表字段；生产应使用 Alembic。
  • with Session(engine)：打开短生命周期会话执行种子数据逻辑。
  • seed_items(session)：空库插入三条样例，保证新手启动后 GET /items 有可观察结果。

④ 核心机制运行全流程拆解

应用启动
  -> FastAPI lifespan 开始
  -> create_db_and_tables()
  -> SQLModel.metadata.create_all(engine)
  -> SQLite 文件 first_api.db 不存在则创建
  -> 根据 Item(table=True) 创建 item 表
  -> migrate_runtime_schema() 补历史字段
  -> seed_items() 检查是否已有 Item
       ├─ 有数据：直接返回，避免重复插入
       └─ 无数据：add_all 三条样例并 commit
  -> 应用开始接收请求

⑤ 重难点、易混淆点对比辨析

  • SQLite 文件 vs Python 文件：first_api.db 是运行数据，不是源码；通常不提交 GitHub。
  • Engine vs Session：engine 是连接工厂和数据库入口；Session 是一次工作单元，负责查询、追踪对象、提交事务。
  • create_all vs migration：create_all 适合首次建表；生产结构演进应使用迁移工具，而不是指望它自动改表。
  • Field(index=True) vs 唯一约束：索引提高查询速度，不等于保证值唯一。唯一性需要 unique 约束或业务校验。

⑥ 开发常见报错+坑点+解决方案

  • no such table: item：未执行建表。解决：确认 lifespan 调用了 create_db_and_tables()，或测试里执行 SQLModel.metadata.create_all(engine)。
  • SQLite objects created in a thread...：SQLite 线程检查冲突。解决：教学/TestClient 场景使用 check_same_thread=False。
  • 样例数据重复：种子函数没有先检查已有数据。解决：像 seed_items 一样先查第一条记录。
  • 数据库文件被提交：.gitignore 未忽略运行产物。解决：忽略 *.db 或具体数据库文件。
  • 改了模型但表结构没变：create_all 不负责复杂迁移。解决：开发期重建本地库，生产用 Alembic。

⑦ 生产环境 vs 教学环境 核心差异

  • 数据库选择：教学用 SQLite 降低门槛；生产常用 PostgreSQL/MySQL，支持并发、权限、备份、监控和高可用。
  • 迁移方案：教学可用 create_all 和轻量补字段；生产必须有版本化 migration。
  • 连接管理：教学 engine 配置简单；生产要考虑连接池、超时、事务隔离、慢查询和连接泄漏。
  • 数据初始化：教学 seed 样例数据；生产 seed 通常只用于基础字典、管理员初始化或迁移脚本。

⑧ 课后记忆习题+标准答案

1. table=True 的作用是什么？

让 SQLModel 把该模型纳入表元数据，并在 create_all 时创建对应数据库表。

2. 为什么 id 类型是 int | None？

对象创建前数据库还没生成主键，所以可能是 None；提交刷新后才有整数 id。

3. 为什么 SQLite 适合教学但不等于生产首选？

它零部署、单文件、易运行，但高并发写入、权限治理和分布式部署能力有限。

⑨ 面试高频真题+解析

1. ORM 解决了什么问题？

ORM 把数据库表和 Python 对象建立映射，让业务代码通过对象操作数据，同时由 ORM 生成和执行 SQL。

2. SQLModel 和 SQLAlchemy、Pydantic 的关系是什么？

SQLModel 基于 SQLAlchemy 和 Pydantic，既能做 ORM 表映射，也能利用类型注解和数据校验能力。

3. 为什么数据库迁移不能只靠 create_all？

create_all 主要负责缺表建表，不可靠处理删列、改列、迁移数据等结构演进。
