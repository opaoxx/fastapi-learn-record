前端任务历史：把后端响应信封投影为浏览器 UI 状态

本节不是“写一点 JavaScript 展示列表”，而是把 GET /tasks 的 API 契约转成用户可理解的任务历史界面。核心思想：UI 状态是 API 契约的投影，前端必须按 items/count/limit/offset 的语义渲染，而不是凭感觉解析 JSON。

① 本节核心知识框架

前端任务历史链路
├─ 静态页面：first_api/frontend/index.html
├─ 鉴权输入：getApiKey() -> X-API-Key
├─ 请求入口：fetch("/tasks?limit=5&offset=0")
├─ 错误边界：parseJsonResponse(response)
│  ├─ response.json()
│  └─ response.ok 判断 HTTP 成败
├─ API 契约：SummaryTaskListResponse
│  ├─ page.items：任务数组
│  ├─ page.count：总数
│  ├─ page.limit：页大小
│  └─ page.offset：偏移量
├─ DOM 渲染：renderTaskHistory(page)
└─ 安全习惯：createElement + textContent，不拼 innerHTML

② 核心概念底层原理

API 契约是后端承诺的 JSON 形状，UI 状态是浏览器为了渲染页面保存和计算的状态。GET /tasks 返回 envelope 后，前端不能再把响应当数组；它必须把 page.items 映射为任务行，把 page.count 映射为总数提示，把 page.limit/page.offset 映射为分页位置。

fetch() 的成功只代表网络层拿到了 HTTP 响应，不代表业务成功。401、422、500 等响应通常不会自动进入 catch，所以必须检查 response.ok。这和后端 HTTPException 的结构化错误契约是一对：后端给错误体，前端要把非 2xx 响应转成 UI 可显示错误。

DOM 渲染使用 createElement() 和 textContent，不是为了复古，而是为了明确数据到节点的映射，并避免把任务结果当 HTML 执行。真实框架 React/Vue 会封装这些操作，但底层仍是“状态变化 -> 重新渲染视图”。

③ 全套代码逐行精读解析

<div id="history-meta" class="history-meta">尚未加载</div>
<div id="task-list" class="task-list" aria-live="polite"></div>
  • history-meta 承载页级信息，例如状态、limit、offset、本页条数；不适合塞进每个任务行。
  • task-list 是任务行容器，每次刷新应清空后重建。

function getApiKey() {
  return document.querySelector("#api-key").value.trim();
}

把 API Key 读取集中到一个函数，避免上传、查询详情、刷新历史各自重复 DOM 查询。若输入框 ID 改变，只改这里。

async function parseJsonResponse(response) {
  const body = await response.json().catch(() => ({}));
  if (!response.ok) {
    throw new Error(`${response.status} ${response.statusText}: ${JSON.stringify(body)}`);
  }
  return body;
}
  • await response.json() 把 JSON 文本解析成 JS 对象；它不是返回字符串。
  • .catch(() => ({})) 防止错误响应不是 JSON 时二次崩溃。
  • response.ok 只在 200-299 为真；401/422 不会自动抛异常。
  • throw new Error(...) 把 HTTP 失败转换成 JS 异常，交给调用处展示。
async function loadTaskHistory() {
  const apiKey = getApiKey();
  if (!apiKey) {
    historyMeta.textContent = "缺少 X-API-Key";
    taskList.replaceChildren();
    return;
  }
  ...
}

没有 API Key 时前端直接阻止请求，避免发送必然 401 的调用。replaceChildren() 清空旧列表，否则错误状态下会残留旧任务。

const response = await fetch("/tasks?limit=5&offset=0", {
  headers: {"X-API-Key": apiKey},
});
const page = await parseJsonResponse(response);
renderTaskHistory(page);

请求头名必须与后端 APIKeyHeader(name="X-API-Key") 对齐。page 是 envelope 对象，真正数组在 page.items。

function renderTaskHistory(page) {
  taskList.replaceChildren();
  historyMeta.textContent =
    `显示 ${page.items.length} 条，共 ${page.count} 条；limit=${page.limit}，offset=${page.offset}`;
  for (const task of page.items) {
    const row = document.createElement("article");
    ...
    taskList.append(row);
  }
}
  • replaceChildren() 保证刷新是替换视图，不是追加重复节点。
  • page.items.length 是当前页条数，page.count 是总数，二者不能混用。
  • for (const task of page.items) 体现响应信封契约；写 for (const task of page) 会报错。
  • document.createElement 创建结构，textContent 写文本，避免 XSS 风险。

④ 核心机制运行全流程拆解

浏览器加载 /app/
  -> StaticFiles 返回 index.html
  -> JS 执行 loadTaskHistory()
  -> getApiKey() 读取 X-API-Key
  -> fetch GET /tasks?limit=5&offset=0
  -> FastAPI 鉴权、校验 query、查询数据库
  -> 返回 SummaryTaskListResponse JSON
  -> parseJsonResponse 检查 response.ok
  -> renderTaskHistory(page)
       -> page.items 渲染任务行
       -> page.count/limit/offset 渲染元信息
       -> task.error 或 task.result 决定展示失败或结果

⑥ 开发常见报错+坑点+解决方案

  • page is not iterable：响应顶层是对象。解决：遍历 page.items。
  • 401 Unauthorized：缺少 X-API-Key。解决：从输入框读取并设置请求头。
  • 错误响应被当成功渲染：未检查 response.ok。解决：统一封装 parseJsonResponse。
  • 列表越刷越多重复项：刷新前没清空 DOM。解决：调用 replaceChildren()。
  • 任务结果含 HTML 被执行：使用 innerHTML。解决：外部数据写入 textContent。

⑧ 课后记忆习题+标准答案

1. 为什么 UI 状态是 API 契约的投影？

因为界面显示的任务行、总数、错误、分页位置都来自后端响应字段，前端只是把这些字段转换成人能读的页面结构。

2. 为什么 fetch 后要检查 response.ok？

HTTP 错误状态通常不会让 fetch 自动抛异常；必须手动判断非 2xx 响应。

3. 为什么要读 page.items？

GET /tasks 返回 envelope，顶层是对象，任务数组位于 items 字段。

4. 为什么不用 innerHTML 渲染任务结果？

任务结果来自数据源，使用 innerHTML 可能把文本当 HTML 执行；textContent 更安全。

⑨ 面试高频真题+解析

题 1：前端如何可靠处理 API 错误？

统一封装响应解析：先尝试解析 JSON，再检查 status/ok；非 2xx 转成错误对象，UI 根据错误码和 message 展示，并记录必要上下文。

题 2：后端列表 envelope 改动会影响前端哪里？

影响数据读取路径、循环渲染、分页按钮、总数提示和测试。前端要从裸数组遍历改为 page.items，并使用 count/limit/offset。

题 3：原生 DOM 和前端框架的本质差异是什么？

原生 DOM 手动创建和更新节点；框架声明式描述状态到 UI 的映射并自动协调 DOM。底层仍是数据状态驱动视图。
