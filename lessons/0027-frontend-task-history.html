Lesson 0027

前端任务历史：把 API 响应信封变成界面状态

上一节我们让 GET /tasks 返回 items、count、limit、offset。本节不急着堆新接口，而是研究一个关键问题：浏览器拿到这个 API 契约以后，怎样把它变成用户能看懂的任务历史界面？

0. 本节学习目标

  • 理解“UI 状态是 API 契约的投影”这句话是什么意思。
  • 理解浏览器 fetch 调用后端时，请求、响应、JSON 解析、错误处理的完整链路。
  • 看懂为什么前端要读取 page.items，而不是直接把整个 JSON 当成数组。
  • 学会用 DOM API 动态渲染任务历史，而不是只把 JSON 打印在 <pre> 里。
  • 把第 25、26 课的分页响应信封落到真实页面中，为下一课分页按钮和状态筛选打基础。

1. 背景：为什么这不是“写一点前端代码”

到目前为止，我们的后端已经有了一个很像真实 AI 服务的工作流：

  1. 用户上传一个文本文件。
  2. 后端创建一个摘要任务，并返回任务 ID。
  3. 后台任务运行，任务状态可能从 queued 到 running，再到 completed 或 failed。
  4. 用户需要回来看历史任务、失败原因和摘要结果。

如果页面只能显示“刚才那一次请求的 JSON”，它其实还不是一个工作台。真实用户会问：我刚才提交过哪些任务？哪个完成了？哪个失败了？历史记录在哪里？

所以本节的核心不是“学习几个 JavaScript 函数”，而是学习一个工程思想：后端 API 给出稳定的数据契约，前端把这个契约投影成界面状态。

2. 核心概念：UI 状态是 API 契约的投影

API 契约 是后端承诺给调用方的结构。例如 GET /tasks?limit=5&offset=0 返回：

{
  "items": [
    {
      "id": 12,
      "status": "completed",
      "text_length": 86,
      "source_file_id": 3,
      "result": "FastAPI lets us build...",
      "error": null
    }
  ],
  "count": 18,
  "limit": 5,
  "offset": 0
}

UI 状态 是浏览器为了渲染页面需要保存或计算的信息。例如：

  • items 决定页面上有几行任务卡片。
  • count 决定“共多少条”的提示。
  • limit 和 offset 决定当前展示哪一页。
  • 每个任务的 status 决定状态标签写什么。
  • result 和 error 决定显示摘要结果还是失败原因。

“投影”这个词可以理解为：后端返回的是数据对象，前端不改变事实，只选择一种适合人阅读的形态把它呈现出来。

3. 运行机制：一次任务历史刷新发生了什么

当页面加载或点击“刷新任务历史”时，浏览器和 FastAPI 之间会发生这条链路：

  1. 前端从输入框读取 X-API-Key。
  2. 前端调用 fetch("/tasks?limit=5&offset=0", {headers: {"X-API-Key": apiKey}})。
  3. FastAPI 匹配 GET /tasks 路由。
  4. Depends(require_api_key) 校验请求头。
  5. Query 解析并校验 limit 和 offset。
  6. service 层查询数据库，得到当前页 items 和总数 count。
  7. FastAPI 按 SummaryTaskListResponse 序列化响应。
  8. 浏览器拿到 Response 对象，检查 response.ok。
  9. 前端调用 response.json() 把响应体解析成 JavaScript 对象。
  10. renderTaskHistory(page) 把对象渲染成 DOM 节点。

这一条链路串起来以后，你就能看出“前端”和“后端”不是两个孤立知识点。它们之间靠 HTTP、JSON 和契约连接。

4. 官方资料中的关键点

MDN 把 Fetch API 描述为浏览器获取资源的接口，它围绕 Request 和 Response 这些对象工作；fetch() 返回的是一个 Promise，成功后得到 Response。这意味着“请求完成”和“JSON 解析完成”是两个步骤。

Response.ok 表示状态码是否在 200 到 299 范围内，适合用来判断 HTTP 层是否成功。Response.json() 会读取响应体并把 JSON 文本解析成 JavaScript 对象。注意，它返回的不是“JSON 字符串”，而是解析后的对象。

FastAPI 这边，response_model 会用于响应数据的文档、校验、转换和过滤。上一节的 SummaryTaskListResponse 不是装饰用的类型，它决定了前端应该期待怎样的 JSON 形状。

5. 代码精读一：HTML 结构为什么这样放

我们在原来的上传工作台下面增加了一个任务历史区域：

<section class="panel history" aria-labelledby="task-history-title">
  <div class="history-head">
    <div>
      <h2 id="task-history-title">任务历史</h2>
      <p class="muted">
        读取 GET /tasks?limit=5&amp;offset=0，把响应信封渲染成界面。
      </p>
    </div>
    <div id="history-meta" class="history-meta">尚未加载</div>
  </div>
  <div id="task-list" class="task-list" aria-live="polite"></div>
</section>

这里有三个设计点：

  • history-meta 放元信息：显示几条、总共几条、当前 limit 和 offset。
  • task-list 放真实列表：每个任务会被渲染成一个 article。
  • aria-live="polite" 是可访问性提示：列表更新时，辅助技术可以以较温和的方式感知变化。

这就是把“响应信封”拆成两层 UI：外层元信息和内层数据项。

6. 代码精读二：统一读取 API Key

function getApiKey() {
  return document.querySelector("#api-key").value.trim();
}

这段看起来很小，但它体现了一个工程习惯：重复的读取逻辑不要散落在每个事件处理函数里。上传、创建任务、查询详情、刷新历史都需要同一个请求头，所以我们用一个函数集中表达。

如果以后 API Key 输入框的 ID 变了，只需要改这一处。

7. 代码精读三：fetch 的错误边界

async function parseJsonResponse(response) {
  const body = await response.json().catch(() => ({}));
  if (!response.ok) {
    throw new Error(`${response.status} ${response.statusText}: ${JSON.stringify(body)}`);
  }
  return body;
}

初学者很容易误以为 fetch 遇到 401、404、422、500 都会自动抛异常。它不会。只要网络层成功拿到 HTTP 响应，Promise 通常会正常 resolve。你必须自己检查 response.ok。

这里的逻辑是：

  1. 先尽量把响应体解析成对象。
  2. 如果状态码不是 2xx，把状态码、状态文本和错误响应体拼成错误消息。
  3. 如果状态码成功，就返回解析后的对象。

这就是前端侧的“错误边界”：后端可以返回结构化错误，前端要把它转成用户和开发者都能看懂的提示。

8. 代码精读四：请求任务历史

async function loadTaskHistory() {
  const apiKey = getApiKey();
  if (!apiKey) {
    historyMeta.textContent = "缺少 X-API-Key";
    taskList.replaceChildren();
    return;
  }

  refreshHistoryButton.disabled = true;
  historyMeta.textContent = "加载中...";

  try {
    const response = await fetch("/tasks?limit=5&offset=0", {
      headers: {"X-API-Key": apiKey},
    });
    const page = await parseJsonResponse(response);
    renderTaskHistory(page);
  } catch (error) {
    historyMeta.textContent = "加载失败";
    taskList.replaceChildren();
    const message = document.createElement("p");
    message.className = "task-error";
    message.textContent = error.message;
    taskList.append(message);
  } finally {
    refreshHistoryButton.disabled = false;
  }
}

这段代码可以拆成四个阶段：

  • 输入检查：没有 API Key 就不发请求。
  • 加载状态：按钮禁用，提示“加载中”。
  • 成功路径：请求后端，解析响应，交给渲染函数。
  • 失败路径：清空旧列表，显示错误消息，最后恢复按钮。

这比“直接 fetch 然后 console.log”多了很多代码，但这些代码都不是啰嗦。它们对应真实应用里必须面对的用户体验：等待、失败、重试、状态恢复。

9. 代码精读五：把 page.items 渲染成 DOM

function renderTaskHistory(page) {
  taskList.replaceChildren();
  historyMeta.textContent =
    `显示 ${page.items.length} 条，共 ${page.count} 条；limit=${page.limit}，offset=${page.offset}`;

  if (page.items.length === 0) {
    const empty = document.createElement("p");
    empty.className = "muted";
    empty.textContent = "还没有任务。上传一个 .txt 文件后再刷新。";
    taskList.append(empty);
    return;
  }

  for (const task of page.items) {
    const row = document.createElement("article");
    row.className = "task-row";
    ...
    taskList.append(row);
  }
}

这里最重要的是第一行：taskList.replaceChildren()。每次刷新历史都先清空旧 DOM，再渲染新 DOM。否则重复点击刷新会把相同任务一遍遍追加到页面里。

第二个关键点是 page.items。因为第 25 课我们把响应从裸数组升级成了响应信封，所以现在不能写：

for (const task of page) {
  ...
}

正确写法必须是：

for (const task of page.items) {
  ...
}

这就是 API 契约变更对调用方的影响。后端 JSON 形状变了，前端读取路径也必须变。

10. 为什么不用 innerHTML 拼字符串

本节使用 document.createElement()、textContent 和 append() 创建 DOM，而不是把一整段 HTML 字符串塞进 innerHTML。

原因有两个：

  • textContent 会把内容当普通文本，不会把任务结果里的字符当 HTML 执行。
  • 逐个创建节点更适合初学者理解“数据对象如何变成页面结构”。

等你熟悉 Vue、React 或其他前端框架后，这类 DOM 操作会被框架封装掉。但底层思想仍然一样：数据变化，界面重新渲染。

11. 和后端代码的对应关系

前端这行请求：

fetch("/tasks?limit=5&offset=0", {
  headers: {"X-API-Key": apiKey},
})

对应后端路由：

@router.get("", response_model=SummaryTaskListResponse)
def list_summary_tasks(
    session: Annotated[Session, Depends(get_session)],
    task_status: Annotated[TaskStatus | None, Query(alias="status")] = None,
    offset: Annotated[int, Query(ge=0)] = 0,
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
) -> SummaryTaskListResponse:
    return list_summary_task_records(
        session=session,
        task_status=task_status,
        offset=offset,
        limit=limit,
    )

这两个片段是一对。前端发送的 limit=5 和 offset=0 会进入后端的同名参数；请求头 X-API-Key 会被路由级依赖校验；返回值会按 SummaryTaskListResponse 变成 JSON。

能把这两段对应起来，你就不再只是“会写接口”或“会调接口”，而是在理解端到端的数据流。

13. 常见错误与诊断

错误一：把响应当数组

现象：控制台提示 page is not iterable 或页面没有任务。

原因：现在响应是对象，任务列表在 page.items 里。

错误二：忘记 X-API-Key

现象：返回 401。

原因：/tasks 路由被 API Key 保护，前端必须带请求头。

错误三：只看 JSON 不看状态码

现象：错误响应也被当成功数据渲染。

原因：没有检查 response.ok。

错误四：刷新时不清空旧 DOM

现象：每点一次刷新，列表重复一遍。

原因：缺少 replaceChildren() 或等价清理逻辑。

14. 小测：先回忆，再展开

1. 为什么前端现在要读取 page.items，而不是直接遍历 page？

因为 GET /tasks 返回的是响应信封对象，真正的任务数组在 items 字段里。

2. response.json() 返回 JSON 字符串吗？

不是。它会把响应体中的 JSON 文本解析成 JavaScript 对象，并通过 Promise 返回。

3. 为什么 fetch 后还要检查 response.ok？

因为 HTTP 401、404、422 等错误响应通常不会让 fetch 自动抛异常，前端需要自己判断状态码是否成功。

4. count 在任务历史界面里有什么作用？

它告诉前端符合当前条件的任务总数，让界面能显示“共多少条”，并为后续分页按钮提供依据。

5. 为什么用 textContent 渲染任务结果更适合本节？

因为它把内容当普通文本处理，避免把任务结果误当 HTML，同时更容易看清数据到 DOM 的映射过程。

15. 本节总结

本节把前两节的分页响应信封真正接到了浏览器页面中。你现在看到的不只是一个新列表，而是一条完整链路：FastAPI 定义响应模型，service 查询数据库，浏览器用 fetch 请求 JSON，前端把 items 和分页元信息渲染成任务历史。

这节课的关键心法是：API 不是后端自己玩的数据结构，它是前后端共同遵守的契约。 一旦契约稳定，前端 UI、自动化测试、文档和未来的 AI 服务调用方都可以围绕它构建。
