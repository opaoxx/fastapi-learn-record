0072 Provider Metrics Practice Session Package：把 Runbook 训练流水线打包成端到端会话

本节新增 build_provider_metrics_runbook_practice_session()，把 samples、findings、题卡、答案键、评分汇总、summary 校验和 Markdown 报告串成一个完整 practice session。它是教学项目里的“编排层”，负责调用顺序，不复制任何下层业务规则。

② 核心概念底层原理

编排层解决的是“多个纯函数如何按正确顺序协作”。前面几节分别实现了样本、解释、题卡、答案键、评分、汇总、校验和报告；本节把这些能力合成一条端到端训练流水线。

编排层不应该重新实现下层逻辑。它不判断 outcome，不生成 severity，不重新计算 passed，也不手写 Markdown 表格。它只负责传递每一步的产物，保证数据流方向清晰。

practice session 的价值是可复盘：一个返回对象同时包含原始教学样本、解释结果、题卡、标准答案、评分汇总和报告文本。读者可以从任意层回看“为什么这一题这么判”。

层级核心问题本节处理方式
数据层练习事实从哪里来固定 scenario samples
解释层指标说明什么findings + next_action
训练层如何出题和给答案cards + answer_key
评估层学习结果如何衡量summary + validation + report

③ 全套代码逐行精读解析

def build_provider_metrics_runbook_practice_session(
    responses_by_card_id: dict[str, str],
) -> dict[str, object]:
    samples = build_provider_metrics_runbook_scenario_samples()

函数只接收作答字典，说明练习场景本身由课程固定样本提供。这样端到端练习可重复、可测试，不依赖线上 provider 状态。

findings = build_provider_metrics_runbook_findings(samples)
findings_markdown = render_provider_metrics_runbook_findings_markdown(findings)

先把 metric samples 翻译成 findings，再生成 findings 报告。注意报告来自 findings，而不是 samples；这是为了保持“事实 → 解释 → 展示”的单向链路。

cards = build_provider_metrics_runbook_exercise_cards(findings)
answer_key = render_provider_metrics_runbook_exercise_answer_key(cards)

题卡也从 findings 生成，标准答案再从 cards 渲染。这样题干、评分锚点和答案键共享同一份结构化来源，避免手写题库和代码契约漂移。

summary = summarize_provider_metrics_runbook_exercise_grades(
    cards,
    responses_by_card_id,
)

评分汇总复用 0069 的批量函数。编排层不关心每题如何判分，只传入 cards 和 responses，得到 summary。

validation = validate_provider_metrics_runbook_grading_summary(summary)
report_markdown = (
    render_provider_metrics_runbook_grading_summary_markdown(summary)
    if validation["valid"]
    else ""
)

报告渲染前先校验 summary。若 validation 失败，返回空报告而不是强行渲染，这是为了避免坏数据被包装成看似可信的复盘材料。

return {
    "samples": samples,
    "findings": findings,
    "findings_markdown": findings_markdown,
    "cards": cards,
    "answer_key": answer_key,
    "summary": summary,
    "validation": validation,
    "report_markdown": report_markdown,
}

返回完整会话包。每个字段都是前面某一层的稳定产物，方便测试、文档和调试。若只返回 report_markdown，读者无法追溯评分结果来自哪些样本和题卡。

④ 核心机制运行全流程拆解

responses_by_card_id
  + fixed scenario samples
  → findings
  → findings_markdown
  → cards
  → answer_key
  → summary
  → validation
  → report_markdown
  → practice session package

这条链路是课程后半段 runbook 训练能力的闭环。它把“观测事实”“排障解释”“训练题目”“标准答案”“学生表现”和“复盘报告”放进同一个可测试对象。

⑤ 重难点、易混淆点对比辨析

易混点错误理解正确理解
编排层应该顺手写业务规则只负责调用顺序和数据传递。
practice session只是 Markdown 报告它包含从样本到报告的全部中间产物。
validation 失败仍然渲染报告更方便坏数据不应被包装成可信报告。
responses应写进最终报告当前会话只传播评分结果，避免原始文本扩散。
固定 samples不如实时 counter 真实教学练习优先可重复；实时数据适合生产演练。

⑥ 开发常见报错+坑点+解决方案

坑点后果解决方案
跳过 validation 直接渲染坏 summary 变成可信报告先校验，valid 后再渲染。
编排层复制评分逻辑与单题评分函数漂移调用已有评分/汇总函数。
只返回最终 Markdown无法调试中间步骤返回完整 session package。
从全局 counter 取样本练习结果不可复现使用固定 scenario samples。
把 response 原文写进 session隐私风险和报告噪声增加只通过 summary 传播评分结果。

维度教学环境生产环境

编排方式同步纯函数调用workflow、队列、任务状态机

⑧ 课后记忆习题+标准答案

1. 为什么本节称为编排层？

因为它主要负责把多个已存在的纯函数按正确顺序串起来，不复制下层业务规则。

2. 为什么 practice session 返回中间产物？

中间产物能支持调试和复盘，读者可以追溯 report 来自哪些 samples、findings、cards 和 summary。

3. 为什么 validation 失败时不渲染报告？

报告会给人可信感。坏 summary 若被渲染，会把错误数据包装成正式复盘材料。

4. 为什么不读取全局 counter？

全局 counter 会随测试或运行顺序变化，教学练习需要固定、可重复、可测试的样本。

5. 生产环境如何升级这个 practice session？

可引入 workflow、数据库批次、用户身份、题库版本、任务状态、权限控制和审计日志。

⑨ 面试高频真题+解析

面试题 1：什么是编排层，为什么它不应该写业务规则？

编排层负责连接多个业务能力并控制调用顺序。若它复制业务规则，会形成多个事实源，导致维护和测试成本上升。

面试题 2：如何设计可复盘的训练会话对象？

应同时返回关键中间产物和最终报告，包括输入样本、解释、题卡、答案键、评分汇总、校验结果和展示文本。

面试题 3：为什么教学流水线适合纯函数？

纯函数可复现、易测试、无外部状态依赖，特别适合 GitHub 离线课程和新手调试。

面试题 4：生产 workflow 与本节纯函数编排有什么差异？

生产 workflow 还要处理持久化、幂等、失败重试、权限、审计、并发和跨服务调用，本节只保留核心数据流。
