AI 客户端依赖边界：用 Protocol、dataclass 与 Depends 隔离模型实现

本节把“情感预测”和“摘要生成”封装进 AIClient 边界。当前实现是 DemoAIClient 规则模型，但路由只依赖协议方法，不绑定具体模型。这个设计为测试替身、真实 provider、超时重试、模型版本切换打下基础。

① 本节核心知识框架

AI 客户端依赖设计
├─ 数据结果：PredictionResult(dataclass, frozen=True)
│  ├─ label：positive | neutral
│  ├─ score：置信分
│  └─ source：模型来源
├─ 配置对象：AIClientConfig(dataclass)
│  ├─ provider：demo
│  ├─ timeout_seconds / max_attempts
│  └─ retry delay 参数
├─ 行为协议：AIClient(Protocol)
│  ├─ predict_sentiment(text, mode)
│  └─ summarize(text)
├─ 具体实现：DemoAIClient
│  ├─ 规则词表 positive_words
│  ├─ failure_marker 模拟失败
│  └─ 摘要截断前 12 个词
└─ 注入入口：get_ai_client() -> AIClient

② 核心概念底层原理

AI 能力在后端系统中属于外部能力边界：它可能来自本地规则、远程模型、第三方 API、GPU 服务或多模型路由。router 如果直接写模型推理细节，就会把 HTTP 协议和模型实现强绑定，后续接真实 provider 时会牵连接口、测试、异常处理和前端契约。

Protocol 表示结构化子类型：对象不必继承某个基类，只要实现 predict_sentiment() 和 summarize()，就能被当作 AIClient 使用。这比强制继承抽象类更轻，也适合 Python 的 duck typing。

PredictionResult 是内部服务结果，PredictionResponse 是外部 API 响应模型。二者必须区分：内部结果可以随着模型架构演进，对外响应应尽量稳定，避免前端和调用方被模型替换破坏。

③ 全套代码逐行精读解析

PredictionLabel = Literal["positive", "neutral"]
AIProvider = Literal["demo"]
  • Literal 把普通字符串收窄为有限集合；如果随便返回 "good" 或 "bad"，类型检查和响应契约都会失去约束。
  • AIProvider 目前只有 demo，但先建立 provider 名称边界，后续可扩展为 openai、local 等。
@dataclass(frozen=True)
class PredictionResult:
    label: PredictionLabel
    score: float
    source: str
  • @dataclass 自动生成初始化和可读表示，适合承载服务层内部值对象。
  • frozen=True 让结果不可变，避免后续流程意外修改模型输出。
  • label 使用收窄类型，保证只出现 API 认可的分类。
  • score 是分数，不在此处做 Pydantic 校验，因为它不是请求/响应模型。
  • source 记录输出来源，便于日志、排障、前端展示和多模型灰度。
class AIClient(Protocol):
    def predict_sentiment(self, text: str, mode: str) -> PredictionResult:
        ...

    def summarize(self, text: str) -> str:
        ...
  • Protocol 定义调用方需要的最小行为集合；这就是依赖倒置，router 依赖抽象行为，不依赖具体模型类。
  • ... 表示协议方法没有实现体；如果误以为这里会运行，就会理解错 Protocol。
  • summarize() 也放入协议，是因为后台摘要任务同样需要 AI 能力。
class DemoAIClient:
    positive_words = {"good", "great", "love", "pleasant", "nice", "happy"}
    failure_marker = "simulate-ai-failure"

positive_words 是类级规则词表，避免每次请求重复构造集合；failure_marker 是教学用故障开关，用来稳定测试错误路径。生产环境不能用文本魔法词触发故障。

def predict_sentiment(self, text: str, mode: str) -> PredictionResult:
    self._raise_if_failure_requested(text)
    tokens = {word.strip(".,!?;:").lower() for word in text.split()}
    is_positive = bool(tokens & self.positive_words)
    careful_bonus = 0.04 if mode == "careful" else 0.0
    score = (0.91 if is_positive else 0.55) + careful_bonus
    return PredictionResult(...)
  • 先调用失败检查，保证错误路径在任何模型计算前中断。
  • text.split() 做最小分词；它不处理中文、词形还原或复杂标点，是 demo 而非 NLP 生产算法。
  • strip(...).lower() 让 Nice! 能匹配 nice；不处理会导致标点和大小写影响结果。
  • tokens & self.positive_words 是集合交集，复杂度低且语义直接。
  • careful_bonus 演示 mode 对模型分数的影响；真实系统可能对应不同模型或推理参数。
  • round(score, 2) 控制输出精度，避免浮点表示噪声进入 API 响应。
def get_ai_client() -> AIClient:
    return DemoAIClient(config=build_ai_client_config())

get_ai_client() 是依赖注入入口。路由声明 Depends(get_ai_client) 后，FastAPI 每次请求解析依赖并传入客户端。测试可以通过 app.dependency_overrides[get_ai_client] 替换实现，不需要 monkey patch 业务代码。

④ 核心机制运行全流程拆解

POST /predict
  -> FastAPI 校验 PredictionRequest
  -> Depends(get_settings) 读取配置
  -> Depends(get_ai_client)
       -> build_ai_client_config()
       -> get_settings()
       -> DemoAIClient(config=...)
  -> run_prediction(...)
       -> 调用 ai_client.predict_sentiment(text, mode)
       -> 得到 PredictionResult
  -> build_prediction_response()
       -> 转成 PredictionResponse
  -> 返回稳定 JSON 契约

后台摘要任务的流程不同：它不是路由函数，不能在参数上写 Depends。因此 task_worker.py 显式调用 get_ai_client()，再把客户端传给 run_summary_task()。依赖注入是 FastAPI 路由生命周期能力，普通 Python 函数要显式组装依赖。

⑤ 重难点、易混淆点对比辨析

概念本课含义易错理解
AIClient行为协议，规定调用方式以为必须是父类继承体系
DemoAIClient教学实现，可被替换当成真实机器学习模型
PredictionResult内部值对象直接当作 API 响应模型
PredictionResponse外部 JSON 契约随着内部模型随意变化
Depends路由依赖解析机制以为 BackgroundTasks 函数也能自动解析 Depends

⑥ 开发常见报错+坑点+解决方案

  • 测试结果不稳定：直接调用真实模型或外部 API。解决：用 dependency_overrides 注入 FixedAIClient。
  • AttributeError: ... summarize：替身客户端只实现了预测方法。解决：凡是要当 AIClient 使用，就实现协议里的全部方法。
  • 响应字段随模型变化：直接把模型原始输出返回给前端。解决：内部结果统一映射到 PredictionResponse。
  • 后台任务拿不到依赖：在普通函数参数里写 Depends(get_ai_client)。解决：在 worker 中显式调用依赖工厂。
  • 误把配置写死在 DemoAIClient：后续无法切换 provider 或超时策略。解决：使用 AIClientConfig 作为边界配置对象。

⑧ 课后记忆习题+标准答案

1. 为什么 router 不应该直接写 AI 推理逻辑？

因为推理逻辑涉及模型实现、配置、错误、超时和替换策略，直接写在 router 会把 HTTP 协议和模型边界耦合，难测试也难演进。

2. Protocol 和普通父类继承有什么区别？

Protocol 关注对象是否具备指定方法，不要求显式继承；普通父类继承关注继承关系。Protocol 更贴近 Python 结构化类型和替身测试。

3. 为什么内部结果和外部响应要分开？

内部结果服务于业务编排，可随实现变化；外部响应是 API 契约，应稳定服务前端和调用方。

4. 测试如何替换 AI 客户端？

定义实现同样方法的替身客户端，然后设置 app.dependency_overrides[get_ai_client] = override_ai_client，测试结束后清空覆盖。

⑨ 面试高频真题+解析

题 1：如何设计一个可替换的 AI provider 边界？

先定义稳定协议或接口，路由和 service 只依赖协议；具体 provider 通过依赖注入创建；内部 provider 输出映射到稳定响应模型；错误统一转换为业务异常或 HTTP 错误；测试用替身覆盖依赖。

题 2：依赖注入对测试有什么价值？

它把不可控外部依赖变成可替换参数。测试可以注入固定返回、固定失败、延迟或边界值客户端，从而稳定覆盖成功和失败路径。

题 3：为什么 demo 模型仍然要按真实模型结构封装？

课程项目的价值在于架构可演进。即使当前算法简单，也要提前把 provider、配置、结果、异常边界放好，后续接真实模型时无需重写 API 层。
