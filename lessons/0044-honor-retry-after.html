0044 Honor Retry-After：优先尊重 provider 的重试时间提示

本地 backoff 是客户端估算；Retry-After 是 provider 明确给出的协议信号。本节让 adapter 解析秒数与 HTTP-date 两种格式，优先使用 provider 提示，同时仍用本地 retry_max_delay_seconds 保护 API 响应边界。

① 本节核心知识框架

Honor Retry-After
├─ 协议来源：HTTP Retry-After response header
├─ 合法格式：delay-seconds / HTTP-date
├─ 解析函数：parse_retry_after_delay()
├─ 决策函数：build_retry_delay_plan()
├─ 优先级：valid Retry-After > local backoff
├─ 边界保护：min(provider_delay, retry_max_delay_seconds)
└─ 回退策略：无效 header → local_backoff

核心结论：provider 的限流与维护信号应该优先于本地估算，但客户端也必须保护自己的同步请求耗时与 worker 资源。

② 核心概念底层原理

Retry-After 常见于 429 Too Many Requests 与 503 Service Unavailable。它告诉客户端“多久以后再试”。格式可以是相对秒数，也可以是一个绝对 HTTP 日期。

格式示例含义
delay-secondsRetry-After: 120从现在开始等待 120 秒
HTTP-dateRetry-After: Wed, 21 Oct 2015 07:28:00 GMT等到该绝对时间后再请求
无效值Retry-After: later忽略 provider hint，回退本地 backoff

本节的工程原则是“协议信号优先，本地边界兜底”。如果 provider 提示等 3600 秒，同步 API 不应占着 worker 等 1 小时，而应被本地上限截断或进入更高级的后台任务机制。

③ 全套代码逐行精读解析

def parse_retry_after_delay(
    retry_after: str | None,
    now: datetime | None = None,
) -> float | None:

函数返回可等待秒数；返回 None 表示没有可用 provider hint，而不是等待 0 秒。now 可注入，避免日期测试依赖真实当前时间。

if retry_after is None:
    return None

value = retry_after.strip()
if value.isdecimal():
    return float(value)

空 header 直接回退。strip() 去掉多余空格；isdecimal() 只接受非负整数字符串，避免 -1、1.5 被误解析。

try:
    retry_at = parsedate_to_datetime(value)
except (TypeError, ValueError, IndexError, OverflowError):
    return None

parsedate_to_datetime 来自 Python 标准库，适合解析 HTTP-date 风格字符串。解析失败不应让 provider 调用崩溃，应回退到本地 backoff。

if retry_at.tzinfo is None:
    retry_at = retry_at.replace(tzinfo=timezone.utc)

current_time = now or datetime.now(timezone.utc)
if current_time.tzinfo is None:
    current_time = current_time.replace(tzinfo=timezone.utc)

日期计算必须处理时区。没有时区的日期统一按 UTC；当前时间也归一到 aware datetime，避免 Python 抛出 naive/aware datetime 不能相减的错误。

delay_seconds = (retry_at - current_time).total_seconds()
return max(delay_seconds, 0.0)

绝对时间转换为相对秒数。若 provider 给出的时间已经过去，则等待 0 秒，而不是返回负数。

provider_delay = parse_retry_after_delay(retry_after, now=now)
if provider_delay is not None:
    return RetryDelayPlan(
        delay_seconds=min(provider_delay, max_delay),
        source="retry_after",
    )

可解析的 Retry-After 优先；但仍被 max_delay 截断。source="retry_after" 为后续日志解释等待来源。

④ 核心机制运行全流程拆解

HTTP 429/503
  → response.raise_for_status()
  → except HTTPStatusError
  → retry_after = response.headers.get("Retry-After")
  → wait_before_retry(..., retry_after=retry_after)
  → build_retry_delay_plan()
      ├─ valid header：delay = min(provider_delay, max_delay)
      └─ invalid/missing：delay = local_backoff
  → sleep(delay)
  → next attempt

注意：timeout 和 request error 没有正常响应对象，无法读取 header；只有 HTTP 状态错误分支适合解析 Retry-After。

⑤ 重难点、易混淆点对比辨析

易混点正确理解后果
无效 Retry-After 是否等 0 秒不是，应该回退本地 backoff否则 provider hint 拼错会导致立即重试
Retry-After 是否突破本地 cap不能突破同步 API 可能占用 worker 过久
HTTP-date 是否直接字符串比较不能，应解析为 datetime时区和格式会出错
所有失败是否都有 header只有收到 HTTP 响应才有 headertimeout 分支读取 header 会报错

⑥ 开发常见报错+坑点+解决方案

  • naive/aware datetime 相减报错：日期对象时区不一致。解决：统一补 UTC。
  • provider 返回超长等待：直接 sleep 导致请求挂住。解决：min(provider_delay, retry_max_delay_seconds)。
  • 解析失败抛到用户：无效 header 让接口返回 500。解决：解析异常返回 None 并回退本地 backoff。
  • 测试日期不稳定：使用真实当前时间。解决：测试传固定 now。

⑦ 生产环境 vs 教学环境 核心差异

维度教学环境生产环境

⑧ 课后记忆习题+标准答案

1. Retry-After: 120 表示什么？

表示客户端应在收到响应后等待 120 秒再请求。

2. 为什么无效 Retry-After 不应当作 0 秒？

无效值表示没有可靠 provider hint，应回退本地 backoff；当作 0 秒会变成立即重试。

3. 为什么 provider hint 仍要被本地 cap 截断？

同步 API 必须保护自己的 worker 和响应时间边界，不能无限等待。

4. 为什么测试 HTTP-date 要传固定 now？

真实时间不断变化，固定 now 才能稳定断言时间差。

⑨ 面试高频真题+解析

题 1：如何处理 provider 返回的 Retry-After？ 答：在 HTTPStatusError 分支读取 header，支持秒数和 HTTP-date；可解析则优先使用，并受本地最大等待截断；不可解析则回退本地 backoff。

题 2：为什么不能无条件按 Retry-After 等待？ 答：provider 可能返回很长时间，同步请求长时间占用 worker 会伤害服务可用性，生产中应结合队列或异步任务处理。

题 3：Retry-After 与 backoff 的关系是什么？ 答：Retry-After 是服务端明确提示，优先级高；backoff 是客户端本地估算，在没有有效提示时使用。
