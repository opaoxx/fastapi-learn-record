0078 GitHub Branch Protection：把 CI 结果升级为 PR 合并硬门禁

本节新增 .github/BRANCH_PROTECTION.md，把 GitHub 仓库页面中的分支保护规则沉淀为可审查、可测试、可复用的工程文档。上一节 CI 负责“自动检查”，本节 required checks 负责“未通过不得合并”。

② 核心概念底层原理

Branch protection 是 GitHub 对分支写入操作设置的服务器端规则。它不改变代码运行逻辑，而是改变“谁能把什么状态的提交写入 main”。没有分支保护时，CI 即使失败，也只是一个红色提示；开启 required checks 后，失败的 CI 会成为合并阻断条件。

required status check 的底层对象是某个 commit 上的检查结果。GitHub Actions job 结束后会把结论写回对应 commit：success、failure、cancelled、skipped 等。分支保护读取这些状态，判断 PR 是否允许 merge。也就是说，CI 负责产生信号，branch protection 负责消费信号并执行治理规则。

本仓库的 required check 名称是 pytest and documentation contracts。它来自上一节 .github/workflows/ci.yml 中 job 的 name。如果 workflow 改名、job 改名或 required check 配置错名，GitHub 可能会一直等待不存在的检查，导致 PR 无法合并。

机制作用层级失败后果
pytest代码行为验证进程返回非 0，CI step 失败。
GitHub Actions job自动化执行层commit 上产生失败 check。
Required status checks分支治理层PR merge 按钮被阻断。
Branch protection仓库写入策略禁止绕过 PR、强推或删除主分支。

教学仓库不能通过普通文件直接启用 GitHub 分支保护，因为该设置保存在 GitHub 服务器侧。可版本化的最佳做法是把规则写进 .github/BRANCH_PROTECTION.md，再用文档契约测试保护关键字段。生产仓库可以进一步用 Terraform、GitHub API 或组织级 rulesets 自动化配置。

③ 全套代码逐行精读解析

# Branch Protection Required Checks

标题直接点明文档职责：不是泛泛介绍 GitHub 设置，而是记录本仓库 required checks 的精确配置。文档名称放在 .github/ 下，是因为该目录天然承载仓库协作、模板、workflow 等 GitHub 相关资产。

This repository uses GitHub Actions as a course quality gate.

第一句定义分支保护的前置条件：本仓库已经有 GitHub Actions 质量门禁。若没有 CI，required checks 没有可消费的状态信号，branch protection 只能约束 PR 流程，无法判断代码和课程文档是否合格。

Configure branch protection for `main` so pull requests cannot be merged until the CI job passes.

这里明确目标分支和治理目标。main 是默认发布分支，保护 main 等价于保护 GitHub 访问者看到的默认项目状态。若只保护临时分支，课程首页、README 和代码仍可能被破坏后直接进入主线。

1. Require a pull request before merging.

要求所有改动通过 PR 进入 main。没有这一条，维护者可以直接 push 到 main，CI 仍会运行，但失败结果只会在破坏已经进入主分支后出现。

2. Require status checks to pass before merging.

把 CI 结果接入合并门禁。它是本节核心规则：测试失败、文档契约失败或 whitespace 检查失败时，PR 不允许合并。

3. Require branches to be up to date before merging.

要求 PR 分支基于最新 main 重新验证。否则 A、B 两个 PR 分别通过 CI，但合并后组合状态可能失败。该规则能减少“单独正确，合并后错误”的集成风险。

4. Require conversation resolution before merging.

要求 review 讨论全部解决。课程仓库中，评论可能指出讲义链接、术语、代码解释或测试缺口；未解决评论不应被静默带入 main。

5. Block force pushes.
6. Block branch deletion.

阻止对 main 的破坏性写入。force push 会重写历史，branch deletion 会删除主分支引用。教学仓库虽然不是线上服务，但学习者依赖稳定历史和入口页，因此也应禁止这两类操作。

Use this required status check name:

```text
pytest and documentation contracts
```

这是最容易配错的字段。required check 必须匹配 GitHub Actions job 展示名称。本仓库 CI job 名为 pytest and documentation contracts，它同时覆盖 provider 行为测试、文档契约测试、全量测试和 whitespace 检查。

def test_branch_protection_documents_required_status_check() -> None:
    protection = Path(".github/BRANCH_PROTECTION.md").read_text(encoding="utf-8")

文档契约测试读取分支保护说明。因为 GitHub 服务器侧设置无法从本地仓库直接断言，所以本节测试保护的是“仓库中必须存在精确配置说明”。这能避免未来贡献者删掉 required check 名称或关键规则。

assert "pytest and documentation contracts" in protection
assert "Require status checks to pass before merging" in protection
assert "Block force pushes" in protection

断言只锁关键治理骨架，不锁每一句解释。这样文档仍可维护、可扩写，但不能丢掉主分支保护、required check 和破坏性操作限制。

④ 核心机制运行全流程拆解

Contributor opens PR
  -> GitHub Actions runs Course CI
     -> job: pytest and documentation contracts
        -> provider tests
        -> docs contract tests
        -> full pytest
        -> git diff --check
  -> GitHub writes check result to PR commit
  -> Branch protection reads required check
     -> success: merge allowed after other rules pass
     -> failure: merge blocked
     -> missing: merge blocked because required check has no result

关键调用链是“job 产出状态，分支保护消费状态”。分支保护本身不懂 pytest，也不解析 HTML 课程结构；它只关心 required check 是否在当前 commit 上成功。具体检查逻辑仍由 CI workflow 和测试文件承担。

数据来源消费者意义
PR head SHA贡献者分支GitHub Actions决定 CI 检查哪一次提交。
job conclusionCI runnerGitHub checks API记录 success 或 failure。
required check namebranch protection 设置merge gate决定必须等待哪个检查。
merge permissionbranch protection 规则汇总GitHub UI/API决定 PR 是否能进入 main。

⑧ 课后记忆习题+标准答案

1. CI 失败但 PR 仍能合并，通常缺了什么？

通常缺少 branch protection 中的 Require status checks to pass before merging，或 required check 名称没有正确匹配 CI job。

2. 本仓库 required status check 的名称是什么？

pytest and documentation contracts，它来自 .github/workflows/ci.yml 中 CI job 的显示名称。

3. 为什么要要求 branch up to date？

因为 PR 基于旧 main 通过 CI，并不代表它与最新 main 合并后仍然通过。要求更新后重跑可以发现组合冲突和集成回归。

4. 为什么要阻止 force push 到 main？

force push 会重写主分支历史，破坏学习记录、回滚依据和协作信任。公共课程仓库应默认禁止。

5. 为什么本节要把 GitHub UI 设置写进仓库文件？

因为 GitHub 分支保护设置保存在服务器侧，普通 clone 看不到。写入仓库文档后，规则可以被审查、复制、迁移，并由文档契约测试保护关键字段。

⑨ 面试高频真题+解析

面试题 1：CI 通过和 PR 可合并之间是什么关系？

CI 通过只是产生 success check；PR 可合并还取决于 branch protection、review、冲突状态、conversation resolution、required checks 是否全部满足。CI 是信号源，分支保护是合并判定器。

面试题 2：required status check 名称为什么容易出问题？

因为 GitHub 显示的 check 名称可能来自 workflow、job、matrix 组合或外部应用。配置时如果写错名称，GitHub 会等待一个不存在的检查，导致 PR 永远不能满足规则。

面试题 3：为什么生产环境会把分支保护配置代码化？

代码化可以避免 UI 手工配置漂移，支持审计、review、跨仓库复用和灾难恢复。常见方式包括 Terraform、GitHub API、组织 rulesets 或内部平台。

面试题 4：为什么文档仓库也要 required checks？

文档仓库的交付物是阅读路径、命令和知识结构。错误链接、过期命令和破坏的索引会直接影响用户，因此文档契约测试应作为 required check 的一部分。

面试题 5：如何处理紧急修复与 required checks 的冲突？

成熟团队会有受审计的紧急流程，例如临时 hotfix 分支、最小化测试集、事后补全测试与复盘。默认不应随意关闭 required checks，否则规则会失去治理可信度。
