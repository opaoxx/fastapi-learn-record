0058 Provider Metrics Production Boundary：教学指标离生产监控还差什么

本节专门划清边界：当前 provider metrics 有 Counter、JSON snapshot、Prometheus text 风格导出和测试，但仍不是生产级 metrics pipeline。我们把这个事实写进 FastAPI OpenAPI description，并用测试保护，避免学习项目被误读成可直接上线的监控方案。

① 本节核心知识框架

Production Boundary
├─ 当前已有
│  ├─ in-process ProviderMetricsCounter
│  ├─ GET /provider/metrics JSON snapshot
│  ├─ GET /provider/metrics/prometheus text export
│  └─ format / endpoint / isolation tests
├─ 当前没有
│  ├─ 多进程聚合
│  ├─ 多实例采集
│  ├─ 服务发现与 scrape 配置
│  ├─ 长期时间序列存储
│  ├─ dashboard / alert rule
│  └─ 权限与网络边界
├─ 文档边界
│  ├─ teaching
│  ├─ in-process
│  ├─ educational export shape
│  └─ not production multi-instance pipeline
└─ 测试落点：tests/test_openapi_contract.py

② 核心概念底层原理

生产监控不是“应用返回一段 metrics 文本”这么简单。完整链路包含应用埋点、导出协议、采集器、服务发现、实例标签、时间序列存储、查询语言、仪表盘、告警规则、权限控制和事故响应流程。

当前课程实现解决的是“指标语义和导出形态”问题：如何记录 provider outcome，如何设计低基数 labels，如何把 snapshot 渲染成文本。它不解决跨实例全局视图，也不提供长期历史查询。

因此，文档必须诚实描述边界。工程文档不是宣传语，而是协作契约：告诉调用者能力范围，也告诉他们不能把这个 endpoint 当作生产监控平台。

③ 全套代码逐行精读解析

@router.get(
    "/provider/metrics",
    response_model=list[ProviderMetricSampleResponse],
    summary="Read teaching provider prediction metrics",
    description=(
        "Returns the current in-process provider prediction metric counter snapshot. "
        "This endpoint is for teaching metric shape and FastAPI response modeling, not "
        "for production multi-instance aggregation."
    ),
)
async def get_provider_metrics() -> list[dict[str, object]]:
    return get_provider_metrics_snapshot()
文本作用
teaching明确该 endpoint 用于教学，不是平台能力声明。
in-process指出数据只来自当前 Python 进程。
snapshot强调它是当前状态快照，不是长期存储。
not for production multi-instance aggregation直接否定生产多实例聚合能力。
 
@router.get(
    "/provider/metrics/prometheus",

    summary="Read teaching provider prediction metrics as text",
    description=(
        "Returns a Prometheus text-style rendering of the current in-process provider "
        "prediction metric counter. This is an educational export shape, not a "

    ),

async def get_provider_metrics_prometheus() -> str:
    return get_provider_metrics_prometheus_text()

Prometheus text-style rendering 的措辞很关键：它说明输出形态接近 Prometheus text，但没有声称已经完成 Prometheus 生产接入。

def test_openapi_documents_provider_metrics_production_boundary() -> None:
    with TestClient(app) as client:
        response = client.get("/openapi.json")

    schema = response.json()
    json_operation = schema["paths"]["/provider/metrics"]["get"]
    text_operation = schema["paths"]["/provider/metrics/prometheus"]["get"]

测试读取 OpenAPI，而不是直接读 HTML 文档。因为 router 的 summary/description 会进入 /openapi.json，这是 FastAPI 对外暴露的机器可读接口文档。

assert "in-process provider prediction metric counter snapshot" in json_operation["description"]
assert "not for production multi-instance aggregation" in json_operation["description"]
assert "educational export shape" in text_operation["description"]
assert "not a production multi-instance metrics pipeline" in text_operation["description"]

这些断言保护的不是业务功能，而是边界语义。删掉这些描述，也是一种契约退化。

④ 核心机制运行全流程拆解

router decorator
  → summary / description / response_model / response_class
  → FastAPI 生成 /openapi.json
  → tests/test_openapi_contract.py 读取 schema
  → 定位 /provider/metrics 和 /provider/metrics/prometheus
  → 断言教学边界、进程内边界、非生产边界、text/plain 边界

文档测试的价值在于防止“说法悄悄变强”。如果未来有人把 description 改成“Production Prometheus metrics endpoint”，测试会失败，提醒开发者这不是文案优化，而是能力声明变化。

⑧ 课后记忆习题+标准答案

1. 为什么 text-style export 不等于生产 Prometheus 接入？

因为生产接入还需要 scrape 配置、服务发现、实例标签、时间序列存储、查询、dashboard、告警和权限边界。文本形态只是其中一个输出环节。

2. 当前 in-process Counter 在多实例下有什么问题？

每个进程或实例只知道自己的计数。单个 endpoint 不能代表全局，需要监控后端采集所有实例并聚合。

3. 为什么 OpenAPI description 也要测试？

因为它是对调用者的机器可读文档契约。边界说明被删掉会误导使用者，也属于接口语义退化。

4. 当前课程实现最适合学习什么？

学习 metrics label 设计、counter 聚合、FastAPI endpoint、Prometheus text 形态、测试边界和生产能力差距。

⑨ 面试高频真题+解析

面试题 1：如何判断一个 metrics endpoint 是否生产可用？

要看是否支持多实例采集、时间序列存储、查询、告警、权限、服务发现、基数治理和故障恢复。只有单进程文本输出通常只是教学或调试能力。

面试题 2：为什么文档要写清“not production pipeline”？

因为调用者会根据文档判断能力边界。明确限制可以防止把教学 endpoint 用作生产监控，也能减少后续维护误解。

面试题 3：生产中如何替代当前实现？

保留低基数 label 和 outcome 语义，但使用成熟 metrics client 或 OpenTelemetry，交给 collector/Prometheus 采集，存入 TSDB，并配置 dashboard、alert 和访问控制。
