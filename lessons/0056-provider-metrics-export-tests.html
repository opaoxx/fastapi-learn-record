0056 Provider Metrics Export Tests：把文本导出契约测试牢

本节只做一件事：为 provider metrics 文本导出补齐边界测试。对 metrics export 而言，200 OK 远远不够；HELP、TYPE、样本行、失败 outcome、label 转义、text/plain 都是对采集器和运维人员可见的契约。

① 本节核心知识框架

Export Tests
├─ helper 层测试
│  ├─ stable text shape
│  ├─ empty snapshot header only
│  └─ special label value escaping
├─ endpoint 层测试
│  ├─ success export text
│  ├─ empty export text
│  ├─ retry failure outcome text
│  └─ content-type starts with text/plain
├─ 测试隔离
│  ├─ autouse reset provider metrics
│  └─ dependency_overrides finally clear
└─ 核心思想：文本响应正文就是 API 契约

② 核心概念底层原理

普通 JSON API 可以用 response model 帮你约束字段；文本导出没有 Pydantic schema 自动兜底。文本格式是否正确，主要依赖显式字符串测试。

metrics export 的测试必须覆盖“无数据”和“异常数据”。无数据时仍应有 HELP/TYPE；失败路径最能体现指标价值；特殊字符最容易破坏格式。只测 success happy path，会让关键风险沉到生产阶段。

③ 全套代码逐行精读解析

def test_render_provider_metrics_prometheus_text_handles_empty_snapshot() -> None:
    text = render_provider_metrics_prometheus_text([])

    assert text == (
        "# HELP provider_prediction_total Provider prediction outcomes recorded by "
        "the teaching metrics counter.\n"
        "# TYPE provider_prediction_total counter\n"
    )

空列表输入代表当前 Counter 没有样本。测试要求输出 header-only text，而不是空字符串。否则调用者无法从导出中获知指标名和类型。

def test_render_provider_metrics_prometheus_text_escapes_special_label_values() -> None:
    samples = [
        {
            "labels": build_provider_metric_labels(
                outcome="fail_fast",
                failure_category="invalid_response",
                error_code='line\\one\n"two"',
                status_code=200,
            ).as_dict(),
            "count": 1,
        }
    ]

    text = render_provider_metrics_prometheus_text(samples)

    assert 'error_code="line\\\\one\\n\\"two\\""' in text
输入字符文本中应变成原因
反斜杠\\避免和后续转义序列混淆
真实换行\n避免一条 sample 被拆成多行
双引号\"避免 label value 提前结束

endpoint 测试保护 HTTP 层：路由存在、content-type 正确、实际响应正文完整。helper 测试无法发现 response_class 配错的问题，所以两层测试不能互相替代。

def test_provider_metrics_prometheus_endpoint_exposes_retry_failure_outcomes() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(status_code=503, request=request)

    def override_retry_settings() -> Settings:
        return Settings(

            ai_retry_base_delay_seconds=0.0,

503 测试用 MockTransport 制造可重试失败，并把 retry delay 设为 0，避免测试真实等待。它验证 text export 能表达 retry_scheduled 与 retry_exhausted。

④ 核心机制运行全流程拆解

helper test
  → 构造 samples
  → render_provider_metrics_prometheus_text(samples)
  → 精确断言字符串

endpoint test
  → MockTransport 模拟 provider
  → POST /provider/predict 制造 metrics
  → GET /provider/metrics/prometheus
  → 断言 status_code + content-type + text

isolation
  → autouse fixture reset_provider_metrics()
  → finally 清 dependency_overrides
  → close replacement AsyncClient

失败 outcome 的文本测试同时覆盖两条样本：

provider_prediction_total{error_code="ai_provider_http_error",failure_category="retryable_http_status",operation="prediction",outcome="retry_exhausted",status_code="503"} 1
provider_prediction_total{error_code="ai_provider_http_error",failure_category="retryable_http_status",operation="prediction",outcome="retry_scheduled",status_code="503"} 1

这证明文本导出没有只服务 success 场景，而是承接了前面建立的 failure outcome taxonomy。

⑤ 重难点、易混淆点对比辨析

测试类型验证对象不能替代的原因
helper 测试纯字符串渲染规则无法发现 FastAPI 路由、Content-Type、依赖覆盖问题
endpoint 测试真实 HTTP 响应边界构造特殊转义样本不如 helper 直接
OpenAPI 测试文档与契约描述不验证运行时正文
全量测试集成回归失败定位慢，不应替代小测试

⑥ 开发常见报错+坑点+解决方案

坑点表现解决方案
只断言 200正文格式坏了仍通过精确断言关键文本输出
只测 success失败指标导出无人保护加入 retry_exhausted、retry_scheduled、fail_fast 场景
不测空快照无样本时导出不合法断言 HELP/TYPE header-only 输出
不测转义特殊字符破坏格式构造包含反斜杠、换行、双引号的 label value
不清全局 Counter测试顺序影响输出使用 reset fixture 隔离每个测试

⑦ 生产环境 vs 教学环境 核心差异

维度教学测试生产验证
格式测试精确字符串断言结合 client library、promtool、采集器回归
失败场景MockTransport 控制 503/400压测、混沌、真实 provider 故障演练
隔离方式reset 进程内 Counter隔离 registry、测试 exporter、独立环境
告警验证暂不涉及验证 alert rule、dashboard query、SLO 计算
兼容性单项目内断言需要下游查询迁移和版本窗口

⑧ 课后记忆习题+标准答案

1. 为什么 text export 不能只测 HTTP 200？

因为正文格式本身才是采集契约。HELP、TYPE、metric name、label、转义、换行错误都可能在 200 状态下发生。

2. 为什么空 snapshot 仍然要测试？

空状态是系统启动后常见状态。它应输出合法 header-only 文本，不能返回空字符串或错误响应。

3. helper 测试和 endpoint 测试如何分工？

helper 测试验证纯渲染规则；endpoint 测试验证路由、HTTP 状态、Content-Type、依赖覆盖和真实业务触发后的输出。

4. 为什么 retry delay 在 endpoint 测试中设为 0？

本测试目标是验证 metrics 输出，不是验证等待时间。将 delay 设为 0 可避免测试变慢和关注点混淆。

⑨ 面试高频真题+解析

面试题 1：你会如何测试一个 Prometheus text endpoint？

我会分 helper 与 endpoint 两层：helper 精确测试 HELP、TYPE、样本行、label 顺序、转义和空快照；endpoint 用 TestClient 验证状态码、Content-Type、真实业务触发后的响应正文，并覆盖成功和失败 outcome。

面试题 2：为什么要测试特殊字符转义？

因为文本协议依赖引号、换行和反斜杠表达结构。未转义字符会破坏 label value 边界，导致采集器解析失败或样本语义错误。

面试题 3：如何避免 metrics endpoint 测试互相污染？

识别进程内全局 Counter，使用 autouse fixture 在每个测试前后 reset；同时在 finally 中清 FastAPI dependency overrides 并关闭替换的 AsyncClient。
