0054 Provider Metrics Test Isolation：全局进程内指标的测试隔离

本节不新增业务能力，而是修复测试结构风险。教学版 provider metrics 使用模块级 Counter，便于理解指标聚合，但它也是进程内全局状态。只要测试之间共享状态，endpoint 快照就可能被前一个测试污染。因此必须用 pytest fixture 在每个测试前后重置指标。

① 本节核心知识框架

Test Isolation for Provider Metrics
├─ 问题根源
│  ├─ provider_prediction_metrics 是模块级对象
│  ├─ Counter 状态跨测试保留
│  └─ endpoint snapshot 读取当前进程全部样本
├─ 隔离原则
│  ├─ 测试开始前：清空历史状态
│  ├─ 测试执行中：只验证自己制造的事件
│  └─ 测试结束后：清理自己留下的状态
├─ pytest 机制
│  ├─ fixture
│  ├─ yield setup/teardown
│  └─ autouse=True 文件级自动应用
└─ 工程边界
   ├─ reset metrics 不等于清 dependency overrides
   ├─ autouse 适合共同环境规则
   └─ 作用域过大将制造隐式副作用

② 核心概念底层原理

自动化测试的可信度来自隔离性：一个测试应该尽量只受自己 arrange 阶段创建的数据影响。若测试 A 记录了 success metric，测试 B 读取 metrics 时也看到 success，那么测试 B 的断言不再只描述自己的行为。

本项目的根因是模块级对象：

provider_prediction_metrics = ProviderMetricsCounter()

Python 进程加载模块后，这个对象会一直存在。pytest 在同一进程内运行多个测试时，Counter 的内容不会自动清空。数据库测试常用事务回滚，文件测试常用临时目录，依赖覆盖测试要清 app.dependency_overrides；进程内 metrics 测试则要清 provider_prediction_metrics。

fixture 是 pytest 提供的测试环境管理机制。yield 前的代码是 setup，yield 后的代码是 teardown。autouse=True 表示测试无需显式声明参数，该 fixture 会自动应用到当前作用域内的测试。

③ 全套代码逐行精读解析

@pytest.fixture(autouse=True)
def reset_provider_metrics_counter() -> object:
    reset_provider_metrics()
    yield
    reset_provider_metrics()
代码作用为什么这样写不这样写的隐患
@pytest.fixture声明测试夹具把环境准备和清理从测试主体抽离每个测试手写 reset，重复且容易漏
autouse=True自动应用到当前文件测试该文件中多个 provider endpoint 测试都需要相同隔离规则某个测试忘记声明 fixture 参数会被污染
def reset_provider_metrics_counter()用清晰函数名说明隔离目标读测试文件时能立即知道它清理 provider metrics counter模糊命名如 setup() 不利于维护
reset_provider_metrics() before测试开始前清空历史样本防止前一个测试失败或遗漏 teardown 后污染当前测试只做 after 清理时，崩溃测试可能影响后续测试
yield把控制权交给测试函数pytest 在测试完成后继续执行 yield 后代码用普通 return 无法表达 teardown
reset_provider_metrics() after测试结束后清理本测试样本保持测试文件和调试环境干净只做 before 清理时，当前测试状态会留给后续手动调试或其他作用域

被隔离后的测试主体更聚焦：

def test_provider_metrics_endpoint_exposes_fail_fast_outcome() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(status_code=400, request=request)

    replacement_client = httpx.AsyncClient(
        transport=httpx.MockTransport(handler),
        trust_env=False,
    )
    app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

    try:
        with TestClient(app) as client:
            prediction_response = client.post("/provider/predict", json={...})
            metrics_response = client.get("/provider/metrics")
    finally:
        app.dependency_overrides.clear()
        asyncio.run(replacement_client.aclose())

注意这里仍然要手动清 dependency_overrides 和关闭 replacement_client。metrics reset 只负责 Counter，不负责 FastAPI 依赖覆盖和 HTTP 客户端生命周期。

④ 核心机制运行全流程拆解

pytest 收集 tests/test_ai_client_dependency.py
  → 发现 autouse fixture reset_provider_metrics_counter
  → 执行某个测试前
       └─ reset_provider_metrics()
  → 运行测试主体
       ├─ MockTransport 模拟 provider
       ├─ POST /provider/predict 记录 metrics
       └─ GET /provider/metrics 读取本测试样本
  → 测试结束后
       └─ reset_provider_metrics()
  → 下一个测试从空 Counter 开始

若没有 before reset，测试执行顺序会影响结果；若没有 after reset，测试运行后的进程状态会残留，不利于后续调试和同文件其他测试。

隔离后，三个典型测试都能独立成立：success 测试只看到 success；503 测试只看到 retry_scheduled 与 retry_exhausted；400 测试只看到 fail_fast。

autouse=True 不是越多越好。它适合表达“当前文件所有测试共同依赖的环境规则”。若某个清理只服务一两个测试，显式 fixture 参数更透明。

⑧ 课后记忆习题+标准答案

1. 为什么 metrics endpoint 测试需要隔离 Counter？

因为 Counter 是进程内全局对象，多个测试共享。如果不清理，后一个测试读取的 snapshot 会包含前一个测试制造的样本。

2. fixture 中 yield 前后分别是什么阶段？

yield 前是 setup，测试函数运行前执行；yield 后是 teardown，测试函数结束后执行。

3. autouse=True 的优势和风险是什么？

优势是自动应用共同环境规则，减少重复和遗漏；风险是行为隐式，作用域过大时会影响不相关测试，必须用清晰命名和小作用域控制。

4. reset_provider_metrics() 能否替代 app.dependency_overrides.clear()？

不能。前者清 metrics Counter，后者清 FastAPI 依赖覆盖。它们处理的是不同共享状态。

⑨ 面试高频真题+解析

面试题 1：如何保证使用全局状态的测试可靠？

要识别全局状态的生命周期，并在每个测试前后恢复干净状态。pytest 中可用 yield fixture 表达 setup/teardown；如果该规则适用于整个文件，可使用文件级 autouse fixture，但要避免扩大作用域。

面试题 2：为什么测试前后都 reset，而不是只 reset 一次？

测试前 reset 防止历史状态污染当前测试；测试后 reset 防止当前测试状态污染后续测试或调试环境。前后都做可以抵御前一测试异常中断造成的残留。

面试题 3：你如何评价 autouse fixture？

autouse 适合共同且无条件的测试环境规则，例如当前文件所有测试都需要清 metrics Counter。它不适合隐藏复杂业务依赖，因为隐式行为会降低测试可读性和可维护性。
