② 核心概念底层原理

开源仓库的贡献入口不是单一文档，而是一组协作协议。Issue 定义问题，Discussion 澄清想法，Branch 承载修改，PR 提交证据，CI 验证行为，CODEOWNERS 路由 review，CHANGELOG 沉淀公共变更。CONTRIBUTING.md 的职责是把这些协议压成贡献者能执行的路径。

课程仓库比普通代码仓库更容易被改坏，因为它有三类同步对象：可运行代码、课程解释、离线导航。贡献者只改 lesson 不改 reference，会破坏复习闭环；只改 README 不改 docs contract，会让入口缺少测试保护；只写 Release Notes 不改 CHANGELOG，会让本地账本丢失。

贡献指南的底层价值是降低维护者审查成本。它把“什么能改、改前看什么、改后跑什么、PR 提交什么证据”前置给贡献者。这样 review 可以聚焦正确性、边界、教学质量和测试，而不是反复追问缺失信息。

协作文件主要读者职责失败后果
README.md学习者给出学习路线和运行入口新手找不到课程入口
DEVELOPMENT.md维护者给出本地命令和维护清单贡献者无法验证变更
CONTRIBUTING.md贡献者给出从想法到 PR 的规则PR 缺证据、范围发散
CHANGELOG.md学习者与贡献者记录公共可见变更发布说明缺素材
CODEOWNERSReviewer按路径路由审查关键文件无人负责

高质量贡献指南必须区分“教学 Demo 贡献”和“生产项目贡献”。教学仓库允许占位 owner、手工 changelog、简化 release；生产仓库通常还需要 DCO/CLA、代码行为兼容性、漏洞披露、合规审查、签名提交和正式行为准则。

③ 全套代码逐行精读解析

# Contributing Guide

标题使用社区约定。GitHub 会在贡献相关流程中识别 CONTRIBUTING.md，新贡献者也会自然寻找该文件。若贡献规则散落在 README 段落中，可发现性和可维护性都会降低。

Thank you for improving this course repository.

开头保持简洁欢迎，但不展开口号。贡献指南的主体不是社交文本，而是可执行规则。过度寒暄会稀释关键流程。

This repository is both a runnable FastAPI project and a structured learning archive.

这行定义双重属性：既是可运行项目，也是结构化课程档案。后续所有要求都从这里推出：代码、测试、课程、速查、导航必须同步。

Contributions must keep code behavior, lesson content, quick references, navigation, tests, and GitHub workflow documents synchronized.

这是贡献不变量。若贡献者只改一个面向，仓库就会出现“代码能跑但课程过时”或“课程写了但入口没链接”的断裂。

## Contribution Scope

Contribution Scope 限定适合贡献的工作类型。它把“欢迎贡献”转化为可选择任务，降低新手进入成本。

Fix a broken course link, typo that changes meaning, or outdated command.

该条定义低风险贡献。注意不是所有 typo 都值得进入贡献流程，只有改变含义、导航或命令正确性的修改才值得公开审查。

Improve an existing lesson while keeping the fixed nine-section structure.

该条绑定用户要求的课程标准。任何 lesson 改进都不能破坏九段结构，否则会让文档契约测试失败，也会降低期末复习式密度。

Open a discussion first when the idea is exploratory, broad, or primarily educational.

这行复用第 84 节的分流规则。开放性想法先进 Discussion，避免把未成形需求塞进 issue 队列。

Open an issue when the work is reproducible, scoped, and can be reviewed against a clear definition of done.

这行定义 issue 入口。可复现、范围明确、可验收是任务化的三个条件，缺一则先讨论。

## Before You Start

Before You Start 是贡献前置依赖。贡献者先读 README、DEVELOPMENT、CHANGELOG，再搜索 issue/discussion，最后绑定 scoped issue。

Check CHANGELOG.md for unreleased public changes.

这一步防止重复劳动。若同类变更已经在 Unreleased 中，贡献者应基于现有方向补充，而不是开出冲突 PR。

## Branch and Commit Workflow

分支与提交规则限制变更范围。短分支名能表达工作类型，独立 PR 能降低 review 和回滚成本。

docs/fix-lesson-0086-link
course/add-lesson-0087-contributing
tests/protect-contributing-guide

这些示例体现命名语义：docs、course、tests 先说明变更域，再说明具体目标。若分支名写成 update，无法从列表中判断工作内容。

Avoid mixing unrelated lesson rewrites, source changes, and GitHub governance changes in one pull request.

这是 PR 原子性原则。混合无关修改会让 reviewer 无法判断风险，也会让回滚粒度变粗。

## Lesson Contribution Standard

该节复制九段结构作为硬约束。贡献指南必须把课程标准前置，否则外部贡献很容易退回网课式浅讲。

Lessons must use the Final Review + CSDN Deep Dive style marker.

style marker 是文档契约的一部分，测试会保护它。它让课程风格从自然语言要求变成可检测约束。

## Documentation Checklist

Documentation Checklist 是 lesson 变更的提交算法。课程不是单文件产物，必须同步 lesson、reference、index、learning record、NOTES、COURSE_PROGRESS、CHANGELOG 和测试。

Update CHANGELOG.md when the change is learner-visible or contributor-visible.

这行把第 86 节的 changelog 接入贡献流程。不是所有内部步骤都写 changelog，但公共可见变化必须记录。

## Required Verification

Required Verification 定义 PR 前质量门禁。贡献者必须先本地跑测试，不能把 CI 当作第一次验证。

.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q
.\.venv\Scripts\python.exe -m pytest tests\test_course_docs_contract.py -q
.\.venv\Scripts\python.exe -m pytest -q
git diff --check

四条命令覆盖核心业务、文档契约、跨模块回归和空白质量。课程仓库的“可合并”必须同时满足代码和文档质量。

## Pull Request Requirements

PR Requirements 定义 reviewer 需要的证据。summary、why、change type、verification、checklist、risk notes、linked issue/discussion 缺一都会增加沟通成本。

The repository pull request template in .github/PULL_REQUEST_TEMPLATE.md is the source of truth for the required evidence.

这行防止贡献指南和 PR 模板分叉。贡献指南讲规则，PR 模板收集证据；真正提交时以模板为准。

## Review and Ownership

Review and Ownership 把贡献流接入 CODEOWNERS 和 branch protection。贡献者知道为什么某些文件需要特定 reviewer，也知道 CI 与 conversation resolution 不是形式主义。

## Changelog and Release Notes

该节说明贡献合入后如何进入发布链路。贡献先进入 CHANGELOG Unreleased，维护者发布时再通过 RELEASE_PROCESS 生成 tag 和 Release Notes。

## Code of Conduct Boundary

当前仓库尚未定义完整行为准则，因此这里先给出最低边界：技术、尊重、证据、安全。后续可在第 88 节或后续课程升级为 CODE_OF_CONDUCT。

⑤ 重难点、易混淆点对比辨析

易混淆点错误理解正确理解
CONTRIBUTING vs DEVELOPMENT都是开发说明，二选一。CONTRIBUTING 面向外部贡献者；DEVELOPMENT 面向维护者日常命令和规则。
Issue vs Discussion提问题都开 issue。可复现、可验收、会改文件才开 issue；探索性问题先 discussion。
PR Template vs CONTRIBUTING有 PR 模板就不需要贡献指南。贡献指南讲完整流程；PR 模板收集本次 PR 证据。
CODEOWNERS只是自动 @ 人。它是 review ownership 映射，可与 branch protection 形成强制审查。
小修可以不测试只改文档不用跑测试。课程文档由 docs contract 保护，小修也可能破坏导航或九段结构。
CHANGELOG只有 release 时才写。公共变更落地后应进入 Unreleased，release 时再迁移。

⑥ 开发常见报错+坑点+解决方案

坑点表现解决方案
PR 范围过大同时改课程、代码、CI、README，review 无法聚焦。拆成独立 issue 和 PR，每个 PR 只解决一个可验收目标。
缺少验证输出PR 描述写“应该没问题”。按模板粘贴 targeted、docs contract、full pytest、diff check 结果。

贡献前不搜 issue重复实现或方向冲突。Before You Start 阶段先搜索 issue/discussion 和 CHANGELOG Unreleased。
把 secrets 贴进 issue公开泄露密钥、日志或个人信息。贡献指南明确禁止 secrets、private logs、personal data 和 credentials。

⑧ 课后记忆习题+标准答案

1. CONTRIBUTING.md 的核心职责是什么？

把贡献者从想法引导到可审查 PR，明确贡献范围、开始前检查、课程标准、验证命令、PR 证据、review 所有权和发布记录边界。

2. 为什么贡献指南要同时提 README、DEVELOPMENT 和 CHANGELOG？

README 说明学习入口，DEVELOPMENT 说明维护命令，CHANGELOG 说明未发布公共变更。贡献者必须理解三者才能避免重复、漏测和漏记录。

3. Lesson 贡献为什么必须保留九段结构？

九段结构是本课程“期末复习干货 + CSDN 深度技术文”标准的结构化载体，也是文档契约测试保护的核心内容。

4. PR 必须提供哪些证据？

至少提供 summary、why、change type、verification output、course documentation checklist、risk notes，以及相关 issue 或 discussion 链接。

5. 贡献指南和 PR 模板如何分工？

贡献指南解释完整协作流程和规则，PR 模板在每次提交中收集具体证据。二者互补，PR 模板是提交证据的直接入口。

⑨ 面试高频真题+解析

面试题 1：一个成熟开源项目为什么需要 CONTRIBUTING.md？

它降低贡献者进入成本，统一 issue/branch/PR/test/review/changelog 流程，减少维护者反复追问缺失信息，让 review 聚焦质量和风险。

面试题 2：如何设计贡献指南中的测试要求？

测试要求应覆盖项目最高风险行为、公共文档契约、全量回归和格式质量。本仓库对应 provider HTTP、docs contract、full pytest、git diff --check。

面试题 3：为什么不建议一个 PR 混合多个无关变更？

混合 PR 会扩大 review 面积、模糊风险来源、增加回滚难度，并让 CI 失败原因难以定位。更好的做法是按可验收目标拆分。

面试题 4：CONTRIBUTING.md 与 CODEOWNERS 有什么关系？

CONTRIBUTING 告诉贡献者如何提交合格变更；CODEOWNERS 告诉 GitHub 和维护者哪些路径需要谁审查。前者是流程说明，后者是 ownership 规则。

面试题 5：教学仓库的贡献指南和生产仓库有什么不同？

教学仓库强调学习路径、九段课程结构、文档契约和新手安全；生产仓库还要加入 CLA/DCO、安全披露、合规、性能、部署和 SLA 等要求。
