0088 CODE_OF_CONDUCT：为新手友好技术社区建立安全协作边界

本节新增 CODE_OF_CONDUCT.md。它不是空泛礼仪声明，而是公开学习社区的安全层：规定期望行为、不可接受行为、适用范围、举报证据、维护者响应、技术 review 规范，以及 secrets、私有日志、个人数据和安全问题的公开边界。

② 核心概念底层原理

行为准则的底层目标不是“让所有人说话好听”，而是降低公开协作中的社会风险。技术社区的核心资产是知识、证据和互信；一旦新手因为羞辱不敢提问，或维护者因为骚扰无法 review，项目质量会直接下降。

课程仓库尤其需要行为准则，因为它面向初学者。初学者的问题往往不够精准、复现材料不完整、概念边界模糊。健康社区应把这些情况视为教学输入，而不是攻击对象。强 review 可以指出错误，但必须把矛头指向代码、文档、证据和推理链。

Code of Conduct 与 CONTRIBUTING 的区别在于治理层级。CONTRIBUTING 规定“如何提交工作”，Code of Conduct 规定“协作中什么行为可接受”。一个解决流程质量，一个解决参与安全。二者缺一，仓库都会失衡。

文档约束对象核心问题失败后果
CONTRIBUTING.md贡献流程如何提交合格 PR？PR 缺证据、范围失控
CODE_OF_CONDUCT.md协作行为如何保证安全讨论？新手流失、维护者受压、讨论失控
SECURITY.md漏洞披露安全问题如何私下报告？漏洞细节公开扩散
ISSUE_TRIAGE.md任务分流问题如何进入执行队列？issue 队列被噪声污染
DISCUSSIONS.md学习交流探索性问题如何沉淀知识？开放问题挤压可执行任务

行为准则必须具备可执行性。只写“请友善”不足以治理复杂场景；必须写明不可接受行为、举报要包含什么证据、维护者可以采取哪些动作、如何保护报告人隐私、哪些内容不能出现在公开渠道。

③ 全套代码逐行精读解析

# Code of Conduct

标题采用 GitHub 社区常见文件名。根目录 CODE_OF_CONDUCT.md 是公开协作规则入口，贡献者、学习者和维护者都能直接发现。

This course repository is intended to be a beginner-safe technical learning space.

首句明确受保护对象：beginner-safe technical learning space。这里的“安全”不是降低技术要求，而是保证新手可以在不被羞辱的情况下学习高质量技术。

The goal is not only to keep conversations polite.

这行排除浅层理解。行为准则不是礼貌清单，而是学习质量、心理安全、公开协作安全和 review 可执行性的共同保护层。

## Expected Behavior

Expected Behavior 定义正向标准。只列禁止项会让规则显得惩罚导向；先列期望行为可以告诉贡献者怎样做才是合格协作。

Be respectful, patient, and precise when discussing beginner questions.

这条针对课程仓库最常见场景：新手问题。尊重和耐心保护学习动机，precise 保护技术质量，三者必须同时存在。

Critique code, documentation, claims, and evidence rather than attacking people.

这是技术 review 的核心边界。可以强烈否定错误实现，但不能把错误上升为对人的攻击。否则 review 会从质量控制变成关系冲突。

Link lessons, source files, tests, issues, discussions, or pull requests when making technical claims.

这条把行为准则和本课程的证据文化绑定。技术判断要有可追溯依据，而不是用资历、语气或情绪压人。

## Unacceptable Behavior

Unacceptable Behavior 定义红线。没有红线，维护者无法解释为什么要隐藏评论、暂停 review 或限制参与。

Mocking beginner questions or using shame as a teaching method.

这条是教学社区关键规则。羞辱可能短期让人闭嘴，但长期会破坏学习空间和问题反馈质量。

Publishing private information, credentials, tokens, private logs, personal data, or proprietary source code.

这条是公开平台安全底线。行为准则必须覆盖隐私和密钥边界，因为很多新手会把完整日志或配置直接贴出来。

Pressuring maintainers to bypass tests, review, branch protection, or documented release rules.

这条保护工程治理。绕过测试、review 和发布规则不是“加快进度”，而是在破坏仓库质量契约。

Submitting AI-generated or copied material without checking accuracy, licensing, and fit with the course standard.

该条处理现代贡献风险。AI 生成内容和复制材料都可能存在事实错误、许可证问题或风格不合格，必须人工审查。

## Scope

Scope 定义适用范围。行为准则覆盖 issues、discussions、PR、reviews、commits、release notes、documentation examples 和明确服务于仓库的公开渠道。

Private behavior may affect repository participation when it directly threatens learner safety, maintainer safety, or the integrity of the project.

这行说明边界外行为何时进入治理范围。它不是审查私人生活，而是当外部行为直接威胁项目安全时，维护者可以处理参与权限。

## Reporting

Reporting 定义报告流程。安全事件不应在公开评论区拉扯升级，应保存证据并寻找私下维护者路径。

Do not escalate the conflict in public.

公开升级会扩大伤害、污染讨论、暴露隐私。正确做法是保存证据并转入维护者处理。

Save links, screenshots, timestamps, and the affected repository area.

证据必须可检查。没有链接、时间和位置，维护者只能根据情绪判断，容易误判。

If no private channel exists, open a short issue asking for a maintainer contact path without posting sensitive details.

这是教学仓库的现实兜底。仓库可能尚未配置私密报告入口，因此允许创建简短 issue 请求联系路径，但禁止公开敏感细节。

## Maintainer Response

Maintainer Response 给出处理动作集合。维护者需要从轻到重的梯度，而不是只有“无视”和“封禁”两种选项。

Ask participants to clarify, edit, or move a comment.

轻量处理适合误发、跑题或措辞不当。先纠正渠道或表达，保留学习空间。

Pause review until behavior or safety concerns are resolved.

review 可以被安全问题阻断。代码正确不代表 PR 可以合并；协作风险也属于项目风险。

Maintainers should prefer proportionate action, preserve evidence when possible, and avoid exposing private reporter details.

这行规定维护者约束：比例原则、证据保全、报告人隐私。行为准则不仅约束贡献者，也约束维护者。

## Teaching and Review Norms

该节把行为准则落到技术教学场景。强 review 不是问题，缺证据、讽刺、围攻和人身排序才是问题。

Strong review is welcome. Contempt is not.

这句是本节核心边界。可以严格，可以要求补测试，可以拒绝 PR；但不能把轻蔑当作教学手段。

## Safety Boundary

Safety Boundary 单独列出公开渠道禁发内容。与第 87 节 CONTRIBUTING 的安全边界一致，并为后续 SECURITY.md 留出接口。

If a report involves a security issue, avoid public reproduction details and ask maintainers for a private disclosure path.

安全问题不能在公开 issue 中贴完整利用细节。第 88 节先建立边界，后续可用 SECURITY.md 进一步细化披露流程。

## Relationship to Other Documents

最后用关系表把行为准则接入仓库治理系统。它不是孤立文件，而是包在 CONTRIBUTING、DEVELOPMENT、DISCUSSIONS、ISSUE_TRIAGE、RELEASE_PROCESS 外层的安全协议。

流程的关键是先分类，再响应。正常技术分歧应回到证据和测试；渠道混乱应移动到 issue 或 discussion；安全和隐私问题则必须停止公开扩散，转入私下处理。

⑥ 开发常见报错+坑点+解决方案

坑点表现解决方案
规则过空只写“请友善”，维护者无法执行。列出期望行为、不可接受行为、举报证据和处理动作。
规则过重所有轻微误会都升级为惩罚。使用比例原则，从澄清、编辑、移动到限制参与逐级处理。
公开处理隐私事件报告人和敏感信息二次暴露。保存证据，私下处理，公开区只保留必要说明。
把技术争论当违规有效 review 被压制。区分证据驱动的严格 review 和人身攻击。
忽略 AI 内容风险复制错误或侵权材料进入课程。要求检查准确性、许可证和课程标准适配。
未链接其他治理文档行为准则孤立，贡献者不知道下一步。链接 CONTRIBUTING、DEVELOPMENT、DISCUSSIONS、ISSUE_TRIAGE、RELEASE_PROCESS。

⑦ 生产环境 vs 教学环境 核心差异

维度教学环境生产环境
目标保护新手提问、学习质量和公开协作安全。还要覆盖员工、客户、生态合作方和品牌风险。
处理团队通常由仓库维护者兼职处理。可能有社区团队、法务、安全、HR 或信任安全团队。
举报路径可能先用维护者私下联系路径。通常有专用邮箱、表单、SLA、案件系统和审计记录。
安全披露先提示不要公开漏洞细节。配套 SECURITY.md、漏洞赏金、embargo 和 CVE 流程。
执行动作编辑、隐藏、暂停 review、关闭 PR、限制参与。还可能涉及账号封禁、合同措施、法律升级和跨平台协作。
文档成熟度自定义轻量规则，贴合课程仓库。常采用标准行为准则模板并配套正式执行政策。

⑧ 课后记忆习题+标准答案

1. Code of Conduct 的核心职责是什么？

定义公开协作中的安全行为边界，保护学习质量、心理安全、证据驱动讨论和维护者治理能力。

2. 为什么不能把行为准则理解成“说话客气”？

因为它还覆盖隐私、密钥、安全问题、公开升级、review 中断、维护者响应和参与限制等项目治理问题。

3. 遇到不安全行为时报告者应提供哪些信息？

应提供发生了什么、发生在哪里、谁受影响、是否涉及隐私或安全风险、可检查的链接或证据。

4. 强 review 与违反行为准则的边界在哪里？

强 review 针对代码、文档、证据和推理链；违规行为攻击个人、羞辱新手、围攻或用轻蔑替代证据。

5. 为什么安全问题不应在公开 issue 中贴完整细节？

公开复现细节可能扩大攻击风险。正确做法是请求维护者私下披露路径，并避免公开 secrets、漏洞利用步骤或敏感数据。

⑨ 面试高频真题+解析

面试题 1：开源项目为什么需要 Code of Conduct？

它为公开协作提供可执行安全边界，降低骚扰、泄露、讨论失控和维护者压力，让技术 review 能回到证据和质量。

面试题 2：Code of Conduct 与 CONTRIBUTING.md 如何分工？

CONTRIBUTING.md 规定如何提交合格贡献；CODE_OF_CONDUCT.md 规定协作行为和安全边界。前者管流程，后者管参与环境。

面试题 3：维护者处理行为准则事件应遵守什么原则？

应遵守比例原则、证据保全、报告人隐私保护和最小公开扩散原则，根据严重程度选择澄清、编辑、隐藏、暂停 review、关闭或限制参与。

面试题 4：为什么技术社区要保护新手问题？

新手问题是课程质量反馈的重要来源。羞辱会减少反馈并降低学习参与度；耐心追问和证据链接能把低清晰度问题转化为教学材料。

面试题 5：生产级社区治理还需要哪些补充？

通常还需要 SECURITY.md、专用举报渠道、正式执行政策、社区团队、法务支持、安全披露流程、SLA、审计记录和跨平台处理能力。
