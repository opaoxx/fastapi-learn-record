Lesson 0043

Backoff Before Retry：重试之前先等一等

第 42 课解决了“哪些错误可以重试”。本节解决另一个真实系统问题：如果所有客户端在失败后马上重试，短暂故障会被放大成请求风暴。我们会给 provider retry 加入指数退避，让每次下一次尝试之前等待一小段时间。

0. 本节学习目标

  • 理解 immediate retry 为什么可能伤害 provider 和自己的系统。
  • 掌握 backoff、exponential backoff、cap 三个概念。
  • 理解 await asyncio.sleep(...) 为什么适合 async provider adapter。
  • 学会把等待时间做成配置，而不是写死在业务代码里。
  • 用假的 sleep 函数测试 retry 等待逻辑，避免测试真的变慢。

1. 背景：立即重试会把小故障放大

上一节我们允许 provider prediction 在 timeout、网络错误、429、部分 5xx 时重试。这个方向是对的，但还缺一个关键保护：两次尝试之间要不要等待？

想象一个 AI provider 短暂返回 503。你的服务同时有 100 个请求失败，如果每个请求都立刻重试 3 次，provider 瞬间会从 100 个请求变成 300 个额外请求。provider 本来只是短暂拥塞，现在反而被客户端一起压得更慢。

Google Cloud 的 retry 指南也提醒：如果失败请求不等待就重试，可能在短时间内发送大量 retry，触发配额和限制。真实系统里这类现象常被称为 thundering herd，也就是许多客户端同步重试造成的请求浪潮。

2. 核心概念：backoff 是“失败后放慢脚步”

Backoff 是指 retry 之间的等待策略。它不是取消重试，而是让下一次尝试稍微晚一点发生，给 provider 恢复和限流窗口留出空间。

Immediate retry

fail -> retry now
fail -> retry now
fail -> retry now

实现简单，但多个客户端会同步冲击 provider。

Exponential backoff

fail -> wait 0.1s
fail -> wait 0.2s
fail -> wait 0.4s

失败越多，等待越长，直到达到最大等待上限。

本项目这节先实现“有上限的指数退避”：第 1 次失败等 base，第 2 次失败等 base * 2，第 3 次失败等 base * 4，但不能超过 max delay。jitter 暂时不加，因为随机性会让初学者和测试更难理解；后续再专门讲。

3. 数学模型：指数增长加上上限

本节新增的计算规则是：

delay = min(base_delay * 2 ** (attempt - 1), max_delay)

如果 base_delay=1.0，max_delay=3.0，那么第 1 到第 4 次失败后的等待是：

失败 attempt 未截断计算 最终等待
1 1.0 * 2^0 = 1.0 1.0 秒
2 1.0 * 2^1 = 2.0 2.0 秒
3 1.0 * 2^2 = 4.0 3.0 秒
4 1.0 * 2^3 = 8.0 3.0 秒

上限很重要。如果没有 max_delay，等待时间会快速变得不可接受。对 API 请求来说，用户不可能一直等你指数级睡下去。

4. 运行机制：retry 现在多了一个等待步骤

request_provider_prediction()
  -> attempt 1
       -> provider fails with retryable error
       -> calculate delay for attempt 1
       -> await asyncio.sleep(delay)
  -> attempt 2
       -> provider fails with retryable error
       -> calculate delay for attempt 2
       -> await asyncio.sleep(delay)
  -> attempt 3
       -> provider succeeds or final error is raised

注意：最后一次失败不会再 sleep，因为已经没有下一次尝试了。等待只发生在“还有下一次 retry”的情况下。

5. 代码精读一：配置项进入 Settings

class Settings(BaseSettings):
    ai_max_attempts: int = Field(default=1, ge=1, le=5)
    ai_retry_base_delay_seconds: float = Field(default=0.1, ge=0)
    ai_retry_max_delay_seconds: float = Field(default=2.0, ge=0)

max_attempts 控制最多尝试几次，base_delay 控制第一次失败后的等待，max_delay 控制等待上限。它们都属于 provider 调用策略，所以放进 settings，再传入 AIClientConfig。

这也意味着读者可以在 .env 中调整策略，而不需要修改 Python 代码：

FIRST_API_AI_MAX_ATTEMPTS="3"
FIRST_API_AI_RETRY_BASE_DELAY_SECONDS="0.1"
FIRST_API_AI_RETRY_MAX_DELAY_SECONDS="2.0"

6. 代码精读二：延迟计算函数

def calculate_retry_delay(attempt: int, config: AIClientConfig) -> float:
    base_delay = max(config.retry_base_delay_seconds, 0.0)
    max_delay = max(config.retry_max_delay_seconds, 0.0)
    exponential_delay = base_delay * (2 ** (attempt - 1))
    return min(exponential_delay, max_delay)

把计算单独抽成函数，是为了让策略可以被直接测试。否则你只能通过“跑一个 provider 请求”间接推断等待时间，测试会更脆弱。

这里用 max(..., 0.0) 做了一层保护。Settings 已经用 Field(ge=0) 限制环境变量，但 AIClientConfig 是普通 dataclass，测试或内部代码仍然可能直接构造它。这个保护让计算函数面对负数时不会生成负等待。

7. 代码精读三：等待函数和 sleep 注入

RetrySleep = Callable[[float], Awaitable[None]]

async def wait_before_retry(
    attempt: int,
    config: AIClientConfig,
    sleep: RetrySleep,
) -> None:
    delay_seconds = calculate_retry_delay(attempt, config)
    if delay_seconds > 0:
        await sleep(delay_seconds)

这里没有直接在测试里真的等 1 秒、2 秒，而是把 sleep 当成可注入函数。生产代码默认传 asyncio.sleep，测试代码传一个假的 async 函数记录参数。

Python 官方文档说明 asyncio.sleep() 会挂起当前 task，让其他 task 继续运行。也就是说，在 async route 或 async adapter 中，应该使用 await asyncio.sleep(...)，不要用会阻塞线程的 time.sleep(...)。

8. 代码精读四：失败分支里的等待位置

except httpx.HTTPStatusError as exc:
    status_code = exc.response.status_code
    if (
        attempt < config.max_attempts
        and is_retryable_provider_status(status_code)
    ):
        await wait_before_retry(attempt, config, sleep)
        continue
    raise AIClientError(...) from exc

等待发生在 continue 之前。也就是说：确认这次错误可重试、确认还有下一次尝试、先等待、再进入下一轮循环。

timeout 和 request error 分支也是同样逻辑。不可重试错误不会 sleep；最后一次失败也不会 sleep。这样既减少无意义等待，也让失败响应更快返回给调用方。

9. 测试精读一：直接测试数学规则

def test_calculate_retry_delay_uses_exponential_backoff_with_cap():
    config = AIClientConfig(
        retry_base_delay_seconds=1.0,
        retry_max_delay_seconds=3.0,
    )

    delays = [calculate_retry_delay(attempt, config) for attempt in range(1, 5)]

    assert delays == [1.0, 2.0, 3.0, 3.0]

这条测试不需要 HTTPX，也不需要 FastAPI。它直接保护 backoff 数学规则：指数增长，同时被最大等待截断。

10. 测试精读二：假的 sleep 让测试又快又准

sleep_calls = []

async def fake_sleep(delay_seconds: float) -> None:
    sleep_calls.append(delay_seconds)

然后让 provider 前两次返回 503，第三次成功：

result = run_async_prediction(
    client,
    max_attempts=3,
    retry_base_delay_seconds=1.0,
    retry_max_delay_seconds=3.0,
    sleep=fake_sleep,
)

assert sleep_calls == [1.0, 2.0]

这条测试验证了两个事实：第一，adapter 确实在 retryable failure 后等待；第二，等待只发生在前两次失败后，因为第三次已经成功，没有下一次 retry。

12. 常见错误与诊断

错误一：用 time.sleep

在 async adapter 中会阻塞当前线程，影响其他请求。应该使用 await asyncio.sleep。

错误二：最后一次失败后还等待

如果没有下一次 retry，等待没有意义。判断 attempt < max_attempts 后再 sleep。

错误三：没有最大等待上限

指数增长会很快变大。必须有 retry_max_delay_seconds 作为截断。

错误四：测试真的睡几秒

测试会变慢又不稳定。把 sleep 函数注入进去，测试只记录 delay 参数。

13. 面试视角：怎么解释 backoff

我不会让外部 provider 失败后立刻连续重试，因为这可能在 provider 过载时制造更多请求。我的做法是在 adapter 层对 retryable failure 使用有上限的指数退避：第 n 次失败后等待 base * 2 ** (n - 1)，并用最大等待时间截断。异步代码里用 await asyncio.sleep，测试里注入 fake sleep 来验证等待时间而不拖慢测试。

14. 小测：先回忆，再展开

1. 为什么立即重试可能伤害系统？

多个客户端会在同一时间重复请求，把短暂 provider 故障放大成请求风暴。

2. base=1、max=3 时前四次失败后等待多少？

分别是 1、2、3、3 秒；第三次开始被最大等待上限截断。

3. 为什么 async 代码里不用 time.sleep？

time.sleep 会阻塞线程；await asyncio.sleep 会挂起当前 task，让事件循环运行其他任务。

4. 为什么测试里要注入 fake sleep？

这样可以验证等待时间是否正确，同时避免测试真的等待几秒。

15. 本节总结

本节把 provider retry 从“可以再试一次”升级成“有节制地再试”。现在 adapter 会在 retryable failure 后根据 retry_base_delay_seconds 和 retry_max_delay_seconds 计算指数退避，并通过 await asyncio.sleep 等待后再进入下一次尝试。你也学到了一个重要测试技巧：把时间等待注入进去，用 fake sleep 记录行为。
