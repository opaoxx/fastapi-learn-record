0043 Backoff Before Retry：重试之前必须先退避

第 42 课解决“哪些错误可以重试”。本节解决“什么时候重试”：外部 provider 暂时不可用时，立即重试会放大故障；指数退避用可配置等待时间把重试压力摊开，并通过 fake sleep 让等待策略可测试。

① 本节核心知识框架

Backoff Before Retry
├─ 问题：immediate retry 会制造请求峰值
├─ 策略：exponential backoff = base * 2 ** (attempt - 1)
├─ 边界：retry_max_delay_seconds 截断最大等待
├─ 异步等待：await asyncio.sleep(delay)
├─ 可测试性：sleep 函数注入，测试不真实等待
└─ 落点：wait_before_retry() 只在还有下一次 attempt 时执行

核心结论：重试预算控制“最多试几次”，backoff 控制“每次失败后隔多久再试”。没有 backoff 的 retry 会在 provider 已经拥塞时继续加压。

② 核心概念底层原理

Immediate retry 会让大量客户端在同一瞬间再次冲向 provider；exponential backoff 让连续失败后的等待时间逐步变长，给 provider、网络和限流窗口恢复空间。

策略公式效果
立即重试delay = 0最容易形成请求风暴
固定间隔delay = base简单，但多客户端仍可能同步
指数退避delay = base * 2 ** (attempt - 1)失败越多，请求越慢
截断退避min(delay, max_delay)避免等待时间无限膨胀

在 async provider adapter 中，等待必须使用 await asyncio.sleep()。它挂起当前协程，不阻塞事件循环；time.sleep() 会阻塞线程，影响同进程其他请求。

③ 全套代码逐行精读解析

class Settings(BaseSettings):
    ai_retry_base_delay_seconds: float = Field(default=0.1, ge=0)
    ai_retry_max_delay_seconds: float = Field(default=2.0, ge=0)

退避时间是部署策略，不应写死在业务代码中。ge=0 禁止负等待；默认值较小，保证教学项目不会因为 retry 变得难以运行。

@dataclass(frozen=True)
class AIClientConfig:
    retry_base_delay_seconds: float = 0.1
    retry_max_delay_seconds: float = 2.0

settings 是环境入口，AIClientConfig 是 provider adapter 接收的运行时配置。把配置集中传入 adapter，避免 adapter 自己读取全局环境。

def build_retry_delay_plan(... ) -> RetryDelayPlan:
    base_delay = max(config.retry_base_delay_seconds, 0.0)
    max_delay = max(config.retry_max_delay_seconds, 0.0)
    exponential_delay = base_delay * (2 ** (attempt - 1))
    jitter_delay = calculate_retry_jitter(config, random_fraction=random_fraction)
    return RetryDelayPlan(
        delay_seconds=min(exponential_delay + jitter_delay, max_delay),
        source="local_backoff",
    )

max(..., 0.0) 是第二道防线，防止测试或内部代码直接构造负数配置。2 ** (attempt - 1) 让第 1 次失败等待 base，第 2 次等待 2 倍 base。min 是上限保护。

async def wait_before_retry(
    attempt: int,
    config: AIClientConfig,
    sleep: RetrySleep,
    ...
) -> None:
    delay_plan = build_retry_delay_plan(...)
    if delay_plan.delay_seconds > 0:
        await sleep(delay_plan.delay_seconds)

sleep 被注入而不是写死，生产默认用 asyncio.sleep，测试传 fake async 函数记录参数。没有注入，测试只能真的等，既慢又不稳定。

if attempt < config.max_attempts:
    await wait_before_retry(...)
    continue

只有还存在下一次 attempt 时才等待。最后一次失败后再 sleep 没有意义，只会拖慢错误返回。

④ 核心机制运行全流程拆解

attempt 1 失败
  → 判断错误可重试
  → attempt < max_attempts
  → delay = min(base * 2 ** 0, max_delay)
  → await sleep(delay)
  → continue 到 attempt 2

attempt 2 失败
  → delay = min(base * 2 ** 1, max_delay)
  → await sleep(delay)
  → continue 到 attempt 3

最后一次失败
  → 不 sleep
  → retry_exhausted / AIClientError

数据流从 Settings 进入 AIClientConfig，再进入 request_provider_prediction()，最后由 wait_before_retry() 消费。调用链保持在 adapter 层，不污染 router 和 service。

⑤ 重难点、易混淆点对比辨析

易混点正确理解隐患
retry vs backoffretry 是再试，backoff 是再试前等待只有 retry 会放大故障
attempt 编号第 1 次失败指数为 2 ** 0写错会让等待翻倍或少一档
最后一次失败是否 sleep不 sleep无意义拖慢错误响应
asyncio.sleep vs time.sleep前者让出事件循环，后者阻塞线程高并发下吞吐下降

⑥ 开发常见报错+坑点+解决方案

  • 测试耗时突然变长：真实执行了 asyncio.sleep。解决：注入 fake_sleep，只记录 delay。
  • 等待时间无限变大：只做指数增长，没有 cap。解决：用 retry_max_delay_seconds 截断。
  • 负数等待：内部构造了非法 config。解决：settings 层 Field(ge=0)，计算层再 max(..., 0)。
  • 不可重试错误也 sleep：等待逻辑放错位置。解决：先判断 retryable 与 attempt 预算，再调用 wait_before_retry()。

⑦ 生产环境 vs 教学环境 核心差异

维度教学环境生产环境
退避参数小默认值，便于测试和演示结合 SLA、总超时、供应商限流配置
等待位置请求内 await sleep长等待可能改为队列任务或后台重试
策略完整度先讲指数退避通常再加 Retry-After、jitter、熔断
测试fake sleep 验证 delay还会验证负载、超时预算、监控指标

⑧ 课后记忆习题+标准答案

1. 为什么不能在失败后立即重试？

大量请求会同步重试，放大 provider 故障，形成请求风暴。

2. base=1、max=3 时前四次失败后的 delay 是多少？

分别是 1、2、3、3 秒；第 3 次开始被最大等待上限截断。

3. 为什么 async 代码中不用 time.sleep？

time.sleep 阻塞线程；await asyncio.sleep 挂起当前协程，让事件循环处理其他任务。

4. 为什么测试要注入 fake sleep？

这样可以断言等待参数，同时避免测试真实等待。

⑨ 面试高频真题+解析

题 1：什么是指数退避？ 答：连续失败后按指数增长等待时间，例如 base、2*base、4*base，并使用最大值截断，避免等待无限增长。

题 2：为什么 retry 要有 backoff？ 答：外部服务故障时立即重试会提高瞬时流量；backoff 能降低 provider 压力，提高整体恢复概率。

题 3：如何测试 backoff 而不拖慢测试？ 答：把 sleep 函数作为依赖注入，测试传入 async fake sleep 收集 delay 列表并断言。
