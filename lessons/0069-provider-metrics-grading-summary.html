0069 Provider Metrics Grading Summary：把单题评分聚合成训练汇总

本节新增 summarize_provider_metrics_runbook_exercise_grades()，把多张题卡与按 card_id 索引的作答合并，输出总题数、已作答数、通过数、失败数、未作答数和每题评分结果。它把 0068 的单题锚点评分升级为可复盘的批量训练统计。

① 本节核心知识框架

Grading Summary
├─ 输入
│  ├─ cards: exercise cards
│  └─ responses_by_card_id: dict[card_id, response]
├─ 复用单题评分
│  └─ grade_provider_metrics_runbook_exercise_answer(card, response)
├─ 汇总字段
│  ├─ total
│  ├─ answered
│  ├─ passed
│  ├─ failed
│  ├─ unanswered
│  └─ grades
├─ 每题 grade 补充
│  └─ answered: 是否收到作答
└─ 设计边界
   ├─ 不保存原始 response
   ├─ 不做持久化
   ├─ 不调用 provider
   └─ 不改变题卡或标准答案

② 核心概念底层原理

单题评分回答“这一题是否覆盖关键锚点”，批量汇总回答“这一轮训练整体完成得怎样”。二者是不同层级：单题评分关注局部判断，汇总层关注训练完成度和失败分布。

本节使用 responses_by_card_id 而不是列表索引，是为了让作答与题卡通过稳定 ID 对齐。列表位置适合临时展示，但一旦插题、删题或分页，位置索引就容易错位；ID 映射更适合工程数据流。

unanswered 与 failed 必须区分。未作答是一种训练完成度问题；已作答但缺锚点是知识掌握问题。若把两者混在一个失败数里，复盘时无法判断是学生没交，还是排障思路错误。

指标含义复盘价值
total本轮训练题卡总数确定统计分母
answered收到 response 的题目数衡量完成度
passed两个锚点都命中的题目数衡量掌握度
failed未通过题目数定位需要复习的题
unanswered没有 response 的题目数定位漏答或数据缺口

③ 全套代码逐行精读解析

def summarize_provider_metrics_runbook_exercise_grades(
    cards: list[dict[str, object]],
    responses_by_card_id: dict[str, str],
) -> dict[str, object]:
    grades = []
    answered = 0

函数输入是题卡列表和按题卡 ID 索引的作答字典。grades 收集每题评分结果，answered 单独统计已提交题数。若只依赖 grades 事后推断，会把“未作答”和“作答错误”混淆。

for card in cards:
    card_id = str(card["id"])
    if card_id in responses_by_card_id:
        answered += 1

每张题卡都从 card["id"] 取稳定标识，并转成字符串对齐 responses_by_card_id 的 key 类型。若题卡缺少 id，这里会抛出 KeyError，说明题卡契约已经损坏，不能继续静默评分。

grade = grade_provider_metrics_runbook_exercise_answer(
    card,
    responses_by_card_id.get(card_id, ""),
)

汇总层不重新实现锚点评分，而是复用 0068 的单题函数。未作答时传入空字符串，让单题评分自然得到缺少两个锚点的结果；同时汇总层会额外标记 answered=False。

grade["answered"] = card_id in responses_by_card_id
grades.append(grade)

这里给每题结果补充 answered 字段。它不是单题评分算法的一部分，而是批量训练上下文：同样未通过，已作答失败和未作答失败的教学处理完全不同。

passed = sum(1 for grade in grades if grade["passed"])
total = len(cards)
return {
    "total": total,
    "answered": answered,
    "passed": passed,
    "failed": total - passed,
    "unanswered": total - answered,
    "grades": grades,
}

passed 从每题结果计算，避免重复实现通过逻辑。failed = total - passed 表示未通过总数，包含未作答；unanswered 单独暴露漏答数量，避免复盘时误读失败原因。

④ 核心机制运行全流程拆解

cards
  + responses_by_card_id
  → for each card
      → find response by card_id
      → grade single answer
      → attach answered flag
  → aggregate counts
  → summary report object

调用链坚持单向数据流：题卡定义标准，作答字典提供输入，单题评分产生 grade，汇总函数只做聚合。任何一步都不反向修改题卡、findings、metrics counter 或 provider 状态。

⑧ 课后记忆习题+标准答案

1. 为什么 responses 使用 card_id 作为 key？

因为题目顺序可能变化，ID 映射能保证作答和题卡稳定对齐，避免列表下标错位。

2. 为什么未作答也要生成 grade？

未作答是训练结果的一部分，跳过会缩小统计分母，让通过率和失败数失真。

3. failed 和 unanswered 的区别是什么？

failed 表示未通过总数，unanswered 表示没有提交作答的子集。前者衡量掌握结果，后者衡量完成度缺口。

4. 为什么 summary 不保存 response 原文？

原始作答可能包含敏感文本，也会增加报告噪声。教学汇总只需要评分结果、缺失锚点和统计数字。

5. 为什么批量函数要复用单题评分函数？

复用可以保证评分规则只有一个事实源，避免单题评分和批量评分出现不同判断。

⑨ 面试高频真题+解析

面试题 1：如何设计批量评分的结果结构？

结果应包含总体统计和逐题明细。总体字段用于看完成度和通过情况，逐题 grade 用于定位缺失锚点和具体失败原因。

面试题 2：为什么不能只统计通过率？

通过率会掩盖未作答、题目难度和失败原因。工程系统需要 answered、failed、unanswered 和 missing 才能支持复盘。

面试题 3：批量评分为什么不应复制单题评分逻辑？

复制会产生规则漂移。单题评分函数应是评分规则的唯一入口，批量层只负责遍历、补上下文和统计。

面试题 4：生产演练系统如何扩展本节设计？

需要加入题库版本、用户身份、批次 ID、持久化、审计、并发提交处理、作答脱敏和人工复核流程。
