Lesson 0019

服务层：让 router 变薄，让业务逻辑有地方住

项目刚开始时，把逻辑写在路由函数里很自然。但当上传、任务、摘要、数据库和前端都出现后，我们需要给业务逻辑单独安排一层。

0. 本节学习目标

  • 理解 router、service、database model 各自负责什么。
  • 看懂为什么把任务创建逻辑抽到 services/summary_tasks.py。
  • 理解“瘦路由”为什么更适合求职项目和 AI 服务项目。
  • 能判断一段代码应该留在 router，还是搬到 service。

1. 为什么现在需要服务层

前面的项目已经有不少能力：

  • API Key 鉴权。
  • 数据库 Session。
  • 上传文件。
  • 创建后台摘要任务。
  • 静态前端调用 API。

如果所有逻辑都继续塞在 router 里，路由函数会越来越长。长路由函数最明显的问题不是“丑”，而是难复用、难测试、难替换模型。

2. 三层职责

Router 层

接 HTTP 请求，读路径参数、请求体、依赖项，返回响应或 HTTPException。

Service 层

表达业务动作，例如创建摘要任务、运行摘要任务、查询任务记录。

Model / Database 层

定义数据长什么样，以及如何通过 Session 保存和读取数据。

一句话记忆：router 管 HTTP，service 管业务，model 管数据形状。

3. 本节新增目录

first_api/
  services/
    __init__.py
    summary_tasks.py
    ai_clients.py

services/summary_tasks.py 放任务相关业务动作；services/ai_clients.py 放模型客户端，下一课会重点讲。

4. 抽出任务创建逻辑

原来 router 里直接写：

task = SummaryTask(
    text=payload.text,
    text_length=len(payload.text),
    status="queued",
)
session.add(task)
session.commit()
session.refresh(task)

现在改成服务函数：

def create_summary_task(
    session: Session,
    text: str,
    source_file_id: int | None = None,
) -> SummaryTask:
    task = SummaryTask(
        text=text,
        text_length=len(text),
        source_file_id=source_file_id,
        status="queued",
    )
    session.add(task)
    session.commit()
    session.refresh(task)
    return task

这段代码现在可以被“直接文本摘要任务”和“文件摘要任务”一起复用。

5. router 变薄之后

@router.post("/summaries", response_model=SummaryTaskRead, status_code=202)
def create_summary_task(
    payload: SummaryRequest,
    background_tasks: BackgroundTasks,
    session: Annotated[Session, Depends(get_session)],
) -> SummaryTask:
    task = create_summary_task_record(
        session=session,
        text=payload.text,
    )
    background_tasks.add_task(process_summary_task, task.id)
    return task

现在路由函数的结构很清楚：接收请求、调用服务、安排后台任务、返回响应。它不再关心任务对象怎么保存进数据库。

6. 命名冲突的小坑

服务函数叫 create_summary_task，路由函数也叫 create_summary_task 时，Python 名字会互相遮住。课程代码里用了别名：

from ..services.summary_tasks import (
    create_summary_task as create_summary_task_record,
)

这个别名让读者知道：一个是 HTTP 路由函数，一个是数据库任务记录的创建函数。

7. 判断规则：什么放 service

  • 如果代码在描述“业务动作”，适合放 service。
  • 如果代码只在处理 HTTP 参数、状态码、请求头，适合留在 router。
  • 如果代码在定义字段、表、校验范围，适合放 schema/model。
  • 如果代码要被多个 router 或 worker 复用，优先考虑 service。

9. 小测：先回忆再展开

1. 为什么 router 不适合越来越胖？

因为它会混合 HTTP、数据库和业务逻辑，导致复用、测试和替换都更困难。

2. service 层主要表达什么？

表达业务动作，例如创建任务、运行任务、查询任务记录。

3. 为什么服务函数和路由函数重名会出问题？

因为 Python 名字会被后定义的同名函数遮住，调用时可能递归调用自己或找错对象。

10. 本节总结

服务层不是 FastAPI 强制要求的语法，而是真实项目里常用的组织方式。它让 router 专心处理 HTTP，让业务逻辑更容易被测试、复用和替换。
