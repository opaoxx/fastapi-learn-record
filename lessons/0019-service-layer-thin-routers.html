服务层与瘦路由：FastAPI 项目从脚本式接口走向可维护分层

本节将任务创建、文件摘要任务创建、任务读取等业务动作从 HTTP 路由中剥离到 services/summary_tasks.py。核心目标不是“多建一个目录”，而是让 API 层只表达协议，让 service 层承载业务不变量，让数据库模型只描述持久化结构。

① 本节核心知识框架

服务层分层模型
├─ Router 层：HTTP 协议适配
│  ├─ 读取 path/query/body/header/dependency
│  ├─ 设置 status_code、response_model、OpenAPI 元数据
│  └─ 把 HTTPException 映射给客户端
├─ Service 层：业务用例编排
│  ├─ create_summary_task：创建 queued 任务记录
│  ├─ create_summary_task_from_file：复用任务创建逻辑
│  ├─ read_summary_task：按 id 读取任务
│  └─ run_summary_task：执行摘要并落库状态
├─ Schema/Model 层：数据形状
│  ├─ SummaryTask：数据库表模型
│  ├─ SummaryTaskRead：对外响应模型
│  └─ SummaryTaskListResponse：列表 envelope
└─ 判断口诀：router 管协议，service 管业务，model 管结构

② 核心概念底层原理

FastAPI 不强制使用 service 层，但真实项目必须区分“协议逻辑”和“业务逻辑”。HTTP 路由函数天然依赖请求体、路径参数、状态码、鉴权依赖和响应模型；业务函数天然关心任务如何创建、如何关联文件、如何切换状态、如何写入数据库。二者混在一起会导致路由越来越胖，测试必须通过 HTTP 才能验证业务规则，复用也会被协议细节卡住。

服务层的本质是 application service：它不是数据库仓储，也不是领域实体，而是一个面向用例的编排层。比如“从上传文件创建摘要任务”不是 SQLModel 的职责，也不是 HTTP 的职责；它是业务动作，适合放入 create_summary_task_from_file()。

与相似概念区分：APIRouter 负责路由注册和 HTTP 适配；Depends 负责依赖解析；SQLModel 负责模型和查询；service 负责把这些能力组合成稳定用例。service 不是“工具函数垃圾桶”，只放有业务语义、可复用、值得测试的动作。

③ 全套代码逐行精读解析

def create_summary_task(
    session: Session,
    text: str,
    source_file_id: int | None = None,
) -> SummaryTask:
    task = SummaryTask(
        text=text,
        text_length=len(text),
        source_file_id=source_file_id,
        status="queued",
    )
    session.add(task)
    session.commit()
    session.refresh(task)
    return task
  • session: Session：由外部传入数据库会话，service 不自己创建连接，便于测试和事务控制；如果在函数内部随意创建 Session，会让调用方难以控制生命周期。
  • text: str：业务输入是纯文本，不依赖 SummaryRequest。这样同一函数可被文本接口、文件接口、测试代码复用。
  • source_file_id: int | None = None：允许任务不关联文件；如果强制传文件 ID，直接文本摘要场景会被破坏。
  • SummaryTask(...)：构造数据库表对象，此时还没有主键；主键由数据库在 commit/flush 后生成。
  • text_length=len(text)：把文本长度固化为查询友好的字段；不保存则每次响应都要重新计算，且历史输入变更时含义不稳定。
  • status="queued"：任务初始状态必须是排队中；如果直接写 running，前端会误判后台已经开始处理。
  • session.add(task)：把对象加入当前 Session 的持久化上下文；漏掉后 commit 不会插入记录。
  • session.commit()：提交事务并让插入落库；漏掉后另一个请求查询不到任务。
  • session.refresh(task)：从数据库回填生成后的 id 等字段；漏掉可能返回 id=None 或陈旧对象。
  • return task：返回持久化后的任务，router 可直接交给 FastAPI 按 SummaryTaskRead 序列化。
from ..services.summary_tasks import (
    create_summary_task_from_file,
    create_summary_task as create_summary_task_record,
)

create_summary_task as create_summary_task_record 解决命名遮蔽：router 里也有 def create_summary_task(...)。如果不改别名，函数名可能读起来像递归调用自己，维护者很难分辨“HTTP 处理函数”和“数据库记录创建函数”。

@router.post("/summaries", response_model=SummaryTaskRead, status_code=status.HTTP_202_ACCEPTED)
def create_summary_task(...):
    task = create_summary_task_record(session=session, text=payload.text)
    background_tasks.add_task(process_summary_task, task.id)
    return task
  • @router.post：声明 HTTP 入口，仍然留在 router 层。
  • response_model=SummaryTaskRead：对外隐藏原始文本，只返回任务状态、长度、结果和错误字段；若直接返回表模型，可能泄露 text。
  • status.HTTP_202_ACCEPTED：表示任务被接受但未完成；如果用 200，会混淆提交成功与处理成功。
  • create_summary_task_record(...)：router 委托 service 创建任务，不直接写数据库插入细节。
  • background_tasks.add_task(...)：注册响应后执行的后台工作；不在 service 中调用是因为它属于 FastAPI HTTP 生命周期能力。
  • return task：返回 queued 任务回执，让客户端拿到 id 后继续查询。

④ 核心机制运行全流程拆解

POST /tasks/summaries
  -> require_api_key 鉴权
  -> Pydantic 校验 SummaryRequest.text
  -> get_session 注入 SQLModel Session
  -> router.create_summary_task()
  -> service.create_summary_task_record()
       -> 构造 SummaryTask(status="queued")
       -> add/commit/refresh
       -> 返回带 id 的任务记录
  -> BackgroundTasks 注册 process_summary_task(task.id)
  -> FastAPI 按 SummaryTaskRead 序列化响应
  -> HTTP 202 返回客户端
  -> 响应结束后后台任务继续执行

调用链的分界点非常清楚：HTTP 输入解析、鉴权和响应状态在 router；任务记录创建的不变量在 service；表字段与响应字段在 schema/model。这样一来，任何一层变化都不必牵连全部代码。

⑤ 重难点、易混淆点对比辨析

问题正确边界常见误解
router vs servicerouter 适配 HTTP，service 表达业务用例把所有代码都写在路由函数里
service vs CRUD helperservice 可以编排多个动作和业务状态以为 service 只能是一层薄薄的数据库 wrapper
表模型 vs 响应模型SummaryTask 入库，SummaryTaskRead 对外直接暴露表模型导致字段泄露
commit 位置本课 service 内部提交，教学简单清晰生产中所有 service 都必须自己 commit
命名别名用 create_summary_task_record 表达语义忽视 Python 名字遮蔽导致读错调用对象

⑥ 开发常见报错+坑点+解决方案

  • 任务返回 id: null：通常是没有 commit/refresh。解决：插入后提交并刷新对象。
  • 路由函数递归调用自己：service 函数和 router 函数同名且未起别名。解决：导入时使用 as create_summary_task_record。
  • 测试必须启动 HTTP 才能验证业务：业务逻辑写死在 router。解决：把可复用规则移动到 service，测试直接传 Session 调用。
  • 返回体泄露原文 text：错误地把表模型作为响应模型。解决：使用 SummaryTaskRead 控制字段。
  • 数据库会话已关闭：在后台任务里复用请求 Session。解决：后台任务单独创建 Session，见后续 task_worker.py。

⑦ 生产环境 vs 教学环境 核心差异

⑧ 课后记忆习题+标准答案

1. 为什么说 service 层不是 FastAPI 语法，而是工程设计？

FastAPI 可以直接在 router 中完成所有逻辑，但当业务动作需要复用、测试、替换实现时，把用例抽到 service 能降低协议层和业务层耦合，这是工程可维护性设计。

2. session.refresh(task) 解决什么问题？

它从数据库重新加载对象状态，拿到自增主键等数据库生成字段，避免返回未刷新或陈旧对象。

3. 为什么任务创建接口返回 202 而不是 200？

因为接口只保证任务已被接收和排队，不保证摘要已经生成；最终结果需要通过任务查询接口读取。

4. 哪些逻辑应该留在 router？

路径参数、查询参数、请求体、依赖注入、状态码、响应模型、HTTPException 等协议适配逻辑应留在 router。

⑨ 面试高频真题+解析

题 1：FastAPI 项目为什么要做瘦路由？

瘦路由把 HTTP 适配和业务规则拆开，路由更短，service 可复用可单测，后续替换数据库、队列、AI 客户端时影响范围更小。面试回答要强调边界，不要只说“代码更好看”。

题 2：service 层应该直接抛 HTTPException 吗？

通常不建议。HTTPException 是协议层概念，service 更适合返回业务结果或抛业务异常，由 router 映射成 HTTP 状态码。本课程早期为了简洁，404 仍在 router 中处理。

题 3：为什么不能把请求级数据库 Session 传给后台任务长期使用？

请求结束后 Session 生命周期可能结束，连接也可能被释放。后台任务应在自己的执行上下文中重新创建 Session，确保事务和连接独立。
