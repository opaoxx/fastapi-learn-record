Lesson 0010

自动化测试：用 TestClient 证明 API 没坏

手动点 /docs 很适合学习，但真实项目不能每次改代码都靠人肉测试。本节学习用 pytest 和 TestClient 自动验证接口行为。

0. 本节学习目标

  • 理解为什么后端项目需要自动化测试。
  • 理解 TestClient 像一个“代码里的浏览器/HTTP 客户端”。
  • 看懂如何用 app.dependency_overrides 替换数据库依赖。
  • 能运行 python -m pytest -q 并读懂通过/失败结果。

1. 为什么需要测试

当接口只有一个时，手动点一下就够了。但现在项目已经有创建、读取、更新、删除、数据库、依赖注入。每次改代码都手动检查全部接口，很容易漏掉。

自动化测试的价值是：把“这个接口应该怎样工作”写成代码。以后你改任何地方，只要跑测试，就能快速知道核心行为有没有坏。

求职项目视角：有测试的后端项目会显得更专业，因为它不仅能跑，还能自证关键路径可靠。

2. TestClient 是什么

FastAPI 官方文档推荐用 TestClient 测试应用。它让你不用真的打开浏览器，也不用启动独立服务器，就能向 app 发请求。

from fastapi.testclient import TestClient
from first_api.main import app

with TestClient(app) as client:
    response = client.get("/health")
    assert response.status_code == 200

client.get、client.post、client.patch、client.delete 的使用方式很像真实 HTTP 客户端。

3. 测试为什么不能直接用正式数据库

如果测试直接使用 first_api.db，会有几个问题：

  • 测试会污染你的本地开发数据。
  • 不同测试之间可能互相影响。
  • 测试结果会依赖你机器上数据库当前有什么数据。

所以第 10 课用了一个临时内存 SQLite 数据库，并通过依赖覆盖把正式 get_session 换掉。

4. 依赖覆盖：dependency_overrides

app.dependency_overrides[get_session] = override_get_session

FastAPI 官方文档说明，测试时可以用 app.dependency_overrides 替换原依赖。这里的意思是：当接口里写 Depends(get_session) 时，测试期间不要调用正式数据库 session，而是调用我们提供的测试 session。

这也是第 6 课 Depends 的延伸：依赖抽得越清楚，测试时越容易替换。

5. 测试数据库怎么创建

engine = create_engine(
    "sqlite://",
    connect_args={"check_same_thread": False},
    poolclass=StaticPool,
)
SQLModel.metadata.create_all(engine)

sqlite:// 表示内存数据库。StaticPool 让测试过程中复用同一个连接，否则内存数据库可能在不同连接之间丢失表和数据。

测试启动时创建表，再插入两条种子商品，保证每次测试都有稳定起点。

6. 测试完整 CRUD

created = client.post("/items", json={...})
assert created.status_code == 201
item_id = created.json()["id"]

updated = client.patch(f"/items/{item_id}", json={"price": 18.0})
assert updated.status_code == 200
assert updated.json()["price"] == 18.0

deleted = client.delete(f"/items/{item_id}")
assert deleted.status_code == 204

missing = client.get(f"/items/{item_id}")
assert missing.status_code == 404

这不是随便测几个状态码，而是在验证一个完整用户故事：创建一个商品，更新它，删除它，再确认它已经不存在。

7. 测试非法输入

response = client.patch("/items/1", json={"price": -1})
assert response.status_code == 422

这条测试证明 Pydantic/SQLModel 的校验仍然有效。接口不仅要能处理正确请求，也要能拒绝错误请求。

8. 运行测试

.\.venv\Scripts\Activate.ps1
python -m pytest -q

如果看到类似：

2 passed

说明当前测试全部通过。如果看到 FAILED，就要看失败的 assert 和错误堆栈，定位是接口行为变了，还是测试预期写错了。

9. 课堂练习

请你新增一个测试：创建商品时传 "price": -1，断言返回 422。

完成标准：运行 python -m pytest -q 后所有测试通过。

10. 小测：先回忆再展开

1. TestClient 的作用是什么？

它让测试代码像 HTTP 客户端一样向 FastAPI app 发请求，并检查响应。

2. 为什么测试要覆盖 get_session？

为了让测试使用独立测试数据库，避免污染或依赖正式开发数据库。

3. assert response.status_code == 422 在证明什么？

它证明非法输入被接口模型拦住，没有进入正常业务流程。

11. 本节总结

自动化测试把接口预期写成代码。TestClient 负责发请求，pytest 负责发现和运行测试，dependency_overrides 让测试可以替换真实依赖。对后端和 AI 服务来说，这都是工程可靠性的基础。

推荐主资源：FastAPI Testing、FastAPI Testing Dependencies with Overrides。
