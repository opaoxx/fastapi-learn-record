0079 GitHub Pull Request Template：把贡献证据写进 PR 审查入口

本节新增 .github/PULL_REQUEST_TEMPLATE.md，把变更摘要、变更类型、测试证据、课程文档清单和风险说明固化为 PR 默认模板。CI 负责自动检查，branch protection 负责阻断合并，PR template 负责让人类审查拿到足够上下文。

① 本节核心知识框架

GitHub Pull Request Template
├── 入口文件
│   └── .github/PULL_REQUEST_TEMPLATE.md
├── 审查信息
│   ├── Summary: 改了什么，为什么改
│   ├── Change Type: 代码、课程、文档、测试/CI
│   ├── Verification: 测试命令执行证据
│   ├── Course Documentation Checklist: 课程配套文件完整性
│   └── Risk Notes: API、导航、后续工作风险
├── 与前两节的关系
│   ├── CI 自动验证
│   ├── branch protection 阻断失败合并
│   └── PR template 提供人工 review 上下文
└── 工程目标
    ├── 降低 reviewer 追问信息成本
    ├── 防止新课只加 HTML 漏掉 reference/index/record
    ├── 让测试证据进入协作流程
    └── 让开源新手知道合格 PR 应交付什么

② 核心概念底层原理

Pull request 是 GitHub 中“变更进入主分支前的审查单元”。它不仅承载 diff，也承载提交意图、测试证据、风险说明和 reviewer 讨论。没有 PR 模板时，贡献者经常只写“update files”，reviewer 必须从 diff 中反推上下文，审查成本极高。

PR template 的底层机制很简单：GitHub 在创建 PR 时读取仓库中的 .github/PULL_REQUEST_TEMPLATE.md，把文件内容预填到 PR description。它不强制用户勾选，但它把仓库期望显式放在提交入口处，属于低成本高收益的协作约束。

本仓库是课程型代码仓库，PR 风险不只来自代码 bug，还来自课程导航断裂、速查页漏改、README 入口过期、文档契约测试缺失。模板必须覆盖代码、课程、文档、测试和风险五类信息，否则 reviewer 只能人工凭经验排查。

工具职责不能替代什么
PR template收集变更说明与测试证据不能替代自动测试。
CI workflow自动执行测试与检查不能解释业务意图。
branch protection阻断不合格合并不能告诉 reviewer 变更背景。
review comment讨论具体问题不能稳定沉淀每次 PR 必填信息。

模板中的 checkbox 不是形式主义。它把“该不该更新 reference”“有没有跑文档契约测试”“是否存在 schema 风险”等隐性经验转成显性清单。对 GitHub 新手而言，这等价于一份最小贡献规范。

③ 全套代码逐行精读解析

# Pull Request Checklist

标题明确模板用途：这是提交 PR 前的检查清单，不是自由文本留言板。reviewer 打开 PR 时首先看到该结构，能快速判断贡献者是否给出了必要证据。

## Summary

- What changed:
- Why it changed:

Summary 要求同时说明“改了什么”和“为什么改”。只写改动内容无法判断设计动机，只写动机又无法快速定位 diff。两者同时存在，reviewer 才能把需求、实现和测试对应起来。

## Change Type

- [ ] Code behavior
- [ ] Lesson or reference page
- [ ] README, DEVELOPMENT, or GitHub documentation
- [ ] Tests or CI

Change Type 将 PR 分类。课程仓库中常见变更跨越代码、课程、根文档、测试和 CI。分类后 reviewer 可以决定应重点看源码、HTML 结构、导航链路还是质量门禁。

## Verification

- [ ] Ran provider HTTP tests: `.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q`
- [ ] Ran documentation contract tests: `.\.venv\Scripts\python.exe -m pytest tests\test_course_docs_contract.py -q`
- [ ] Ran full test suite: `.\.venv\Scripts\python.exe -m pytest -q`
- [ ] Ran whitespace check: `git diff --check`

Verification 把本仓库维护底线写进 PR。这里使用 Windows PowerShell 命令，是因为当前课程维护环境以该路径为准；CI 中会用 Linux 命令再次验证。模板要求贡献者报告本地验证，CI 提供独立复核。

## Course Documentation Checklist

- [ ] Updated the lesson HTML in `lessons/` when lesson content changed.
- [ ] Updated the quick reference page in `reference/` when lesson content changed.
- [ ] Updated `index.html` navigation when adding or renaming lessons.
- [ ] Updated `NOTES.md` and `COURSE_PROGRESS.md`.
- [ ] Added or updated documentation contract tests when public links, commands, or required structures changed.

这是课程仓库特有的清单。普通后端项目通常只关心源码和测试；本项目还要维护 lesson、reference、index、learning-records、NOTES、COURSE_PROGRESS 和文档契约测试。模板能防止“新增一课但没有入口”的孤岛页面。

## Risk Notes

- Breaking API or schema changes:
- Documentation or navigation risks:
- Follow-up work:

Risk Notes 要求贡献者主动暴露风险。API/schema 风险影响代码调用者，文档/导航风险影响学习者，follow-up work 避免未完成事项被默默遗忘。生产环境还会扩展安全、性能、迁移和回滚风险。

def test_pull_request_template_guides_course_contributors() -> None:
    template = Path(".github/PULL_REQUEST_TEMPLATE.md").read_text(encoding="utf-8")

文档契约测试读取 PR 模板。模板是 GitHub 协作入口，一旦关键清单被删，贡献流程会退化，因此它必须像 README 和 CI workflow 一样纳入测试。

assert "Summary" in template
assert "Verification" in template
assert "Course Documentation Checklist" in template
assert "python -m pytest -q" in template
assert "git diff --check" in template

断言保护模板骨架，不锁死完整文案。未来可以改进措辞，但不能丢掉摘要、验证、课程文档清单、全量测试和 whitespace 检查。

④ 核心机制运行全流程拆解

Contributor opens pull request
  -> GitHub reads .github/PULL_REQUEST_TEMPLATE.md
  -> PR description is prefilled
  -> Contributor writes summary, checks type, records verification
  -> CI workflow runs automatically
  -> Branch protection waits for required check
  -> Reviewer compares:
       PR description intent
       code and lesson diff
       CI result
       documentation checklist
       risk notes
  -> PR can merge only when required checks and review rules pass

PR template 不参与代码执行路径，但参与协作数据流。它把贡献者知道的信息提前结构化，减少 reviewer 在 diff、聊天记录和历史上下文之间来回查找。

阶段输入输出价值
创建 PR模板文件预填 description让贡献要求前置。
填写模板变更意图和本地验证审查证据降低 reviewer 追问成本。
CI 执行提交 diff自动检查结果验证描述是否与仓库状态一致。
Reviewdescription、diff、checks批准或要求修改将人类判断集中在设计和风险。

⑥ 开发常见报错+坑点+解决方案

坑点后果解决方案
模板路径写错GitHub 创建 PR 时不会预填模板。使用 .github/PULL_REQUEST_TEMPLATE.md 或 GitHub 支持的模板目录结构。
模板过长贡献者机械删除内容，review 成本反而上升。只保留高价值信息：summary、type、verification、docs checklist、risk。
缺少测试命令贡献者不知道该报告哪些验证。把仓库维护底线命令直接写入 Verification。
缺少课程文档清单新增课程容易漏 reference、index 或进度记录。把课程配套文件列为显式 checkbox。
模板更新后无测试未来误删关键项无人发现。使用 test_pull_request_template_guides_course_contributors 保护模板骨架。
把模板当作审批替代品坏 PR 勾选所有项后仍可能合并。模板只提供审查输入，仍需 CI、required checks 和 reviewer 判断。

⑧ 课后记忆习题+标准答案

1. PR template 解决的核心问题是什么？

它把变更意图、类型、测试证据、课程文档完整性和风险说明前置到 PR 入口，降低 reviewer 从 diff 中反推上下文的成本。

2. PR template 能否替代 CI？

不能。模板是贡献者自述和自检，CI 是独立环境中的自动验证。两者互补。

3. 为什么课程仓库模板必须包含 Course Documentation Checklist？

因为新增或修改课程通常牵动 lesson、reference、index、NOTES、COURSE_PROGRESS 和文档契约测试，缺少清单容易产生断链或孤岛页面。

4. 为什么模板中要写测试命令？

它让贡献者明确本仓库的本地验证底线，也让 reviewer 能快速判断是否缺少关键验证。

5. Risk Notes 至少应该覆盖哪些风险？

至少覆盖 breaking API/schema changes、documentation or navigation risks、follow-up work。本仓库重点关注代码契约和学习路径完整性。

⑨ 面试高频真题+解析

面试题 1：PR template、CI、branch protection 如何分工？

PR template 收集人类上下文和验证声明；CI 在独立环境执行测试；branch protection 把 CI 结果变成合并门禁。三者分别对应描述、验证和治理。

面试题 2：为什么优秀 PR 需要解释 why 而不只是 what？

what 只能说明 diff 内容，why 才能让 reviewer 判断设计选择是否合理、替代方案是否考虑充分、风险是否可接受。

面试题 3：如何避免 PR 模板变成形式主义？

模板应短而硬，只保留 reviewer 必须知道的信息；再用 CI、required checks 和 review 规则验证勾选内容是否真实。

面试题 4：为什么开源项目常把模板放在 .github/ 目录？

.github/ 是 GitHub 约定目录，可承载 workflow、issue template、PR template、CODEOWNERS 等协作资产，使仓库治理文件集中可见。

面试题 5：课程仓库和业务系统的 PR 风险有什么不同？

业务系统更关注线上行为、数据、性能和安全；课程仓库除代码行为外，还关注学习路径、讲义结构、导航链接、参考页和文档契约是否同步。
