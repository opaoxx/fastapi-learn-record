② 核心概念底层原理

Git 的核心模型是内容寻址对象数据库。commit 表示一次完整快照及其父提交；branch 是可移动引用，通常随开发继续向前；tag 是给某个对象命名的引用，适合标记稳定里程碑。课程仓库需要 tag，因为学习者需要知道“这批课程、代码、测试在某一刻完全一致”。

lightweight tag 本质上只是一个名字指向对象，缺少发布者、发布时间和说明。annotated tag 是独立 tag object，包含 tagger、message，可进一步 GPG/SSH 签名。课程批次发布属于交付事件，不只是临时书签，因此应优先使用 annotated tag。

GitHub Release 是 GitHub 托管层建立在 tag 之上的发布页面。tag 负责不可变定位，Release Notes 负责解释“为什么这个点值得发布”。二者不可互相替代：只有 tag 没有说明，学习者不知道改了什么；只有 Release 文案没有 tag，无法精确复现源码状态。

概念底层含义适用场景风险点
Branch可移动引用日常开发、PR 合并不能表示稳定发布点
Commit Hash内容寻址快照精确定位代码状态对新手不可读，不表达业务语义
Lightweight Tag命名引用本地临时标记缺少发布元数据
Annotated Tag带元数据的 tag object正式发布、课程批次发布后不应静默移动
GitHub Release基于 tag 的远程发布记录面向学习者说明变更与验证不能替代测试和 review

发布流程的核心不是“生成一个版本号”，而是把变更范围、验证证据和回滚路径变成公开契约。对课程项目而言，Release Notes 至少要说明 changed lessons、changed references、code/workflow changes、verification、known warnings、reading notes、rollback。

③ 全套代码逐行精读解析

# GitHub Release Process

标题定义文件职责：这不是普通备忘录，而是发布流程规范。放在 .github/ 下，表示它属于仓库协作治理的一部分，与 CI、PR 模板、分支保护、Discussions 同层。

This document defines how maintainers publish course batches with Git tags and GitHub Releases.

首句绑定业务对象：发布的不是 Python 包，而是 course batches。若写成泛泛的 release process，会让学习者误以为这是生产部署流程；这里明确是课程批次交付。

GitHub Releases are remote state.

该句是边界声明。Release 页面、发布资产、远程 tag 都在 GitHub 端；本地 Markdown 不能自动创建 Release。忽略这个边界，会误以为提交文件就完成了远程发布。

## Release Purpose

Purpose 先解释“为什么发布”。没有目的约束时，Release Notes 容易退化为流水账，无法支撑学习者判断是否需要更新、从哪一课继续读、哪些测试证明内容可用。

Releases turn a set of lesson, reference, code, test, and workflow changes into a traceable learning checkpoint.

这一行列出发布批次的完整组成：lesson、reference、code、test、workflow。课程仓库的交付单元不是单一代码包，而是学习资料和可运行项目的一致快照。

## Tag Naming

Tag Naming 规定命名空间。课程项目不用语义化版本号 v1.2.3 强行模拟库 API 兼容性，而使用日期批次号表达学习资料的发布节奏。

course-vYYYY.MM.DD-N

course 区分课程发布与未来可能出现的应用版本；YYYY.MM.DD 便于按时间排序；N 解决同一天多次发布。若没有 N，同日补丁发布会发生命名冲突。

Use annotated tags for course batch releases:

这里选择 annotated tag。正式课程发布需要 tagger、message 和可签名能力；lightweight tag 缺少审计元数据，不适合公开交付。

Never move a published tag silently.

这是发布治理底线。学习者和贡献者可能已经基于 tag 下载、引用或复现。如果静默移动 tag，同一个版本名会指向不同内容，破坏可追溯性。

git tag -a course-v2026.07.05-1 -m "Course batch 2026.07.05-1"

-a 创建 annotated tag；-m 写入 tag message。若漏掉 -a，Git 会创建 lightweight tag，发布元数据不足。

git push origin course-v2026.07.05-1

tag 不会随普通 git push 自动全部推送，必须显式推送 tag 或使用 --tags。教学环境推荐显式推送单个 tag，避免把本地临时 tag 一并发布。

## Release Notes Structure

Release Notes Structure 是发布文案模板。它让每次发布都能回答三个问题：改了什么、怎么验证、出问题怎么退。

## Changed Lessons
## Changed References
## Code and Workflow Changes
## Verification
## Known Warnings
## Upgrade or Reading Notes
## Rollback

这些章节按依赖顺序排列：先给学习内容范围，再给工程变更，再给证据与风险。若把 Verification 放在末尾且无结构，读者很难快速判断发布可信度。

## Verification Checklist

Verification Checklist 把发布前检查从“记得跑一下测试”变成可复用命令清单。课程项目必须保护代码行为和文档契约，不能只跑单一测试。

.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q

provider HTTP 是当前项目高风险业务模块，覆盖 provider retry、metrics、runbook training 等核心逻辑。先跑它能快速发现后端业务退化。

.\.venv\Scripts\python.exe -m pytest tests\test_course_docs_contract.py -q

docs contract 保护课程入口、九段结构、GitHub 治理文档链接。课程仓库里文档就是产品，因此文档测试与代码测试同级。

.\.venv\Scripts\python.exe -m pytest -q

full pytest 捕获跨模块回归。只跑目标测试会漏掉 OpenAPI、前端集成、系统 API、文件 API 等相邻行为。

git diff --check

该命令检查尾随空格、空白错误和补丁格式问题。Release 前必须执行，因为这些问题会污染公开仓库质量。

## Release Flow

Release Flow 把命令串成调用链：先完成批次内容，再更新入口文档，再验证，再审查 diff，最后创建 tag 和 GitHub Release。顺序不能颠倒。

finish lesson batch
-> update lessons, references, README, DEVELOPMENT, NOTES, COURSE_PROGRESS, index
-> run targeted tests
-> run docs contract
-> run full pytest
-> run git diff --check
-> review git status and diff
-> create annotated tag
-> push branch and tag
-> create GitHub Release from tag
-> paste release notes with verification evidence

流程中的每一步都对应可审计证据。若跳过 review git status and diff，可能把无关文件或半成品课程带入发布；若先打 tag 再测试，tag 可能固定了失败状态。

## Rollback

Rollback 说明发布错误后的处理方式。GitHub Release 不是本地草稿，发布后要面向已下载或引用该版本的学习者负责。

Do not move the published tag silently.

重复强调不可静默移动 tag，是因为这是最常见的版本治理事故。正确做法是新提交、新 tag、新 Release，并在旧 Release 标注 superseded。

## Production Notes

Production Notes 区分教学仓库和企业生产。真实组织还需要签名 tag、审批、制品校验、依赖扫描、SBOM、部署门禁；本课程只保留与学习交付直接相关的最小闭环。

⑧ 课后记忆习题+标准答案

1. 为什么课程批次发布优先使用 annotated tag？

因为 annotated tag 有独立 tag object，包含发布者、时间、message，并可签名。课程批次是公开交付事件，需要审计元数据，不适合只用 lightweight tag。

2. course-vYYYY.MM.DD-N 中 N 的作用是什么？

解决同一天多次发布的命名冲突。N 从 1 开始递增，使同日补丁或二次发布仍有唯一 tag。

3. Release Notes 必须包含哪些验证证据？

至少包含 provider HTTP、docs contract、full pytest、git diff --check 的结果，并说明 Known Warnings，例如非阻塞的 Windows LF-to-CRLF 提示。

4. 为什么不能静默移动已发布 tag？

因为外部学习者可能已经基于该 tag 下载、引用或复现。静默移动会让同一版本名指向不同内容，破坏可追溯性和信任。

5. 发布流程中为什么要先测试再打 tag？

tag 会固定某个历史快照。如果先打 tag 后测试，可能把失败状态标记为发布版本，后续需要额外补丁发布修正。

⑨ 面试高频真题+解析

面试题 1：Git tag 和 GitHub Release 的关系是什么？

tag 是 Git 对象数据库中的命名引用，用于定位 commit；GitHub Release 是 GitHub 基于 tag 提供的发布页面，用 Release Notes、资产和说明解释该 tag 的交付意义。

面试题 2：为什么生产发布不建议重写已经推送的 tag？

tag 一旦公开就可能被 CI、部署、依赖方、学习者或审计系统引用。重写 tag 会导致同一版本不可复现，正确策略是追加修复提交并发布新 tag。

面试题 3：Release Notes 和 commit message 的职责有什么不同？

commit message 解释一次提交为什么发生；Release Notes 面向用户或学习者解释一组提交形成的交付批次、影响范围、验证结果、风险和回滚方式。

面试题 4：课程仓库为什么要把 docs contract 纳入发布门禁？

因为课程仓库的文档是核心产品。docs contract 能保护九段结构、入口链接、GitHub 工作流说明和学习路径，防止代码可运行但课程不可读。

面试题 5：一个可靠发布流程至少包含哪些阶段？

完成变更、更新入口文档、运行目标测试、运行文档契约、运行全量测试、检查 diff、审查变更范围、创建 annotated tag、推送 tag、创建 Release Notes、记录回滚策略。
