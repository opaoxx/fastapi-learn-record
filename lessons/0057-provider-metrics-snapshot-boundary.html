0057 Provider Metrics Snapshot Boundary：快照不是内部状态后门

本节聚焦一个小但关键的封装边界：ProviderMetricsCounter.snapshot() 返回给外部调用方的应是当前状态的数据投影，而不是内部 Counter 或可变结构引用。读接口如果泄漏内部可变对象，本质上也会变成写入口。

① 本节核心知识框架

Snapshot Boundary
├─ 内部状态
│  └─ _counts: Counter[ProviderMetricLabels]
├─ 写入口
│  ├─ increment()
│  └─ reset()
├─ 读出口
│  └─ snapshot()
├─ 边界要求
│  ├─ 不返回内部 Counter
│  ├─ 不返回共享可变 labels
│  ├─ 返回 list/dict 数据投影
│  └─ 调用方修改 snapshot 不影响真实计数
└─ 测试保护
   └─ 修改 snapshot 后重新读取 counter，真实 count 仍为 1

② 核心概念底层原理

封装的核心不是把字段名前加下划线，而是让外部只能通过模块允许的操作改变状态。对于 counter，合法写入应只有 increment() 和 reset()。如果 snapshot() 返回内部对象引用，调用方就可以绕过这些方法直接改状态。

快照是“某一时刻状态的外部表示”。它可以被序列化、导出、断言、修改用于测试，但不应该与内部状态保持共享可变引用。这个原则同样适用于分页结果、配置对象、缓存视图、权限上下文和统计报表。

③ 全套代码逐行精读解析

@dataclass
class ProviderMetricsCounter:
    _counts: Counter[ProviderMetricLabels] = field(default_factory=Counter)

_counts 是内部状态。下划线表达“不要从外部依赖它”的约定，但 Python 不会强制阻止访问，所以仍需要方法边界和测试约束。

def snapshot(self) -> list[dict[str, object]]:
    samples = sorted(
        self._counts.items(),
        key=lambda item: (
            item[0].operation,
            item[0].outcome,
            item[0].failure_category,
            item[0].error_code,
            item[0].status_code,
        ),
    )
    return [
        {
            "labels": labels.as_dict(),
            "count": count,
        }
        for labels, count in samples
    ]
代码边界意义如果省略
sorted(self._counts.items())读取内部状态并生成稳定顺序输出顺序依赖内部遍历，测试不稳定
labels.as_dict()把不可变 label 对象投影成新字典可能泄漏内部 key 对象或共享结构
"count": count复制数值到外部样本调用方需要知道 Counter 内部结构
return [...]返回新 list外部可能修改内部集合
 
def test_provider_metrics_counter_snapshot_is_detached_from_counter_state() -> None:
    counter = ProviderMetricsCounter()
    labels = build_provider_metric_labels(outcome="success")

    counter.increment(labels)
    snapshot = counter.snapshot()
    snapshot[0]["count"] = 999
    snapshot[0]["labels"]["outcome"] = "fail_fast"

    assert counter.snapshot() == [
        {
            "labels": labels.as_dict(),
            "count": 1,
        }
    ]

测试故意篡改返回的 snapshot：把 count 改成 999，把 outcome 改成 fail_fast。重新读取 counter 后仍应是 success/count=1，证明 snapshot 与内部状态断开。

④ 核心机制运行全流程拆解

counter.increment(success_labels)
  → _counts[success_labels] = 1

snapshot = counter.snapshot()
  → 遍历 _counts
  → labels.as_dict() 生成新 dict
  → 返回新 list: [{"labels": new_dict, "count": 1}]

外部修改 snapshot
  → snapshot[0]["count"] = 999
  → snapshot[0]["labels"]["outcome"] = "fail_fast"

counter.snapshot()
  → 再次从 _counts 生成新快照
  → 仍然是 success + count 1

JSON endpoint 与 text export 都基于 snapshot：

def get_provider_metrics_snapshot() -> list[dict[str, object]]:
    return provider_prediction_metrics.snapshot()

def get_provider_metrics_prometheus_text() -> str:
    return render_provider_metrics_prometheus_text(get_provider_metrics_snapshot())

因此 snapshot 边界一旦泄漏，JSON 和 text 两个出口都会继承风险。

⑤ 重难点、易混淆点对比辨析

概念正确理解常见误区
下划线字段访问约定，不是强制安全以为 _counts 外部绝对无法访问
snapshot状态投影，适合读和序列化把它当成实时视图或内部引用
浅拷贝/深拷贝当前返回新 list 与新 dict，足够隔离现有结构盲目深拷贝所有对象，不分析数据结构
不可变 labels作为 Counter key 稳定可哈希把可变 dict 直接作为 key
读接口也需要防止泄漏可变状态认为只读函数天然安全

⑧ 课后记忆习题+标准答案

1. 为什么读接口也可能破坏状态？

如果读接口返回内部可变对象引用，调用方拿到后可以直接修改它，等价于绕过正式写入口。

2. snapshot() 为什么要返回新 list 和新 dict？

这样调用方修改返回值只会改变自己的副本，不会影响 Counter 内部状态。

3. snapshot 隔离解决不了什么问题？

它不解决多进程、多实例、服务重启、采集延迟、长期存储等生产监控问题，只解决模块封装边界。

4. 为什么要故意修改 snapshot 来测试？

这是对封装边界的压力测试。若修改返回值能改变内部状态，说明 snapshot 泄漏了可变引用。

⑨ 面试高频真题+解析

面试题 1：如何避免 getter 泄漏内部状态？

不要返回内部可变对象本身。根据数据结构返回只读视图、不可变对象或新的数据投影，并用测试验证外部修改返回值不会影响内部状态。

面试题 2：snapshot 和实时视图区别是什么？

snapshot 是某一时刻的状态投影，生成后与内部状态脱钩；实时视图则可能持续反映内部变化。metrics endpoint 更适合返回 snapshot。

面试题 3：为什么这类小测试有价值？

它保护模块封装边界。未来为了省代码而直接返回内部 Counter 时，测试会立刻失败，避免可变状态泄漏进入 endpoint 和导出层。
