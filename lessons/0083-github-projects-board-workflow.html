0083 GitHub Projects Board：把 issue 标签队列升级为可视化项目工作流

本节新增 .github/PROJECT_BOARD.md，记录 GitHub Projects 看板字段、视图、状态映射和运行规则。Issue Templates 收集事实，Issue Triage 定义标签语义，Projects Board 负责把 issue 和 PR 变成可排期、可跟踪、可复盘的执行队列。

② 核心概念底层原理

GitHub Projects 是 issue、pull request 和字段元数据组成的项目管理层。它不替代 issue，不替代 PR，也不替代 CI。它的核心作用是把分散在仓库中的任务对象投影到一个可视化队列里，让维护者看到“有哪些事、卡在哪、谁可以开工、哪些已完成”。

上一节的 label taxonomy 解决的是任务语义：type 表示做什么，area 表示影响哪里，priority 表示多紧急，status 表示流转状态。Projects 进一步把这些语义复制到看板字段中。字段与标签保持一致，可以让 issue 列表、看板视图和维护文档使用同一套语言。

看板不是任务真相的唯一来源。Issue 正文保存原始问题，labels 保存分类信号，PR 保存修复 diff，CI 保存验证结果，Projects 保存执行视图。若把所有信息只写在看板卡片里，离开 Projects 页面后上下文会丢失，自动关闭和审查链路也会断开。

对象职责不能替代什么
Issue保存问题、需求、讨论和验收背景。不能替代代码修复。
Label提供可筛选的分类和状态信号。不能保存完整执行计划。
Project field提供看板维度、排序、分组和视图。不能作为质量门禁。
Pull Request提交修复、证据、风险说明和 review 对话。不能替代 issue 输入。
CI自动验证代码、文档契约和 whitespace。不能判断业务优先级。

字段设计必须克制。教学仓库只需要 Status、Priority、Area、Type、Lesson 五个字段。字段过多会把维护者拖进表格管理，字段过少又无法支持筛选和排期。高质量项目管理的核心不是字段数量，而是字段能否稳定触发行动。

Projects 的远程属性也必须明确：提交 .github/PROJECT_BOARD.md 不会自动创建 GitHub Projects 看板。该文件是“配置契约文档”，用于告诉维护者在远程 GitHub 上应如何创建字段、视图和自动化规则。生产仓库可进一步使用 GitHub API 或 Terraform 类工具同步项目配置。

③ 全套代码逐行精读解析

# GitHub Projects Board Workflow

标题定义该文档的范围：它描述 Projects 看板工作流，而不是 issue 模板或标签分类法。范围明确能避免把所有 GitHub 协作规则塞进一个文件。

This document defines the recommended GitHub Projects board for the course repository.

recommended 表示这是课程仓库推荐看板模型。由于 GitHub Projects 是远程状态，本地仓库不能仅通过提交文件创建看板，所以文档必须说明它是维护者应复制的配置规范。

## Board Purpose

Board Purpose 先说明看板为什么存在。看板不是为了“看起来正规”，而是把 labeled issues 和 pull requests 转换为 visible maintenance queue。没有目的说明时，看板很容易退化成没人更新的卡片堆。

The project board turns labeled issues and pull requests into a visible maintenance queue.

这句话绑定前两节成果：Issue Forms 收集事实，Issue Triage 贴标签，Projects 展示执行状态。它把三者关系写成单向链路，避免 Projects 绕过 issue 和 PR 流程。

## Required Fields

Required Fields 是看板的 schema。字段一旦被维护者和自动化依赖，就不能随意改名。课程仓库用文档契约测试保护字段名称，防止未来误删关键配置。

| Status | Single select | `Intake`, `Needs Repro`, `Needs Design`, `Ready`, `In Progress`, `In Review`, `Blocked`, `Done` | Current execution stage. |

Status 是项目执行状态，使用 Single select 是为了保证一个卡片只有一个当前阶段。若允许多个状态同时存在，看板分组会失真，维护者无法判断下一步动作。

| Priority | Single select | `P0`, `P1`, `P2`, `P3` | Work ordering signal aligned with `priority:*` labels. |

Priority 与 priority:* 标签对齐。标签用于 issue 列表筛选，字段用于 Projects 排序和分组。两者名字不同但语义一致：priority:p1 映射为 P1。

| Area | Single select | `API`, `Tests`, `Lessons`, `Reference`, `GitHub` | Primary repository surface affected. |

Area 使用 Single select 表示看板上的主影响区域。issue labels 可以有多个 area:*，但 Projects 通常需要一个主 area 便于分组。多影响范围可在 issue labels 和正文中保留。

| Type | Single select | `Bug`, `Lesson`, `Docs`, `Workflow` | Work category aligned with `type:*` labels. |

Type 与 type:* 标签对齐。它决定任务在看板中按缺陷、课程、文档、流程分类，方便维护者按工作类型批量处理。

| Lesson | Text | Lesson number or range | Course location for lesson and documentation tasks. |

Lesson 是课程仓库特有字段。普通业务仓库可能用 milestone 或 component；本仓库更需要记录 Lesson 0083、Lessons 0077-0082 等学习位置，便于课程索引、速查和学习记录同步。

## Views

Views 是同一批任务的不同投影。不要为每个临时问题创建新视图，视图应服务高频维护动作：分流、开工、课程文档、GitHub 流程、完成复核。

| Triage Queue | Status | `Status != Done` | Daily or weekly maintenance intake. |

Triage Queue 是默认维护入口，展示所有未完成任务。它用于定期清理 Intake、Needs Repro、Needs Design、Ready、Blocked 等状态，防止 issue 队列沉积。

| Ready Work | Priority | `Status = Ready` | Contributor-friendly implementation queue. |

Ready Work 是贡献者最该看的视图。只展示已经澄清、可实现的任务，并按 Priority 分组。这样新手不会误接 still needs design 的任务。

| Course Docs | Area | `Area in Lessons, Reference` | Lesson, reference, README, and navigation work. |

Course Docs 视图聚焦课程交付物。课程仓库的核心质量不只在代码，也在 lesson、reference、README、index 和 learning-records 的一致性。

| GitHub Workflow | Status | `Area = GitHub` | CI, branch protection, templates, CODEOWNERS, and issue workflow work. |

GitHub Workflow 视图集中治理仓库协作设施。CI、branch protection、PR template、Issue Forms、CODEOWNERS、Projects 都属于此类，改动风险和普通课程文本不同。

## Status Mapping

Status Mapping 保证 labels 与 Projects 字段不会说两套话。若 status:ready 的 issue 在看板里还是 Needs Design，维护者会收到冲突信号。

| `status:ready` | `Ready` |
| `status:blocked` | `Blocked` |
| `status:done` | `Done` |

映射表让状态同步具备可检查性。教学阶段可以手动同步；生产阶段可以用 GitHub Actions 或 GitHub Apps 监听 label changes 并更新 Projects 字段。

## Operating Rules

Operating Rules 定义看板使用纪律。没有运行规则的项目看板很容易开局整齐、后期失真。规则把“何时加卡、何时移动、何时完成”写成可复用动作。

Every accepted issue should be added to the project board.

该规则保证看板覆盖维护队列。不是所有 issue 都必须接受；重复、无效、无法复现且长期无反馈的问题可以关闭。但一旦接受，就应进入 Projects。

The project item Status field must agree with the current `status:*` label.

这是本节最重要的不变量。标签和项目字段不一致时，issue 列表和 Projects 视图会给出不同状态，维护者不知道应相信哪一个。

Pull requests should be linked to the issue item before review.

PR 必须链接 issue，才能形成从问题输入到修复交付的追踪链。否则代码合并后很难判断修复了哪个问题，也无法可靠关闭 issue。

Move work to `Done` only after tests, documentation, navigation, and progress records are complete.

Done 的定义与课程仓库质量标准绑定。只合并代码不代表完成；课程页、速查页、索引、NOTES、COURSE_PROGRESS 和契约测试都可能是完成条件。

## Automation Boundary

Automation Boundary 防止过早自动化。流程不清时上自动化，只会把错误流程跑得更快。本节要求先手动稳定字段模型，再考虑自动移动卡片。

## Production Notes

生产说明指出本地文档与远程 Projects 的边界，并强调看板不是质量门禁。真正阻断不合格合并的仍是 CI、文档契约测试、CODEOWNERS 和 branch protection。

④ 核心机制运行全流程拆解

Issue created from template
  -> maintainer triages labels
  -> accepted issue is added to GitHub Projects
  -> project fields are filled:
       Type, Area, Priority, Status, Lesson
  -> board view exposes the item:
       Triage Queue / Ready Work / Course Docs / GitHub Workflow
  -> contributor picks a Ready item
  -> pull request links the issue
  -> project status moves to In Review
  -> CI + docs contract + CODEOWNERS review run
  -> merged PR updates docs and progress records
  -> issue label becomes status:done
  -> project status becomes Done

完整链路中，Projects 位于 triage 之后、PR 之前和 review 期间。它不负责收集原始事实，也不负责证明代码正确；它负责让维护队列可见，并让每个 accepted issue 有当前执行位置。

⑧ 课后记忆习题+标准答案

1. GitHub Projects 在本仓库协作链路中的职责是什么？

它把已 triage 的 issue 和 PR 投影为可视化执行队列，帮助维护者查看状态、优先级、影响范围和可开工任务。

2. 为什么 Projects 不能替代 labels？

labels 是 issue 层面的分类信号，可用于筛选和自动化；Projects 是看板视图。二者应同步，而不是互相替代。

3. 本节推荐的五个 Projects 字段是什么？

Status、Priority、Area、Type、Lesson。前四个与 triage 标签语义对齐，Lesson 是课程仓库特有定位字段。

4. 为什么提交 .github/PROJECT_BOARD.md 不会自动创建看板？

GitHub Projects 是远程仓库配置状态，Markdown 文件只是配置契约文档。需要维护者在 GitHub 上手动创建或用 API 同步。

5. 什么时候可以把项目项移动到 Done？

只有当相关测试、文档、导航、学习记录和进度记录全部同步完成，并且 PR 已通过 review 与 CI 后，才能移动到 Done。

⑨ 面试高频真题+解析

面试题 1：GitHub Projects、Issues、Labels、Pull Requests 如何分工？

Issues 保存问题和讨论，Labels 提供分类和状态信号，Projects 提供可视化执行视图，Pull Requests 提交修复和验证证据。四者共同形成从输入到交付的闭环。

面试题 2：为什么项目看板字段要与标签体系对齐？

字段和标签使用同一语义，可以避免 issue 列表与 Projects 视图出现冲突状态，也方便后续用自动化同步。

面试题 3：如何判断一个 issue 是否适合进入 Ready Work 视图？

它必须范围清楚、验收标准明确、依赖可满足，并且具有 type、area、priority、status 信号。缺复现或缺方案的任务不能进入 Ready。

面试题 4：为什么看板不能作为质量门禁？

看板状态是项目管理信号，不能证明代码或文档正确。质量门禁应由 CI、文档契约测试、CODEOWNERS review 和 branch protection 执行。

面试题 5：生产环境如何进一步增强 Projects 工作流？

可增加 owner、team、severity、release、SLA 字段，并用 GitHub Actions、GitHub Apps 或 API 自动同步 labels、PR 状态和项目字段。
