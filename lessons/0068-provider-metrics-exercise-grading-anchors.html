0068 Provider Metrics Exercise Grading Anchors：用确定性锚点给排障作答评分

本节新增 grade_provider_metrics_runbook_exercise_answer()，把学生作答与题卡里的 expected_severity、expected_next_action 做严格锚点匹配。它不是智能阅卷器，而是一个可解释、可测试、适合教学项目的最小评分边界。

② 核心概念底层原理

评分系统的本质是把“自由文本答案”映射成“可比较结果”。在教学项目里，最危险的设计是过早引入模糊语义评分：看似智能，实际不可解释、不可稳定测试，也不利于初学者理解代码边界。

本节采用锚点评分：题卡已经给出标准优先级 expected_severity 和标准动作 expected_next_action，评分函数只检查学生答案是否包含这两个锚点。这个策略牺牲表达自由度，换取确定性、可复现和测试稳定。

casefold() 是比 lower() 更强的大小写归一化方法，适合做不区分大小写的字符串比较。当前答案主要是英文技术短语，使用 casefold() 可以让 ACTION_REQUIRED 与 action_required 被视为同一锚点。

评分层级教学实现生产实现
答案来源题卡结构化字段题库、演练记录、人工 rubric
匹配方式确定性子串匹配规则引擎、人工复核、语义评估组合
失败解释列出缺失锚点给出扣分项、证据、复核流程

复核补强：本节必须把“可解释评分”放在“智能评分”之前。教学项目最怕评分器看似高级却不可复现；锚点评分虽然朴素，但每个扣分项都能直接追溯到题卡字段，适合初学者理解评分系统的最小闭环。

评分设计收益代价
严格包含 expected_severity保证优先级判断被考核同义表达不会自动通过。
严格包含 expected_next_action保证下一步动作可核对回答必须贴近标准动作文本。
返回 missing失败原因透明需要维护字段名作为反馈契约。
不保存原始 response降低隐私和噪声传播后续深度复盘需要另建受控存储。

③ 全套代码逐行精读解析

def grade_provider_metrics_runbook_exercise_answer(
    card: dict[str, object],
    response: str,
) -> dict[str, object]:
    normalized_response = response.casefold()

函数接收一张题卡和一段学生作答，返回结构化评分结果。response.casefold() 先把作答归一化，避免大小写造成误判；若不归一化，学生写 ACTION_REQUIRED 会因为大小写与标准答案不同而失败。

expected_severity = str(card["expected_severity"])
expected_next_action = str(card["expected_next_action"])

评分只依赖两个锚点字段：优先级和下一步动作。使用 card["..."] 而不是 get() 是有意暴露题卡契约错误；如果题卡缺字段，应尽早失败，而不是把 None 当成可评分内容。

severity_matched = expected_severity.casefold() in normalized_response
next_action_matched = expected_next_action.casefold() in normalized_response

这里是核心评分算法：把标准锚点归一化后检查是否出现在学生答案中。它不是语义相似度，只是确定性包含判断；优点是测试可完全锁定，缺点是同义改写不会通过。

missing = []
if not severity_matched:
    missing.append("expected_severity")
if not next_action_matched:
    missing.append("expected_next_action")
passed = not missing

missing 是评分解释层。它把布尔结果转换成可读缺失项，避免只返回 False 让学习者不知道错在哪里。passed = not missing 表示两个锚点全部命中才通过。

return {
    "card_id": card["id"],
    "severity_matched": severity_matched,
    "next_action_matched": next_action_matched,
    "passed": passed,
    "missing": missing,
    "feedback": (
        "Answer includes the expected severity and next action."
        if passed
        else "Missing: {items}.".format(items=", ".join(missing))
    ),
}

返回值不是裸布尔，而是完整评分报告。card_id 让评分结果可追溯到题卡；两个 *_matched 字段支持细粒度断言；feedback 给出人类可读反馈。若只返回 True/False，后续页面、测试和学习记录都无法解释失败原因。

逐行精读补充：str(card["expected_severity"]) 不是为了容忍坏类型，而是把结构化字段统一转换为可比较文本。真正的字段存在性仍由索引读取保证；如果字段缺失，函数会抛出异常，阻止错误题卡继续参与评分。

passed = not missing 把通过条件绑定到缺失列表，而不是重复写成 severity_matched and next_action_matched。这样后续如果增加第三个锚点，只需维护 missing 生成逻辑，通过判断仍保持一致。

⑧ 课后记忆习题+标准答案

1. 为什么本节不用 NLP 语义评分？

教学项目优先保证确定性和可测试性。NLP 评分会引入不可解释和不稳定结果，不适合作为初学者理解评分边界的第一步。

2. casefold() 在本函数中的作用是什么？

它把 response 和标准锚点做大小写归一化，避免同一技术词因为大小写不同而匹配失败。

3. 为什么必须同时检查 severity 和 next_action？

排障答案不仅要知道下一步做什么，还要知道问题优先级。只检查 action 会漏掉处理紧急程度。

4. 为什么返回 missing 而不是只返回 False？

缺失项能告诉学习者到底少了 severity 还是 next_action，是评分可解释性的核心。

5. 生产环境能否直接用这个函数做演练评分？

不能直接照搬。生产演练需要题库版本、rubric、人工复核、审计和隐私治理；本函数只是教学级最小评分边界。

⑨ 面试高频真题+解析

面试题 1：如何设计一个可测试的教学评分函数？

先定义结构化标准答案字段，再选择确定性匹配策略，最后返回可解释评分报告。不要一开始引入语义模型，否则测试稳定性和错误解释都会变差。

面试题 2：为什么评分结果不应该只是布尔值？

布尔值只能说明通过或失败，不能说明失败原因。工程系统需要可追踪的 card_id、细粒度 matched 字段、missing 列表和用户可读 feedback。

面试题 3：教学评分和生产演练评分的最大区别是什么？

教学评分追求低复杂度和可解释；生产演练评分还要处理同义表达、上下文证据、人工复核、题库版本、隐私和合规留档。

面试题 4：为什么缺字段时应该暴露异常而不是默认通过？

题卡缺字段是上游数据契约损坏。静默默认值会把系统错误伪装成评分结果，导致后续排查困难。
