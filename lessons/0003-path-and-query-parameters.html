路径参数与查询参数：URL 资源建模的基础语法

本节把 URL 参数从“会写”提升到“会设计”：路径参数定位单个资源，查询参数描述列表筛选、分页和搜索条件。掌握这部分，才能写出可读、可测试、可演进的 REST 风格 API。

① 本节核心知识框架

URL 参数建模
├─ Path Parameter：路径参数
│  ├─ 写法：/items/{item_id}
│  ├─ 语义：定位唯一资源
│  └─ 校验：Path(ge=1)
├─ Query Parameter：查询参数
│  ├─ 写法：/items?q=api&limit=10
│  ├─ 语义：筛选、分页、排序、搜索
│  └─ 校验：Query(max_length=40), Query(ge=0)
├─ FastAPI 推断规则
│  ├─ 出现在路径模板中 -> path
│  ├─ 简单类型且不在路径中 -> query
│  └─ Pydantic 模型 -> body
└─ 当前项目落点：routers/items.py 的 read_item 与 list_items

② 核心概念底层原理

URL 是资源定位语言。/items/2 表示“ID 为 2 的商品资源”；/items?category=book&limit=10 表示“商品集合中满足条件的一页”。路径参数改变资源身份，查询参数改变集合视图。

路径匹配先于参数校验。FastAPI 先用路由表匹配 /items/{item_id}，再把 URL 里的字符串片段转换成 int，最后执行 Path(ge=1) 约束。转换或约束失败返回 422，函数不执行。

查询参数天然是字符串。HTTP URL 中 max_price=50 原始值是字符串 "50"，FastAPI 根据类型注解转换成 float；in_stock=true 会按布尔规则转换。没有类型注解，后端就只能得到弱类型字符串。

和请求体的边界。路径/查询适合轻量、可出现在 URL 中的条件；复杂对象、长文本、嵌套结构应放 body。把长文本塞进 query 会导致 URL 可读性差、长度限制和日志泄露风险。

③ 全套代码逐行精读解析

@router.get("/{item_id}", response_model=ItemRead)
def read_item(
    item_id: Annotated[int, Path(ge=1, description="The numeric ID of the item.")],
    session: Annotated[Session, Depends(get_session)],
) -> Item:
    item = session.get(Item, item_id)
    if item is None:
        raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail=f"Item {item_id} was not found.")
    return item
  • @router.get("/{item_id}")：在 /items router 前缀下形成 GET /items/{item_id}。若路径变量名与函数参数名不一致，FastAPI 无法正确注入。
  • response_model=ItemRead：对外只返回读模型。若直接暴露数据库模型，未来内部字段可能泄漏。
  • Annotated[int, Path(...)]：把类型 int 和参数元数据组合起来。ge=1 保证 ID 从 1 开始。
  • session: Annotated[Session, Depends(get_session)]：数据库会话来自依赖注入，不来自 URL。
  • session.get(Item, item_id)：按主键查询。若 ID 合法但数据库无记录，返回 None。
  • if item is None：业务不存在分支。不能让 None 继续返回，否则与 ItemRead 响应模型不匹配。
  • raise HTTPException(...404...)：合法 ID 找不到资源，语义是 404，而不是 422。
  • return item：返回 ORM/SQLModel 对象，FastAPI 按 ItemRead 序列化。
@router.get("", response_model=list[ItemRead])
def list_items(
    filters: Annotated[ItemFilters, Depends(ItemFilters)],
    pagination: Annotated[Pagination, Depends(Pagination)],
    session: Annotated[Session, Depends(get_session)],
) -> list[Item]:
    statement = select(Item)
    items = list(session.exec(statement))

    if filters.q is not None:
        keyword = filters.q.lower()
        items = [item for item in items if keyword in item.name.lower()]
    if filters.category is not None:
        items = [item for item in items if item.category == filters.category]
    if filters.max_price is not None:
        items = [item for item in items if item.price <= filters.max_price]
    if filters.in_stock is not None:
        items = [item for item in items if item.in_stock == filters.in_stock]

    return items[pagination.skip : pagination.skip + pagination.limit]
  • @router.get("")：空字符串与 router 前缀 /items 合成列表接口。
  • filters：把多个 query 参数封装成对象，避免函数签名膨胀。
  • pagination：分页参数同样来自 query。
  • select(Item)：构造 SQLModel 查询语句。当前为了教学，先查询全部再在 Python 中过滤。
  • if filters.q is not None：只在调用者传入搜索词时启用条件；空缺条件不应改变结果。
  • keyword = filters.q.lower()：大小写归一。若不做归一，API 和 api 匹配结果会不一致。
  • items[pagination.skip : pagination.skip + pagination.limit]：用切片实现 offset/limit。生产中应下推到 SQL，而不是先取全量。

④ 核心机制运行全流程拆解

GET /items/2
  -> 匹配 /items/{item_id}
  -> "2" 转 int 2
  -> Path(ge=1) 通过
  -> get_session() 产出数据库 Session
  -> session.get(Item, 2)
  -> 找到：200 + ItemRead JSON
  -> 找不到：404 + detail

GET /items?category=book&skip=0&limit=10
  -> 匹配 /items
  -> ItemFilters 从 query 解析 category
  -> Pagination 从 query 解析 skip/limit
  -> 数据库查询 Item
  -> Python 层过滤 category
  -> 切片分页
  -> 200 + list[ItemRead]

⑤ 重难点、易混淆点对比辨析

  • /items/abc vs /items/999：前者类型转换失败是 422；后者参数合法但资源不存在是 404。
  • 路径参数 vs 查询参数：item_id 是资源身份，不应设计成 /items?id=2；筛选条件不是资源身份，不应设计成 /items/category/book 起步。
  • skip/limit vs 页码：offset 模式直接表示跳过多少条；page/page_size 需要转换。两者都常见，但响应契约要一致。
  • 教学过滤 vs SQL 过滤：当前先取全量再 Python 过滤便于教学；生产应把 where/limit/offset 下推到数据库。

⑥ 开发常见报错+坑点+解决方案

  • 422 int_parsing：路径 ID 传了非数字。解决：传 /items/1 这种整数片段。
  • 422 greater_than_equal：/items/0 不满足 Path(ge=1)。
  • 422 literal_error：category 不是 book/course/tool。解决：遵守 ItemCategory 枚举。
  • 布尔 query 传错：in_stock=maybe 无法转换。解决：使用 true/false。
  • 分页参数过大：limit=1000 会被依赖中的 Query(le=50) 拦截。

⑦ 生产环境 vs 教学环境 核心差异

  • 过滤位置：教学可以 Python 过滤；生产必须 SQL 层过滤并配索引，否则数据量变大后性能崩溃。
  • 分页策略：教学用 offset；生产大表可能使用 cursor/keyset pagination，避免深分页性能问题。
  • 参数上限：教学限制 limit <= 50；生产还会按用户权限、接口成本、缓存策略设置更严格边界。
  • 错误格式：教学保留 FastAPI 默认 422；生产常统一错误码、国际化消息、trace id。

⑧ 课后记忆习题+标准答案

1. /items/{item_id} 中 {item_id} 为什么必须和函数参数同名？

FastAPI 根据路径模板变量名把 URL 片段注入到同名函数参数。

2. 查询参数不传时为什么通常使用 None？

None 表示筛选条件缺省，业务代码可以据此跳过该过滤分支。

3. GET /items/0 为什么不是 404？

因为 0 在进入业务查询前已经违反 Path(ge=1) 参数约束，所以是 422。

⑨ 面试高频真题+解析

1. REST API 中路径参数和查询参数如何取舍？

路径参数表达资源身份和层级，查询参数表达集合筛选、分页、排序和可选视图控制。

2. 为什么生产中不建议先查全表再 Python 过滤？

全表读取会浪费内存和网络 I/O，无法利用数据库索引，数据量增大后延迟和资源占用会急剧恶化。

3. FastAPI 如何知道一个参数来自 path 还是 query？

参数名出现在路径模板中就是 path；简单类型参数未出现在路径模板且不是依赖/body 时通常推断为 query。
