PATCH 更新与 DELETE 删除：补齐 CRUD 的状态变更语义

本节关注 CRUD 中最容易写错的两类接口：部分更新与删除。核心不是记住 PATCH 和 DELETE，而是理解可选字段、未传字段、空值、事务提交、204 响应和 404 边界。

① 本节核心知识框架

Update/Delete CRUD
├─ PATCH：部分更新
│  ├─ ItemUpdate：所有字段可选
│  ├─ model_dump(exclude_unset=True)：只取客户端传入字段
│  ├─ sqlmodel_update(update_data)：写回 ORM 对象
│  └─ add + commit + refresh：提交并返回最新对象
├─ DELETE：删除资源
│  ├─ 先 get：确认资源存在
│  ├─ session.delete(item)：标记删除
│  ├─ commit：提交删除
│  └─ 204 No Content：成功且无响应体
└─ 共同边界：合法 ID 不存在 -> 404；参数非法 -> 422

② 核心概念底层原理

CRUD 是资源状态机。Create 产生资源，Read 观察资源，Update 改变资源，Delete 移除资源。HTTP 方法不是装饰语，而是客户端、缓存、网关、文档和测试理解接口语义的协议。

PATCH 与 PUT 的关键区别是“局部”与“整体”。PUT 通常表示用客户端提交的完整表示替换资源；PATCH 表示只提交需要修改的字段。本项目用 PATCH，因此更新模型必须允许字段缺省。

未传字段与显式传 null 必须区分。部分更新里，未传字段应保持原值；显式传 null 是否允许取决于业务模型。exclude_unset=True 的价值就在于只处理“客户端真的传了”的字段。

删除不是把对象从 Python 列表移除。ORM 删除需要把对象标记为 deleted，再提交事务。提交前数据库状态尚未最终改变。

③ 全套代码逐行精读解析

class ItemUpdate(SQLModel):
    name: str | None = Field(default=None, min_length=2, max_length=80)
    category: ItemCategory | None = None
    price: float | None = Field(default=None, ge=0)
    in_stock: bool | None = None
  • ItemUpdate(SQLModel)：更新请求模型，不是数据库表模型。
  • str | None = None：字段可缺省，支持部分更新。
  • min_length/max_length：如果客户端传了 name，仍必须满足名称规则。
  • ItemCategory | None：分类可不改；一旦改，只能是合法分类。
  • price ... ge=0：价格可不改；一旦改，不能为负。
@router.patch(
    "/{item_id}",
    response_model=ItemRead,
    dependencies=[Depends(require_api_key)],
)
def update_item(
    item_id: Annotated[int, Path(ge=1, description="The numeric ID of the item.")],
    payload: ItemUpdate,
    session: Annotated[Session, Depends(get_session)],
) -> Item:
    item = session.get(Item, item_id)
    if item is None:
        raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail=f"Item {item_id} was not found.")

    update_data = payload.model_dump(exclude_unset=True)
    item.sqlmodel_update(update_data)
    session.add(item)
    session.commit()
    session.refresh(item)
    return item
  • @router.patch("/{item_id}")：更新指定资源，资源身份来自路径。
  • dependencies=[Depends(require_api_key)]：状态变更接口必须鉴权。
  • Path(ge=1)：非法 ID 在业务前返回 422。
  • payload: ItemUpdate：请求体只包含要更新字段。
  • session.get：先确认资源存在；不存在返回 404。
  • model_dump(exclude_unset=True)：避免未传字段被默认 None 覆盖数据库原值。
  • sqlmodel_update(update_data)：把变更应用到 ORM 对象。
  • session.add(item)：确保对象处于 Session 跟踪中。
  • commit/refresh：提交事务并返回最新状态。
@router.delete(
    "/{item_id}",
    status_code=status.HTTP_204_NO_CONTENT,
    dependencies=[Depends(require_api_key)],
)
def delete_item(
    item_id: Annotated[int, Path(ge=1, description="The numeric ID of the item.")],
    session: Annotated[Session, Depends(get_session)],
) -> None:
    item = session.get(Item, item_id)
    if item is None:
        raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail=f"Item {item_id} was not found.")

    session.delete(item)
    session.commit()
  • status_code=204：删除成功但没有响应体。若返回 JSON，应使用 200 或 202 等其他语义。
  • session.delete(item)：标记删除，等待提交。
  • commit()：真正提交删除。忘记 commit 会导致删除没有落库。
  • -> None：函数不返回 body，与 204 保持一致。

④ 核心机制运行全流程拆解

PATCH /items/3 {"price": 18.0}
  -> API Key 校验
  -> item_id 校验
  -> ItemUpdate 校验 price >= 0
  -> session.get(Item, 3)
  -> payload.model_dump(exclude_unset=True) = {"price": 18.0}
  -> item.sqlmodel_update(...)
  -> commit + refresh
  -> 200 + ItemRead

DELETE /items/3
  -> API Key 校验
  -> item_id 校验
  -> session.get(Item, 3)
  -> session.delete(item)
  -> commit
  -> 204 No Content
  -> 再 GET /items/3 返回 404

⑤ 重难点、易混淆点对比辨析

  • 未传字段 vs null：未传表示不修改；传 null 是否允许由字段类型决定。当前字段允许 None 进入模型，但实际更新语义要谨慎。
  • PATCH vs PUT：PATCH 局部修改；PUT 整体替换。面试中不要混用。
  • 204 vs 200：204 不带 body；若要返回删除后的对象或状态消息，使用 200 更合理。
  • 404 vs 幂等删除：当前删除不存在资源返回 404；有些 API 会把重复删除设计成 204。关键是文档和测试要一致。

⑥ 开发常见报错+坑点+解决方案

  • PATCH 后字段被清空：忘记 exclude_unset=True。解决：只 dump 已设置字段。
  • DELETE 返回 body 导致客户端困惑：204 不应返回内容。解决：函数返回 None。
  • 更新不存在 ID 返回 500：未处理 None。解决：先查再 404。
  • 写接口返回 401：缺少 X-API-Key。解决：带正确请求头。
  • PATCH 负价格返回 422：这是正确行为，说明模型约束生效。

⑦ 生产环境 vs 教学环境 核心差异

  • 并发更新：教学直接覆盖；生产要考虑乐观锁、版本号、更新时间戳和冲突检测。
  • 删除策略：教学物理删除；生产常用软删除以保留审计、恢复能力和外键历史。
  • 审计日志：教学不记录操作者；生产应记录谁在何时改了什么。
  • 权限控制：教学只校验 API Key；生产要校验资源级权限，例如用户只能改自己的数据。

⑧ 课后记忆习题+标准答案

1. exclude_unset=True 为什么是 PATCH 的关键？

它只提取客户端实际传入的字段，避免未传字段被默认 None 覆盖。

2. 删除成功为什么使用 204？

204 表示请求成功且没有响应体，符合删除后不返回内容的设计。

3. 更新前为什么必须先查询资源？

需要确认资源存在，并拿到 ORM 对象才能应用变更；不存在应返回 404。

⑨ 面试高频真题+解析

1. PATCH 和 PUT 的区别是什么？

PUT 通常表示完整替换资源，PATCH 表示局部修改资源字段。

2. 软删除和硬删除如何取舍？

硬删除简单直接；软删除保留历史、支持恢复和审计，但查询必须过滤删除状态。

3. 如何避免并发更新互相覆盖？

可使用版本号、更新时间条件、乐观锁、事务隔离或业务冲突检测。
