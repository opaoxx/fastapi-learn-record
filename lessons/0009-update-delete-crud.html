Lesson 0009

更新和删除：补齐 CRUD 闭环

前面已经会创建和读取数据库记录。本节补上更新和删除，形成完整 CRUD：Create、Read、Update、Delete。

0. 本节学习目标

  • 理解 CRUD 四个动作和 HTTP 方法的对应关系。
  • 理解为什么部分更新通常用 PATCH。
  • 看懂 ItemUpdate 为什么所有字段都是可选的。
  • 掌握 exclude_unset、sqlmodel_update、delete 的作用。

1. CRUD 是什么

CRUD 是后端最基础的数据操作模型：

  • Create：创建资源，例如 POST /items。
  • Read：读取资源，例如 GET /items、GET /items/{item_id}。
  • Update：更新资源，例如 PATCH /items/{item_id}。
  • Delete：删除资源，例如 DELETE /items/{item_id}。

求职项目里，只要你能把一个资源完整做出 CRUD，就已经具备了普通后端接口的主干能力。

2. PUT 和 PATCH 的区别

更新接口常见两种方法：

  • PUT：更偏向“整体替换”，客户端通常要提交完整资源。
  • PATCH：更偏向“部分更新”，客户端只提交要改的字段。

本项目使用 PATCH，因为新手最容易理解：只想改价格，就只传价格；只想改库存，就只传库存。

3. ItemUpdate 模型

class ItemUpdate(SQLModel):
    name: str | None = Field(default=None, min_length=2, max_length=80)
    category: ItemCategory | None = None
    price: float | None = Field(default=None, ge=0)
    in_stock: bool | None = None

这里所有字段都是可选的，因为 PATCH 支持部分更新。客户端可以只传：

{
  "price": 18.0
}

但可选不等于随便传。比如 price 如果传了，就必须大于等于 0；category 如果传了，就必须是 book、course 或 tool。

4. 更新接口逐行讲解

@router.patch("/{item_id}", response_model=ItemRead)
def update_item(
    item_id: Annotated[int, Path(ge=1)],
    payload: ItemUpdate,
    session: Annotated[Session, Depends(get_session)],
) -> Item:
    item = session.get(Item, item_id)
    if item is None:
        raise HTTPException(status_code=404, detail=f"Item {item_id} was not found.")

    update_data = payload.model_dump(exclude_unset=True)
    item.sqlmodel_update(update_data)
    session.add(item)
    session.commit()
    session.refresh(item)
    return item

关键点：

  • session.get(Item, item_id)：先找到要更新的记录。
  • exclude_unset=True：只取客户端真的传了的字段。
  • sqlmodel_update：把更新字段写回数据库对象。
  • commit：保存改动。
  • refresh：刷新对象，返回最新数据。

5. 删除接口逐行讲解

@router.delete("/{item_id}", status_code=status.HTTP_204_NO_CONTENT)
def delete_item(
    item_id: Annotated[int, Path(ge=1)],
    session: Annotated[Session, Depends(get_session)],
) -> None:
    item = session.get(Item, item_id)
    if item is None:
        raise HTTPException(status_code=404, detail=f"Item {item_id} was not found.")

    session.delete(item)
    session.commit()

204 No Content 表示删除成功，但响应体为空。这是删除接口很常见的设计：客户端只需要知道删除成功，不一定需要服务器再返回被删除的对象。

6. 实验：更新一条记录

先创建一个商品，再更新它：

POST /items
{
  "name": "Patch Practice Item",
  "category": "tool",
  "price": 10,
  "in_stock": true
}

PATCH /items/{new_id}
{
  "price": 12.5,
  "in_stock": false
}

你应该看到名称和分类保持不变，价格和库存发生变化。

7. 实验：删除一条记录

DELETE /items/{new_id}
GET /items/{new_id}

删除请求应该返回 204。再次读取同一个 ID 应该返回 404，因为这个资源已经不存在。

8. 课堂练习

请你完成：

  1. 创建一个 category="course" 的商品。
  2. 只用 PATCH 修改它的 name。
  3. 只用 PATCH 修改它的 price。
  4. 删除它，并解释为什么后续读取是 404。

9. 小测：先回忆再展开

1. 为什么 ItemUpdate 的字段都是可选的？

因为 PATCH 是部分更新，客户端只需要提交想改的字段。

2. exclude_unset=True 避免了什么问题？

它避免未传字段被当作 None 写回数据库，导致原数据被误清空。

3. 删除成功为什么返回 204？

因为 204 No Content 表示操作成功，但服务器没有响应体要返回。

10. 本节总结

现在 items 已经具备完整 CRUD。PATCH 用于部分更新，DELETE 用于删除资源。更新时先查、再改、再提交；删除时先查、再删、再提交。下一节会用自动化测试证明这些接口真的按预期工作。
