Final Review + CSDN Deep Dive

Schema 演进与向后兼容：JSON 形状变化就是 API 变更

当 GET /tasks 从裸数组升级成响应信封时，后端只是多包一层对象，调用方却必须重写解析路径。Schema 演进的核心判断标准不是“代码改得多不多”，而是“旧客户端是否还能不用修改继续工作”。

① 本节核心知识框架

Schema 演进判断模型
├─ Schema：JSON 形状、字段、类型、必填性、约束、语义
├─ 兼容变更
│  ├─ 新增可选响应字段
│  ├─ 新增可选查询参数且默认行为不变
│  └─ 扩展文档描述
├─ 破坏性变更
│  ├─ 顶层类型变化：array -> object
│  ├─ 删除字段 / 字段改名
│  ├─ 字段类型改变
│  ├─ 新增必填请求字段
│  └─ 默认语义改变
├─ 项目案例：list[SummaryTaskRead] -> SummaryTaskListResponse
└─ 防线：运行时契约测试 + OpenAPI schema 测试

② 核心概念底层原理

Schema 不只是“有哪些字段”，还包括顶层类型、字段类型、是否必填、取值范围和业务语义。count 是整数并且 ge=0，这属于结构约束；count 表示当前筛选条件下总数，这属于语义约束。

向后兼容是调用方视角的概念：服务端升级后，旧客户端不改代码仍能工作，才叫兼容。后端觉得“只是字段改名”没有意义，因为旧客户端读取旧字段会失败。

版本化、弃用、迁移窗口是处理不兼容变更的工程手段。课程项目调用方少，可以直接升级；真实团队通常要新增版本或新 endpoint，保留旧契约一段时间，并用文档和测试引导迁移。

③ 全套代码逐行精读解析

class SummaryTaskListResponse(BaseModel):
    items: list[SummaryTaskRead] = PydanticField(description="The current page of summary tasks.")
    count: int = PydanticField(ge=0, description="Total number of tasks matching the current filter.")
    limit: int = PydanticField(ge=1, le=100, description="Maximum number of tasks requested for this page.")
    offset: int = PydanticField(ge=0, description="Zero-based number of matching tasks skipped before this page.")
  • items 新增了页级容器语义；旧裸数组客户端必须迁移到 payload.items。
  • count 是新增字段，但由于顶层已经从数组变对象，整体仍是不兼容变更。
  • ge/le 把约束写入 schema，让 OpenAPI 可见，也能被测试保护。
  • 字段描述补充语义，防止后续维护者把 count 错写成当前页条数。
def test_list_summary_tasks_returns_response_envelope_contract() -> None:
    response = client.get("/tasks?limit=5&offset=0", headers=API_KEY_HEADERS)
    payload = response.json()
    assert isinstance(payload, dict)
    assert set(payload) == {"items", "count", "limit", "offset"}
    assert isinstance(payload["items"], list)
  • isinstance(payload, dict) 保护顶层类型；若有人改回裸数组，该断言立刻失败。
  • set(payload) 保护字段集合；字段改名、删除、增加未评审字段都会暴露。
  • items 类型断言保护前端遍历路径。
envelope_schema = schema["components"]["schemas"]["SummaryTaskListResponse"]
assert envelope_schema["required"] == ["items", "count", "limit", "offset"]
assert envelope_schema["properties"]["limit"]["maximum"] == 100

运行时测试保护真实响应，OpenAPI 测试保护文档契约。两者必须同时存在，否则可能出现“接口返回对了，文档错了”或“文档写对了，运行时漂移”。

④ 核心机制运行全流程拆解

需求：前端需要分页按钮
  -> 裸数组无法表达总数和偏移
  -> 后端设计 SummaryTaskListResponse
  -> 修改 response_model 和 service 返回
  -> 前端从 response 改为 response.items
  -> OpenAPI schema 从 array 变 object
  -> 契约测试锁定 items/count/limit/offset
  -> 后续变更先判断兼容性
       兼容：新增可选字段/参数 + 测试
       不兼容：版本化/弃用/迁移期

⑤ 重难点、易混淆点对比辨析

变更通常结论原因
新增可选查询参数，默认行为不变兼容旧客户端不传也能继续工作
新增必填请求字段不兼容旧客户端请求会 422
字段改名不兼容等价于删除旧字段
新增可选响应字段通常兼容宽松客户端会忽略未知字段
数组改对象不兼容顶层解析逻辑变化
枚举新增值谨慎旧客户端可能没有 default 分支

⑥ 开发常见报错+坑点+解决方案

  • “只是改字段名”导致前端挂掉：字段名是公共契约。解决：新增字段并保留旧字段，或版本化。
  • OpenAPI 与真实响应不一致：response_model 或实际返回漂移。解决：运行时和 OpenAPI 双测试。
  • 新增状态后前端白屏：枚举值扩展未处理 default。解决：前端对未知状态有 fallback。
  • 新增必填参数导致旧客户端 422：不兼容变更未评审。解决：先做可选参数，默认行为保持不变。
  • 版本号改了但迁移失败：版本化不是迁移策略。解决：写差异说明、保留旧接口、设置弃用窗口。

⑧ 课后记忆习题+标准答案

1. 什么是 schema？

schema 是数据形状说明，包括顶层类型、字段、字段类型、必填性、约束范围和字段语义。

2. 为什么裸数组到 envelope 是破坏性变化？

顶层 JSON 类型从数组变对象，旧客户端遍历数组的代码会失败。

3. 新增可选字段一定安全吗？

通常兼容，但严格客户端可能拒绝未知字段；仍需评估调用方解析策略。

4. 运行时契约测试和 OpenAPI 测试分别保护什么？

运行时测试保护真实 wire format；OpenAPI 测试保护文档、工具和 SDK 看到的 schema。

⑨ 面试高频真题+解析

题 1：如何判断一个 API 变更是否向后兼容？

站在旧客户端视角判断：不改代码还能否成功请求、解析响应并保持语义一致。删除字段、改名、改类型、新增必填字段、改变默认行为通常不兼容。

题 2：必须做不兼容变更怎么办？

新增版本或新路径，旧接口标记 deprecated 并保留迁移窗口；提供迁移文档、兼容测试和发布计划，必要时双写/双读过渡。

题 3：FastAPI 项目如何防止 schema 漂移？

用 Pydantic 模型表达请求响应，用 OpenAPI 暴露契约，用测试断言运行时 JSON 和 OpenAPI schema 的关键字段，并在 CI 中阻断未评审变更。
