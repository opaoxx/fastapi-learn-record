Lesson 0030

Schema 演进和向后兼容：JSON 形状变化不是小事

第 29 课我们把 /openapi.json 看成 API 契约。这一课继续深入：当业务要继续长大，接口返回的 JSON 能不能随便改？什么时候可以加字段，什么时候必须升级版本，什么时候要写迁移计划？

0. 本节学习目标

  • 理解 schema、schema 演进、向后兼容、破坏性变更、弃用和版本号。
  • 能判断常见 API 变更是兼容还是不兼容。
  • 理解为什么 GET /tasks 从裸数组升级成响应信封是一种契约变化。
  • 学会用 response_model、Field 和测试守住响应形状。
  • 形成真实项目里的变更流程：先分类，再迁移，再删除。

1. 背景：为什么“改个字段名”会出事故

后端初学阶段很容易有一个错觉：接口是我写的，所以我可以随时改返回值。比如把任务列表从：

[
  {"id": 1, "status": "completed"}
]

改成：

{
  "items": [
    {"id": 1, "status": "completed"}
  ],
  "count": 12,
  "limit": 5,
  "offset": 0
}

从后端角度看，这只是“为了分页多包一层”。但从前端角度看，旧代码可能写的是：

for (const task of tasks) {
  renderTask(task);
}

如果 tasks 突然从数组变成对象，循环就会失败。也就是说，JSON 形状不是内部实现细节，而是调用方代码依赖的公共接口。

2. 核心概念：schema 是数据的形状说明

在本课程里，schema 可以先理解为“JSON 应该长什么样”。它至少描述：

  • 顶层类型：对象、数组、字符串、数字、布尔值。
  • 字段集合：有哪些字段，字段名是什么。
  • 字段类型：items 是数组，count 是整数。
  • 是否必填：调用方能不能假设字段一定存在。
  • 取值范围：limit 最小 1，最大 100。
  • 语义：count 是当前筛选条件下的总数，而不是当前页条数。

OpenAPI Specification 的价值就在这里：它提供一种机器可读的 API 描述格式，让路径、参数、请求体、响应体、安全方案等信息可以被文档、测试和客户端工具共同理解。

3. 六个术语先讲清楚

术语 含义 本项目例子
Schema 数据形状、字段、类型和约束的说明。 SummaryTaskListResponse
Schema 演进 接口形状随着业务变化而改变。 从任务裸数组到分页响应信封。
向后兼容 旧客户端不用改代码，也能继续和新服务通信。 新增一个可选查询参数，默认行为不变。
破坏性变更 旧客户端可能无法继续工作。 把 count 改名为 total。
弃用 暂时保留旧能力，但告诉调用方未来会移除。 旧参数保留一段时间并标记 deprecated。
版本化 用 v1、v2 或版本号表达不兼容边界。 未来可能出现 /api/v2/tasks。

Google AIP 的向后兼容准则把 API 视为和用户之间的契约，并区分源码兼容、传输兼容和语义兼容。对我们现在的 FastAPI 项目，最重要的是传输兼容和语义兼容：JSON 能不能被旧客户端解析，以及字段含义有没有悄悄变掉。

4. 兼容性判断表

下面这张表是你以后做接口变更前可以先过一遍的“变更体检表”。

变更 通常是否兼容 原因
新增可选响应字段 通常兼容 宽松客户端会忽略不认识的字段；严格客户端仍需谨慎。
新增可选查询参数，默认行为不变 通常兼容 旧客户端不传新参数时仍得到旧行为。
新增必填请求字段 通常不兼容 旧客户端不会发送这个字段，请求会失败。
删除响应字段 不兼容 旧客户端可能正在读取该字段。
字段改名 不兼容 等价于删除旧字段再新增新字段。
字段类型改变 不兼容 解析逻辑会变，例如整数变字符串。
默认值改变 通常不兼容 旧客户端不传参数时观察到的行为变了。
枚举响应新增值 谨慎处理 旧客户端可能没有处理未知状态。

SemVer 的基本思路也可以借来理解 API：补丁版本用于向后兼容的修复，次版本用于向后兼容的新功能，主版本用于不兼容变化。我们的课程项目还没有真正发布包版本，但这个思维很重要。

5. 回看本项目：裸数组到信封为什么是破坏性变化

早期任务列表如果返回裸数组，调用方会把响应当作数组：

[
  {"id": 3, "status": "completed"},
  {"id": 2, "status": "failed"}
]

第 25、26 课之后，我们为了支持前端分页，把响应升级成：

{
  "items": [
    {"id": 3, "status": "completed"},
    {"id": 2, "status": "failed"}
  ],
  "count": 12,
  "limit": 5,
  "offset": 0
}

这个设计本身是对的，因为前端需要 count 才能判断下一页。但它不是“无痛小改”。它改变了顶层类型：从数组变成对象。任何旧客户端如果还把响应当数组处理，都会坏。

真实项目里遇到这种变化，常见做法不是直接改原接口，而是：

  • 保留旧接口一段时间，例如 GET /tasks 仍返回旧数组。
  • 新增新接口，例如 GET /tasks/search 或 GET /api/v2/tasks 返回信封。
  • 或者先让旧接口支持兼容输出，通知调用方迁移，再移除旧格式。

课程项目因为调用方只有我们自己的前端和测试，所以可以直接升级。但学习时必须知道：在真实团队里，这会被当成契约变更来评审。

6. 本节项目改动概览

这一课做了一个保守但很实用的改动：不改变 JSON 返回值，只强化契约表达和测试。

  • 给 SummaryTaskListResponse 的四个字段补充 Field 描述和数值边界。
  • 新增测试，确认 GET /tasks 返回的是对象信封，而不是裸数组。
  • 新增 OpenAPI 测试，确认响应信封 schema 仍然要求 items/count/limit/offset。

这类改动很适合教学项目和真实项目：对调用方没有新增负担，但让契约更清楚、更容易被工具检查。

7. 代码精读一：用 Field 把字段语义写进 schema

class SummaryTaskListResponse(BaseModel):
    items: list[SummaryTaskRead] = PydanticField(
        description="The current page of summary tasks."
    )
    count: int = PydanticField(
        ge=0,
        description="Total number of tasks matching the current filter."
    )
    limit: int = PydanticField(
        ge=1,
        le=100,
        description="Maximum number of tasks requested for this page."
    )
    offset: int = PydanticField(
        ge=0,
        description="Zero-based number of matching tasks skipped before this page."
    )

这段代码没有改变响应 JSON 的字段名，也没有新增字段。它改变的是 schema 的清晰度：

  • items：说明这是当前页，不是全部任务。
  • count：说明这是当前筛选条件下的总数，不是当前页数量。
  • limit：把最大 100 的边界写进响应模型。
  • offset：说明它是从 0 开始跳过的匹配记录数。

FastAPI 的 response_model 会把这些模型信息用于文档、验证、序列化和过滤。也就是说，模型不是“只给 Python 看”，它会影响 OpenAPI 契约。

8. 代码精读二：运行时契约测试

def test_list_summary_tasks_returns_response_envelope_contract() -> None:
    with TestClient(app) as client:
        response = client.get("/tasks?limit=5&offset=0", headers=API_KEY_HEADERS)

    assert response.status_code == 200
    payload = response.json()
    assert isinstance(payload, dict)
    assert set(payload) == {"items", "count", "limit", "offset"}
    assert isinstance(payload["items"], list)
    assert isinstance(payload["count"], int)
    assert isinstance(payload["limit"], int)
    assert isinstance(payload["offset"], int)

这个测试的目标很明确：守住 wire format，也就是网络上传输的 JSON 形状。

如果以后有人把接口改回：

return items

这个测试会在 assert isinstance(payload, dict) 失败。它不是在挑剔实现细节，而是在保护前端依赖的响应契约。

9. 代码精读三：OpenAPI schema 测试

def test_openapi_documents_task_list_response_envelope_schema() -> None:
    with TestClient(app) as client:
        response = client.get("/openapi.json")

    schema = response.json()
    envelope_schema = schema["components"]["schemas"]["SummaryTaskListResponse"]

    assert envelope_schema["required"] == ["items", "count", "limit", "offset"]
    assert set(envelope_schema["properties"]) == {"items", "count", "limit", "offset"}
    assert envelope_schema["properties"]["count"]["minimum"] == 0
    assert envelope_schema["properties"]["limit"]["minimum"] == 1
    assert envelope_schema["properties"]["limit"]["maximum"] == 100
    assert envelope_schema["properties"]["offset"]["minimum"] == 0

这和上一个测试关注点不同：

  • 运行时测试：真实接口现在返回什么。
  • OpenAPI 测试：文档和工具看到的契约是什么。

两个都要有。否则可能出现一种尴尬状态：接口实际返回信封，但 OpenAPI 文档还像裸数组；或者文档写得很好，代码实际已经漂移。

10. 变更流程：真实项目怎么改 API

  1. 先写出当前契约：路径、参数、响应字段、字段语义、错误状态码。
  2. 判断变更类型：兼容、可能兼容、不兼容。
  3. 如果兼容，直接加测试和文档，然后发布。
  4. 如果不兼容，优先寻找兼容方案，例如新增字段而不是改名。
  5. 如果必须不兼容，新增版本或新路径，保留旧接口一段时间。
  6. 在文档里标记弃用，告诉调用方迁移目标和时间。
  7. 用测试守住旧版本和新版本各自的契约。

FastAPI 的 Query(deprecated=True) 可以把旧查询参数标记为弃用，并显示在 OpenAPI 文档里。它不等于删除参数，而是给调用方一个迁移信号。

11. 课堂实验一：观察响应信封

运行服务后请求：

GET http://127.0.0.1:8000/tasks?limit=5&offset=0
X-API-Key: dev-secret-key

你应该观察到顶层是对象，不是数组：

{
  "items": [...],
  "count": 0,
  "limit": 5,
  "offset": 0
}

请特别看 count 和 items.length 的区别：当前页可能只有 5 条，但符合筛选条件的总数可能是 20 条。

12. 课堂实验二：故意制造一次破坏性变更

只做实验，不要提交这个改动。把 SummaryTaskListResponse 里的 count 临时改名为 total，然后运行：

.\.venv\Scripts\python.exe -m pytest tests\test_tasks_api.py tests\test_openapi_contract.py -q

你会看到契约测试失败。这就是我们想要的效果：测试不是阻止你演进 API，而是提醒你“这次演进需要迁移策略”。

13. 常见错误与诊断

错误一：把字段改名当作小重构

字段名属于公共契约。改名等于删除旧字段再新增新字段。

错误二：新增必填参数

旧客户端不会发送新参数，所以请求可能从 200 变成 422。

错误三：只测业务成功，不测响应形状

业务仍然成功，但 JSON 形状变了，前端一样会坏。

错误四：只版本化，不迁移

v2 不是魔法。你仍然要告诉调用方差异、迁移路径和保留时间。

14. 面试视角：如何回答“怎么做 API 向后兼容？”

可以这样答：

我会先把 API 当成调用方依赖的契约，而不是后端内部实现。兼容变更通常包括新增可选字段或可选参数，并保持默认行为不变；破坏性变更包括删除字段、字段改名、字段类型变化、新增必填请求字段和改变默认语义。对 FastAPI 项目，我会用 Pydantic 响应模型表达 schema，用 OpenAPI 暴露契约，并用测试检查运行时响应和 OpenAPI schema 是否保持一致。如果必须做不兼容变化，我会新增版本或新路径，旧接口标记弃用并保留迁移窗口。

15. 小测：先判断，再展开

1. 把 count 改名为 total 是兼容变更吗？

不是。旧客户端读取 count 会失败；这等价于删除旧字段。

2. 新增可选查询参数，但默认行为不变，通常兼容吗？

通常兼容。旧客户端不传新参数时仍然得到旧行为。

3. 为什么从裸数组改成响应信封是破坏性变化？

因为顶层 JSON 类型从数组变成对象，旧客户端的解析代码可能直接失败。

4. 运行时契约测试和 OpenAPI 契约测试分别守什么？

前者守真实响应形状，后者守文档和工具看到的 schema。

16. 本节总结

这一节的核心观点很硬：API 返回的 JSON 形状不是随手可改的实现细节，而是调用方依赖的公共契约。新增可选能力通常容易兼容；删除、改名、改类型、改默认语义往往是破坏性变更。
