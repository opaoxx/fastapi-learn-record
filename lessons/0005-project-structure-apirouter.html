项目结构与 APIRouter：从单文件 Demo 到 GitHub 可读工程

本节解决“代码能跑但不像项目”的问题。核心是把入口、路由、模型、数据库、依赖、安全、服务层分开，让 FastAPI 仓库具备可维护、可测试、可展示的工程结构。

① 本节核心知识框架

FastAPI 工程拆分
├─ main.py：应用组装层
│  ├─ FastAPI(...)
│  ├─ middleware
│  ├─ include_router(...)
│  └─ StaticFiles mount
├─ routers/：HTTP 接口层
│  ├─ system.py：健康检查、配置、provider health
│  ├─ items.py：商品 CRUD
│  ├─ predictions.py：预测与 provider metrics
│  ├─ tasks.py：后台任务状态
│  └─ files.py：文件上传
├─ schemas.py：请求/响应/表模型
├─ dependencies.py：可复用依赖
├─ database.py：数据库 engine/session/建表/seed
├─ services/：业务与外部 provider 边界
└─ tests/：契约验证与回归保护

② 核心概念底层原理

单文件 Demo 的问题不是“低级”，而是扩展边界很快耗尽。当接口从 /health 扩展到 CRUD、鉴权、任务、上传、AI provider、metrics 后，继续堆在 main.py 会导致导入混乱、职责不清、测试困难和 Git 冲突频发。

APIRouter 是路由聚合器，不是第二个应用。router 保存一组路径操作、统一前缀、tag、依赖和响应配置；只有被 app.include_router(...) 挂载后，它的路由才进入主应用路由表。

项目结构是架构约束。把 HTTP 层放在 routers/，把业务放在 services/，把数据模型放在 schemas.py，不是形式主义，而是限制不同职责互相污染。面试中可以把它解释为“分层架构的最小版本”。

GitHub 友好结构的判断标准。新读者打开仓库，应能从 README 找启动命令，从 main.py 找应用组装，从 routers/ 找接口，从 tests/ 找行为证明。这比“一个大文件能跑”更适合推广。

③ 全套代码逐行精读解析

from .routers import files, items, predictions, system, tasks

openapi_tags = [
    {"name": "system", "description": "Health checks and public runtime configuration."},
    {"name": "items", "description": "Database-backed learning endpoints for CRUD and filtering."},
    {"name": "predictions", "description": "Model-like synchronous endpoints for AI service practice."},
    {"name": "tasks", "description": "Protected background summary tasks with status, filtering, and pagination."},
    {"name": "files", "description": "Protected UTF-8 text uploads used as summary-task inputs."},
]

app = FastAPI(..., openapi_tags=openapi_tags, lifespan=lifespan)

app.include_router(system.router)
app.include_router(items.router)
app.include_router(predictions.router)
app.include_router(tasks.router)
app.include_router(files.router)
  • from .routers import ...：相对导入当前包下的路由模块。若用错误的绝对路径，运行方式变化时容易导入失败。
  • openapi_tags：声明文档分组及说明。没有它接口仍能跑，但 /docs 的可读性下降。
  • system tag：放健康检查、公开配置和 provider health，属于运行状态而非业务资源。
  • items tag：放数据库驱动的 CRUD 学习接口，是早期后端基础主线。
  • predictions tag：放 AI-like 同步接口和 provider metrics，连接后续 AI 服务章节。
  • tasks tag：放后台任务，体现异步处理与状态查询边界。
  • files tag：放文件上传，用于摘要任务输入。
  • app = FastAPI(...)：主应用只负责全局配置，不承载具体业务函数。
  • include_router(...)：把各模块 router 的路径操作合并进主路由表。漏掉某行，对应模块所有接口都会 404。
router = APIRouter(prefix="/items", tags=["items"])

@router.get("", response_model=list[ItemRead])
def list_items(...):
    ...

@router.get("/{item_id}", response_model=ItemRead)
def read_item(...):
    ...
  • APIRouter(prefix="/items", tags=["items"])：定义商品模块统一 URL 前缀和文档标签。
  • @router.get("")：空子路径与前缀组合成 GET /items。
  • @router.get("/{item_id}")：子路径与前缀组合成 GET /items/{item_id}。
  • 如果把这些接口直接写在 main.py，短期少一个文件，长期会让入口层变成业务层。

④ 核心机制运行全流程拆解

应用启动
  -> 导入 first_api.main
  -> 导入 routers/files/items/predictions/system/tasks
  -> 每个 router 模块创建 APIRouter 并注册路径操作
  -> main.py 创建 app
  -> app.include_router(...) 合并路由表
  -> /docs 读取统一 OpenAPI schema

请求 GET /items/1
  -> 主 app 路由表匹配 /items/{item_id}
  -> 实际执行 first_api.routers.items.read_item
  -> 返回 ItemRead JSON

⑤ 重难点、易混淆点对比辨析

  • router prefix vs 装饰器 path：最终路径是二者拼接。prefix="/items" + @router.get("") 等于 /items。
  • router tag vs URL：tag 只影响文档分组，不影响 URL。
  • schemas.py vs routers.py：模型是接口契约，应被多个路由、测试、文档复用；不要塞进单个路由文件导致循环导入。
  • services vs routers：router 处理 HTTP 细节；service 处理业务流程和外部依赖。把业务放 router 会让测试和复用变难。

⑥ 开发常见报错+坑点+解决方案

  • GET /items 404：忘记 app.include_router(items.router)。解决：检查 main.py 挂载。
  • ImportError: attempted relative import beyond top-level package：从错误工作目录直接运行子文件。解决：从项目根启动 first_api/main.py。
  • 接口出现在 /docs 但 tag 混乱：router 未设置 tags 或 tag 名不一致。
  • 循环导入：router 导入 main，main 又导入 router。解决：router 不应依赖 app；公共对象放 schemas/dependencies/services。
  • 静态前端路径错误：使用当前文件相对路径 Path(__file__).parent，不要依赖启动命令所在目录。

⑦ 生产环境 vs 教学环境 核心差异

  • 模块规模：教学可按单文件模块拆；生产常进一步按领域包拆分，例如 items/router.py、items/service.py、items/models.py。
  • 依赖方向：教学可以轻量约束；生产应严格禁止 service 反向依赖 router，避免架构腐化。
  • 配置与启动：教学入口简单；生产会有应用工厂、环境配置、日志初始化、监控和中间件顺序约束。
  • 文档维护：教学用自动文档即可；生产需要版本化文档、变更记录、兼容性说明和契约测试。

⑧ 课后记忆习题+标准答案

1. APIRouter 的本质是什么？

它是一组路径操作的容器，用于模块化组织接口，挂载后成为主应用路由表的一部分。

2. 为什么 main.py 应该保持薄？

入口层只负责应用组装，业务分散到路由和服务模块后更容易维护、测试和协作。

3. prefix="/items" 与 @router.get("/{item_id}") 合成什么？

合成 GET /items/{item_id}。

⑨ 面试高频真题+解析

1. FastAPI 项目为什么要拆 router？

为了按业务领域组织接口，降低 main.py 复杂度，减少冲突，并让每个模块能独立阅读和测试。

2. 如何避免 FastAPI 项目循环导入？

保持依赖方向清晰：main 导入 router，router 导入 schemas/dependencies/services；router 不导入 main 或 app。

3. GitHub 上一个 FastAPI 学习项目至少应具备哪些结构？

README、requirements、应用包、routers、schemas、dependencies/database/services、tests、课程或文档入口。
