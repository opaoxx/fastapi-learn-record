Lesson 0005

项目结构和 APIRouter：让 main.py 变薄

前几节课把所有代码写在 main.py 里，这是入门时最直观的方式。但真实项目会不断增长，如果不拆分，代码很快会变成难维护的一大坨。本节学习 FastAPI 官方推荐的大项目拆分方式：APIRouter。

0. 本节学习目标

  • 理解为什么后端项目需要按职责拆分文件。
  • 理解 APIRouter 是“接口分组器”，不是一个新的 app。
  • 看懂当前项目拆分后的目录结构。
  • 知道 main.py 应该主要负责创建应用和挂载路由。

1. 背景问题：为什么不能一直写在 main.py

如果只有 3 个接口，所有代码放在 main.py 里没问题。但当项目发展到登录、用户、商品、订单、AI 推理、任务队列、文件上传时，一个文件会同时包含模型、数据、业务逻辑、路由和错误处理。

这种代码会有几个问题：

  • 查找困难：想改商品接口，却要在几百行代码里搜索。
  • 职责混乱：模型、数据和接口函数互相挤在一起。
  • 协作困难：多人同时改一个大文件，很容易冲突。
  • 测试困难：想测试一个模块时，很难单独拿出来。

工程原则：文件拆分不是为了显得高级，而是为了让每个文件只承担清晰的职责。

2. 当前项目结构

这节课已经把示例项目拆成了下面这样：

first_api/
  main.py
  schemas.py
  store.py
  dependencies.py
  routers/
    __init__.py
    system.py
    items.py
    predictions.py

每个文件的职责是：

  • main.py：创建 FastAPI() 应用，并挂载所有 router。
  • schemas.py：放 Pydantic 模型，例如 Item、PredictionRequest。
  • store.py：放临时内存数据和数据查询函数。
  • dependencies.py：放可复用的依赖逻辑，第 6 课会重点讲。
  • routers/items.py：放商品相关接口。
  • routers/predictions.py：放 AI 预测相关接口。
  • routers/system.py：放根接口、健康检查这类系统接口。

3. APIRouter 是什么

APIRouter 可以理解成“一组接口的收纳盒”。你先在某个模块里创建 router，把相关接口挂到这个 router 上，再在 main.py 里统一挂载。

from fastapi import APIRouter

router = APIRouter(prefix="/items", tags=["items"])

@router.get("")
def list_items():
    ...

注意这里用的是 @router.get，不是 @app.get。原因是这个文件不直接创建总应用，它只声明“商品模块有哪些接口”。

prefix="/items" 表示这个 router 下的接口统一以 /items 开头。这样 @router.get("") 最终就是 GET /items，@router.get("/{item_id}") 最终就是 GET /items/{item_id}。

4. main.py 现在做什么

拆分后，main.py 变得很薄：

from fastapi import FastAPI

from .routers import items, predictions, system

app = FastAPI(
    title="First FastAPI API",
    description="A tiny learning API for backend basics and future AI service endpoints.",
    version="0.1.0",
)

app.include_router(system.router)
app.include_router(items.router)
app.include_router(predictions.router)

include_router 的意思是：把某个 router 中声明的所有接口加入到主应用里。官方文档也强调，router 被 include 后，它里面的路径操作会成为主应用的一部分。

这个结构对求职项目很加分，因为别人打开项目时能快速判断：入口在哪、模型在哪、业务接口在哪。

5. import 语法小补课

你会看到这种写法：

from .routers import items, predictions, system
from ..schemas import Item, ItemCreate

前面的点是 Python 包内相对导入：

  • . 表示当前包，例如 first_api。
  • .. 表示上一级包，例如从 first_api/routers/items.py 回到 first_api。

这也是为什么目录里需要 __init__.py：它告诉 Python，这个目录可以作为包来导入。

6. 实验：确认拆分后接口没变

运行服务：

.\.venv\Scripts\Activate.ps1
fastapi dev first_api/main.py

然后打开 http://127.0.0.1:8000/docs。你应该看到接口按 tag 分组，例如 items、predictions、system。

再测试这些接口：

GET /health
GET /items
GET /items/2
POST /predict

如果拆分正确，接口行为应该和前几节课一样。拆文件不应该改变业务行为，这是重构的重要原则。

8. 小测：先回忆再展开

1. APIRouter 的核心作用是什么？

它把一组相关接口收纳到一个模块里，再由主应用统一挂载。

2. main.py 变薄有什么好处？

入口更清晰，模块职责更明确，后续扩展、测试和协作更容易。

3. prefix="/items" 和 @router.get("/{item_id}") 合起来是什么路径？

合起来是 GET /items/{item_id}。

9. 本节总结

本节的核心是项目结构。APIRouter 让接口按领域分组，main.py 只负责应用组装。以后做 AI 服务时，普通业务接口、模型推理接口、任务接口都可以拆成不同 router。
