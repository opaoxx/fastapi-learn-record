0077 GitHub Actions CI Workflow：把本地质量门禁升级为仓库级自动化

本节新增 .github/workflows/ci.yml，把 provider 测试、文档契约测试、全量 pytest 和 whitespace 检查固化为 GitHub Actions 工作流。重点不是背 YAML，而是理解 CI 如何把协作规范变成可阻断合并的工程契约。

① 本节核心知识框架

GitHub Actions CI Workflow
├── 触发器
│   ├── push: main 分支更新后执行
│   └── pull_request: PR 指向 main 时执行
├── 执行环境
│   ├── runner: ubuntu-latest
│   ├── Python: actions/setup-python@v5
│   └── dependencies: requirements.txt
├── 质量门禁
│   ├── tests/test_provider_http.py
│   ├── tests/test_course_docs_contract.py

│   └── git diff --check
├── 契约测试
│   └── test_github_actions_ci_runs_course_quality_gates
└── 工程目标

    ├── PR 合并前暴露问题
    ├── 防止课程导航和文档结构退化
    └── 为 GitHub 新手提供可复制 CI 模板

② 核心概念底层原理

CI，Continuous Integration，持续集成，本质是“每次代码进入共享仓库前，由独立环境重新拉取代码、安装依赖、执行检查，并用进程退出码决定通过或失败”。它解决的不是语法问题，而是协作问题：本地能跑不等于别人 clone 后能跑，作者记得跑测试不等于每个贡献者都会跑测试。

GitHub Actions 的抽象层级是 workflow、event、job、runner、step。workflow 是一份 YAML 文件；event 决定什么时候执行；job 是一组在同一 runner 上执行的步骤；runner 是一次性的虚拟机；step 是具体命令或复用 action。测试失败时，pytest 返回非 0 退出码，GitHub 将该 step 标记为失败，整个 job 失败。

本仓库上一节已经把维护流程写入 DEVELOPMENT.md。这一节进一步把流程迁移到 .github/workflows/ci.yml。文档是人读的契约，CI 是机器执行的契约。只有二者同时存在，开源课程仓库才不会依赖某个维护者的记忆。

概念底层含义本节落点
workflow仓库级自动化流程定义.github/workflows/ci.yml
event触发 workflow 的 GitHub 事件push 与 pull_request
job在 runner 上执行的一组 steptest 质量门禁 job
runnerGitHub 提供的临时执行环境ubuntu-latest
step复用 action 或执行 shell 命令checkout、setup-python、pytest、diff check

CI 与单元测试不是同一层概念。单元测试描述“代码行为是否正确”，CI 描述“何时、在哪个环境、按什么顺序执行这些验证”。因此 CI 不是 pytest 的替代品，而是 pytest、文档契约、格式检查、构建命令的编排层。

③ 全套代码逐行精读解析

name: Course CI

定义工作流显示名称。名称会出现在 GitHub Actions 页面和 PR 检查列表中。若名称含糊，失败时贡献者无法快速判断是哪类质量门禁失败。

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

on 是触发器入口。push 保护 main 分支上的直接提交，pull_request 保护合并前变更。只监听 main 可以避免教学仓库在任意实验分支上频繁消耗 CI 资源。

jobs:
  test:
    name: pytest and documentation contracts
    runs-on: ubuntu-latest

jobs 下可以有多个任务。本节只需要一个 test job。runs-on 选择 Linux runner，因为本项目是纯 Python/FastAPI 课程仓库，跨平台风险较低。若项目依赖 Windows 路径或系统库，应增加 matrix 测试。

- name: Checkout repository
  uses: actions/checkout@v4
  with:
    fetch-depth: 0

actions/checkout 把仓库代码拉到 runner。fetch-depth: 0 保留完整历史，便于后续 git diff --check 比较 PR base 与 HEAD。若只浅克隆最新提交，PR diff 检查可能找不到 base ref。

- name: Set up Python
  uses: actions/setup-python@v5
  with:
    python-version: "3.11"
    cache: pip
    cache-dependency-path: requirements.txt

设置 Python 版本并开启 pip 缓存。cache-dependency-path 指向 requirements.txt，依赖文件不变时可复用缓存。教学仓库固定 3.11，降低初学者理解成本；生产仓库可使用 matrix 同时测试多个版本。

- name: Install dependencies
  run: |
    python -m pip install --upgrade pip
    python -m pip install -r requirements.txt

先升级 pip，再安装项目依赖。这里使用 python -m pip 而不是裸 pip，可以保证 pip 与当前 Python 解释器一致，避免多 Python 环境下安装到错误位置。

- name: Run provider HTTP tests
  run: python -m pytest tests/test_provider_http.py -q

先跑 provider HTTP 目标测试。这组测试覆盖 provider health、prediction adapter、retry、metrics、runbook、practice package 等课程后半段核心业务逻辑。若这里失败，问题通常集中在服务边界或 metrics 训练包。

- name: Run documentation contract tests
  run: python -m pytest tests/test_course_docs_contract.py -q

再跑文档契约测试。课程仓库的产品不仅是 API，还包括 lesson、reference、README、DEVELOPMENT 和索引导航。该测试能防止课程九段结构、GitHub 学习入口、runbook 链接和 CI 契约被误删。

- name: Run full test suite
  run: python -m pytest -q

最后跑全量 pytest。目标测试用于快速定位，全量测试用于发现跨模块回归。只跑目标测试会遗漏 CRUD、OpenAPI、静态前端、任务接口等不在当前主题内的破坏。

- name: Check whitespace in committed diff
  run: |
    if [ "${{ github.event_name }}" = "pull_request" ]; then
      git diff --check "origin/${{ github.base_ref }}...HEAD"
    elif git rev-parse HEAD^ >/dev/null 2>&1; then
      git diff --check HEAD^ HEAD
    else
      git diff --check
    fi

最后检查 diff 中的尾随空格等 whitespace 问题。PR 场景比较 base 分支与当前 HEAD；普通 push 场景比较当前提交与父提交；初始提交没有父提交时退回到工作区检查。若直接在 CI 中只写 git diff --check，干净 checkout 往往没有工作区 diff，检查价值会下降。

def test_github_actions_ci_runs_course_quality_gates() -> None:
    workflow = Path(".github/workflows/ci.yml").read_text(encoding="utf-8")

文档契约测试直接读取 CI 文件。这里不引入 YAML 解析库，是为了避免为了测试 workflow 而增加新依赖。测试保护的是关键门禁短语，而不是 YAML 的完整语义。

assert "actions/checkout@v4" in workflow
assert "actions/setup-python@v5" in workflow
assert "python -m pytest tests/test_provider_http.py -q" in workflow
assert "python -m pytest tests/test_course_docs_contract.py -q" in workflow
assert "python -m pytest -q" in workflow
assert "git diff --check" in workflow

这些断言把 CI 的核心职责锁住：拉代码、装 Python、跑业务测试、跑文档测试、跑全量测试、检查 whitespace。未来若有人删掉文档契约测试或全量测试，CI 契约测试会先失败。

④ 核心机制运行全流程拆解

Developer pushes / opens PR
  -> GitHub receives event
  -> Match .github/workflows/ci.yml
  -> Create ubuntu-latest runner
  -> checkout repository
  -> setup Python 3.11
  -> install requirements.txt
  -> run targeted provider tests
  -> run course documentation contract tests
  -> run full pytest
  -> run committed diff whitespace check
  -> report pass/fail status to commit or PR

CI 的关键在“全新环境”。本地机器可能有隐藏依赖、旧虚拟环境、未提交文件和特殊环境变量；runner 每次从仓库重新开始，能暴露“只在我电脑上能跑”的问题。

阶段输入输出失败含义
checkoutGitHub 仓库runner 本地工作区仓库权限或 ref 获取失败
setup-python版本号与缓存配置可用 Python 解释器版本不可用或 action 配置错误
installrequirements.txt依赖环境依赖声明缺失、版本冲突、网络问题
pytest测试文件与源码行为验证结果业务逻辑、文档结构或契约回归
diff check提交 diffwhitespace 诊断尾随空格等提交质量问题

⑥ 开发常见报错+坑点+解决方案

报错或坑点原因解决方案
ModuleNotFoundError: No module named 'first_api'CI 工作目录不在仓库根目录，或包导入路径被破坏。保持 workflow 在仓库根目录运行，测试命令使用 python -m pytest。
pip install -r requirements.txt 失败依赖文件缺包、网络临时错误或版本冲突。先本地删除虚拟环境重装验证，必要时固定关键依赖版本。
文档契约测试失败新增课程未更新九段结构、README、索引或 workflow 契约。按 DEVELOPMENT.md 的 Lesson Change Checklist 补齐配套文件。
git diff --check 报 trailing whitespace提交中存在尾随空格或空白格式问题。删除尾随空格，本地运行同一命令后再提交。
PR 中 CI 没触发workflow 文件不在 .github/workflows/，或目标分支不匹配。确认文件路径、YAML 缩进、PR base 分支与 branches 配置。
本地通过但 CI 失败本地有未提交文件、缓存依赖或不同 Python 版本。在干净 clone 中复现，检查 requirements 与 CI Python 版本。

⑦ 生产环境 vs 教学环境 核心差异

维度教学仓库实现生产仓库常见做法
Python 版本固定 3.11，降低理解成本。使用 matrix 覆盖多个版本或锁定运行时版本。
测试范围provider 测试、文档契约、全量 pytest。增加 lint、type check、security scan、build、e2e。
密钥管理当前 CI 不需要 secrets。通过 GitHub Secrets 或 OIDC 获取云资源权限。
部署只做质量门禁，不自动发布。测试通过后进入 artifact、staging、production 部署流水线。
分支保护提供 CI workflow 模板。在 GitHub branch protection 中强制 required checks。
缓存策略仅缓存 pip。细分依赖缓存、构建缓存、远程缓存和失效策略。

⑧ 课后记忆习题+标准答案

1. CI 与本地测试的本质区别是什么？

本地测试依赖开发者主动执行和本机环境；CI 在共享仓库事件触发后，用独立 runner 自动重建环境并执行检查，结果会反馈到 commit 或 PR。

2. 为什么本节同时监听 push 和 pull_request？

push 覆盖 main 分支上的直接提交，pull_request 覆盖合并前审查场景。二者共同保护主干质量。

3. 为什么使用 python -m pytest 而不是直接 pytest？

它保证 pytest 由当前 Python 解释器执行，减少多 Python 环境下命令绑定错误。

4. 为什么文档契约测试应进入 CI？

课程仓库的 lesson、reference、README 和索引是核心交付物，文档结构退化会直接破坏学习体验，因此必须像代码行为一样被自动检查。

5. fetch-depth: 0 在本节 workflow 中解决什么问题？

它保留完整 Git 历史，使 PR 场景可以比较 base 分支与当前 HEAD，避免浅克隆导致 diff 检查缺少参照。

⑨ 面试高频真题+解析

面试题 1：CI/CD 中 CI 和 CD 的边界是什么？

CI 关注集成前后的自动验证，例如安装依赖、测试、静态检查、构建。CD 关注验证通过后的交付或部署，例如发布 artifact、部署 staging、灰度生产。本节只实现 CI，不包含自动部署。

面试题 2：为什么 CI 必须从干净环境安装依赖？

干净环境能暴露依赖声明缺失、隐式本地配置和未提交文件问题。如果 CI 复用开发者本机环境，它就无法证明仓库本身是可复现的。

面试题 3：为什么生产仓库常用 required checks？

required checks 将 CI 结果与分支保护绑定，未通过检查的 PR 不能合并。这样质量门禁从“建议执行”升级为“制度性阻断”。

面试题 4：如何设计一个不会拖慢开发节奏的 CI？

常见做法是分层执行：快速单元测试和契约测试作为 PR 必跑，耗时 e2e 或安全扫描按路径、标签或定时任务执行；同时使用缓存降低依赖安装时间。

面试题 5：为什么要把文档检查纳入工程质量门禁？

对课程、SDK、API 平台和开源项目而言，文档是用户接触系统的公共接口。文档链接、命令、结构错误会造成真实使用失败，因此应通过测试和 CI 保护。
