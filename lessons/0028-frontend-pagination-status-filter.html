Lesson 0028

前端分页与状态筛选：让 UI 控件驱动查询参数

第 27 课把 GET /tasks 渲染成任务历史。本节继续往前走：把 status、limit、offset 变成真实控件，让用户通过按钮和下拉框控制后端查询。

0. 本节学习目标

  • 理解“查询参数是前后端共享的状态语言”。
  • 掌握前端如何用一个 historyState 对象管理 status、limit、offset。
  • 看懂 URLSearchParams 如何安全构造查询字符串。
  • 理解状态筛选为什么要重置 offset。
  • 能解释上一页、下一页按钮为什么需要 count 和 items.length。

1. 背景：用户不应该手写 URL

第 23 到 26 课已经把任务列表接口设计成了这样：

GET /tasks?status=failed&limit=5&offset=10

对开发者来说，这个 URL 很清楚；但普通用户不应该手写它。用户更自然的操作是：

  • 点“Failed”按钮，只看失败任务。
  • 选择“每页 10 条”。
  • 点“下一页”。
  • 点“上一页”。

本节就是把这些界面动作翻译成查询参数。换句话说，前端控件不是装饰，它们是 API 查询状态的可视化入口。

2. 核心概念：查询参数是前后端共享的状态语言

后端已经认识这些参数：

status: queued | running | completed | failed | 不传
limit: 1 到 100 之间的整数
offset: 大于等于 0 的整数

前端需要维护同样的状态：

const historyState = {
  status: "",
  limit: 5,
  offset: 0,
  count: 0,
  itemsLength: 0,
};

这里的 status/limit/offset 是发给后端的查询状态；count/itemsLength 是后端响应后，前端用来判断分页按钮的派生状态。

3. 运行机制：点击筛选按钮之后发生了什么

  1. 用户点击 Failed 按钮。
  2. 前端从按钮的 data-status 读取值 failed。
  3. 前端把 historyState.status 改成 failed。
  4. 前端把 historyState.offset 重置为 0。
  5. buildTaskHistoryUrl() 生成 /tasks?limit=5&offset=0&status=failed。
  6. 浏览器用 fetch 请求后端。
  7. FastAPI 校验查询参数并查询数据库。
  8. 前端收到响应信封，渲染任务列表、元信息和分页按钮。

这条链路的重点是：按钮本身不直接“筛选 DOM”。它改变查询状态，再由后端返回真实数据。

4. 官方资料中的关键点

MDN 的 URLSearchParams 是专门处理 URL 查询字符串的 Web API。它可以设置、删除、迭代参数，并生成适合放进 URL 的查询字符串。

MDN 的 dataset 允许 JavaScript 读取 HTML 元素上的 data-* 自定义属性。例如 data-status="failed" 会变成 button.dataset.status。

FastAPI 的查询参数文档说明：查询参数不是路径的固定部分，它们可以有默认值、可选值、类型转换和校验规则。我们的前端控件最终就是在给这些后端参数赋值。

5. 代码精读一：HTML 控件如何表达查询状态

<div class="status-filters" role="group" aria-label="任务状态筛选">
  <button class="status-filter" type="button" data-status="" aria-pressed="true">全部</button>
  <button class="status-filter" type="button" data-status="queued" aria-pressed="false">Queued</button>
  <button class="status-filter" type="button" data-status="running" aria-pressed="false">Running</button>
  <button class="status-filter" type="button" data-status="completed" aria-pressed="false">Completed</button>
  <button class="status-filter" type="button" data-status="failed" aria-pressed="false">Failed</button>
</div>

每个按钮都用 data-status 保存它代表的后端状态。全部 的值是空字符串，意思是不传 status 查询参数。

aria-pressed 表示按钮是否处于选中状态，既能辅助样式，也能让辅助技术理解当前筛选。

6. 代码精读二：页大小控件

<label for="history-limit">每页数量</label>
<select id="history-limit">
  <option value="3">3</option>
  <option value="5" selected>5</option>
  <option value="10">10</option>
</select>

limit 是一个选项集合，所以用下拉框很合适。我们没有给用户一个任意数字输入框，是因为后端已经限制 limit 最大为 100，而学习项目里 3、5、10 更便于观察分页效果。

页大小一旦改变，也应该回到第一页。否则用户在第 5 页时把每页 5 条改成每页 10 条，原来的 offset 可能不再符合用户直觉。

7. 代码精读三：用一个对象保存历史状态

const historyState = {
  status: "",
  limit: Number(historyLimit.value),
  offset: 0,
  count: 0,
  itemsLength: 0,
};

这个对象是本节最重要的设计。它把散落在按钮、下拉框、响应结果里的信息收拢到一个地方。

  • status：当前筛选条件。
  • limit：每页最多多少条。
  • offset：当前从哪里开始。
  • count：当前筛选条件下的总数。
  • itemsLength：当前页实际返回了多少条。

状态对象让页面行为变得可推理。你可以问：下一次请求要发什么？看 historyState 就知道。

8. 代码精读四：构造查询字符串

function buildTaskHistoryUrl() {
  const params = new URLSearchParams({
    limit: String(historyState.limit),
    offset: String(historyState.offset),
  });

  if (historyState.status) {
    params.set("status", historyState.status);
  }

  return `/tasks?${params.toString()}`;
}

这里不用手动字符串拼接：

"/tasks?limit=" + limit + "&offset=" + offset + "&status=" + status

原因是手动拼接容易漏掉 &、忘记编码、或者在 status 为空时留下奇怪的 URL。URLSearchParams 把“键值对变查询字符串”这件事交给浏览器标准 API。

注意 status 只有在非空时才加入。这样“全部”对应的 URL 是：

/tasks?limit=5&offset=0

而不是：

/tasks?limit=5&offset=0&status=

9. 代码精读五：筛选变化为什么要 offset=0

for (const button of statusFilterButtons) {
  button.addEventListener("click", async () => {
    historyState.status = button.dataset.status;
    historyState.offset = 0;
    await loadTaskHistory();
  });
}

假设你正在“全部任务”的第 5 页，offset=40。然后你点击 Failed。失败任务可能总共只有 3 条。如果仍然请求 status=failed&offset=40，你会看到空列表，误以为没有失败任务。

所以筛选条件变化时，应该回到第一页。这是分页 UI 的基本规则：改变结果集，就重置当前位置。

10. 代码精读六：上一页和下一页

previousPageButton.addEventListener("click", async () => {
  historyState.offset = Math.max(historyState.offset - historyState.limit, 0);
  await loadTaskHistory();
});

nextPageButton.addEventListener("click", async () => {
  if (historyState.offset + historyState.itemsLength < historyState.count) {
    historyState.offset += historyState.limit;
    await loadTaskHistory();
  }
});

上一页很直观：从当前 offset 减去 limit，但不能小于 0。

下一页需要更小心。只有当 offset + itemsLength < count 时，才说明后面还有数据。这里不能只看 itemsLength === limit，因为最后一页可能刚好满页，也可能因为筛选条件不同而变化。

11. 代码精读七：渲染按钮状态

function renderPager() {
  const start = historyState.count === 0 ? 0 : historyState.offset + 1;
  const end = historyState.offset + historyState.itemsLength;
  pagerInfo.textContent = `第 ${start}-${end} 条 / 共 ${historyState.count} 条`;
  previousPageButton.disabled = historyState.offset === 0;
  nextPageButton.disabled =
    historyState.offset + historyState.itemsLength >= historyState.count;
}

按钮是否可点不是写死的，而是由响应信封计算出来。后端给出 count，前端结合 offset 和 items.length 得出 UI 状态。

这就是第 25、26 课的真正用途：没有响应信封和 count 查询，前端分页按钮只能猜。

12. 常见错误与诊断

错误一：筛选后不重置 offset

现象：切换状态后看到空列表。原因：新结果集没有那么多页。

错误二：手写查询字符串

现象：漏掉 &、空参数或编码问题。用 URLSearchParams 更稳。

错误三：下一页永远可点

现象：点到最后还有下一页。原因：没有用 count 判断边界。

错误四：按钮状态和请求状态不同步

现象：按钮显示 Failed，但请求仍是全部。原因：没有用同一个状态对象驱动 UI 和请求。

13. 课堂实验

运行服务后打开：

http://127.0.0.1:8000/app/

按顺序观察：

  1. 点 Completed，看元信息里的状态和任务列表是否变化。
  2. 点 Failed，观察空列表或失败任务。
  3. 把每页数量从 5 改成 3，看是否回到第一页。
  4. 连续点击“下一页”，直到按钮禁用。
  5. 点击“上一页”，观察 offset 如何减少。

同时可以打开浏览器开发者工具的 Network 面板，查看每次请求的 URL 是否和控件状态一致。

14. 小测：先回忆，再展开

1. 为什么状态筛选变化时要把 offset 重置为 0？

因为筛选会改变结果集，旧 offset 可能指向新结果集不存在的位置。

2. URLSearchParams 在本节解决了什么问题？

它把查询参数键值对转换成规范 URL 查询字符串，避免手写拼接出错。

3. “下一页”按钮为什么需要 count？

因为前端要知道当前页后面是否还有符合条件的数据。

4. data-status 和 dataset.status 是什么关系？

HTML 中的 data-status 会通过 DOM 的 dataset.status 暴露给 JavaScript。

15. 本节总结

本节把任务历史页面从“显示列表”升级成“可交互查询工具”。你现在有了一个完整闭环：用户操作控件，前端更新 historyState，URLSearchParams 生成查询字符串，FastAPI 解析查询参数，后端返回响应信封，前端再根据 items/count/limit/offset 更新界面。
