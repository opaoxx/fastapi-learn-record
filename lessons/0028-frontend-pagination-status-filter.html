前端分页与状态筛选：用 UI 控件驱动后端查询参数

第 23-26 课把 status/limit/offset 设计成 API 查询参数。本节把它们变成按钮、下拉框和分页控件。重点不是控件样式，而是用单一状态对象维护前后端共享查询语言。

① 本节核心知识框架

前端查询状态管理
├─ HTML 控件
│  ├─ status buttons：data-status
│  ├─ limit select：history-limit
│  └─ previous/next buttons
├─ 状态对象：historyState
│  ├─ status：后端 status 查询参数
│  ├─ limit：后端 limit 查询参数
│  ├─ offset：后端 offset 查询参数
│  ├─ count：后端响应总数
│  └─ itemsLength：当前页长度
├─ URL 构造：URLSearchParams
├─ 状态规则：筛选或页大小变化 -> offset = 0
└─ 按钮规则：next disabled when offset + itemsLength >= count

② 核心概念底层原理

查询参数是前后端共享的状态语言。后端定义 status/limit/offset 的合法值，前端控件负责把用户动作翻译成这些参数。用户点击 Failed，不是前端从当前 DOM 里筛掉任务，而是发起 GET /tasks?status=failed，让后端按真实数据库状态返回结果。

单一状态对象可以避免 UI、URL、按钮状态彼此漂移。historyState 中的 status/limit/offset 用于构造请求，count/itemsLength 来源于响应，用于推导上一页/下一页是否可用。

筛选条件改变时必须重置 offset。因为结果集变了，旧偏移量可能指向新集合不存在的位置。页大小变化也一样：每页 5 条的第 5 页，不等价于每页 10 条的第 5 页。

③ 全套代码逐行精读解析

<button class="status-filter" type="button" data-status="failed" aria-pressed="false">Failed</button>
  • data-status 保存后端认识的枚举值；显示文本可以变，但 data 值必须和 TaskStatus 对齐。
  • aria-pressed 表达当前筛选按钮是否被选中，兼顾状态渲染和可访问性。
  • type="button" 防止按钮在表单上下文中默认提交。
const historyState = {
  status: "",
  limit: Number(historyLimit.value),
  offset: 0,
  count: 0,
  itemsLength: 0,
};
  • status: "" 表示“不传 status”，即查看全部任务。
  • Number(historyLimit.value) 将 select 字符串值转为数字；不转换会导致分页计算出现字符串拼接隐患。
  • offset: 0 初始从第一页开始。
  • count/itemsLength 是响应后更新的派生状态，不应由用户控件直接写入。
function buildTaskHistoryUrl() {
  const params = new URLSearchParams({
    limit: String(historyState.limit),
    offset: String(historyState.offset),
  });
  if (historyState.status) {
    params.set("status", historyState.status);
  }
  return `/tasks?${params.toString()}`;
}
  • URLSearchParams 负责编码和拼接，避免手写 ?、&、空值导致错误。
  • String(...) 明确 URL 参数都是字符串。
  • 只有 status 非空才加入参数；全部任务请求保持 /tasks?limit=5&offset=0。
button.addEventListener("click", async () => {
  historyState.status = button.dataset.status;
  historyState.offset = 0;
  await loadTaskHistory();
});

事件处理的顺序是：读取按钮状态、更新统一状态对象、重置页码、重新请求后端。若省略 offset = 0，切换到较小结果集时会出现空页假象。

nextPageButton.disabled =
  historyState.offset + historyState.itemsLength >= historyState.count;

下一页是否可用由响应事实推导。不能只看当前页是否满页，因为最后一页可能刚好满页；也不能只看 offset，因为筛选条件会改变总数。

④ 核心机制运行全流程拆解

用户点击 Failed
  -> button.dataset.status == "failed"
  -> historyState.status = "failed"
  -> historyState.offset = 0
  -> buildTaskHistoryUrl()
       /tasks?limit=5&offset=0&status=failed
  -> fetch 请求后端
  -> FastAPI 校验 Literal["queued","running","completed","failed"]
  -> service where(status == "failed") + count
  -> 返回 envelope
  -> renderTaskHistory 更新 count/itemsLength
  -> renderStatusFilters 更新 aria-pressed
  -> renderPager 更新按钮 disabled

⑥ 开发常见报错+坑点+解决方案

  • 切换筛选后空列表：旧 offset 超出新结果集。解决：筛选变化时设置 offset = 0。
  • 请求 URL 出现 status=：全部状态也塞入空参数。解决：status 非空再 params.set。
  • 按钮选中态错误：UI 没从 historyState 渲染。解决：集中调用 renderStatusFilters()。
  • 分页计算变字符串拼接：select 值未转数字。解决：使用 Number(historyLimit.value)。
  • 下一页请求越界：没有用 count 禁用按钮。解决：响应后更新 count/itemsLength 并渲染 pager。

⑧ 课后记忆习题+标准答案

1. 为什么查询参数是前后端共享状态语言？

后端用查询参数定义可查询维度，前端控件把用户操作翻译成这些参数，请求和响应围绕同一状态模型工作。

2. 为什么筛选变化要重置 offset？

筛选改变了结果集，旧 offset 可能在新结果集中越界或不符合用户预期。

3. URLSearchParams 相比手写拼接有什么优势？

它负责规范编码和拼接参数，减少漏写分隔符、空参数和特殊字符编码错误。

4. 下一页按钮禁用条件是什么？

当 offset + itemsLength >= count 时，当前页后面没有更多数据，应禁用下一页。

⑨ 面试高频真题+解析

题 1：前端筛选应该在本地做还是请求后端做？

如果数据量小且已全量加载，可以本地筛；但分页列表通常应让后端筛选，因为前端只有当前页，无法代表完整结果集。

题 2：如何避免前端分页状态混乱？

用单一状态对象或 URL 状态作为事实来源；筛选和页大小变化重置 offset；响应后用 envelope 更新 count/itemsLength；按钮状态从状态推导，不手写。

题 3：为什么生产环境常把筛选状态同步到 URL？

URL 可刷新恢复、可分享、可支持浏览器前进后退，也便于埋点和排障。页面内变量丢失后无法恢复用户视图。
