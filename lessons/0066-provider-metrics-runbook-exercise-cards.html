0066 Provider Metrics Runbook Exercise Cards：把排障判断拆成可训练题卡

本节新增 build_provider_metrics_runbook_exercise_cards()。它把 findings 转成题卡：给出 outcome 和 count，要求学习者写出下一步检查动作。课程从“看懂报告”推进到“能独立回答排障题”。

① 本节核心知识框架

Runbook Exercise Cards
├─ 输入：findings
├─ 输出：exercise cards
│  ├─ id
│  ├─ prompt
│  ├─ expected_severity
│  ├─ expected_next_action
│  └─ answer
├─ 训练目标
│  ├─ 从 outcome 识别排障分支
│  ├─ 从 count 判断观察规模
│  ├─ 从 severity 判断优先级
│  └─ 从 next_action 写出下一步
└─ 测试目标
   ├─ id 稳定
   ├─ prompt 稳定
   ├─ severity 顺序稳定
   └─ answer 不丢 finding

② 核心概念底层原理

题卡是认知训练工具。它把完整 runbook 拆成最小可回答单元：看到一个 outcome 和 count，学生必须说出 severity 与 next_action。这个过程非常接近面试中的故障排查追问。

题卡不是测试业务代码，而是测试学习者的判断路径。代码层面仍然要保证题卡从真实 findings 生成，避免题目与项目指标契约脱节。

题卡生成函数的输入不是原始 metric sample，而是解释后的 findings。这一层选择很重要：如果直接从 samples 出题，题卡就必须重复判断 severity 和 next_action，导致解释规则分裂；从 findings 出题则复用单一事实源，降低漂移概率。

对象职责风险
Finding保存解释结果文案可能漂移
Exercise Card把解释结果变成问题题目可能丢失标准答案
Answer给出标准作答如果不测试会与 next_action 不一致

③ 全套代码逐行精读解析

def build_provider_metrics_runbook_exercise_cards(
    findings: list[dict[str, object]],
) -> list[dict[str, object]]:
    cards = []
    for index, finding in enumerate(findings, start=1):

enumerate(..., start=1) 让题号从 1 开始，更符合教学材料习惯。若使用 0 开始，读者容易把内部索引误认为题号。

cards.append(
    {
        "id": f"provider-metrics-runbook-{index}",
        "prompt": (
            "Outcome {outcome} has count {count}. What should you check next?"
        ).format(
            outcome=finding["outcome"],
            count=finding["count"],
        ),
        "expected_severity": finding["severity"],
        "expected_next_action": finding["next_action"],
        "answer": (
            "{finding_text} Next action: {next_action}"
        ).format(
            finding_text=finding["finding"],
            next_action=finding["next_action"],
        ),
    }
)

prompt 只暴露 outcome 和 count，避免把答案泄露给学生；expected_severity 和 expected_next_action 是评分锚点；answer 把 finding 与 action 拼接成完整标准答案。

return cards

返回结构化列表而不是直接渲染文本，是为了让后续答案键、网页、测试都能复用同一份题卡数据。

代码点作用不这样写的隐患
cards = []建立可追加的题卡集合，空 findings 时自然返回空列表。若返回 None，下游答案键无法统一迭代处理。
enumerate(..., start=1)生成面向读者的 1 基题号。0 基题号会暴露实现细节，不利于文档引用。
prompt只给 outcome/count 两个观察事实。若把 next_action 放进题干，题目失去训练判断的意义。
expected_severity保存优先级标准答案。只检查 action 会漏掉“该不该升级处理”的关键能力。
expected_next_action保存评分锚点，供后续自动/人工评估使用。若只保留自然语言 answer，机器评估时还要重新解析文本。
answer把事实解释和下一步动作合成完整作答。只写 action 会让学生无法说明判断依据。

④ 核心机制运行全流程拆解

samples
  → findings
  → exercise cards
    → prompt: 只给题目
    → expected_*: 评分锚点
    → answer: 标准答案
  → 学生作答
  → 对照 answer_key

这条链路把学习材料做成结构化数据。结构化题卡的好处是：既能生成 HTML，也能生成 Markdown，还能被测试保护。

调用链中最重要的分工是：findings 决定“正确判断是什么”，cards 决定“如何向学习者提问”。若把两者合并，题干、标准答案、排障规则会互相缠绕，后续很难单独修改题目呈现方式。

⑤ 重难点、易混淆点对比辨析

易混点错误理解正确理解

⑥ 开发常见报错+坑点+解决方案

坑点后果解决方案

题卡 id 不稳定文档链接和测试难维护用顺序编号生成稳定 id。
answer 丢 finding只剩操作，缺少事实解释答案拼接 finding 与 next_action。
题卡手写与代码契约漂移从 findings 自动生成。

缺失 expected_next_action无法机器化核对关键动作保留结构化评分字段，不只依赖自然语言答案。

维度教学环境生产环境
题卡用途训练排障判断演练、值班培训、事故复盘题库
题目来源固定 findings真实事故、演练场景、历史告警

ID 策略顺序号足够支撑课程阅读需要题库版本、场景标识、迁移策略和历史引用兼容。

⑧ 课后记忆习题+标准答案

1. 为什么题卡 prompt 不直接包含 next_action？

因为 next_action 是学生要推导的答案，放进 prompt 会破坏训练价值。

2. 为什么题卡要有 expected_severity？

severity 是优先级判断锚点，可以检查学生是否把 scheduled、exhausted、fail_fast 混淆。

3. 为什么题卡从 findings 生成？

这样能复用真实解释层契约，避免手写题目与项目语义漂移。

4. 为什么 answer 需要包含 finding？

排障答案不能只有动作，还要说明观察事实是什么。

5. 教学题卡为什么可以用顺序 ID，生产题库为什么不够？

教学页面顺序稳定且规模小；生产题库会插题、删题、归档历史演练，必须用版本化稳定 ID 保护引用。

⑨ 面试高频真题+解析

面试题 1：如何把 runbook 变成团队训练材料？

把常见信号拆成结构化题卡：给观察事实，要求说出 severity、判断依据和下一步动作。题卡从真实 findings 生成，避免脱离系统契约。

面试题 2：题卡和单元测试有什么相似点？

二者都固定输入、预期输出和验证标准。题卡验证人的判断，单元测试验证代码行为。

面试题 3：为什么 runbook 培训需要标准答案？

没有标准答案，团队成员会形成不同排障路径，事故时难以协作。标准答案能统一术语和动作顺序。
