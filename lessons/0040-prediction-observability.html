Lesson 0040

Prediction Observability：给 AI 调用留下可排查的证据

第 39 课让 /predict 可以在 demo 和 provider 后端之间切换。本节解决下一个真实工程问题：当预测接口变慢、失败或返回异常结果时，我们怎样知道它走了哪个后端、用了多久、失败码是什么？答案不是随手 print()，而是有意识地设计响应头和日志。

0. 本节学习目标

  • 理解可观测性和普通调试输出的区别。
  • 学会用 time.perf_counter() 测量预测调用耗时。
  • 学会用 FastAPI Response 设置响应头，同时保持 response_model 生效。
  • 学会用 Python logging 记录成功和失败的结构化上下文。
  • 学会用 pytest caplog 和响应头断言可观测性行为。

1. 背景：能调用 provider 还不够，还要能排查

一个 AI 服务上线后，常见问题不是“完全不能运行”，而是更模糊的现象：

  • 同一个 /predict 有时很快，有时很慢。
  • 本地 demo 正常，provider 环境偶发 503。
  • 前端只看到失败，不知道是 demo 失败、provider 失败还是响应体校验失败。
  • 测试失败时，只知道状态码不对，不知道接口走了哪条后端分支。

这就是可观测性要解决的问题：让系统在运行时留下足够证据，帮助我们回答“发生了什么、在哪里发生、花了多久、为什么失败”。

本节采用低侵入策略：不修改 PredictionResponse 的 JSON body，只增加响应头和服务端日志。这样公共 JSON 契约保持稳定，但排查信息更丰富。

2. 核心概念：响应头给调用方，日志给服务端

本节新增两类观察信号：

响应头

跟着 HTTP 响应返回。适合让调用方或测试快速看到本次请求走了哪个 backend、source 是什么、耗时多少毫秒、错误码是什么。

服务端日志

留在服务端。适合线上排查、聚合检索、报警和审计。日志可以记录更多上下文，但不要写入敏感请求内容。

二者的关系不是互相替代，而是分工协作：

client sees response headers
server keeps logs
tests assert both when needed

3. 为什么不把这些字段加进 JSON body

我们当然可以把 backend、elapsed_ms 加到 PredictionResponse 里。但这会改变公共 JSON 契约。前面刚学过 schema evolution：响应体字段变化虽然未必一定破坏兼容性，但它会让 API 语义变复杂。

本节先把观察信息放进 header：

X-Prediction-Backend: demo
X-Prediction-Source: demo-ai-client
X-Prediction-Elapsed-Ms: 1.23

失败时还会多一个：

X-Prediction-Error-Code: ai_provider_http_error

这样响应体仍然是稳定的 PredictionResponse 或 ErrorResponse，调试信息则放在更轻量的位置。

4. 本节项目改动概览

  • 在 routers/predictions.py 中加入模块级 logger。
  • 用 perf_counter() 测量 /predict 和 /provider/predict 的耗时。
  • 成功时设置 X-Prediction-Backend、X-Prediction-Source、X-Prediction-Elapsed-Ms。
  • 失败时通过 HTTPException(headers=...) 设置 backend、elapsed、error-code。
  • 成功和失败都写日志：prediction_completed 或 prediction_failed。
  • 测试断言响应头，并用 caplog 捕获成功日志。

5. 代码精读一：logger 与 perf_counter

import logging
from time import perf_counter

logger = logging.getLogger(__name__)

logging.getLogger(__name__) 会创建一个以当前模块名命名的 logger，例如 first_api.routers.predictions。这样日志来源清楚，测试也能只捕获这个模块的日志。

perf_counter() 适合测量短时间间隔。我们在请求处理开始时记录一次，成功或失败时再记录一次，两者相减得到耗时毫秒。

started_at = perf_counter()
elapsed_ms = (perf_counter() - started_at) * 1000

6. 代码精读二：统一构造可观测响应头

def build_prediction_headers(
    backend: str,
    elapsed_ms: float,
    source: str | None = None,
    error_code: str | None = None,
) -> dict[str, str]:
    headers = {
        "X-Prediction-Backend": backend,
        "X-Prediction-Elapsed-Ms": f"{elapsed_ms:.2f}",
    }
    if source is not None:
        headers["X-Prediction-Source"] = source
    if error_code is not None:
        headers["X-Prediction-Error-Code"] = error_code
    return headers

这个函数把 header 名称集中起来，避免成功路径、失败路径、/predict 和 /provider/predict 各写一遍字符串。小项目里看起来只是少几行代码；真实项目里，这能减少拼写不一致和字段含义漂移。

注意 header 值必须是字符串，所以耗时被格式化为 "1.23" 这种文本。

7. 代码精读三：成功时设置 Response headers

def set_prediction_headers(
    response: Response,
    backend: str,
    elapsed_ms: float,
    source: str | None = None,
) -> None:
    for name, value in build_prediction_headers(
        backend=backend,
        elapsed_ms=elapsed_ms,
        source=source,
    ).items():
        response.headers[name] = value

FastAPI 支持在 path operation function 中声明 response: Response 参数，然后设置响应头。即使函数返回 Pydantic 模型，response_model 仍会照常过滤和序列化响应体；设置的 header 会进入最终 HTTP 响应。

这就是本节的低侵入点：body 不变，header 增加。

8. 代码精读四：/predict 成功路径

started_at = perf_counter()
backend = settings.ai_prediction_backend

try:
    if backend == "provider":
        prediction = await request_provider_prediction(...)
    else:
        prediction = ai_client.predict_sentiment(...)
except AIClientError:
    ...

elapsed_ms = (perf_counter() - started_at) * 1000
set_prediction_headers(
    response=response,
    backend=backend,
    elapsed_ms=elapsed_ms,
    source=prediction.source,
)
logger.info(
    "prediction_completed",
    extra={
        "prediction_backend": backend,
        "prediction_source": prediction.source,
        "prediction_elapsed_ms": round(elapsed_ms, 2),
    },
)

成功路径做了两件事：

  • 给调用方返回 headers，方便立刻看到本次预测走了哪个后端。
  • 写一条 prediction_completed 日志，方便服务端聚合和排查。

extra={...} 会把额外字段挂到日志记录上。后面接 JSON 日志系统时，这些字段就可以变成可检索的结构化字段。

9. 代码精读五：失败路径也要有证据

except AIClientError as exc:
    elapsed_ms = (perf_counter() - started_at) * 1000
    logger.warning(
        "prediction_failed",
        extra={
            "prediction_backend": backend,
            "prediction_elapsed_ms": round(elapsed_ms, 2),
            "prediction_error_code": exc.error_code,
        },
    )
    raise HTTPException(
        status_code=503,
        detail={
            "error_code": exc.error_code,
            "message": exc.message,
        },
        headers=build_prediction_headers(
            backend=backend,
            elapsed_ms=elapsed_ms,
            error_code=exc.error_code,
        ),
    ) from exc

失败时尤其需要可观测性。否则线上只看到 503，很难知道是 demo client 失败、provider HTTP 失败、provider 响应体非法，还是超时。

这里有三个层次：

  • HTTP 状态码：客户端知道请求失败，状态是 503。
  • JSON detail：客户端知道稳定错误码和错误消息。
  • headers/logs：客户端和服务端都能看到 backend、耗时、error code。

10. 测试精读：断言 header 和日志

def test_predict_uses_overridable_ai_client_dependency(caplog) -> None:
    caplog.set_level(logging.INFO, logger=PREDICTIONS_LOGGER)
    ...
    assert response.headers["x-prediction-backend"] == "demo"
    assert response.headers["x-prediction-source"] == "fixed-test-client"
    assert float(response.headers["x-prediction-elapsed-ms"]) >= 0

    log_record = next(
        record for record in caplog.records
        if record.message == "prediction_completed"
    )
    assert log_record.prediction_backend == "demo"
    assert log_record.prediction_source == "fixed-test-client"

caplog 是 pytest 提供的日志捕获工具。它让我们不只测试 HTTP 响应，还能测试服务端是否真的写出了我们需要的日志上下文。

注意响应头在测试客户端中通常用小写读取，例如 x-prediction-backend。HTTP header 名称本身大小写不敏感。

12. 常见错误与诊断

错误一：用 print 当日志

print() 难分级、难过滤、难聚合。服务端代码应优先使用 logging。

错误二：只记录成功，不记录失败

失败路径更需要上下文。至少记录 backend、elapsed_ms 和 error_code。

错误三：把敏感文本写进日志

预测文本可能包含隐私数据。本节只记录元信息，不记录完整输入文本。

错误四：为了调试改响应 body

调试信息优先放 header 或日志，避免轻易改变公共 JSON 契约。

13. 面试视角：怎样解释预测可观测性

我会在预测接口中记录 backend、source、elapsed_ms 和 error_code 这类元信息。成功时通过响应头和 info 日志暴露 backend/source/耗时，失败时通过 HTTPException headers 和 warning 日志暴露 backend/耗时/错误码。这样不改变 JSON 响应契约，也能让调用方和服务端排查路径都有证据。

14. 小测：先回忆，再展开

1. 为什么本节不把 elapsed_ms 加进 PredictionResponse？

因为那会改变公共 JSON 契约；放进 header 可以提供调试信息，同时保持响应体稳定。

2. perf_counter() 在这里解决什么问题？

它用于测量一次预测调用从开始到成功或失败所花的耗时，适合短时间间隔测量。

3. 为什么失败路径也要设置 header 和写日志？

因为失败时更需要知道 backend、耗时和错误码，否则线上只能看到 503，很难定位原因。

4. 为什么不要把完整预测文本写入日志？

因为用户输入可能包含隐私或敏感信息；日志应优先记录可排查的元信息。

15. 本节总结

本节给预测接口加入了低侵入可观测性：响应体保持稳定，响应头提供 backend/source/elapsed/error-code，服务端日志记录成功和失败上下文。现在 /predict 不只是能调用 demo 或 provider，也能留下可排查证据。
