0040 Prediction Observability：给 AI 调用留下可排查证据

能调用 provider 只是第一步；能解释“走了哪个 backend、耗时多少、失败码是什么”才接近真实工程。本节通过 X-Prediction-* 响应头与结构化日志建立低侵入可观测性，不改变 PredictionResponse 的 JSON 契约。

① 本节核心知识框架

Prediction Observability
├─ 耗时测量：time.perf_counter()
├─ 调用方证据：X-Prediction-Backend / Source / Elapsed-Ms / Error-Code
├─ 服务端证据：logger.info("prediction_completed")
├─ 失败证据：logger.warning("prediction_failed")
├─ 契约策略：body 不变，headers 增强
└─ 测试手段：response.headers + pytest caplog

核心结论：可观测性不是随手打印，而是设计可被测试、可被聚合、可被排查的信号。响应头解决单次调用定位，日志解决服务端聚合分析。

② 核心概念底层原理

可观测性 回答四个问题：发生了什么、发生在哪里、花了多久、为什么失败。AI provider 调用具有外部依赖、网络等待、供应商限流、schema 漂移等不确定性，因此必须把 backend、source、elapsed、error_code 作为一等诊断信号。

信号位置适合内容不适合内容
响应 body业务结果或稳定错误对象频繁变化的调试字段
响应 header单次请求元信息、耗时、后端来源敏感输入文本、密钥
服务端日志结构化上下文、错误码、耗时、来源完整用户隐私文本
metrics可聚合计数和分布高基数字段、原始 prompt

③ 全套代码逐行精读解析

import logging
logger = logging.getLogger(__name__)

模块级 logger 会生成 first_api.routers.predictions 这类层级名称，便于测试用 caplog 精确捕获，也便于生产日志按模块过滤。

def build_prediction_headers(
    backend: str,
    elapsed_ms: float,
    source: str | None = None,
    error_code: str | None = None,
) -> dict[str, str]:

把 header 构造集中化。若每个端点手写 X-Prediction-Elapsed-Ms，字段命名、格式化精度和失败路径很容易不一致。

headers = {
    "X-Prediction-Backend": backend,
    "X-Prediction-Elapsed-Ms": f"{elapsed_ms:.2f}",
}

HTTP header 值必须是字符串，所以耗时格式化为两位小数文本。backend 表示 demo 或 provider，是定位分支的第一信号。

if source is not None:
    headers["X-Prediction-Source"] = source
if error_code is not None:
    headers["X-Prediction-Error-Code"] = error_code

成功路径通常有 source，失败路径通常有 error_code。使用可选字段避免把空字符串误当成真实信号。

def set_prediction_headers(response: Response, ...) -> None:
    for name, value in build_prediction_headers(...).items():
        response.headers[name] = value

FastAPI 允许在返回 Pydantic 模型的同时修改 Response headers。body 由 response_model 控制，headers 由该函数补充。

logger.info(
    "prediction_completed",
    extra={
        "prediction_backend": result.backend,
        "prediction_source": result.prediction.source,
        "prediction_elapsed_ms": round(result.elapsed_ms, 2),
    },
)

extra 将字段挂到日志记录对象上。接入 JSON 日志时，这些字段可以直接被检索、聚合和告警。注意没有记录完整输入文本，避免隐私泄漏。

④ 核心机制运行全流程拆解

成功请求
  → run_prediction() 计算 elapsed_ms
  → router set_prediction_headers(...)
  → logger.info("prediction_completed", extra=...)
  → 返回 PredictionResponse body + X-Prediction-* headers

失败请求
  → run_prediction() 抛 PredictionServiceError
  → router log_prediction_failed(...)
  → HTTPException(503, detail=..., headers=...)
  → 返回 ErrorResponse shape + X-Prediction-Error-Code

成功和失败都要留证据。只记录成功会导致线上 503 只能看到状态码；只记录失败会失去基线耗时，无法判断“慢”是否异常。

⑤ 重难点、易混淆点对比辨析

易混点正确理解本项目做法
print vs loggingprint 无级别、难过滤；logging 可结构化使用 logger.info/warning
body vs headerbody 是业务契约；header 适合元信息调试字段放 X-Prediction-*
elapsed_ms 来源应由 service 包装整个调用用例耗时PredictionServiceResult.elapsed_ms

⑥ 开发常见报错+坑点+解决方案

  • headers 设置后丢失：直接返回自定义 JSONResponse 覆盖了 Response。解决：当前模式返回 Pydantic 模型，使用注入的 Response 设置 header。
  • caplog 捕获不到日志：logger 名称或级别不对。解决：caplog.set_level(logging.INFO, logger="first_api.routers.predictions")。
  • 失败路径没有 header：只在成功函数里设置 header。解决：HTTPException(headers=build_prediction_headers(...))。
  • 日志泄漏输入文本：把完整 prompt/text 写进日志。解决：只记录长度、来源、错误码、耗时等低敏元信息。

⑧ 课后记忆习题+标准答案

1. 为什么不把 elapsed_ms 加进 PredictionResponse？

那会改变公共 JSON 契约。把它放入 header 可以提供调试信息，同时保持业务响应体稳定。

2. logging.getLogger(__name__) 的好处是什么？

日志来源与模块路径绑定，便于过滤、测试和生产检索。

3. 失败路径为什么也要设置 headers？

失败时更需要知道 backend、耗时和 error_code，否则只能看到 503，难以定位原因。

4. 为什么不要记录完整预测文本？

用户输入可能包含隐私或敏感数据，日志应优先记录低敏元信息。

⑨ 面试高频真题+解析

题 1：如何为 AI 调用设计可观测性？ 答：记录 backend、source、elapsed、error_code；成功与失败都要覆盖；调用方可见信息放 header，服务端聚合信息放结构化日志，敏感输入不进日志。

题 2：为什么不建议用 print 调试服务端接口？ 答：print 缺少级别、上下文、结构化字段和过滤能力，生产环境难以检索和关联。

题 3：如何测试日志行为？ 答：使用 pytest caplog 设置目标 logger 和级别，执行请求后断言日志 message 与 extra 字段。
