API Key 鉴权：用请求头和依赖注入保护状态变更接口

本节讲清楚最小鉴权链路：客户端在 X-API-Key 请求头携带密钥，FastAPI 的 APIKeyHeader 提取凭据，require_api_key 与 Settings 中的服务端密钥比较，写接口通过 Depends 强制执行检查。

① 本节核心知识框架

API Key 鉴权
├─ 威胁模型：未授权调用会创建/修改/删除数据，或消耗 AI 服务额度
├─ 请求头：X-API-Key: dev-secret-key
├─ APIKeyHeader
│  ├─ name="X-API-Key"
│  ├─ scheme_name="ApiKeyAuth"
│  ├─ auto_error=False
│  └─ OpenAPI security scheme
├─ require_api_key
│  ├─ Depends(api_key_header)：读取客户端 key
│  ├─ Depends(get_settings)：读取服务端 key
│  └─ 不匹配 -> 401
├─ 路由保护：dependencies=[Depends(require_api_key)]
└─ 测试保护：成功路径带 headers，失败路径断言 401

② 核心概念底层原理

鉴权回答“你是谁/你是否有凭据”。API Key 是最简单的共享密钥方案：客户端和服务端都知道同一个 secret，请求时客户端证明自己知道它。

认证与授权不要混淆。当前 API Key 只证明调用者持有密钥，属于认证雏形；它没有用户身份、角色、资源权限。生产系统还需要授权：这个调用者能不能改这条资源。

请求头比 query 更适合放凭据。query 会进入 URL、浏览器历史、代理日志和分享链接；header 更符合认证凭据的传输习惯。即便如此，生产也必须使用 HTTPS。

依赖放在装饰器 dependencies 中表示“执行检查但不使用返回值”。require_api_key 成功时返回 None，路径函数不需要它的结果，只需要它失败时阻断请求。

③ 全套代码逐行精读解析

api_key_header = APIKeyHeader(
    name="X-API-Key",
    scheme_name="ApiKeyAuth",
    description="Local learning API key required by protected write, file, and task endpoints.",
    auto_error=False,
)
  • APIKeyHeader：FastAPI 安全工具，负责从请求头读取 API Key，并生成 OpenAPI 安全方案。
  • name="X-API-Key"：约定客户端必须发送的 header 名。
  • scheme_name="ApiKeyAuth"：OpenAPI 中安全方案的名字，影响 /docs Authorize 面板。
  • description=...：说明保护范围，帮助 GitHub 读者理解哪些接口需要 key。
  • auto_error=False：缺少 header 时不由工具自动报错，而是交给 require_api_key 返回统一错误。
def require_api_key(
    api_key: Annotated[str | None, Depends(api_key_header)],
    settings: Annotated[Settings, Depends(get_settings)],
) -> None:
    if api_key != settings.api_key:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid or missing API key.",
        )
  • api_key：客户端传来的 header 值，缺失时是 None。
  • Depends(api_key_header)：把“从 header 取 key”的动作声明成依赖。
  • settings：服务端配置对象，包含正确的 api_key。
  • api_key != settings.api_key：统一处理缺失和错误 key。
  • HTTP_401_UNAUTHORIZED：认证失败。不要用 422，因为请求格式可能完全合法，只是凭据不对。
  • detail：错误信息保持简洁，不区分“缺失”还是“错误”，避免泄漏过多认证判断细节。
@router.post(
    "",
    response_model=ItemRead,
    status_code=status.HTTP_201_CREATED,
    dependencies=[Depends(require_api_key)],
)
def create_item(...):
    ...
  • dependencies=[...]：注册前置检查，不把返回值传给路径函数。
  • 同样模式用于 PATCH 和 DELETE，因为它们都会改变数据。
  • GET 暂时公开，是教学取舍：读接口方便学习，写接口先保护。

④ 核心机制运行全流程拆解

POST /items
Headers:
  X-API-Key: dev-secret-key
Body:
  {...}

请求进入
  -> 匹配 POST /items
  -> 执行 dependencies=[Depends(require_api_key)]
  -> api_key_header 从请求头读取 X-API-Key
  -> get_settings 读取服务端 Settings.api_key
  -> 比较两者
       ├─ 不相等：raise HTTPException(401)，create_item 不执行
       └─ 相等：继续解析 ItemCreate、Session，执行 create_item
  -> 成功返回 201

⑤ 重难点、易混淆点对比辨析

  • 401 vs 403：401 通常表示认证凭据缺失或无效；403 表示已认证但无权限。当前场景用 401。
  • API Key vs JWT：API Key 是共享静态密钥；JWT 可携带用户身份、过期时间和声明，更适合用户级认证。
  • Header vs Query：认证凭据应放 header；query 更容易被日志和分享链接泄漏。
  • 鉴权依赖 vs 中间件：依赖可按路由选择性启用；中间件更适合全局横切逻辑。

⑥ 开发常见报错+坑点+解决方案

  • 写接口返回 401：缺少或写错 X-API-Key。解决：带上当前 Settings 中的 key。
  • /docs 中请求仍 401：没有点击 Authorize 或 header 未附加。解决：在 Swagger UI 的 Authorize 填入 key。
  • 把 key 写进源码：安全风险。解决：使用 .env / 环境变量。
  • 测试成功路径失败：忘记 headers=API_KEY_HEADERS。
  • 所有接口都被保护导致学习不便：依赖挂载范围过大。解决：只给写接口或需要保护的 router 加依赖。

⑦ 生产环境 vs 教学环境 核心差异

  • 密钥强度：教学 key 简短固定；生产 key 应随机、高熵、可轮换。
  • 传输安全：教学本地 HTTP；生产必须 HTTPS，避免 header 被窃听。
  • 权限模型：教学只有一个共享 key；生产要有用户、角色、资源权限、审计日志。
  • 滥用防护：生产还需要限流、配额、IP/用户维度监控，尤其 AI 接口会消耗真实成本。

⑧ 课后记忆习题+标准答案

1. 为什么 API Key 放在 header 而不是 query？

query 容易进入 URL 历史、代理日志和分享链接，header 更适合携带认证凭据。

2. auto_error=False 的作用是什么？

让缺失 key 的情况交给自定义依赖统一处理，而不是由安全工具直接返回默认错误。

3. 为什么写接口需要优先保护？

写接口会改变数据或消耗资源，未授权调用的破坏性比只读接口更大。

⑨ 面试高频真题+解析

1. API Key 鉴权的优缺点是什么？

优点是简单、易接入；缺点是共享静态密钥难表达用户身份和细粒度权限，泄漏后风险大。

2. 401 和 403 如何区分？

401 表示认证凭据缺失或无效；403 表示凭据有效但没有访问该资源的权限。

3. AI 服务接口为什么必须做鉴权和限流？

AI 调用有真实成本和滥用风险，未保护接口可能导致额度耗尽、账单异常和服务不可用。
