Lesson 0012

API Key 鉴权：保护会修改数据的接口

读接口可以公开，但创建、更新、删除通常不能让任何人随便调用。本节用 X-API-Key 请求头和 Depends 做一个入门级 API Key 鉴权。

0. 本节学习目标

  • 理解鉴权和接口保护的基本目的。
  • 理解请求头 X-API-Key 如何携带密钥。
  • 看懂 APIKeyHeader 和 require_api_key。
  • 理解为什么写接口需要保护，读接口暂时保持公开。

1. 为什么需要 API Key

如果没有保护，任何人都可以调用：

POST /items
PATCH /items/{item_id}
DELETE /items/{item_id}

这意味着别人可以随便创建、改动或删除你的数据。对于 AI 服务也一样：如果推理接口不保护，别人可能滥用你的模型额度和 API Key。

API Key 是最简单的一类鉴权方式：客户端在请求里带一个约定好的密钥，后端检查它是否正确。

2. 请求头是什么

HTTP 请求除了 URL、方法、请求体，还有请求头。请求头适合携带元信息，例如认证凭据：

POST /items
X-API-Key: dev-secret-key
Content-Type: application/json

{
  "name": "Protected Item",
  "category": "book",
  "price": 10,
  "in_stock": true
}

FastAPI 官方 Header 文档说明，读取请求头要使用 Header 或安全工具，而不能把它当普通查询参数。

3. APIKeyHeader

api_key_header = APIKeyHeader(name="X-API-Key", auto_error=False)

APIKeyHeader 来自 FastAPI 的 security 工具。它会从请求头里提取 X-API-Key，并且能集成到 OpenAPI 文档中。

auto_error=False 表示缺少 header 时不要让工具自动返回错误，而是交给我们自己的 require_api_key 统一处理。

4. require_api_key 逐行讲解

def require_api_key(
    api_key: Annotated[str | None, Depends(api_key_header)],
    settings: Annotated[Settings, Depends(get_settings)],
) -> None:
    if api_key != settings.api_key:
        raise HTTPException(
            status_code=status.HTTP_401_UNAUTHORIZED,
            detail="Invalid or missing API key.",
        )

这里用到了两个依赖：

  • api_key_header：从请求头读取客户端传来的 key。
  • get_settings：读取服务端配置里的正确 key。

如果两者不相等，就返回 401 Unauthorized。这表示请求没有通过身份凭据检查。

5. 把鉴权挂到写接口上

@router.post(
    "",
    response_model=ItemRead,
    status_code=status.HTTP_201_CREATED,
    dependencies=[Depends(require_api_key)],
)
def create_item(...):
    ...

这里把依赖放在路径装饰器的 dependencies 里。这样 FastAPI 会执行 require_api_key，但不会把它的返回值传给路径函数。

这很适合“只需要执行检查，不需要在函数里使用结果”的场景。现在 POST、PATCH、DELETE 都被保护了，GET 仍然公开。

6. 实验：无 key 请求

POST /items
{
  "name": "No Key Item",
  "category": "book",
  "price": 5,
  "in_stock": true
}

不带 X-API-Key 时应该返回 401。

7. 实验：带 key 请求

在 /docs 里可以点击 Authorize，填写 API Key；或者在 HTTP 请求里加 header：

X-API-Key: dev-secret-key

然后再请求：

POST /items
{
  "name": "Protected Item",
  "category": "book",
  "price": 10,
  "in_stock": true
}

如果 key 正确，应该返回 201。

8. 测试也要带 key

API_KEY_HEADERS = {"X-API-Key": "dev-secret-key"}

created = client.post(
    "/items",
    headers=API_KEY_HEADERS,
    json={...},
)

第 10 课的测试现在也升级了：成功创建、更新、删除时要带正确 key；另外新增了“不带 key 创建商品返回 401”的测试。

9. 课堂练习

请你完成：

  1. 不带 X-API-Key 调用 POST /items，观察 401。
  2. 带错误 key 调用，观察 401。
  3. 带正确 key 调用，观察 201。

10. 小测：先回忆再展开

1. API Key 应该放在代码里还是配置里？

应该放在配置或环境变量里，不能把真实密钥硬编码并提交。

2. 为什么写接口比读接口更需要先保护？

因为写接口会改变数据，误用或滥用的破坏性更大。

3. 401 和 422 的区别是什么？

401 是认证失败，422 是请求数据不符合参数或模型校验。

11. 本节总结

API Key 是最小可用的接口保护方式。客户端通过请求头发送 key，后端通过 APIKeyHeader 提取，再和 Settings 中的服务端 key 比较。真实生产系统通常还会用 OAuth2、JWT、权限角色、速率限制等更完整的安全方案，但 API Key 是入门 AI 服务保护的好起点。
