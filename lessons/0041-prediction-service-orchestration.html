0041 Prediction Service Orchestration：把预测用例编排从 router 抽到 service

当 /predict 同时负责 HTTP、依赖注入、backend 选择、耗时测量、错误包装、日志和响应构造时，router 已经变成业务大杂烩。本节用 services/predictions.py 建立用例层，让 router 回到 HTTP 翻译职责。

① 本节核心知识框架

Prediction Service Orchestration
├─ Router：HTTP 协议翻译
│  ├─ request model / response model
│  ├─ HTTPException / headers / logs
│  └─ OpenAPI 文档
├─ Service：预测用例编排
│  ├─ backend 选择
│  ├─ demo/provider 调用
│  ├─ elapsed_ms 测量
│  └─ AIClientError → PredictionServiceError
└─ Adapter：provider HTTP 细节与 JSON 契约

核心结论：分层不是为了文件数量，而是为了让变化归位。HTTP 变化改 router，预测业务编排改 service，provider 传输细节改 adapter。

② 核心概念底层原理

用例层 service 表达“系统要完成什么业务动作”。对本项目来说，预测用例包括选择 backend、调用对应实现、测量耗时、把底层 AI 错误包装成带上下文的 service 错误。这些逻辑不属于 HTTP 协议本身。

层允许知道不应该知道
Router状态码、headers、请求/响应模型、FastAPI 依赖provider 重试细节、HTTPX 异常分类
Servicebackend、AIClient、provider client、耗时、业务错误HTTPException、OpenAPI、响应头命名
Adapterprovider URL、payload、响应 schema、HTTP 异常FastAPI 路由、前端响应体

③ 全套代码逐行精读解析

PredictionBackend = Literal["demo", "provider"]

后端类型被显式收窄。service 内部不接受任意字符串，避免未知 backend 进入业务分支。

@dataclass(frozen=True)
class PredictionServiceResult:
    prediction: PredictionResult
    backend: PredictionBackend
    elapsed_ms: float

frozen=True 表示结果对象不可变，降低跨层传递时被误改的风险。它携带的不只是预测值，还携带 router 设置 header 和日志所需的 backend、耗时。

class PredictionServiceError(Exception):
    def __init__(self, message: str, error_code: str, backend: PredictionBackend, elapsed_ms: float) -> None:
        super().__init__(message)
        self.message = message
        self.error_code = error_code
        self.backend = backend
        self.elapsed_ms = elapsed_ms

失败路径也必须携带上下文。若只抛普通 AIClientError，router 不一定知道本次走的是哪个 backend、已经耗时多少，失败 header 和日志就会缺证据。

async def run_prediction(
    text: str,
    mode: str,
    settings: Settings,
    ai_client: AIClient,
    provider_http_client: httpx.AsyncClient,
    backend: PredictionBackend | None = None,
) -> PredictionServiceResult:

函数签名完整声明用例依赖：输入文本、预测模式、配置、demo client、provider HTTP client、可选强制 backend。它不依赖 FastAPI 的 Request 或 Response，因此可以被单元测试直接调用。

active_backend = backend or settings.ai_prediction_backend
started_at = perf_counter()

backend 参数优先级高于 settings，用于 /provider/predict 强制 provider；perf_counter() 测量用例级耗时，而不是只测 router 开销。

try:
    if active_backend == "provider":
        config = AIClientConfig(...)
        prediction = await request_provider_prediction(...)
    else:
        prediction = ai_client.predict_sentiment(text, mode)
except AIClientError as exc:
    elapsed_ms = (perf_counter() - started_at) * 1000
    raise PredictionServiceError(...) from exc

service 捕获 adapter/client 统一错误，并包装成 router 可使用的错误对象。注意 service 不抛 HTTPException，因为那是 HTTP 层职责。

④ 核心机制运行全流程拆解

router.predict()
  → run_prediction(...)
      → active_backend = settings 或强制参数
      → started_at = perf_counter()
      → demo 分支或 provider 分支
      → 成功：PredictionServiceResult
      → 失败：PredictionServiceError
  → router 把 result/error 翻译为 HTTP headers、logs、body

重构前，router 关心“预测怎么跑”；重构后，router 只关心“service 结果怎么变成 HTTP 响应”。这就是薄 router、深 service 的核心。

⑤ 重难点、易混淆点对比辨析

问题正确边界判断标准
service 能不能抛 HTTPException不建议service 应保持框架无关
router 能不能测耗时可测 HTTP 总耗时，但业务耗时更适合 service当前 headers 需要预测用例耗时
dataclass vs dict内部契约优先 dataclass字段清晰、类型明确、IDE 可提示
单测 service vs 测 HTTP两者都要service 测业务编排，HTTP 测 FastAPI 翻译

⑥ 开发常见报错+坑点+解决方案

  • service 抛 HTTPException：业务层被 FastAPI 污染。解决：service 抛内部异常，router 负责 HTTP 翻译。
  • 错误包装丢失 backend：失败 header 只能写 unknown。解决：PredictionServiceError 必须携带 backend 与 elapsed。
  • 只测路由不测 service：失败时难定位是依赖注入问题还是业务分支问题。解决：增加 tests/test_prediction_service.py。
  • service 偷偷读取全局 settings：测试难控制。解决：settings 作为参数传入，让依赖边界显式化。

⑧ 课后记忆习题+标准答案

1. router 和 service 的职责分别是什么？

router 负责 HTTP 协议翻译；service 负责预测用例编排，包括 backend 选择、调用实现、耗时和错误包装。

2. 为什么 PredictionServiceError 要携带 elapsed_ms？

失败路径也要给 router 提供耗时证据，用于响应头、日志和排查慢失败。

3. 为什么 service 不直接读取 get_settings()？

显式传入 settings 更容易测试，也让依赖边界清楚，不把全局状态藏进业务函数。

4. 为什么 HTTP 测试仍然必要？

service 测业务编排，HTTP 测 FastAPI 依赖注入、状态码、headers、响应模型和异常翻译。

⑨ 面试高频真题+解析

题 1：什么是薄 router、深 service？ 答：router 只处理协议层输入输出，业务用例、分支选择、错误包装放在 service。这样 HTTP 变化与业务变化不会互相污染。

题 2：如何判断一段逻辑该放 service？ 答：如果它描述“业务怎么完成”，且不依赖 HTTP 语义，就应放 service；如果它描述“请求怎么进来、响应怎么出去”，就放 router。

题 3：这次重构对测试有什么收益？ 答：可以绕过 TestClient 直接测 run_prediction() 的 demo/provider 分支和错误包装，定位更快，HTTP 测试只保留协议翻译断言。
