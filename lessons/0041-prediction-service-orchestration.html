Lesson 0041

Prediction Service Orchestration：把路由里的业务编排抽到 service 层

第 40 课给预测接口加了可观测性，但也让 predictions.py 变胖了：它既要处理 HTTP，又要选择 demo/provider 后端，还要测耗时和包装错误。本节做一次架构收束：把预测业务编排抽到 services/predictions.py，让 router 重新专注 HTTP 翻译。

0. 本节学习目标

  • 理解“瘦 router，深 service”的模块边界。
  • 学会用 service 函数承接后端选择、耗时测量和错误包装。
  • 理解为什么 service error 可以携带 backend、elapsed_ms、error_code。
  • 学会用 dataclass 表达 service 层返回结果。
  • 学会为 service 层直接写单元测试，而不总是通过 HTTP 测试。

1. 背景：上一课之后，router 开始变胖

上一课的 /predict 做了很多事：

  • 读取 PredictionRequest。
  • 读取 settings。
  • 选择 demo 或 provider 后端。
  • 构造 AIClientConfig。
  • 调用 demo client 或 provider adapter。
  • 测量耗时。
  • 把错误转成 503。
  • 设置 headers。
  • 写日志。
  • 返回 PredictionResponse。

这些代码都能运行，但职责已经混在一起。真实项目里，如果 router 继续变胖，后面加缓存、重试、限流、审计、A/B 实验时，路由函数会越来越难读。

所以本节做一个关键调整：把“预测业务怎么跑”交给 service；router 只负责“HTTP 请求怎么进来，HTTP 响应怎么出去”。

2. 核心概念：router 是协议层，service 是用例层

可以把后端分成两层看：

Router

处理 HTTP 相关的事情：路径、请求体、依赖注入、状态码、headers、OpenAPI、HTTPException。

Service

处理用例相关的事情：选择后端、调用 adapter、测量耗时、统一内部成功结果和内部错误结果。

本节重构后的链路是：

router receives HTTP
  -> service runs prediction use case
  -> router maps service result to HTTP response
  -> router maps service error to HTTP 503

这个边界不是为了“多建文件显得高级”，而是为了让变化有地方放。以后预测业务复杂了，优先长在 service 里，而不是把 router 写成大杂烩。

3. 本节项目改动概览

  • 新增 first_api/services/predictions.py。
  • 新增 PredictionServiceResult，表达 service 成功结果。
  • 新增 PredictionServiceError，表达 service 失败结果。
  • 新增 run_prediction()，承接 demo/provider 后端选择和耗时测量。
  • 重构 routers/predictions.py，让 /predict 和 /provider/predict 调用 service。
  • 新增 tests/test_prediction_service.py，直接测试 service 层。

4. 代码精读一：service 成功结果

@dataclass(frozen=True)
class PredictionServiceResult:
    prediction: PredictionResult
    backend: PredictionBackend
    elapsed_ms: float

这个 dataclass 表示一次预测用例成功完成后，service 要交给 router 的信息。

  • prediction 是内部稳定预测结果。
  • backend 告诉 router 本次走了 demo 还是 provider。
  • elapsed_ms 告诉 router 本次业务耗时。

为什么不用普通 dict？因为 dataclass 有明确字段、类型提示和构造方法。它比随手传字典更适合作为 service 层和 router 层之间的内部契约。

5. 代码精读二：service 失败结果

class PredictionServiceError(Exception):
    def __init__(
        self,
        message: str,
        error_code: str,
        backend: PredictionBackend,
        elapsed_ms: float,
    ) -> None:
        super().__init__(message)
        self.message = message
        self.error_code = error_code
        self.backend = backend
        self.elapsed_ms = elapsed_ms

失败路径也需要携带上下文。上一课我们已经知道，失败时最需要 backend、elapsed_ms 和 error_code。问题是这些信息不能丢在 service 里，也不能让 router 自己重新猜。

所以 service 捕获底层 AIClientError 后，包装成 PredictionServiceError。router 拿到它以后，可以稳定地生成 HTTP 503、响应头和日志。

6. 代码精读三：run_prediction 编排后端选择

async def run_prediction(
    text: str,
    mode: str,
    settings: Settings,
    ai_client: AIClient,
    provider_http_client: httpx.AsyncClient,
    backend: PredictionBackend | None = None,
) -> PredictionServiceResult:
    active_backend = backend or settings.ai_prediction_backend
    started_at = perf_counter()

    try:
        if active_backend == "provider":
            config = AIClientConfig(...)
            prediction = await request_provider_prediction(...)
        else:
            prediction = ai_client.predict_sentiment(text, mode)
    except AIClientError as exc:
        elapsed_ms = (perf_counter() - started_at) * 1000
        raise PredictionServiceError(...) from exc

    elapsed_ms = (perf_counter() - started_at) * 1000
    return PredictionServiceResult(
        prediction=prediction,
        backend=active_backend,
        elapsed_ms=elapsed_ms,
    )

这就是本节的核心函数。它把之前散在 router 里的“预测怎么跑”集中起来：

  • 决定 active backend。
  • 记录开始时间。
  • 根据 backend 调用 demo client 或 provider adapter。
  • 成功时返回带耗时的 service result。
  • 失败时抛出带耗时的 service error。

backend 参数是可选的。普通 /predict 使用 settings 决定后端；/provider/predict 可以传入 backend="provider"，强制走 provider 分支。

7. 代码精读四：router 变回 HTTP 翻译层

try:
    result = await run_prediction(
        text=payload.text,
        mode=payload.mode,
        settings=settings,
        ai_client=ai_client,
        provider_http_client=provider_http_client,
    )
except PredictionServiceError as exc:
    log_prediction_failed(exc)
    raise HTTPException(
        status_code=503,
        detail={
            "error_code": exc.error_code,
            "message": exc.message,
        },
        headers=build_prediction_headers(
            backend=exc.backend,
            elapsed_ms=exc.elapsed_ms,
            error_code=exc.error_code,
        ),
    ) from exc

现在 router 不再关心 demo/provider 分支怎么执行。它只关心：

  • 调用 service。
  • service 成功时设置 headers、写成功日志、返回响应模型。
  • service 失败时设置错误 headers、写失败日志、抛出 HTTPException。

这就是“router 负责 HTTP 翻译”的意思。

8. 测试精读一：直接测 service 的 demo 分支

def test_run_prediction_uses_demo_backend() -> None:
    async def run() -> object:
        provider_client = httpx.AsyncClient(
            transport=httpx.MockTransport(lambda request: httpx.Response(500))
        )
        try:
            return await run_prediction(
                text="FastAPI is great and pleasant.",
                mode="fast",
                settings=Settings(ai_prediction_backend="demo"),
                ai_client=DemoAIClient(config=AIClientConfig(provider="demo")),
                provider_http_client=provider_client,
            )
        finally:
            await provider_client.aclose()

    result = asyncio.run(run())

    assert result.backend == "demo"
    assert result.prediction.source == "demo-ai-client"
    assert result.elapsed_ms >= 0

这条测试不经过 HTTP，也不创建 TestClient。它直接测试 service 用例。这种测试更快、更聚焦，失败时也更容易定位到业务编排层。

9. 测试精读二：service 错误必须保留上下文

def test_run_prediction_wraps_provider_errors_with_backend_and_elapsed_time() -> None:
    async def run() -> None:
        def handler(request: httpx.Request) -> httpx.Response:
            return httpx.Response(status_code=503, request=request)

        provider_client = httpx.AsyncClient(
            transport=httpx.MockTransport(handler)
        )
        try:
            await run_prediction(
                text="FastAPI is great and pleasant.",
                mode="fast",
                settings=Settings(ai_prediction_backend="provider"),
                ai_client=DemoAIClient(config=AIClientConfig(provider="demo")),
                provider_http_client=provider_client,
            )
        finally:
            await provider_client.aclose()

    try:
        asyncio.run(run())
    except PredictionServiceError as exc:
        error = exc

    assert error.backend == "provider"
    assert error.error_code == "ai_provider_http_error"
    assert error.elapsed_ms >= 0

这个测试保护了本节最重要的失败语义：provider 失败以后，router 仍然能知道 backend、error_code 和 elapsed_ms。

11. 常见错误与诊断

错误一：router 什么都做

router 变胖后，业务变化和 HTTP 变化混在一起。优先把用例编排抽到 service。

错误二：service 抛 HTTPException

HTTPException 属于 HTTP 层。service 更适合抛内部错误，由 router 翻译成 HTTP。

错误三：错误包装丢失上下文

如果只保留 message，router 就无法设置 backend、elapsed 和 error-code headers。

错误四：只写 HTTP 测试

HTTP 测试很重要，但 service 层单元测试能更快定位业务编排问题。

12. 面试视角：怎样解释这次重构

我把预测后端选择和耗时测量从 FastAPI router 抽到了 service 层。service 返回 PredictionServiceResult 或抛 PredictionServiceError，里面包含 prediction、backend、elapsed_ms 和 error_code。router 不再关心 demo/provider 分支细节，只负责把 service 结果翻译成 HTTP response、headers、logs 或 HTTPException。这样路由更薄，业务编排更容易测试。

13. 小测：先回忆，再展开

1. router 和 service 的职责分别是什么？

router 负责 HTTP 协议翻译，service 负责业务用例编排，例如选择后端、调用 adapter、测量耗时和包装内部错误。

2. 为什么 service 不直接抛 HTTPException？

HTTPException 属于 HTTP 层；service 应该保持框架无关，抛内部错误，再由 router 转成 HTTP 响应。

3. PredictionServiceError 为什么要携带 elapsed_ms？

失败路径同样需要耗时证据，router 要用它设置响应头和日志，帮助定位慢失败或 provider 超时。

4. 为什么还要保留 HTTP 测试？

service 测试验证业务编排，HTTP 测试验证 FastAPI 依赖、headers、状态码和响应 JSON 翻译。

14. 本节总结

本节把预测业务编排从 router 抽到 service 层。现在 run_prediction() 负责选择后端、调用 demo 或 provider、测量耗时和包装错误；router 负责把 service 结果转换成 HTTP 响应、headers 和日志。这个结构更接近真实后端项目，也为下一步加入缓存、重试、限流或更复杂 provider 策略留下了清晰位置。
