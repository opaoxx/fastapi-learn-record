0060 Provider Metrics Naming Refactor Boundary：指标改名是迁移，不是普通重构

本节回答一个真实工程问题：如果未来必须修改 metric name 或 label names，能不能直接改字符串？通常不能。metrics 命名被外部查询、告警和文档依赖，因此改名属于兼容性迁移。课程通过 get_provider_prediction_metric_contract() 把 rename-sensitive 字段集中暴露并测试保护。

② 核心概念底层原理

内部变量重命名只要不改变行为，通常是重构。metrics 名称重命名不同：metric name 与 label names 进入时间序列身份，会被外部系统长期引用。改名会产生新时间序列，旧查询不会自动迁移。

因此 metrics rename 的正确心智是“兼容迁移”：先盘点契约字段，再设计并行期，再迁移 dashboard、alert、文档、runbook，最后下线旧名。没有迁移窗口的直接改名会让监控突然失明。

③ 全套代码逐行精读解析

PROVIDER_PREDICTION_METRIC_LABEL_NAMES = (
    "operation",
    "outcome",
    "failure_category",
    "error_code",
    "status_code",
)

将 label names 提取为常量，不只是减少重复，而是把“哪些 label 属于公共契约”显式化。未来改 failure_category 或新增 label 时，review 能看到这是 contract 变化。

def get_provider_prediction_metric_contract() -> dict[str, object]:
    return {
        "metric_name": PROVIDER_PREDICTION_METRIC_NAME,
        "help": PROVIDER_PREDICTION_METRIC_HELP,
        "label_names": list(PROVIDER_PREDICTION_METRIC_LABEL_NAMES),
    }
字段含义为什么 rename-sensitive
metric_name指标主名称查询、dashboard、告警直接引用
help人类说明误导性说明会改变使用者理解
label_names标签维度列表改变时间序列身份和查询过滤条件
list(...)返回普通列表更像 JSON/文档快照，也避免暴露 tuple 对象本身
 
def test_provider_metric_contract_collects_rename_sensitive_fields() -> None:
    contract = get_provider_prediction_metric_contract()

    assert contract == {
        "metric_name": "provider_prediction_total",
        "help": "Provider prediction outcomes recorded by the teaching metrics counter.",
        "label_names": list(PROVIDER_PREDICTION_METRIC_LABEL_NAMES),
    }
    assert contract["label_names"] == [
        "operation",
        "outcome",
        "failure_category",
        "error_code",
        "status_code",
    ]

测试集中保护“改名前必须审查”的字段。若未来有人调整 metric name 或 label_names，测试失败点会把这次修改从普通字符串变化提升为契约变更讨论。

④ 核心机制运行全流程拆解

开发者准备改名
  → 查看 get_provider_prediction_metric_contract()
  → 识别 metric_name/help/label_names 影响面
  → 搜索 dashboard / alert / docs / tests / runbook 依赖
  → 决定是否需要双写或兼容窗口
  → 更新代码与文档
  → 更新合同测试
  → 验证新旧消费者迁移完成
  → 下线旧名

本课程没有真的做双写，因为当前没有真实外部消费者。强行加入旧指标别名会增加噪声。本节只建立迁移审查面：如果以后真的要改名，知道从哪里开始盘点。

⑧ 课后记忆习题+标准答案

1. 为什么 metrics rename 是迁移而不是普通重构？

因为 metric name 和 label names 被外部查询、dashboard、alert 和文档依赖。直接改名会破坏外部消费者。

2. contract helper 收集哪些字段？

收集 metric_name、help、label_names，它们都是重命名前必须审查的字段。

3. 安全改名为什么需要兼容窗口？

兼容窗口允许旧查询继续工作，同时给 dashboard、alert、runbook 和使用者迁移到新名称的时间。

4. 为什么本节不真的做旧指标别名？

当前没有真实外部消费者。加入别名会增加课程复杂度。本节目标是建立审查边界，而不是制造不存在的迁移压力。

⑨ 面试高频真题+解析

面试题 1：如何安全修改一个生产 metrics 名称？

先盘点 metric name、labels、dashboard、alert、文档依赖；新增新名称并与旧名称并行导出；迁移消费者；验证旧名称无关键依赖；最后按计划下线旧名称。

面试题 2：为什么 label name 改动也危险？

label name 是时间序列身份的一部分，也是查询过滤条件。改名会导致旧查询无法匹配，新旧序列割裂。

面试题 3：contract snapshot helper 的价值是什么？

它把 rename-sensitive 字段集中暴露，让测试和 review 能明确看到哪些字段属于外部观测契约，避免无意识破坏。
