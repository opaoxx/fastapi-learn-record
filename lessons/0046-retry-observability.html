Lesson 0046

Retry Observability：让 provider 重试过程可被看见

上一课我们已经会计算更稳妥的重试等待时间。本节继续向生产级 AI 服务靠近：当 provider 超时、限流或临时 5xx 时，我们不仅要重试，还要把“为什么重试、这是第几次、等多久、等待来自哪里”记录成结构化日志。

0. 本节学习目标

  • 理解 observability 和普通 print 调试的区别。
  • 理解为什么 retry 行为必须记录 attempt、reason、delay 等上下文。
  • 掌握 Python logging 的 extra 如何把结构化字段放进 LogRecord。
  • 学会用 pytest 的 caplog 捕获日志并断言日志字段。
  • 知道为什么本节只改内部日志，不改外部 JSON 响应契约。

1. 背景：会重试还不够，还要知道重试发生了什么

在真实后端和 AI 服务项目里，外部 provider 的失败通常不是一个简单的“报错”。它可能是短暂网络抖动、连接超时、限流、模型服务滚动发布、区域性故障，也可能是客户端请求写错导致的 400。前几节课已经让我们能区分可重试错误和不可重试错误，并能用 backoff、Retry-After、jitter 控制等待时间。

但是，如果线上用户说“预测接口偶尔很慢”，只知道最终返回 200 或 503 还不够。我们需要回答更细的问题：这次请求是不是经历了重试？第几次失败后重试？失败原因是 timeout 还是 HTTP 429？等待时间来自 provider 的 Retry-After，还是来自客户端本地 backoff？最终 sleep 了多久？

这就是本节要补上的知识：重试策略负责让系统更稳，重试可观测性负责让系统可解释。

2. 核心概念：Observability 不是“多打几行日志”

Observability 可以先理解为：系统运行之后，我们能不能从外部留下的信号推断内部发生了什么。常见信号包括 logs、metrics、traces。本课程现在还不引入完整监控平台，先从最基础也最常用的 logs 做起。

初学者容易把日志理解成 print("retrying")。这在本地调试时有一点用，但生产排查时很弱，因为它缺少结构：无法稳定筛选“只看 provider_http 模块”、无法统计“429 重试次数”、无法聚合“Retry-After 命中比例”。

更好的方式是：日志消息保持稳定，关键上下文放在结构化字段里。

message: provider_prediction_retry_scheduled
fields:
  attempt: 1
  max_attempts: 2
  retry_reason: http_status
  status_code: 429
  retry_after: 2
  delay_seconds: 2.0
  delay_source: retry_after

Python 官方 logging 文档说明，extra 参数可以把自定义字典写入 LogRecord 的属性中；pytest 官方文档说明，caplog.records 可以读取测试期间捕获到的 LogRecord。这正好让我们既能写结构化日志，也能测试这些字段。

3. 为什么不把这些信息直接塞进响应体

你可能会问：既然这些信息有用，为什么不在 POST /predict 的 JSON 响应里返回 attempts、retry_delay 呢？原因是它们属于内部诊断事实，不一定属于公共 API 契约。

公共响应体是给调用者稳定依赖的，例如 label、score、source。如果把内部 retry 细节暴露出去，以后改重试策略、换 provider、换队列或换 SDK 时，前端和外部调用方可能都被迫跟着改。

所以本节采用更稳的边界：外部 JSON 契约不变，内部日志增强。这样既方便排查，又不会扩大 API 表面。

4. 运行机制：一次 provider 重试日志如何产生

现在 provider 预测调用的关键路径可以拆成 7 步：

  1. request_provider_prediction() 发起 provider HTTP POST。
  2. HTTPX 遇到 timeout、request error，或者 response.raise_for_status() 抛出 HTTPStatusError。
  3. adapter 判断当前 attempt 是否还小于 max_attempts。
  4. 如果是 HTTP 状态错误，还要判断状态码是否属于可重试集合，例如 429、500、502、503、504。
  5. build_retry_delay_plan() 计算等待计划：优先使用有效 Retry-After，否则使用本地 backoff 加 jitter。
  6. wait_before_retry() 在 sleep 前写入结构化日志。
  7. 等待结束后进入下一轮 attempt。
provider POST
  -> failed
  -> retryable?
  -> build retry delay plan
  -> log provider_prediction_retry_scheduled
  -> await sleep(delay)
  -> next attempt

5. 代码精读一：把等待结果建模为 RetryDelayPlan

之前 calculate_retry_delay() 只返回一个数字。数字能告诉我们“等多久”，但不能告诉我们“这个等待来自哪里”。本节新增了一个小的 dataclass：

RetryDelaySource = Literal["retry_after", "local_backoff"]

@dataclass(frozen=True)
class RetryDelayPlan:
    delay_seconds: float
    source: RetryDelaySource

这段代码有两个关键点：

  • delay_seconds 是最终等待秒数。
  • source 是等待来源，只允许 retry_after 或 local_backoff 两种值。

这比直接返回 tuple 更清楚。初学者读到 RetryDelayPlan(delay_seconds=2.0, source="retry_after") 时，不需要猜第一个值、第二个值分别是什么意思。

6. 代码精读二：保留旧函数，新增更丰富的内部函数

为了不破坏已有测试和已有调用，我们保留 calculate_retry_delay() 的返回值仍然是 float。它内部委托给新的 build_retry_delay_plan()：

def calculate_retry_delay(... ) -> float:
    return build_retry_delay_plan(...).delay_seconds

这种设计非常适合课程项目：外层旧接口保持简单，内部新接口承载更多语义。以后如果别的模块只关心“等多久”，继续调用 calculate_retry_delay()；如果日志或监控需要知道“为什么等这么久”，就使用 build_retry_delay_plan()。

def build_retry_delay_plan(... ) -> RetryDelayPlan:
    provider_delay = parse_retry_after_delay(retry_after, now=now)
    if provider_delay is not None:
        return RetryDelayPlan(
            delay_seconds=min(provider_delay, max_delay),
            source="retry_after",
        )

    exponential_delay = base_delay * (2 ** (attempt - 1))
    jitter_delay = calculate_retry_jitter(config, random_fraction=random_fraction)
    return RetryDelayPlan(
        delay_seconds=min(exponential_delay + jitter_delay, max_delay),
        source="local_backoff",
    )

7. 代码精读三：在 sleep 前记录结构化日志

日志写在 wait_before_retry() 里，因为这是“等待即将发生”的统一位置。无论失败原因是 timeout、request error，还是 HTTP 429/503，只要真的准备重试，都会走到这里。

logger.info(
    "provider_prediction_retry_scheduled",
    extra={
        "attempt": attempt,
        "max_attempts": config.max_attempts,
        "retry_reason": retry_reason,
        "status_code": status_code,
        "retry_after": retry_after,
        "delay_seconds": delay_plan.delay_seconds,
        "delay_source": delay_plan.source,
    },
)

这里的 message 是稳定事件名：provider_prediction_retry_scheduled。字段才是变化的上下文。这样以后无论日志系统是终端、文件、云日志平台，还是测试里的 caplog，都能围绕同一个事件名做筛选。

注意：日志发生在 await sleep() 之前。这样如果程序在等待期间被取消，或者服务进程被重启，至少日志里已经留下“本来准备等多久”的事实。

8. 代码精读四：不同失败路径传入不同 retry_reason

provider adapter 现在会把失败原因压缩成几个稳定字符串：

retry_reason="timeout"
retry_reason="http_status"
retry_reason="request_error"

HTTP 状态错误还会额外传入 status_code 和 Retry-After：

await wait_before_retry(
    attempt,
    config,
    sleep,
    retry_reason="http_status",
    retry_after=exc.response.headers.get("Retry-After"),
    status_code=status_code,
    random_fraction=random_fraction,
)

这就是结构化日志的价值：不是把异常对象整段打印出去，而是把排查最需要的字段挑出来，保持字段名稳定、含义稳定。

在项目根目录运行：

retry_records = [
    record

    if record.getMessage() == "provider_prediction_retry_scheduled"

这里不是只检查日志文本，而是检查 LogRecord 上的字段：

10. 常见错误

  • 只写 print()：本地能看见，测试和生产日志系统很难稳定捕获。
  • 日志消息拼接太多变量：人能读，但机器筛选和聚合困难。
  • 在日志里写入请求全文或密钥：可观测性不能牺牲隐私和安全。
  • 只记录最终失败，不记录中间重试：慢请求的真正原因会被隐藏。
  • 把诊断字段加入公共响应体：容易让内部实现细节变成外部契约。
  • 测试只断言有日志文本：没有验证 attempt、delay、reason 等真正关键的诊断字段。

11. 面试视角：如何回答“你们怎么排查 provider 不稳定”

一个比较成熟的回答可以这样组织：

我会把 provider 调用封装在 adapter 或 service 边界里，在那里统一处理 timeout、HTTP 状态错误、request error、非法响应和 retry。对于 retry，我不会只做等待，还会记录结构化日志，例如 attempt、max_attempts、retry_reason、status_code、Retry-After、delay_seconds 和 delay_source。这样线上排查时可以区分是 provider 限流、临时 5xx、网络问题，还是客户端本地 backoff 导致的等待。

这段回答背后的核心不是“我会打日志”，而是“我知道日志应该放在哪里、记录哪些稳定字段、如何避免污染公共 API 契约”。

12. 记忆练习

1. 为什么 retry observability 不只是打印 retrying？

因为生产排查需要稳定事件名和结构化字段，才能筛选、聚合、测试和解释失败路径。

2. delay_source 为什么有价值？

它能说明等待来自 provider 的 Retry-After，还是客户端本地 backoff 计算，排查含义完全不同。

3. 为什么日志写在 sleep 前更合适？

因为 sleep 前已经确定重试计划，即使等待期间程序被取消，也留下了准备重试的诊断事实。

4. 为什么本节不修改 PredictionResponse？

因为 retry 细节是内部诊断事实，不应轻易变成外部调用方依赖的公共 JSON 契约。

13. 本节总结

本节把 provider retry 从“会自动重试”升级成“重试过程可解释”。我们新增了 RetryDelayPlan 来保存等待秒数和等待来源，并在真正 sleep 前写入结构化日志。测试使用 caplog 断言日志字段，证明可观测性也可以成为自动化测试的一部分。
