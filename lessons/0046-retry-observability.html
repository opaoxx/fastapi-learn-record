0046 Retry Observability：让 provider 重试过程可解释

重试策略不能只“悄悄执行”。当线上接口变慢或 provider 限流时，我们需要知道第几次失败、失败原因、等待多久、等待来源是 Retry-After 还是本地 backoff。本节把 retry 中间事件写成结构化日志，并用 caplog 测试。

① 本节核心知识框架

Retry Observability
├─ 事件名：provider_prediction_retry_scheduled
├─ 等待计划：RetryDelayPlan(delay_seconds, source)
├─ 关键字段：attempt / max_attempts / retry_reason
├─ HTTP 字段：status_code / retry_after
├─ 等待字段：delay_seconds / delay_source
├─ 写入位置：sleep 之前
└─ 测试方式：pytest caplog.records 断言 LogRecord 字段

核心结论：可观测性不是多打印，而是用稳定事件名和稳定字段表达系统内部发生的关键决策。

② 核心概念底层原理

retry 过程有两类事实：一类是“最终成功/失败”，另一类是“中间发生过什么”。如果只记录最终 503，就无法解释请求为什么慢；如果记录 retry scheduled，就能知道慢来自限流、超时还是本地等待。

字段含义排查价值
attempt第几次尝试失败后准备重试判断失败发生在早期还是接近耗尽
retry_reasontimeout、http_status、request_error区分网络、限流、上游错误
delay_sourceretry_after 或 local_backoff判断等待来自 provider 还是客户端策略
delay_seconds最终等待秒数定位接口变慢原因

③ 全套代码逐行精读解析

RetryDelaySource = Literal["retry_after", "local_backoff"]

@dataclass(frozen=True)
class RetryDelayPlan:
    delay_seconds: float
    source: RetryDelaySource

单纯返回 float 只能表示“等多久”；RetryDelayPlan 同时表达“等多久”和“为什么是这个等待来源”。Literal 让来源值稳定可枚举。

delay_plan = build_retry_delay_plan(
    attempt,
    config,
    retry_after=retry_after,
    random_fraction=random_fraction,
)

等待计划统一由一个函数构造，避免日志、sleep、测试各自计算一遍导致不一致。

logger.info(
    "provider_prediction_retry_scheduled",
    extra={
        "attempt": attempt,
        "max_attempts": config.max_attempts,
        "retry_reason": retry_reason,
        "status_code": status_code,
        "retry_after": retry_after,
        "delay_seconds": delay_plan.delay_seconds,
        "delay_source": delay_plan.source,
    },
)

事件名固定，字段结构固定。日志系统可以按 message 聚合，也可以按 extra 字段过滤。此处不记录用户输入文本，避免隐私进入日志。

if delay_plan.delay_seconds > 0:
    await sleep(delay_plan.delay_seconds)

日志写在 sleep 之前。即使等待期间任务被取消，日志也已经记录“系统准备等待多久再重试”的事实。

retry_records = [
    record
    for record in caplog.records
    if record.getMessage() == "provider_prediction_retry_scheduled"
]

caplog 捕获的是 LogRecord，不是裸字符串。测试可以直接断言 record.delay_source、record.status_code 等结构化字段。

④ 核心机制运行全流程拆解

provider 返回 429 + Retry-After: 2
  → HTTPStatusError
  → wait_before_retry(retry_reason="http_status", status_code=429, retry_after="2")
  → build_retry_delay_plan()
      → RetryDelayPlan(delay_seconds=2.0, source="retry_after")
  → logger.info("provider_prediction_retry_scheduled", extra=...)
  → fake/real sleep(2.0)
  → 下一次 attempt

这条链路把 provider 信号、客户端决策、日志字段和实际等待统一起来。测试若断言 sleep_calls == [2.0] 且日志 delay_source == "retry_after"，就能证明机制完整闭环。

⑤ 重难点、易混淆点对比辨析

易混点正确理解隐患
日志文本 vs 结构化字段message 稳定，细节放 extra拼接文本难检索和聚合
scheduled vs exhaustedscheduled 是准备再试，exhausted 是没机会再试混淆会误导排查
delay_source说明等待来自哪里没有来源就无法判断限流还是本地策略
日志写入时机sleep 前写任务取消时丢失重试计划证据

⑥ 开发常见报错+坑点+解决方案

  • caplog 抓不到日志：logger 名称或级别不对。解决：caplog.set_level(logging.INFO, logger="first_api.services.provider_http")。
  • 日志只有 retrying：没有 attempt/reason/delay 字段。解决：使用 extra 记录结构化上下文。
  • 日志记录原始 text：AI 输入可能敏感。解决：只记录元信息，不记录完整请求文本。
  • 日志字段命名漂移：不同分支字段不一致。解决：所有 retry scheduled 都走 wait_before_retry()。

⑧ 课后记忆习题+标准答案

1. 为什么 retry 日志不能只写 retrying？

生产排查需要 attempt、reason、delay、source 等结构化字段，否则无法过滤、聚合和解释慢请求。

2. delay_source 的价值是什么？

它区分等待来自 provider 的 Retry-After，还是客户端本地 backoff。

3. 为什么日志写在 sleep 之前？

这样即使等待期间任务取消，也保留了准备重试和等待多久的证据。

4. 为什么不把 retry 细节加进 PredictionResponse？

retry 细节是内部诊断事实，不应轻易变成外部 API 契约。

⑨ 面试高频真题+解析

题 1：如何排查 provider 调用偶发变慢？ 答：检查 retry scheduled 日志，查看 attempt、retry_reason、status_code、retry_after、delay_seconds、delay_source，判断慢来自限流、上游 5xx、超时还是本地 backoff。

题 2：结构化日志比普通字符串日志强在哪里？ 答：字段稳定可检索、可聚合、可测试，也方便接入日志平台和告警。

题 3：为什么可观测性也要写测试？ 答：日志字段是线上排查契约。测试能防止重构时丢失关键诊断字段。
