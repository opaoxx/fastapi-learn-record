0038 Provider Predict Endpoint：把异步 provider adapter 接入 FastAPI

本节把 request_provider_prediction() 接入 POST /provider/predict。重点不是新增一个 URL，而是掌握“渐进式接入外部依赖”：旧 /predict 保持稳定，新端点单独验证 provider 链路，返回体仍然使用项目公共 PredictionResponse。

① 本节核心知识框架

POST /provider/predict
├─ 请求体：PredictionRequest
├─ 依赖：Settings, AIClient, provider_http_client
├─ 编排：run_prediction(..., backend="provider")
├─ 成功：PredictionServiceResult → headers → PredictionResponse
├─ 失败：PredictionServiceError → HTTP 503 + ErrorResponse
└─ 测试：dependency_overrides + httpx.MockTransport

核心结论：公共端点只暴露本项目稳定 API 契约，provider 的 URL、HTTP client、响应校验和错误码都被限制在后端边界内。

② 核心概念底层原理

渐进式接入 是生产系统演进的基本策略。直接替换旧 /predict 会让已有前端、测试、文档和读者突然依赖外部网络；新增 /provider/predict 可以单独验证 provider 链路，同时不破坏旧 demo 行为。

概念本节体现底层意义
公共 API 契约PredictionRequest / PredictionResponse调用方稳定依赖的 JSON 形状
内部实现run_prediction(..., backend="provider")服务端可以演进、切换、替换
依赖注入Depends(get_provider_http_client)生产复用真实 client，测试注入 mock client
错误翻译PredictionServiceError → HTTPException内部异常不直接泄漏到 HTTP 调用方

③ 全套代码逐行精读解析

@router.post(
    "/provider/predict",
    response_model=PredictionResponse,
    responses={status.HTTP_503_SERVICE_UNAVAILABLE: {"model": ErrorResponse}},
    summary="Call the configured AI provider prediction endpoint",
)

response_model 约束成功响应，避免 provider 原始字段外泄；responses 把 503 错误模型写进 OpenAPI；summary 让文档读者知道该端点是 provider 链路，不是普通 demo 预测。

async def predict_with_provider(
    payload: PredictionRequest,
    response: Response,
    settings: Annotated[Settings, Depends(get_settings)],
    ai_client: Annotated[AIClient, Depends(get_ai_client)],
    provider_http_client: Annotated[httpx.AsyncClient, Depends(get_provider_http_client)],
) -> PredictionResponse:

async def 允许等待外部 HTTP I/O；payload 由 Pydantic 校验；response 用于设置诊断 header；settings 提供 URL 和重试配置；provider_http_client 来自 lifespan，避免每个请求新建连接池。

result = await run_prediction(
    text=payload.text,
    mode=payload.mode,
    settings=settings,
    ai_client=ai_client,
    provider_http_client=provider_http_client,
    backend="provider",
)

backend="provider" 强制该端点走 provider 分支。即使全局配置默认是 demo，这个教学端点也能稳定验证真实 provider 调用链。

except PredictionServiceError as exc:
    log_prediction_failed(exc)
    raise HTTPException(
        status_code=status.HTTP_503_SERVICE_UNAVAILABLE,
        detail={"error_code": exc.error_code, "message": exc.message},
        headers=build_prediction_headers(...),
    ) from exc

service 错误被转换为 HTTP 503。from exc 保留异常链，便于调试；detail 保持稳定错误体；headers 带出 backend、elapsed、error-code 诊断信息。

set_prediction_headers(...)
log_prediction_completed(result)
return build_prediction_response(result.prediction, payload.text)

成功路径先设置 headers，再记录结构化日志，最后把内部 PredictionResult 转成公共 PredictionResponse。不直接返回 result.prediction，因为公共响应还需要 text_length。

④ 核心机制运行全流程拆解

HTTP request
  → FastAPI 路由匹配 /provider/predict
  → Pydantic 校验 PredictionRequest
  → Depends 注入 settings / ai_client / provider_http_client
  → run_prediction(backend="provider")
  → request_provider_prediction()
  → provider JSON 校验为 PredictionResult
  → router 设置 X-Prediction-* headers
  → PredictionResponse JSON

失败链路：provider 超时、HTTP 错误、无效响应先在 adapter 收敛为 AIClientError，service 包装为 PredictionServiceError，router 最终翻译为 HTTP 503。这种分层保证每层只处理自己理解的错误语义。

问题正确做法原因

⑥ 开发常见报错+坑点+解决方案

  • 忘记注入 lifespan client：端点内部临时创建 AsyncClient，高并发下连接池无法复用。解决：通过 Depends(get_provider_http_client) 获取共享 client。
  • 错误响应不是文档模型：只抛 HTTPException(503)，没有稳定 error_code。解决：统一使用 ErrorResponse 形状。
  • 测试偶发失败：访问真实 provider 或 example URL。解决：使用 httpx.MockTransport 和 app.dependency_overrides。
  • 在同步函数里 await：路由写成 def 却调用异步 adapter。解决：外部 HTTP I/O 链路使用 async def。

⑧ 课后记忆习题+标准答案

1. 为什么新增 /provider/predict 而不是直接替换 /predict？

为了渐进式验证 provider 链路，避免旧端点行为、测试和本地学习体验突然依赖外部网络。

2. response_model=PredictionResponse 的价值是什么？

它固定成功响应体契约，防止 provider 原始 JSON 或内部字段直接泄漏给调用方。

3. 端点为什么需要 get_provider_http_client？

它复用应用生命周期内创建的 HTTP client，同时允许测试用依赖覆盖注入 MockTransport client。

4. provider 无效 JSON 为什么映射为 503？

客户端请求本身通过校验，失败发生在上游 provider 契约或服务质量上，因此作为依赖不可用处理。

⑨ 面试高频真题+解析

题 1：如何在不破坏旧 API 的情况下接入新 provider？ 答：新增专用验证端点或使用配置开关灰度；保持请求/响应模型稳定；通过依赖注入隔离外部 client；用 MockTransport 覆盖测试。

题 2：FastAPI 端点里为什么可以同时返回模型并设置 header？ 答：函数参数注入 Response 后可以写 header，返回值仍由 response_model 过滤和序列化，二者作用在 HTTP 响应的不同部分。

题 3：provider endpoint 的单测应该断言什么？ 答：断言状态码、响应 JSON、X-Prediction-* headers、错误码映射，并确保请求没有访问真实网络。
