Lesson 0038

Provider Predict Endpoint：把真实 provider 调用接进 FastAPI

第 37 课我们已经有了 provider prediction adapter。本节把它接进一个新的 FastAPI endpoint：POST /provider/predict。重点不是多写一个接口，而是学习如何在不破坏已有 /predict 的前提下，把异步外部 provider 调用渐进接入项目。

0. 本节学习目标

  • 理解为什么新增 /provider/predict，而不是直接替换已有 /predict。
  • 学会把 async provider adapter 接入 FastAPI async endpoint。
  • 理解 endpoint、adapter、settings、lifespan HTTP client 的职责分工。
  • 学会保持外部响应契约稳定：仍然返回 PredictionResponse。
  • 学会用依赖覆盖和 MockTransport 测试 provider 路由，不访问真实网络。

1. 背景：为什么不直接改旧的 /predict

当前项目已经有一个 POST /predict，它通过 DemoAIClient 做本地演示预测。现在我们有了真实 provider adapter，一个冲动做法是：直接把 /predict 改成调用 provider。

但这在真实项目里通常不是最稳的第一步，因为已有接口可能已经被前端、测试、文档、GitHub 读者使用。贸然替换实现，会引入几个风险：

  • 旧测试可能突然依赖外部网络。
  • 本地学习体验可能因为 provider 不可用而变差。
  • 同步 demo client 与异步 provider call 混在一个函数里，初学者更难理解。
  • 如果 provider schema 还在试验中，旧接口的稳定性会被牵连。

所以本节采用渐进式做法：保留 /predict，新增 /provider/predict。这样旧接口继续稳定，新接口专门用来学习和验证真实 provider 链路。

2. 核心概念：稳定契约与渐进接入

后端演进时要区分两个问题：

公共 API 契约

调用方看到的 URL、方法、请求体、响应体、状态码。例如 PredictionResponse 是前端和调用方能依赖的稳定 JSON。

内部实现

服务内部到底用 demo client、本地模型、外部 provider、缓存还是队列。内部实现可以演进，但不应随意震动公共契约。

本节的设计是：新 endpoint 使用真实 provider adapter，但仍返回同一个 PredictionResponse。也就是说，内部实现变真实了，外部 JSON 形状仍然稳定。

3. 运行机制：/provider/predict 的完整链路

POST /provider/predict
  -> FastAPI parses PredictionRequest
  -> Depends(get_settings)
  -> Depends(get_provider_http_client)
  -> build AIClientConfig
  -> await request_provider_prediction(...)
       -> provider POST JSON
       -> validate provider response
       -> return PredictionResult
  -> map PredictionResult to PredictionResponse
  -> return stable JSON

这条链路把之前很多课串起来了：Pydantic request body、settings、lifespan-managed HTTP client、dependency bridge、HTTPX async POST、provider adapter、错误映射、response model。

4. 本节项目改动概览

  • 新增配置 FIRST_API_AI_PROVIDER_PREDICTION_URL。
  • 新增 POST /provider/predict。
  • 复用已有 PredictionRequest 和 PredictionResponse。
  • 复用上一课的 request_provider_prediction()。
  • 新增测试：provider 成功返回时响应稳定；provider 返回非法 JSON 时映射为 503。

注意：默认配置里的 provider prediction URL 是占位地址。真实运行时需要换成兼容本课程约定的 provider endpoint；自动化测试里我们用 MockTransport 模拟 provider，所以不会访问外网。

5. 代码精读一：新增预测 URL 配置

class Settings(BaseSettings):
    ai_provider: Literal["demo"] = "demo"
    ai_provider_health_url: str = "https://example.com"
    ai_provider_prediction_url: str = "https://example.com/predict"
    ai_timeout_seconds: float = Field(default=10.0, gt=0)
    ai_max_attempts: int = Field(default=1, ge=1, le=5)

为什么 prediction URL 也放进配置？原因和 health URL 一样：请求目标是服务端基础设施配置，不应该由普通用户输入控制。这样可以避免把服务器变成“帮用户请求任意地址”的代理，也让本地、测试、生产环境能使用不同 provider。

6. 代码精读二：新的 provider predict endpoint

@router.post(
    "/provider/predict",
    response_model=PredictionResponse,
    responses={status.HTTP_503_SERVICE_UNAVAILABLE: {"model": ErrorResponse}},
)
async def predict_with_provider(
    payload: PredictionRequest,
    settings: Annotated[Settings, Depends(get_settings)],
    provider_http_client: Annotated[
        httpx.AsyncClient,
        Depends(get_provider_http_client),
    ],
) -> PredictionResponse:
    config = AIClientConfig(
        provider=settings.ai_provider,
        timeout_seconds=settings.ai_timeout_seconds,
        max_attempts=settings.ai_max_attempts,
    )

    try:
        prediction = await request_provider_prediction(
            url=settings.ai_provider_prediction_url,
            text=payload.text,
            mode=payload.mode,
            config=config,
            client=provider_http_client,
        )
    except AIClientError as exc:
        raise HTTPException(
            status_code=status.HTTP_503_SERVICE_UNAVAILABLE,
            detail={
                "error_code": exc.error_code,
                "message": exc.message,
            },
        ) from exc

    return PredictionResponse(
        label=prediction.label,
        score=prediction.score,
        source=prediction.source,
        text_length=len(payload.text),
    )

这个 endpoint 的结构很清楚：

  • payload 是客户端请求体，由 FastAPI 和 Pydantic 解析校验。
  • settings 提供 provider URL、timeout、provider 名称等配置。
  • provider_http_client 来自 lifespan-managed HTTP client。
  • request_provider_prediction() 负责真正的外部 provider 调用。
  • endpoint 负责把内部 PredictionResult 转换成公共 PredictionResponse。

这就是瘦路由的做法：endpoint 不直接解析 provider JSON，不直接处理 HTTPX 细节，只负责把请求、依赖、service 和响应契约接起来。

7. 为什么 endpoint 必须是 async def

上一课的 provider adapter 是异步函数：

prediction = await request_provider_prediction(...)

因此路由函数也应该是 async def。这不是为了让 CPU 计算更快，而是因为外部 HTTP 请求是 I/O 等待。等待 provider 响应时，事件循环可以去处理其他请求。

如果你在同步 def 路由里硬塞异步调用，会遇到“coroutine 没有 await”“事件循环嵌套”“阻塞等待”等问题。设计上更清晰的做法是：provider HTTP 调用链路用 async endpoint 承接。

8. 成功测试：provider 结果变成稳定响应

def test_provider_predict_endpoint_uses_provider_adapter() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(
            status_code=200,
            json={"label": "positive", "score": 0.873},
            request=request,
        )

    replacement_client = httpx.AsyncClient(
        transport=httpx.MockTransport(handler),
        trust_env=False,
    )
    app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

    try:
        with TestClient(app) as client:
            response = client.post(
                "/provider/predict",
                json={
                    "text": "FastAPI is great and pleasant.",
                    "mode": "careful",
                    "temperature": 0.2,
                },
            )
    finally:
        app.dependency_overrides.clear()
        asyncio.run(replacement_client.aclose())

    assert response.status_code == 200
    assert response.json() == {
        "label": "positive",
        "score": 0.87,
        "source": "demo-provider",
        "text_length": 30,
    }

这个测试说明：provider 返回 label/score，adapter 转成 PredictionResult，endpoint 再转成 PredictionResponse。调用方完全不需要知道 provider 原始 JSON 来自哪里。

9. 失败测试：非法 provider JSON 变成 503

def test_provider_predict_endpoint_maps_provider_errors_to_503() -> None:
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(
            status_code=200,
            json={"label": "confused", "score": 1.5},
            request=request,
        )

    replacement_client = httpx.AsyncClient(
        transport=httpx.MockTransport(handler),
        trust_env=False,
    )
    app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

    try:
        with TestClient(app) as client:
            response = client.post("/provider/predict", json={...})
    finally:
        app.dependency_overrides.clear()
        asyncio.run(replacement_client.aclose())

    assert response.status_code == 503
    assert response.json() == {
        "detail": {
            "error_code": "ai_provider_invalid_response",
            "message": "The AI provider returned an invalid prediction response.",
        }
    }

这里 provider 的 HTTP 状态码是 200，但业务 JSON 不合法。adapter 会用 Pydantic 拦下它，endpoint 再把 AIClientError 转成稳定 503。这样前端不需要区分 HTTPX、Pydantic、provider schema 这些内部细节。

11. 常见错误与诊断

错误一：直接替换旧 /predict

学习阶段可以先新增 provider endpoint。等新链路稳定后，再考虑是否合并或切换默认实现。

错误二：同步路由里调用 async provider

外部 HTTP adapter 是 async，就用 async def endpoint 和 await 清楚承接。

错误三：让测试访问真实 provider

接口测试应该用 MockTransport 和依赖覆盖模拟 provider，保证快速稳定。

错误四：忘记保持响应契约稳定

即使内部改成 provider，外部仍应返回项目自己的 PredictionResponse。

12. 面试视角：怎样解释渐进接入 provider

我不会一开始就把已有 /predict 直接替换成真实 provider 调用，而是会先新增 /provider/predict 验证 provider 链路。这个 endpoint 是 async 的，通过 Depends 获取 settings 和 lifespan-managed HTTP client，调用 provider adapter，再把内部 PredictionResult 映射成稳定的 PredictionResponse。这样旧接口不受影响，新链路也能被 MockTransport 测试覆盖。

13. 小测：先回忆，再展开

1. 为什么本节新增 /provider/predict，而不是直接改 /predict？

为了渐进接入真实 provider，避免突然改变已有接口的行为、测试稳定性和本地学习体验。

2. /provider/predict 为什么要写成 async def？

因为它要 await 异步 HTTP provider adapter，等待外部 I/O 时可以不阻塞事件循环。

3. 为什么 endpoint 返回 PredictionResponse，而不是 provider 原始 JSON？

因为外部调用方应该依赖本项目稳定 API 契约，而不是依赖某个 provider 的字段形状。

4. 测试里为什么要覆盖 get_provider_http_client？

为了让 endpoint 使用带 MockTransport 的测试 client，从而不访问真实外部网络。

14. 本节总结

本节把 provider prediction adapter 接进了 FastAPI，形成了新的 POST /provider/predict。它复用已有请求和响应模型，使用配置里的 provider prediction URL，通过 lifespan-managed HTTP client 发起异步 provider 请求，并把结果转换成稳定 PredictionResponse。

下一课建议学习：用配置开关选择预测后端。也就是让 /predict 在 demo provider 和真实 provider 之间切换，而不是长期保留两个相似入口。
