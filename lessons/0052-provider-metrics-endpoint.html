0052 Provider Metrics Endpoint：将内部指标快照暴露为只读 API

本节把 provider prediction 的成功、重试、耗尽、快速失败事件写入教学 Counter，并通过 GET /provider/metrics 暴露当前进程内快照。核心设计是分层：service 负责记录事实，counter 负责聚合标签，router 只负责 HTTP 展示与响应模型约束。

① 本节核心知识框架

Provider Metrics Endpoint
├─ 写入路径
│  ├─ request_provider_prediction()
│  ├─ wait_before_retry()
│  ├─ log_retry_exhausted()
│  └─ log_prediction_fail_fast()
├─ 聚合层
│  ├─ provider_prediction_metrics
│  ├─ record_provider_metric()
│  └─ get_provider_metrics_snapshot()
├─ HTTP 展示层
│  ├─ GET /provider/metrics
│  ├─ response_model=list[ProviderMetricSampleResponse]
│  └─ 只读，不触发 provider 调用
└─ 合同边界
   ├─ labels 固定 5 个 key
   ├─ count 非负
   ├─ endpoint 返回当前进程快照
   └─ 教学实现，不是生产多实例聚合

② 核心概念底层原理

metrics endpoint 是观测接口，不是业务接口。业务接口会改变业务状态或触发外部调用；观测接口读取已经发生的事实。如果 GET /provider/metrics 自己去调用 provider，就会污染指标：一次“查看指标”的行为反而制造新指标。

本项目把指标写入放在 service 边界，因为 service 最接近真实事件：provider 返回成功、需要重试、重试耗尽、fail fast。router 不应该重新推导 provider 失败语义；它只调用 get_provider_metrics_snapshot() 并交给 FastAPI response model 序列化。

教学 endpoint 返回 JSON，是为了让初学者直接看懂数据形状。生产监控通常还会暴露 Prometheus 文本、OpenTelemetry exporter 或由 agent/collector 采集。

③ 全套代码逐行精读解析

provider_prediction_metrics = ProviderMetricsCounter()

def record_provider_metric(labels: ProviderMetricLabels, amount: int = 1) -> None:
    provider_prediction_metrics.increment(labels, amount=amount)

def get_provider_metrics_snapshot() -> list[dict[str, object]]:
    return provider_prediction_metrics.snapshot()

def reset_provider_metrics() -> None:
    provider_prediction_metrics.reset()
代码职责设计原因缺失后的隐患
provider_prediction_metrics模块级教学 Counter用最小状态模型解释 provider metrics初学者需要先理解 Counter，再接复杂监控后端
record_provider_metric()统一写入入口调用方不直接依赖全局对象未来替换 backend 时要到处改调用点
amount=1默认一次事件加 1符合 counter 的事件计数语义调用方反复写数字，容易传错
get_provider_metrics_snapshot()统一读取入口router 只依赖快照函数，不知道内部 CounterHTTP 层与存储实现耦合
reset_provider_metrics()测试清理入口隔离进程内全局状态endpoint 测试之间计数互相污染
 
class ProviderMetricLabelsResponse(BaseModel):
    operation: str
    outcome: str
    failure_category: str
    error_code: str
    status_code: str

class ProviderMetricSampleResponse(BaseModel):
    labels: ProviderMetricLabelsResponse
    count: int = PydanticField(ge=0)

响应模型固定 JSON 形状：labels 是嵌套对象，count 是非负整数。如果没有 response model，字段名从 count 漂移到 value、labels 缺字段、count 变负数等问题都不容易被 OpenAPI 和测试捕获。

④ 核心机制运行全流程拆解

POST /provider/predict
  → run_prediction(... backend="provider")
  → request_provider_prediction()
       ├─ 成功：record_provider_metric(success)
       ├─ retry 前：wait_before_retry() → record retry_scheduled
       ├─ 重试耗尽：log_retry_exhausted() → record retry_exhausted
       └─ fail fast：log_prediction_fail_fast() → record fail_fast
  → 调用方再请求 GET /provider/metrics
  → router 调用 get_provider_metrics_snapshot()
  → ProviderMetricsCounter.snapshot()
  → FastAPI 用 ProviderMetricSampleResponse 序列化 JSON

成功路径的写入点：

body = ProviderPredictionBody.model_validate(response.json())
record_provider_metric(build_provider_metric_labels(outcome="success"))
return PredictionResult(...)

失败路径复用同一 helper 生成 labels，而不是在每个分支手写 dict：

record_provider_metric(
    build_provider_metric_labels(
        outcome="retry_exhausted",
        failure_category=classify_retry_metric_failure(...),
        error_code=error_code,
        status_code=status_code,
    )
)

这条链路体现分层原则：provider adapter 识别事件，metrics helper 规范 label，counter 聚合，router 展示。

⑧ 课后记忆习题+标准答案

1. 为什么 metrics endpoint 必须是只读的？

因为它是观测接口，职责是读取已发生事件的聚合结果。如果读取指标会触发 provider 调用，就会制造新事件并污染计数。

2. 为什么 metrics 写入应放在 service/adapter 层？

service/adapter 最清楚 provider 调用真实结果，能准确判断 success、retry_scheduled、retry_exhausted、fail_fast。router 只负责 HTTP 展示。

3. ProviderMetricSampleResponse 的作用是什么？

它固定 endpoint JSON 合同：每个样本包含 labels 和非负 count，让 OpenAPI、测试和读者都能依赖稳定结构。

4. 为什么说本 endpoint 不是生产多实例 metrics 聚合？

它只读取当前进程内 Counter，不跨进程、不跨实例、不持久化，也没有成熟的采集、查询、告警能力。

⑨ 面试高频真题+解析

面试题 1：如何设计一个指标读取 endpoint？

应保持只读，不触发业务行为；写入发生在业务事件现场，读取只返回 snapshot；用 response model 固定输出结构；文档中说明数据范围、权限和生产边界。

面试题 2：为什么 router 不应该拼装 metrics labels？

router 属于 HTTP 适配层，不应掌握 provider retry/fail-fast 语义。labels 由 service/adapter 在真实事件发生处构造，router 只负责返回已有快照，避免层间职责错位。

面试题 3：教学版 in-process metrics 迁移到生产需要补什么？

需要成熟 metrics client、标准导出格式、跨实例聚合、采集器、权限控制、采样/基数治理、长期存储、仪表盘和告警规则。
