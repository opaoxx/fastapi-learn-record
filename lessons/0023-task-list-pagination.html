Lesson 0023

任务列表分页：用 limit 和 offset 控制返回数量

任务历史会不断增长。分页的本质不是“少返回几条”这么简单，而是在数据库、网络、前端渲染和用户体验之间建立一个可控边界。

0. 本节学习目标

  • 理解列表接口为什么必须限制返回规模。
  • 掌握 limit 与 offset 的数学含义和 URL 表达。
  • 理解 FastAPI 如何用 Query 在进入业务函数前拒绝非法分页参数。
  • 看懂 SQLModel 查询中 order_by、offset、limit 的组合。
  • 理解为什么分页必须配合稳定排序。

1. 背景：列表接口最怕“无限制”

学习项目里可能只有几个任务，所以全量返回暂时没问题。但真实系统里的任务表、日志表、订单表、文件表会快速增长。如果列表接口没有分页，调用方可能一次拉走几万条数据。

这会同时伤到三层：

  • 数据库层：查询结果集过大，占用更多 CPU、内存和磁盘 IO。
  • 网络层：响应体变大，传输变慢，移动网络下尤其明显。
  • 前端层：浏览器要解析大量 JSON，还要渲染大量 DOM，页面会卡。

分页就是给列表接口加上“每次只看一小段”的规则。它不是高级功能，而是后端 API 的基本卫生习惯。

2. 核心概念：limit 是窗口大小，offset 是窗口起点

请求：

GET /tasks?limit=20&offset=40

可以理解成：在排序后的任务序列里，跳过前 40 条，然后最多拿 20 条。

offset=0,  limit=20  -> 第 1 页
offset=20, limit=20  -> 第 2 页
offset=40, limit=20  -> 第 3 页

如果把任务列表想成一条长队，offset 是从队头跳过多少个人，limit 是接下来允许进入视野的人数。

3. 查询参数为什么适合表达分页

路径 /tasks 表达“任务集合”这个资源。分页不改变资源是谁，只改变“以什么视角查看集合”。所以它适合放在查询参数里：

GET /tasks?limit=10&offset=0
GET /tasks?limit=10&offset=10

这和搜索、排序、状态筛选属于同一类问题：它们都是集合视图的条件，而不是新的资源路径。

4. 代码精读一：router 层只负责 HTTP 参数

@router.get("", response_model=SummaryTaskListResponse)
def list_summary_tasks(
    session: Annotated[Session, Depends(get_session)],
    task_status: Annotated[TaskStatus | None, Query(alias="status")] = None,
    offset: Annotated[int, Query(ge=0)] = 0,
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
) -> SummaryTaskListResponse:
    return list_summary_task_records(
        session=session,
        task_status=task_status,
        offset=offset,
        limit=limit,
    )

这里有三层意思：

  1. offset 和 limit 是查询参数，不是请求体。
  2. Query(ge=0) 让负数 offset 在进入业务逻辑前就被拒绝。
  3. Query(ge=1, le=100) 给 limit 设置下限和上限，防止调用方请求 0 条或 100000 条。

FastAPI 官方文档把这类参数称为 query parameters。它们出现在 URL 的 ? 后面，并且可以有默认值和校验规则。

5. 为什么要限制 limit 的最大值

很多初学者会问：既然客户端想要更多数据，为什么不允许 limit=10000？

原因是后端 API 要保护系统边界。服务端不是只服务一个请求，而是同时服务很多用户。如果一个请求拿走太多资源，其他请求也会变慢。le=100 这样的上限是一种服务端保护。

这也是 API 设计中的重要原则：客户端可以提出偏好，但服务端必须保留资源控制权。

6. 代码精读二：service 层构造数据库查询

statement = select(SummaryTask).order_by(SummaryTask.id.desc())
statement = statement.offset(offset).limit(limit)
items = list(session.exec(statement))

select(SummaryTask) 表示查询任务表中的任务记录。order_by(SummaryTask.id.desc()) 表示最新任务排在前面。然后 offset() 和 limit() 截取其中一段。

SQLModel 的分页教程也使用了 offset 和 limit 的组合。它们最终会转成数据库可以理解的 SQL 片段。

7. 为什么分页前必须有稳定排序

如果没有 order_by，数据库不承诺每次返回顺序都一样。你第一页拿到 A、B、C，第二页可能因为默认顺序变化又看到 B，或者漏掉 D。

分页的数学前提是：先有一个稳定序列，再从序列里切片。我们的稳定序列是：

按 id 从大到小排列的 SummaryTask 列表

这让“第一页是最新任务”这个用户体验也更自然。

8. offset 分页的局限

limit + offset 很适合入门和中小规模列表，但它不是万能方案。它有两个常见限制：

  • 数据量很大时，跳过很多行可能变慢。
  • 翻页过程中如果有新任务插入，用户可能看到重复或错位。

真实大规模系统常用游标分页，比如“从某个 id 之后继续取”。但对当前课程来说，先学会 limit + offset 是最合适的，因为它直观、容易测试，也正好服务前端分页控件。

9. 和响应信封的关系

本课聚焦“怎么取当前页”。第 25 课会解释为什么响应形状要变成：

{
  "items": [...],
  "count": 18,
  "limit": 20,
  "offset": 0
}

先记住一句话：limit 和 offset 决定当前页有哪些 items；count 则帮助前端知道总共有多少条。

10. 常见错误与诊断

错误一：limit 允许无限大

这会让一个请求消耗过多资源。用 Query(le=100) 设置上限。

错误二：offset 允许负数

负数没有清晰业务含义。用 Query(ge=0) 拒绝。

错误三：没有排序就分页

数据库默认顺序不可靠，分页可能重复或漏项。

错误四：把分页逻辑写满 router

router 应接收 HTTP 参数，数据库查询细节更适合放 service。

11. 课堂实验

先创建 3 个摘要任务，然后分别请求：

GET /tasks?limit=1&offset=0
GET /tasks?limit=1&offset=1
GET /tasks?limit=1&offset=2

观察：

  1. 每次 items 是否最多只有 1 条。
  2. 三个请求返回的任务 id 是否不同。
  3. id 是否按从新到旧排列。

再试一个非法请求：

GET /tasks?limit=0&offset=-1

它应该返回 422，因为参数在进入 service 前就被 FastAPI 校验失败。

12. 小测：先回忆再展开

1. limit 和 offset 分别控制什么？

limit 控制最多返回多少条，offset 控制从排序后的结果里跳过多少条。

2. 分页前为什么要排序？

因为分页需要稳定序列；没有排序时数据库返回顺序不可靠，可能重复或漏项。

3. 为什么要限制 limit 最大值？

为了保护数据库、网络和前端渲染资源，避免单个请求拉取过多数据。

4. 分页参数为什么放查询参数而不是请求体？

因为它们描述的是如何查看同一个任务集合，属于列表视图条件。

13. 本节总结

本节把任务列表从“全量返回”升级为“按窗口读取”。你需要记住的不是两个参数名，而是分页背后的边界意识：后端要控制每次查询、传输和渲染的数据规模，并用稳定排序保证分页语义清楚。
