Lesson 0023

任务列表分页：用 limit 和 offset 控制返回数量

任务越来越多时，接口不能一次把所有历史记录都返回。分页让列表接口更快、更稳定，也更适合前端页面展示。

0. 本节学习目标

  • 理解为什么列表接口需要分页。
  • 掌握 limit 和 offset 的含义。
  • 看懂 FastAPI 如何用 Query 校验分页参数。
  • 看懂 SQLModel 的 offset() 和 limit()。

1. 为什么不能一次返回所有任务

学习项目里可能只有几个任务，但真实系统中，任务表可能很快变成几千、几万条。一次返回所有记录会带来三个问题：

  • 数据库查询变慢。
  • 网络响应变大。
  • 前端渲染变卡。

所以列表接口通常要支持分页，让客户端按页读取。

2. limit 和 offset

GET /tasks?limit=20&offset=0
  • limit：这次最多返回多少条。
  • offset：跳过前面多少条。

如果每页 20 条：

第 1 页: offset=0,  limit=20
第 2 页: offset=20, limit=20
第 3 页: offset=40, limit=20

3. router 接收查询参数

@router.get("", response_model=list[SummaryTaskRead])
def list_summary_tasks(
    session: Annotated[Session, Depends(get_session)],
    task_status: Annotated[TaskStatus | None, Query(alias="status")] = None,
    offset: Annotated[int, Query(ge=0)] = 0,
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
) -> list[SummaryTask]:
    return list_summary_task_records(
        session=session,
        task_status=task_status,
        offset=offset,
        limit=limit,
    )

这里的 Query(ge=0) 表示 offset 不能小于 0。Query(ge=1, le=100) 表示 limit 至少 1，最多 100。

4. service 执行数据库分页

statement = select(SummaryTask).order_by(SummaryTask.id.desc())
statement = statement.offset(offset).limit(limit)
return list(session.exec(statement))

order_by(SummaryTask.id.desc()) 让最新任务排在前面。没有稳定排序时，分页容易出现重复或漏项，因为数据库不保证默认顺序。

5. 为什么分页逻辑放 service

router 负责把 HTTP 查询参数变成 Python 参数；service 负责构造数据库查询。这样以后要把分页规则复用到别的接口，或者改成游标分页时，router 不需要知道太多数据库细节。

6. 常见错误

  • /tasks?limit=0：返回 422，因为 limit 至少是 1。
  • /tasks?offset=-1：返回 422，因为 offset 不能为负数。
  • /tasks?limit=10000：返回 422，因为服务端限制最多 100。

这就是 Pydantic/FastAPI 自动校验的价值：错误请求不会进入业务函数。

7. 课堂练习

先创建 3 个摘要任务，然后分别请求：

GET /tasks?limit=1&offset=0
GET /tasks?limit=1&offset=1
GET /tasks?limit=1&offset=2

观察每次返回的任务 id 是否不同，以及它们是否按从新到旧排列。

8. 小测：先回忆再展开

1. limit 控制什么？

控制这次请求最多返回多少条记录。

2. offset 控制什么？

控制查询结果从前面跳过多少条记录。

3. 分页前为什么要排序？

因为稳定排序能减少分页时重复或漏掉记录的风险。

9. 本节总结

本节让任务列表接口从“全量返回”升级为“可分页返回”。这是任何任务历史、日志列表、订单列表、文件列表都会用到的后端基础能力。

推荐主资源：FastAPI Query Parameters、SQLModel Limit and Offset。
