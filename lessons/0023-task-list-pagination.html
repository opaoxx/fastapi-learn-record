任务列表分页：limit/offset、稳定排序与列表 API 资源边界

列表接口不能无限制返回全部数据。分页不是 UI 小功能，而是数据库负载、网络传输、浏览器渲染和 API 契约共同的资源边界。本节用 GET /tasks?limit=&offset= 建立任务历史的分页查询模型。

① 本节核心知识框架

limit/offset 分页
├─ API 入口：GET /tasks?limit=20&offset=0
├─ 参数语义
│  ├─ limit：窗口大小，最多返回多少条
│  └─ offset：窗口起点，跳过多少条匹配记录
├─ FastAPI 校验
│  ├─ offset: Query(ge=0)
│  └─ limit: Query(ge=1, le=100)
├─ SQLModel 查询
│  ├─ select(SummaryTask)
│  ├─ order_by(SummaryTask.id.desc())
│  ├─ offset(offset)
│  └─ limit(limit)
├─ 响应 envelope：items/count/limit/offset
└─ 底层原则：先形成稳定序列，再切片

② 核心概念底层原理

limit 表示最多取多少条，offset 表示从排序后的结果集中跳过多少条。它们不是数据库独有概念，而是“有序集合切片”的 API 表达：offset=40&limit=20 就是从第 41 条开始取最多 20 条。

分页必须建立在稳定排序之上。关系型数据库如果没有 ORDER BY，不承诺返回顺序；同样的查询可能因为执行计划、插入、删除、索引变化而出现不同顺序。没有稳定序列，分页会重复、漏项或错位。

与游标分页区分：offset 分页简单、直观、适合教学和中小列表；游标分页通过 after_id 或 opaque cursor 继续读取，适合大数据量和频繁插入场景。本课先掌握 offset 分页，因为它与前端页码控件最直观。

③ 全套代码逐行精读解析

@router.get("", response_model=SummaryTaskListResponse)
def list_summary_tasks(
    session: Annotated[Session, Depends(get_session)],
    task_status: Annotated[TaskStatus | None, Query(alias="status")] = None,
    offset: Annotated[int, Query(ge=0)] = 0,
    limit: Annotated[int, Query(ge=1, le=100)] = 20,
) -> SummaryTaskListResponse:
    return list_summary_task_records(
        session=session,
        task_status=task_status,
        offset=offset,
        limit=limit,
    )
  • @router.get("") 表示 /tasks 集合资源；空字符串配合 router 前缀，避免重复写路径。
  • response_model=SummaryTaskListResponse 指定列表响应 envelope，不再返回裸数组。
  • session: Annotated[Session, Depends(get_session)] 注入数据库会话，router 不关心连接创建细节。
  • task_status 暂时允许筛选条件传入，下一课重点讲；分页与筛选共用同一个列表接口。
  • Query(ge=0) 在进入 service 前拒绝负数 offset；如果允许负数，不同数据库行为可能不一致。
  • Query(ge=1, le=100) 防止 limit=0 的无意义请求和 limit=100000 的资源攻击。
  • return list_summary_task_records(...) 让 router 只转发已验证参数，数据库查询细节留在 service。
def list_summary_tasks(
    session: Session,
    task_status: TaskStatus | None = None,
    offset: int = 0,
    limit: int = 20,
) -> SummaryTaskListResponse:
    statement = select(SummaryTask).order_by(SummaryTask.id.desc())
    count_statement = select(func.count()).select_from(SummaryTask)
    if task_status is not None:
        statement = statement.where(SummaryTask.status == task_status)
        count_statement = count_statement.where(SummaryTask.status == task_status)
    statement = statement.offset(offset).limit(limit)
    items = list(session.exec(statement))
    count = session.exec(count_statement).one()
    return SummaryTaskListResponse(items=items, count=count, limit=limit, offset=offset)
  • select(SummaryTask) 构造任务表查询，不立即访问数据库；SQLModel 语句是惰性查询描述。
  • order_by(SummaryTask.id.desc()) 让最新任务排在前面；缺失会破坏分页稳定性。
  • count_statement 单独计算总数；当前页 items 的长度不等于总数。
  • if task_status is not None 只有传筛选时才加 where；否则返回全部状态。
  • statement.offset(offset).limit(limit) 在筛选和排序之后切片，表达“匹配结果中的这一页”。
  • list(session.exec(statement)) 真正执行查询并把结果物化为列表；不转 list 可能返回迭代器式结果对象。
  • session.exec(count_statement).one() 读取聚合 count 的单行结果；count 用于前端判断是否还有下一页。
  • SummaryTaskListResponse(...) 把当前页数据和分页元信息绑定为稳定响应契约。

④ 核心机制运行全流程拆解

GET /tasks?limit=10&offset=20
  -> FastAPI 解析 query 参数
  -> 校验 offset >= 0、1 <= limit <= 100
  -> router 调用 list_summary_task_records()
  -> service 构造基础查询 select(SummaryTask)
  -> 应用 order_by(id desc)，形成稳定序列
  -> 应用 offset(20).limit(10)，截取窗口
  -> 执行 items 查询
  -> 执行 count 查询
  -> 返回 {items, count, limit, offset}
  -> 前端用 count 和 items.length 决定下一页按钮

关键顺序是：校验参数在 HTTP 层，查询构造在 service 层，排序先于分页，总数查询独立于当前页查询。只要顺序错了，分页体验就会立刻失真。

⑤ 重难点、易混淆点对比辨析

概念正确含义常见误解
limit最多返回条数总页数或总数量
offset跳过多少条匹配记录页码本身
count当前筛选条件下总数当前页条数
order_by分页的稳定序列前提可有可无的显示优化
offset 分页简单直观大规模场景永远最佳

⑥ 开发常见报错+坑点+解决方案

  • GET /tasks?limit=0 返回 422：这是正确行为。解决：客户端把 limit 控件下限设为 1。
  • 翻页重复或漏项：缺少稳定排序。解决：按唯一或近似唯一字段排序，例如 id desc。
  • 下一页按钮判断错误：只看 items.length === limit。解决：使用 offset + items.length < count。
  • 一次请求拖垮页面：limit 无上限。解决：服务端设置 le=100，不要只依赖前端限制。
  • 分页和筛选组合错位：先分页再筛选。解决：先 where 过滤，再 order_by 排序，再 offset/limit。

⑦ 生产环境 vs 教学环境 核心差异

维度教学环境生产环境

⑧ 课后记忆习题+标准答案

1. limit=20&offset=40 表示什么？

在排序后的匹配结果中跳过前 40 条，然后最多返回 20 条。

2. 为什么分页必须配合 order_by？

数据库无排序时不保证返回顺序，分页切片会基于不稳定序列，导致重复或漏项。

3. 为什么服务端要限制 limit 最大值？

服务端需要保护数据库、网络和前端资源，不能让单个客户端请求无限消耗系统容量。

4. count 为什么不能用 len(items) 替代？

len(items) 是当前页条数，count 是当前筛选条件下总条数，二者语义不同。

⑨ 面试高频真题+解析

题 1：offset 分页有什么缺点？

深页查询可能慢，因为数据库要跳过大量行；数据频繁插入或删除时，页间可能重复或漏项。大规模动态列表通常考虑 cursor/keyset pagination。

题 2：如何设计一个列表接口的分页响应？

返回当前页 items，同时返回 limit、offset、count 或 next cursor，让前端能渲染分页控件并判断是否还有后续数据。

题 3：为什么分页参数放 query 而不是 path？

分页描述的是同一集合资源的视图条件，不定位新的资源实体；因此适合 /tasks?limit=...&offset=...。
