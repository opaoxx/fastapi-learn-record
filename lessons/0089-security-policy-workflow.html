② 核心概念底层原理

安全策略的底层目标是控制信息扩散半径。普通 bug 可以公开复现，因为公开复现能帮助协作；安全问题若公开完整攻击步骤，会把“帮助修复”变成“帮助利用”。因此安全报告必须先进入私下通道。

教学仓库不是生产服务，但仍有安全风险。学习者可能复制代码到真实项目；仓库可能误提交密钥；课程可能给出不安全生产建议；上传、鉴权、CORS、依赖配置都可能被误学误用。SECURITY.md 的作用是把这些风险纳入可处理流程。

安全报告与普通 issue 的核心区别是敏感细节。普通 issue 追求公开、完整、可复现；安全报告追求私密、最小披露、证据可检查。报告者应提供影响范围、最小复现、受影响文件和缓解建议，但不能在公开区发布可武器化细节。

入口适用内容是否公开细节处理目标
Issue普通 bug、断链、课程错误可以公开复现协作修复
Discussion概念问题、开放讨论不贴敏感信息知识沉淀
Security report漏洞、密钥、隐私、可利用风险私下披露降低利用风险并修复
Code of Conduct report骚扰、安全协作事件私下处理保护参与者

生产环境安全流程通常有 SLA、漏洞赏金、CVE、embargo、专用邮箱和安全团队。本教学仓库采用轻量流程，但仍保留核心原则：私下报告、分级响应、证据保全、最小公开披露、修复后验证。

③ 全套代码逐行精读解析

# Security Policy

标题采用 GitHub 识别的安全策略文件名。根目录 SECURITY.md 会成为贡献者寻找漏洞披露规则的默认入口。

This repository is a teaching project, but security reports still need a private and evidence-based workflow.

这行说明教学项目不等于无安全责任。即使没有线上服务，也可能影响学习者复制实践、仓库公开内容和贡献协作安全。

Do not publish exploit details, secrets, private logs, personal data, customer data, or working attack steps in public issues, discussions, pull requests, or comments.

这是安全策略的红线。公开渠道一旦出现 exploit detail 或 secrets，就可能被搜索引擎、fork、通知邮件和第三方缓存扩散。

## Supported Scope

Supported Scope 定义哪些问题按安全流程处理。没有范围，报告者会把普通问题都当漏洞，维护者会浪费 triage 成本。

Exposed credentials, API keys, tokens, or secret-like values committed to the repository.

密钥暴露是最高频仓库安全事故。即使是教学 key，也要训练学习者形成“公开仓库不放 secrets”的习惯。

Authentication or authorization bypass in the FastAPI teaching project.

鉴权绕过属于真实安全风险。课程已有 API key、依赖注入和受保护接口，若行为与文档不一致，应走安全流程。

Unsafe file upload behavior that could expose private data or execute unintended content.

上传入口常见风险包括过大文件、路径处理、内容解析、私有数据暴露和非预期执行。本项目虽是教学实现，仍要明确边界。

Documentation that instructs learners to use unsafe production practices.

课程文档本身也可能制造安全问题。如果教程鼓励把 demo key 用到生产、公开日志或关闭鉴权，这就是教学层安全风险。

This teaching repository does not provide a production service, hosted customer environment, bug bounty program, or guaranteed response SLA.

这行定义非承诺范围。教学仓库不能承诺生产 SLA 或漏洞赏金，避免报告者产生错误预期。

## Not Security Reports

Not Security Reports 用于分流普通问题。安全流程应处理可利用风险，不应承载所有概念疑问和泛化建议。

Beginner questions about API keys, CORS, authentication, or GitHub permissions.

概念问题不等于漏洞。它们应进入 Discussion 或普通 issue，除非包含具体可利用行为或敏感信息。

When unsure, avoid posting sensitive details publicly and ask maintainers for a private contact path.

不确定时采用安全默认值：少公开，先问私下路径。这个规则能避免误把漏洞细节贴到公开区。

## Private Reporting Workflow

Private Reporting Workflow 是本节执行核心。它规定优先使用 GitHub 私密漏洞报告；没有私密通道时，只能公开请求联系路径，不能公开细节。

Open a short public issue asking for a maintainer security contact path.

这是兜底策略。公开 issue 只表达“需要安全联系路径”，不包含复现步骤、payload、密钥、日志或个人数据。

Include the following in the private report:

私下报告字段保证可处理性。没有 affected area、impact、minimal reproduction、mitigation，维护者难以分级和修复。

Affected file, route, dependency, lesson, or GitHub workflow.

该字段覆盖课程仓库的全部风险对象：源码、接口、依赖、课程文档和 GitHub 工作流。安全风险不只存在于后端代码。

## Maintainer Triage

Maintainer Triage 定义分级。分级让维护者用不同优先级处理密钥泄露、数据暴露、文档硬化和证据不足的报告。

critical: credential exposure, public exploit path, or severe access-control bypass.

critical 对应立即风险：密钥、公开利用路径或严重访问控制绕过。处理目标是快速止血、删除暴露、旋转密钥并修复。

medium: unsafe documented practice that learners may copy into production.

medium 体现教学仓库特性。即使代码不在线运行，不安全文档也可能被学习者复制到生产，因此需要修复。

## Fix and Disclosure Flow

Fix and Disclosure Flow 把安全报告转成修复链路：私下报告、分级、复现、修补、验证、合并、最小公开披露。

Public disclosure should avoid weaponized detail.

公开说明要帮助学习者修复和理解影响，而不是提供可直接攻击的细节。应说明范围、缓解、固定版本或 commit。

## Required Verification

安全修复仍需正常质量门禁。安全问题不能成为绕过测试、review、branch protection 的理由。

Add focused tests when the fix changes behavior, validation, authentication, file handling, dependency configuration, or documented security guidance.

这句要求回归保护。安全修复若没有测试，很容易在后续重构中复发。

## Relationship to Other Documents

最后把 SECURITY 接入治理体系：CODE_OF_CONDUCT 处理协作安全，CONTRIBUTING 处理正常贡献，DEVELOPMENT 处理命令，ISSUE_TRIAGE 处理普通 issue，RELEASE_PROCESS 处理发布。

⑧ 课后记忆习题+标准答案

1. SECURITY.md 的核心职责是什么？

定义漏洞和敏感安全问题的私下报告、维护者分级、修复验证和最小公开披露流程，避免在公开渠道扩散可利用细节。

2. 为什么教学仓库仍然需要安全策略？

学习者会复制代码和文档；仓库可能暴露密钥；课程可能传播不安全生产实践。因此即使没有线上服务，也需要安全报告边界。

3. 不确定是否为漏洞时应如何处理？

不要公开敏感细节，先请求维护者私下联系路径；确认不是安全问题后再转为普通 issue 或 discussion。

4. 安全报告私下应包含哪些字段？

受影响文件/接口/依赖/课程/工作流、影响摘要、最小复现、是否涉及 secrets 或私有数据、缓解建议、是否已公开。

5. 安全修复为什么仍要跑测试？

安全修复可能改变鉴权、校验、上传、依赖或文档契约；没有测试会导致回归。紧急性不能替代验证。

⑨ 面试高频真题+解析

面试题 1：SECURITY.md 和普通 issue template 的区别是什么？

SECURITY.md 处理敏感漏洞和私下披露，强调最小公开细节；issue template 处理普通可公开复现问题，强调完整上下文和验收条件。

面试题 2：为什么公开漏洞复现细节有风险？

公开细节会被搜索、通知、fork 和缓存扩散，攻击者可直接利用。私下披露能让维护者先修复和验证，再发布最小安全说明。

面试题 3：安全报告如何分级？

可按影响和可利用性分为 critical、high、medium、low。凭据泄露和严重绕过最高，文档不安全指导通常为中等，证据不足硬化建议较低。

面试题 4：安全修复的发布说明应包含什么？

应包含受影响范围、缓解措施、固定版本或 commit、学习者需要修改的做法，避免 payload、完整攻击链和敏感数据。

面试题 5：生产级安全策略还需要哪些能力？

需要专用报告通道、SLA、漏洞赏金、CVE/advisory、embargo、依赖扫描、事件响应、密钥轮换、客户通知和合规记录。
