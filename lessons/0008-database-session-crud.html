Lesson 0008

数据库 Session 和 CRUD：让接口真正读写数据库

本节把第 6 课的 Depends 和第 7 课的 SQLModel 连起来：用依赖为每个请求提供数据库 Session，然后用 Session 完成查询、创建、提交和刷新。

0. 本节学习目标

  • 理解 Session 是一次请求访问数据库的工作单元。
  • 理解 yield 依赖如何创建并关闭 Session。
  • 看懂 GET /items、GET /items/{item_id}、POST /items 的数据库版本。
  • 理解 add、commit、refresh 的基本作用。

1. Session 是什么

你可以把数据库 Session 理解成“这一轮请求和数据库沟通的上下文”。路径函数不应该自己到处创建连接，而是通过依赖拿到一个 Session。

本项目的依赖是：

def get_session() -> Generator[Session, None, None]:
    with Session(engine) as session:
        yield session

这段代码的含义是：

  1. 请求进来时，创建一个 Session。
  2. yield session 把 Session 交给路径函数使用。
  3. 路径函数执行完后，离开 with 代码块，Session 自动关闭。

连接第 6 课：这就是 Depends 的价值。路径函数只声明需要 Session，创建和关闭由依赖统一管理。

2. 查询列表：select + session.exec

@router.get("", response_model=list[ItemRead])
def list_items(
    filters: Annotated[ItemFilters, Depends(ItemFilters)],
    pagination: Annotated[Pagination, Depends(Pagination)],
    session: Annotated[Session, Depends(get_session)],
) -> list[Item]:
    statement = select(Item)
    items = list(session.exec(statement))
    ...
    return items[pagination.skip : pagination.skip + pagination.limit]

select(Item) 表示构造一条“查询所有 Item”的 SQL 查询。session.exec(statement) 执行查询并返回结果。

当前为了保持小白友好，筛选和分页仍然在 Python 列表里完成。后面课程可以再把这些筛选下推到 SQL 查询中，让数据库做更多工作。

3. 查询单个资源：session.get

item = session.get(Item, item_id)
if item is None:
    raise HTTPException(
        status_code=status.HTTP_404_NOT_FOUND,
        detail=f"Item {item_id} was not found.",
    )
return item

session.get(Item, item_id) 表示按主键查找一条 Item。查不到时返回 None，这时我们主动抛出 404。

这也复习了第 4 课：999 是合法 ID，但数据库里没有对应记录，所以是业务层面的 404，不是参数校验 422。

4. 创建资源：add + commit + refresh

@router.post("", response_model=ItemRead, status_code=status.HTTP_201_CREATED)
def create_item(
    payload: ItemCreate,
    session: Annotated[Session, Depends(get_session)],
) -> Item:
    item = Item.model_validate(payload)
    session.add(item)
    session.commit()
    session.refresh(item)
    return item

逐行解释：

  • Item.model_validate(payload)：把创建请求模型转换成数据库表模型。
  • session.add(item)：告诉 Session 准备新增这条记录。
  • session.commit()：真正提交到数据库文件。
  • session.refresh(item)：从数据库刷新对象，拿到自动生成的 id。

如果忘记 commit，数据通常不会真正保存；如果忘记 refresh，返回对象可能拿不到数据库生成后的最新值。

5. 为什么数据库 Session 要用 Depends

如果每个接口都手动写 Session(engine)，会有三个问题：

  • 重复代码多。
  • 容易忘记关闭 Session。
  • 以后测试时不容易替换成测试数据库。

用 Depends(get_session) 后，路径函数只关心“我要一个 Session”，至于怎么创建、怎么关闭、以后怎么替换，都集中在依赖函数里。

6. 实验：确认数据能持久化

在 /docs 里执行：

POST /items
{
  "name": "Persistent FastAPI Notes",
  "category": "book",
  "price": 12.5,
  "in_stock": true
}

然后执行：

GET /items
GET /items/{new_id}

重启 FastAPI 服务后再次执行 GET /items。如果新商品还在，说明它已经写入 SQLite 文件，而不是只存在内存里。

7. 课堂练习

请你完成三个小实验：

  1. 创建一个 category="tool" 的商品。
  2. 用 GET /items?category=tool 查到它。
  3. 重启服务后再查一次，确认数据仍然存在。

完成标准：你能解释“重启后仍然存在”为什么说明数据库持久化成功。

8. 小测：先回忆再展开

1. yield session 做了什么？

它把创建好的数据库 Session 交给路径函数使用，函数结束后依赖继续清理资源。

2. commit 的作用是什么？

它把 Session 中准备好的数据库改动真正提交保存。

3. refresh 为什么常跟在创建后面？

它从数据库重新读取对象，让自动生成的主键等最新字段回到 Python 对象上。

9. 本节总结

现在你的 items 接口已经从内存字典升级为 SQLite 数据库。Depends(get_session) 让每个请求获得一个可控的数据库 Session；select、get、add、commit、refresh 是数据库 CRUD 的基本动作。
