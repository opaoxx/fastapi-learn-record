数据库 Session 与 CRUD：请求级工作单元的读写闭环

本节把第 6 课的依赖注入和第 7 课的数据库模型合并：每个请求通过 Depends(get_session) 获得一个 Session，再用 select/get/add/commit/refresh 完成读写。重点是 Session 生命周期和事务语义，而不是机械背 CRUD。

① 本节核心知识框架

Session CRUD
├─ get_session：yield 依赖，创建并关闭请求级 Session
├─ Read list：select(Item) + session.exec(...)
├─ Read one：session.get(Item, item_id)
├─ Create：Item.model_validate(payload)
│  ├─ session.add(item)：加入事务跟踪
│  ├─ session.commit()：提交写入
│  └─ session.refresh(item)：读取数据库生成字段
├─ Response：Item/SQLModel -> ItemRead JSON
└─ 错误边界：合法 ID 查不到 -> 404

② 核心概念底层原理

Session 是 ORM 工作单元。它不只是“数据库连接”，还负责对象状态追踪、事务边界、SQL 执行和结果映射。一次请求拿一个短生命周期 Session，是 Web API 最常见的安全起点。

事务提交是写入的分界线。session.add(item) 只是把对象放进待处理队列；commit() 才把改动提交给数据库。若请求过程中发生异常且未提交，改动不应被持久化。

refresh() 解决 Python 对象与数据库状态同步。自增主键、数据库默认值、触发器生成字段只有数据库知道；提交后刷新对象，API 才能返回完整、最新的数据。

教学实现与生产查询的边界。当前 list_items 先查全量再 Python 过滤，利于新手理解；生产要把过滤、排序、分页下推到 SQL，否则数据量一大就会退化。

③ 全套代码逐行精读解析

def get_session() -> Generator[Session, None, None]:
    with Session(engine) as session:
        yield session
  • Generator[Session, None, None]：说明这是 yield 依赖，产出值是 Session。
  • with Session(engine)：创建会话并交给上下文管理器关闭。若手动创建不关闭，会造成连接泄漏。
  • yield session：FastAPI 将该值注入路径函数；请求结束后继续退出 with。
statement = select(Item)
items = list(session.exec(statement))
  • select(Item)：构造查询 Item 表全部列的 SQLModel/SQLAlchemy statement。
  • session.exec(statement)：通过当前 Session 执行 SQL 并返回 ORM 对象结果。
  • list(...)：消费结果迭代器，得到 Python 列表。教学清晰，但生产大结果集要谨慎。
item = session.get(Item, item_id)
if item is None:
    raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail=f"Item {item_id} was not found.")
  • session.get(Item, item_id)：按主键查询，适合详情页读取。
  • item is None：参数合法但记录不存在。
  • HTTPException(404)：把业务不存在转换为 HTTP 语义，不让 None 进入响应模型。
item = Item.model_validate(payload)
session.add(item)
session.commit()
session.refresh(item)
return item
  • Item.model_validate(payload)：把 ItemCreate 输入模型转成 Item 表模型。若直接拼字典，模型边界更脆弱。
  • session.add(item)：进入 Session 的 pending 状态，等待 flush/commit。
  • session.commit()：提交事务，数据库生成主键。
  • session.refresh(item)：刷新 Python 对象，拿到 id 等数据库侧值。
  • return item：FastAPI 按 response_model=ItemRead 输出 JSON。

④ 核心机制运行全流程拆解

POST /items
  -> require_api_key 通过
  -> Pydantic 校验 ItemCreate
  -> get_session() yield Session
  -> Item.model_validate(payload)
  -> session.add(item)
  -> commit: INSERT INTO item ...
  -> refresh: SELECT 新生成 id
  -> ItemRead 序列化
  -> 201 + JSON
  -> 请求结束，Session 关闭

GET /items/2
  -> get_session() yield Session
  -> session.get(Item, 2)
  -> 命中：200 + ItemRead
  -> 未命中：404 + detail

⑤ 重难点、易混淆点对比辨析

  • add vs commit：add 只是登记变更，commit 才提交事务。
  • flush vs commit：flush 发送 SQL 但不结束事务；commit 提交事务。当前课程先掌握 commit。
  • refresh vs 重新查询：refresh 针对已有对象刷新字段；重新查询会得到另一个查询结果对象。
  • Session vs Engine：engine 管连接配置，Session 管一次工作单元和 ORM 对象状态。

⑥ 开发常见报错+坑点+解决方案

  • 创建后返回 id: null：通常忘记 commit 或 refresh。解决：提交后刷新。
  • 测试数据污染开发库：测试未覆盖 get_session。解决：使用内存 SQLite 和 dependency override。
  • Session 泄漏：手写 Session 但异常路径没关闭。解决：用 yield 依赖或上下文管理器。
  • 查询列表越来越慢：Python 层过滤全量数据。解决：生产中改为 SQL where/limit/offset。
  • 合法 ID 返回 500：未处理 None。解决：显式 404。

⑦ 生产环境 vs 教学环境 核心差异

  • 查询下推：教学先全量读取再过滤；生产应构造 SQL 条件、分页和索引。
  • 事务边界：教学每个写接口简单 commit；生产要考虑异常 rollback、唯一约束冲突、幂等写入。
  • 连接池：教学 SQLite 配置极简；生产数据库需要连接池大小、超时、回收策略。
  • 观测性：教学不记录 SQL 指标；生产应关注慢查询、错误率、连接耗尽和事务锁等待。

⑧ 课后记忆习题+标准答案

1. 为什么 Session 适合通过 Depends 注入？

Session 需要请求级创建和清理，Depends 能统一生命周期并方便测试替换。

2. 创建数据时 commit 和 refresh 分别解决什么？

commit 提交写入，refresh 把数据库生成或更新后的字段同步回 Python 对象。

3. 查不到主键记录为什么是 404？

因为请求参数合法，但业务资源不存在，属于资源未找到。

⑨ 面试高频真题+解析

1. Web API 中数据库 Session 生命周期应如何设计？

通常每个请求创建一个短生命周期 Session，请求结束后关闭，写操作在明确事务边界提交或回滚。

2. 为什么不建议路径函数里到处手动创建 Session？

重复、易泄漏、难测试，也会让事务和资源生命周期散落在业务代码中。

3. 为什么生产列表查询要把过滤分页交给数据库？

数据库能利用索引和查询优化器，避免应用层加载大量无关数据。
