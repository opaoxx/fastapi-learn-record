列表响应信封：从裸数组到可分页 API 契约

当列表接口只返回裸数组时，它只能表达“当前有哪些记录”。一旦前端需要总数、页大小、偏移量、下一页判断，响应就必须升级为 envelope。响应信封不是包装癖，而是列表 API 从 demo 走向工程可用的契约升级。

① 本节核心知识框架

列表响应信封
├─ 旧形态：list[SummaryTaskRead]
│  └─ 只能表达当前页任务数组
├─ 新形态：SummaryTaskListResponse
│  ├─ items：当前页任务
│  ├─ count：当前筛选条件下的总数
│  ├─ limit：服务端接受的页大小
│  └─ offset：当前页跳过的匹配记录数
├─ router 契约：response_model=SummaryTaskListResponse
├─ service 契约：返回模型对象而非裸 list
├─ frontend 契约：page.items 才是数组
└─ 兼容性结论：裸数组 -> 对象是破坏性 JSON 形状变更

② 核心概念底层原理

裸数组的顶层 JSON 类型是 array，响应信封的顶层 JSON 类型是 object。顶层类型变化会直接改变调用方解析方式：旧前端可以 for (const task of response)，新前端必须 for (const task of response.items)。所以这不是后端内部重构，而是 API wire format 改变。

Envelope 的本质是把业务数据和控制元数据放在同一个响应体里。items 是数据面，count/limit/offset 是控制面。分页 UI、SDK、自动化测试都依赖控制面判断“现在在哪一页、后面还有没有数据”。

与 HTTP header 分页元数据相比，body envelope 更适合本课程：它能被 Pydantic 模型描述，能进入 OpenAPI schema，浏览器和 Swagger UI 也更容易直接观察。生产系统可以选择 Link header 或 cursor 字段，但关键原则不变：列表响应必须显式携带翻页所需事实。

③ 全套代码逐行精读解析

class SummaryTaskListResponse(BaseModel):
    items: list[SummaryTaskRead] = PydanticField(description="The current page of summary tasks.")
    count: int = PydanticField(ge=0, description="Total number of tasks matching the current filter.")
    limit: int = PydanticField(ge=1, le=100, description="Maximum number of tasks requested for this page.")
    offset: int = PydanticField(ge=0, description="Zero-based number of matching tasks skipped before this page.")
  • BaseModel：这是纯响应契约模型，不是数据库表；如果继承 table 模型，容易把持久化字段和对外字段混在一起。
  • items: list[SummaryTaskRead]：当前页任务数组；用 read model 避免暴露原始 text。
  • PydanticField(description=...)：把字段语义写进 OpenAPI；没有描述时调用方只能猜 count 是总数还是当前页长度。
  • count: int = PydanticField(ge=0)：总数不能为负；如果允许负数，分页按钮逻辑没有意义。
  • limit: int = PydanticField(ge=1, le=100)：响应也承诺页大小边界，与请求参数约束保持一致。
  • offset: int = PydanticField(ge=0)：偏移量从 0 开始，负数没有稳定业务语义。
@router.get(
    "",
    response_model=SummaryTaskListResponse,
    response_description="A paginated task-list response envelope.",
)
def list_summary_tasks(...) -> SummaryTaskListResponse:
    return list_summary_task_records(...)
  • response_model 是对外契约，不是装饰。FastAPI 会用它生成文档、过滤字段、声明 schema。
  • response_description 明确 200 响应含义；OpenAPI 中会显示这不是普通数组。
  • 返回类型注解让读代码的人也知道该函数不返回 list。
return SummaryTaskListResponse(
    items=items,
    count=count,
    limit=limit,
    offset=offset,
)

service 层显式构造响应模型，避免临时字典字段拼错。若把 count 写成 total，Pydantic/测试会尽早暴露契约偏移。

④ 核心机制运行全流程拆解

GET /tasks?status=completed&limit=5&offset=0
  -> router 校验 query 参数
  -> service 查询当前页 items
  -> service 查询筛选后总数 count
  -> 构造 SummaryTaskListResponse
  -> FastAPI 按 response_model 序列化
  -> JSON 响应：
     {
       "items": [...],
       "count": 18,
       "limit": 5,
       "offset": 0
     }
  -> 前端读取 page.items 渲染列表
  -> 前端用 page.count/page.offset 计算分页按钮

⑤ 重难点、易混淆点对比辨析

概念正确理解常见误区
裸数组只适合极简单列表以为所有列表 API 都该返回数组
Envelope数据 + 元数据的响应对象以为只是多包一层无意义对象
items当前页数据把它当全部数据
count当前筛选条件下总数写成当前页长度
响应变更调用方契约变更当作后端内部小重构

⑥ 开发常见报错+坑点+解决方案

  • page is not iterable：前端仍把响应当数组遍历。解决：遍历 page.items。
  • 下一页按钮误禁用：把 count 写成 len(items)。解决：用独立 count 查询计算总数。
  • OpenAPI 显示数组但实际返回对象：忘改 response_model。解决：设置为 SummaryTaskListResponse 并加契约测试。
  • 字段名漂移：后端临时返回 total、前端读 count。解决：统一模型和测试保护字段名。
  • 把元数据放到每个 item 内：重复且语义错误。解决：页级元数据放 envelope 顶层。

⑧ 课后记忆习题+标准答案

1. 为什么列表接口常用 envelope 而不是裸数组？

因为分页、筛选和 UI 控件需要总数、页大小、偏移量等页级元数据，裸数组只能表达当前页有哪些 item。

2. 从数组改成 envelope 为什么是破坏性变更？

顶层 JSON 类型从 array 变成 object，旧客户端的遍历和索引代码会直接失效。

3. count 与 items.length 的区别是什么？

items.length 是当前页条数；count 是当前筛选条件下的总记录数。

4. response_model 在本节保护了什么？

它保护成功响应的外部 schema，让文档、序列化和测试围绕同一响应结构工作。

⑨ 面试高频真题+解析

题 1：设计分页列表接口时响应体应该包含什么？

至少包含当前页数据和分页元信息，例如 items、count、limit、offset；游标分页可换成 next_cursor/has_more。重点是让客户端不靠猜测判断翻页状态。

题 2：为什么后端返回格式变化要通知前端？

JSON 形状是前后端共享契约。字段改名、顶层类型变化、必填字段变化都会破坏调用方解析逻辑，需要版本、迁移或同步发布。

题 3：响应元数据放 body 还是 header？

两者都可行。业务 JSON API 常放 body，便于模型化和文档化；REST 风格分页链接可用 Link header。选择取决于团队规范和客户端便利性。
