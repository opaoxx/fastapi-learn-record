Lesson 0025

列表响应信封：不只返回数组，还返回分页元数据

分页和筛选让后端可以只返回一部分任务，但前端还需要知道“这部分数据处在整个结果集的什么位置”。响应信封就是为这件事服务的 API 契约。

0. 本节学习目标

  • 理解为什么很多列表接口返回对象，而不是裸数组。
  • 掌握 items、count、limit、offset 的职责分工。
  • 看懂 SummaryTaskListResponse 如何把列表响应建模成 Pydantic 契约。
  • 理解响应形状变化为什么是 API 破坏性变更。
  • 能解释前端分页按钮为什么需要响应元数据。

1. 背景：裸数组只能表达“有什么”，不能表达“在哪里”

裸数组响应长这样：

[
  {"id": 3, "status": "completed"},
  {"id": 2, "status": "completed"}
]

它能告诉前端当前拿到了哪些任务，却不能回答这些问题：

  • 符合当前筛选条件的任务一共有多少条？
  • 这一页从第几条开始？
  • 服务端实际接受的 limit 是多少？
  • 还有没有下一页？

如果前端只展示静态列表，裸数组勉强够用；如果前端要做任务历史、分页按钮、总数提示、状态筛选，它就不够了。

2. 核心概念：响应信封把数据和元数据放在一起

新的响应形状：

{
  "items": [
    {"id": 3, "status": "completed"},
    {"id": 2, "status": "completed"}
  ],
  "count": 18,
  "limit": 2,
  "offset": 0
}

这里可以分成两类字段：

  • items 是业务数据：当前页的任务记录。
  • count、limit、offset 是元数据：描述当前页和整个结果集的关系。

“信封”这个说法的意思是：真正的信件内容是 items，信封上的地址、页码、总数等说明信息则帮助调用方正确使用这批数据。

3. 为什么不用响应头放这些信息

有些 API 会把分页元数据放到 HTTP 响应头里。但对初学项目和许多业务 API 来说，放在 JSON body 里更直观：

  • Swagger UI 和前端调试工具里更容易看到。
  • 前端只解析一次 JSON 就能拿到数据和元数据。
  • Pydantic 模型可以直接描述整个响应结构。

所以本课程选择 body envelope。重点不是“世界上只有这一种做法”，而是让 API 契约清晰、可测试、可文档化。

4. 代码精读一：响应模型不是装饰品

class SummaryTaskListResponse(BaseModel):
    items: list[SummaryTaskRead]
    count: int
    limit: int
    offset: int

FastAPI 的 response_model 会参与文档生成、响应转换和过滤。这个模型一旦声明，就表示后端对调用方承诺：任务列表响应会有这些字段，并且字段类型应符合声明。

SummaryTaskRead 负责单个任务的形状；SummaryTaskListResponse 负责列表整体的形状。这样模型层次更清楚：

SummaryTaskRead          -> 一条任务
list[SummaryTaskRead]    -> 当前页任务数组
SummaryTaskListResponse  -> 当前页 + 分页元数据

5. 代码精读二：router 的合约变化

@router.get("", response_model=SummaryTaskListResponse)
def list_summary_tasks(...) -> SummaryTaskListResponse:
    return list_summary_task_records(
        session=session,
        task_status=task_status,
        offset=offset,
        limit=limit,
    )

旧合约是：

response_model=list[SummaryTaskRead]

新合约是：

response_model=SummaryTaskListResponse

这个变化会直接影响调用方。前端原来可能写：

renderTasks(response);

现在必须写：

renderTasks(response.items);

这就是为什么响应模型不是后端内部实现细节，而是前后端共享契约。

6. 前端如何从信封构建分页 UI

const page = await fetch("/tasks?limit=10&offset=0").then(r => r.json());

renderTasks(page.items);
renderMeta(`显示 ${page.items.length} 条，共 ${page.count} 条`);

const hasPrevious = page.offset > 0;
const hasNext = page.offset + page.items.length < page.count;

这里 items 决定渲染几行任务；count 决定总数提示；offset 和 limit 决定上一页、下一页按钮。

第 27 课已经把这个思想落实到静态前端页面中：任务历史区域读取 page.items，同时显示 count、limit 和 offset。

7. 为什么这是破坏性变更

对后端来说，从裸数组改成信封对象可能只是几行代码。但对调用方来说，JSON 结构变了，这就是破坏性变更。

破坏性变更的判断标准不是“后端改得多不多”，而是“调用方原来的代码是否还能工作”。如果前端还在写 response[0]，它会失败；因为现在第一条任务在 response.items[0]。

真实项目中常见处理方式包括：

  • 提前沟通并同步升级前端。
  • 提供兼容期。
  • 使用新路径，例如 /tasks/v2。
  • 用测试和 OpenAPI 文档明确新契约。

8. 测试为什么必须跟着改

response = client.get("/tasks?limit=2", headers=API_KEY_HEADERS)
page = response.json()

assert set(page.keys()) == {"items", "count", "limit", "offset"}
assert len(page["items"]) <= 2
assert page["limit"] == 2

测试在这里不是只验证“接口能跑”。它在守 API 契约：响应必须是信封对象，而不是裸数组；任务记录必须在 items 里；元数据必须存在。

9. 常见错误与诊断

错误一：前端仍按数组读取

旧代码 response[0] 会失效。新结构要读 response.items[0]。

错误二：信封字段命名随意

字段名应该稳定。items/count/limit/offset 简短清楚，适合长期维护。

错误三：count 含义不清

count 应表示当前筛选条件下的总数，不是当前页长度。

错误四：只改代码不改文档

响应结构是对外契约，课程、速查、README 和测试都应同步。

10. 课堂实验

请求：

GET /tasks?limit=1&offset=0
X-API-Key: dev-secret-key

回答：

  1. items 是数组还是对象？
  2. count 是否可能大于 items.length？
  3. 如果要渲染第一条任务，应该读取 page[0] 还是 page.items[0]？
  4. 如果 offset + items.length < count，前端应该允许点击下一页吗？

11. 小测：先回忆再展开

1. 为什么列表接口常返回对象而不是裸数组？

因为对象可以同时承载当前页数据和分页元数据，裸数组只能表达当前有哪些项。

2. items 和 count 的职责有什么不同？

items 是当前页记录，count 是当前筛选条件下的总记录数。

3. 为什么响应信封升级属于破坏性变更？

因为 JSON 结构变了，旧调用方如果仍按数组读取就会失败。

4. response_model 在这里的意义是什么？

它声明并文档化响应契约，让 FastAPI 按模型处理输出数据。

12. 本节总结

本节真正讲的是 API 契约设计。裸数组适合非常简单的列表；一旦列表需要分页、筛选、总数、上一页、下一页，就应该用响应信封把业务数据和元数据一起返回。
