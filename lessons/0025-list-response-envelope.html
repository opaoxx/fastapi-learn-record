Lesson 0025

列表响应信封：不只返回数组，还返回分页元数据

上一节的 GET /tasks 已经能分页和筛选，但前端还不知道总共有多少条。本节把列表响应从数组升级成响应信封。

0. 本节学习目标

  • 理解为什么列表接口常返回对象，而不是裸数组。
  • 看懂 SummaryTaskListResponse 响应模型。
  • 理解 items、count、limit、offset 的职责。
  • 理解 API 合约升级为什么要同步更新测试和文档。

1. 裸数组有什么问题

原来的任务列表响应长这样：

[
  {"id": 3, "status": "completed"},
  {"id": 2, "status": "completed"}
]

它能展示任务，但前端不知道：

  • 总共有多少条符合条件的任务。
  • 当前使用的 limit 是多少。
  • 当前页从哪个 offset 开始。
  • 后面是否还可能有下一页。

所以真实项目的列表接口经常会返回一个对象，把数据和分页元信息包起来。

2. 新响应形状

{
  "items": [
    {"id": 3, "status": "completed"},
    {"id": 2, "status": "completed"}
  ],
  "count": 18,
  "limit": 2,
  "offset": 0
}

这个外层对象就像一个信封：items 是信里的任务列表，其他字段是这封信的说明信息。

3. 响应模型

class SummaryTaskListResponse(BaseModel):
    items: list[SummaryTaskRead]
    count: int
    limit: int
    offset: int

SummaryTaskRead 仍然负责单个任务长什么样；SummaryTaskListResponse 负责整个列表响应长什么样。这样单项模型和列表模型分工清楚。

4. router 的变化

@router.get("", response_model=SummaryTaskListResponse)
def list_summary_tasks(...) -> SummaryTaskListResponse:
    return list_summary_task_records(
        session=session,
        task_status=task_status,
        offset=offset,
        limit=limit,
    )

注意 response_model 不再是 list[SummaryTaskRead]。这代表 API 合约发生了变化，调用方也要从 response[0] 改成 response.items[0]。

5. 前端如何使用

const page = await fetch("/tasks?limit=10&offset=0").then(r => r.json());

renderTasks(page.items);
renderPagination({
  count: page.count,
  limit: page.limit,
  offset: page.offset,
});

如果 offset + items.length < count，前端就可以显示“下一页”。如果 offset 大于 0，就可以显示“上一页”。

6. 为什么这是破坏性变更

对后端来说，这只是换了一个响应模型。但对已经接入这个接口的前端来说，这是破坏性变更，因为 JSON 结构变了。

学习项目里可以直接升级；真实项目里通常要考虑版本号、兼容期或新接口路径，例如 /tasks/v2。

7. 课堂练习

请求：

GET /tasks?limit=1&offset=0

回答三个问题：

  1. items 里有几条？
  2. count 是否可能大于 items.length？
  3. 下一页的 offset 应该是多少？

8. 小测：先回忆再展开

1. items 表示什么？

表示当前页实际返回的任务记录列表。

2. count 表示什么？

表示符合当前筛选条件的总记录数。

3. 为什么响应结构变化要更新测试？

因为测试是 API 合约的守门员，响应 JSON 结构变了，断言也必须跟着变。

9. 本节总结

本节把任务列表响应升级为响应信封。它让前端不仅能展示当前页任务，还能构建分页控件和任务历史页。
