真实 AI Provider 边界：Protocol、配置对象与可替换客户端

真实 AI provider 是不稳定外部依赖：会超时、限流、变更响应格式、产生费用。后端不能让 router 直接认识厂商 SDK，而要把外部世界封装进 AI client 边界。本节把 AIClient 协议、AIClientConfig 配置对象和 get_ai_client() 工厂串成可替换、可测试、可演进的 provider 入口。

① 本节核心知识框架

AI Provider Boundary
├─ 外部问题：timeout / rate limit / 5xx / invalid response / cost
├─ 内部协议：AIClient(Protocol)
│  ├─ predict_sentiment(text, mode) -> PredictionResult
│  └─ summarize(text) -> str
├─ 配置对象：AIClientConfig
│  ├─ provider
│  ├─ timeout_seconds
│  ├─ max_attempts
│  └─ retry_* 参数
├─ 适配器：DemoAIClient(config)
├─ 工厂函数：get_ai_client() -> AIClient
├─ FastAPI 注入：Depends(get_ai_client)
└─ 测试替身：FixedAIClient / FailingAIClient

② 核心概念底层原理

边界层的职责是翻译：把项目内部的预测/摘要请求翻译成 provider 调用，把 provider 原始响应翻译成 PredictionResult 或摘要字符串，把 provider 失败翻译成稳定 AIClientError。边界层越清晰，router 越不需要知道厂商细节。

Protocol 是结构化类型：对象不需要继承 AIClient，只要实现协议要求的方法，就可被当作 AI client 使用。这非常适合测试替身和多 provider adapter，因为我们关心行为形状，而不是继承树。

AIClientConfig 把外部调用策略集中化。timeout、max_attempts、retry delay 不是随手写在 provider 函数里的魔法数字，而是 AI 边界的一部分。生产里这些参数决定响应时间、成本、重试风暴风险和错误恢复策略。

③ 全套代码逐行精读解析

@dataclass(frozen=True)
class AIClientConfig:
    provider: AIProvider = "demo"
    timeout_seconds: float = 10.0
    max_attempts: int = 1
    retry_base_delay_seconds: float = 0.1
    retry_max_delay_seconds: float = 2.0
    retry_jitter_seconds: float = 0.0
  • @dataclass(frozen=True)：配置对象不可变，避免请求处理中被误改。
  • provider：当前只允许 demo，但字段已经为真实 provider 切换留位置。
  • timeout_seconds：外部调用最大等待时间；没有 timeout 会让请求无限挂住。
  • max_attempts：重试次数上限；不设上限会放大 provider 故障。
  • retry_*：后续课程用于退避、封顶和抖动，提前放入边界配置。
class AIClient(Protocol):
    def predict_sentiment(self, text: str, mode: str) -> PredictionResult:
        ...

    def summarize(self, text: str) -> str:
        ...
  • Protocol 声明能力形状，不实现逻辑。
  • predict_sentiment 服务同步预测接口。
  • summarize 服务后台摘要任务。
  • ... 表示协议占位；如果在运行时直接调用协议方法，本身不会产生业务结果。
def build_ai_client_config() -> AIClientConfig:
    settings = get_settings()
    return AIClientConfig(
        provider=settings.ai_provider,
        timeout_seconds=settings.ai_timeout_seconds,
        max_attempts=settings.ai_max_attempts,
        retry_base_delay_seconds=settings.ai_retry_base_delay_seconds,
        retry_max_delay_seconds=settings.ai_retry_max_delay_seconds,
        retry_jitter_seconds=settings.ai_retry_jitter_seconds,
    )

这段代码把环境变量配置转成服务层配置对象。router 不读取环境变量，provider adapter 不散落 get_settings()，配置流向变得可测试、可替换。

def get_ai_client() -> AIClient:
    return DemoAIClient(config=build_ai_client_config())

get_ai_client() 是 FastAPI 依赖入口。今天返回 demo adapter，未来可根据 config.provider 返回真实 provider adapter。调用方依赖的是 AIClient 协议，不依赖 DemoAIClient 具体类。

④ 核心机制运行全流程拆解

POST /predict
  -> Depends(get_settings)
  -> Depends(get_ai_client)
       -> build_ai_client_config()
       -> DemoAIClient(config)
  -> run_prediction()
       -> ai_client.predict_sentiment(text, mode)
       -> PredictionResult
  -> build_prediction_response()
  -> PredictionResponse JSON

测试路径
  -> app.dependency_overrides[get_ai_client] = override_ai_client
  -> router 收到 FixedAIClient / FailingAIClient
  -> 不访问真实 provider
  -> 稳定验证成功与失败路径

⑥ 开发常见报错+坑点+解决方案

  • 测试慢且偶发失败：测试访问真实 AI provider。解决：用 dependency_overrides 注入替身 client。
  • provider 错误泄露到前端：adapter 未统一错误映射。解决：边界层转换为 AIClientError。
  • 切 provider 改动范围大：router 依赖具体类。解决：router 注入 AIClient 协议。
  • 请求长期卡住：没有 timeout。解决：把 timeout_seconds 放入配置并由 adapter 使用。
  • 重试无限放大故障：没有最大尝试次数。解决：配置 max_attempts 上限。

⑧ 课后记忆习题+标准答案

1. 为什么真实 AI provider 不应直接写在 router 里？

router 会耦合厂商协议、异常、配置和测试细节，导致换 provider、写测试和统一错误处理都变困难。

2. AIClient Protocol 的价值是什么？

它让调用方依赖行为形状，具体实现可以是 demo、真实 provider 或测试替身。

3. timeout 为什么属于 provider 边界？

timeout 决定外部调用最多占用请求多久，是保护 API 延迟和资源的关键策略。

4. 测试替身为什么不必继承 DemoAIClient？

因为 Protocol 关注方法形状，只要实现同名方法并返回符合契约的结果即可。

⑨ 面试高频真题+解析

题 1：后端如何接入可替换 AI provider？

定义内部协议，写 provider adapter 负责请求/响应/错误翻译；router 通过依赖注入拿协议对象；配置集中在 Settings/Config；测试用替身覆盖依赖，不直接访问真实 provider。

题 2：为什么外部 AI 调用要封装 timeout 和 retry？

外部网络和模型服务不可靠。timeout 防止无限等待，有限 retry 应对临时失败，但必须有上限和退避，避免放大故障。

题 3：Protocol 和抽象基类怎么选？

Protocol 更轻，适合 Python 结构化类型和测试替身；抽象基类适合需要共享实现或强制显式继承的场景。
