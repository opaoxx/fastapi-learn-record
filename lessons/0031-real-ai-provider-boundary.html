Lesson 0031

真实 AI Provider 边界：把不稳定的外部调用关进服务层

前面的 DemoAIClient 只是本地模拟。真实项目里，AI 能力通常来自外部服务：它会超时、限流、返回 5xx、改响应格式，甚至在你演示项目时刚好不可用。本节学习如何先设计边界，再考虑接入真实 provider。

0. 本节学习目标

  • 理解为什么外部 AI provider 不能直接写在 router 里。
  • 理解边界、协议、适配器、配置、超时、重试、错误映射和测试替身。
  • 学会用 Python Protocol 描述 AI 客户端必须具备的能力。
  • 学会把 provider、timeout、max_attempts 放进配置，而不是散落成魔法数字。
  • 理解 FastAPI 的 Depends 和 dependency_overrides 如何让外部服务可替换、可测试。

1. 背景：AI 服务最容易把项目拖乱

普通 CRUD 接口主要依赖数据库，失败模式相对清楚。AI 服务接口多了一层不稳定性：

  • 网络请求可能超时。
  • provider 可能返回 429、500、503。
  • 模型响应可能为空、格式不符合预期，或字段含义变化。
  • 不同 provider 的请求参数和响应结构不同。
  • 测试时不能每次都真的花钱调用外部模型。

如果把真实 AI 调用直接写在 routers/predictions.py 或 task_worker.py 里，后面会很痛苦：路由会越来越胖，测试会越来越慢，错误处理会到处复制，换 provider 时要改很多地方。

所以本节的设计原则是：router 只认识“项目自己的 AI 客户端边界”，不认识具体厂商。

2. 核心概念：边界是把外部世界翻译成内部语言

真实 provider 讲的是外部语言：HTTP 状态码、厂商错误码、token 限制、模型名、响应 JSON。我们的项目应该讲内部语言：

predict_sentiment(text, mode) -> PredictionResult
summarize(text) -> str
AIClientError(error_code, message)

这层翻译边界的职责是：

  • 把项目请求转换成 provider 请求。
  • 把 provider 响应转换成项目内部结果。
  • 把 provider 失败转换成稳定的 AIClientError。
  • 集中处理 timeout、retry、provider 配置和日志。
  • 让测试可以用假客户端替代真实 provider。

边界设计得好，后面接 OpenAI、Claude、本地模型或公司内部模型，router 都不用大改。

3. 术语表：先把工程词说清楚

术语 含义 本项目例子
Provider 提供 AI 能力的外部或内部服务。 现在是 demo，未来可以是真实模型服务。
Boundary 系统内部和外部服务之间的隔离层。 services/ai_clients.py
Protocol 按“方法形状”定义能力，不要求继承同一个父类。 AIClient
Adapter 把某个 provider 的 API 翻译成项目内部接口。 DemoAIClient 是本地 demo adapter。
Timeout 外部调用最多等多久。 FIRST_API_AI_TIMEOUT_SECONDS
Retry 遇到特定临时失败时是否重试。 FIRST_API_AI_MAX_ATTEMPTS
Test double 测试中替代真实外部服务的假对象。 FixedAIClient、FailingAIClient

4. 官方资料里的关键原则

FastAPI 的依赖系统适合把共享组件注入到路径操作函数中，测试时可以用 app.dependency_overrides 替换依赖。这正好适合替换 AI provider。

Python 的 typing.Protocol 支持结构化子类型：只要对象实现了协议要求的方法，类型检查就可以把它视为该协议的实现，而不要求显式继承。

HTTPX 的 timeout 文档强调，网络请求必须有超时边界；transport 文档也提到连接层失败和连接超时可以在 transport 层重试。真实 AI provider 本质上也是 HTTP 外部依赖，所以 timeout 和 retry 必须提前进入设计。

5. 本节项目改动概览

  • 新增 AIClientConfig，集中保存 provider、timeout_seconds、max_attempts。
  • 新增 AIClient Protocol，声明项目需要的 AI 能力。
  • 让 DemoAIClient 接收配置，并继续返回稳定结果。
  • 在 Settings 和 .env.example 中加入 AI provider 配置项。
  • 让 /predict 和后台摘要任务依赖 AIClient，而不是硬绑定 DemoAIClient。
  • 新增测试，确认配置留在 AI 客户端边界内。

注意：本节还没有真正联网调用 AI provider。我们先把边界打好，再接真实服务。这个顺序很重要。

6. 代码精读一：AIClientConfig

@dataclass(frozen=True)
class AIClientConfig:
    provider: AIProvider = "demo"
    timeout_seconds: float = 10.0
    max_attempts: int = 1

这三个字段看起来简单，但它们代表外部调用的三个关键决策：

  • provider：当前选择哪个 provider。
  • timeout_seconds：最多等多久，避免请求无限挂住。
  • max_attempts：最多尝试几次，为后续重试策略留位置。

配置对象的价值不是“现在立刻用满”，而是提前把未来真实 provider 的不稳定因素放在一个清楚的位置。

7. 代码精读二：Protocol 定义能力形状

class AIClient(Protocol):
    def predict_sentiment(self, text: str, mode: str) -> PredictionResult:
        ...

    def summarize(self, text: str) -> str:
        ...

这个协议没有实现任何逻辑，只定义“项目需要 AI 客户端会做什么”。它表达两件事：

  • 同步预测接口需要 predict_sentiment。
  • 后台摘要任务需要 summarize。

测试里的 FixedAIClient 没有继承 AIClient，但只要它实现了这两个方法，就能作为替身使用。这就是结构化类型的好处：关注行为形状，而不是继承关系。

8. 代码精读三：DemoAIClient 变成一个 adapter

class DemoAIClient:
    positive_words = {"good", "great", "love", "pleasant", "nice", "happy"}
    failure_marker = "simulate-ai-failure"

    def __init__(self, config: AIClientConfig | None = None) -> None:
        self.config = config or AIClientConfig()

DemoAIClient 仍然是本地模拟，但它现在更像一个 provider adapter。未来如果新增 OpenAIClient，它也应该实现同样的 AIClient 协议，把真实 API 响应翻译成 PredictionResult 或摘要字符串。

预测结果仍然返回：

PredictionResult(
    label="positive",
    score=0.91,
    source="demo-ai-client",
)

也就是说，内部响应契约没有变，变的是服务层的可替换性。

9. 代码精读四：配置从 Settings 进入边界

class Settings(BaseSettings):
    app_name: str = "First FastAPI API"
    environment: str = "development"
    api_key: str = "dev-secret-key"
    ai_provider: Literal["demo"] = "demo"
    ai_timeout_seconds: float = Field(default=10.0, gt=0)
    ai_max_attempts: int = Field(default=1, ge=1, le=5)

这里把 AI 配置放进环境变量系统：

FIRST_API_AI_PROVIDER="demo"
FIRST_API_AI_TIMEOUT_SECONDS="10"
FIRST_API_AI_MAX_ATTEMPTS="1"

真实项目里，不同环境可能使用不同 provider、timeout 和尝试次数。开发环境可以用 demo，测试环境用 fake，生产环境用真实 provider。

10. 代码精读五：get_ai_client 是工厂函数

def build_ai_client_config() -> AIClientConfig:
    settings = get_settings()
    return AIClientConfig(
        provider=settings.ai_provider,
        timeout_seconds=settings.ai_timeout_seconds,
        max_attempts=settings.ai_max_attempts,
    )


def get_ai_client() -> AIClient:
    return DemoAIClient(config=build_ai_client_config())

get_ai_client 现在可以理解为 provider 工厂。今天它只返回 DemoAIClient，以后可以变成：

if config.provider == "demo":
    return DemoAIClient(config)
if config.provider == "real":
    return RealProviderClient(config)

重要的是：router 不需要知道这些选择逻辑。router 只通过 Depends(get_ai_client) 拿到一个会 predict_sentiment 的对象。

11. 代码精读六：router 依赖协议，不依赖具体类

def predict(
    payload: PredictionRequest,
    ai_client: Annotated[AIClient, Depends(get_ai_client)],
) -> PredictionResponse:

这一行是本节的关键变化。旧写法是依赖 DemoAIClient，新写法是依赖 AIClient。这让路由表达的是“我需要一个具备 AIClient 能力的对象”，而不是“我只能用 DemoAIClient”。

错误处理仍然保持稳定：

except AIClientError as exc:
    raise HTTPException(
        status_code=503,
        detail={
            "error_code": exc.error_code,
            "message": exc.message,
        },
    )

这就是边界的另一个价值：无论真实 provider 内部失败长什么样，API 调用方看到的失败形状都稳定。

12. 测试：替身对象不需要继承具体类

class FixedAIClient:
    def predict_sentiment(self, text: str, mode: str) -> PredictionResult:
        return PredictionResult(
            label="positive",
            score=0.99,
            source="fixed-test-client",
        )

    def summarize(self, text: str) -> str:
        return "fixed summary"

测试通过 app.dependency_overrides[get_ai_client] 把真实依赖替换成 FixedAIClient。它不需要联网，不需要真实模型，不需要等待 provider 响应。

这就是外部服务测试的基本策略：接口层测试验证“我们的系统如何处理成功和失败”，而不是每次都真的打外部 API。

13. 真实 provider 接入时的失败分类

失败类型 边界层应该怎么处理
连接失败 映射成 AIClientError("ai_provider_connect_error")，必要时按策略重试。
请求超时 立刻停止等待，返回稳定错误；不要让 API 请求无限挂起。
429 限流 通常不要盲目重试；要考虑退避、队列或提示稍后再试。
5xx 临时故障 可以考虑有限重试；超过次数后映射为可恢复失败。
响应格式变化 在 adapter 内校验并转换，不让脏数据进入 router。

HTTPX 文档中 timeout 和 transport retry 的设计提醒我们：网络不可靠是常态。AI 服务更应该把这个现实写进边界层。

14. 课堂实验

运行测试：

.\.venv\Scripts\python.exe -m pytest tests\test_ai_client_dependency.py -q

观察三件事：

  1. FixedAIClient 可以替换 get_ai_client。
  2. FailingAIClient 抛出的 AIClientError 会变成稳定 503 响应。
  3. DemoAIClient 可以携带 AIClientConfig，但路由不需要知道配置细节。

15. 常见错误与诊断

错误一：在 router 里直接写 provider SDK

router 会变胖，测试会依赖网络，换 provider 时到处改。

错误二：没有 timeout

外部调用可能长期挂住，用户请求也跟着卡住。

错误三：把 provider 错误原样透给前端

前端会依赖厂商细节，也可能泄露不该暴露的信息。

错误四：测试真的调用外部 AI

测试会慢、贵、不稳定。应该用依赖覆盖和测试替身。

16. 面试视角：如何回答“AI 服务怎么接入后端？”

我不会把外部 AI provider 直接写进 FastAPI router，而是放在服务层 adapter 后面。router 依赖项目自己的 AIClient 协议，provider adapter 负责把内部请求转换为外部请求，把外部响应转换为内部结果，并把 timeout、限流、5xx、格式错误等失败映射为稳定的业务错误。测试中我会用 FastAPI dependency_overrides 替换 AI 客户端，避免真实联网调用。

17. 小测：先回忆，再展开

1. 为什么 router 不应该直接调用真实 AI provider？

因为会让路由耦合外部细节，导致测试慢、不稳定，换 provider 时修改范围变大。

2. AIClient Protocol 解决了什么问题？

它定义项目需要的 AI 能力形状，让具体实现和测试替身都能被替换使用。

3. 为什么 timeout 应该进入 AI 客户端配置？

因为外部调用可能无限等待，timeout 是保护 API 响应时间和系统资源的边界。

4. 为什么测试里应该用 dependency_overrides？

因为它能替换外部服务依赖，让测试快速、稳定、便宜。

18. 本节总结

本节把 AI 能力从“一个 demo 类”推进到“可替换 provider 边界”。你现在应该能解释：真实 AI provider 是不稳定外部依赖，应该被封装在服务层 adapter 后面；FastAPI router 只依赖项目自己的协议；timeout、retry 和错误映射属于边界层职责；测试用替身对象验证我们自己的行为。
