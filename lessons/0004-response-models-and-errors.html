响应模型、状态码与错误处理：API 对外契约的另一半

请求模型约束输入，响应模型约束输出；状态码表达结果类别，错误响应表达失败原因。本节从当前 items CRUD 接口出发，掌握后端项目里最容易被新手写乱的响应边界。

① 本节核心知识框架

响应与错误
├─ response_model：成功响应结构契约
├─ HTTP status code：请求结果类别
│  ├─ 200 OK：查询/更新成功
│  ├─ 201 Created：创建成功
│  ├─ 204 No Content：删除成功且无响应体
│  ├─ 404 Not Found：合法请求但资源不存在
│  └─ 422 Validation Error：参数/请求体违反模型约束
├─ HTTPException：主动中断并返回业务错误
├─ Pydantic/SQLModel：输入校验与输出序列化
└─ 当前项目落点：ItemRead / ItemCreate / ItemUpdate / read_item / create_item

② 核心概念底层原理

HTTP 响应由状态行、响应头和响应体组成。状态码是机器可判断的结果类别；响应体是业务数据或错误细节。只返回 JSON 不设计状态码，会让前端和测试无法可靠区分成功、客户端错误和服务端错误。

response_model 是输出白名单。FastAPI 会按响应模型对返回对象做序列化，文档也按它生成。生产项目中响应模型常用于隐藏数据库内部字段、统一字段类型、避免偶然泄漏。

HTTPException 是控制流，不是普通异常泄漏。它告诉 FastAPI 立即结束当前请求并生成指定状态码的 HTTP 响应。与未捕获异常不同，HTTPException 是预期内业务分支。

404 与 422 的分层。422 是进入业务前的契约校验失败；404 是契约合法但资源不存在。这个区分是面试高频，也是 API 设计基本功。

③ 全套代码逐行精读解析

class ItemCreate(ItemBase):
    name: str = Field(min_length=2, max_length=80, schema_extra={"examples": ["FastAPI Project Notes"]})
    category: ItemCategory = "course"
    price: float = Field(ge=0, schema_extra={"examples": [29.9]})
    in_stock: bool = True

class ItemRead(ItemBase):
    id: int
  • ItemCreate：创建请求模型。它不包含 id，因为 ID 应由数据库生成，不应由客户端指定。
  • name 长度约束：避免空名称、单字符脏数据和过长文本进入数据库。
  • category: ItemCategory：限制分类为 book/course/tool，防止任意字符串破坏筛选逻辑。
  • price: Field(ge=0)：价格不能为负。没有该约束，业务层和前端都要额外处理非法价格。
  • ItemRead：读取响应模型，包含数据库生成的 id。这体现输入输出模型分离。
@router.post(
    "",
    response_model=ItemRead,
    status_code=status.HTTP_201_CREATED,
    dependencies=[Depends(require_api_key)],
)
def create_item(
    payload: ItemCreate,
    session: Annotated[Session, Depends(get_session)],
) -> Item:
    item = Item.model_validate(payload)
    session.add(item)
    session.commit()
    session.refresh(item)
    return item
  • @router.post("")：在 /items 下注册创建接口。
  • response_model=ItemRead：创建成功后返回带 id 的读模型。
  • status_code=201：明确表达“资源已创建”。若省略，默认是 200，语义不够准确。
  • dependencies=[Depends(require_api_key)]：要求 API Key。若缺失，写接口会返回 401。
  • payload: ItemCreate：请求体必须符合创建模型。
  • session：数据库会话由依赖注入提供。
  • Item.model_validate(payload)：把输入模型转换为表模型。若直接信任字典，容易绕过模型边界。
  • session.add(item)：把对象加入事务跟踪。
  • session.commit()：提交事务，写入数据库并生成主键。
  • session.refresh(item)：从数据库刷新对象，拿到生成后的 id。
  • return item：FastAPI 按 ItemRead 输出。
if item is None:
    raise HTTPException(status_code=status.HTTP_404_NOT_FOUND, detail=f"Item {item_id} was not found.")

这两行是业务错误边界：ID 参数合法、数据库查询执行成功，但目标资源不存在，所以返回 404。若返回 None，会让响应模型校验失败，变成服务端问题。

④ 核心机制运行全流程拆解

POST /items
  -> API Key 依赖校验
  -> ItemCreate 校验请求体
  -> get_session() 提供数据库会话
  -> Item.model_validate(payload)
  -> session.add + commit + refresh
  -> ItemRead 序列化
  -> 201 + JSON

GET /items/999
  -> item_id 类型和 ge=1 通过
  -> session.get(Item, 999)
  -> None
  -> raise HTTPException(404)
  -> 404 + {"detail":"Item 999 was not found."}

⑤ 重难点、易混淆点对比辨析

  • 201 vs 200：创建资源用 201 更精确；查询和普通更新常用 200。
  • 204 vs 返回空 JSON：删除成功用 204 时不应再返回响应体；返回 {} 更适合 200。
  • 401 vs 422：缺 API Key 是认证失败 401；请求体字段非法是 422。
  • HTTPException vs Python ValueError：前者是可控 HTTP 错误；后者未处理时通常变成 500。

⑥ 开发常见报错+坑点+解决方案

  • POST /items 返回 401：缺少 X-API-Key: dev-secret-key。解决：按 README 配置请求头。
  • POST /items 返回 422：name 太短、price 为负、category 非法。解决：按 ItemCreate 字段约束传参。
  • 创建后 id 为 None：忘记 commit 或 refresh。解决：写入后刷新数据库生成字段。
  • 找不到资源却返回 500：没有处理 None。解决：显式抛 HTTPException(404)。
  • 响应字段比预期多：直接返回内部模型且未使用响应模型。解决：定义 ItemRead 作为输出契约。

⑦ 生产环境 vs 教学环境 核心差异

  • 错误体结构：教学可用 FastAPI 默认；生产常定义统一 error_code/message/request_id。
  • 鉴权：教学 API Key 足够展示依赖；生产要用 OAuth2/JWT/会话/权限模型。
  • 事务：教学每次简单 commit；生产要考虑事务边界、幂等性、唯一约束冲突和回滚。
  • 数据暴露：教学模型字段少；生产必须严格区分 DB model、domain model、request DTO、response DTO。

⑧ 课后记忆习题+标准答案

1. 为什么创建商品请求模型不包含 id？

id 是数据库生成的资源标识，客户端不应在创建请求中指定。

2. GET /items/abc 和 GET /items/999 分别是什么错误？

abc 无法转 int，是 422；999 合法但资源不存在，是 404。

3. session.refresh(item) 的作用是什么？

提交后从数据库刷新对象，拿到数据库生成或更新后的字段，例如自增 id。

⑨ 面试高频真题+解析

1. 为什么要区分请求模型和响应模型？

二者面向不同边界：请求模型限制外部输入，响应模型控制对外输出，分离后可以避免字段泄漏并支持独立演进。

2. 什么时候应该抛出 404？

当请求路径和参数合法，但业务资源不存在，例如数据库按 ID 查询为空。

3. 为什么 204 删除接口不返回 body？

HTTP 204 的语义是成功但无内容，客户端不应期待响应体；这能减少歧义并符合标准语义。
