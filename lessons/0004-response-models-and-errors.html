Lesson 0004

响应模型、状态码和错误处理：接口要会说清楚结果

请求进来以后，后端不仅要“算出结果”，还要用稳定的响应结构和正确的 HTTP 状态码告诉调用者发生了什么。这是后端工程感很重要的一步。

0. 本节学习目标

  • 理解 response_model 的作用：声明、校验和文档化响应数据。
  • 理解 200、201、404、422 的基本语义。
  • 学会用 HTTPException 主动返回业务错误。
  • 理解为什么错误处理对普通后端和 AI 服务都很重要。

1. 背景知识：响应不只是 return 一个字典

一个 HTTP 响应至少包含状态码和响应体：

HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 2,
  "name": "Python Backend Handbook",
  "category": "book",
  "price": 39.9,
  "in_stock": true
}

状态码告诉客户端“这次请求整体上发生了什么”，响应体告诉客户端“具体数据是什么”。如果只返回数据，不管状态码和结构，接口就很难被前端、测试脚本和其他服务稳定调用。

2. 响应模型：后端对外的承诺

在代码里，我们用 response_model=Item 声明响应结构：

@app.get("/items/{item_id}", response_model=Item)
def read_item(item_id: Annotated[int, Path(ge=1)]) -> Item:
    return get_item_or_404(item_id)

Item 模型如下：

class Item(BaseModel):
    id: int
    name: str
    category: ItemCategory
    price: float
    in_stock: bool

这表示接口对外承诺：成功时会返回一个包含这些字段的商品对象。FastAPI 会用这个模型生成 /docs 文档，也会按这个模型处理返回数据。

工程直觉：请求模型管“别人能传什么进来”，响应模型管“你承诺返回什么出去”。

3. 状态码：用数字表达请求结果

先掌握这四个常见状态码：

  • 200 OK：请求成功，常见于查询。
  • 201 Created：资源创建成功，常见于 POST /items。
  • 404 Not Found：客户端要的资源不存在。
  • 422 Unprocessable Entity：请求格式能理解，但参数或请求体不符合模型约束。

在新接口里，创建商品使用 201：

@app.post("/items", response_model=Item, status_code=status.HTTP_201_CREATED)
def create_item(payload: ItemCreate) -> Item:
    next_id = max(ITEMS) + 1
    item = Item(id=next_id, **payload.model_dump())
    ITEMS[next_id] = item
    return item

status.HTTP_201_CREATED 比直接写 201 更可读。面试或团队项目里，这种写法也更容易让别人看懂你的意图。

4. 主动抛出错误：HTTPException

如果用户请求 /items/999，参数本身是合法整数，但业务上找不到这个商品。这不是 422，而是 404。

def get_item_or_404(item_id: int) -> Item:
    item = ITEMS.get(item_id)
    if item is None:
        raise HTTPException(
            status_code=status.HTTP_404_NOT_FOUND,
            detail=f"Item {item_id} was not found.",
        )
    return item

raise HTTPException(...) 会中断当前请求处理，FastAPI 会把它转换成标准 JSON 错误响应。这里的 detail 是给调用者看的错误说明。

5. 404 和 422 的区别

这两个错误很容易混，要用“错在语法/参数”还是“错在业务资源”来区分：

  • GET /items/abc 返回 422，因为 abc 不能变成整数 ID。
  • GET /items/0 返回 422，因为 ID 不满足 ge=1。
  • GET /items/999 返回 404，因为 ID 合法，但这个商品不存在。

普通后端里，这能帮助前端准确提示用户。AI 服务里，这能区分“请求参数不合法”和“任务 ID / 文件 ID / 模型 ID 不存在”。

6. 实验一：创建商品

在 /docs 中测试 POST /items：

{
  "name": "FastAPI Interview Pack",
  "category": "course",
  "price": 49.9,
  "in_stock": true
}

你应该看到 201 状态码，并得到一个带 id 的新商品。然后再访问 GET /items，可以看到它已经进入内存列表。

注意：这里用的是内存字典，重启服务后新增商品会消失。数据库会在后续课程再讲。

7. 实验二：触发不同错误

GET /items/abc
GET /items/0
GET /items/999

POST /items
{
  "name": "A",
  "category": "course",
  "price": -1,
  "in_stock": true
}

请观察状态码和错误响应。前三个请求能帮你区分路径参数校验和业务 404；最后一个请求会展示请求体校验错误。

8. 课堂练习

请你完成两个小任务：

  1. 创建一个 category 为 "tool" 的商品，然后用 GET /items?category=tool 查出来。
  2. 访问一个不存在但合法的 ID，例如 /items/9999，说出为什么它是 404 而不是 422。

9. 小测：先回忆再展开

1. response_model 主要解决什么问题？

它声明成功响应的数据结构，让 FastAPI 能生成文档，并按模型处理返回值。

2. 创建资源为什么常用 201？

因为 201 Created 表示请求成功，并且服务器创建了新的资源。

3. 什么时候用 HTTPException(status_code=404)？

当请求本身合法，但业务资源不存在时，例如数据库里找不到指定 ID。

10. 本节总结

请求模型让输入可控，响应模型让输出稳定，状态码让调用者知道请求结果。HTTPException 是你主动表达业务错误的工具。到这里，你已经具备一个小型 CRUD API 的基本接口表达能力。
