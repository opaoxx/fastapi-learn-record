0091 .env.example：为 GitHub 新手建立本地配置与密钥卫生边界

本节补强 .env.example，并把它与 .gitignore、first_api/settings.py 和 README 配置说明串成闭环。核心目标：提交配置形状，不提交真实秘密；允许本地覆盖，不污染仓库历史；让新手能运行项目，也能理解生产配置边界。

② 核心概念底层原理

配置管理的底层问题是“同一份代码在不同环境中需要不同参数”。代码应稳定，配置应可变。把 API key、外部 provider URL、超时和重试策略写死在代码里，会让开发、测试、生产互相污染，也会让密钥泄露风险上升。

.env.example 是配置 schema 的文档化副本。它告诉新手需要哪些变量、变量名如何拼写、示例值长什么样。它应该可以提交到 Git，因为它不包含真实秘密。.env 是本地实际覆盖文件，包含开发者自己的值，必须被 .gitignore 排除。

pydantic-settings 把字符串环境变量转换成 typed settings。项目中的 SettingsConfigDict(env_file=".env", env_prefix="FIRST_API_") 定义了读取位置和前缀。前缀能避免变量名与系统环境或其他项目冲突。

文件/机制是否提交职责风险
.env.example提交说明配置字段和安全样例误放真实 secret 会泄露
.env不提交本地私有配置覆盖若进 Git 会污染历史
.gitignore提交阻止本地状态进仓库漏掉 .env 会增加泄露概率
Settings提交类型化读取配置字段与 example 不一致会误导新手
部署 secret store远程配置生产密钥管理不能用教学 .env 代替

教学默认值不是生产安全值。dev-secret-key 只用于本地练习，作用是让新手能快速调用受保护接口。生产环境应由部署平台、CI secret、云密钥管理或容器编排系统注入真实 secrets。

③ 全套代码逐行精读解析

# Copy this file to `.env` for local learning overrides.

第一行定义操作方式：复制样例文件生成本地覆盖。直接编辑 .env.example 会污染公共样例；正确做法是复制到 .env 后在本机改。

# Keep `.env` out of Git. Do not commit real secrets, production URLs, private tokens, or customer data.

这行是安全边界。真实密钥、生产 URL、私有 token、客户数据都不应进入仓库。即使后续删除，Git 历史中仍可能保留痕迹。

# The values below are safe teaching defaults only.

该行解释示例值语义。dev-secret-key 是教学默认值，不是生产密钥。没有这句，新手可能误以为示例值可以直接上生产。

FIRST_API_APP_NAME="First FastAPI API"

应用名称配置。它映射到 Settings.app_name，用于展示和系统信息。放进配置而非写死，便于不同环境显示不同名称。

FIRST_API_ENVIRONMENT="development"

环境标识。它映射到 Settings.environment，用于区分 development、test、production 等运行上下文。当前教学项目只用作学习信号。

FIRST_API_API_KEY="dev-secret-key"

本地 API key 示例。受保护接口通过 X-API-Key 请求头校验它。真实环境必须替换，并通过 secret store 注入。

FIRST_API_AI_PROVIDER="demo"

AI provider 类型。当前 Settings.ai_provider 使用 Literal["demo"]，表示课程只启用 demo provider，避免新手误以为已经接入真实模型服务。

FIRST_API_AI_PREDICTION_BACKEND="demo"

预测后端选择。它允许 demo 或 provider，用于课程中切换本地 demo 与 provider 适配层。默认 demo 能保证离线学习可运行。

FIRST_API_AI_PROVIDER_HEALTH_URL="https://example.com"

provider health URL 示例。使用 example.com 是安全占位，不指向真实私有服务。真实 URL 不应写入样例文件。

FIRST_API_AI_PROVIDER_PREDICTION_URL="https://example.com/predict"

provider prediction URL 示例。它说明字段形状，但不暴露真实 provider 地址。生产环境应从部署配置注入。

FIRST_API_AI_TIMEOUT_SECONDS="10"

超时配置。Settings.ai_timeout_seconds 用 Field(gt=0) 校验必须大于 0；若填 0 或负数，配置加载会失败。

FIRST_API_AI_MAX_ATTEMPTS="1"

最大尝试次数。代码中约束 ge=1, le=5，防止无意义的 0 次尝试和过高重试造成教学项目卡顿。

FIRST_API_AI_RETRY_BASE_DELAY_SECONDS="0.1"

基础退避延迟。它配合 retry 课程使用。设置为小数是为了本地测试快速完成。

FIRST_API_AI_RETRY_MAX_DELAY_SECONDS="2.0"

最大退避延迟。它限制重试等待上限，避免指数退避无限变大。生产环境通常根据 provider SLA 和业务超时预算调整。

FIRST_API_AI_RETRY_JITTER_SECONDS="0.1"

抖动配置。jitter 用于避免并发重试同时打向 provider。教学默认值很小，便于观察机制。

env_file=".env"

first_api/settings.py 中的该配置告诉 pydantic-settings 从根目录 .env 读取本地覆盖。若文件不存在，则使用模型默认值。

env_prefix="FIRST_API_"

前缀将外部变量 FIRST_API_API_KEY 映射到字段 api_key。不使用前缀时，通用变量名容易和其他工具冲突。

.env

.gitignore 中这一行确保本地私有配置默认不会被 Git 追踪。它是密钥卫生的最后一道低成本防线。

④ 核心机制运行全流程拆解

Learner clones repository
  -> reads README configuration section
  -> copies .env.example to .env
  -> optionally changes FIRST_API_API_KEY
  -> starts FastAPI app
  -> Settings loads defaults + .env overrides
  -> protected endpoints compare X-API-Key with settings.api_key
  -> learner runs tests
  -> .gitignore prevents .env from being committed

运行链路从 README 开始，到 Settings 结束。.env.example 不参与运行覆盖，除非被复制为 .env；.env 参与运行，但不进入 Git。这种分层同时满足可上手和安全边界。

⑧ 课后记忆习题+标准答案

1. .env.example 的核心职责是什么？

作为可提交的配置字段清单和安全样例，告诉新手需要哪些环境变量，但不保存真实密钥或生产地址。

2. 为什么 .env 必须进入 .gitignore？

.env 保存本地真实覆盖，可能包含密钥、私有 URL 或 token。提交到 Git 后会泄露到历史记录和远程仓库。

3. FIRST_API_ 前缀有什么作用？

它把项目配置与系统变量、其他项目变量隔离，并让 pydantic-settings 能把外部变量映射到 Settings 字段。

4. 为什么教学默认 key 不能用于生产？

教学 key 是公开样例，任何人都能看到；生产必须使用不可公开、可轮换、由平台安全注入的真实 secret。

5. 如果密钥已经提交到 Git，应该怎么办？

立即停止发布，移除密钥，轮换密钥；若已推送，应按 SECURITY 流程处理，并考虑历史清理和影响评估。

⑨ 面试高频真题+解析

面试题 1：为什么配置不应该写死在代码里？

不同环境需要不同参数。写死会导致开发、测试、生产耦合，并增加密钥泄露和部署变更风险。

面试题 2：.env.example 和 .env 如何分工？

.env.example 是提交到仓库的字段说明和安全样例；.env 是本地私有配置，参与运行但不提交。

面试题 3：pydantic-settings 的价值是什么？

它把字符串环境变量加载为有类型、有默认值、有校验规则的 Settings 对象，减少配置解析和类型错误。

面试题 4：生产环境如何管理 secrets？

应使用云 Secret Manager、CI/CD secrets、容器编排 secret、平台环境变量和权限控制，不应依赖提交到仓库的文件。

面试题 5：为什么 .gitignore 不能完全解决密钥泄露？

.gitignore 只能阻止未跟踪文件被自动加入；已被 Git 跟踪或已推送的密钥仍在历史中，需要轮换和清理。
