Lesson 0026

count 查询：当前页长度不等于总记录数

列表响应里的 count 不是 len(items)。它表示数据库中符合当前筛选条件的总数，需要单独查询。

0. 本节学习目标

  • 理解 len(items) 和 count 的区别。
  • 看懂 select(func.count())。
  • 理解 count 查询为什么要复用筛选条件。
  • 能解释 count 对前端分页控件的意义。

1. 一个容易混淆的问题

如果请求：

GET /tasks?limit=1&offset=0

响应可能是：

{
  "items": [{"id": 40, "status": "completed"}],
  "count": 18,
  "limit": 1,
  "offset": 0
}

这里 items.length 是 1，因为当前页只拿了 1 条；count 是 18，因为数据库里符合条件的任务总共有 18 条。

2. 两个查询

为了返回这种响应，service 里其实做了两个查询：

statement = select(SummaryTask).order_by(SummaryTask.id.desc())
count_statement = select(func.count()).select_from(SummaryTask)
  • statement：查当前页的任务记录。
  • count_statement：查符合条件的总数。

3. count 也要带筛选条件

if task_status is not None:
    statement = statement.where(SummaryTask.status == task_status)
    count_statement = count_statement.where(SummaryTask.status == task_status)

如果请求 /tasks?status=failed，那么 count 应该表示 failed 任务总数，而不是全部任务总数。所以 page 查询和 count 查询必须使用同样的筛选条件。

4. 分页只作用于 items

statement = statement.offset(offset).limit(limit)
items = list(session.exec(statement))
count = session.exec(count_statement).one()

offset() 和 limit() 只用于当前页查询，不用于 count 查询。如果 count 也加了 limit，那么总数就不准了。

5. 返回响应信封

return SummaryTaskListResponse(
    items=items,
    count=count,
    limit=limit,
    offset=offset,
)

这里 service 返回的是 Pydantic 响应模型，不是裸字典。这样字段名和字段类型都更清楚，FastAPI 文档也能自动展示。

6. 测试 count

completed = client.get("/tasks?status=completed&limit=100")
failed = client.get("/tasks?status=failed&limit=100")

assert completed.json()["count"] == len(completed.json()["items"])
assert failed.json()["count"] == len(failed.json()["items"])

测试里用 limit=100 是为了让本页能装下全部测试数据。这样可以直接比较 count 和 len(items)，确认 count 没有混入其他状态。

7. 课堂练习

请求：

GET /tasks?status=completed&limit=1&offset=0

观察 items.length 和 count。如果 count 大于 1，说明当前筛选条件下还有下一页。

8. 小测：先回忆再展开

1. 为什么 count 不能直接等于 len(items)？

因为 items 只是当前页，count 是符合筛选条件的总数。

2. count 查询为什么不能加 limit？

因为加了 limit 后只能数当前页，不能表示总记录数。

3. status 筛选为什么要同时作用于 count 查询？

因为 count 必须表示当前筛选条件下的总数。

9. 本节总结

本节补上了分页接口最容易漏掉的 count 查询。现在 GET /tasks 能同时告诉前端：当前页有哪些任务，以及当前筛选条件下一共有多少任务。
