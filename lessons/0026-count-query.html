Lesson 0026

count 查询：当前页长度不等于总记录数

响应信封里最容易误解的字段是 count。它不是 len(items)，而是“当前筛选条件下，数据库里一共有多少条记录”。

0. 本节学习目标

  • 理解 items.length 和 count 的语义差异。
  • 看懂为什么分页接口通常需要两条查询：数据查询和计数查询。
  • 掌握 select(func.count()).select_from(...) 的作用。
  • 理解 count 查询为什么必须复用筛选条件，但不能使用分页截取。
  • 能解释 count 如何支撑前端分页按钮和总数提示。

1. 背景：为什么 len(items) 不够

假设任务表里有 18 个 completed 任务，请求：

GET /tasks?status=completed&limit=1&offset=0

响应可能是：

{
  "items": [{"id": 40, "status": "completed"}],
  "count": 18,
  "limit": 1,
  "offset": 0
}

这里 items.length 是 1，因为当前页只拿了 1 条；count 是 18，因为符合筛选条件的记录总共有 18 条。

如果把 count 写成 len(items)，前端会误以为总共只有 1 条，于是不会显示下一页。

2. 核心概念：当前页数量和总结果数量是两个问题

数据查询回答的问题是：

这一页应该显示哪些任务？

计数查询回答的问题是：

当前筛选条件下一共有多少任务？

这两个问题相关，但不是同一个问题。因此它们通常需要两条查询。

3. 代码精读一：两条查询从同一个表出发

statement = select(SummaryTask).order_by(SummaryTask.id.desc())
count_statement = select(func.count()).select_from(SummaryTask)

statement 查询真实任务记录，后面会加排序、筛选、分页。

count_statement 查询数量，不需要返回每一行任务，所以它选择的是 func.count()。SQLAlchemy 的 func 用来生成 SQL 函数表达式，func.count() 对应数据库里的 COUNT(...)。

select_from(SummaryTask) 明确告诉 SQLAlchemy：这个计数来自 SummaryTask 表。

4. 为什么 count 查询也要带筛选条件

if task_status is not None:
    statement = statement.where(SummaryTask.status == task_status)
    count_statement = count_statement.where(SummaryTask.status == task_status)

如果请求是：

GET /tasks?status=failed&limit=10&offset=0

那么 count 应该表示 failed 任务总数，而不是全部任务总数。否则前端会显示一个错误的总数。

这条规则可以概括为：items 查询和 count 查询必须共享同一套筛选条件。

5. 为什么 count 查询不能加 offset 和 limit

statement = statement.offset(offset).limit(limit)
items = list(session.exec(statement))
count = session.exec(count_statement).one()

offset() 和 limit() 只用于当前页数据查询。如果把它们加到 count_statement 上，计数就会变成“当前页最多几条”，而不是“总共几条”。

所以规则是：

筛选条件 -> items 查询和 count 查询都要用
排序      -> items 查询需要，count 查询通常不需要
分页      -> 只用于 items 查询，不用于 count 查询

6. session.exec(...).one() 表示什么

count = session.exec(count_statement).one()

计数查询不会返回一组任务，而是返回一个数字。.one() 表示我们期望这条查询正好有一个结果。

对 COUNT 来说，这个预期是合理的：无论表里有没有数据，数据库都会返回一个计数值，例如 0、3、18。

7. 返回响应信封

return SummaryTaskListResponse(
    items=items,
    count=count,
    limit=limit,
    offset=offset,
)

这里 service 返回的是 Pydantic 响应模型，而不是随手拼一个字典。好处是字段结构清楚、类型明确，也能和 FastAPI 的 response_model 保持一致。

这就是从数据库查询到 API 契约的最后一步：把数据查询的结果和计数查询的结果组装成一个稳定响应。

8. 前端如何使用 count

const hasPrevious = page.offset > 0;
const hasNext = page.offset + page.items.length < page.count;

如果 hasNext 为真，说明当前页后面还有数据，前端可以启用“下一页”。如果 offset 大于 0，说明不是第一页，前端可以启用“上一页”。

没有 count，前端只能猜。猜出来的分页体验通常会出错。

9. 测试 count 的正确性

completed = client.get("/tasks?status=completed&limit=100")
failed = client.get("/tasks?status=failed&limit=100")

assert completed.json()["count"] == len(completed.json()["items"])
assert failed.json()["count"] == len(failed.json()["items"])

测试里用 limit=100 是为了让当前页容纳全部测试数据。这样在测试数据规模可控的情况下，count 应该等于 len(items)。

更重要的是，它验证了 count 没有混入其他状态。例如 failed 的 count 不应该把 completed 任务也算进去。

10. 常见错误与诊断

错误一：count 等于 len(items)

这只能表示当前页长度，不能表示总数。分页时尤其错误。

错误二：count 忘记筛选条件

status=failed 时，count 必须只数 failed 任务。

错误三：count 也加 limit

这样数出来的是当前页，不是总结果集。

错误四：把 count 查询写到前端

前端不应该自己反复请求推断总数。总数应由后端统一计算。

11. 课堂实验

先创建多个任务，然后请求：

GET /tasks?limit=1&offset=0
X-API-Key: dev-secret-key

观察：

  1. items.length 是否为 1。
  2. count 是否可能大于 1。
  3. 如果 offset + items.length < count，是否说明还有下一页。

再请求：

GET /tasks?status=failed&limit=1&offset=0

思考：这个响应里的 count 应该表示全部任务总数，还是 failed 任务总数？

12. 小测：先回忆再展开

1. 为什么 count 不能直接等于 len(items)？

因为 items 是当前页记录，count 是当前筛选条件下的总记录数。

2. count 查询为什么要复用 status 筛选？

因为总数必须对应当前筛选条件，筛选 failed 时就只能统计 failed 任务。

3. count 查询为什么不能加 offset 和 limit？

因为分页截取只属于当前页数据查询，加到 count 上会让总数变成当前页数量。

4. func.count() 在这里解决什么问题？

它生成数据库计数函数，让数据库直接返回符合条件的记录数量。

13. 本节总结

本节补上了分页接口最容易漏掉的一块：总数查询。现在你应该能清楚区分两个问题：当前页有哪些任务，以及当前筛选条件下一共有多少任务。前者由 items 表达，后者由 count 表达。
