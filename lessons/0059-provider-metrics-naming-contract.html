0059 Provider Metrics Naming Contract：指标名和标签名也是 API

本节把 metric name、HELP 文本、label names 提升为监控契约。它们会被 Prometheus 查询、dashboard、告警规则、runbook 和课程文档引用。随手改名不是普通重构，而是对外观测接口变更。

① 本节核心知识框架

Metrics Naming Contract
├─ metric name
│  └─ provider_prediction_total
├─ HELP text
│  └─ Provider prediction outcomes recorded by the teaching metrics counter.
├─ label names
│  ├─ operation
│  ├─ outcome
│  ├─ failure_category
│  ├─ error_code
│  └─ status_code
├─ 约束原则
│  ├─ counter 使用 _total
│  ├─ label names 表达维度
│  ├─ label values 保持低基数
│  └─ 命名变更必须显式迁移
└─ 测试落点
   ├─ test_provider_metric_name_and_help_text_are_stable_contract
   └─ test_provider_metric_label_names_are_stable_low_cardinality_contract

② 核心概念底层原理

指标命名一旦暴露，就进入外部依赖链。Prometheus 查询可能写死 provider_prediction_total；告警规则可能按 outcome="retry_exhausted" 过滤；runbook 可能指导值班人员查看 failure_category。

metric name 定义“测量对象”，label name 定义“分组维度”，label value 定义“具体分类”。三者职责不能混淆。把 outcome 塞进 metric name 会导致指标碎片化；把 request id 塞进 label value 会导致高基数爆炸。

def test_provider_metric_name_and_help_text_are_stable_contract() -> None:
    assert PROVIDER_PREDICTION_METRIC_NAME == "provider_prediction_total"
    assert PROVIDER_PREDICTION_METRIC_NAME.endswith("_total")
    assert (
        PROVIDER_PREDICTION_METRIC_HELP
        == "Provider prediction outcomes recorded by the teaching metrics counter."
    )

测试不仅检查字符串，还检查 _total 语义。若有人把 counter 改成无后缀名称，测试会提示这是监控命名契约变化。

def test_provider_metric_label_names_are_stable_low_cardinality_contract() -> None:
    labels = build_provider_metric_labels(
        outcome="retry_exhausted",
        failure_category="retryable_http_status",
        error_code="ai_provider_http_error",
        status_code=503,
    )

    assert list(labels.as_dict()) == [
        "operation",
        "outcome",
        "failure_category",
        "error_code",
        "status_code",
    ]

该测试保护 label key 集合与顺序。顺序对 Prometheus 查询语义不是必须，但对本课程的文本导出、测试快照和学习材料很有价值。

⑧ 课后记忆习题+标准答案

1. 为什么 metric name 接近 API？

因为它会被外部查询、dashboard、告警、文档和 runbook 依赖。改名会影响外部消费者，不只是内部代码重构。

2. _total 后缀表达什么？

表达这是累计型 counter，语义上记录总次数，通常只增不减。

3. label name 和 label value 的区别是什么？

label name 表达维度，如 outcome；label value 表达该维度下的分类，如 retry_exhausted。

4. 为什么 label value 要低基数？

每组 label value 都可能形成一条时间序列。高基数会导致存储、查询、告警成本暴涨。

⑨ 面试高频真题+解析

面试题 1：如何设计 metrics name 和 labels？

metric name 表达同一被测对象和指标类型，例如 provider_prediction_total；labels 表达低基数维度，例如 operation、outcome、failure_category、error_code、status_code。避免把动态文本、用户 ID、request id 放入 labels。

面试题 2：为什么命名需要测试保护？

因为命名进入监控查询和告警规则。测试让无意改名显式失败，迫使开发者按兼容迁移处理，而不是静默破坏下游。

面试题 3：metric name 中应该包含 outcome 吗？

通常不应把 outcome 拆进多个 metric name。更好的做法是一个 metric name 表达 provider prediction 总事件，通过 outcome label 区分 success、retry_exhausted、fail_fast 等分类。
