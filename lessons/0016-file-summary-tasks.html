文件摘要任务：上传资源、任务资源与 source_file_id 溯源链路

文件型 AI 服务不是“上传完立刻算完”，而是把文件资源和处理任务拆开：POST /files/text 创建输入资源，POST /tasks/files/{file_id}/summary 创建处理任务，GET /tasks/{task_id} 查询结果。本节聚焦这条链路的资源建模、错误语义和生产扩展。

① 本节核心知识框架

文件摘要闭环
├─ 输入资源：UploadedTextFile 保存上传文件内容与元数据
├─ 处理资源：SummaryTask 保存处理状态、结果和错误
├─ 关联字段：SummaryTask.source_file_id -> UploadedTextFile.id
├─ 创建接口：POST /tasks/files/{file_id}/summary
├─ 调度机制：创建 queued 任务后 add_task(process_summary_task, task.id)
├─ 查询接口：GET /tasks/{task_id} / GET /tasks
└─ 扩展方向：PDF、图片、音频、表格都可复用“文件 -> 任务 -> 状态”模型

② 核心概念底层原理

上传文件和处理文件应拆成两个资源生命周期。上传成功只说明输入进入系统；摘要成功说明某个处理任务完成。二者失败原因完全不同：上传可能因为鉴权、格式、大小、编码失败；处理可能因为模型超时、外部依赖失败或文本无法摘要失败。

source_file_id 是溯源字段。它让任务结果能回指输入文件，解决“这个摘要来自哪个文件”的问题。没有溯源字段，任务结果、用户上传记录、调试日志和审计信息会断裂。

该设计体现资源组合思想：文件是可复用输入，任务是一次处理动作。一个文件未来可以创建摘要、关键词提取、分类、向量化等多个任务；如果上传时强行处理，就会限制复用和错误恢复。

    responses={status.HTTP_404_NOT_FOUND: {"description": "The uploaded text file does not exist."}},

def create_file_summary_task(
    file_id: Annotated[int, Path(ge=1, description="The uploaded text file ID.")],
    background_tasks: BackgroundTasks,
    session: Annotated[Session, Depends(get_session)],
) -> SummaryTask:
  • "/files/{file_id}/summary"：路径表达“基于某个文件创建摘要任务”，不是读取文件本身。
  • response_model=SummaryTaskRead：返回任务视图，包含 source_file_id，但不返回文件完整内容。
  • 202：创建的是后台处理任务，结果稍后查。
  • Path(ge=1)：文件 id 必须是正整数；非法路径参数直接 422。
uploaded_file = session.get(UploadedTextFile, file_id)
if uploaded_file is None:
    raise HTTPException(
        status_code=status.HTTP_404_NOT_FOUND,
        detail=f"File {file_id} was not found.",
    )

这里先验证输入资源存在。file_id 合法但库中没有文件，所以是 404。不能直接创建空任务，否则后续 worker 没有输入文本。

task = create_summary_task_from_file(
    session=session,
    uploaded_file=uploaded_file,
)

background_tasks.add_task(process_summary_task, task.id)

任务创建仍然走服务层，路由函数只负责编排：查文件、建任务、注册后台处理、返回任务。

def create_summary_task_from_file(
    session: Session,
    uploaded_file: UploadedTextFile,
) -> SummaryTask:
    return create_summary_task(
        session=session,
        text=uploaded_file.content,
        source_file_id=uploaded_file.id,
    )

这一层把文件内容转换为任务输入，把文件 id 写入 source_file_id。如果不写该字段，前端和运维无法从任务追溯来源。

④ 核心机制运行全流程拆解

POST /files/text
  -> 保存 UploadedTextFile(id=7, content="...", preview="...")
  -> 返回 file_id=7

POST /tasks/files/7/summary
  -> Path 校验 file_id >= 1
  -> session.get(UploadedTextFile, 7)
  -> create_summary_task(text=file.content, source_file_id=7)
  -> SummaryTask(id=15, status="queued")
  -> add_task(process_summary_task, 15)
  -> 返回 202 + task_id=15

后台 worker
  -> 读取 task 15
  -> status=running
  -> ai_client.summarize(task.text)
  -> status=completed/result=...

GET /tasks/15
  -> 返回 source_file_id=7 + completed/result

⑥ 开发常见报错+坑点+解决方案

  • 404 File was not found：任务接口引用了不存在的文件。解决：先上传文件，并使用响应里的 id。
  • 422 file_id：路径参数不是正整数。解决：使用 /tasks/files/1/summary 这类合法路径。
  • source_file_id 为 null：调用的是原始文本任务 /tasks/summaries，不是文件任务。解决：确认调用路径。
  • 任务失败但上传成功：这是两个阶段，上传成功不保证处理成功。解决：查询任务 error 字段。
  • 重复处理同一文件：教学项目允许重复创建任务。生产环境若要防重复，应增加幂等键或唯一任务约束。

⑦ 生产环境 vs 教学环境 核心差异

维度教学环境生产环境

⑧ 课后记忆习题+标准答案

1. 为什么上传和处理要拆成两个接口？

因为上传是输入资源创建，处理是异步任务执行；拆开后更容易复用文件、定位错误、扩展多种处理任务。

2. source_file_id 解决什么问题？

它建立任务到上传文件的溯源关系，使结果、日志、前端展示和审计都能追踪输入来源。

3. 文件不存在时为什么返回 404？

路径参数合法，但服务器没有该文件资源；这符合资源不存在的 HTTP 语义。

⑨ 面试高频真题+解析

题 1：如何设计一个文档摘要服务的最小闭环？

先上传文件并保存文件元数据和内容引用；再创建摘要任务并记录 source_file_id；后台 worker 读取文件内容执行摘要；客户端通过任务状态接口查询结果。

题 2：为什么任务表要保存 source_file_id 而不是只保存 text？

只保存 text 会丢失输入资源身份，无法追溯文件名、上传者、版本和复用关系；source_file_id 能把处理结果与原始输入资源连接起来。

题 3：一个文件触发多个 AI 任务时如何建模？

文件表保存输入资源，任务表保存任务类型、参数、状态、结果与 source_file_id；必要时增加 task_type、model_version、idempotency_key 和任务参数快照。
