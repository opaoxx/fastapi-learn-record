Lesson 0016

上传文件创建摘要任务：文件型 AI 服务入口

这一节把第 15 课的文件上传和第 13/14 课的后台任务合起来：先上传文本文件，再基于文件内容创建摘要任务。

0. 本节学习目标

  • 理解“上传文件”和“处理文件”为什么通常拆成两个接口。
  • 看懂 POST /tasks/files/{file_id}/summary。
  • 理解 source_file_id 如何把任务和文件关联起来。
  • 建立文件型 AI 服务的基本工作流。

1. 为什么拆成两步

你可能会问：为什么不上传文件时立刻处理？真实项目里，拆成两步更灵活：

  • 一个文件可能被多次处理，例如摘要、分类、关键词提取。
  • 上传成功和处理成功是两件事，错误原因也不同。
  • 处理可能很慢，适合放到后台任务。
  • 前端可以先展示“文件已上传”，再展示“任务处理中”。

2. 文件摘要接口

@router.post("/files/{file_id}/summary", response_model=SummaryTaskRead, status_code=202)
def create_file_summary_task(
    file_id: Annotated[int, Path(ge=1)],
    background_tasks: BackgroundTasks,
    session: Annotated[Session, Depends(get_session)],
) -> SummaryTask:
    uploaded_file = session.get(UploadedTextFile, file_id)
    if uploaded_file is None:
        raise HTTPException(status_code=404, detail=f"File {file_id} was not found.")
    ...

它先根据 file_id 找上传文件。如果文件不存在，就返回 404。如果存在，就用文件内容创建摘要任务。

3. source_file_id 的作用

task = SummaryTask(
    text=uploaded_file.content,
    text_length=len(uploaded_file.content),
    source_file_id=uploaded_file.id,
    status="queued",
)

source_file_id 表示这个任务来自哪个上传文件。以后你看到任务结果时，就能追溯输入来源。

这也是后端常见的关联关系：任务属于某个文件，评论属于某篇文章，订单属于某个用户。

4. 完整工作流

POST /files/text
  -> 得到 file_id

POST /tasks/files/{file_id}/summary
  -> 得到 task_id

GET /tasks/{task_id}
  -> 查看 status 和 result

这就是文件型 AI 服务的最小闭环。把 demo 摘要函数替换成真实模型调用后，它就可以变成文档摘要 API 的雏形。

5. 实验：上传后创建任务

先上传文本文件：

POST /files/text
X-API-Key: dev-secret-key
file = notes.txt

拿到响应里的 id 后，创建摘要任务：

POST /tasks/files/{file_id}/summary
X-API-Key: dev-secret-key

最后查询任务：

GET /tasks/{task_id}
X-API-Key: dev-secret-key

6. 和 VLM / 文档 AI 的关系

当前只处理文本文件，但结构可以扩展：

  • 图片上传后创建 VLM 描述任务。
  • PDF 上传后创建解析和摘要任务。
  • 音频上传后创建转写任务。
  • 表格上传后创建数据分析任务。

核心思想一样：文件进入系统，任务处理文件，状态接口返回结果。

7. 课堂练习

  1. 上传一个 .txt 文件。
  2. 基于它创建摘要任务。
  3. 查询任务结果，并指出 source_file_id 是哪个文件。

8. 小测：先回忆再展开

1. 为什么上传和处理常拆成两个接口？

因为文件保存和文件处理是两个不同阶段，拆开后更容易复用、排错和异步执行。

2. source_file_id 表示什么？

它表示任务来源于哪个上传文件，方便追踪输入和结果。

3. 文件不存在时为什么返回 404？

因为 file_id 合法，但数据库里没有对应文件资源。

9. 本节总结

本节完成了文件型 AI 服务的最小闭环：上传文本文件，创建文件摘要任务，查询任务结果。后续可以把存储从 SQLite 换成对象存储，把 demo 摘要函数换成真实大模型。
