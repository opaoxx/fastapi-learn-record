0053 Provider Metrics Failure Outcomes：从快照识别失败路径

本节专门拆 provider failure metrics。真正有用的指标不是一个笼统的 failed，而是区分 retry_scheduled、retry_exhausted、fail_fast。同样是 provider 调用失败，503 重试耗尽与 400 快速失败代表完全不同的系统状态和处理策略。

① 本节核心知识框架

Failure Outcomes
├─ retry_scheduled
│  ├─ 可重试失败
│  ├─ 尚未最终失败
│  └─ 表示系统正在尝试恢复
├─ retry_exhausted
│  ├─ 可重试失败持续发生
│  ├─ max_attempts 已用完
│  └─ 表示最终失败
├─ fail_fast
│  ├─ 不适合重试
│  ├─ 400 / invalid_response 等
│  └─ 表示立即停止
├─ 典型场景
│  ├─ 连续 503 + max_attempts=2 → scheduled + exhausted
│  └─ 400 → fail_fast
└─ 测试边界
   ├─ 先 POST /provider/predict 制造事件
   ├─ 再 GET /provider/metrics 读取快照
   └─ 断言调用方真实可见 JSON 合同

② 核心概念底层原理

失败不是单一状态，而是系统生命周期中的不同阶段。retry_scheduled 表示本次失败后系统仍准备继续尝试；retry_exhausted 表示可恢复路径已经用尽；fail_fast 表示该错误从策略上不应继续尝试。

outcome是否最终失败典型原因排障含义
retry_scheduled不是timeout、request_error、429、503 等可重试失败系统正在吸收临时抖动
retry_exhausted是可重试错误超过 max_attemptsprovider 持续不可用或重试预算不足
fail_fast是400、401、404、422、invalid_response请求契约、鉴权、路径或响应格式需要修复

如果只统计 failed，你无法判断系统是在短暂恢复中，还是已经最终失败，也无法判断问题来自 provider 临时故障还是请求/契约错误。

③ 全套代码逐行精读解析

代码解释关键点
wait_before_retry()只在还会继续尝试时执行因此 outcome 是 retry_scheduled，不是最终失败
classify_retry_metric_failure()把 timeout/request_error/http_status 转成 failure category保留错误性质
retry_metric_error_code()把 retry reason 转成稳定错误码与 API error_code 体系对齐
 
def log_retry_exhausted(...):
    ...
    record_provider_metric(
        build_provider_metric_labels(
            outcome="retry_exhausted",
            failure_category=classify_retry_metric_failure(
                retry_reason,
                status_code=status_code,
            ),
            error_code=error_code,
            status_code=status_code,
        )
    )

retry_exhausted 只在最后一次 attempt 失败后记录。它与 retry_scheduled 可能出现在同一个业务调用链中，但语义不同：一个表示“准备恢复”，一个表示“恢复失败”。

def log_prediction_fail_fast(...):
    ...
    record_provider_metric(
        build_provider_metric_labels(
            outcome="fail_fast",
            failure_category=classify_fail_fast_metric_failure(
                fail_fast_reason,
                status_code=status_code,
            ),
            error_code=error_code,
            status_code=status_code,
        )
    )

fail_fast 不经过等待和重试。HTTP 400、非法响应体等错误继续重试通常只会浪费资源，并放大 provider 压力。

prediction_response = client.post("/provider/predict", json={...})
metrics_response = client.get("/provider/metrics")

测试必须先制造 provider 事件，再读取 metrics。/provider/metrics 是只读窗口，不应该自己生成失败样本。

④ 核心机制运行全流程拆解

场景 A：provider 连续返回 503，ai_max_attempts=2。

attempt 1
  → provider status 503
  → classify_provider_http_status(503) = retryable_http_status
  → attempt < max_attempts
  → record retry_scheduled{status_code="503"}

attempt 2
  → provider status 503
  → retryable_http_status
  → attempt == max_attempts
  → record retry_exhausted{status_code="503"}
  → API 返回 503 + ai_provider_http_error

对应 JSON 快照：

[
  {
    "labels": {
      "operation": "prediction",
      "outcome": "retry_exhausted",
      "failure_category": "retryable_http_status",
      "error_code": "ai_provider_http_error",
      "status_code": "503"
    },
    "count": 1
  },
  {
    "labels": {
      "operation": "prediction",
      "outcome": "retry_scheduled",
      "failure_category": "retryable_http_status",
      "error_code": "ai_provider_http_error",
      "status_code": "503"
    },
    "count": 1
  }
]

场景 B：provider 返回 400。

attempt 1
  → provider status 400
  → classify_provider_http_status(400) = non_retryable_http_status
  → 不等待，不重试
  → record fail_fast{status_code="400"}
  → API 返回 503 + ai_provider_http_error

对应快照只有一条 fail_fast 样本。若出现 retry_scheduled，说明不可重试边界被破坏。

⑤ 重难点、易混淆点对比辨析

场景failure_categoryoutcome解释
503 第一次失败且仍有预算retryable_http_statusretry_scheduled错误可重试，系统将继续尝试
503 最后一次失败retryable_http_statusretry_exhausted错误性质不变，但重试预算耗尽

注意：provider 原始 400 不等于本项目最终响应 400。本项目把 provider 调用失败映射为对当前 API 调用方的 503，但 metrics 中保留 status_code="400" 作为内部诊断事实。

⑧ 课后记忆习题+标准答案

1. 为什么 retry_scheduled 不等于失败总数？

它表示系统准备重试，后续 attempt 仍可能成功。最终失败应看 retry_exhausted 或 fail_fast。

2. 连续 503 且最大尝试次数为 2 时，为什么会有两条样本？

第一次 503 触发 retry_scheduled，第二次 503 用完重试预算触发 retry_exhausted。它们描述同一调用链中的两个观测事件。

3. provider 400 的 metrics 应如何记录？

记录 outcome="fail_fast"、failure_category="non_retryable_http_status"、error_code="ai_provider_http_error"、status_code="400"。

4. 为什么 endpoint 级测试比只测 helper 更有价值？

helper 测试只能证明内部函数正确；endpoint 级测试证明真实调用者通过 GET /provider/metrics 能看到正确 JSON 合同。

⑨ 面试高频真题+解析

面试题 1：provider 失败指标为什么不能只用 failed？

因为 failed 混淆了临时失败、最终失败和不可重试失败。高质量观测需要区分 retry_scheduled、retry_exhausted、fail_fast，这样才能判断系统是否在自恢复、是否预算不足、是否请求契约错误。

面试题 2：如何测试 retry failure metrics？

用 httpx.MockTransport 让 provider 返回 503，用 settings 覆盖最大尝试次数和 retry delay；通过 POST /provider/predict 触发真实业务路径，再用 GET /provider/metrics 断言 scheduled 与 exhausted 样本。

面试题 3：为什么 provider 400 对外可能返回 503？

从当前 API 调用方视角看，AI provider 能力不可用，因此服务层映射为稳定 503；但 metrics 保留 provider 原始 status_code="400"，用于定位是上游请求/契约问题。
