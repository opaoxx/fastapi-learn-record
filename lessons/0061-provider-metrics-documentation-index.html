0061 Provider Metrics Documentation Index：把监控知识组织成可验证入口

本节不再新增业务代码，而是把 0050-0060 累积的 provider metrics 知识收束为一个可维护文档入口：reference/provider-metrics-index.html。核心思想是 docs-as-contract：关键文档不是“写完给人看”的静态说明，而是项目契约的一部分，必须被测试保护，必须能追溯到 endpoint、指标名、生产边界、课程来源和代码测试。

② 核心概念底层原理

documentation index 的本质是导航型契约，不是教程正文。教程正文负责按知识依赖展开原理；文档入口负责把“读者现在要解决的问题”路由到正确资源。对于 metrics 主题，读者通常不会从第一节慢慢翻，而是带着问题进入：端点在哪里、指标名是什么、能不能用于生产告警、改名会不会破坏 dashboard、对应测试在哪。

docs-as-contract 的底层逻辑来自接口稳定性治理。代码契约通常由类型、测试和 OpenAPI 保护；文档契约则保护“人类依赖的信息入口”。如果指标名变了但入口页面仍写旧名字，读者会配置错误查询；如果文档漏掉生产边界，教学用 in-process counter 可能被误认为生产级 metrics pipeline；如果入口不链接测试，读者无法判断文档陈述是否仍由项目验证。

概念关注点与本节的区别
课程正文解释一个知识点的因果链、代码、流程偏学习路径，篇幅可长
Documentation Index把分散资源聚合为入口偏检索路径，必须短、准、可追溯
Runbook遇到现象后按步骤排查偏操作路径，强调判断分支
OpenAPI机器可读 HTTP 合同偏接口形状，不负责完整学习路线
README项目全局介绍范围过大，不能替代专题索引

本节的设计取舍是：只保护入口页的关键事实，不逐字测试整篇文档。逐字测试会把排版修改、措辞优化都变成失败；关键事实测试则只锁住端点、指标名、改名 helper、生产边界、课程链接和测试链接这些真正属于契约的内容。

③ 全套代码逐行精读解析

本节主要代码是文档合同测试。它使用 Python 标准库 HTMLParser，不引入 BeautifulSoup 等额外依赖，符合教学项目“少依赖、够表达”的原则。

class LinkAndTextParser(HTMLParser):
    def __init__(self) -> None:
        super().__init__()
        self.links: list[str] = []
        self.text_parts: list[str] = []
代码行作用为什么这样写 / 不这样写的隐患
class LinkAndTextParser(HTMLParser)继承标准库 HTML 解析器比正则解析 HTML 更稳；正则容易被属性顺序、换行、嵌套标签破坏。
super().__init__()初始化父类状态机省略后 parser 内部状态可能不完整，属于继承初始化错误。
self.links收集所有 <a href>入口文档的核心能力之一是“可追溯”，因此链接是合同对象。
self.text_parts收集正文文本片段正文关键事实可能分布在表格、段落、列表中，统一抽取更稳定。
 
def handle_starttag(
    self,
    tag: str,
    attrs: list[tuple[str, str | None]],
) -> None:
    if tag != "a":
        return
    attributes = dict(attrs)
    href = attributes.get("href")
    if href is not None:
        self.links.append(href)
代码行作用工程含义
handle_starttag解析起始标签时回调只在结构层抽取链接，不关心 CSS 和视觉排版。
if tag != "a": return过滤非链接标签减少测试噪声；文档入口只需要验证导航能力。
dict(attrs)把属性列表转为字典避免依赖属性顺序；HTML 属性顺序不属于语义契约。
attributes.get("href")读取链接目标允许没有 href 的 a 标签存在，不因为无关结构直接报错。
self.links.append(href)记录链接后续断言可以直接检查课程和测试是否被链接。
 
def handle_data(self, data: str) -> None:
    if data.strip():
        self.text_parts.append(data.strip())

@property
def text(self) -> str:
    return " ".join(self.text_parts)
代码行作用不这么写的隐患
data.strip()去掉纯空白文本避免换行缩进进入断言文本，降低测试脆弱性。
append(data.strip())保留有意义正文表格单元格、列表项、段落都能被统一检查。
" ".join(...)把片段拼成可检索文本如果直接比较原始 HTML，会被标签和排版细节干扰。

这一组断言锁住可追溯性。没有课程链接，读者只能看到结论；没有测试链接，读者无法判断结论是否被当前代码保护。文档入口的价值不在于“看起来完整”，而在于能把读者带到真实证据。

④ 核心机制运行全流程拆解

读者进入 GitHub 仓库
  → 打开 provider metrics 文档入口
    → 定位运行端点
      → /provider/metrics 或 /provider/metrics/prometheus
    → 定位指标契约
      → provider_prediction_total + label keys + HELP text
    → 定位边界说明
      → educational export shape, not production multi-instance pipeline
    → 定位变更审查点
      → get_provider_prediction_metric_contract()
    → 定位证据链
      → 课程页 + provider_http 测试 + OpenAPI 合同测试
    → 若要排障
      → 跳转 provider-metrics-runbook.html

从数据流角度看，provider metrics 的真实运行数据来自 provider prediction 过程，先被计入 in-process counter，再由 JSON endpoint 和 Prometheus text endpoint 读取；documentation index 不参与运行链路，它参与知识链路。两条链路必须互相指向：运行端点提供事实，文档入口组织事实，合同测试保护入口不脱离事实。

阶段输入处理输出
运行层provider prediction outcome记录低基数 metric labelscounter snapshot
接口层counter snapshot渲染 JSON 或 text/provider/metrics*
文档层端点、指标、边界、测试聚合为入口页provider-metrics-index.html
测试层HTML 文本和链接断言关键事实存在CI 中发现文档漂移

⑤ 重难点、易混淆点对比辨析

易混点错误理解正确理解
文档入口 vs 长教程入口页越长越好入口页应该高信噪比，负责路由，不负责重复讲完整课程。
文档测试 vs 文案测试逐字锁死整篇文章只保护端点、指标、边界、链接等契约事实。
README vs 专题索引README 写了就够README 是全局入口；专题索引负责某一技术域的深链接与证据链。
OpenAPI vs 文档入口有 Swagger 就不需要文档页OpenAPI 描述接口形状；文档入口还要组织课程、测试、生产边界和迁移说明。
教学指标 vs 生产指标能导出文本就是生产监控当前是教学级 in-process counter，不是多实例聚合、持久存储、告警治理体系。

⑥ 开发常见报错+坑点+解决方案

问题典型表现根因解决方案
链接漂移点击课程或测试链接 404文件改名后文档未同步把关键链接放入 parser.links 断言；改名时同步测试。
指标名漂移文档写旧的 metric name只改代码，漏改文档入口断言 provider_prediction_total 或新兼容名必须存在。
边界丢失读者误把教学 counter 用于生产告警文档只写端点，不写限制强制保留 not a production multi-instance metrics pipeline。
测试过细改一个标点导致测试失败把文案当作合同全文比较只断言关键事实和关键链接，不测试排版措辞。
正则解析 HTML属性换行或顺序变化导致误判HTML 是结构化文本，正则脆弱使用 HTMLParser 抽取文本和链接。

⑧ 课后记忆习题+标准答案

1. documentation index 的核心职责是什么？

把分散的端点、指标契约、生产边界、课程说明和测试证据聚合为一个高信噪比入口，回答“从哪里开始查”。

2. 为什么文档入口也需要测试？

因为入口页承载外部依赖信息。若端点、指标名、生产边界或测试链接漂移，读者会做出错误配置或错误判断。测试能在 CI 中提前发现明显失真。

3. 为什么不逐字测试整篇 HTML？

逐字测试会把排版、换行、措辞优化都变成失败，导致测试脆弱。合理做法是只锁定真正属于合同的关键事实和链接。

4. 本节文档入口必须包含哪些 provider metrics 关键事实？

/provider/metrics、/provider/metrics/prometheus、provider_prediction_total、get_provider_prediction_metric_contract 和 not a production multi-instance metrics pipeline。

5. documentation index 与 runbook 的区别是什么？

documentation index 解决“资源在哪里”；runbook 解决“看到某个现象后下一步怎么查”。前者偏导航，后者偏操作流程和判断分支。

⑨ 面试高频真题+解析

面试题 1：如何防止技术文档与代码长期漂移？

把关键文档当作合同维护。对外部依赖信息，例如 endpoint、metric name、label keys、生产边界、迁移 helper 和测试链接，编写轻量文档合同测试；测试只锁关键事实，不锁整篇文案。变更代码契约时要求同 PR 更新文档和测试。

面试题 2：为什么有 OpenAPI 之后仍然需要专题文档入口？

OpenAPI 主要描述 HTTP 接口形状，无法完整表达教学路径、生产边界、命名迁移、测试证据和排障入口。专题文档入口把机器合同、人类解释、课程来源和测试证据组织在一起。

面试题 3：文档合同测试应该锁哪些内容，不应该锁哪些内容？

应该锁 endpoint、指标名、稳定 label、生产边界、关键 helper、课程和测试链接。不应该锁整段措辞、视觉排版、HTML 标签顺序、换行和非契约性描述。

面试题 4：如果团队要修改 provider_prediction_total，文档入口应如何参与变更？

入口页应链接改名审查 helper 和命名契约课程，标注旧名/新名兼容窗口，说明 dashboard、alert、runbook、测试的迁移状态。文档合同测试应随兼容策略更新，避免旧入口误导使用者。
