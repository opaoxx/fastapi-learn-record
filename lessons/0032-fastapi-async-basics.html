Lesson 0032

FastAPI async 基础：等待时让出执行权，而不是让 CPU 更快

上一课我们把真实 AI provider 放到边界层。真实 provider 往往是外部 HTTP 调用，所以会遇到“等待”。这一节讲清楚 async def、await、事件循环、阻塞调用和同步数据库的关系。

0. 本节学习目标

  • 理解并发和并行的区别。
  • 理解事件循环为什么能在等待 I/O 时处理其他任务。
  • 知道什么时候用 async def，什么时候普通 def 也可以。
  • 理解 await asyncio.sleep() 和 time.sleep() 的本质区别。
  • 知道当前项目里的 SQLModel 数据库代码仍然是同步代码，不能因为路由写了 async def 就自动变异步。

1. 背景：AI 服务为什么绕不开 async

一个真实 AI 服务接口经常要等待外部事情完成：

  • 等待浏览器上传文件。
  • 等待数据库查询。
  • 等待外部 AI provider 返回结果。
  • 等待对象存储、消息队列或另一个微服务。

这些等待不一定消耗 CPU。比如请求外部 AI 时，程序大部分时间不是在计算，而是在等网络响应。如果服务器在等待时什么都不能做，就会浪费吞吐量。

async 的意义就在这里：当一个任务等待 I/O 时，它可以主动让出执行权，让事件循环去处理别的任务。

2. 核心概念：async 是等待效率，不是计算加速

先把一句话记牢：

async helps with waiting; it does not make CPU work faster.

如果你的函数主要在等网络、等文件、等数据库、等外部 API，async 很有价值。如果你的函数主要在做 CPU 密集计算，比如压缩大文件、跑复杂算法、处理大图像，单纯写成 async def 并不会让它更快。

这也是为什么 AI 后端常常同时需要两种思维：

  • I/O 等待型任务：适合 async，例如外部模型 API 调用。
  • CPU 计算型任务：需要线程池、进程池、任务队列或专门推理服务。

3. 术语表

术语 含义 本节例子
Concurrency 并发 多个任务交替推进，看起来同时处理。 等待一个请求时处理另一个请求。
Parallelism 并行 多个任务真的同时在多个 CPU 核上运行。 多进程并行跑 CPU 计算。
Event loop 事件循环 负责调度 async task 的循环。 一个任务 await 时切去跑别的任务。
Coroutine 协程 由 async def 创建的可暂停函数。 async_wait()
await 等待一个可等待对象，并把执行权交还给事件循环。 await asyncio.sleep(delay)
Blocking 阻塞 当前执行线程被占住，不能做别的事。 time.sleep()、同步网络请求。

4. 官方资料里的关键点

FastAPI 官方 async 文档说明：你可以在路径操作函数中使用 async def，也可以使用普通 def；FastAPI 会根据函数形态做合适处理。重点不是所有函数都强行 async，而是选择和内部调用相匹配的写法。

Python 官方 asyncio 文档说明：事件循环使用协作式调度。一个事件循环一次运行一个 task；当某个 task 在 await 等待 Future 完成时，事件循环可以运行其他 task、回调或 I/O 操作。

这两点合起来，就是 FastAPI async 的基础心智模型：你的代码需要在可等待的地方主动让出执行权。

5. 本节项目改动概览

  • 新增响应模型 AsyncWaitResponse。
  • 新增公开接口 GET /async/wait?delay=0.05。
  • 接口内部使用 await asyncio.sleep(delay) 模拟非阻塞等待。
  • 新增测试验证响应结构和 delay 参数边界。

这个接口不是业务功能，而是教学探针。它让我们能用项目自己的代码解释 async 机制。

6. 代码精读一：响应模型

class AsyncWaitResponse(BaseModel):
    status: Literal["ok"]
    delay_seconds: float
    message: str

这个模型保持简单：状态、等待秒数和说明文字。即使是教学接口，我们仍然用响应模型把 JSON 形状固定下来，这延续了前面 OpenAPI 契约和 schema 演进的思路。

7. 代码精读二：async path operation

@router.get(
    "/async/wait",
    response_model=AsyncWaitResponse,
    summary="Demonstrate non-blocking async waiting",
)
async def async_wait(
    delay: Annotated[
        float,
        Query(ge=0, le=1, description="Seconds to await.")
    ] = 0.05,
) -> AsyncWaitResponse:
    await asyncio.sleep(delay)
    return AsyncWaitResponse(
        status="ok",
        delay_seconds=delay,
        message="The endpoint awaited without doing CPU work.",
    )

这段代码有三个重点：

  • async def：告诉 Python 这是协程函数，可以在内部使用 await。
  • Query(ge=0, le=1)：限制等待时间，避免学习接口被故意请求很久。
  • await asyncio.sleep(delay)：模拟异步 I/O 等待，等待期间事件循环可以处理别的任务。

如果你把 await asyncio.sleep(delay) 换成 time.sleep(delay)，它就会变成阻塞等待。函数签名仍然可以是 async def，但内部阻塞调用会占住执行线程，破坏 async 的意义。

8. 运行机制：一次 /async/wait 请求发生了什么

  1. 浏览器或测试客户端请求 /async/wait?delay=0.05。
  2. FastAPI 解析查询参数，把字符串 "0.05" 转成 float。
  3. Query 校验 delay 是否在 0 到 1 之间。
  4. FastAPI 调用 async_wait 协程。
  5. 代码执行到 await asyncio.sleep(delay)。
  6. 这个协程暂停，事件循环可以调度其他任务。
  7. 等待时间到后，协程恢复，返回 AsyncWaitResponse。
  8. FastAPI 用响应模型序列化 JSON。

这里最重要的是第 5 和第 6 步：await 不是“卡在那里什么都不能做”，而是“我在等，你先去处理别的”。

9. def 和 async def 怎么选

场景 建议 原因
内部调用 async HTTP 客户端 async def 可以 await 网络 I/O。
内部调用 async 数据库驱动 async def 数据库等待可以让出执行权。
内部全是同步库 普通 def 通常更清楚 写成 async 也不能自动把同步库变异步。
CPU 密集计算 不要只靠 async 需要线程、进程、队列或专门计算服务。
教学演示非阻塞等待 async def await asyncio.sleep 能清楚展示让出执行权。

FastAPI 允许混用 def 和 async def。成熟项目里真正重要的是：不要在 async def 里面悄悄塞同步阻塞调用。

10. 同步数据库 caveat：当前 SQLModel 代码仍然是同步的

本项目现在的数据库依赖是同步 SQLModel/SQLite 形态：

with Session(engine) as session:
    ...

这意味着现有 items 和 tasks 相关数据库操作不是 async 数据库操作。你不应该只是把它们的 router 函数改成 async def，然后以为数据库等待就会自动非阻塞。

如果未来要把数据库层改成真正 async，需要成套考虑 async engine、async session、驱动支持、事务管理和测试改造。这属于后面的高级主题。

def test_async_wait_endpoint_returns_wait_metadata() -> None:
    with TestClient(app) as client:

    assert response.status_code == 200
    assert response.json() == {

测试里用 delay=0 是为了验证接口行为，同时不让测试真的等待。另一个测试请求 delay=2，应该得到 422，因为我们把最大等待限制为 1 秒。

12. 课堂实验

运行服务后访问：

http://127.0.0.1:8000/async/wait?delay=0.2

你应该看到：

{
  "status": "ok",
  "delay_seconds": 0.2,
  "message": "The endpoint awaited without doing CPU work."
}

再访问：

http://127.0.0.1:8000/async/wait?delay=2

你应该看到 422 校验错误。这个错误来自 Query(le=1)，说明异步端点仍然享受 FastAPI 的参数校验和 OpenAPI 文档生成。

13. 常见错误与诊断

错误一：以为 async 会让计算更快

async 主要优化等待，不会让 CPU 密集计算自动加速。

错误二：在 async 函数里调用 time.sleep

time.sleep 是阻塞等待，会占住执行线程。

错误三：把同步数据库函数直接改成 async

同步数据库驱动不会因为外层是 async def 就变成非阻塞。

错误四：忘记 await

调用协程但不 await，通常得到的是协程对象，而不是执行结果。

14. 面试视角：如何回答 FastAPI async

FastAPI 支持普通 def 和 async def。async def 适合内部有可 await 的 I/O 操作，比如 async HTTP 客户端、async 数据库驱动或其他协程。await 时当前协程会暂停并把执行权交还给事件循环，使服务器能处理其他任务。async 不会让 CPU 密集计算变快，也不能把同步阻塞库自动变成异步；在 async endpoint 中调用阻塞函数反而会影响并发。

15. 小测：先判断，再展开

1. async def 会让 CPU 密集计算更快吗？

不会。async 主要提升等待型 I/O 的并发效率，不是计算加速器。

2. await asyncio.sleep() 和 time.sleep() 的关键区别是什么？

前者让出执行权，后者阻塞当前执行线程。

3. 当前 SQLModel 同步数据库代码适合直接改成 async def 吗？

不适合。同步数据库调用不会因为外层 async 就自动变成非阻塞。

4. FastAPI 能混用 def 和 async def 吗？

可以。关键是根据内部调用是否可 await 来选择合适写法。

16. 本节总结

这节课的核心是：async 不是神奇加速按钮，而是一种等待型并发工具。async def 创建协程，await 在等待点让出执行权，事件循环再调度其他任务。它适合外部 HTTP、async 数据库、文件或网络等待；不适合单靠它解决 CPU 密集计算。
