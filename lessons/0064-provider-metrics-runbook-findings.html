0064 Provider Metrics Runbook Findings：把指标样本翻译成排障结论

上一节提供了稳定 metrics scenario samples。本节继续向前一步：新增 build_provider_metrics_runbook_findings()，把样本中的 outcome、failure_category、status_code 转换成 severity、finding 和 next_action。目标不是让程序自动代替人排障，而是训练初学者把观测事实转换成工程判断。

① 本节核心知识框架

Provider Metrics Runbook Findings
├─ 输入
│  └─ provider metric samples: [{labels, count}, ...]
├─ 核心字段
│  ├─ outcome：分支入口
│  ├─ count：样本数量
│  ├─ failure_category：失败分类
│  └─ status_code：HTTP 状态码或 none
├─ 输出
│  ├─ severity：教学级判断级别
│  ├─ finding：观测事实的人类解释
│  └─ next_action：下一步排障动作
├─ 四个稳定分支
│  ├─ success → baseline
│  ├─ retry_scheduled → investigate
│  ├─ retry_exhausted → action_required
│  └─ fail_fast → contract_check
├─ 防御分支
│  ├─ unknown outcome → unknown
│  ├─ labels 非 dict → ValueError
│  └─ count 非 int → ValueError
└─ 工程边界
   ├─ 纯转换函数
   ├─ 不读取全局 counter
   ├─ 不发出告警
   └─ 不替代生产 runbook/incident workflow

② 核心概念底层原理

监控排障的关键不是“看到一个数字”，而是把数字放回语义上下文。provider_prediction_total 的 count 只是事实；outcome="retry_exhausted" 才把事实推向“可重试失败已经耗尽尝试次数”的判断；next_action 则把判断进一步落到可执行动作。

本节新增的 findings helper 是一个解释层。它位于 raw metrics 与 runbook 之间：raw metrics 保持机器可读、低基数、稳定；runbook findings 把机器字段转成人类排障语言。两者不能混为一谈。metrics 不应该存长句子，findings 也不应该伪装成真实监控数据。

层级代表字段职责
观测事实labels、count记录发生过什么，保持机器可处理。
分类判断outcome、failure_category决定进入哪个 runbook 分支。
人类解释finding把 label 组合翻译成一句明确判断。
行动建议next_action提示下一步查什么，不直接自动修复。
治理边界severity教学级优先级，不等于生产告警级别。

面试里经常考“可观测性数据如何变成行动”。标准答案不是把所有逻辑塞进告警规则，而是分层：采集层保持低基数事实，解释层形成可读结论，runbook 层规定动作，生产告警层再结合 SLO、阈值、持续时间和升级策略。

③ 全套代码逐行精读解析

def build_provider_metrics_runbook_findings(
    samples: list[dict[str, object]],
) -> list[dict[str, object]]:
    findings = []
    for sample in samples:
        labels = sample.get("labels")
        count = sample.get("count")
        if not isinstance(labels, dict):
            raise ValueError("Metric sample labels must be a dictionary.")
        if not isinstance(count, int):
            raise ValueError("Metric sample count must be an integer.")
代码行作用为什么这样写 / 不这样写的隐患
samples接收 metrics snapshot-like 数据复用上一节 fixture 和真实 endpoint snapshot shape。
findings = []准备输出列表一个 sample 对应一个 finding，保留逐样本可追溯性。
sample.get("labels")读取 label 字典label 是所有 runbook 分支判断的入口。
sample.get("count")读取样本数量finding 需要保留观测规模，不能只给结论。
isinstance(labels, dict)校验 labels shape没有 dict 就无法读取 outcome；继续执行会产生含糊异常。
isinstance(count, int)校验 count shapecount 不是整数时无法表示 counter 样本。
 
outcome = labels.get("outcome")
failure_category = str(labels.get("failure_category", "none"))
status_code = str(labels.get("status_code", "none"))

outcome 决定主分支；failure_category 和 status_code 被保留到输出中，方便读者把 finding 追溯回原始 label。这里对缺省值使用 none，与前面 metrics label contract 保持一致。

if outcome == "success":
    finding = "Provider prediction requests are succeeding."
    next_action = (
        "Compare the success count with expected request volume before "
        "treating failure samples as system-wide."
    )
    severity = "baseline"

success 不代表“无问题”，而是建立基线：当失败样本存在时，成功样本可以帮助判断故障是否系统性。若不保留 success 分支，学习者容易看到失败样本就直接下全局故障结论。

elif outcome == "retry_scheduled":
    finding = "Retryable provider failures were observed and retry was scheduled."
    next_action = (
        "Check failure_category, status_code, retry delay logs, and whether "
        "retry_exhausted also appears."
    )
    severity = "investigate"

retry_scheduled 的重点是“需要调查”，不是“已经失败”。下一步必须继续查 retry_exhausted，否则会把一次可恢复抖动误判成最终失败。

elif outcome == "retry_exhausted":
    finding = "Retryable provider failures exhausted the configured attempts."
    next_action = (
        "Check max_attempts, provider availability, and the last retryable "
        "failure before changing retry policy."
    )
    severity = "action_required"

retry_exhausted 比 scheduled 更接近真实影响，但仍然不能自动等同“应该增加重试次数”。正确动作是先检查最大尝试次数、provider 可用性和最后一次错误类别。

elif outcome == "fail_fast":
    finding = "A non-retryable provider failure or invalid response was recorded."
    next_action = (
        "Do not increase retries first; inspect the request or response "
        "contract and the status_code."
    )
    severity = "contract_check"

fail_fast 的关键词是“契约检查”。它可能来自 4xx 或 invalid response。盲目重试只会放大错误，应该先检查请求参数、provider 响应结构和状态码语义。

else:
    finding = "The metric sample uses an outcome without a runbook branch."
    next_action = (
        "Add an explicit runbook branch before treating this metric as "
        "operationally actionable."
    )
    severity = "unknown"

未知 outcome 不能静默忽略。可观测性字段一旦新增，就必须同步 runbook，否则团队会看到新信号却不知道如何行动。

findings.append(
    {
        "outcome": str(outcome),
        "count": count,
        "failure_category": failure_category,
        "status_code": status_code,
        "severity": severity,
        "finding": finding,
        "next_action": next_action,
    }
)

输出同时保留原始分类字段和解释字段。这样页面、测试或后续练习既能展示人类可读结论，也能追溯到机器 label。

④ 核心机制运行全流程拆解

scenario samples
  → build_provider_metrics_runbook_findings(samples)
    → 校验 labels 是 dict
    → 校验 count 是 int
    → 读取 outcome/failure_category/status_code
    → outcome 分支匹配
      ├─ success → baseline finding
      ├─ retry_scheduled → investigate finding
      ├─ retry_exhausted → action_required finding
      ├─ fail_fast → contract_check finding
      └─ unknown → unknown finding
    → 输出 [{outcome,count,severity,finding,next_action}, ...]
  → 学习者对照 runbook
    → 从 finding 理解事实
    → 从 next_action 确定下一步检查

数据流上，build_provider_metrics_runbook_scenario_samples() 负责“造观测样本”，build_provider_metrics_runbook_findings() 负责“解释观测样本”。这两个函数都不读取真实全局 counter，因此它们是教学练习工具，不是运行时监控入口。

输入 outcomeseverity核心判断下一步
successbaseline成功请求仍存在对比预期请求量，避免过度扩大失败影响。
retry_scheduledinvestigate发生过可重试失败查失败分类、状态码、retry delay log 和 exhausted 样本。
retry_exhaustedaction_required重试预算耗尽查 provider 可用性、max_attempts、最后一次错误。
fail_fastcontract_check非重试失败或响应无效先查契约，不先增加重试。
未知 outcomeunknownrunbook 没有分支先补 runbook 分支，再把信号用于行动。

⑧ 课后记忆习题+标准答案

1. 为什么不能把 finding 文本直接放进 metrics label？

metrics label 需要低基数、稳定、便于聚合查询。finding 是人类解释文本，容易变长和变化，应该在解释层生成。

2. retry_scheduled 对应的 severity 为什么是 investigate？

因为它只表示安排过重试，不代表最终失败。必须继续查 retry_exhausted、成功样本和日志。

3. fail_fast 的 next_action 为什么强调先查契约？

fail-fast 通常来自非重试型 HTTP 错误或无效响应。盲目重试会放大错误，正确动作是先检查请求/响应契约和状态码语义。

4. 为什么未知 outcome 不应该静默忽略？

未知 outcome 表示 metrics contract 已出现新信号，但 runbook 没有解释分支。静默忽略会让团队错过需要治理的新状态。

5. 本节 helper 为什么不读取全局 counter？

它的目标是解释传入样本，服务课程练习和测试；读取全局 counter 会把解释函数变成运行时状态读取函数，增加副作用和测试不确定性。

⑨ 面试高频真题+解析

面试题 1：可观测性数据如何转化为可执行行动？

先保持采集层低基数、稳定；再用解释层把 label 组合映射成 finding；再由 runbook 定义 next_action；生产环境还要结合 SLO、时间窗口、阈值、影响范围和升级策略，不能只靠单个 count 下结论。

面试题 2：为什么解释层应与 metrics 采集层分离？

采集层服务机器聚合查询，必须稳定、低基数；解释层服务人类理解，可以包含长文本和行动建议。混在一起会导致指标不可控、查询困难、文案变更影响监控契约。

面试题 3：看到新的 outcome，你会怎么处理？

先确认它是否属于稳定 metrics contract；然后补充 label builder、测试、runbook finding、文档入口和排障流程。上线前确保未知 outcome 不会被误当成已治理信号。

面试题 4：教学 severity 与生产告警 severity 有什么区别？

教学 severity 用来训练判断分支，例如 baseline、investigate、action_required；生产告警 severity 要结合业务影响、SLO、持续时间、用户范围、升级路径和值班规则，不能由单个教学函数直接决定。
