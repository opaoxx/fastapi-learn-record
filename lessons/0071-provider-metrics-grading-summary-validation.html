0071 Provider Metrics Grading Summary Validation：渲染报告前先校验训练汇总契约

本节新增 validate_provider_metrics_runbook_grading_summary()，对 0069 生成的 summary 做结构校验和计数一致性校验。它不渲染报告、不重新评分，只负责把坏数据挡在 Markdown renderer 之前。

① 本节核心知识框架

Grading Summary Validation
├─ 输入：summary dict
├─ 校验维度
│  ├─ 必备数字字段
│  │  ├─ total
│  │  ├─ answered
│  │  ├─ passed
│  │  ├─ failed
│  │  └─ unanswered
│  ├─ grades 必须是 list
│  ├─ 每个 grade 必备字段
│  │  ├─ card_id
│  │  ├─ answered
│  │  ├─ passed
│  │  ├─ missing
│  │  └─ feedback
│  └─ 统计一致性
│     ├─ failed == total - passed
│     ├─ unanswered == total - answered
│     └─ total == len(grades)
└─ 输出
   ├─ valid
   └─ errors

② 核心概念底层原理

校验层的目标不是“修复数据”，而是“暴露契约破坏”。summary 会被 Markdown renderer、学习记录、PR 说明等多个消费者使用，越靠近入口发现坏数据，后续排查成本越低。

本节选择返回 {"valid": bool, "errors": list}，而不是遇到第一个错误立刻抛异常。原因是复盘材料更需要完整诊断：一次性看到所有字段缺失、类型错误和计数不一致，学习者才能系统修正数据。

校验函数不重新评分，也不重新生成 summary。它只检查 summary 是否满足结构和数学约束。这样能保持职责单一：评分层负责判断答案，汇总层负责统计，校验层负责守住数据契约，渲染层负责展示。

层级职责错误处理方式
summary helper生成统计结果通过测试保护计算语义
validator检查结构和一致性返回完整 errors 列表
renderer渲染 Markdown只做最小防御校验

③ 全套代码逐行精读解析

def validate_provider_metrics_runbook_grading_summary(
    summary: dict[str, object],
) -> dict[str, object]:
    errors = []
    numeric_fields = ["total", "answered", "passed", "failed", "unanswered"]

函数接收 summary 并返回结构化校验结果。errors 收集所有问题，numeric_fields 固定总体统计字段，避免后续校验散落硬编码字符串。

for field in numeric_fields:
    if field not in summary:
        errors.append(f"Missing summary field: {field}.")
    elif not isinstance(summary[field], int):
        errors.append(f"Summary field must be an integer: {field}.")

第一轮检查字段存在性和类型。使用 elif 是为了避免字段缺失时继续取值导致 KeyError；校验函数应该把错误整理成可读列表，而不是在第一个字段缺失时中断。

grades = summary.get("grades")
if not isinstance(grades, list):
    errors.append("Summary field must be a list: grades.")

grades 使用 get() 是因为这里正在做校验，缺失字段应该转成错误信息。它必须是列表，因为后续逐题明细需要按顺序遍历。

else:
    for index, grade in enumerate(grades, start=1):
        if not isinstance(grade, dict):
            errors.append(f"Grade #{index} must be a dictionary.")
            continue

逐题校验从 1 开始编号，方便错误信息对应用户看到的题号。若某个 grade 不是字典，继续检查字段没有意义，因此用 continue 跳到下一项。

for field in ["card_id", "answered", "passed", "missing", "feedback"]:
    if field not in grade:
        errors.append(f"Grade #{index} missing field: {field}.")
if "answered" in grade and not isinstance(grade["answered"], bool):
    errors.append(f"Grade #{index} field must be a boolean: answered.")
if "passed" in grade and not isinstance(grade["passed"], bool):
    errors.append(f"Grade #{index} field must be a boolean: passed.")
if "missing" in grade and not isinstance(grade["missing"], list):
    errors.append(f"Grade #{index} field must be a list: missing.")

每个 grade 必须包含追踪字段、布尔结果、缺失锚点和反馈。answered 与 passed 必须是布尔值，missing 必须是列表，否则渲染层和复盘逻辑都会误读数据。

if all(field in summary and isinstance(summary[field], int) for field in numeric_fields):
    total = summary["total"]
    answered = summary["answered"]
    passed = summary["passed"]
    failed = summary["failed"]
    unanswered = summary["unanswered"]

只有数字字段都存在且类型正确时，才进入数学一致性校验。否则继续计算会把类型错误变成二次异常，降低错误信息质量。

if total < 0 or answered < 0 or passed < 0 or failed < 0 or unanswered < 0:
    errors.append("Summary counts must be non-negative.")
if answered > total:
    errors.append("Summary answered count cannot exceed total.")
if passed > total:
    errors.append("Summary passed count cannot exceed total.")
if failed != total - passed:
    errors.append("Summary failed count must equal total - passed.")
if unanswered != total - answered:
    errors.append("Summary unanswered count must equal total - answered.")
if isinstance(grades, list) and total != len(grades):
    errors.append("Summary total must equal the number of grades.")

这些判断保护 summary 的数学语义。failed 和 unanswered 不是独立随便填的字段，它们必须能由 total、passed、answered 推导出来。

return {
    "valid": not errors,
    "errors": errors,
}

最终返回统一结构。valid 适合程序快速判断，errors 适合教学复盘和测试断言。

④ 核心机制运行全流程拆解

summary
  → validate_provider_metrics_runbook_grading_summary(summary)
  → validation result
    ├─ valid: True
    │  └─ render Markdown report
    └─ valid: False
       └─ inspect errors before rendering

校验层应位于 summary 生成之后、Markdown 渲染之前。这样 renderer 不需要承担完整 schema 检查，也不会把错误数据渲染成看似正常的报告。

⑧ 课后记忆习题+标准答案

1. 为什么 validator 不直接修复坏 summary？

校验层的职责是暴露契约破坏，自动修复会隐藏上游错误来源，导致后续数据更难追踪。

2. 为什么要一次性返回多个 errors？

教学复盘需要完整诊断。一次只报一个错误会降低修复效率，也不利于理解 schema 的整体约束。

3. 为什么 failed 必须等于 total - passed？

failed 表示未通过题目数，数学上必须由总题数和通过题数推导出来，否则汇总统计自相矛盾。

4. 为什么 missing 必须是 list？

missing 表示缺失锚点集合，可能包含多个字段。字符串无法表达集合语义，还会破坏后续 join 渲染。

5. 生产环境会如何升级本节校验？

可引入 Pydantic 或 JSON Schema，增加字段路径、错误码、题库版本、数据库约束和审计日志。

⑨ 面试高频真题+解析

面试题 1：为什么需要在渲染前校验 summary？

渲染层会把数据变成人类可信的报告。如果坏数据先被渲染，读者容易把错误统计当成事实，后续排查成本更高。

面试题 2：返回 errors 和抛异常有什么取舍？

返回 errors 适合批量诊断和教学反馈；抛异常适合不可恢复的运行时错误。这里目标是复盘输入契约，所以返回完整错误列表更合适。

面试题 3：如何判断一个汇总对象是否自洽？

检查字段存在、类型正确、计数非负、answered/passed 不超过 total、failed 等于 total - passed、unanswered 等于 total - answered、total 等于明细数量。

面试题 4：生产系统为什么不能只靠前端校验？

前端校验可以提升体验，但不能作为可信边界。后端仍必须校验 schema、权限、一致性和持久化约束。
