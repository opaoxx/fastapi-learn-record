# Learning Record 0016: Course Quality Recalibration

## Date
2026-07-03

## Context
The learner noticed that earlier lessons had stronger explanations, while later lessons became more like code snippets plus short analysis. The learner wants the course to feel more like reading a serious technical article or academic-style explanation: first understand the knowledge and principles, then use code to consolidate the idea.

## What changed
- Added `COURSE-STANDARD.md` as the quality contract for future lessons.
- Reframed future lessons around principle, mechanism, code reading, experiments, common mistakes, and retrieval practice.
- Clarified that two-lesson batches are acceptable only when quality remains high.
- Planned the next stage around frontend/API integration, OpenAPI contracts, schema evolution, AI provider boundaries, and async basics.

## Key insight
FastAPI learning should not be reduced to memorizing decorators and code blocks. A useful lesson should help the learner answer "why is the API designed like this?", "what happens at runtime?", and "how do I verify this behavior?".

## Retrieval prompts
- What is the difference between learning syntax and learning a mechanism?
- Why should a backend lesson explain request flow, response shape, and failure behavior?
- Why can two lessons in one batch still be worse than one deep lesson?
- What does it mean to treat an API response shape as a contract?

## Next possible step
Start lesson 0027 with the new standard: teach frontend task history as an API-contract and UI-state lesson, not merely as DOM rendering code.
