# Code of Conduct

This course repository is intended to be a beginner-safe technical learning space.

The goal is not only to keep conversations polite. The goal is to protect learning quality, psychological safety, public collaboration safety, and the ability of maintainers to review technical work with evidence.

## Expected Behavior

Contributors, learners, and maintainers should:

- Be respectful, patient, and precise when discussing beginner questions.
- Critique code, documentation, claims, and evidence rather than attacking people.
- Explain tradeoffs when correcting an answer or implementation.
- Keep public discussions safe for learners who are new to FastAPI, backend engineering, GitHub workflow, and testing.
- Link lessons, source files, tests, issues, discussions, or pull requests when making technical claims.
- Accept maintainer decisions about scope, triage, review, moderation, and release readiness.

## Unacceptable Behavior

The following behavior is not acceptable:

- Harassment, insults, threats, sexualized comments, discriminatory language, or repeated personal attacks.
- Mocking beginner questions or using shame as a teaching method.
- Publishing private information, credentials, tokens, private logs, personal data, or proprietary source code.
- Deliberately derailing issues, discussions, pull requests, or reviews.
- Repeatedly ignoring maintainer requests to move a topic to the correct channel.
- Pressuring maintainers to bypass tests, review, branch protection, or documented release rules.
- Submitting AI-generated or copied material without checking accuracy, licensing, and fit with the course standard.

## Scope

This code of conduct applies to repository issues, discussions, pull requests, reviews, commits, release notes, documentation examples, and any public channel that is clearly used for this course repository.

Private behavior may affect repository participation when it directly threatens learner safety, maintainer safety, or the integrity of the project.

## Reporting

If you see unsafe or unacceptable behavior:

1. Do not escalate the conflict in public.
2. Save links, screenshots, timestamps, and the affected repository area.
3. Contact maintainers through the most appropriate private channel available for the repository.
4. If no private channel exists, open a short issue asking for a maintainer contact path without posting sensitive details.

Reports should include:

- What happened.
- Where it happened.
- Who was affected.
- Whether private data, secrets, or safety risks are involved.
- Links or evidence that maintainers can inspect.

## Maintainer Response

Maintainers may take one or more actions:

- Ask participants to clarify, edit, or move a comment.
- Hide, delete, or lock unsafe content when the platform allows it.
- Redirect a discussion to the correct issue, discussion, or pull request.
- Pause review until behavior or safety concerns are resolved.
- Reject or close a pull request when the contributor ignores the documented workflow.
- Temporarily or permanently restrict participation when behavior threatens learners, maintainers, or repository integrity.

Maintainers should prefer proportionate action, preserve evidence when possible, and avoid exposing private reporter details.

## Teaching and Review Norms

Technical disagreement is expected. High-quality disagreement should:

- State the exact claim being challenged.
- Provide source, test, specification, or runtime evidence.
- Distinguish teaching-demo constraints from production requirements.
- Offer a concrete correction or follow-up path.
- Avoid sarcasm, dogpiling, and personal ranking language.

Strong review is welcome. Contempt is not.

## Safety Boundary

Do not post secrets, API keys, private logs, personal data, customer data, security exploit details, or proprietary code in public repository channels.

If a report involves a security issue, avoid public reproduction details and ask maintainers for a private disclosure path.

## Relationship to Other Documents

- `CONTRIBUTING.md` explains how to submit acceptable work.
- `DEVELOPMENT.md` explains local maintenance commands and quality gates.
- `.github/DISCUSSIONS.md` explains exploratory learning conversations.
- `.github/ISSUE_TRIAGE.md` explains actionable issue workflow.
- `.github/RELEASE_PROCESS.md` explains verified course batch publishing.

This code of conduct defines the collaboration safety layer around those workflows.
