Reference 0054

Provider Metrics Test Isolation 速查

进程内 metrics counter 是共享状态。测试前后都 reset，才能让 endpoint 测试互不污染。

核心问题

provider_prediction_metrics = ProviderMetricsCounter()

# Same Python process
test A records success
test B can still see test A's count unless reset

Fixture 写法

@pytest.fixture(autouse=True)
def reset_provider_metrics_counter() -> object:
    reset_provider_metrics()
    yield
    reset_provider_metrics()

时间线

  1. 测试开始前：清空 counter。
  2. 测试运行：制造 provider 事件并读取 metrics。
  3. 测试结束后：再次清空 counter。

什么时候用 autouse

情况 选择
整个测试文件都共享同一环境规则 autouse=True
只有少数测试需要 显式 fixture 参数
规则会影响不相关测试 缩小 fixture 作用域

测试主体

def test_provider_metrics_endpoint_exposes_fail_fast_outcome() -> None:
    # fixture already reset metrics
    client.post("/provider/predict", json=payload)
    response = client.get("/provider/metrics")
    assert response.json() == expected_samples

不要混淆的清理

  • reset_provider_metrics() 清 provider metrics counter。
  • app.dependency_overrides.clear() 清 FastAPI dependency overrides。
  • AsyncClient.aclose() 关闭测试 HTTP client。

常见错误

  • 只在测试前 reset，不在测试后 reset。
  • 只在测试后 reset，不在测试前 reset。
  • 把 autouse fixture 放到过大的作用域。
  • 用 fixture 隐藏太多不相关环境行为。
  • 误以为 metrics reset 会清理 dependency overrides。

面试短答

全局进程内状态会跨测试保留，所以我会用 yield fixture 在每个测试前后清理。这里 autouse fixture 表达的是当前测试文件的共同环境规则：provider metrics counter 必须为空开始，也必须在测试后清理。

来源：pytest fixtures、 pytest autouse fixtures、 FastAPI testing dependencies。
