Reference 0043

Backoff Before Retry 速查

retry 之前先等待，避免把短暂故障放大成请求风暴。

心智模型

retry policy says: should we retry?
backoff says: how long should we wait before retrying?

公式

delay = min(base_delay * 2 ** (attempt - 1), max_delay)

例子

attempt base=1, max=3
1 1 秒
2 2 秒
3 3 秒
4 3 秒

配置

FIRST_API_AI_MAX_ATTEMPTS="3"
FIRST_API_AI_RETRY_BASE_DELAY_SECONDS="0.1"
FIRST_API_AI_RETRY_MAX_DELAY_SECONDS="2.0"

核心函数

def calculate_retry_delay(attempt, config):
    base_delay = max(config.retry_base_delay_seconds, 0.0)
    max_delay = max(config.retry_max_delay_seconds, 0.0)
    exponential_delay = base_delay * (2 ** (attempt - 1))
    return min(exponential_delay, max_delay)

等待位置

if attempt < config.max_attempts and error_is_retryable:
    await wait_before_retry(attempt, config, sleep)
    continue

async 原则

  • 在 async adapter 中使用 await asyncio.sleep(...)。
  • 不要用 time.sleep(...) 阻塞线程。
  • 最后一次失败后不 sleep，因为没有下一次 retry。

测试技巧

sleep_calls = []

async def fake_sleep(delay_seconds):
    sleep_calls.append(delay_seconds)

assert sleep_calls == [1.0, 2.0]

常见坑

  • 失败后立即连续 retry。
  • 指数增长没有最大上限。
  • async 代码里用 time.sleep。
  • 测试真的等待几秒。

面试短答

retry 决定是否再试，backoff 决定再试前等多久。我会用有上限的指数退避降低 provider 过载时的请求冲击，并在 async 代码中用 await asyncio.sleep；测试中注入 fake sleep，验证等待时间而不拖慢测试。

来源：Google Cloud Retry Strategy、 Python asyncio Sleeping、 RFC 9110 Retry-After。
