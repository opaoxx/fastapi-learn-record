Reference 0048

Fail-fast Observability 速查

Fail fast 表示错误从分类上不该重试。它应该快速停止，同时留下清楚的结构化日志。

心智模型

retry exhausted = retryable error + no attempts left
fail fast = non-retryable error + stop immediately

日志事件

provider_prediction_fail_fast

字段清单

  • attempt：在哪一次 attempt 上停止。
  • max_attempts：配置允许的最大 attempt 数。
  • fail_fast_reason：http_status 或 invalid_response。
  • status_code：拿到 HTTP 响应时记录状态码。
  • error_code：上抛给 service/router 的稳定错误码。

核心规则

if retryable and attempt < max_attempts:
    log scheduled retry
    sleep
    continue

if retryable:
    log retry exhausted
    raise AIClientError

if not retryable:
    log fail fast
    raise AIClientError

典型场景

场景 是否重试 日志语义
HTTP 400 否 fail_fast_reason="http_status"
HTTP 503 到最后一次 先重试，最后停止 provider_prediction_retry_exhausted
非法响应体 否 fail_fast_reason="invalid_response"

代码形状

logger.warning(
    "provider_prediction_fail_fast",
    extra={
        "attempt": attempt,
        "max_attempts": config.max_attempts,
        "fail_fast_reason": fail_fast_reason,
        "status_code": status_code,
        "error_code": error_code,
    },
)

caplog 断言

assert attempts == 1
assert fail_fast_records[0].fail_fast_reason == "http_status"
assert fail_fast_records[0].status_code == 400
assert exhausted_records == []

常见错误

  • 把 400 误记成 retry exhausted。
  • 非法响应体不记录诊断日志。
  • 日志里写完整用户输入或 provider 响应体。
  • 只测试不重试，不测试日志语义。
  • 把内部 fail-fast 字段加入公共响应体。

面试短答

我会把不可重试错误和 retry exhausted 分开。400 和非法响应体会 fail fast，只尝试一次，并记录 attempt、max_attempts、fail_fast_reason、status_code 和 error_code。这样既避免无意义重试，也不会误导日志读者以为 retry budget 被用完。

来源：Python logging、 pytest logging、 HTTPX exceptions、 Pydantic models。
