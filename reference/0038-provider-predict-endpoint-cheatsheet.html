Reference 0038

Provider Predict Endpoint 速查

新增 POST /provider/predict，把 async provider adapter 接进 FastAPI，同时保持 PredictionResponse 稳定。

核心链路

POST /provider/predict
  -> PredictionRequest
  -> get_settings()
  -> get_provider_http_client()
  -> await request_provider_prediction(...)
  -> PredictionResult
  -> PredictionResponse

配置项

FIRST_API_AI_PROVIDER_PREDICTION_URL="https://example.com/predict"

路由形状

@router.post("/provider/predict", response_model=PredictionResponse)
async def predict_with_provider(
    payload: PredictionRequest,
    settings: Annotated[Settings, Depends(get_settings)],
    provider_http_client: Annotated[
        httpx.AsyncClient,
        Depends(get_provider_http_client),
    ],
) -> PredictionResponse:
    prediction = await request_provider_prediction(...)

为什么新增端点

  • 保留旧 /predict 的 demo 学习体验。
  • 单独验证真实 provider 调用链路。
  • 避免旧测试突然依赖外部网络。
  • 让 async provider 路由和 sync demo 路由边界清楚。

测试模式

replacement_client = httpx.AsyncClient(
    transport=httpx.MockTransport(handler),
    trust_env=False,
)
app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

常见坑

  • 直接替换旧 /predict，导致学习路径和测试不稳定。
  • 同步路由里硬调用 async provider adapter。
  • 测试访问真实 provider。
  • 把 provider 原始 JSON 当成公共 API 响应。

面试短答

我会先新增 /provider/predict 验证真实 provider 链路，而不是直接替换旧 /predict。新端点用 async def，注入 settings 和 lifespan-managed HTTP client，调用 provider adapter，再映射成稳定 PredictionResponse；测试用 MockTransport 和依赖覆盖隔离外部网络。
