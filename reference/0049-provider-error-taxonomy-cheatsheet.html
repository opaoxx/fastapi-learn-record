Reference 0049

Provider Error Taxonomy 速查

Provider 错误分类让 retry、fail fast、日志事件和稳定错误码使用同一套语言。

核心分类

timeout
request_error
retryable_http_status
non_retryable_http_status
invalid_response

HTTP 状态分类

RETRYABLE_PROVIDER_STATUS_CODES = {429, 500, 502, 503, 504}

def classify_provider_http_status(status_code: int):
    if status_code in RETRYABLE_PROVIDER_STATUS_CODES:
        return "retryable_http_status"
    return "non_retryable_http_status"

分类总表

类别 例子 策略
timeout 等待 provider 超时 有限重试
request_error 连接失败、网络错误 有限重试
retryable_http_status 429、500、502、503、504 有限重试
non_retryable_http_status 400、401、404、422 fail fast
invalid_response 非法 JSON、字段越界 fail fast

类别 vs 事件

category = what failed
event = what the adapter did

retryable_http_status + attempts left -> retry_scheduled
retryable_http_status + no attempts left -> retry_exhausted
non_retryable_http_status -> fail_fast
invalid_response -> fail_fast

稳定错误码

  • ai_provider_timeout：timeout。
  • ai_provider_request_error：request error。
  • ai_provider_http_error：HTTP 状态码错误。
  • ai_provider_invalid_response：响应体不符合内部模型。

测试保护

assert classify_provider_http_status(429) == "retryable_http_status"
assert classify_provider_http_status(503) == "retryable_http_status"
assert classify_provider_http_status(400) == "non_retryable_http_status"
assert classify_provider_http_status(404) == "non_retryable_http_status"

常见错误

  • 只说 provider error，不继续分类。
  • 把错误类别和日志事件混用。
  • 把所有 HTTP 错误都重试。
  • 把所有 4xx 都当成调用方一定有错。
  • 分类名没有测试，后续重构随意改散。

面试短答

我会把 provider 失败分成 timeout、request error、retryable HTTP status、non-retryable HTTP status 和 invalid response。分类之后再决定 retry、exhausted 或 fail fast；对外保持稳定错误码，对内用结构化日志保留诊断细节。

来源：HTTPX exceptions、 Python logging、 pytest logging。
