Reference 0050

Provider Metrics Basics 速查

Metrics 用低基数标签统计一段时间内的数量、比例和趋势；日志负责解释单次请求细节。

Logs vs Metrics

信号 适合回答 例子
Logs 这一次请求发生了什么 attempt、delay、Retry-After
Metrics 一段时间发生了多少次 success/fail_fast 计数

低基数 Label

good labels:
  outcome
  failure_category
  error_code
  status_code

bad labels:
  user text
  full provider response body
  full exception message
  request id

ProviderMetricLabels

ProviderMetricLabels(
    operation="prediction",
    outcome="fail_fast",
    failure_category="non_retryable_http_status",
    error_code="ai_provider_http_error",
    status_code="400",
)

成功默认值

build_provider_metric_labels(outcome="success").as_dict()

{
  "operation": "prediction",
  "outcome": "success",
  "failure_category": "none",
  "error_code": "none",
  "status_code": "none",
}

失败标签

build_provider_metric_labels(
    outcome="fail_fast",
    failure_category="non_retryable_http_status",
    error_code="ai_provider_http_error",
    status_code=400,
)

测试重点

assert labels.as_dict()["outcome"] == "fail_fast"
assert labels.as_dict()["status_code"] == "400"
assert "FastAPI is great and pleasant." not in labels.as_dict().values()

常见错误

  • 把用户输入放进 label。
  • 把完整异常消息放进 label。
  • 没有统一使用 none 默认值。
  • label 名称随重构变化。
  • 还没设计 label 就先接监控平台。

面试短答

日志解释单次 provider 请求，指标统计一段时间内的趋势。metrics label 必须低基数，所以我只放 outcome、failure_category、error_code、status_code 等稳定字段，不放用户输入、响应体或完整异常文本。
