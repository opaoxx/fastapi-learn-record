Reference 0057

Provider Metrics Snapshot Boundary 速查

snapshot 是内部 counter 状态的外部投影，不应该成为修改内部状态的入口。

四层边界

ProviderMetricsCounter._counts
  -> counter.snapshot()
  -> GET /provider/metrics
  -> GET /provider/metrics/prometheus

核心原则

读取函数也要保护封装。返回给外部的 snapshot 应该是一份数据表示，而不是内部可变对象。

关键代码

return [
    {
        "labels": labels.as_dict(),
        "count": count,
    }
    for labels, count in samples
]

labels.as_dict() 会生成新的字典，外部修改它不会改变原始 label 对象。

边界测试

snapshot = counter.snapshot()
snapshot[0]["count"] = 999
snapshot[0]["labels"]["outcome"] = "fail_fast"

assert counter.snapshot() == [
    {"labels": labels.as_dict(), "count": 1}
]

它保护什么

  • 防止调用方通过 snapshot 修改真实计数。
  • 防止导出层依赖内部 Counter 实现细节。
  • 防止未来重构把内部可变对象泄漏出去。
  • 让 JSON endpoint 和 text export 都建立在稳定快照上。

它不解决什么

  • 不解决多进程指标合并。
  • 不解决多实例聚合。
  • 不解决服务重启后的计数丢失。
  • 不替代 Prometheus 或 OpenTelemetry。

运行命令

.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q

面试短答

snapshot 是模块边界对象。它把内部 counter 投影成外部可读结构，但不能泄漏内部可变状态。测试可以故意修改返回的 snapshot，再确认真实 counter 仍然不变。

来源：Python Counter、 Python dataclasses、 FastAPI response models。
