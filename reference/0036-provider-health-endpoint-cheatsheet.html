Reference 0036

Provider Health Endpoint 速查

把外部 provider health check 接进 FastAPI，并把内部错误翻译成稳定 HTTP 契约。

核心链路

GET /provider/health
  -> get_settings()
  -> get_provider_http_client()
  -> check_provider_health(url, config, client)
  -> 200 ProviderHealthResponse
  -> 503 ErrorResponse

配置项

FIRST_API_AI_PROVIDER_HEALTH_URL="https://example.com"

health URL 来自服务端配置，不来自用户请求参数，避免把 API 做成可请求任意地址的代理。

响应模型

class ProviderHealthResponse(BaseModel):
    ok: bool
    provider: str
    status_code: int
    message: str

错误转换

except AIClientError as exc:
    raise HTTPException(
        status_code=503,
        detail=ErrorDetail(
            error_code=exc.error_code,
            message=exc.message,
        ).model_dump(),
    ) from exc

测试模式

replacement_client = httpx.AsyncClient(
    transport=httpx.MockTransport(handler),
    trust_env=False,
)
app.dependency_overrides[get_provider_http_client] = lambda: replacement_client

常见坑

  • 把 provider URL 放到用户可控查询参数里。
  • 让 endpoint 直接处理所有 HTTPX 细节。
  • 测试时真的访问外部网络。
  • 失败时泄漏内部异常，而不是返回稳定错误契约。

面试短答

Provider health endpoint 应该读取服务端配置中的 provider URL，使用 lifespan-managed HTTP client 发起检查，通过 adapter 隔离外部 HTTP 细节，并在 API 边界把内部错误转换成稳定的 503 JSON。测试时用 dependency_overrides 和 MockTransport 替换真实网络。

来源：FastAPI Handling Errors、 FastAPI Testing Dependencies、 HTTPX Transports、 MDN 503。
