Reference 0060

Provider Metrics Naming Refactor Boundary 速查

metrics rename 是兼容性迁移，不是普通变量重命名。

一句话模型

先盘点契约字段，再设计兼容窗口，最后迁移消费者。

契约字段

PROVIDER_PREDICTION_METRIC_LABEL_NAMES = (
    "operation",
    "outcome",
    "failure_category",
    "error_code",
    "status_code",
)

Contract snapshot

def get_provider_prediction_metric_contract() -> dict[str, object]:
    return {
        "metric_name": PROVIDER_PREDICTION_METRIC_NAME,
        "help": PROVIDER_PREDICTION_METRIC_HELP,
        "label_names": list(PROVIDER_PREDICTION_METRIC_LABEL_NAMES),
    }

测试模板

contract = get_provider_prediction_metric_contract()

assert contract == {
    "metric_name": "provider_prediction_total",
    "help": "Provider prediction outcomes recorded by the teaching metrics counter.",
    "label_names": list(PROVIDER_PREDICTION_METRIC_LABEL_NAMES),
}

安全改名五步

  1. 新增新名字。
  2. 并行导出旧新名字。
  3. 迁移 Dashboard、告警、文档和查询。
  4. 验证旧名字没有关键消费者。
  5. 下线旧名字并更新测试。

什么时候不要加旧别名

  • 没有真实外部消费者。
  • 只是教学项目里的契约讲解。
  • 新增别名会让课程代码复杂度高于收益。

运行命令

.\.venv\Scripts\python.exe -m pytest tests\test_provider_http.py -q

常见错误

  • 直接改 metric name 字符串。
  • 没有兼容窗口。
  • 忘记迁移 Dashboard 和告警。
  • 忘记文档和课程材料。
  • 改 label name 后没有意识到时间序列身份变化。

面试短答

metrics rename 应作为兼容性迁移处理。先盘点 metric_name、help、label_names，再并行导出旧新名字，迁移消费者，验证旧名字不再被依赖，最后下线。

来源：Prometheus metric and label naming、 Prometheus data model、 Prometheus instrumentation practices。
