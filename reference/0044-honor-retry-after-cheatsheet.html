Reference 0044

Honor Retry-After 速查

provider 给了可解析的重试时间提示时，优先听 provider，但仍受本地 cap 保护。

心智模型

local backoff = client estimate
Retry-After  = provider hint
final delay  = provider hint first, capped locally

两种格式

Retry-After: 120
Retry-After: Wed, 21 Oct 2015 07:28:00 GMT

解析结果

  • 合法秒数：返回秒数。
  • 合法 HTTP-date：返回目标时间与当前时间的差值。
  • 过去的 HTTP-date：返回 0.0。
  • 无效 header：返回 None，回退本地 backoff。

核心代码

provider_delay = parse_retry_after_delay(retry_after, now=now)
if provider_delay is not None:
    return min(provider_delay, max_delay)

return min(local_exponential_delay, max_delay)

HTTP 分支

await wait_before_retry(
    attempt,
    config,
    sleep,
    retry_after=exc.response.headers.get("Retry-After"),
)

测试清单

  • Retry-After: 5 解析为 5.0。
  • HTTP-date 用固定 now 计算差值。
  • Retry-After: 5 且 max delay 为 3 时，最终等待 3。
  • 无效 header 回退到本地指数退避。
  • 429 响应带 Retry-After: 2 时，fake sleep 记录 2.0。

常见坑

  • 把无效 header 当作 0 秒。
  • 完全不设置本地最大等待。
  • 用真实当前时间写日期测试。
  • 在 timeout 分支读取不存在的响应头。

面试短答

Retry-After 是 provider 的标准重试时间提示。我会在 429 或可重试 5xx 的 HTTP 响应分支读取它，支持秒数和 HTTP-date；可解析时优先使用它，并被本地 max delay 截断；不可解析时回退到本地指数退避。

来源：RFC 9110 Retry-After、 MDN Retry-After、 Python email.utils。
