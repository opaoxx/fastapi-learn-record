Reference 0053

Provider Metrics Failure Outcomes 速查

失败 metrics 要区分系统动作：安排重试、重试耗尽、快速失败。不要把所有失败压成一个模糊的 failed。

Outcome 表

outcome 含义 典型场景
retry_scheduled 可重试失败后安排下一次尝试 第 1 次 HTTP 503，且还有剩余尝试次数
retry_exhausted 最大尝试次数用完 连续 HTTP 503 后最终失败
fail_fast 错误不适合重试，立即停止 HTTP 400 或非法 provider response

测试流程

reset_provider_metrics()
client.post("/provider/predict", json=payload)
response = client.get("/provider/metrics")

assert response.status_code == 200
assert response.json() == expected_samples

关键规则

  • retry_scheduled 是中间事件，不等于最终失败。
  • retry_exhausted 是可重试错误的最终失败。
  • fail_fast 表示不应该继续尝试。
  • provider 原始状态码放在 metrics 的 status_code label 里。
  • 当前项目对外 HTTP 响应状态和 provider 原始状态不一定相同。

常见错误

  • 所有失败都叫 failed。
  • 把 provider 400 和本项目对外 503 混为一谈。
  • 请求 metrics endpoint 前没有先制造 provider 事件。
  • 测试失败路径时忘记重置 in-process counter。
  • 把异常全文、请求文本或 request id 放进 labels。

面试短答

失败 metrics 应该区分系统动作。retry_scheduled 表示系统正在恢复，retry_exhausted 表示重试已用完，fail_fast 表示继续尝试没有意义。这样排查时能分清临时抖动、最终不可用和契约错误。

来源：FastAPI testing、 HTTPX MockTransport、 Prometheus naming best practices。
