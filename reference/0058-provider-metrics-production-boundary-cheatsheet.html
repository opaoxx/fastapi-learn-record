Reference 0058

Provider Metrics Production Boundary 速查

当前 provider metrics 是教学级 in-process counter，不是生产级多实例 metrics pipeline。

一句话模型

教学实现可以模拟生产输出形状，但必须诚实标注生产缺口。

当前实现能做什么

  • 记录 provider prediction outcome。
  • 暴露 GET /provider/metrics JSON snapshot。
  • 暴露 GET /provider/metrics/prometheus text export。
  • 用测试保护格式、转义、失败 outcome 和 snapshot 边界。

当前实现不解决什么

缺口 原因
多进程 每个 worker 有自己的内存 counter。
多实例 单个 endpoint 不能代表所有机器或容器。
重启连续性 内存计数会随进程重启归零。
长期分析 需要时间序列存储和查询系统。
告警 需要规则、阈值、通知和运行平台。

OpenAPI 边界测试

json_operation = schema["paths"]["/provider/metrics"]["get"]
text_operation = schema["paths"]["/provider/metrics/prometheus"]["get"]

assert "in-process provider prediction metric counter snapshot" in json_operation["description"]
assert "not for production multi-instance aggregation" in json_operation["description"]
assert "educational export shape" in text_operation["description"]
assert "not a production multi-instance metrics pipeline" in text_operation["description"]

运行命令

.\.venv\Scripts\python.exe -m pytest tests\test_openapi_contract.py -q

常见错误

  • 把 Prometheus text 风格输出等同于生产 Prometheus 接入。
  • 把单进程 count 当成全局 count。
  • 忽略重启后内存计数丢失。
  • API 文档不写清用途限制。
  • 不测试文档里的生产边界说明。

面试短答

当前实现是教学级 metrics path：能展示 outcome、snapshot 和 text export，但不解决多 worker、多实例、重启、scrape、长期存储、告警和权限。边界应该写进 OpenAPI，并由契约测试保护。
