Reference 0062

Provider Metrics Runbook Checklist 速查

先看入口，再按 outcome 分支排查，最后追溯日志和测试。

一句话模型

metrics 告诉你发生了什么；runbook 告诉你下一步查什么。

观测入口

  • GET /provider/metrics：优先用于人工排查 JSON snapshot。
  • GET /provider/metrics/prometheus：用于检查 provider_prediction_total 的文本导出形状。

Outcome 判断

outcome 下一步
success 确认成功样本是否符合请求量。
retry_scheduled 看失败类别、状态码和 retry delay log。
retry_exhausted 确认最大尝试次数、provider 可用性和最终错误。
fail_fast 区分非重试 HTTP 错误和无效响应体。

五步流程

  1. 读 /provider/metrics。
  2. 按 outcome 分组。
  3. 失败时看 failure_category、error_code、status_code。
  4. retry 相关时回看日志和 provider HTTP 测试。
  5. 需要确认导出格式时读 /provider/metrics/prometheus。

测试命令

.\.venv\Scripts\python.exe -m pytest tests\test_course_docs_contract.py -q

常见误判

  • retry_scheduled 不等于最终失败。
  • fail_fast 不应默认靠增加重试解决。
  • 教学 counter 不是生产全局指标。
  • 只看 count 不看 label 会丢掉原因。
来源：Prometheus instrumentation practices、 Prometheus alerting practices、 Google SRE Book - Monitoring Distributed Systems。
