Reference 0042

Provider Retry Policy 速查

retry 是有限预算下的错误分类策略，不是无脑循环。

心智模型

retry policy =
  which failures can be retried
  + which failures must fail fast
  + how many total attempts are allowed

可重试状态码

RETRYABLE_PROVIDER_STATUS_CODES = {429, 500, 502, 503, 504}

本项目规则

情况 是否重试
httpx.TimeoutException 是，直到 max_attempts
httpx.RequestError 是，直到 max_attempts
429/500/502/503/504 是，直到 max_attempts
400/401/403/404 否，立刻失败
非法 JSON 或 Pydantic 校验失败 否，立刻失败

核心代码形状

for attempt in range(1, config.max_attempts + 1):
    try:
        response = await active_client.post(...)
        response.raise_for_status()
        body = ProviderPredictionBody.model_validate(response.json())
        return PredictionResult(...)
    except httpx.HTTPStatusError as exc:
        status_code = exc.response.status_code
        if attempt < config.max_attempts and is_retryable_provider_status(status_code):
            continue
        raise AIClientError(...) from exc

职责位置

  • adapter：识别 HTTPX 异常、HTTP 状态码、provider 响应体。
  • service：选择 demo/provider，编排预测用例，包装 service 错误。
  • router：把 service 成功或失败翻译成 HTTP 响应、headers、日志。

测试清单

  • 第一次 503、第二次 200：应该成功，调用 2 次。
  • 连续 503 且 max_attempts=3：应该失败，调用 3 次。
  • 400：应该失败，调用 1 次。
  • 非法响应体：应该失败，调用 1 次。

常见坑

  • 把所有异常都重试。
  • 没有最大尝试次数。
  • 在 router 里处理 provider retry 细节。
  • 忘记 POST 可能有副作用或计费成本。

面试短答

我会在 provider adapter 层做有限 retry：超时、网络错误、429 和常见临时 5xx 可以重试到 max_attempts；400、认证错误、权限错误、路径错误和非法响应体不重试。这样既提升临时故障下的成功率，又不会掩盖确定性错误。

来源：HTTPX Exceptions、 HTTPX Timeouts、 RFC 6585、 RFC 9110。
