Runbook

Provider Metrics 排障清单

这是一张教学项目里的排障 runbook：看到 provider prediction metrics 变化时，先确认观测入口，再按 outcome 分类定位原因。当前指标是 teaching signal, not a paging alert。

1. 观测入口

入口 看什么 什么时候用
GET /provider/metrics JSON snapshot，方便手工检查 labels 和 count。 课堂实验、测试排查、确认某个 outcome 是否出现。
GET /provider/metrics/prometheus provider_prediction_total 的 Prometheus text-style rendering。 观察 HELP、TYPE、label 格式和 text/plain export 形状。

2. Outcome 排查表

outcome 通常说明 下一步
success provider prediction 请求完成并返回有效响应。 确认成功样本是否和预期测试或请求数量一致。
retry_scheduled 出现 timeout、request error、429 或临时 5xx，系统准备重试。 看 failure_category、status_code 和 retry delay log。
retry_exhausted 可重试失败已经用完最大尝试次数。 确认 max_attempts、provider 可用性和最后一次错误类别。
fail_fast 非重试型 HTTP 错误或 provider 响应体无效。 区分 non_retryable_http_status 与 invalid_response。

3. 五步排查流程

  1. 先读取 /provider/metrics，确认 snapshot 是否为空。
  2. 按 outcome 分组，先看是否只有 success，再看失败 outcome。
  3. 失败时读取 failure_category、error_code 和 status_code。
  4. 如果是 retry 相关 outcome，再回看 retry scheduled、retry exhausted 和 fail-fast 日志测试。
  5. 需要检查导出格式时，再读取 /provider/metrics/prometheus，确认 text export 和 label 转义。

4. 相关测试

  • tests/test_provider_http.py：保护 outcome label、counter、snapshot、text rendering、retry/fail-fast 分类。
  • tests/test_ai_client_dependency.py：保护 provider metrics endpoint 在集成路径里的行为和测试隔离。
  • tests/test_course_docs_contract.py：保护本 runbook 必须保留的排障入口和 outcome 分类。

5. 训练评分入口

完成题卡后，可以用 grade_provider_metrics_runbook_exercise_answer() 检查作答是否覆盖 expected_severity 和 expected_next_action 两个锚点。该评分只服务教学自测，不代表生产演练平台的完整评分。

如果需要复盘多题训练结果，再使用 summarize_provider_metrics_runbook_exercise_grades() 统计 total、answered、passed、failed 和 unanswered。

如果需要把训练结果贴到 GitHub 或 PR 中，再使用 render_provider_metrics_runbook_grading_summary_markdown() 生成 Markdown 报告。

渲染报告前，可以用 validate_provider_metrics_runbook_grading_summary() 检查 summary 字段、grade 明细和统计一致性。

6. 不要误用

  • 当前 counter 是进程内教学实现，不代表生产全局计数。
  • 当前 runbook 是学习排障顺序，不是值班升级策略。
  • 看到 retry_scheduled 不等于用户请求最终失败，它只说明发生过一次可重试失败。
  • 看到 fail_fast 时不要盲目增加重试次数，先确认错误是不是本来就不该重试。
