Reference 0046

Retry Observability 速查

重试可观测性的目标：让每次“准备重试”都有稳定事件名和可测试的诊断字段。

心智模型

retry policy        = should we try again?
retry delay         = how long should we wait?
retry observability = why did we wait, and how can we prove it?

日志事件

provider_prediction_retry_scheduled

推荐字段

  • attempt：当前失败发生在第几次尝试。
  • max_attempts：最多允许尝试几次。
  • retry_reason：timeout、http_status 或 request_error。
  • status_code：HTTP 状态失败时记录，例如 429、503。
  • retry_after：provider 返回的 Retry-After 原始值。
  • delay_seconds：本次重试前最终等待秒数。
  • delay_source：retry_after 或 local_backoff。

核心类型

RetryDelaySource = Literal["retry_after", "local_backoff"]

@dataclass(frozen=True)
class RetryDelayPlan:
    delay_seconds: float
    source: RetryDelaySource

logging extra

logger.info(
    "provider_prediction_retry_scheduled",
    extra={
        "attempt": attempt,
        "delay_seconds": delay_plan.delay_seconds,
        "delay_source": delay_plan.source,
    },
)

caplog 断言

caplog.set_level(logging.INFO, logger="first_api.services.provider_http")

retry_records = [
    record
    for record in caplog.records
    if record.getMessage() == "provider_prediction_retry_scheduled"
]

assert retry_records[0].delay_source == "retry_after"

边界规则

  • 公共 JSON 响应保持稳定，retry 细节优先放日志。
  • 日志消息名保持稳定，变化数据放 extra 字段。
  • 不要记录 API key、完整敏感请求体或用户隐私文本。
  • 在 sleep 前记录日志，保留即将等待的事实。

面试短答

我会在 provider adapter 边界记录 retry 的结构化日志，包括 attempt、原因、状态码、Retry-After、等待秒数和等待来源。这样既不污染公共响应体，又能在日志系统或测试中解释慢请求和 provider 不稳定问题。

来源：Python logging、 pytest logging、 HTTPX exceptions。
