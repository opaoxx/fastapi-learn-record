Reference 0030

Schema 演进与兼容性速查

判断 API 变更时先问：旧客户端不改代码，还能不能继续工作？

核心心智模型

API schema = 调用方依赖的数据形状
兼容变更 = 旧客户端不用改也能继续工作
破坏性变更 = 旧客户端可能解析失败或语义出错

兼容性判断

变更 判断
新增可选响应字段 通常兼容
新增可选查询参数且默认行为不变 通常兼容
新增必填请求字段 不兼容
删除响应字段 不兼容
字段改名 不兼容
字段类型改变 不兼容
默认行为改变 通常不兼容

任务列表契约

GET /tasks?limit=5&offset=0

{
  "items": [],
  "count": 0,
  "limit": 5,
  "offset": 0
}

保护响应形状

payload = response.json()
assert isinstance(payload, dict)
assert set(payload) == {"items", "count", "limit", "offset"}
assert isinstance(payload["items"], list)

变更流程

  1. 写出现有契约。
  2. 判断兼容或不兼容。
  3. 兼容变更补测试和文档。
  4. 不兼容变更优先寻找兼容方案。
  5. 必须不兼容时新增版本或新路径。
  6. 标记弃用，给迁移窗口。

常见坑

  • 把字段改名当成内部重构。
  • 新增必填参数导致旧客户端 422。
  • 只测业务成功，不测 JSON 形状。
  • 只有 v2，没有迁移说明和保留时间。
