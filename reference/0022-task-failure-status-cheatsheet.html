Reference 0022

任务失败状态速查

202 表示任务被接收，不表示任务最终成功。最终结果写在任务记录里。

核心心法

  • 同步接口：本次 HTTP 响应可以直接表达成功或失败。
  • 后台任务：提交响应已经发出，最终结果要通过任务状态查询。
  • failed 是业务结果，不是提交接口的 HTTP 状态码。

状态流转

queued
  -> running
    -> completed
    -> failed

字段组合

completed -> result 有值, error 为 null
failed    -> result 为 null, error 有值
queued    -> result/error 通常都为空
running   -> result/error 通常都为空

失败处理骨架

except AIClientError as exc:
    task.status = "failed"
    task.error = f"{exc.error_code}: {exc.message}"
except Exception:
    task.status = "failed"
    task.error = "unexpected_task_error: The summary task failed unexpectedly."
finally:
    session.add(task)
    session.commit()

常见坑

  • 把 202 Accepted 当作任务成功。
  • 异常分支忘记 commit()，导致状态没有落库。
  • 把 Python 原始异常直接暴露给前端。
  • failed 状态还保留旧 result，造成语义冲突。
来源：FastAPI Background Tasks、 MDN 202 Accepted。
